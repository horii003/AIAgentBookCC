## 2. 目的・役割定義

### 2.1 エージェントの目的
- 経費精算申請に必要な情報を対話形式で収集する（BRL-07）
- LLMの画像認識機能で領収書画像から情報を自動抽出し（BRL-16）、経費区分を自動判断する（BRL-17）
- TOOL-002bを用いてExcel申請書を自動生成し、申請書のチェックを実施する

### 2.2 主な責務
- **必須情報収集**: BRL-07の必須項目（店舗名・経費区分・金額・日付・領収書画像・業務目的）を対話で収集する
- **領収書自動抽出**: LLMの画像認識機能で領収書画像（D-009）から店舗名・金額・日付・品目を自動抽出する（BRL-16・ツール不使用）
- **経費区分自動判断**: 抽出した品目から経費区分を自動判断する（BRL-17）
- **申請期限チェック**: 経費発生日から90日以内であることを確認する（BRL-12）
- **申請書自動作成**: TOOL-002b（generate_expense_application）でExcel申請書を生成する（BRL-02）
- **申請書チェック**: 必須項目（JD-06/JD-07）・整合性（JD-08/JD-09）・ルール適合性（BRL-03）・上長承認要否（BRL-18）を順次チェックしてフィードバックする

### 2.3 非責務（やらないこと）
- **申請種別判断**: AG-001（オーケストレーター）の責務
- **交通費運賃計算**: AG-002の責務
- **申請書提出**: 社員（人）の責務（ACT-EXEC-02禁止。GRD-011）

## 3. 自律度・判断権限

### 3.1 自律度レベル

| レベル | 内容 | 適用範囲 |
|------|------|----------|
| 参照 | 社内申請ルールナレッジ（D-002）参照 | ACT-READ-01 |
| 実行（自動） | 必須情報充足判断・領収書自動抽出・経費区分自動判断・申請期限判定・申請書チェック・上長承認要否判定 | ACT-DEC-02, ACT-DEC-03, ACT-DEC-04, ACT-DEC-05 |
| 実行（承認必須） | 申請書自動作成・ファイル出力（社員確認後のみ実行） | ACT-GEN-01（COND-002）, ACT-EXEC-01（COND-004） |

### 3.2 判断権限の範囲
- **自律判断可能**: 必須情報充足判断・領収書情報自動抽出・経費区分自動判断・申請期限チェック・申請書チェック・上長承認要否判定
- **人間確認必要**: 自動抽出結果の確認（CF-003 Step 3）。申請書内容確認（HumanApprovalHook。COND-002/COND-004）。最終確認・提出
- **実行不可**: 交通費計算ツール（TOOL-001）の呼び出し。申請書提出（ACT-EXEC-02禁止。GRD-011）

## 4. 処理フロー設計

### 4.1 基本フロー（概要）

```text
開始（AG-001から委譲。invocation_stateで申請者名・申請日・session_id取得）
→ [CF-003: 経費精算情報収集]
  → [不足項目確認（BRL-07）（Step 1）]
  → [領収書画像受付（Step 2）]
  → [領収書自動抽出・経費区分自動判断（LLM画像認識）・確認提示（Step 3）]
  → [全必須情報充足確認（JD-04）→ 未充足: 不足確認ループへ]
→ [申請期限チェック（JD-11）（Step 4）→ 超過: 通知・停止]
→ [CF-004: 申請書ドラフト提示（Step 1）]
  → 収集済み申請情報をテキストとして整理・提示する（ツール呼び出しなし）
→ [HumanApprovalHook: 申請者承認待ち（BeforeToolCallEventでトリガー）（Step 2）]
  → [OK選択 → TOOL-002b（generate_expense_application）実行（Step 3）]
  → [修正選択 → CF-003（情報収集）へ戻る]
  → [キャンセル選択 → セッションTERMINATED・申請停止]
→ [CF-005: 申請書チェック（Step 1〜4）]
  → [必須項目チェック（JD-06/JD-07）]
  → [整合性チェック（JD-08/JD-09）]
  → [ルール適合性チェック（BRL-03）]
  → [上長承認要否チェック（BRL-18）]
→ [合格済み申請書提示（Step 5）]
→ 終了（AG-001へ完了を返す）
```

**フローの詳細**: 詳細設計書 agents/expense_agent.py を参照

### 4.2 フロー制御の考え方
- **原則的な処理順序**: 情報収集 → 期限チェック → ドラフト提示 → 承認（HumanApprovalHook） → 申請書生成（TOOL-002b） → チェックの逐次処理
- **処理をスキップする考え方**: 全必須情報が充足した場合（JD-04）は収集ループをスキップして申請期限チェックへ進む。領収書から全情報が自動抽出できた場合は追加質問をスキップする
- **同一処理を複数回実行**: 未充足項目がある間は不足確認ループを繰り返す。チェック不備時は修正ループを繰り返す
- **ドラフト提示とツール呼び出しの分離**: 収集済み申請情報はまずテキストとして提示する（ツール呼び出しなし）。HumanApprovalHookのBeforeToolCallEventが発火し、申請者がOKを選択した後にはじめてTOOL-002bを実行する。承認前にTOOL-002bを実行してはならない
- **Human-in-the-Loop承認の発生タイミング**: HumanApprovalHookによる承認は1回のみ発生する（CF-004ステップ内、TOOL-002b呼び出し直前のBeforeToolCallEventで1回のみ）

### 4.3 状態管理
- **セッション状態（D-006）**: SessionManager（session_id=invocation_stateから取得）でdata/sessions/に保存する。flow_step・収集済み申請情報・申請書ファイルパスを管理する
- **invocation_state**: AG-001から`{"applicant_name": str, "application_date": str, "session_id": str}`を受け取る
- **会話履歴**: SlidingWindowConversationManager（window_size=15）で管理する

## 6. ツール設計

### 6.1 利用ツール

AG-003が`tools`リストに登録するツールは業務機能（生成）に特化したツールのみとする。

- **TOOL-002b（TL-002b: generate_expense_application）**: CF-004 HumanApprovalHookによる申請者承認（OK）後に、D-005（収集済み申請情報）からExcel申請書（経費精算申請書）を生成する。HumanApprovalHookのBeforeToolCallEventで申請者承認を取得した後にのみ実行する（COND-002/COND-004）。承認前にTOOL-002bを実行してはならない

TOOL-001（交通費計算ツール）はAG-003の担当ではない（自律度・権限定義 7章 P-READ-002はAG-003に付与されない）。領収書画像認識はLLMの画像認識機能を直接使用する（ツール不使用）。

**ツールの詳細**: 申請書生成ツール基本設計書（BD-01b）を参照

## 7. プロンプト設計

### 7.1 システムプロンプトの目的
- AG-003の役割（経費精算申請の情報収集・領収書自動抽出・申請書作成・チェック）を定義する
- 社内申請ルールナレッジ（D-002）を埋め込み、必須項目・チェックルールを提供する
- 申請期限チェック（BRL-12）で実行時の日付を計算するために動的生成が必要

### 7.2 プロンプト構成要素
- **役割定義**: 経費精算申請エージェントとしての役割（情報収集・領収書自動抽出・経費区分自動判断・ドラフト提示・承認取得・申請書作成・チェック）
- **処理フロー**: CF-003〜CF-005の処理手順。ドラフト提示（テキスト提示のみ）→ HumanApprovalHook承認待ち → TOOL-002b実行の順序を明示する
- **判断プロトコル**: 必須情報充足判断（JD-04/JD-05）・申請期限チェック（JD-11）・各チェック基準（JD-06〜JD-09/JD-13）・経費区分自動判断（BRL-17）
- **社内申請ルールナレッジ埋め込み**: D-002をシステムプロンプトに直接埋め込む
- **動的情報**: 実行時の申請日（application_date）を埋め込む
- **エラー発生時の振る舞い**: ツールエラー・システム障害等のシステム系エラーが発生した場合、エラー内容を要約して呼び出し元エージェント（AG-001）に返す

**プロンプト全文**: 詳細設計書 agents/expense_agent.py 2.3.1章を参照

### 7.3 システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 動的生成関数 |
| 動的生成の理由 | CF-003 Step 4（申請期限チェック・BRL-12）でapplication_dateを基準とした90日以内判定が必要。invocation_stateから受け取るapplication_dateを実行時にプロンプトに埋め込む |
| 動的に埋め込む情報 | application_date（YYYY-MM-DD形式）・申請期限日（application_dateの90日前の日付） |

### 7.4 主要な設計意図
- **LLM画像認識の活用**: 領収書自動抽出（BRL-16）はStrands Agentsのフレームワーク組み込み画像認識機能をLLMで直接実施する（専用ツール不要）
- **経費区分判断**: BRL-17の経費区分自動判断はプロンプトによるLLM判断で実施する。判断不可時は「その他経費」として扱う
- **ナレッジ埋め込み方式**: D-002をRAGではなくシステムプロンプトに直接埋め込む

## 13. Agent as Tools設計

### 13.1 ツールとしての公開
AG-003はAG-001からAgent as Toolsとして呼び出される。

- **ツール名**: expense_agent_tool
- **デコレータ**: @tool(context=True)
- **入力**: query (str)（委譲メッセージ：申請種別・申請意図テキスト）, tool_context (ToolContext)
- **出力**: str（AG-003からの最終応答：合格済み申請書の提示）

### 13.2 ToolContextの使用
- invocation_stateから`applicant_name`・`application_date`・`session_id`を取得する
- `session_id`を使用してSessionManagerを初期化する
- `applicant_name`・`application_date`をシステムプロンプトの動的生成に使用する
- 専門エージェント内部ではinvocation_stateに`applicant_name`・`application_date`（`session_id`除く）を設定する

**詳細**: 詳細設計書 agents/expense_agent.py 13章を参照

## 14. テスト観点

### 14.1 テスト観点の概要
- **機能テスト**: 領収書画像から情報が正しく抽出される。経費区分が正しく判断される。ドラフト提示時にTOOL-002bが呼び出されないこと。HumanApprovalHookがBeforeToolCallEventで発火し、OK選択後にTOOL-002bが実行される
- **Human-in-the-Loop承認テスト**: OK選択→TOOL-002b実行・ファイル生成。修正選択→情報収集ステップへ戻ること。キャンセル選択→セッションTERMINATED
- **承認前ツール実行禁止テスト**: HumanApprovalHookのOK前にTOOL-002bが実行されないこと（BeforeToolCallEventで制御されること）
- **異常系テスト**: 領収書画像の自動抽出失敗→手動入力を促す。申請期限91日超過→申請停止。5,001円以上→上長承認案内。ツールエラー発生時→AG-001へ要約を返す
- **性能テスト**: 要件上未定義

## 16. 制約事項・前提条件

### 16.1 技術的制約
- LLMバックエンドはAmazon Bedrock（Claude Sonnet 4.5）のみ
- Strands AgentsのAgent as Toolsパターンで実装する
- ストリーミング無効（`callback_handler=None`）
- 領収書画像認識はStrands Agentsフレームワークの画像読み取り機能（Pillow経由）を使用する

### 16.2 業務的制約
- 経費精算申請専用（交通費精算申請は担当しない）
- TOOL-001（交通費計算ツール）は使用しない（権限なし）
- 申請書提出は社員のみ実施可能（ACT-EXEC-02禁止）

### 16.3 前提条件
- invocation_stateに`applicant_name`・`application_date`・`session_id`が設定されていること
- 社内申請ルールナレッジ（D-002）がシステムプロンプトに埋め込まれていること
