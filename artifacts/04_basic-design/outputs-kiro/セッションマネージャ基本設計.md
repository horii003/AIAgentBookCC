## 2. 目的・役割定義

### 2.1 コンポーネントの目的
- セッション単位で会話状態（申請者名・申請日・申請種別・収集済み情報・フロー位置・対話回数）を永続化し、セッション再開を可能とする
- AG-001 が生成した session_id を invocation_state 経由で専門エージェントへ引き継ぎ、マルチエージェント間でのセッション共有を実現する
- セッションライフサイクル（CREATED/ACTIVE/WAITING/CLOSED/TERMINATED）を管理し、状態遷移を制御する

### 2.2 主な責務
- **セッション生成**: AG-001 がセッション開始時に UUID v4 形式の session_id を生成し、セッションストア（data/sessions/）に初期状態を保存する
- **セッション状態の更新**: 各エージェントからの更新要求を受け取り、セッションストアに反映する（applicant_name は上書き禁止）
- **セッション状態の読み込み**: session_id を指定してセッションストアからセッション状態を読み込む
- **セッション終了処理**: セッション終了（CLOSED/TERMINATED）時に機微情報（D-2）をマスキングし、セッションストアを更新する

### 2.3 非責務（やらないこと）
- **会話履歴の LLM への渡し方**: strands-agents の SlidingWindowConversationManager が担当する
- **例外処理**: ErrorHandler（HD-001）が担当する
- **ログ出力**: 各エージェント・ツールが担当する（SM-001 はセッション状態の管理のみ）

### 2.4 必要性
- セッション情報をローカルファイルに永続化することで、セッション再開・中断からの復旧を可能とする
- AG-001 と専門エージェント（AG-002・AG-003）が同一セッション情報を参照できるよう、session_id をキーとした一元管理が必要
- 対話回数上限（30回、BRL-16）の管理を一元化し、全エージェントで一貫した制御を実現する

### 2.5 主要な設計意図
- **ローカルファイル永続化**: セッション情報を `data/sessions/{session_id}/session.json` に保存し、セッション再開を可能とする（セッション管理方針 5.1節）
- **session_id によるディレクトリ分離**: 複数セッションの混在を防止するため、session_id をキーとしたディレクトリ構造で管理する
- **applicant_name の不変性**: アプリ起動時に取得済みの申請者名は上書き禁止とし、セッション中の変更を防止する（セッション管理方針 6.2節）

## 3. SessionManagerクラス

### 3.1 クラス概要
- **クラス名**: `SessionManager`
- **説明**: セッション状態の生成・読み込み・更新・終了処理を担当するクラス。`session/session_manager.py` に実装する

### 3.2 クラス変数
- **session_id**: セッション識別子（UUID v4 形式の文字列）（初期値: 生成時に設定）
- **storage_path**: セッションストアのベースパス（初期値: `"data/sessions/"`）
- **session_data**: 現在のセッション状態（SessionState モデルのインスタンス）（初期値: None）

**詳細**: 詳細設計書 1章を参照

## 4. 主要メソッド

### 4.1 create_session
- **目的**: 新規セッションを生成し、セッションストアに初期状態を保存する
- **引数**: `applicant_name: str`（申請者名）、`application_date: str`（申請日、YYYY-MM-DD）
- **戻り値**: `session_id: str`（生成した UUID v4 形式のセッション識別子）
- **フォーマット**: `data/sessions/{session_id}/session.json` に JSON 形式で保存する

### 4.2 load_session
- **目的**: session_id を指定してセッションストアからセッション状態を読み込む
- **引数**: `session_id: str`（セッション識別子）
- **戻り値**: `SessionState`（セッション状態モデルのインスタンス）または `None`（セッションが存在しない場合）
- **デフォルトパス**: `data/sessions/{session_id}/session.json`

### 4.3 update_session
- **目的**: セッション状態の特定フィールドを更新してセッションストアに保存する
- **引数**: `session_id: str`、`updates: dict`（更新するフィールドと値のマッピング）
- **戻り値**: `None`
- **注意**: `applicant_name` フィールドの更新は禁止（上書き禁止）

### 4.4 close_session
- **目的**: セッションを正常終了（CLOSED）または異常終了（TERMINATED）させ、機微情報をマスキングしてセッションストアを更新する
- **引数**: `session_id: str`、`status: Literal["CLOSED", "TERMINATED"]`
- **戻り値**: `None`
- **フォーマット**: 終了時に D-2（機微）情報をマスキングして保存する（セッション管理方針 12節）

### 4.5 increment_turn_count
- **目的**: 対話回数をインクリメントし、上限（30回）チェックを行う
- **引数**: `session_id: str`
- **戻り値**: `turn_count: int`（インクリメント後の対話回数）
- **上限到達時**: `turn_count >= 30` の場合は ErrorHandler（HD-001）に EX-05 を通知する（BRL-16）

**詳細**: 詳細設計書 2.1章〜2.5章を参照

## 5. セッション管理の仕組み

**システム設計書の方針**: `artifacts/03_system-design/outputs/セッション管理方針.md` を参照してください。

### 5.1 セッション生成
1. AG-001 が `create_session(applicant_name, application_date)` を呼び出す
2. SM-001 が UUID v4 形式の session_id を生成する
3. 初期状態（CREATED・turn_count=0・current_flow=CF-001）を `data/sessions/{session_id}/session.json` に保存する
4. session_id を AG-001 に返却する

### 5.2 セッション状態の更新
1. 各エージェントが `update_session(session_id, updates)` を呼び出す
2. SM-001 が `data/sessions/{session_id}/session.json` を読み込む
3. `applicant_name` 以外のフィールドを更新する（applicant_name は上書き禁止）
4. 更新後の状態をアトミック操作（一時ファイルへの書き込み後にリネーム）で保存する

### 5.3 セッション再開
1. アプリ起動時に `load_session(session_id)` を呼び出す
2. SM-001 が `data/sessions/{session_id}/session.json` を読み込む
3. セッション状態が CLOSED または TERMINATED の場合は再開不可（None を返す）
4. turn_count が30以上の場合は再開不可（None を返す）
5. 有効なセッション状態を返却し、current_flow から処理を再開する

### 5.4 セッション終了
1. 各エージェントが `close_session(session_id, status)` を呼び出す
2. SM-001 がセッション状態を CLOSED または TERMINATED に更新する
3. D-2（機微）情報（申請者名・申請内容・申請書内容）をマスキングする（GRD-014）
4. 更新後の状態をセッションストアに保存する

**詳細**: 詳細設計書 3章を参照

## 6. マルチエージェント連携

### 6.1 セッション共有の仕組み

**AG-001（オーケストレーター）**:
1. セッション開始時に `create_session()` を呼び出して session_id を生成する
2. session_id を invocation_state に含めて専門エージェントへ渡す
3. 申請種別確定後に `update_session()` で application_type を更新する
4. セッション終了時に `close_session()` を呼び出す

**AG-002・AG-003（専門エージェント）**:
1. invocation_state から session_id を取得する
2. `load_session(session_id)` でセッション状態を読み込む
3. 収集済み情報・フロー位置を `update_session()` で更新する（session_id は invocation_state に含めない）

### 6.2 会話履歴の共有範囲
- **共有される情報**: applicant_name・application_date・session_id・application_type・collected_info・current_flow・turn_count（セッションストア経由）
- **共有されない情報**: 各エージェントの LLM 会話ウィンドウ（SlidingWindowConversationManager が独立管理）・領収書画像の実体（処理完了後に削除）

- **パス**: `data/sessions/`

- **フォーマット**: JSON 形式（UTF-8 エンコーディング）
- **例**: `{"session_id": "...", "status": "ACTIVE", "applicant_name": "***", "application_date": "2026-04-19", "application_type": null, "current_flow": "CF-001", "turn_count": 0, "collected_info": {}}`

## 11. 詳細設計への引き渡し事項

### 11.1 実装時の識別子
- **クラス名**: `SessionManager`
- **実装ファイル**: `session/session_manager.py`

### 11.2 SDK実装時の注意事項
- セッションファイルの書き込みはアトミック操作（一時ファイルへの書き込み後にリネーム）を使用し、ファイル破損を防止する（セッション管理方針 14節）
- `applicant_name` フィールドの更新は `update_session()` 内で禁止チェックを実装する
- セッション終了時の機微情報マスキングは `close_session()` 内で実装する（GRD-014）
- `data/sessions/` ディレクトリが存在しない場合は自動作成する

### 11.3 モック／スタブ要否
- テスト時はセッションストアのパスをテスト用ディレクトリ（例：`tests/data/sessions/`）に変更する

## 13. 制約事項・前提条件

### 13.1 技術的制約
- セッションストアはローカルファイル（data/sessions/）のみ（外部ストレージ不使用）
- セッションファイルは JSON 形式（UTF-8 エンコーディング）
- ファイル書き込みはアトミック操作を使用する

### 13.2 業務的制約
- セッション終了（CLOSED/TERMINATED）時にセッションストアの機微情報をマスキングする（GRD-014）
- applicant_name は上書き禁止（セッション管理方針 6.2節）
- 対話回数上限は30回（BRL-16）

### 13.3 前提条件
- `data/sessions/` ディレクトリが書き込み可能であること
- SessionState モデル（models/data_models.py）が定義済みであること
