## 1. 目的・適用範囲

### 1.1 目的

システム全体で共通するバリデーション方針を定義し、データの整合性・型安全性・業務ルール適合性を一貫した方法で保証する。

### 1.2 適用範囲

- AG-001〜AG-003の全エージェント（`invocation_state` の型保証）
- TOOL-001（交通費計算ツールのパラメータ）
- TOOL-002（申請書生成ツールのパラメータ）
- models/data_models.py の全Pydanticモデル

### 1.3 本方針で定義すること

- バリデーションの基本原則とタイミング
- Pydanticモデルの使用方針（定義場所・命名規則・基本構造）
- データ型別バリデーション（日付・金額・文字列・列挙型）
- 業務ルールに基づくカスタムバリデーション
- バリデーションエラーメッセージ方針

### 1.4 本方針で定義しないこと

- Pydanticモデルの具体的なフィールド定義（データモデル基本設計.mdで定義する）
- バリデーション処理の実装詳細（ツール詳細設計.mdで定義する）

## 2. バリデーションの基本原則

### 2.1 基本方針

- **早期検証**: システム境界（ツール関数入口・`invocation_state` 受取時）で必ずバリデーションを実施し、不正データの伝播を防止する
- **型安全性**: 全入力データにPydanticモデルを適用し、型の不一致を実行前に検出する
- **ユーザーフレンドリー**: ValidationError発生時は日本語のエラーメッセージで再入力を促す
- **自動正規化**: 駅名の接尾語除去（BRL-15）・交通手段表記の正規化等を `field_validator` で自動変換する
- **一貫性**: 同一フィールドには同じバリデーターを共通関数として定義し、複数モデルで再利用する

### 2.2 バリデーションのタイミング

| タイミング | 対象 | 実装場所 |
|----------|------|---------|
| ツール関数入口 | TOOL-001/TOOL-002のパラメータ | `tools/transport_tools.py`・`tools/output_generator.py` のPydanticモデル |
| `invocation_state` 受取時 | エージェント状態（申請者名・申請日等）。専用の Pydantic モデルは定義しない。辞書リテラルで受け取り、必要なフィールド（`applicant_name`・`application_date`）の存在確認とNone/空文字チェックのみ実施する | `agents/` 配下の各エージェントクラス |
| アプリケーション起動時（静的読み込み） | DATA-009（電車経路・運賃）・DATA-010（固定運賃）の運賃データ。起動時に静的読み込みを行い、読み込み失敗時は例外（EX-06）として処理する | `tools/transport_tools.py` 内部 |
| 申請書生成前 | 収集済み申請情報の完全性確認 | `tools/output_generator.py` 内部 |

### 2.3 バリデーション失敗時の扱い

**ValidationError発生時**:
- `ValidationError` をキャッチして `ErrorHandler.handle_validation_error()` に委譲する
- ErrorHandlerが日本語エラーメッセージを生成してユーザーに提示する
- スタックトレースはエラーログ（ERR-001等）にのみ記録し、ユーザーには非表示とする
- ツール関数は `{"success": False, "message": 日本語エラーメッセージ}` を返却する

## 3. Pydanticモデルの使用方針

### 3.1 基本原則

- Pydantic v2+ を使用する
- `BaseModel` を継承して全モデルを定義する
- `field_validator` でカスタムバリデーション・正規化を実装する
- モデルカテゴリ（マスタデータ・ツール入力・エージェント状態・出力生成）ごとに分類して定義する

### 3.2 Pydanticモデルの定義場所

**定義場所**: `models/data_models.py`（全モデルを集約する）

**命名規則**: 
- マスタデータモデル: `{データ名}Master`（例: `RailwayRouteMaster`）
- ツール入力モデル: `{ツール名}Input`（例: `FareCalculationInput`）
- 出力生成モデル: `{申請種別}ApplicationData`（例: `TransportApplicationData`）

> ※ `invocation_state` は辞書リテラルで管理する。専用の Pydantic モデル（`{エージェント略称}InvocationState` 等）は定義しない

### 3.3 Pydanticモデルの基本構造

```python
from pydantic import BaseModel, field_validator, Field
from typing import Optional, Literal
from datetime import date

class FareCalculationInput(BaseModel):
    departure: str = Field(..., description="出発地（正規化前でも可）")
    destination: str = Field(..., description="目的地（正規化前でも可）")
    transport_type: Literal["電車", "バス", "タクシー", "飛行機"]
    travel_date: Optional[date] = None

    @field_validator("departure", "destination")
    @classmethod
    def normalize_station_name(cls, v: str) -> str:
        # 駅名の接尾語を除去する（BRL-15）
        for suffix in ["駅前", "駅"]:
            if v.endswith(suffix):
                return v[:-len(suffix)]
        return v
```

## 4. 日付のバリデーション

### 4.1 基本原則

- 日付はPythonの `date` 型（`datetime.date`）で管理する
- 文字列入力（`YYYY-MM-DD` 形式）は `python-dateutil` で自動変換する
- 申請期限チェック（BRL-12）は `field_validator` または `model_validator` で実施する

### 4.2 日付の定義

**標準形式**: `YYYY-MM-DD`（ISO 8601）

**許容値**: 
- 移動日・経費発生日: 過去日付（申請日から90日以内）
- 申請日: 本日以前

### 4.3 日付バリデーションの実装

```python
from pydantic import field_validator
from datetime import date
from dateutil.parser import parse as parse_date

@field_validator("travel_date", "expense_date", mode="before")
@classmethod
def parse_date_field(cls, v):
    if isinstance(v, str):
        try:
            return parse_date(v).date()
        except Exception:
            raise ValueError("日付のエラー: 正しい日付形式（YYYY-MM-DD）で入力してください")
    return v
```

### 4.4 日付の特殊ルール

- **申請期限チェック（BRL-12）**: 経費発生日から申請日までが90日以内であること
  - `(申請日 - 経費発生日).days <= 90` を検証する
  - 90日超過の場合は `ValueError("申請のエラー: 申請期限（経費発生日から90日以内）を超過しています")` を発生させる

## 5. 金額のバリデーション

### 5.1 基本原則

- 金額は `int` 型（円単位の整数）で管理する
- 負の値・ゼロは許容しない
- 上長承認要否の閾値チェック（BRL-11/BRL-18）はバリデーションではなくビジネスロジックで実施する

### 5.2 金額の定義

**標準形式**: 正の整数（円単位）

**許容値**:
- 交通費（1区間あたり）: 1円以上の正整数
- 経費金額: 1円以上の正整数

### 5.3 金額バリデーションの実装

```python
@field_validator("fare", "amount")
@classmethod
def validate_positive_amount(cls, v: int) -> int:
    if v <= 0:
        raise ValueError("金額のエラー: 1円以上の金額を入力してください")
    return v
```

### 5.4 金額の特殊ルール

- **交通費上長承認要否（BRL-11）**: 交通費合計 > 10,000円 → 上長承認案内（バリデーションではなくビジネスロジックで対応）
- **経費上長承認要否（BRL-18）**: 経費金額 > 5,000円 → 上長承認案内（バリデーションではなくビジネスロジックで対応）
- **固定運賃の許容値**: バス230円・タクシー10,000円・飛行機50,000円（DATA-010から読み込むため、バリデーションではなくJSONスキーマ検証で保証する）

## 6. 文字列のバリデーション

### 6.1 基本原則

- 空文字・None は必須フィールドとして禁止する（GRD-001）
- 入力文字数は500文字以内に制限する（GRD-012）
- 前後の空白は `strip()` で自動除去する

### 6.2 文字列の定義

**標準形式**: 空白を除去した非空文字列

**許容値**:
- 申請者名: 1文字以上・500文字以内
- 申請意図テキスト: 1文字以上・500文字以内（GRD-012）
- 業務目的: 1文字以上・500文字以内

### 6.3 文字列バリデーションの実装

```python
@field_validator("applicant_name", "purpose", mode="before")
@classmethod
def validate_required_string(cls, v: str) -> str:
    if v is None or str(v).strip() == "":
        raise ValueError("入力のエラー: 必須項目が入力されていません。入力してください")
    stripped = str(v).strip()
    if len(stripped) > 500:
        raise ValueError("入力のエラー: 入力は500文字以内で入力してください")
    return stripped
```

## 7. カスタムバリデーションルール

### 7.1 列挙型フィールドのバリデーションと正規化

**交通手段のバリデーションと正規化（BRL-14）**:

LLMが渡す可能性のある表記バリエーション（「電車」「鉄道」「JR」等）を許容し、内部統一表記に正規化する。

```python
from typing import Literal
from pydantic import field_validator

transport_type: Literal["電車", "バス", "タクシー", "飛行機"]

@field_validator("transport_type")
@classmethod
def normalize_transport_type(cls, v: str) -> str:
    mapping = {
        "電車": "電車", "鉄道": "電車", "JR": "電車", "地下鉄": "電車", "電鉄": "電車",
        "バス": "バス", "路線バス": "バス",
        "タクシー": "タクシー", "taxi": "タクシー",
        "飛行機": "飛行機", "航空機": "飛行機", "airplane": "飛行機",
    }
    normalized = mapping.get(v)
    if normalized is None:
        raise ValueError(f"交通手段のエラー: 「{v}」は対応していません。電車・バス・タクシー・飛行機のいずれかを指定してください")
    return normalized
```

**経費区分のバリデーションと正規化（BRL-17）**:

```python
expense_category: Literal["事務用品費", "宿泊費", "資格精算費", "その他経費"]

@field_validator("expense_category")
@classmethod
def normalize_expense_category(cls, v: str) -> str:
    mapping = {
        "事務用品費": "事務用品費", "事務用品": "事務用品費",
        "宿泊費": "宿泊費", "宿泊": "宿泊費",
        "資格精算費": "資格精算費", "資格費": "資格精算費", "資格": "資格精算費",
        "その他経費": "その他経費", "その他": "その他経費",
    }
    normalized = mapping.get(v)
    if normalized is None:
        return "その他経費"  # 判断不可時は「その他経費」として扱う（BRL-17）
    return normalized
```

### 7.2 駅名正規化バリデーション（BRL-15）

```python
@field_validator("departure", "destination")
@classmethod
def normalize_station_name(cls, v: str) -> str:
    for suffix in ["駅前", "駅"]:
        if v.endswith(suffix):
            return v[:-len(suffix)]
    return v
```

### 7.3 申請種別のバリデーション

```python
application_type: Literal["交通費精算申請", "経費精算申請"]

@field_validator("application_type")
@classmethod
def normalize_application_type(cls, v: str) -> str:
    mapping = {
        "交通費精算申請": "交通費精算申請", "交通費": "交通費精算申請",
        "経費精算申請": "経費精算申請", "経費": "経費精算申請",
    }
    normalized = mapping.get(v)
    if normalized is None:
        raise ValueError(f"申請種別のエラー: 「{v}」は対応していません。交通費精算申請・経費精算申請のいずれかを指定してください")
    return normalized
```

## 8. バリデーションエラーメッセージ

### 8.1 基本原則

- **わかりやすさ**: 技術的な詳細を隠蔽し、社員が理解できる日本語で記述する
- **具体性**: 何が問題でどう対処すればよいかを明示する
- **一貫性**: フォーマットを統一し、全エラーメッセージで同じ形式を使用する

### 8.2 エラーメッセージの形式

**基本形式**:
```
{フィールド名}のエラー: {エラー内容}
対処方法: {対処方法}
```

### 8.3 エラーメッセージの実装

```python
from pydantic import ValidationError
from handlers.error_handler import ErrorHandler

_error_handler = ErrorHandler()

try:
    model = FareCalculationInput(**data)
except ValidationError as e:
    error_message = _error_handler.handle_validation_error(e)
    return {"success": False, "message": error_message}
```

**具体的なエラーメッセージ例**:
```
# 空入力エラー
「申請者名のエラー: 申請者名が入力されていません。
対処方法: 申請者名を入力してください。」

# 申請期限超過エラー
「申請のエラー: 申請期限（経費発生日から90日以内）を超過しています（経費発生日: YYYY-MM-DD、申請日: YYYY-MM-DD）。
対処方法: 担当部署へご相談ください。」

# 交通手段エラー
「交通手段のエラー: 「新幹線」は対応していません。
対処方法: 電車・バス・タクシー・飛行機のいずれかを選択してください。」

# 金額エラー
「金額のエラー: 金額に0円以下の値は入力できません。
対処方法: 1円以上の正しい金額を入力してください。」
```

## 9. 設計上の留意点・リスク

### 9.1 列挙型の表記揺れ

**リスク**: LLMが想定外の表記（英語・略称等）で交通手段・経費区分を渡した場合、バリデーションエラーが発生するリスクがある

**対策**: `field_validator` の正規化マッピングに想定される表記バリエーションを網羅する。新たな表記が発生した場合は随時マッピングを追加する

### 9.2 日付フォーマットの多様性

**リスク**: LLMが `YYYY/MM/DD` や `MM/DD/YYYY` 等のフォーマットで日付を渡した場合に変換失敗するリスクがある

**対策**: `python-dateutil` の `parse` 関数を `mode="before"` の `field_validator` で使用して、主要な日付フォーマットを自動解析する
