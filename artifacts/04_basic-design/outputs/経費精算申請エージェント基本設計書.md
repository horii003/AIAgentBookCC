## 2. 目的・役割定義

### 2.1 エージェントの目的
- AG-001 から委譲された経費精算申請書作成フロー全体を担当し、社員との対話を通じて経費情報を収集・確認し、最終的に経費精算申請書（下書き）を生成する
- Strands SDK 組み込みの LLM 推論（image_reader）を使用して領収書画像から経費情報を自動抽出（OCR）し、社員の手入力負荷を軽減する
- 品目名称から経費区分（事務用品費・宿泊費・資格精算費・その他経費）を自動判断し（BRL-09）、判断困難な場合は社員に選択を委ねる
- 生成した申請書に対して自動チェック（EVT-001/FR-010）を実施し、業務ルール違反を申請前に検出する

### 2.2 主な責務
- **経費情報の収集**: CF-003 フローに沿って不足情報を対話で収集する（BRL-02）。必要情報一覧（D-005）と照合して未収集項目を特定する
- **領収書OCR情報抽出**: 社員がアップロードした領収書画像から金額・支払先・経費発生日を自動抽出する（BRL-08）。LLM 組み込みの画像認識機能を利用する
- **経費区分の自動判断**: 品目名称から経費区分（BRL-09/JD-07/JD-17）を自動判断する。判断困難な場合は社員に4択の選択肢を提示する
- **高額申請の案内**: 5,000円を超える費用がある場合は事前上長承認を案内する（BRL-12/JD-11）
- **APR-001 承認の取得**: 全情報収集完了後、HumanApprovalHook 経由で社員の申請書生成前確認を取得する（ABAC-001）
- **申請書の生成**: APR-001 承認後に TOOL-002（generate_expense_application）を呼び出して申請書を生成する（BRL-07）
- **申請内容の自動チェック**: 申請書生成後に EVT-001 として申請内容チェック（FR-010）を自動実施する（BRL-03/BRL-16）

### 2.3 非責務（やらないこと）
- **申請者名・申請日の収集**: AG-001 が収集し invocation_state で引き継ぐ
- **申請種別の判断**: AG-001 が実施する（委譲時点で経費精算申請と確定済み）
- **移動情報（交通費）の収集**: AG-002 が実施する
- **交通費の計算**: AG-002 が TOOL-001 経由で実施する

## 3. 自律度・判断権限

### 3.1 自律度レベル

| レベル | 内容 | 適用範囲 |
|------|------|----------|
| 参照 | 社内申請ルール（D-002）・申請書作成必要情報一覧（D-005）の参照 | 不足情報特定・経費区分判断時 |
| 提案 | 経費区分の判断困難時の選択肢提示（JD-07/JD-17） | BRL-09 で経費区分を特定できない場合 |
| 実行 | 不足情報収集・OCR実行・経費区分自動判断・申請内容チェック（EVT-001） | 情報収集・OCR・経費区分判断・チェック |
| 実行（承認必須） | 申請書生成（TOOL-002 呼び出し） | APR-001 承認後のみ（HumanApprovalHook） |

### 3.2 判断権限の範囲
- **自律判断可能**: 不足情報の特定（JD-05）、OCR実行タイミング（CF-003 Step 6）、経費区分自動判断（BRL-09）、申請期限チェック（BRL-11）、高額判定（JD-11）
- **人間確認必要**: APR-001（申請書生成前の社員確認）、経費区分の判断困難時の社員選択（JD-17）、OCR失敗時の手動入力
- **実行不可**: APR-001 承認なしでの TOOL-002 実行（EX-05）

## 4. 処理フロー設計

### 4.1 基本フロー（概要）

```text
開始（AG-001からの委譲受付）
  → invocation_stateから申請者名・申請日を取得
  → 不足情報収集ループ（BRL-02: JD-05で不足項目を特定）
    → 不足情報の対話ヒアリング（申請者名・業務目的・経費発生日・金額・店舗名・品目等）
    → 領収書画像アップロード確認（任意）
      → 画像あり: OCR自動抽出（BRL-08）
        → 抽出成功: 経費情報に反映
        → 抽出失敗: 再アップロード促し/手動入力案内
      → 画像なし: スキップ
    → 経費区分自動判断（BRL-09/JD-07/JD-17）
      → 判断成功: 経費区分を自動設定
      → 判断困難: 社員に4択選択肢を提示
    → 申請期限チェック（BRL-11: 購入日から90日以内）
    → 高額判定（5,000円超: BRL-12案内）
    → 追加経費の有無確認
  → 全経費収集完了
  → 収集済み申請情報をテキストとして整理・提示（ドラフト確認用。ツール呼び出しなし）
  → APR-001承認取得（HumanApprovalHook / BeforeToolCallEvent）
    → OK: はじめてTOOL-002呼び出し（申請書生成）※承認前にTOOL-002を実行してはならない
    → 修正: 修正対象を再収集してAPR-001再実施
    → キャンセル: セッションをTERMINATEDに遷移
  → 申請書生成完了
  → EVT-001: 申請内容自動チェック（FR-010/BRL-03/BRL-16）
  → 結果をAG-001に返却
  → 終了
```

**フローの詳細**: 詳細設計書 2.3.1章を参照

### 4.2 フロー制御の考え方
- **原則的な処理順序**: 不足情報収集 → OCR（任意）→ 経費区分判断 → 期限・高額チェック → 追加経費確認の逐次処理。全経費収集完了後に APR-001 を実施する
- **処理をスキップする考え方**: 領収書画像がない場合は OCR ステップをスキップして手動入力フローを継続する（CF-003 Step 5〜6）。経費区分の自動判断が成功した場合は社員への選択肢提示をスキップする
- **同一処理を複数回実行**: 経費件数分だけ情報収集ループを繰り返す。追加経費の意思確認を各経費完了後に実施する
- **ドラフト提示とツール呼び出しの分離**: 全経費収集完了後、収集済み申請情報をテキストとして整理・提示する（この時点ではツールを呼び出さない）。その後 APR-001 承認（HumanApprovalHook / BeforeToolCallEvent）を取得し、承認（OK）後にはじめて TOOL-002 を呼び出す。Human-in-the-Loop 承認は1回のみ発生する
- **承認前のツール実行禁止**: APR-001 承認なしで TOOL-002 を実行してはならない（ABAC-001）

### 4.3 状態管理
- **経費情報リスト（D-007）**: `List[ExpenseItem]` として AG-003 のメモリ内で保持。各経費収集完了ごとに追加する。session_{id}.json にも反映する
- **invocation_state（applicant_name / application_date）**: ToolContext 経由で AG-001 から受け取り、TOOL-002 入力に使用する
- **APR-001 承認済みフラグ**: session_{id}.json の `approval_granted` フィールドに記録

## 6. ツール設計

### 6.1 利用ツール

**ツール割り当て方針**: AG-003 が使用するツールは業務機能（生成）に特化したツールのみとする。

- **TOOL-002（generate_expense_application）**: 全経費収集完了・APR-001 承認後に呼び出す。申請者名・申請日・経費情報リストを渡して経費精算申請書 Excel ファイルを生成する。**承認（OK）後にはじめて呼び出すこと。承認前に呼び出してはならない**

TOOL-002 は APR-001 承認後に1回のみ呼び出す（CF-005 Step 1）。

> **注記**: 領収書OCR（BRL-08）は Strands SDK 組み込みの LLM 推論（画像認識機能）で実現する。外部ツール（機能ツール一覧.md 登録外）のため、本ツール設計セクションへの記載対象外とする。

**ツールの詳細**: 詳細設計書 2.4.2章および申請書生成ツール基本設計書を参照

## 7. プロンプト設計

### 7.1 システムプロンプトの目的
- 経費精算申請書作成フローの手順（CF-003/CF-005）をエージェントに定義する
- 不足情報対話収集（BRL-02）・OCR情報抽出（BRL-08）・経費区分自動判断（BRL-09）・申請期限チェック（BRL-11）・高額判定（BRL-12）をプロンプト内で指示する
- 申請書生成後の自動チェック（EVT-001/BRL-03/BRL-16）の手順を定義する

### 7.2 プロンプト構成要素
- **役割定義**: 経費精算申請書作成専門エージェントとしての役割
- **処理フロー**: CF-003 Step 1〜10 / CF-005 Step 1〜3 の手順
- **判断プロトコル**: JD-05（不足情報特定）、JD-07/JD-17（経費区分判断）、JD-09（申請期限チェック）、JD-11（高額判定）の手順
- **主な責任**: BRL-02（対話収集）、BRL-07（全情報収集後の申請書生成）、BRL-08（OCR情報抽出）、BRL-09（経費区分自動判断）、BRL-13（業務目的必須）
- **動的情報（本日の日付）**: 申請期限チェック（BRL-11）で「購入日から90日以内」を判定するため、現在日付を実行時に埋め込む
- **エラー発生時の振る舞い**: システム系エラー（ツールエラー・システム障害等）が発生した場合は、呼び出し元エージェント（AG-001）に要約メッセージを返す

**プロンプト全文**: 詳細設計書 2.3.1章を参照

### 7.3 システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 動的生成関数（呼び出しのたびに関数を実行して生成する） |
| 動的生成の理由 | 申請期限チェック（BRL-11）で購入日から90日以内であることを確認するため、現在日付（date.today()）をプロンプトに埋め込む必要がある |
| 動的に埋め込む情報 | 本日の日付（`date.today().isoformat()`）、申請期限の基準日（90日前の日付） |

### 7.4 主要な設計意図
- **動的生成の採用**: 申請期限チェック（BRL-11）はシステム日付に依存する業務ルールであるため、プロンプトに現在日付を埋め込んで LLM が正確な期限判断を行えるようにする
- **OCR処理の組み込み**: BRL-08 に従い、領収書画像がアップロードされた場合の自動抽出手順をプロンプトで明示する。OCR失敗時の手動入力フォールバックも定義する
- **経費区分判断の明示**: BRL-09 に従い、品目名称から経費区分を判断する手順とフォールバック（社員選択肢提示）をプロンプトで定義する

## 10. 入力データ設計

### 10.1 初期化時の入力
- **なし**: AG-003 は AG-001 からの委譲を受けて起動するため、特別な初期化パラメータは不要

### 10.2 対話ループでの入力
- **社員の回答（D-006）**: 経費情報収集における各項目の回答（経費発生日・金額・店舗名・品目・経費区分・業務目的・追加経費の有無等）
- **領収書画像（D-014）**: 社員がアップロードした領収書の画像データ（任意。JPEG/PNG等）。OCR処理の入力となる

### 10.3 invocation_stateの入力
AG-001 から ToolContext 経由で受け取る情報（`InvocationStateModel` でバリデーション）:
```python
{
    "applicant_name": str,    # 申請者名
    "application_date": date  # 申請日（システム日付）
}
```
> `session_id` はファクトリ関数内でのみ使用し、エージェントの LLM プロンプトには含めない。

## 11. 出力データ設計

### 11.1 出力形式
- **出力先**: AG-001 への返却文字列（Agent as Tools の戻り値）および標準出力（社員への経過表示）
- **出力内容**: 申請書生成完了通知（ファイルパス）と申請内容チェック結果（EVT-001）

### 11.2 出力メッセージ
- 不足情報ヒアリング時: `"{不足項目}をご入力ください。"`
- 領収書アップロード案内: `"領収書の画像をお持ちでしたら、アップロードしてください。（スキップも可能です）"`
- OCR失敗時: `"領収書の読み取りに失敗しました。再度アップロードするか、手動でご入力ください。"`
- 経費区分判断不能時（BRL-09）: `"品目から経費区分を特定できませんでした。「事務用品費」「宿泊費」「資格精算費」「その他経費」のいずれかを選択してください。"`
- 高額案内時（BRL-12）: `"申請金額が5,000円を超えています。申請前に上長の事前承認を取得していることを確認してください。"`
- 申請書生成完了時: `"経費精算申請書を生成しました。{ファイルパス} に保存されています。"`

### 11.3 生成ファイル
- **ファイル名**: `{申請者名}_経費精算申請書_{タイムスタンプ}.xlsx`
- **保存先**: `data/outputs/`
- **内容**: 申請者情報・経費情報リストが記入された経費精算申請書（下書き）

## 12. バリデーション設計

**システム設計書の方針**: システム設計書 バリデーション方針.md（`artifacts/03_system-design/outputs/バリデーション方針.md`）を参照してください。

### 12.1 バリデーション方針
- **金額**: `gt=0`（1円以上）。高額判定（5,000円超）は ValidationError でブロックせずエージェントが案内（BRL-12）
- **経費区分**: `Literal["事務用品費", "宿泊費", "資格精算費", "その他経費"]`。判断困難時は社員選択後に Literal チェック（BRL-09）
- **業務目的**: min_length=1（BRL-13）
- **申請期限**: `check_application_deadline`（model_validator）で購入日から90日以内チェック（BRL-11）

**バリデーションの詳細**: 詳細設計書 4章を参照

### 12.2 申請ルール一覧

| ルールID | ルール名 | 条件 | 対応 |
|----------|----------|------|------|
| BRL-02 | 不足情報の対話確認 | 未収集の必須項目が1つ以上存在する | 不足項目を対話で一つずつ確認する |
| BRL-03 | 申請内容妥当性チェック実施 | 申請書（下書き）が生成済み | 社内申請ルールと照合し結果と修正案を提示する |
| BRL-07 | 全必要情報収集後の申請書生成 | 全必須項目収集済み・APR-001 承認完了後のみ | TOOL-002 を呼び出す |
| BRL-08 | 領収書OCRによる情報自動抽出 | 領収書画像がアップロードされた場合 | LLM 画像認識で金額・支払先・日付を自動抽出し収集済み情報に反映する |
| BRL-09 | 品目からの経費区分自動判断 | 品目名称が入力された場合（JD-07/JD-17） | 経費区分定義と照合して自動設定する。判断困難な場合は社員に4択選択肢を提示する |
| BRL-11 | 申請期限超過検出・警告 | 購入日から申請日まで90日超過の場合 | ErrorHandler 経由で超過を提示し管理部門への相談を案内する |
| BRL-12 | 高額申請時の事前上長承認案内 | 費用が5,000円を超える場合（JD-11） | 事前上長承認取得を確認する旨を社員に提示する |

| BRL-16 | 申請書生成後の自動チェック実行 | 申請書生成後（EVT-001） | 社内申請ルールと照合し結果を提示する |

## 13. Agent as Tools設計

### 13.1 ツールとしての公開
- **ツール名**: handle_expense_application
- **デコレータ**: `@tool(context=True)`
- **入力**: `query: str`（AG-001 からの指示文字列）、`tool_context: ToolContext`
- **出力**: `str`（申請書生成完了通知・申請内容チェック結果を含む文字列）

### 13.2 ToolContextの使用
- `invocation_state` から `applicant_name`・`application_date` を取得し `InvocationStateModel` でバリデーション
- `invocation_state["session_id"]` をファクトリ関数内で使用して `FileBasedSessionManager` を初期化（session_id は LLM には渡さない）
- セッションファイル（session_{id}.json）から既存の経費情報リストを読み込み（会話中断・再開対応）

**詳細**: 詳細設計書 2.5章を参照

## 14. テスト観点

### 14.1 テスト観点の概要
- **機能テスト**: 経費情報収集（2件）→ OCR 自動抽出 → 経費区分判断 → 申請情報のドラフト提示（テキスト整理、ツール呼び出しなし）→ APR-001 承認（BeforeToolCallEvent 経由で1回のみ発生）→ TOOL-002 生成 → EVT-001 チェックの一連フローが正常に動作すること
- **Human-in-the-Loop テスト**: ドラフト提示後に APR-001 承認が1回だけ発生し、承認（OK）後にはじめて TOOL-002 が実行されること。承認前に TOOL-002 が実行されないこと
- **異常系テスト**: OCR 抽出失敗時に手動入力フローに切り替わること。経費区分の判断困難時に4択選択肢が提示されること。申請期限超過（BRL-11）の警告が提示されること。高額（5,000円超）の案内（BRL-12）が提示されること。システム系エラー発生時に AG-001 へ要約メッセージが返却されること
- **性能テスト**: LoopControlHook により10回上限でループが終了すること

**詳細なテストケース**: 詳細設計書 7章を参照

## 16. 制約事項・前提条件

### 16.1 技術的制約
- Strands Agents SDK v1.25.0 を使用し、`@tool(context=True)` デコレータで AG-001 からツールとして呼び出されること
- HumanApprovalHook を必須設定すること（TOOL-002 呼び出し前の APR-001 承認のため）
- LoopControlHook（max 10回）を必須設定すること
- OCR は Strands SDK に組み込みの LLM 画像認識機能を利用し、外部 OCR ツールは使用しないこと

### 16.2 業務的制約
- APR-001 承認なしでの TOOL-002 実行は EX-05（実行不可）として処理する（ABAC-001）
- 経費区分の自動判断が困難な場合は、社員に「事務用品費」「宿泊費」「資格精算費」「その他経費」の4択を提示して選択させること（BRL-09）
- 領収書画像のアップロードは任意とし、アップロードしない場合は手動入力フローを継続すること（BRL-08）

### 16.3 前提条件
- invocation_state に `applicant_name`（str、min_length=1）と `application_date`（date）が含まれていること
- `data/templates/expense_template.xlsx` がファイルストアに存在すること
- `data/knowledge/` ディレクトリに社内申請ルール（D-002）・申請書作成必要情報一覧（D-005）が配置されていること
