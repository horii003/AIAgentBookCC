## 1. 概要

### 1.1 コンポーネントの目的

申請者名を収集した後、申請内容（自由文）から申請種別（交通費精算申請・経費精算申請）を判定し、対応する専門エージェント（AG-002/AG-003）へ委任する。申請フロー全体のオーケストレーターとして、セッション全体を統括する。判定不能時は2択の選択肢を提示してユーザーに確認し、申請種別を確定させる。

### 1.2 主要な責務

1. **申請者名の収集**: アプリ起動時（対話ループ開始前）に申請者名を取得し、invocation_stateに含めて専門エージェントへ渡す
2. **申請日の自動取得**: システム日付をYYYY-MM-DD形式で自動取得し、invocation_stateに含める
3. **セッションIDの生成**: `{タイムスタンプ（秒単位）}_{UUID（8文字）}` 形式（例: `20260323153045_a1b2c3d4`）でセッションIDを生成し、FileSessionManagerの初期化とinvocation_stateの受け渡しに使用する
4. **申請種別の判定**: ユーザーの申請内容（自由文）からBRL-01に基づき申請種別を判定する
5. **申請種別の確認（判定不能時）**: BRL-02に基づき2択を提示し、ユーザー選択で申請種別を確定する
6. **専門エージェントへの委任**: 申請種別確定後にAG-002またはAG-003をツールとして呼び出す（LNK-001/LNK-002）
7. **リセット処理**: `reset`/`リセット`/`最初から` コマンドを受けて会話履歴と申請者情報をリセットし、申請者名の収集から再開する

### 2.2 インターフェース設計

#### 2.2.1 入力

**初期化時の入力（アプリ起動時）**:
- `applicant_name` (str): 申請者名（必須、min_length=1）。空文字の場合は再入力を促す

**対話ループでの入力**:
- 申請内容（自由文）: ユーザーが申請したい内容を自然言語で入力する
- 特殊コマンド:
  - `reset`, `リセット`, `最初から`: 会話履歴と申請者情報をリセットし、申請者名の収集から再開する
  - 判定不能時の2択選択: `1`（交通費精算申請）または `2`（経費精算申請）

**パラメータ**:
- `user_input` (str): ユーザーからの入力（`UserInputText` モデルで max_length=500・min_length=1 を検証）

#### 2.2.2 出力

**出力先**: CLI（標準出力）

**出力メッセージ**:
- ウェルカムメッセージ（アプリ起動時・申請者名入力前に一度だけ表示）:
  ```
  ============================================================
  こちらは申請受付窓口エージェントです
  社内の様々な申請作業をサポートします

  最初に申請者名を入力してください。その後、申請したい内容をお知らせください。キーワードでも構いません

  ※終了するには 'exit' または 'quit' と入力ください
  ※最初からやり直すには 'reset' と入力ください
  ============================================================
  ```
- 申請者名収集: `"申請者名を入力してください。"`
- 申請書・申請先提示: `"[申請種別]の申請が必要です。申請書：[申請書名]、申請先：[申請先名]"`
- 2択提示（判定不能時）: `"申請種別が特定できませんでした。以下から選択してください：\n1. 交通費精算申請\n2. 経費精算申請"`
- 専門エージェント委任開始: `"[申請種別]の手続きを開始します。"`
- エラー通知（AG-002/AG-003内部エラー）: `"申し訳ありません。処理中にエラーが発生しました。システム管理者にご連絡ください。"`
- ループ上限到達: `"処理が複雑すぎるため終了します。"`

**戻り値の形式**: `str`（エージェントからの応答）

### 2.3 ビジネスロジック

#### 2.3.1 システムプロンプト

##### システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 静的定数 |
| 動的生成の理由 | 不要。申請期限チェック（BRL-13/BRL-18）はAG-002/AG-003が担当するため、AG-001のプロンプトに日付・期限判断は含まれない |

##### プロンプト全文

```
あなたは社内申請システムの申請受付窓口エージェントです。
申請者からの申請内容（自由文）を受け付け、申請種別（交通費精算申請・経費精算申請）を判定し、対応する専門エージェントへ委任することが役割です。

【役割と責任範囲】
- 申請種別の判定と専門エージェントへの委任のみを担当します
- 申請情報の収集・申請書の生成・申請書のチェック・申請書の提出は担当しません
- これらはすべて専門エージェント（交通費精算申請エージェント・経費精算申請エージェント）が担当します

【申請種別判定プロトコル（BRL-01/BRL-02）】
1. ユーザーの申請内容（自由文）から申請種別を判定します
   - 交通費精算申請に該当する場合（電車代・交通費・バス代・タクシー代・飛行機代など）: AG-002（交通費精算申請エージェント）へ委任します
   - 経費精算申請に該当する場合（会議費・文房具・宿泊費・書籍代・接待費など）: AG-003（経費精算申請エージェント）へ委任します

2. 申請種別が判定不能の場合（BRL-02）:
   - 以下の2択のみをユーザーに提示します
     「申請種別が特定できませんでした。以下から選択してください：
      1. 交通費精算申請
      2. 経費精算申請」
   - ユーザーの選択（1または2）で申請種別を確定します
   - 上記2択以外の回答は受け付けず、再選択を促します

【申請書・申請先の特定（BRL-03）】
- 申請種別確定後、申請書名と申請先をユーザーに提示します
  - 交通費精算申請: 申請書「交通費精算申請書」、申請先は要件上未定義
  - 経費精算申請: 申請書「経費精算申請書」、申請先は要件上未定義
- 提示後、対応する専門エージェントへ委任します

【専門エージェントへの委任】
- 申請種別が「交通費精算申請」に確定した場合: travel_application_agent_tool を呼び出します
- 申請種別が「経費精算申請」に確定した場合: expense_application_agent_tool を呼び出します
- 委任は1回のみ実行します。同一フローで二重委任は禁止です
- AG-002とAG-003を同時に呼び出してはいけません

【禁止事項】
- 申請情報（移動情報・経費明細）の収集: 専門エージェントが担当
- 申請書（Excel）の生成: 専門エージェントが担当
- 申請期限チェック: 専門エージェントが担当
- 申請書の提出操作: 利用者のみが実施可能（BRL-09）
- TOOL-001（交通費計算ツール）の直接呼び出し: AG-002が担当
- TOOL-002（申請書生成ツール）の直接呼び出し: AG-002/AG-003が担当

【リセット処理】
- ユーザーが「reset」「リセット」「最初から」と入力した場合: 会話をリセットして申請者名の収集から再開します

【エラー発生時の対応（BRL-10）】
- 専門エージェント（AG-002/AG-003）からエラーが返された場合: 「申し訳ありません。処理中にエラーが発生しました。システム管理者にご連絡ください。」をユーザーに提示します
```

##### プロンプト設計の意図

| 要素 | 意図 |
|------|------|
| 役割と責任範囲 | AG-001が申請種別判定・委任のみを担い、申請フロー詳細（情報収集・生成・チェック）は専門エージェントに委ねることを明示する |
| 申請種別判定プロトコル | BRL-01/BRL-02に基づく判定手順を明示し、判定不能時の2択提示ループを制御する |
| 禁止事項 | TOOL-001/TOOL-002の直接呼び出しや申請書提出操作をAG-001が実行しないことを明示する |
| 委任の単純化 | 申請種別確定後は専門エージェントへ全委任し、AG-001が申請フロー詳細に関与しないことを明示する |

#### 2.3.2 申請種別判定ルール

| ルール | 条件 | 対応 |
|--------|------|------|
| BRL-01: 自動判定 | 申請内容から交通費精算申請・経費精算申請が特定できる場合 | 申請種別を確定してBRL-03（申請書・申請先提示）へ進む |
| BRL-02: 判定不能 | 申請種別が特定できない場合 | 2択（交通費精算申請/経費精算申請）をユーザーに提示。選択結果で申請種別を確定する |
| BRL-03: 提示 | 申請種別確定後 | 申請書名・申請先をユーザーへ提示する（申請先は要件上未定義） |
| BRL-09: 提出禁止 | 申請書提出操作が発生する場合 | AG-001は提出操作を実行しない。利用者に提出操作を促す |
| BRL-10: エスカレーション | AG-002/AG-003の内部エラー、ユーザーが離脱した場合 | エスカレーション案内を提示する（エスカレーション先は要件上未定義） |

```

  ↓

  ↓
セッションIDを生成
  datetime.now().strftime("%Y%m%d%H%M%S") + "_" + uuid.uuid4().hex[:8]
  ↓

  ↓
Agent インスタンスを生成（orchestrator_agent）

  - tools=[travel_application_agent_tool, expense_application_agent_tool]
  - conversation_manager=SlidingWindowConversationManager(window_size=30, should_truncate_results=True, per_turn=False)

  ↓

  ↓

  ↓

  ↓

  Yes → ERRORログ出力（exc_info=True）。ErrorHandler呼び出しユーザー向けメッセージを表示。ループ継続（continue）
  ↓
Exception 発生？
  Yes → ERRORログ出力（exc_info=True）。ErrorHandler呼び出しユーザー向けメッセージを表示。ループ継続（continue）
  ↓
ループ先頭へ
```

### 2.4 設定・構成

#### 2.4.1 共通設定

| 設定項目 | 設定値 | 備考 |
|---------|--------|------|
| 会話管理 | SlidingWindowConversationManager(window_size=30, should_truncate_results=True, per_turn=False) | 複数の専門エージェントとのやり取りを保持するため |
| リトライ戦略 | 標準設定（Strands Agents SDK組み込み） | 6回/4秒/240秒。Amazon Bedrockの一時的エラーへのリトライのみ適用 |
| ループ制御 | LoopControlHook(max_iterations=10) | 暴走防止。申請フロー正常ステップ数に対して十分な余裕 |
| 承認フック | なし | 申請書生成はAG-002/AG-003の責務のため（共通設定方針 6.2節） |
| セッション管理 | FileSessionManager(session_id=session_id, storage_path="data/sessions/") | 会話履歴の永続化・セッション再開のため |

| ツールID | ツール名 | 利用目的 | 呼び出し条件 |

## 3. 実装詳細

### 3.1 クラス/関数設計

#### 3.1.1 main 関数

##### 説明
アプリケーションのエントリーポイント。申請者名収集・セッション初期化・対話ループを管理する。

##### 引数
なし

##### 戻り値
なし

##### 処理内容
1. `data/output/{session_id}/` ディレクトリが存在しない場合は `os.makedirs(f"data/output/{session_id}", exist_ok=True)` で作成する
2. `load_fare_data()` を呼び出してDATA-011/DATA-012を静的読み込みする（失敗時はアプリ起動を中断）
3. ウェルカムメッセージを表示する（アプリ起動時に一度だけ）
4. 申請者名をループで収集する（空文字の場合は再促し）
5. `datetime.date.today().isoformat()` で申請日を自動取得する
6. `datetime.now().strftime("%Y%m%d%H%M%S") + "_" + uuid.uuid4().hex[:8]` でセッションIDを生成する
7. `FileSessionManager` と `Agent` インスタンスを生成する
8. 対話ループを開始する。ユーザー入力受付には `input("\n\n入力内容（終了時はquit）: ")` を使用する

#### 3.1.2 create_orchestrator_agent 関数

##### 説明
AG-001（申請受付窓口エージェント）の `Agent` インスタンスを生成するファクトリ関数。

##### 引数
- `session_id` (str): セッションID（必須）
- `applicant_name` (str): 申請者名（必須）
- `application_date` (str): 申請日（YYYY-MM-DD形式、必須）

##### 戻り値
- `Agent`: 初期化済みの申請受付窓口エージェントインスタンス

##### 処理内容
1. `FileSessionManager(session_id=session_id, storage_path="data/sessions/")` を生成する
2. `Agent(agent_id="orchestrator_agent", ...)` を生成して返す（`callback_handler` は省略、標準出力への直接表示はAG-001が行う）

### 3.2 エラーハンドリング

| 例外種別 | ログレベル | ループ制御 | 備考 |
|---------|-----------|-----------|------|
| `KeyboardInterrupt` | INFO | break（ループ終了） | ユーザーによる強制割り込み。ErrorHandlerを呼び出しユーザー向けメッセージをprintで表示 |
| `LoopLimitError` | WARNING | continue（ループ継続） | ReActループ上限超過。ErrorHandlerを呼び出しユーザー向けメッセージをprintで表示 |
| `ContextWindowOverflowException` | WARNING | continue（ループ継続） | コンテキストウィンドウ超過。ErrorHandlerを呼び出しユーザー向けメッセージをprintで表示 |
| `MaxTokensReachedException` | WARNING | continue（ループ継続） | 最大トークン数超過。ErrorHandlerを呼び出しユーザー向けメッセージをprintで表示 |
| `RuntimeError` | ERROR（exc_info=True） | continue（ループ継続） | その他のRuntimeError。スタックトレース付き。ErrorHandlerを呼び出しユーザー向けメッセージをprintで表示 |
| `Exception`（その他） | ERROR（exc_info=True） | continue（ループ継続） | 予期しないエラー。スタックトレース付き。ErrorHandlerを呼び出しユーザー向けメッセージをprintで表示 |

**共通仕様**:
- 戻り値はなし（対話ループのため不要）
- 全例外のログにはセッションID（`session_id`）を付加情報として含める
- 全例外でErrorHandlerを呼び出し、ユーザー向けメッセージを `print()` で表示する

## 4. データ設計

### 4.2 セッション初期化データ

**データソース**: アプリ起動時の標準入力・システム日付・UUID生成

**読み込みタイミング**: アプリ起動時（対話ループ開始前）

**渡すデータ（invocation_state）**:
```python
{
    "session_id": str,       # {タイムスタンプ（秒単位）}_{UUID（8文字）} 形式のセッション識別子
                             # 例: "20260323153045_a1b2c3d4"
                             # 生成方法: datetime.now().strftime("%Y%m%d%H%M%S") + "_" + uuid.uuid4().hex[:8]
                             # 生成タイミング: セッション開始時（AG-001が生成）
    "applicant_name": str,   # 申請者名（min_length=1）
    "application_date": str  # 申請日（YYYY-MM-DD形式。システム日付自動取得）
}
```

> ※ `application_type`（申請種別）は invocation_state に含めない。専門エージェントは自身の役割を固定で把握しているため不要
> ※ invocation_state は辞書リテラルで渡す。専用 Pydantic モデルは定義しない

## 5. 補足情報

### 5.1 実装上の注意点

1. **invocation_stateの渡し方**
   - `agent(user_input, invocation_state={...})` の形式で渡す
   - `invocation_state` はツール関数の `ToolContext` 経由でのみアクセス可能。LLMプロンプトには含まれない

2. **セッションIDの生成タイミング**
   - アプリ起動時（申請者名収集後）に1回だけ生成する
   - リセット時は新しいセッションIDを生成する（`datetime.now().strftime("%Y%m%d%H%M%S") + "_" + uuid.uuid4().hex[:8]` で再生成）
   - 形式: `{タイムスタンプ（秒単位）}_{UUID（8文字）}`（例: `20260323153045_a1b2c3d4`）

3. **callback_handler の設定**
   - AG-001自身はCLI出力を担うため `callback_handler` をデフォルト（省略）にする
   - AG-002/AG-003（専門エージェント）は `callback_handler=None` を指定して二重出力を防ぐ

4. **リセット処理**
   - `reset`/`リセット`/`最初から` コマンドは対話ループの冒頭でチェックし、`Agent` インスタンスを再生成する
   - リセット後は申請者名の収集から再開する

---

### 5.2 パフォーマンス考慮事項

1. **LoopControlHookのカウンタリセット**
   - `BeforeInvocationEvent` と `AfterInvocationEvent` でループカウンタをリセットすることで、エージェント呼び出し間でカウンタが引き継がれない

2. **FileSessionManagerの起動コスト**
   - 申請者名収集後に1回だけFileSessionManagerを初期化し、以後は同一インスタンスを使用する

---

### 5.3 セキュリティ考慮事項

1. **invocation_stateの安全な受け渡し**
   - session_id・applicant_name・application_date はLLMプロンプトには含めずinvocation_state経由で渡す

2. **ユーザー入力の長さ制限**
   - `UserInputText`（max_length=500）で長大なユーザー入力を拒否する（GRD-017）
