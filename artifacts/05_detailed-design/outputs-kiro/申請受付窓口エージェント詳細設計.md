## 1. 概要

### 1.1 コンポーネントの目的

申請者名を受け取り、申請内容テキストを受け付けて社内申請ルールに基づき申請種別（交通費精算申請・経費精算申請）を判断し、対応する専門エージェント（AG-002・AG-003）へ処理を委譲するオーケストレーターエージェント。

### 1.2 主要な責務

- アプリ起動時に取得済みの申請者名を初期化パラメータとして受け取る
- 申請内容テキスト（D-001）を受け付け、LLMで意図を分類する
- 社内申請ルール（D-002）を参照して申請種別を判断する（BRL-01、JD-01、JD-02）
- 申請種別確定後にAG-002またはAG-003へinvocation_stateを渡して委譲する
- 申請種別が判断不能の場合（JD-03）、2択を提示してユーザーに選択させる（BRL-04）

### 2.2 インターフェース設計

#### 2.2.1 入力

**初期化時の入力**:
- `applicant_name` (str): 必須。アプリ起動時に取得済みの申請者名。空文字列・空白のみ不可

**対話ループでの入力**:
- ユーザー入力（自然言語での申請内容テキスト）
- 特殊コマンド:
  - `exit`, `quit`, `終了`: システム終了
  - `reset`, `リセット`, `最初から`: 会話履歴と申請者情報をリセットし、セッションを最初から開始する

**パラメータ**:
- `query` (str): ユーザーからの申請内容テキスト

#### 2.2.2 出力

**出力先**: CLI（標準出力）

**出力メッセージ**:
- ウェルカムメッセージ（起動時）:
```
============================================================
こちらは申請受付窓口エージェントです
社内の様々な申請作業をサポートします

最初に申請者名を入力してください。
その後、申請したい内容をお知らせください。キーワードでも構いません

※終了するには 'exit' または 'quit' と入力ください
※最初からやり直すには 'reset' と入力ください
============================================================
```
- 入力プロンプト: `\n\n入力内容（終了時はquit）: `
- 申請種別提示: `"申請種別：[申請種別]、申請書：[申請書名]、申請先：[申請先]"`
- 2択提示: `"申請種別を選択してください：1. 交通費精算申請 / 2. 経費精算申請"`
- エラー通知: `"エラーが発生しました。管理部門にお問い合わせください。"`
- セッション終了: `"申請処理を終了します。"`

**戻り値の形式**: `str`（エージェントからの応答）

### 2.3 ビジネスロジック

#### 2.3.1 システムプロンプト

##### システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 静的定数 |
| 動的生成の理由 | 不要（AG-001は申請種別判断・振り分けのみ担当。日付・期限を基準とした判断処理は含まない） |

##### プロンプト全文

```
あなたは経費精算AIエージェントシステム（ECAAS）の申請受付窓口エージェントです。
申請者からの申請内容を受け付け、社内申請ルールに基づいて申請種別を判断し、
適切な専門エージェントへ処理を委譲します。

【役割】
1. 申請内容の受付
   - 申請者からの自然言語による申請内容テキストを受け付ける
   - 空入力の場合は再入力を促す
   - 500文字を超える入力の場合は500文字以内での再入力を促す（BRL-15）

2. 申請種別の判断
   - 社内申請ルール（BRL-01）に基づき、申請内容から申請種別を判断する
   - 交通費精算申請（JD-01）：電車・バス・タクシー・飛行機等の交通費精算に該当する場合
   - 経費精算申請（JD-02）：領収書・備品・宿泊費等の経費精算に該当する場合
   - 判断不能（JD-03）：申請種別が判断できない場合は2択を提示する（BRL-04）

3. 専門エージェントへの委譲
   - 申請種別が「交通費精算申請」に確定した場合：transport_application_agentツールを呼び出す
   - 申請種別が「経費精算申請」に確定した場合：expense_application_agentツールを呼び出す

【申請種別判断ルール（BRL-01）】
1. 交通費精算申請に該当するキーワード・特徴
   - 電車・バス・タクシー・飛行機等の交通手段に関する申請
   - 出張・移動・通勤等の交通費精算
   - 「交通費」「電車代」「バス代」「タクシー代」「出張費」等のキーワード

2. 経費精算申請に該当するキーワード・特徴
   - 領収書・レシートを伴う経費の申請
   - 備品・消耗品・宿泊費・資格取得費等の経費精算
   - 「経費」「領収書」「備品」「宿泊」「資格」等のキーワード

3. 判断不能の場合（JD-03）
   - 申請内容が交通費精算申請・経費精算申請のどちらにも該当しない場合
   - 申請内容が曖昧で判断できない場合
   - 2択（交通費精算申請 / 経費精算申請）を提示し、申請者に選択させる（BRL-04）

【禁止事項（BRL-14）】
- 申請ルールに定義されていない内容を推測・補完してはならない
- 申請ルール外の申請種別を独自に定義してはならない
- 申請者の意図を勝手に解釈して申請種別を確定してはならない（判断不能時は必ず2択提示）

【対話回数制限（BRL-16）】
- 対話回数が30回に達した場合はセッションを終了する

【エラー対応】
- 空入力：「申請内容を入力してください。どのような申請をご希望か、具体的にお知らせください。」
- 500文字超過：「入力が長すぎます。500文字以内で入力してください。」
- システムエラー：「システムの処理中にエラーが発生しました。管理部門にお問い合わせください。」
```

##### プロンプト設計の意図

| 要素 | 意図 |
|------|------|
| 役割定義 | 申請受付窓口としての責務を明確化し、情報収集・申請書生成は専門エージェントに委譲することを明示する |
| 申請種別判断ルール | BRL-01・JD-01・JD-02・JD-03に基づく判断基準をLLMに提供し、一貫した判断を実現する |
| 禁止事項 | BRL-14に基づき、申請ルール外の推測・補完を禁止し、誤った申請種別判断を防止する |
| エラー対応 | EX-01・EX-02に対応するメッセージを定義し、ユーザーへの適切なフィードバックを実現する |

#### 2.3.2 申請種別判断ルール

| ルール | 条件 | 対応 |
|--------|------|------|
| JD-01（交通費精算申請判断） | 申請内容が交通費精算申請に該当するキーワード・特徴を含む | transport_application_agentツールを呼び出す |
| JD-02（経費精算申請判断） | 申請内容が経費精算申請に該当するキーワード・特徴を含む | expense_application_agentツールを呼び出す |
| JD-03（判断不能） | 申請内容が交通費精算申請・経費精算申請のどちらにも該当しない、または曖昧 | 2択（交通費精算申請 / 経費精算申請）を提示し、申請者に選択させる（BRL-04） |
| BRL-15（文字数制限） | ユーザー入力が500文字を超える | 500文字以内での再入力を促す |
| BRL-16（対話回数上限） | 対話回数が30回に達する | セッションを終了する |

#### 2.3.3 処理フロー

```
開始（applicant_nameを受け取る）
  ↓
session_idを生成する
  形式: {タイムスタンプ（秒単位）}_{UUID（8文字）}
  例: 20260323153045_a1b2c3d4
  生成方法: datetime.now().strftime("%Y%m%d%H%M%S") + "_" + uuid.uuid4().hex[:8]
  ↓
SessionManagerを初期化する
  ↓
application_dateを取得する（システム日付、YYYY-MM-DD形式）
  ↓
ウェルカムメッセージを表示する
  ↓
対話ループ開始
  ↓
入力プロンプトを表示する（"\n\n入力内容（終了時はquit）: "）
  ↓
ユーザー入力を受け付ける
  ↓
特殊コマンド判定
  reset/リセット/最初から → 会話履歴・セッション情報をリセットして対話ループ先頭へ
  exit/quit/終了 → セッション終了
  ↓
空入力チェック
  → 空の場合: 「申請内容を入力してください。」を表示して対話ループ先頭へ（EX-01）
  ↓
文字数チェック（500文字以内）
  → 超過の場合: 「入力が長すぎます。500文字以内で入力してください。」を表示して対話ループ先頭へ（EX-01）
  ↓
turn_countをインクリメントする
  ↓
turn_countが30回に達した？
  → YES: 「対話回数の上限に達しました。セッションを終了します。」を表示してセッション終了（BRL-16）
  ↓
LLMで申請種別を判断する（BRL-01）
  ↓
申請種別が「交通費精算申請」？（JD-01）
  → YES: transport_application_agentツールを呼び出す
         invocation_state = {applicant_name, application_date, session_id}
  ↓
申請種別が「経費精算申請」？（JD-02）
  → YES: expense_application_agentツールを呼び出す
         invocation_state = {applicant_name, application_date, session_id}
  ↓
申請種別が判断不能？（JD-03）
  → YES: 「申請種別を選択してください：1. 交通費精算申請 / 2. 経費精算申請」を表示する（BRL-04）
         ユーザーの選択を受け付ける
         1選択 → transport_application_agentツールを呼び出す
         2選択 → expense_application_agentツールを呼び出す
         その他 → 再選択を促す
  ↓
専門エージェントの処理完了を待つ
  ↓
処理結果をユーザーに表示する
  ↓
対話ループ先頭へ（次の申請受付）
```

### 2.5 連携設計

#### 2.5.1 Agent as Tools設計

**transport_application_agentツール**:

ツール名: `transport_application_agent`

ツール説明:
```
交通費精算申請の処理を行う専門エージェント。
移動情報の収集・交通費計算・申請書生成・妥当性チェックを担当する。
申請種別が「交通費精算申請」に確定した場合に呼び出す。
```

**expense_application_agentツール**:

ツール名: `expense_application_agent`

ツール説明:
```
経費精算申請の処理を行う専門エージェント。
領収書画像処理・経費情報収集・申請書生成・妥当性チェックを担当する。
申請種別が「経費精算申請」に確定した場合に呼び出す。
```

**ToolContextの使用**:
- `@tool(context=True)` デコレータを使用
- `tool_context` パラメータでToolContextを受け取る
- invocation_stateをToolContext経由で専門エージェントへ渡す

#### 2.5.2 invocation_stateの受け渡し

```python
# AG-001から専門エージェントへ渡す
invocation_state = {
    "applicant_name": applicant_name,       # アプリ起動時取得済み
    "application_date": application_date,   # システム日付自動取得（YYYY-MM-DD）
    "session_id": session_id,               # AG-001がセッション開始時に生成
}
response = transport_application_agent(query, invocation_state=invocation_state)
```

#### 2.5.3 専門エージェントへの振り分けロジック

**振り分け判断基準**:

| 連携先 | キーワード | 特徴 |
|--------|-----------|------|
| AG-002（transport_application_agent） | 電車・バス・タクシー・飛行機・交通費・電車代・バス代・タクシー代・出張費・移動費 | 交通手段を伴う移動に関する費用精算 |
| AG-003（expense_application_agent） | 領収書・レシート・備品・消耗品・宿泊費・資格・経費・購入費 | 領収書を伴う物品・サービスの費用精算 |
| 2択提示（CF-002） | 上記以外・曖昧な表現 | 申請種別が判断不能な場合 |

### 3.2 エラーハンドリング

| エラー種別 | 条件 | ログレベル | ループ制御 | 対応 |
|-----------|------|-----------|-----------|------|
| KeyboardInterrupt | ユーザーによる中断 | INFO | break（ループ終了） | ErrorHandlerでメッセージ生成・printで表示。セッションIDをログに付加 |
| LoopLimitError | LoopControlHookがループ上限到達を検知 | WARNING | continue（ループ継続） | ErrorHandlerでメッセージ生成・printで表示。セッションIDをログに付加 |
| ContextWindowOverflowException | コンテキストウィンドウ超過 | WARNING | continue（ループ継続） | ErrorHandlerでメッセージ生成・printで表示。セッションIDをログに付加 |
| MaxTokensReachedException | 最大トークン数到達 | WARNING | continue（ループ継続） | ErrorHandlerでメッセージ生成・printで表示。セッションIDをログに付加 |
| RuntimeError | その他の実行時エラー | ERROR（exc_info=True） | continue（ループ継続） | ErrorHandlerでメッセージ生成・printで表示。セッションIDをログに付加 |
| Exception（その他） | 予期しないエラー | ERROR（exc_info=True） | continue（ループ継続） | ErrorHandlerでメッセージ生成・printで表示。セッションIDをログに付加 |

> **注意**: 戻り値はなし（対話ループのため不要）。ログにはセッションIDを付加情報として含める。

## 5. 補足情報

### 5.1 実装上の注意点

1. **申請者名の取得タイミング**
   - 申請者名はアプリ起動時（対話ループ開始前）に取得し、AG-001の初期化パラメータとして渡す
   - 対話中に申請者名を収集してはならない

2. **session_idの生成**
   - session_idはAG-001がセッション開始時に生成する
   - 形式: `{タイムスタンプ（秒単位）}_{UUID（8文字）}`（例: `20260323153045_a1b2c3d4`）
   - 生成方法: `datetime.now().strftime("%Y%m%d%H%M%S") + "_" + uuid.uuid4().hex[:8]`
   - 専門エージェントはsession_idを生成しない

3. **application_dateの取得**
   - application_dateはAG-001がシステム日付（実行時の日付）を自動取得する（YYYY-MM-DD形式）
   - LLMに日付を推測させてはならない

4. **例外捕捉の順序**
   - KeyboardInterrupt → LoopLimitError → ContextWindowOverflowException → MaxTokensReachedException → RuntimeError → Exception の順序で個別に捕捉する
   - 各例外でErrorHandlerを呼び出す前に呼び出し元でログ出力を実施する
   - ログにはセッションIDを付加情報として含める

---

### 5.2 パフォーマンス考慮事項

1. **ウィンドウサイズ**
   - SlidingWindowConversationManager(window_size=30)を使用し、古い会話履歴をLLMへの入力から除外する

---

### 5.3 セキュリティ考慮事項

1. **申請者名のマスキング**
   - ログ出力時に申請者名をマスキングする（GRD-014）
