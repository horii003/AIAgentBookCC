## 1. 概要

### 1.1 コンポーネントの目的

収集済み申請情報（D-005）を業務固有フォーマット（Excel）の申請書に変換してファイルを生成する。交通費精算申請書はDATA-002（交通費精算申請書テンプレート）、経費精算申請書はDATA-003（経費精算申請書テンプレート）を使用する。申請者名・申請日はLLMからパラメータとして渡されず、ツール内部でinvocation_stateから取得する。生成した申請書を`data/output/{session_id}/`ディレクトリにExcelファイルとして永続化し、社員が確認・提出できる状態にする。

### 1.2 主要な責務

- **申請書テンプレート読み込み**: 申請種別に応じたExcelテンプレートファイルを`openpyxl`で読み込む。読み込み前に`os.path.exists()`で存在チェックを行う
- **データマッピング**: 収集済み申請情報（`TransportApplicationData` / `ExpenseApplicationData`）を申請書の各セルに書き込む
- **ファイルID・パスの自律生成**: 申請書IDおよび出力ファイルパスはツール内部で`datetime.now().strftime("%Y%m%d%H%M%S")`をもとに生成する（LLMからパスを受け取ることを禁止）
- **ファイル出力**: 生成した申請書を`data/output/{session_id}/`ディレクトリにExcelファイルとして保存する（最大2回リトライ）
- **監査ログ記録**: 申請書ID・生成日時・申請者・申請種別・ファイルパスを強化ログに記録する（DATA-008）

## 3. generate_transport_applicationツール設計

### 3.1 基本情報

#### ツール名（関数名）
`generate_transport_application`

#### 説明（docstring）
```
収集済みの交通費精算申請情報からExcel申請書を生成してファイルに出力します。

申請書IDおよびファイルパスはツール内部で自動生成します。
申請者名・申請日はinvocation_stateから取得します（LLMはこれらをパラメータとして渡しません）。
ファイル書き込みは最大2回リトライします。

Args:
    segments (list[dict]): 移動区間情報リスト。各要素は以下のキーを持つ辞書:
        - travel_date (str): 移動日（YYYY-MM-DD形式）
        - departure (str): 出発地
        - destination (str): 目的地
        - transport_type (str): 交通手段（「電車」「バス」「タクシー」「飛行機」）
        - fare (int): 運賃（1以上の正整数）
    purpose (str): 業務目的（空文字禁止・500文字以内）

Returns:
    dict: 以下のキーを持つ辞書:
        - success (bool): 生成成否フラグ
        - file_path (str): 生成された申請書のファイルパス。エラー時は空文字列
        - application_data (dict): 申請書の各項目と記入値。エラー時は空辞書
        - message (str): エラーメッセージ（success=Trueの場合は空文字列）
```

### 3.2 インターフェース設計

#### 3.2.1 入力

**パラメータ定義**（LLMが渡すパラメータ）:

| 引数名 | 型 | 説明 | デフォルト値 |
|--------|-----|------|--------------|
| `segments` | `list[dict]` | 移動区間情報リスト（1件以上必須） | なし（必須） |
| `purpose` | `str` | 業務目的（validate_required_stringで検証） | なし（必須） |

**invocation_stateから取得するフィールド**（LLMはパラメータとして渡さない）:

| フィールド名 | 型 | 説明 | 取得元 |
|------------|-----|------|-------|
| `applicant_name` | `str` | 申請者名 | `tool_context.invocation_state.get("applicant_name", "")` |
| `application_date` | `str` | 申請日（YYYY-MM-DD形式） | `tool_context.invocation_state.get("application_date", "")` |
| `session_id` | `str` | セッションID（出力ディレクトリパスに使用） | `tool_context.invocation_state.get("session_id", "")` |

**segmentsの各要素フィールド**:

| フィールド名 | 型 | 必須/任意 | 説明 |
|------------|-----|-----------|------|
| `travel_date` | `str` | 必須 | 移動日（YYYY-MM-DD形式） |
| `departure` | `str` | 必須 | 出発地 |
| `destination` | `str` | 必須 | 目的地 |
| `transport_type` | `str` | 必須 | 交通手段（「電車」「バス」「タクシー」「飛行機」） |
| `fare` | `int` | 必須 | 運賃（1以上の正整数） |

##### バリデーションルール詳細

| フィールド | ルール | エラーメッセージ | 例 |
|-----------|--------|----------------|-----|
| `applicant_name`（invocation_stateから取得） | 空文字・None禁止 | `"入力のエラー: 申請者名が設定されていません。"` | NG: `""` |
| `application_date`（invocation_stateから取得） | YYYY-MM-DD等の解析可能な日付文字列 | `"日付のエラー: 申請日が正しく設定されていません。"` | NG: `""` |
| `segments` | 1件以上のリスト。各要素は上記フィールドを必須キーとして持つ辞書 | `"移動区間情報のエラー: 移動区間が1件以上必要です"` | NG: `[]` |
| `segments[].travel_date` | YYYY-MM-DD等の解析可能な日付文字列 | `"日付のエラー: 正しい日付形式（YYYY-MM-DD）で入力してください"` | NG: `"abc"` |
| `segments[].departure` | 空文字・None禁止 | `"入力のエラー: 必須項目が入力されていません。入力してください"` | NG: `""` |
| `segments[].destination` | 空文字・None禁止 | 同上 | NG: `""` |
| `segments[].transport_type` | 「電車」「バス」「タクシー」「飛行機」のいずれか | `"交通手段のエラー: ..."` | NG: `"新幹線"` |
| `segments[].fare` | 1以上の正整数 | `"金額のエラー: 1円以上の金額を入力してください"` | NG: `0` |
| `purpose` | 空文字・None禁止。前後空白自動除去。500文字以内 | `"入力のエラー: 必須項目が入力されていません。入力してください"` | NG: `""` |

#### 3.2.2 出力

**戻り値の型**: `dict[str, bool | str | dict]`

**説明**: 申請書生成結果を辞書形式で返す。ツール関数は例外を再送出せず、必ずこの形式で返す（例外処理方針準拠）

**構造**:
```python
{
    "success": bool,           # 生成成否フラグ
    "file_path": str,          # 生成されたExcelファイルパス（例: "data/output/{session_id}/APP_20260506123456_交通費精算申請.xlsx"）
    "application_data": dict,  # 申請書の各項目と記入値（CF-005のチェック参照用）
    "message": str             # エラーメッセージ（success=Trueの場合は空文字列""）
}
```

#### 3.2.3 例外

| 例外 | 発生条件 | 戻り値 |
|------|---------|-------|
| `ValidationError` | `TransportApplicationData` のバリデーション失敗時（ツール関数内でキャッチ） | `{"success": False, "file_path": "", "application_data": {}, "message": エラーメッセージ}` |
| `KeyError` | `segments`要素に必須キーが存在しない場合（ツール関数内でキャッチ） | `{"success": False, "file_path": "", "application_data": {}, "message": エラーメッセージ}` |
| `IOError` | ファイル保存時のI/Oエラー（ツール関数内でキャッチ） | `(False, エラーメッセージ文字列)` ※内部メソッドの戻り値 |
| `PermissionError` | ファイル保存時のアクセス権限エラー（ツール関数内でキャッチ） | `(False, エラーメッセージ文字列)` ※内部メソッドの戻り値 |
| `Exception` | 想定外の例外（ツール関数内でキャッチ） | `{"success": False, "file_path": "", "application_data": {}, "message": エラーメッセージ}` |

### 3.3 ビジネスロジック

#### 3.3.1 処理フロー

```
開始
  ↓
invocation_stateからapplicant_name・application_date・session_idを取得する
  ↓
TransportApplicationDataでバリデーション
  - applicant_name: 空文字チェック（invocation_stateから取得）
  - application_date: parse_date_field（invocation_stateから取得）
  - segments: 各TransportSegmentのバリデーション（必須キーガード含む）
  - purpose: validate_required_string
  ↓
バリデーション失敗？
  YES → ValidationError / KeyError をキャッチ
         呼び出し元でlogging.warning(f"ValidationError: {e}")を記録
         ErrorHandler.handle_validation_error(e) でメッセージ生成
         {"success": False, "file_path": "", "application_data": {}, "message": エラーメッセージ} を返す
  ↓
申請書IDとファイルパスを自律生成
  - application_id = f"APP_{datetime.now().strftime('%Y%m%d%H%M%S')}"
  - file_name = f"{application_id}_交通費精算申請.xlsx"
  - file_path = f"data/output/{session_id}/{file_name}"
  ↓
data/output/{session_id}/ ディレクトリが存在しない場合は作成する
  失敗 → EX-07として処理。{"success": False, ...} を返す
  ↓
os.path.exists(テンプレートパス) で存在チェック
  存在しない → 呼び出し元でlogging.error(f"Template not found: {path}")を記録
               ErrorHandler.handle_file_save_error(e) でメッセージ生成
               {"success": False, "file_path": "", "application_data": {}, "message": エラーメッセージ} を返す
  ↓
DATA-002（交通費精算申請書テンプレート）をopenpyxlで読み込む
  失敗（EX-06）→ 呼び出し元でlogging.error記録
                  ErrorHandler.handle_unexpected_error(e) でメッセージ生成
                  {"success": False, "file_path": "", "application_data": {}, "message": エラーメッセージ} を返す
  ↓
申請情報を各セルにマッピングする（列定義に従う）
  - 申請者名、申請日、業務目的を固定セルに書き込む
  - segments の各区間を明細行に書き込む（A: No, B: 移動日, C: 出発地, D: 目的地, E: 交通手段, F: 費用, G: 業務目的, H: 承認状況（空欄））
  ↓
application_dataの辞書を構築する（CF-005チェック参照用）
  ↓
Excelファイルを file_path に保存する（最大2回リトライ）
  1回目成功 → 次へ
  1回目失敗（IOError/PermissionError/Exception）→ 2回目リトライ
  2回目成功 → 次へ
  2回目失敗（IOError/PermissionError/Exception）→
    呼び出し元でlogging.error(f"File save failed: file_path={file_path}")を記録
    ErrorHandler.handle_file_save_error(e) でメッセージ生成
    {"success": False, "file_path": "", "application_data": {}, "message": エラーメッセージ} を返す
  ↓
監査ログ（DATA-008）に記録する
  - application_id, generated_at, applicant_name, application_type, file_path
  ↓
{"success": True, "file_path": file_path, "application_data": application_data, "message": ""} を返す
```

#### 3.3.2 segments必須キーガード

`segments`の各要素（辞書）に対して、バリデーション前に必須キーの存在確認を行う。キーが不足している場合は、LLMが再呼び出し時に正しいキーを含められるよう、不足しているキー名を明示したエラーメッセージを返す。

```
必須キー一覧: ["travel_date", "departure", "destination", "transport_type", "fare"]
↓
各要素に対して必須キーを確認する
  キーが存在しない場合 → KeyError:
    "移動区間情報のエラー: 必須キー「{missing_key}」が含まれていません。
    以下の全キーを含めて再度入力してください: travel_date, departure, destination, transport_type, fare"
```

### 3.4 エラーハンドリング

#### 3.4.1 エラー処理フロー

```
エラー発生
  ↓
ValidationError / KeyError（EX-01）
  → 呼び出し元でlogging.warning記録
     ErrorHandler.handle_validation_error(e) でメッセージ生成
     → {"success": False, "file_path": "", "application_data": {}, "message": 日本語メッセージ}
  ↓
テンプレート取得失敗（EX-06）: os.path.exists()で事前チェック
  → 呼び出し元でlogging.error(f"Template not found: {path}")記録
     ErrorHandler.handle_unexpected_error(e) でメッセージ生成
     → {"success": False, "file_path": "", "application_data": {}, "message": 日本語メッセージ}
  ↓
ファイル保存メソッド（_save_file）内のエラー
  IOError → (False, "申請書ファイルの出力に失敗しました。...")
  PermissionError → (False, "申請書ファイルへのアクセス権限がありません。...")
  Exception → (False, "申請書ファイルの保存中に予期しないエラーが発生しました。...")
  ↓
ファイル書き込み失敗（EX-04） - 最大2回リトライ
  → 1回目失敗: 2回目リトライ
  → 2回目失敗: 呼び出し元でlogging.error(f"File save failed (attempt 2): file_path={file_path}")記録
               ErrorHandler.handle_file_save_error(e) でメッセージ生成
               → {"success": False, "file_path": "", "application_data": {}, "message": 日本語メッセージ}
  ↓
output/ディレクトリ作成失敗（EX-07）
  → 呼び出し元でlogging.error記録
     ErrorHandler.handle_unexpected_error(e) でメッセージ生成
     → {"success": False, ...}
  ↓
想定外の例外（EX-08）
  → 呼び出し元でlogging.error記録（exc_info=True）
     ErrorHandler.handle_unexpected_error(e) でメッセージ生成
     → {"success": False, ...}
```

#### 3.4.2 エラーメッセージの詳細

| エラー種別 | 発生条件 | 例外クラス | 戻り値 | ユーザー向けメッセージ |
|-----------|---------|----------|-------|---------------------|
| E-001: テンプレート取得失敗（EX-06） | DATA-002のExcelテンプレートが存在しない | 存在チェック（FileNotFoundErrorではなくos.path.exists()を使用） | `{"success": False, ...}` | `"申請書テンプレートの読み込みに失敗しました。担当部署へご連絡ください。"` |
| E-002: ファイル保存失敗（IOError） | ファイルシステムへの書き込みI/Oエラー | `IOError` | `(False, エラーメッセージ文字列)` | `"申請書ファイルの出力に失敗しました。しばらくしてから再度お試しいただくか、担当部署へご連絡ください。"` |
| E-003: ファイル保存失敗（PermissionError） | アクセス権限エラー | `PermissionError` | `(False, エラーメッセージ文字列)` | `"申請書ファイルへのアクセス権限がありません。担当部署へご連絡ください。"` |
| E-004: ファイル保存失敗（その他） | 2回リトライ後も失敗 | `Exception` | `(False, エラーメッセージ文字列)` | `"申請書ファイルの保存中に予期しないエラーが発生しました。担当部署へご連絡ください。"` |

| E-006: ディレクトリ作成失敗（EX-07） | output/ディレクトリの作成・アクセス失敗 | `Exception` | `{"success": False, ...}` | `"出力ディレクトリの作成に失敗しました。担当部署へご連絡ください。"` |

### 4.2 インターフェース設計

**パラメータ定義**（LLMが渡すパラメータ）:

| 引数名 | 型 | 説明 | デフォルト値 |
|--------|-----|------|--------------|
| `store_name` | `str` | 店舗名（validate_required_stringで検証） | なし（必須） |
| `expense_category` | `str` | 経費区分（normalize_expense_categoryで正規化） | なし（必須） |
| `amount` | `int` | 金額（validate_positive_amountで検証。1以上の正整数） | なし（必須） |
| `expense_date` | `str` | 経費発生日（YYYY-MM-DD形式。parse_date_fieldで変換） | なし（必須） |

| フィールド | ルール | エラーメッセージ | 例 |

### 4.3 ビジネスロジック

#### 処理フロー

```
開始
  ↓
invocation_stateからapplicant_name・application_date・session_idを取得する

ExpenseApplicationDataでバリデーション

  - expense_category: normalize_expense_category（エラーにしない）

  - expense_date: parse_date_field
  - 申請期限チェック: model_validator（BRL-12）
  - purpose: validate_required_string
  ↓
バリデーション失敗？
  YES → ValidationError をキャッチ
         呼び出し元でlogging.warning記録
         ErrorHandler.handle_validation_error(e) でメッセージ生成
         {"success": False, "file_path": "", "application_data": {}, "message": エラーメッセージ} を返す
  ↓

  ↓
data/output/{session_id}/ ディレクトリが存在しない場合は作成する

  ↓

               {"success": False, "file_path": "", "application_data": {}, "message": エラーメッセージ} を返す

  失敗（EX-06）→ 呼び出し元でlogging.error記録

  ↓
```

## 6. 補足情報

### 6.1 実装上の注意点

1. **ファイルパスはツール内部で自律生成する**
   - 出力ファイルパスはLLMから受け取らない。`datetime.now().strftime("%Y%m%d%H%M%S")`をもとに内部生成する（LLM幻覚によるパス汚染防止）

2. **ツール関数は例外を再送出しない**
   - `ValidationError`・`KeyError`・その他の例外はすべてキャッチし、`{"success": False, "message": ...}` を返す
   - ファイル保存メソッドは `(False, エラーメッセージ文字列)` を返す

3. **ファイル書き込みは最大2回リトライする**
   - 1回目失敗後に2回目をリトライし、2回目も失敗した場合のみエラーを返す（実行制御方針 7.1準拠）

4. **ファイル保存メソッドの例外は3種類を個別定義する**
   - `IOError`, `PermissionError`, `Exception` を個別にキャッチする（`IOError` を `PermissionError` に統合しない）

5. **generate()の例外は2種類のみ**
   - `generate()` メソッドの例外クラスは `ValidationError` と `Exception` の2種類のみとする（`FileNotFoundError` は例外テーブルに定義しない。テンプレートの存在チェックは `os.path.exists()` で行う）

6. **ファイル存在チェックは os.path.exists() による事前チェックで実装する**
   - `open()` 実行時の `FileNotFoundError` 捕捉との併用は禁止し、事前チェックを優先する

7. **申請者名・申請日はLLMパラメータとして渡さない**
   - `applicant_name` と `application_date` は invocation_state から取得する
   - LLMがツール呼び出し時にこれらを引数として渡す必要はない

8. **segments必須キーガード**
   - `segments`の各要素辞書に必須キーが不足している場合、不足キー名を明示したエラーメッセージを返し、LLMが正しいキーを含めて再呼び出しできるようにする

9. **HumanApprovalHookはエージェント側で管理する**
   - ツール関数はファイル出力のみ担当する。HumanApprovalHookの制御はAG-002/AG-003が行う

10. **ログ出力はツール関数が担当する**
    - ErrorHandlerはログ出力しないため、ツール関数内でErrorHandler呼び出し前にログを記録する

### 6.2 パフォーマンス考慮事項

1. **ファイルリトライの上限**
   - 最大2回のリトライにより、一時的なファイルシステムエラーへの耐性を確保しつつ、無限ループを防止する

2. **テンプレートの都度読み込み**
   - Excelテンプレートはツール呼び出しごとに読み込む（副作用があるファイルのため静的キャッシュは使用しない）

### 6.3 セキュリティ考慮事項

1. **ファイルパスの自律生成**
   - タイムスタンプベースのファイルパス生成によりディレクトリトラバーサルを防止する

2. **ValidationErrorの内容マスキング**
   - `ErrorHandler.handle_validation_error()` でユーザー向けメッセージを生成し、Pydanticの内部スタックトレースをユーザーに見せない

## 8. テスト観点

### 8.1 機能テスト
- 正しい交通費申請情報（TransportApplicationData）を渡したとき、`data/output/{session_id}/`にExcelファイルが生成され、`success: True`が返ること
- 正しい経費申請情報（ExpenseApplicationData）を渡したとき、同様にExcelファイルが生成されること
- 生成されたExcelファイルの各セルに、入力データが列定義（A〜H列）通りに正しくマッピングされていること（承認状況H列は空欄）
- `application_data` に申請書の全項目と記入値が含まれること
- 監査ログ（DATA-008）にapplication_id・生成日時・申請者名・申請種別・ファイルパスが記録されること
- invocation_stateから申請者名・申請日・セッションIDが正しく取得されること

### 8.2 異常系テスト
- DATA-002（交通費テンプレート）が存在しない場合、`success: False` とエスカレーション促進メッセージが返ること（os.path.exists()で検出）
  - **期待戻り値**: `{"success": False, ...}`
- `segments` に必須キーが不足した辞書が含まれる場合、不足キー名を明示したエラーメッセージが返ること（KeyError）
  - **期待戻り値**: `{"success": False, ...}`
- `amount`に0を指定した場合、ValidationErrorに基づく日本語エラーメッセージが返ること（EX-01）
  - **期待戻り値**: `{"success": False, ...}`
- 申請期限（90日）を超過した `expense_date` を指定した場合、ValidationErrorが発生すること（BRL-12）
  - **期待戻り値**: `{"success": False, ...}`
- ファイル書き込みが2回失敗した場合（IOError）、`success: False` とエラーメッセージが返ること（EX-04）
  - **期待戻り値**: `{"success": False, ...}`
- ファイル書き込みがPermissionErrorで失敗した場合、権限エラーメッセージが返ること
  - **期待戻り値**: `{"success": False, ...}`
- `segments` が空リストの場合、`"移動区間情報のエラー"` が返ること

### 8.3 性能テスト
- 正常ケース（移動区間3件）で500ms以内に応答を返すこと
- ファイル書き込みリトライが発生した場合でも1000ms以内に完了すること

### 8.4 境界値テスト
- `application_date` と `expense_date` が同日（0日差）の場合、バリデーションを通過すること（BRL-12）
- `application_date` と `expense_date` が90日差の場合、バリデーションを通過すること（BRL-12）
- `application_date` と `expense_date` が91日差の場合、申請期限超過エラーが発生すること（BRL-12）
- `amount` が1の場合（最小値）、バリデーションを通過すること
- `segments` が1件（最小）の場合、正常に処理されること
- `expense_category` に判断不可の値（例: `"新幹線代"`）を渡した場合、`"その他経費"` に変換されること（BRL-17）
