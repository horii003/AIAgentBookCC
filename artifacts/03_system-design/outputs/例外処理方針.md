# 例外処理方針

> **参照元（システム要件定義資料）:**
> - 会話フロー一覧.md（例外が発生する会話フロー・エラー会話フロー）

## 1. 目的・適用範囲

### 1.1 目的

システム全体として **「例外とは何か」「どう分類し」「どう収束させるか」** の共通方針を定義する

個々のエージェント設計に **一貫した判断軸** を提供する

### 1.2 適用範囲

- マルチエージェントを含む全システム（AG-001, AG-002, AG-003）
- 業務処理（BIZ-01/BIZ-02）、外部連携（EXT-001/EXT-002/EXT-004）、UI/UX、運用を含む

### 1.3 本方針で定義しないこと

- エージェント内部の例外検知ロジック
- SDK／ライブラリ依存の例外クラス設計
- 具体的な再試行アルゴリズム

---

## 2. 例外処理の基本原則（システム共通）

- **例外は必ず収束させる（無限リトライ禁止）**
- **失敗は状態として管理する**
- **例外＝即エラー表示ではない**
- **ユーザーに内部失敗を見せない**
- **後続処理に影響を残さない**


---

## 3. 例外の定義と分類（システム観点）

### 3.1 例外の定義

「通常の業務フローから逸脱し、継続判断が必要な状態」

### 3.2 例外分類

| 分類ID | 分類名 | 説明 | 例 |
|-------|-------|------|---|
| EX-01 | 入力例外 | 入力不正・不足 | 申請内容テキストの空入力（GRD-001）、入力文字数500文字超（GRD-004）、対応外交通手段入力（GRD-002） |
| EX-02 | 判断不能 | AIが結論を出せない | 申請種別の自動判定不能（JD-02 / ERR-003）、交通費計算時の経路不明（TOOL-001） |
| EX-03 | 業務ルール違反 | 定義済ルール不一致 | 申請期限90日超過（BRL-14 / ERR-001）、申請書生成前の必須項目不足（BRL-04）、上長承認不取得のまま申請書提出を試みる（GRD-012） |
| EX-04 | 外部I/F例外 | ファイル参照失敗 | ナレッジ/マスタファイル参照失敗（ERR-004）、申請書テンプレート取得失敗（ERR-005）、経路/固定運賃テーブル参照失敗（TOOL-001） |
| EX-05 | 実行不可 | 権限・状態不整合 | 申請書提出操作の実行試行（GRD-012 / ACT-EXEC-01）、申請書生成前社員確認未取得の生成試行（GRD-014） |
| EX-06 | データ不整合 | データ形式不正 | 申請書生成失敗（テンプレートのフォーマット不正等）（ERR-006）、LLM画像読み取り失敗（領収書画像不正）（ERR-007） |
| EX-07 | システム障害 | 基盤障害 | Amazon Bedrock（LLMバックエンド）への接続失敗・タイムアウト |
| EX-08 | 想定外 | 未分類事象 | 予期しない例外（ERR-008）、LoopLimitError（ReActループ上限到達） |

## 4. 例外検知レベル

### 4.1 検知レベルの考え方

| レベル | 説明 | 検知主体 |
|-------|------|---------|
| 業務レベル | 業務定義上の異常（入力不正・ルール違反・判断不能） | エージェント（AG-001/AG-002/AG-003）が LLM 推論または条件判断で検知 |
| 処理レベル | 処理失敗（ファイル参照失敗・申請書生成失敗・LLM画像読み取り失敗） | ツール（TOOL-001/TOOL-002）が例外として検知 |
| システムレベル | 基盤・インフラ異常（Bedrock 接続失敗・LoopLimitError） | ErrorHandler が EventLoopException / Exception として検知 |

### 4.2 検知責務

- 検知は **エージェントまたはツール**
- **判断・収束はErrorHandler** が行う

## 5. 例外発生時の基本対応フロー

### 5.1 共通対応フロー

1. 例外検知（エージェント/ツール）
2. 例外分類（ErrorHandler）
3. 継続可否判定（ErrorHandler）
4. 処理分岐（再試行／代替／停止）
5. 状態更新（セッション管理）
6. ログ・監査記録（ErrorHandler）

## 7. 例外ハンドリング実装

### 7.1 ErrorHandlerの責務方針

ErrorHandlerは以下の責務を担う。

- 例外分類（EX-01〜EX-08）ごとに継続可否を判定し、対応方針（再試行・代替処理・エスカレーション・中断）を決定する
- ユーザー向けの日本語メッセージを生成し、技術的な詳細を隠蔽する
- 例外発生時のログ出力を行い、スタックトレースはログにのみ記録する

**エラー伝播の責務分担**:

| 層 | 責務 |
|---|------|
| 各モジュール（エージェント・ツール） | 例外発生箇所でログを出力し、ErrorHandler へ処理を委譲する |
| ErrorHandler | ユーザー向けメッセージを生成して呼び出し元に返す |
| 呼び出し元 | 受け取ったメッセージを戻り値または表示として上位レイヤーへ伝播させる |

具体的なメソッド設計（メソッド名・引数・戻り値）は基本設計（ハンドラー基本設計.md）で定義する。

> ※ 基本設計書への参照: ハンドラー基本設計.md 2.3節


### 7.2 エラーメッセージ設計方針

**ユーザーフレンドリー**:
- 技術的な詳細を隠蔽（スタックトレース・エラーコードは社員に表示しない）
- 具体的な対処方法を提示（再入力・担当部門への問い合わせ先等）
- 丁寧な言葉遣い（「〜できませんでした。〇〇をお試しください」形式）

**情報の充実**:
- エラーの原因を日本語で簡潔に説明
- 次の操作（再入力・担当部門への連絡）を明示
- 必要に応じて問い合わせ先を提示（CF-006）

**例**:
```
申し訳ありません。ファイルの参照中にエラーが発生しました。
担当部門（管理部）にお問い合わせいただくか、しばらく経ってから再度お試しください。
```
```
申請期限（経費発生日から90日以内）を超過しているため、この申請は受け付けられません。
担当部門にご確認ください。
```

## 8. 例外とエスカレーションの関係

### 8.1 エスカレーション対象例外

| 例外分類 | エスカレーション要否 | 行き先 |
|---------|------------------|-------|
| 外部I/F例外（EX-04）：ナレッジ/テンプレート参照失敗 | 要（代替手段なし） | CF-006（エラー・エスカレーション案内）→ 担当部門（管理部門） |
| 実行不可（EX-05）：申請書提出試行・社員確認未取得 | 要 | CF-006 → 担当部門（管理部門） |
| データ不整合（EX-06）：申請書生成失敗（リトライ後） | 要 | CF-006 → 担当部門（管理部門） |
| システム障害（EX-07）：Bedrock接続失敗（リトライ上限後） | 要 | CF-006 → 担当部門（管理部門） |
| 想定外（EX-08）：予期しない例外・LoopLimitError | 要 | CF-006 → 担当部門（管理部門） |

### 8.2 エスカレーション判定責務

**判定主体**: ErrorHandler

**判定基準**: 再試行または代替処理による自動収束が不可能な場合、またはシステムの継続運転が不可能な場合にエスカレーション対象と判定する

## 9. セッション状態への反映方針

### 9.1 例外時の状態更新

| 状態 | 説明 | トリガー |
|-----|------|---------|
| ACTIVE | 処理中（エラー発生前） | - |
| WAITING | 人対応待ち | EX-01（入力例外）・EX-02（判断不能）で社員の再入力・選択を待つ |
| TERMINATED | 異常終了 | EX-04〜EX-08 でエスカレーション後フロー終了・EX-03 で申請期限超過フロー終了・EX-08 でLoopLimitError発生時 |

### 9.2 再開可否

**再開可能な例外**:
- EX-01（入力例外）：社員が再入力すれば継続可能
- EX-02（判断不能）：社員が選択肢から申請種別を選択すれば継続可能（CF-002）
- EX-06（LLM画像読み取り失敗）：手動入力フローへ切り替えて継続可能（JD-15）

**再開不可な例外**:
- EX-03（申請期限超過）：申請不可のためフロー終了（新規セッションで最初から実施）
- EX-04（ファイル参照失敗）：CF-006後に担当部門の対応が必要（担当部門対応後に新規セッション）
- EX-07/EX-08：CF-006後にシステム復旧または担当部門対応が必要

## 10. UX観点の扱い（概要）

**利用者への通知有無**:
- 業務エラー（EX-01/EX-02/EX-03）：社員へ通知する（対応方法を提示）
- 技術エラー（EX-04/EX-06/EX-07）：社員へ通知するが技術的詳細は隠蔽する（CF-006 経由）
- システム内部の例外（EX-05/EX-08）：社員へ通知する（CF-006 経由）

**エラーメッセージの抽象度**:
- 社員向けメッセージは業務的な言葉で記述し、エラーコード・スタックトレース・内部状態は含めない
- ログには詳細な技術情報（エラーコード・スタックトレース要約・発生エージェントID・フローID）を記録する

**次の行動を必ず提示**:
- 再入力が可能な場合：「〇〇を入力してください」
- 手動対応が必要な場合：「担当部門（管理部）にお問い合わせください」
- フロー終了の場合：「最初からやり直してください」

## 11. ログ・監査方針

### 11.1 例外ログ

| 項目 | 内容 | 出力先 |
|-----|------|-------|
| error_type | エラー分類ID（EX-01〜EX-08）と対応ERRコード（ERR-001〜ERR-008） | ログストア（DATA-011: LOG-STD または DATA-012: LOG-HI） |
| error_message | ユーザー向けメッセージ（日本語） | ログストア（DATA-011/DATA-012） |
| context | エラー発生エージェントID・フローID（CF-xxx）・ステップID | ログストア（DATA-011/DATA-012） |
| exc_info | スタックトレース要約（全文は除外） | ログストア（DATA-012: LOG-HI、技術エラー・想定外例外のみ） |
| timestamp | タイムスタンプ（ISO8601） | ログストア（DATA-011/DATA-012） |


### 11.2 監査観点

**再現可能性**:
- request_id を全ログに含め、1申請フロー全体の例外発生経緯を時系列に追跡可能とする

**判断経路の説明可能性**:
- 例外分類 ID（EX-xx）・エラーコード（ERR-xxx）・適用ルールID（BRL-xx/JD-xx）を記録することで、どのルールの何の判断結果として例外が発生したかを説明可能とする

## 12. 設計上の留意点・リスク

**例外過多によるUX低下**:
- 業務エラーと技術エラーが同じ頻度で社員に表示されると混乱が生じる可能性がある
- 対策: EX-01/EX-02（入力例外・判断不能）は会話の中で自然に対応し、技術エラー（EX-04〜EX-08）のみ CF-006 へ移行する

**例外の握り潰し**:
- ツール関数が例外を握り潰して `{"success": False}` のみを返すと、ErrorHandler による適切な分類・ログ記録が行われない可能性がある
- 対策: ツール関数は例外メッセージを `{"success": False, "message": エラーメッセージ}` として返し、ErrorHandler に委譲する（再送出しない）

**想定外例外の扱い**:
- EX-08 で分類されない新たな例外が実装後に発生する可能性がある
- 対策: LoopLimitError の `__cause__` 判定および最外層の Exception キャッチにより、未分類例外を EX-08 として収束させ CF-006 へ誘導する
