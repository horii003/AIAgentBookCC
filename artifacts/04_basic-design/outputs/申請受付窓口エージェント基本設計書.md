## 2. 目的・役割定義

### 2.1 エージェントの目的
- 社員が入力した申請内容テキストから申請種別（交通費精算申請・経費精算申請）を自動判断し、適切な申請先を提示する
- 申請種別判断後、対応する専門エージェント（AG-002またはAG-003）へ申請フローを委譲する
- 社員にとっての「申請受付窓口」として、どの申請フローを進めるべきかをガイドする

### 2.2 主な責務
- **申請内容テキスト受付**: 社員が入力した申請内容テキスト（最大500文字）を受け取る（GRD-001, GRD-004）
- **申請種別判断**: 社内申請ルール（DATA-001）を参照して申請種別（交通費精算申請・経費精算申請）を判断する（BRL-01）
- **判断不能時の選択肢提示**: 申請種別を自動判定できない場合は選択肢を提示して社員に確認する（BRL-09）
- **申請書・申請先提示**: 申請種別が確定したら対応する申請書・申請先を提示する（BRL-02）
- **専門エージェントへの委譲**: 申請種別確定後、AG-002またはAG-003をツールとして呼び出して申請フローを委譲する（LNK-001/LNK-002）

### 2.3 非責務（やらないこと）
- **移動情報・経費情報の収集**: AG-002（交通費）またはAG-003（経費）が担当
- **交通費計算・申請書生成**: AG-002が担当（TOOL-001/TOOL-002）
- **申請書の提出**: 社員が手動で実施（ACT-EXEC-01、P-EXEC-003禁止）
- **申請者名の収集**: アプリケーション起動時（main.py）に取得済み。AG-001は対話で再収集しない

## 3. 自律度・判断権限

### 3.1 自律度レベル

| レベル | 内容 | 適用範囲 |
|------|------|----------|
| 参照（Lv1） | 社内申請ルールファイル・申請先情報の参照 | P-READ-001 / DATA-001, DATA-004 |
| 提案（Lv2） | 申請種別の自動判断（社内申請ルール照合） | ACT-DEC-01 / FR-001 |
| 実行（Lv3） | 不足情報質問の生成（対話継続） | ACT-GEN-02 |

### 3.2 判断権限の範囲
- **自律判断可能**: 申請種別の自動判定（交通費精算申請・経費精算申請の範囲内）、判断不能ケースの検知と選択肢提示、申請先の特定
- **人間確認必要**: 申請種別判断不能時の社員選択（COND-001）
- **実行不可**: 申請書生成（P-EXEC-002禁止）、申請書提出（P-EXEC-003禁止 / ACT-EXEC-01）

## 4. 処理フロー設計

### 4.1 基本フロー（概要）

```text
開始（申請者名は起動時に取得済み）
→ 社員から入力を受け取る
  ├── [特殊コマンド検出: reset / リセット / 最初から]
  │   → 会話履歴と申請者情報をリセットして初期状態に戻す（§4.5参照）
  └── [通常入力]
      → 申請内容テキストを受け取る（CF-001）
      → 申請内容テキストのバリデーション（GRD-001/GRD-004）
      → 対話回数上限チェック（turn_count が上限到達時は TERMINATED へ遷移してフロー終了）
      → 社内申請ルール参照（DATA-001）
      → 申請種別判断（BRL-01 / FR-001）
        ├── [判断可能] 申請種別・申請書・申請先を提示（BRL-02 / CF-001 Step 3）
        │   └── 複数種別該当時は全て提示（BRL-03）
        └── [判断不能] 選択肢提示（BRL-09 / CF-002）→ 社員選択待ち → 申請種別確定
      → 専門エージェント（AG-002またはAG-003）へ委譲（CF-003/CF-004）
      → 申請フロー完了（CF-005完了後、申請先案内）
      → 終了
```

### 4.2 フロー制御の考え方
- **原則的な処理順序**: 逐次処理（申請種別判断 → 委譲という順序は変わらない）
- **処理をスキップする考え方**: 申請種別が入力テキストから明確な場合は選択肢提示をスキップして直接委譲する
- **同一処理を複数回実行**: 判断不能時に社員が選択肢から未選択の場合は再度選択肢を提示する（CF-002のループ）
- **対話回数上限チェック**: turn_count がセッションの上限（30回: GRD-005）に達した場合、セッション状態を TERMINATED に遷移してフローを終了する（EX-08として処理）
- **特殊コマンドの優先処理**: 社員入力が特殊コマンド（§4.5）と一致した場合は、通常フローに入る前に優先して処理する

### 4.5 特殊コマンド設計

| コマンド | 処理内容 |
|---------|---------|
| `reset` / `リセット` / `最初から` | 会話履歴（SlidingWindowConversationManager のバッファ）と申請者情報（invocation_state の applicant_name・application_date・session_id）をリセットし、セッションを初期状態（アプリケーション起動直後）に戻す |

**リセット処理の詳細**:
- SlidingWindowConversationManager の会話バッファをクリアする
- applicant_name は起動時入力に戻す（再入力を促す）
- session_id を SM-001 で新規発行し、新しいセッション状態ファイルを作成する
- セッション状態を CREATED に設定する
- リセット完了後、「リセットしました。最初からやり直します。お名前を入力してください。」と案内する

---

### 4.3 状態管理
- **セッションID**: SessionManager（SM-001）が発行・管理。invocation_state 経由で専門エージェントに伝播
- **申請者名（applicant_name）**: アプリケーション起動時に取得済み。invocation_state に設定して専門エージェントに伝播
- **申請日（application_date）**: システム日付（YYYY-MM-DD）を自動取得。invocation_state に設定して専門エージェントに伝播
- **申請種別**: AG-001が判断・確定後に固定。AG-002/AG-003は変更しない（マルチエージェント連携設計.md §8.1）

## 5. 専門エージェントとの連携設計

### 5.1 連携対象エージェント

| エージェントID | エージェント名 | 連携目的 | 振り分け条件 |
|---------------|---------------|----------|--------------|
| AG-002 | 交通費精算申請エージェント | 交通費精算申請フロー全体（情報収集・計算・申請書生成）を委譲 | 申請種別 = 交通費精算申請（自動判定またはCF-002での社員選択） |
| AG-003 | 経費精算申請エージェント | 経費精算申請フロー全体（情報収集・OCR・申請書生成）を委譲 | 申請種別 = 経費精算申請（自動判定またはCF-002での社員選択） |

### 5.2 振り分け判断基準

- **AG-002へのルーティング**: 申請内容テキストが「出張」「交通費」「電車」「移動」「往復」等、交通費精算に関するキーワードを含む場合、または社員が CF-002 で「交通費精算申請」を選択した場合
- **AG-003へのルーティング**: 申請内容テキストが「領収書」「経費」「食事」「宿泊」「接待費」「事務用品」等、経費精算に関するキーワードを含む場合、または社員が CF-002 で「経費精算申請」を選択した場合
- **判断不能**: いずれにも該当しない、または双方に該当する場合は CF-002（選択肢提示）へ遷移する（BRL-09 / JD-02）

**詳細な判断基準**: 詳細設計書 システムプロンプト全文を参照

### 5.3 invocation_stateの受け渡し

専門エージェントのツール関数呼び出し前に、ToolContext の `invocation_state` に以下を設定する（LLMのプロンプトには含めない）。

```python
context.invocation_state = {
    "session_id": session_id,           # SM-001が発行
    "applicant_name": applicant_name,   # アプリケーション起動時に取得済み
    "application_date": application_date  # date.today().isoformat()
}
```

**データ構造の詳細**: データモデル基本設計書.md §5.6（InvocationState）を参照

## 6. ツール設計

### 6.1 利用ツール

- **transport_agent_tool（AG-002 as tool）**: 申請種別が交通費精算申請と確定した場合に呼び出す。`@tool(context=True)` でデコレートしたファクトリ関数 `_get_transport_agent(session_id)` で生成したエージェントをツールとして呼び出す
- **expense_agent_tool（AG-003 as tool）**: 申請種別が経費精算申請と確定した場合に呼び出す。`@tool(context=True)` でデコレートしたファクトリ関数 `_get_expense_agent(session_id)` で生成したエージェントをツールとして呼び出す

AG-001 は申請種別確定後に上記2ツールのいずれか1つのみを呼び出す。同一セッション内での同時呼び出しは禁止（実行制御方針 §5.1）。

**ツールの詳細**: 各専門エージェント基本設計書（AG-002 / AG-003）を参照

## 7. プロンプト設計

### 7.1 システムプロンプトの目的
- 申請種別判断のルール（BRL-01）と判断不能時のフォールバック（BRL-09）を定義する
- 申請書・申請先の提示方法（BRL-02/BRL-03）を定義する
- 申請書提出の禁止（GRD-012 / ACT-EXEC-01）を明示する
- 専門エージェントへの委譲条件と委譲時のメッセージ形式を定義する

### 7.2 プロンプト構成要素
- **役割定義**: 社内申請受付窓口として、申請種別判断・申請先案内・専門エージェント委譲を行うオーケストレーター
- **処理フロー**: 申請内容受付 → 申請種別判断 → （判断不能時）選択肢提示 → 申請書・申請先提示 → 専門エージェント委譲
- **判断プロトコル**: 社内申請ルール（DATA-001）に基づく申請種別判断ロジック（BRL-01, BRL-03, BRL-09）
- **主な責任**: 申請種別の正確な判定、社員へのわかりやすい案内、専門エージェントへの適切な委譲
- **動的情報**: なし（AG-001のシステムプロンプトは静的定数。申請期限チェック等の日付依存ロジックはAG-002/AG-003が担当）

**プロンプト全文**: 詳細設計書 2.3.1章を参照

### 7.3 システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | **静的定数** |
| 動的生成の理由 | 不要。AG-001は申請種別判断のみを担当し、申請期限チェック（90日）等の日付ベースのルール判断はAG-002/AG-003が担当するため |
| 動的に埋め込む情報 | なし |

### 7.4 主要な設計意図
- **静的プロンプト**: AG-001は日付依存のルールを扱わないため静的プロンプトで十分。動的生成のオーバーヘッドを排除する
- **判断と委譲の分離**: 申請種別判断（AG-001）と申請フロー実行（AG-002/AG-003）を明確に分離し、各エージェントの責務をシンプルに保つ
- **根拠提示**: 申請種別判断時は参照したルールID（BRL-01等）を提示し、社員が確認できるようにする（根拠提示方針参照）

## 10. 入力データ設計

### 10.1 初期化時の入力
- **申請者名（applicant_name）**: 必須。アプリケーション起動時（main.py）に取得済みで、`invocation_state` にセット済み。AG-001の対話では再収集しない
- **申請日（application_date）**: `date.today().isoformat()` で自動取得。`invocation_state` にセット済み
- **セッションID（session_id）**: SM-001が起動時に発行済み

### 10.2 対話ループでの入力
- **申請内容テキスト（D-001）**: 社員が入力した申請の内容。最大500文字（GRD-004）、空入力禁止（GRD-001）
- **申請種別の選択（CF-002時）**: 「交通費精算申請」または「経費精算申請」の選択回答
- **特殊コマンド**: `reset` / `リセット` / `最初から`。通常テキストより優先して処理される（§4.5参照）

## 11. 出力データ設計

### 11.1 出力形式
- **出力先**: 社員（チャットインターフェース）
- **出力内容**: 申請種別・申請書名・申請先の案内メッセージ、判断根拠（参照したルールID）、専門エージェント委譲後の申請先案内

### 11.2 出力メッセージ
- 申請種別提示: `"申請内容を確認しました。「交通費精算申請」に該当します（根拠: BRL-01）。交通費申請書_template.xlsx を使用し、総務部に申請してください。"`
- 判断不能時: `"申請種別の判断ができませんでした。以下からお選びください：1. 交通費精算申請 / 2. 経費精算申請"`
- 委譲完了後の申請先案内: `"申請書（下書き）が生成されました。[申請先情報]に提出してください。"`

## 12. バリデーション設計

**システム設計書の方針**: `artifacts/03_system-design/outputs/バリデーション方針.md` を参照。

### 12.1 バリデーション方針
- **申請内容テキスト**: 空入力禁止（GRD-001）、500文字以内（GRD-004）。`AgentDelegationMessage.user_input_text` の `max_length=500` で保証
- **申請種別**: `normalize_application_type` バリデーターで別表記を正規値に変換。LLMが判断した申請種別は `AgentDelegationMessage.application_type` を通じて型安全に渡す

### 12.2 申請ルール一覧

| ルールID | ルール名 | 条件 | エージェントの対応 |
|----------|----------|------|------------------|
| BRL-01 | 申請種別判断 | 申請内容テキストが存在し社内申請ルールが参照可能な場合 | 社内申請ルール照合で申請種別（交通費/経費）を判断 |
| BRL-02 | 申請書・申請先提示 | 申請種別が確定した直後 | 申請書名・申請先部門を根拠ルールIDとともに提示 |
| BRL-03 | 複数申請種別の全提示 | 申請内容に複数申請種別が該当する場合 | 該当する全申請種別・申請書・申請先を提示 |
| BRL-09 | 申請種別判断不能時の選択肢提示 | 申請種別が自動判定できない場合（JD-02） | 「交通費精算申請」「経費精算申請」の選択肢を提示して社員に確認 |

## 13. Agent as Tools設計

### 13.1 AG-001のオーケストレーター役割
AG-001はオーケストレーターであり、他エージェントをツールとして呼び出す側である。AG-001自身はAgent as Toolsとして他エージェントから呼び出される設計ではない。

### 13.2 専門エージェントのツール化

| ツール名 | 呼び出し対象 | デコレータ | 役割 |
|---------|------------|-----------|------|
| transport_agent_tool | AG-002（交通費精算申請エージェント） | @tool(context=True) | 申請種別=交通費精算申請時に呼び出す |
| expense_agent_tool | AG-003（経費精算申請エージェント） | @tool(context=True) | 申請種別=経費精算申請時に呼び出す |

### 13.3 ToolContextの使用（専門エージェント側）
専門エージェント（AG-002/AG-003）のツール関数内では以下を実施する（AG-001が invocation_state を設定し、専門エージェントが読み取る）:
- `context.invocation_state` から `session_id`, `applicant_name`, `application_date` を取得する
- `InvocationState.model_validate(context.invocation_state)` で型検証を実施する
- `session_id` でファクトリ関数を使用してエージェントインスタンスを生成する

**詳細**: 詳細設計書 2.5章を参照

### 14.1 テスト観点の概要

## 16. 制約事項・前提条件

### 16.1 技術的制約
- Strands Agents SDK v1.25.0 の `Agent` クラスを使用する
- LLMバックエンドは Amazon Bedrock（Claude Sonnet 4.5）のみ
- `SlidingWindowConversationManager(window_size=30, should_truncate_results=True)` を使用する
- AG-002/AG-003 の並列起動は禁止（実行制御方針 §5.1）

### 16.2 業務的制約
- 申請書生成・提出はAG-001が行わない（P-EXEC-002禁止、P-EXEC-003禁止）
- 申請者名の収集は対話で行わず、アプリケーション起動時に取得済みの値を使用する
- 申請種別は交通費精算申請・経費精算申請の2種類のみ対応

### 16.3 前提条件
- アプリケーション起動時に `applicant_name` が取得済みで `invocation_state` にセット済みであること
- セッションID（`session_id`）がSM-001により発行済みで `invocation_state` にセット済みであること
- 社内申請ルールファイル（DATA-001）が `data/` 配下に配置されていること
