## 1. 概要

### 1.1 コンポーネントの目的

経費精算申請に特化したスペシャリストエージェントとして、AG-001からの委任を受けて申請フロー全体を実行する。領収書画像からの情報自動抽出（BRL-16）と経費区分の自動判断（BRL-17）を行い、申請書ドラフト（Excel/.xlsx）を生成する。申請期限チェック（BRL-18）・上長承認要否判定（BRL-19）・差し戻しリスク評価（BRL-08）を実施してチェック結果をユーザーへ提示する。

### 1.2 主要な責務

1. **経費情報の一括収集**: CF-004でユーザーから全経費明細の情報（購入日・店舗名・品目・金額）をまとめて一括収集する。1つずつ順番に聞くのではなく、必要項目を1メッセージで一括収集する（BRL-16支援）
2. **領収書画像からの情報自動抽出**: Strands Agents SDK提供の `image_reader` 相当機能を使って領収書画像から店舗名・金額・日付・品目を自動抽出する（BRL-16）
3. **経費区分の自動判断**: LLMの判断で品目から経費区分（事務用品費・宿泊費・資格精算費・その他経費）を自動判断して提示する（BRL-17）
4. **業務目的の確認**: 全明細収集後に業務目的を収集する（BRL-20）
5. **Human-in-the-Loop承認取得**: 収集済み申請情報をテキストで整理・提示してOK/修正/キャンセルの3択を取得する（BRL-06）
6. **申請書の生成**: HumanApprovalHook（BeforeToolCallEvent）承認後にTOOL-002（generate_expense_form）でExcelドラフトを生成する（BRL-05、FR-010）
7. **申請ルールチェック**: 申請期限（BRL-18）・上長承認要否（BRL-19）・差し戻しリスク（BRL-08）をチェックし結果を提示する（CF-006）
8. **申請書最終提示・提出促し**: 申請書ドラフト・チェック結果を提示し、ユーザーへ提出操作を促す（BRL-09、CF-007）

### 2.2 インターフェース設計

#### 2.2.1 入力

**初期化時の入力（invocation_stateから受け取る）**:
- `applicant_name` (str): 申請者名（min_length=1）
- `application_date` (str): 申請日（YYYY-MM-DD形式）
- `session_id` (str): セッションID（ファクトリ関数で使用、エージェントへは渡さない）

**対話ループでの入力**:
- 領収書画像: ユーザーが画像を提出する（CF-004 Step 2）
- 経費情報一括入力（画像なし・抽出失敗時）: 購入日・店舗名・品目・金額をユーザーが一括で自由文入力する
- 業務目的: ユーザーが自由文で入力する（BRL-20）
- Human-in-the-Loop選択: OK/修正/キャンセルから選択する
- 特殊コマンド:
  - 経費区分判断不能時の4択選択（事務用品費/宿泊費/資格精算費/その他経費）

**パラメータ（Agent as Toolsとして）**:
- `query` (str): AG-001から渡される委任メッセージ

**ToolContext経由で受け取る情報**:
```python
{
    "session_id": str,       # ファクトリ関数でFileSessionManagerの初期化に使用
    "applicant_name": str,   # 申請者名
    "application_date": str  # 申請日（YYYY-MM-DD形式）
}
```

#### 2.2.2 出力

**出力先**: なし（`callback_handler=None` により標準出力への直接表示を抑制。AG-001が代わりに表示する）

**出力メッセージ**:
- 経費情報一括収集促し（手動入力）: `"以下の情報をまとめてご入力ください。\n---\n購入日: YYYY-MM-DD\n店舗名: \n品目: \n金額（円）: \n---\n複数明細ある場合は繰り返してご入力ください。"`
- 領収書抽出結果提示: `"領収書から以下の情報を読み取りました：店舗名:[店舗名]、金額:[金額]円、日付:[日付]、品目:[品目]"`
- 経費区分判断結果提示: `"品目「[品目]」の経費区分は「[経費区分]」と判断しました。"`
- 経費区分4択提示: `"経費区分を選択してください：\n1. 事務用品費\n2. 宿泊費\n3. 資格精算費\n4. その他経費"`
- Human-in-the-Loop確認: `"収集した申請情報を確認してください。\n[申請情報サマリー]\nOK・修正・キャンセルのいずれかを入力してください。"`
- 申請書生成完了: `"申請書ドラフトを生成しました: [file_path]"`
- 申請期限超過通知（CF-008）: `"申し訳ありません。経費発生日から3ヶ月を超えているため、この申請は受け付けられません。総務部門にご相談ください。"`
- 上長承認要警告（CF-009）: `"経費金額が5,000円を超えています。事前に上長の承認を得てください。上長承認済みですか？"`
- 最終提示（CF-007）: `"申請書ドラフトは [file_path] に保存されました。チェック結果を確認の上、申請システムから提出してください。"`

**戻り値の形式**: `str`（申請フロー完了メッセージまたはエラーメッセージ）

### 2.3 ビジネスロジック

#### 2.3.1 システムプロンプト

##### システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 動的生成関数（外部ポリシーファイル参照） |
| 動的生成の理由 | CF-006のルールチェックで申請期限チェック（BRL-18）を実行するため。経費発生日からの経過期間（3ヶ月以内）を判断するために実行時の申請日（application_date）をプロンプトに埋め込む必要がある |
| 外部ポリシーファイル参照 | `agent_knowledge/receipt_policies.py` を初期化時に読み込み、バリデーションルール定義をプロンプトに組み込む |

##### 動的に埋め込む情報
- `application_date`: 申請日（invocation_stateから取得。YYYY-MM-DD形式）
- `deadline_date`: 申請期限基準日（`application_date` - 3ヶ月。BRL-18の判断基準）
- 経費ポリシールール: `agent_knowledge/receipt_policies.py` から読み込んだバリデーションルール定義

##### プロンプト全文

```
あなたは経費精算申請専門エージェントです。
AG-001（申請受付窓口エージェント）から委任を受け、経費精算申請フロー全体を担当します。

【申請情報】
- 申請日: {application_date}
- 申請期限基準日（この日付より前の経費発生は申請不可）: {deadline_date}

【役割と責任範囲】
- 経費精算申請フロー全体（領収書情報抽出・経費区分判断・申請書生成・ルールチェック・最終提示）を担当します
- 交通費精算申請フロー、申請種別判定はAG-001/AG-002が担当します
- 申請書の自動提出は禁止です（BRL-09）。提出操作は利用者のみが実施します
- 交通費計算ツール（calculate_travel_expense）は使用しません

【領収書画像からの情報抽出（BRL-16）】
1. まずユーザーに領収書画像の提出を促してください
2. 画像が提出された場合: image_readerを使用して店舗名・金額・日付・品目を自動抽出する
3. 抽出成功時: 抽出した情報をユーザーに提示し確認を取る
4. 画像なし・抽出失敗時: 以下の手動入力フォームを提示する（1つずつ聞いてはいけません）

手動入力フォーム例:
---
以下の情報をまとめてご入力ください。
購入日: YYYY-MM-DD
店舗名: （例: コンビニA）
品目: （例: ノート、ホテル代）
金額（円）:
---
複数明細ある場合は、明細の数だけ上記フォームを繰り返してください。
全明細の入力が完了したら「完了」と入力してください。

【経費区分の自動判断（BRL-17）】
品目から以下の4区分のいずれかをLLMが判断して提示してください:
- 事務用品費: ノート、ペン、文房具など
- 宿泊費: ホテル代、旅館代など
- 資格精算費: 試験費用、資格取得費用など
- その他経費: 上記に該当しない業務経費

判断不能な場合は以下を提示してください:
「経費区分を選択してください：
1. 事務用品費
2. 宿泊費
3. 資格精算費
4. その他経費」

ユーザーの選択（1〜4）で経費区分を確定します。

【業務目的の確認（BRL-20）】
全明細の収集完了後、業務目的を確認してください。業務目的が入力されるまでTOOL-002を呼び出してはいけません。

【ドラフト提示とHuman-in-the-Loop承認（BRL-06）】
1. 収集した全申請情報をテキストで整理・提示する（この時点ではTOOL-002を呼び出してはいけない）
2. ユーザーにOK・修正・キャンセルの選択を求める
3. OKの場合のみgenerate_expense_formを呼び出す
4. 修正の場合は情報収集に戻る
5. キャンセルの場合はセッションを終了する

【generate_expense_form の引数形式】
generate_expense_form を呼び出す際は、以下の形式で引数を渡してください:
- applicant_name: 申請者名（文字列）
- application_date: 申請日（YYYY-MM-DD形式の文字列）
- business_purpose: 業務目的（文字列）
- items: 経費明細リスト（以下の形式のdictのリスト）
  各itemの必須キー:
  - "purchase_date": 購入日（"YYYY-MM-DD"形式の文字列）
  - "store_name": 店舗名（文字列）
  - "item_name": 品目（文字列）
  - "expense_category": 経費区分（"事務用品費"/"宿泊費"/"資格精算費"/"その他経費"のいずれか）
  - "amount": 金額（整数、円）

【申請ルールチェック（BRL-07）】
申請書生成後、以下をチェックしてユーザーに提示してください:
1. 申請期限チェック（BRL-18）: 各購入日が申請期限基準日（{deadline_date}）以降であることを確認する。超過している場合はCF-008へ遷移し「申し訳ありません。経費発生日から3ヶ月を超えているため、この申請は受け付けられません。総務部門にご相談ください。」を提示する
2. 上長承認要否（BRL-19）: 経費合計が5,000円超の場合は「経費金額が5,000円を超えています。事前に上長の承認を得てください。上長承認済みですか？」を提示する
3. 差し戻しリスク評価（BRL-08）: リスクが高いと判断した場合は警告を提示する（判定基準は要件上未定義）

【最終提示（CF-007）】
申請書ドラフトのファイルパスとチェック結果を提示し、申請システムからの提出操作を促してください。

【エラー発生時の対応】
ツールエラー・システム障害等のシステム系エラーが発生した場合は、エラー内容を要約して呼び出し元エージェント（AG-001）に返してください。AG-001がユーザーへエスカレーション案内を提示します。
```

##### プロンプト設計の意図

| 要素 | 意図 |
|------|------|
| 一括収集フォーム | 1つずつ質問するよりも会話ターン数を削減し、BRL-23のターン数制限リスクを低減する |
| 経費区分判断の4択 | 判断不能時に明確な選択肢を提示し、誤分類を防止する（BRL-17） |
| ドラフト提示分離 | ツール呼び出し前にテキスト提示を行い、HumanApprovalHookとの連携を明確化する |
| dict型引数の明示 | generate_expense_formのitemsフィールドの必須キーをすべて明記し、キー省略によるエラーを防ぐ |
| 申請期限の動的埋め込み | LLMが実行時のapplication_dateを正確に判断できるよう deadline_date を埋め込む |
| 外部ポリシーファイル参照 | バリデーションルールを `agent_knowledge/receipt_policies.py` から読み込むことで、ルール変更時のプロンプト修正を不要にする |

#### 2.3.2 申請ルール

| ルール | 条件 | 対応 |
|--------|------|------|
| BRL-16: 領収書情報自動抽出 | 領収書画像が提出された場合 | image_readerで自動抽出する。抽出失敗時は手動入力フォームへ遷移 |
| BRL-17: 経費区分自動判断 | 品目から経費区分を判断する場合 | LLMが4区分のいずれかを判断する。判断不能時は4択を提示 |
| BRL-18: 申請期限チェック | CF-006実行時 | 購入日が deadline_date 以降かチェックする。超過時はCF-008へ遷移 |
| BRL-19: 上長承認要否判定 | CF-006実行時 | 経費合計が5,000円超の場合にCF-009へ遷移 |
| BRL-20: 業務目的必須 | 業務目的が未入力の場合 | 業務目的の入力を促す。業務目的なしでTOOL-002を呼び出さない |

#### 2.3.3 処理フロー

```
開始（expense_application_agent_tool 呼び出し）
  ↓
ToolContextからsession_id・applicant_name・application_dateを取得
  - session_idはファクトリ関数でFileSessionManagerの初期化に使用
  - applicant_name（min_length=1）・application_date（YYYY-MM-DD形式）を個別バリデーション
    → バリデーション失敗 → エラーメッセージを返す
  ↓
_agent_instances[session_id] が存在しない場合のみ Agent インスタンスを生成
  - build_expense_system_prompt(application_date) でシステムプロンプトを動的生成
  - Agent(callback_handler=None, ...) を生成してキャッシュ
  ↓
agent(query, invocation_state={"applicant_name": applicant_name, "application_date": application_date}) を呼び出す
  ↓
【エージェント内部での処理フロー（システムプロンプトに基づくLLMの動作）】
  1. 領収書画像の提出を促す（BRL-16）
  2-a. 画像あり: image_readerで情報を自動抽出する
  2-b. 画像なし・抽出失敗: 手動入力フォームを提示して経費情報を一括収集する
  3. 品目から経費区分を自動判断する（BRL-17）
     → 判断不能: 4択を提示してユーザー選択で確定
  4. 追加明細の確認・収集をループ
  5. 業務目的を収集する（BRL-20）
  6. 収集済み申請情報をテキストで整理・提示（TOOL-002は呼び出さない）
  7. OK/修正/キャンセルをユーザーから取得
     → 修正: Step 1へ戻る
     → キャンセル: セッション終了メッセージを返す
     → OK: 次ステップへ
  8. generate_expense_form を呼び出す（HumanApprovalHook（BeforeToolCallEvent）が承認後に実行）
     → success=True: ファイルパスを取得
     → success=False: エラーメッセージをAG-001へ返す
  9. 申請期限チェック（BRL-18）・上長承認要否（BRL-19）・差し戻しリスク（BRL-08）を実施
  10. 申請書ドラフト・チェック結果・最終提示をユーザーへ提示（CF-007）
  ↓
処理完了メッセージを AG-001 へ返す
```

### 2.4 設定・構成

#### 2.4.1 共通設定

| 設定項目 | 設定値 | 備考 |
|---------|--------|------|
| 会話管理 | SlidingWindowConversationManager(window_size=15, should_truncate_results=True, per_turn=False) | 経費情報収集・申請書生成・ルールチェックを行うため。AG-002より区間数が少ないため15で十分 |
| リトライ戦略 | 標準設定（Strands Agents SDK組み込み） | 6回/4秒/240秒。Amazon Bedrockの一時的エラーへのリトライのみ適用 |
| ループ制御 | LoopControlHook(max_iterations=10) | 暴走防止。情報収集・ツール呼び出しの正常ステップ数に対して十分な余裕 |
| 承認フック | HumanApprovalHook()（generate_expense_form のみ対象） | TOOL-002（generate_expense_form）のBeforeToolCallEventで利用者確認を取得するため（BRL-06、GRD-016） |
| セッション管理 | FileSessionManager(session_id=session_id, storage_path="data/sessions/") | session_idをinvocation_stateから受け取りファクトリ関数内で初期化する |

#### 2.4.2 利用ツール

| ツールID | ツール名 | 利用目的 | 呼び出し条件 |
|----------|----------|----------|--------------|
| TOOL-002 | `generate_expense_form` | 経費精算申請書ExcelドラフトをOutput/に生成する | HumanApprovalHook（BeforeToolCallEvent）承認後の1回のみ（BRL-05、GRD-016） |

> 【注意】TOOL-001（calculate_travel_expense）はAG-003の利用ツールではない。AG-002が担当する
> `generate_travel_expense_form`（AG-002用ツール）は呼び出さない。`generate_expense_form` のみを使用する

### 2.5 連携設計

#### 2.5.1 Agent as Tools設計

**ツール名**: `expense_application_agent_tool`

**ツール説明**:
```
経費精算申請フロー全体を実行する。
領収書画像からの情報抽出・経費区分判断・申請書Excelドラフト生成・申請ルールチェック・最終提示を担当する。
```

**ToolContextの使用**:
- `@tool(context=True)` デコレータを使用
- `tool_context.invocation_state` から `session_id`・`applicant_name`・`application_date` を取得
- `session_id` はファクトリ関数 `_get_expense_agent(session_id)` 内で FileSessionManager の初期化に直接使用する
- `applicant_name`・`application_date` をエージェントへの invocation_state として設定する

#### 2.5.2 invocation_stateの受け渡しとAgentインスタンスのキャッシュ

```python
# session_id をキーにして Agent インスタンスをキャッシュし、
# セッション内で会話履歴を保持する（情報の再入力を防ぐ）
_agent_instances: dict[str, Agent] = {}

@tool(context=True)
def expense_application_agent_tool(query: str, tool_context: ToolContext) -> str:
    # AG-001から受け取る
    state = tool_context.invocation_state
    session_id = state.get("session_id", "")
    applicant_name = state.get("applicant_name", "")
    application_date = state.get("application_date", "")

    # 個別バリデーション
    if not applicant_name or not applicant_name.strip():
        return "エラー: 申請者名が設定されていません。"
    if not application_date or not _is_valid_date(application_date):
        return "エラー: 申請日の形式が不正です。"

    # session_id 未登録の場合のみ新しい Agent インスタンスを生成する
    # Agent初期化時にcallback_handler=Noneを指定して標準出力への直接表示を抑制する（AG-001による二重出力を防ぐ）
    if session_id not in _agent_instances:
        _agent_instances[session_id] = _get_expense_agent(session_id, application_date)

    internal_state = {
        "applicant_name": applicant_name,
        "application_date": application_date,
    }
    response = _agent_instances[session_id](query, invocation_state=internal_state)
    return str(response)


def _get_expense_agent(session_id: str, application_date: str) -> Agent:
    session_manager = FileSessionManager(
        session_id=session_id,
        storage_path="data/sessions/"
    )
    system_prompt = build_expense_system_prompt(application_date)
    return Agent(
        agent_id="expense_agent",
        name="経費精算申請エージェント",
        model=get_model(),
        system_prompt=system_prompt,
        tools=[generate_expense_form],
        conversation_manager=SlidingWindowConversationManager(
            window_size=15,
            should_truncate_results=True,
            per_turn=False
        ),
        hooks=[HumanApprovalHook(), LoopControlHook(max_iterations=10)],
        session_manager=session_manager,
        callback_handler=None,
    )
```

## 3. 実装詳細

### 3.1 クラス/関数設計

#### 3.1.1 expense_application_agent_tool 関数

##### 説明
AG-001から呼び出されるAgent as Toolsのラッパー関数。invocation_stateからセッション情報を取得し、AG-003エージェントに委任する。

##### 引数
- `query` (str): AG-001から渡される委任メッセージ（必須）
- `tool_context` (ToolContext): AG-001のToolContext（context=Trueにより自動渡し）

##### 戻り値
- `str`: 申請フロー完了メッセージまたはエラーメッセージ

##### 処理内容
1. `tool_context.invocation_state` から `session_id`・`applicant_name`・`application_date` を取得する
2. `applicant_name`（min_length=1）と `application_date`（YYYY-MM-DD形式）を個別バリデーションする
3. `session_id` が `_agent_instances` に未登録の場合のみ `_get_expense_agent(session_id, application_date)` を呼び出してエージェントを生成・キャッシュする
4. `_agent_instances[session_id](query, invocation_state={...})` を呼び出して処理を委任する
5. 結果を文字列で返す

#### 3.1.2 build_expense_system_prompt 関数

##### 説明
申請日と申請期限基準日をシステムプロンプトに動的に埋め込んで返す。外部ポリシーファイル（`agent_knowledge/receipt_policies.py`）を読み込み、バリデーションルール定義をプロンプトに組み込む。

##### 引数
- `application_date` (str): 申請日（YYYY-MM-DD形式、必須）

##### 戻り値
- `str`: 動的情報および外部ポリシールールが埋め込まれたシステムプロンプト全文

##### 処理内容
1. `datetime.date.fromisoformat(application_date)` で申請日を `date` 型に変換する
2. `deadline_date` を `application_date - relativedelta(months=3)` で計算する
3. `os.path.exists("agent_knowledge/receipt_policies.py")` で外部ポリシーファイルの存在を事前チェックする
4. ファイルが存在する場合は `open()` でポリシーファイルを読み込み、ポリシー内容を変数に格納する
5. ファイルが存在しない場合はエラーログを出力し、フォールバック用の空ポリシー文字列を使用する
6. EXPENSE_SYSTEM_PROMPT_TEMPLATE の `{application_date}`・`{deadline_date}`・`{receipt_policies}` を置換して返す

### 3.2 エラーハンドリング

`expense_application_agent_tool` 関数内で捕捉する例外と処理を以下に示す。エラー時の戻り値はすべて `str`（呼び出し元LLMコンテキストに渡されるため）。ログにはクエリの先頭50文字を付加情報として含める。

| エラー種別 | 条件 | ログレベル | 対応 |
|-----------|------|-----------|------|
| `LoopLimitError` | エージェントのループ回数がLoopControlHookの上限を超えた場合 | WARNING | WARNINGレベルでログ出力。エラーメッセージ文字列を返す |
| `ContextWindowOverflowException` | コンテキストウィンドウが溢れた場合 | WARNING | WARNINGレベルでログ出力。エラーメッセージ文字列を返す |
| `MaxTokensReachedException` | 最大トークン数に達した場合 | WARNING | WARNINGレベルでログ出力。エラーメッセージ文字列を返す |
| `RuntimeError` | その他のRuntimeError発生時 | ERROR (exc_info=True) | ERRORレベル（exc_info=True）でログ出力。エラーメッセージ文字列を返す |
| `Exception`（その他） | 上記以外の予期しないエラー発生時 | ERROR (exc_info=True) | ERRORレベル（exc_info=True）でログ出力。エラーメッセージ文字列を返す |

## 5. 補足情報

### 5.1 実装上の注意点

1. **callback_handler=Noneの必須指定**
   - AG-003の `Agent` インスタンスは必ず `callback_handler=None` で初期化する
   - これにより専門エージェント自身の標準出力への直接表示を抑制し、AG-001による二重出力を防ぐ
   - `agent()` 呼び出し時に渡すと `TypeError` になるため必ずコンストラクタで指定する

2. **Agentインスタンスのキャッシュ**
   - `_agent_instances: dict[str, Agent] = {}` をモジュールレベルで定義する
   - ツール関数が呼び出されるたびに `Agent(...)` を新規生成してはならない
   - `session_id` をキーとして初回のみ生成・登録し、2回目以降は再利用する

3. **TOOL-002の呼び出し禁止（承認前）**
   - `generate_expense_form` はHumanApprovalHook（BeforeToolCallEvent）による承認後にのみ呼び出す
   - ドラフト提示（テキスト整理）とツール呼び出しステップを明確に分離する

4. **generate_travel_expense_form の呼び出し禁止**
   - AG-003は `generate_travel_expense_form`（AG-002用）を呼び出してはならない

5. **TOOL-001（calculate_travel_expense）の非利用**
   - AG-003は TOOL-001 を利用しない。tools=[generate_expense_form] のみを指定する

6. **外部ポリシーファイルの読み込み**
   - `agent_knowledge/receipt_policies.py` の存在チェックは `os.path.exists()` による事前チェックで実装すること
   - `open()` 実行時の `FileNotFoundError` 捕捉との併用は禁止（事前チェックと例外捕捉を二重に行わない）
   - ファイルが存在しない場合はフォールバック処理としてエラーログを出力し、空ポリシー文字列を使用する

---

### 5.2 パフォーマンス考慮事項

1. **一括収集による会話ターン数削減**
   - 経費情報を1つずつ聞かずにフォームで一括収集することで、会話ターン数を削減する（BRL-23対策）

2. **Agentインスタンスのキャッシュ**
   - 同一セッション内での複数呼び出し間で `SlidingWindowConversationManager` の会話履歴を保持する

---

### 5.3 セキュリティ考慮事項

1. **invocation_stateの安全な受け渡し**
   - `session_id`・`applicant_name`・`application_date` はLLMプロンプトには含めずinvocation_state経由で渡す

2. **申請書の自動提出禁止**
   - BRL-09に従い、申請書の提出操作はユーザーのみが実施する
