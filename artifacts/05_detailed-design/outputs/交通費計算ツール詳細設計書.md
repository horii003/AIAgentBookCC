## 1. 概要

### 1.1 コンポーネントの目的

出発地・目的地・交通手段・移動日を入力として、交通手段ごとのルールに従い運賃を自動計算する。電車区間はDATA-009（train_routes.json）を検索し正確な運賃を返す。バス・タクシー・飛行機はDATA-010（fixed_fares.json）を参照して固定運賃を返す。AG-002（交通費精算申請エージェント）が収集済み申請情報のD-005に運賃を自動補完できるようにする。申請者名・申請日はLLMからパラメータとして渡されず、ツール内部でinvocation_stateから取得する。

### 1.2 主要な責務

- **電車運賃計算**: アプリケーション起動時に静的読み込みしたDATA-009（train_routes.json）をリスト線形探索で検索し、出発地・目的地に一致する区間の運賃を返す
- **固定運賃参照**: アプリケーション起動時に静的読み込みしたDATA-010（fixed_fares.json）を参照し、バス・タクシー・飛行機の固定運賃を返す
- **交通手段別分岐**: 正規化済み交通手段（電車/バス/タクシー/飛行機）に応じて計算方法を切り替える
- **計算根拠の提供**: 利用したデータソース（経路テーブル検索 or 固定運賃参照）を `calculation_method` フィールドでエージェントに伝える
- **入力バリデーション**: `FareCalculationInput` Pydanticモデルで入力の型安全性と正規化を保証する

## 3. calculate_transport_fareツール設計

### 3.1 基本情報

#### ツール名（関数名）
`calculate_transport_fare`

#### 説明（docstring）
```
指定された移動区間の交通費を計算します。

電車の場合はDATA-009（train_routes.json）をリスト線形探索で検索し、出発地・目的地に一致する
区間の運賃を返します。バス・タクシー・飛行機の場合はDATA-010（fixed_fares.json）
の固定運賃を返します。申請者名・申請日はツール内部でinvocation_stateから取得します。

Args:
    departure (str): 出発地（「渋谷駅」「渋谷」等、駅名前後の「駅」「駅前」は自動除去）
    destination (str): 目的地（「新宿駅」「新宿」等、同上）
    transport_type (str): 交通手段。以下の表記を許容し内部統一表記へ正規化します:
        - 電車系: 「電車」「鉄道」「JR」「地下鉄」「電鉄」→「電車」
        - バス系: 「バス」「路線バス」→「バス」
        - タクシー系: 「タクシー」「taxi」→「タクシー」
        - 飛行機系: 「飛行機」「航空機」「airplane」→「飛行機」
    travel_date (str, optional): 移動日（「2026-05-06」等のYYYY-MM-DD形式）

Returns:
    dict: 以下のキーを持つ辞書:
        - success (bool): 計算成否フラグ
        - fare (int): 運賃（円単位）。エラー時は0
        - calculation_method (str): 計算方法の説明（「電車経路テーブル検索」または「固定運賃参照: タクシー」等）
        - message (str): エラーメッセージ（success=Trueの場合は空文字列）
```

---

### 3.2 インターフェース設計

#### 3.2.1 入力

**パラメータ定義**（LLMが渡すパラメータ）:

| 引数名 | 型 | 説明 | デフォルト値 |
|--------|-----|------|--------------|
| `departure` | `str` | 出発地（正規化前でも可。field_validatorで「駅」「駅前」接尾語を自動除去） | なし（必須） |
| `destination` | `str` | 目的地（正規化前でも可。同上） | なし（必須） |
| `transport_type` | `str` | 交通手段（field_validatorで「電車」「バス」「タクシー」「飛行機」に正規化） | なし（必須） |
| `travel_date` | `str` | 移動日（YYYY-MM-DD形式。parse_date_fieldでdate型に変換） | `None`（任意） |

**invocation_stateから取得するフィールド**（LLMはパラメータとして渡さない）:

| フィールド名 | 型 | 説明 | 取得元 |
|------------|-----|------|-------|
| `applicant_name` | `str` | 申請者名 | `tool_context.invocation_state.get("applicant_name", "")` |
| `application_date` | `str` | 申請日（YYYY-MM-DD形式） | `tool_context.invocation_state.get("application_date", "")` |

##### バリデーションルール詳細

| フィールド | ルール | エラーメッセージ | 例 |
|-----------|--------|----------------|-----|
| `departure` | 空文字・None禁止。前後空白自動除去。500文字以内。接尾語「駅前」「駅」を自動除去 | `"入力のエラー: 必須項目が入力されていません。入力してください"` | NG: `""`, OK: `"渋谷駅"` → `"渋谷"` |
| `destination` | 同上 | 同上 | NG: `""`, OK: `"新宿駅前"` → `"新宿"` |
| `transport_type` | 許容するキーワードにマッピングされること。マッピング外はValueError | `"交通手段のエラー: 「{v}」は対応していません。電車・バス・タクシー・飛行機のいずれかを指定してください"` | NG: `"新幹線"`, OK: `"JR"` → `"電車"` |
| `travel_date` | YYYY-MM-DD等の解析可能な日付文字列（python-dateutil経由）。Noneは許容 | `"日付のエラー: 正しい日付形式（YYYY-MM-DD）で入力してください"` | NG: `"abc"`, OK: `"2026-05-06"` |

##### 正規化処理

| 入力値 | 正規化後 | 処理内容 |
|--------|---------|---------|
| `"渋谷駅"` | `"渋谷"` | `normalize_station_name`: 末尾の「駅」を除去 |
| `"渋谷駅前"` | `"渋谷"` | `normalize_station_name`: 末尾の「駅前」を除去（「駅前」が優先） |
| `"新宿"` | `"新宿"` | `normalize_station_name`: 接尾語なし→そのまま返す |
| `"JR"` | `"電車"` | `normalize_transport_type`: JR→電車にマッピング |
| `"鉄道"` | `"電車"` | `normalize_transport_type`: 鉄道→電車にマッピング |
| `"地下鉄"` | `"電車"` | `normalize_transport_type`: 地下鉄→電車にマッピング |
| `"路線バス"` | `"バス"` | `normalize_transport_type`: 路線バス→バスにマッピング |
| `"taxi"` | `"タクシー"` | `normalize_transport_type`: taxi→タクシーにマッピング |
| `"航空機"` | `"飛行機"` | `normalize_transport_type`: 航空機→飛行機にマッピング |

#### 3.2.2 出力

**戻り値の型**: `dict[str, bool | int | str]`

**説明**: 運賃計算結果を辞書形式で返す。ツール関数は例外を再送出せず、必ずこの形式で返す（例外処理方針準拠）

**構造**:
```python
{
    "success": bool,           # 計算成否フラグ
    "fare": int,               # 運賃（円単位）。エラー時は0
    "calculation_method": str, # 計算方法（例: "電車経路テーブル検索", "固定運賃参照: タクシー"）
    "message": str             # エラーメッセージ（success=Trueの場合は空文字列""）
}
```

#### 3.2.3 例外

| 例外 | 発生条件 | 戻り値 |
|------|---------|-------|
| `ValidationError` | `FareCalculationInput` のバリデーション失敗時（ツール関数内でキャッチ） | `{"success": False, "fare": 0, "calculation_method": "", "message": エラーメッセージ}` |
| `ValueError` | 電車区間が経路テーブルに存在しない場合（ツール関数内でキャッチ） | `{"success": False, "fare": 0, "calculation_method": "", "message": エラーメッセージ}` |
| `Exception` | 想定外の例外（ツール関数内でキャッチ） | `{"success": False, "fare": 0, "calculation_method": "", "message": エラーメッセージ}` |

### 3.3 ビジネスロジック

#### 3.3.1 処理フロー

```
開始
  ↓
invocation_stateからapplicant_name・application_dateを取得する
  ↓
FareCalculationInputでバリデーション・正規化
  - departure, destination: normalize_station_name（接尾語除去）
  - transport_type: normalize_transport_type（表記ゆれ正規化）
  - travel_date: parse_date_field（文字列→date変換）
  ↓
バリデーション失敗？
  YES → ValidationError をキャッチ
         呼び出し元でlogging.warning(f"ValidationError: {e}")を記録（ツール関数）
         ErrorHandler.handle_validation_error(e) で日本語メッセージ生成
         {"success": False, "fare": 0, "calculation_method": "", "message": エラーメッセージ} を返す
  ↓
交通手段の分岐
  ↓
transport_type == "電車" ?
  YES → _railway_routes（DATA-009の静的読み込みデータ）をリスト線形探索で検索
         departure と destination が完全一致するレコードを探す
         ↓
         一致レコードが存在する？
           YES → fare = レコードの fare
                  calculation_method = "電車経路テーブル検索"
           NO  → ValueError を発生させる
                  呼び出し元でlogging.warning(f"Route not found: {departure}→{destination}")を記録
                  {"success": False, "fare": 0,
                   "calculation_method": "",
                   "message": "該当する電車区間が見つかりません。出発地・目的地をご確認いただくか、運賃を手動で入力してください。"} を返す
  ↓
transport_type in ["バス", "タクシー", "飛行機"] ?
  YES → _fixed_fares（DATA-010の静的読み込みデータ）を参照
         fixed_fares[transport_type] で運賃を取得する
         ↓
         fare = _fixed_fares[transport_type]
         calculation_method = f"固定運賃参照: {transport_type}"
  ↓
{"success": True, "fare": fare, "calculation_method": calculation_method, "message": ""} を返す
  ↓
例外発生時（想定外）
  → 呼び出し元でlogging.error(f"Unexpected error: {e}", exc_info=True)を記録
     ErrorHandler.handle_unexpected_error(e) で日本語メッセージ生成
     {"success": False, "fare": 0, "calculation_method": "", "message": エラーメッセージ} を返す
  ↓
終了
```

#### 3.3.2 静的読み込み機構

**機構名**: モジュールレベル静的読み込み

**用途**: アプリケーション起動時にDATA-009・DATA-010をメモリに読み込み、ツール実行ごとのファイルI/Oを排除して応答速度を向上させる（バリデーション方針準拠）

**実装**:
- `tools/transport_tools.py` モジュールレベルで `_railway_routes: list[dict]` と `_fixed_fares: dict[str, int]` を定義する
- `load_fare_data()` 補助関数を定義し、モジュールインポート時に1回だけ呼び出す
- 読み込み失敗時は `(False, エラーメッセージ文字列)` を返す

## 4. 補助関数設計

### 4.1 load_fare_data

**関数名**: `load_fare_data`

**目的**: アプリケーション起動時にDATA-009・DATA-010をメモリに静的読み込みする

**シグネチャ**:
```python
def load_fare_data(
    train_routes_path: str = "data/train_routes.json",
    fixed_fares_path: str = "data/fixed_fares.json"
) -> tuple[bool, str | tuple[list[dict], dict[str, int]]]:
```

**戻り値**:
- 成功: `(True, (train_routes_list, fixed_fares_dict))`
- 失敗: `(False, エラーメッセージ文字列)`

**処理フロー**:
```
開始
  ↓
os.path.exists(train_routes_path) で存在チェック
  存在しない → (False, "運賃データファイルが見つかりません: {train_routes_path}") を返す
  ↓
train_routes_path のJSONファイルを読み込む（routes キー配下のネスト形式）
  失敗（Exceptionキャッチ）→ (False, "電車経路データの読み込みに失敗しました: {e}") を返す
  ↓
os.path.exists(fixed_fares_path) で存在チェック
  存在しない → (False, "運賃データファイルが見つかりません: {fixed_fares_path}") を返す
  ↓
fixed_fares_path のJSONファイルを読み込む（{"バス": 230, "タクシー": 10000, "飛行機": 50000} 形式）
  失敗（Exceptionキャッチ）→ (False, "固定運賃データの読み込みに失敗しました: {e}") を返す
  ↓
(True, (train_routes_list, fixed_fares_dict)) を返す
```

**モジュールレベルでの呼び出し**:
```python
# tools/transport_tools.py モジュールレベル
_load_result = load_fare_data()
if not _load_result[0]:
    import logging
    logging.warning(f"load_fare_data failed: {_load_result[1]}")
    _railway_routes: list[dict] = []
    _fixed_fares: dict[str, int] = {}
else:
    _railway_routes, _fixed_fares = _load_result[1]
```

## 5. データ設計

### 5.1 DATA-009（train_routes.json）

**データソース**: `data/train_routes.json`（アプリケーション起動時に静的読み込み）

**構造**: `routes` キーあり・ネスト形式
```json
{
  "routes": [
    {
      "route_id": "R001",
      "departure": "渋谷",
      "destination": "新宿",
      "fare": 200
    },
    {
      "route_id": "R002",
      "departure": "東京",
      "destination": "品川",
      "fare": 170
    }
  ]
}
```

**検索方式**: リスト線形探索（departure・destination の完全一致）

### 5.2 DATA-010（fixed_fares.json）

**データソース**: `data/fixed_fares.json`（アプリケーション起動時に静的読み込み）

**構造**: フラットな key-value 形式
```json
{
  "バス": 230,
  "タクシー": 10000,
  "飛行機": 50000
}
```

### 5.3 データモデル

#### RailwayRouteMasterモデル（内部表現）

| フィールド名 | 型 | 必須/任意 | 説明 | 制約 |
|--------------|-----|-----------|------|------|
| `route_id` | `str` | 必須 | 区間識別キー | 空文字禁止 |
| `departure` | `str` | 必須 | 出発地（正規化済み駅名） | 空文字禁止 |
| `destination` | `str` | 必須 | 目的地（正規化済み駅名） | 空文字禁止 |
| `fare` | `int` | 必須 | 運賃（円単位） | 1以上の正整数 |

**注意**: `FixedFareMaster` モデルは不要。fixed_fares.jsonはフラットなdict形式で直接参照する。

#### FareCalculationInputモデル

| フィールド名 | 型 | 必須/任意 | 説明 | 制約 |
|--------------|-----|-----------|------|------|
| `departure` | `str` | 必須 | 出発地（field_validatorで接尾語除去） | 空文字禁止・500文字以内 |
| `destination` | `str` | 必須 | 目的地（field_validatorで接尾語除去） | 空文字禁止・500文字以内 |
| `transport_type` | `Literal["電車", "バス", "タクシー", "飛行機"]` | 必須 | 交通手段（field_validatorで正規化） | 4種のみ |
| `travel_date` | `Optional[date]` | 任意 | 移動日（parse_date_fieldで変換） | YYYY-MM-DD等の解析可能形式 |

**バリデーション**:
- `normalize_station_name`: departure・destinationの接尾語（「駅」「駅前」）除去（BRL-15）
- `normalize_transport_type`: transport_typeの表記ゆれ正規化（BRL-14）
- `parse_date_field`: travel_dateの文字列→date変換（mode="before"）

## 6. 補足情報

### 6.1 実装上の注意点

1. **ファイル存在チェックは os.path.exists() による事前チェックで実装する**
   - `open()` 実行時の `FileNotFoundError` 捕捉との併用は禁止し、事前チェックを優先する

2. **ツール関数は例外を再送出しない**
   - `ValidationError`・`ValueError`・その他の例外はすべてキャッチし、`{"success": False, "message": ...}` を返す（例外処理方針準拠）
   - データ読み込み系メソッドは `(False, エラーメッセージ文字列)` を返す

3. **電車区間の検索はリスト線形探索・完全一致**
   - `departure` と `destination` の完全一致でレコードをリスト線形探索する
   - `normalize_station_name` による接尾語除去後の正規化済み文字列で比較する
   - 一致するレコードが存在しない場合は `ValueError` を発生させて戻り値で処理する（例外をraiseして上位に伝播させない）

4. **冪等性の保証**
   - 同一パラメータで再実行しても同一結果を返す（実行制御方針準拠）
   - 再試行は実施しない（実行制御方針 7.1準拠）

5. **申請者名・申請日はLLMパラメータとして渡さない**
   - `applicant_name` と `application_date` は invocation_state から取得する
   - LLMがツール呼び出し時にこれらを引数として渡す必要はない

6. **ログ出力はツール関数が担当する**
   - ErrorHandlerはログ出力しないため、ツール関数内でErrorHandler呼び出し前にログを記録する

### 6.2 パフォーマンス考慮事項

1. **静的読み込みによるI/O排除**
   - モジュールレベルで1回だけファイルを読み込み、ツール実行ごとのディスクI/Oを排除する

2. **リスト線形探索の許容**
   - DATA-009はJSONファイルでデータ件数が少ないため、リスト線形探索で十分な応答速度を確保できる

### 6.3 セキュリティ考慮事項

1. **JSONファイルパスのハードコード**
   - ファイルパスは実装コード内にハードコードし、LLMからパスを受け取る設計にしない（ディレクトリトラバーサル防止）

2. **ValidationErrorの内容マスキング**
   - `ErrorHandler.handle_validation_error()` でユーザー向けメッセージを生成し、Pydanticの内部スタックトレースをユーザーに見せない

## 8. テスト観点

### 8.1 機能テスト
- 電車区間（例: 渋谷→新宿）を入力したとき、DATA-009に存在する区間の運賃が正しく返ること
- バスを指定したとき、DATA-010の固定運賃（230円）が返ること
- タクシーを指定したとき、DATA-010の固定運賃（10,000円）が返ること
- 飛行機を指定したとき、DATA-010の固定運賃（50,000円）が返ること
- `calculation_method` に「電車経路テーブル検索」または「固定運賃参照: {交通手段}」が正しく返ること
- invocation_stateから申請者名・申請日が正しく取得されること

### 8.2 異常系テスト
- DATA-009に存在しない電車区間を入力したとき、`success: False` とともに手動入力を促すメッセージが返ること（ValueError）
  - **期待戻り値**: `{"success": False, "fare": 0, "calculation_method": "", "message": "..."}`
- transport_typeに「新幹線」を指定したとき、ValidationErrorに基づく日本語エラーメッセージが返ること（EX-01）
  - **期待戻り値**: `{"success": False, "fare": 0, "calculation_method": "", "message": "..."}`
- departure/destinationに空文字を指定したとき、ValidationErrorに基づく日本語エラーメッセージが返ること（EX-01）
  - **期待戻り値**: `{"success": False, "fare": 0, "calculation_method": "", "message": "..."}`
- DATA-009が存在しない状態でload_fare_dataを呼んだとき、`(False, エラーメッセージ文字列)` が返ること

### 8.3 性能テスト
- 正常ケースで100ms以内に応答を返すこと（静的読み込みによるI/O排除の効果を確認する）
- 100件の連続呼び出しで応答時間が安定していること

### 8.4 境界値テスト
- departure/destinationが500文字ちょうどの場合、バリデーションを通過すること
- departure/destinationが501文字の場合、ValidationErrorが発生すること
- DATA-009の駅名に「駅」「駅前」の接尾語付きで入力した場合、正規化されて正しく検索されること（例: 「渋谷駅」→「渋谷」）
