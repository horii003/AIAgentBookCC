## 1. 目的・設計方針

### 1.1 目的

- AIエージェントが **セッション単位で保持・参照すべき情報の範囲と方針**を定義する
- 会話継続、再開、引き継ぎ、監査に耐える **一貫した状態管理** を実現する

### 1.2 本方針で定義すること

- セッションの単位・ライフサイクル
- 管理対象情報（会話履歴、判断結果、成果物参照 等）
- メモリの種類と役割分担
- 再開・中断・終了時の扱い

### 1.3 本方針で定義しないこと

- メモリの具体的な保存技術（SessionManager を使用）
- エージェント内部の推論用メモリ構造
- 実装クラス／SDK依存設計

---

## 2. セッションの定義

### 2.1 セッションとは

**本システムにおける「セッション」の定義**:
- 社員が申請を開始してから申請書（下書き）提示または中断・エラー終了までの1申請フロー全体を1セッションとする
- セッションはアプリケーション起動・対話ループ開始を起点とし、申請書提示（CF-005 完了）またはエラー・エスカレーション案内（CF-006 完了）を終点とする（申請者名はアプリケーション起動時に取得済み）
- 1セッションで対応できる申請種別は1種類のみ（交通費精算申請または経費精算申請）

### 2.2 セッション識別子

| 項目 | 内容 | 形式 | 例 |
|-----|------|------|---|
| session_id | セッション識別子 | `{YYYYMMDD}_{HHMMSS}_{uuid4[:8]}` | `20260502_143022_a3f9b1c2` |

## 3. セッションライフサイクル

### 3.1 状態遷移

| 状態 | 説明 | トリガー |
|-----|------|---------|
| CREATED | セッション生成 | 対話ループ開始時に SessionManager が session_id を発行する（申請者名は起動時取得済み） |
| ACTIVE | 処理中 | エージェントが会話ターンを処理している |
| WAITING | 人間対応待ち | エージェントが社員の入力（承認・修正・選択）を待っている |
| CLOSED | 正常終了 | 申請書（下書き）が社員に提示された（CF-005 完了） |
| TERMINATED | 異常終了 | エラー・エスカレーション案内（CF-006）完了またはセッションタイムアウト |

### 3.2 状態遷移条件

**状態遷移のトリガー**:
- 対話ループ開始: CREATED → ACTIVE
- エージェントが社員へ入力を求める: ACTIVE → WAITING
- 社員が入力を返す: WAITING → ACTIVE
- 申請書（下書き）提示完了（CF-005 完了）: ACTIVE → CLOSED
- 社員がキャンセルを選択（CF-005 Step 2）: ACTIVE → CLOSED
- 技術エラー・CF-006 完了: ACTIVE → TERMINATED
- 対話ターン上限（30回）到達: ACTIVE → TERMINATED

**状態管理責務**:
- SM-001（SessionManager）がセッション状態を管理し、data/sessions/{session_id}.json として永続化する（セッション終了時に削除）

## 4. セッションで管理する情報の分類

### 4.1 管理情報カテゴリ

| カテゴリ | 説明 | 保存場所 |
|---------|------|---------|
| 会話履歴 | エージェントと社員の会話ターン（メッセージ）履歴 | SlidingWindowConversationManager（AG-001: 30、AG-002: 20、AG-003: 15）インメモリ。セッション状態ファイルには保存しない |
| 業務コンテキスト | 申請者名・申請日・申請種別・収集済み申請情報（D-010）・申請進捗 | data/sessions/{session_id}.json（SM-001が永続化、セッション終了時に削除） |
| 実行履歴 | ツール実行（TOOL-001/TOOL-002）・ルーティング結果・HITL承認結果 | ログストア（DATA-011: LOG-STD / DATA-012: LOG-HI） |
| 成果物参照 | 生成した申請書（下書き）のファイルパス（D-011） | data/sessions/{session_id}.json（参照IDのみ） |
| 状態管理 | セッション状態（CREATED/ACTIVE/WAITING/CLOSED/TERMINATED）・対話ターン数 | data/sessions/{session_id}.json（SM-001が永続化、セッション終了時に削除） |

## 5. 会話履歴管理方針

### 5.1 会話履歴の範囲

**保存対象**:
- エージェントの発話（システムレスポンス）
- 社員の入力（ユーザーメッセージ）
- ツール呼び出しと結果（ツールメッセージ）

**保存しないもの**:
- LLM内部推論（Chain-of-Thought）
- 領収書画像のバイナリデータ（セッション終了後は破棄。DATA-009）


### 5.2 会話履歴の粒度

| 粒度 | 方針 |
|-----|------|
| フル履歴 | SlidingWindowConversationManager がウィンドウサイズ（AG-001: 30、AG-002: 20、AG-003: 15）の範囲でインメモリ保持する。ウィンドウ超過時は古いメッセージを切り落とす（should_truncate_results=True） |
| 要約 | 適用しない（Strands Agents の SlidingWindowConversationManager に委ねる） |
| 重要発言抽出 | 業務コンテキスト（収集済み申請情報 D-010）として別途 data/sessions/{session_id}.json に保存する |

## 6. 業務コンテキスト管理方針

### 6.1 管理対象

- 申請者名（applicant_name）：アプリケーション起動時に取得し、エージェント初期化パラメータとして渡す
- 申請日（application_date）：YYYY-MM-DD 形式。システム日付（実行時の日付）を自動取得する
- 申請種別（application_type：交通費精算申請 / 経費精算申請）
- セッション状態（session_status：CREATED/ACTIVE/WAITING/CLOSED/TERMINATED）
- 対話ターン数（turn_count：最大30回）
- 収集済み申請情報（collected_info：移動情報 D-014 または経費情報 D-015）
- 申請書ファイルパス（output_file_path：生成済みの場合のみ）


### 6.2 更新ルール

**誰が更新できるか**:
- applicant_name：アプリケーション起動時に設定済み。AG-001 を含む全エージェントは変更しない
- application_date：アプリケーション起動時にシステム日付から自動取得。全エージェントは変更しない
- application_type：AG-001 が確定後に一度だけ設定し、AG-002/AG-003 は変更しない
- collected_info・session_status・turn_count：各専門エージェント（AG-002/AG-003）が自律的に更新する
- output_file_path：AG-002/AG-003 が TOOL-002 実行後に設定する

**上書き／追記のルール**:
- applicant_name・application_date：上書き禁止（起動時取得後は固定）
- application_type：上書き禁止（AG-001 確定後は固定）
- collected_info：各情報収集ステップで上書き更新する（差分追記）
- session_status：状態遷移条件に従い上書きする

**差分管理の考え方**:
- セッション状態ファイルはターン単位で最新状態を上書き保存する（バージョン履歴は保持しない）

## 7. 実行・判断履歴管理方針

### 7.1 実行履歴

| 項目 | 内容 | 保存場所 |
|-----|------|---------|
| tool_name | ツール名（TOOL-001 / TOOL-002） | ログストア（DATA-012: LOG-HI / OPE-002） |
| 実行内容 | ツールパラメータ（交通手段・経路・テンプレートID等） | ログストア（DATA-012: LOG-HI / OPE-002） |
| 実行結果 | ツール戻り値（計算運賃・生成ファイルパス等） | ログストア（DATA-012: LOG-HI / OPE-002） |
| 実行時刻 | タイムスタンプ（ISO8601） | ログストア（DATA-011: LOG-STD / OPE-001） |
| 再実行可否 | TOOL-001：再試行可（参照のみ）。TOOL-002：1回のみリトライ可（非冪等） | — |

### 7.2 判断履歴

**ルーティング結果**:
- 申請種別確定時に AUD-002（申請種別判断ログ）として LOG-STD（DATA-011）に記録する
- 記録内容：request_id、適用ルールID（JUDGE-01等）、判断結果（申請種別）、判断方法（自動/選択肢提示）

**エスカレーション判定**:
- CF-006 発火時に ERR-004〜ERR-008 として LOG-STD または LOG-HI（DATA-011/DATA-012）に記録する

**人間承認の結果**:
- HITL 確認承認（CF-005 Step 2）の結果を AUD-004（HITL 承認ログ）として LOG-HI（DATA-012）に記録する
- 記録内容：request_id、確認結果（OK/修正/キャンセル）、収集情報サマリ（D-010の要約・マスキング後）

## 8. 成果物・ナレッジ参照管理

### 8.1 管理対象

- 生成ドキュメントID: 申請書（下書き）のファイルパス（output/ 配下の .xlsx ファイルパス）
- 参照ナレッジID: 社内申請ルール（DATA-001）・申請先情報（DATA-004）・申請書テンプレート（DATA-002/DATA-003）の参照記録
- 外部I/F参照ID: 運賃テーブル（DATA-006/DATA-007）の参照記録

### 8.2 管理方針

**実体は保持しない**:
- 申請書テンプレート・運賃テーブル・ナレッジファイルの実体はファイルストレージに保持し、セッション状態には含めない

**参照IDのみを保持**:
- 申請書（下書き）ファイルパスを data/sessions/{session_id}.json の output_file_path フィールドに保持する

**再取得可能性を前提**:
- ナレッジ・テンプレート・運賃テーブルはファイルストレージから常時参照可能であることを前提とし、セッション状態にキャッシュしない

## 9. メモリの種類と役割分担（概念）

### 9.1 メモリ分類

| 種類 | 用途 | 実装 |
|-----|------|------|
| セッションメモリ | 会話・状態の継続（業務コンテキスト・申請進捗） | SessionManager（SM-001）が data/sessions/{session_id}.json として永続化 |
| ワーキングメモリ | 一時処理用（会話ターンのコンテキストウィンドウ） | SlidingWindowConversationManager（各エージェントが独立管理） |
| ロングターム参照 | ナレッジ・マスタファイル（申請ルール・テンプレート・運賃テーブル） | ファイルストレージ（knowledge/ / templates/ / data/ 配下）から都度参照 |

## 10. 再開・引き継ぎ方針

### 10.1 再開条件

- data/sessions/{session_id}.json が存在すること: 再開可（収集済み情報から継続）
- セッション状態が CLOSED または TERMINATED であること: 再開不可（新規セッションを開始する）
- 申請書（下書き）生成完了（CLOSED）後: 再開不可（フロー完了のため新規セッションを開始する）

### 10.2 再開時に復元する情報

| 情報 | 復元要否 | 実装状況 |
|-----|---------|---------|
| applicant_name（申請者名） | 要 | data/sessions/{session_id}.json から読み込み |
| application_date（申請日） | 要 | data/sessions/{session_id}.json から読み込み |
| application_type（申請種別） | 要 | data/sessions/{session_id}.json から読み込み |
| collected_info（収集済み申請情報） | 要 | data/sessions/{session_id}.json から読み込み |
| session_status（セッション状態） | 要 | data/sessions/{session_id}.json から読み込み |
| turn_count（対話ターン数） | 要 | data/sessions/{session_id}.json から読み込み |
| 会話履歴（SlidingWindowConversationManager） | 不要 | 再開時は収集済み情報を踏まえて会話を継続する |

## 11. エスカレーション・承認との連携

**エスカレーション発生時に保持すべき情報**:
- エラー発生フロー（CF-xxx）・エラー種別（ERR-xxx）
- エラー発生時点の収集済み申請情報（D-010）

**人間承認時に提示する情報**:
- 収集済み申請情報の一覧（D-010 の全フィールド）
- 申請金額（高額判定が発生した場合）
- 申請期限チェック結果（期限内/超過）

**承認結果の反映方法**:
- 承認（OK）: TOOL-002（申請書生成）を実行して申請書（下書き）を生成する。AUD-004 を LOG-HI に記録する
- 修正要望: 修正対象フィールドの対話ステップ（CF-003/CF-004 の該当 Step）に戻る
- 拒否（キャンセル）: セッション状態を CLOSED に遷移してフローを終了する。AUD-004 を LOG-HI に記録する

## 12. セキュリティ・ガードレール観点

**個人情報の保持方針**:
- 申請者名（DATA-013：D-3 個人情報）はセッション状態ファイル（data/sessions/{session_id}.json）に保持するが、ログ出力時はマスキングする（GRD-011）
- 領収書画像（DATA-009：D-2 機微）はセッション内のみ参照し、セッション終了後は破棄する

**セッション内秘匿情報の扱い**:
- session_id は invocation_state 経由でエージェント間に伝播し、LLM のプロンプトには含めない（DEC-002 / マルチエージェント連携設計参照）
- 申請内容テキスト・収集済み申請情報はセッション状態ファイルに保存するが、ログ出力時は部分マスキングを適用する（AUD-001/AUD-004/AUD-002）

**マスキング／削除方針**:
- 申請内容テキスト（D-001）：ログ出力時に固有名詞・金額を除く要約のみを記録する（マスキング方針：LOG-STD）
- 収集済み申請情報（D-010）：ログ出力時に項目名のみを記録し、値は除外または要約する
- 領収書情報（D-015）：ログ出力時に抽出項目名のみを記録し、全値をマスキングする（OPE-003/TRC-004）

## 13. ログ・監査との関係

**セッション情報とログの関係**:
- 全ログに request_id（1申請フロー全体の相関ID）を含め、セッション単位でログを追跡可能とする
- session_id と request_id の対応はセッション状態ファイルで管理する

**監査時に参照可能な情報**:
- AUD-001（申請書生成ログ）：申請種別・生成ファイルパス・申請金額・収集情報サマリ
- AUD-002（申請種別判断ログ）：入力申請内容要約・判断結果・適用ルールID
- AUD-004（HITL 承認ログ）：確認結果・収集情報サマリ

**保持期間の考え方**:
- セッション状態ファイル（data/sessions/{session_id}.json）：セッション終了時（CLOSED/TERMINATED）に削除
- ログ（DATA-011 LOG-STD / DATA-012 LOG-HI）：要件上未定義

## 14. 設計上の留意点・リスク

**セッション肥大化のリスク**:
- セッション状態ファイルへの収集済み情報の蓄積によりファイルサイズが肥大化する可能性がある
- 対策: 会話履歴はセッション状態ファイルに保存せず SlidingWindowConversationManager に委ねる。ファイルには業務コンテキストのみ保存する

**メモリ誤参照のリスク**:
- 複数エージェント（AG-001/AG-002/AG-003）が同一 session_id のセッション状態ファイルを参照するため、読み取りタイミングによって状態が不整合になる可能性がある
- 対策: AG-002/AG-003 は自身のセッション状態のみを更新する（AG-001 が設定した applicant_name / application_type は変更しない）

**再開時の誤動作リスク**:
- セッション状態が途中の場合（ACTIVE/WAITING）に再開すると、収集済み情報から継続するが、会話履歴が失われるため社員が混乱する可能性がある
- 対策: 再開時は収集済み情報の一覧を社員に提示して確認を取ってから継続する
