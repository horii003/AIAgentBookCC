## 2. 目的・役割定義

### 2.1 エージェントの目的
- 交通費精算申請に必要な情報を対話形式で一区間ずつ収集する（BRL-06）
- TOOL-001を用いて運賃を自動計算し（BRL-14）、駅名を正規化する（BRL-15）
- TOOL-002を用いてExcel申請書を自動生成し、申請書のチェック（必須項目・整合性・ルール適合性・上長承認要否）を実施する

### 2.2 主な責務
- **必須情報収集**: BRL-06の必須項目（移動日・出発地・目的地・交通手段・費用・業務目的）を一区間ずつ対話で収集する
- **運賃自動計算**: TOOL-001（calculate_transport_fare）で区間ごとの運賃を計算しD-005に補完する（BRL-14）
- **申請期限チェック**: 経費発生日から90日以内であることを確認する（BRL-12）
- **申請書自動作成**: TOOL-002a（generate_transport_application）でExcel申請書を生成する（BRL-02）
- **申請書チェック**: 必須項目（JD-06/JD-07）・整合性（JD-08/JD-09）・ルール適合性（BRL-03）・上長承認要否（BRL-11）を順次チェックしてフィードバックする

### 2.3 非責務（やらないこと）
- **申請種別判断**: AG-001（オーケストレーター）の責務
- **領収書画像認識**: AG-003の責務
- **申請書提出**: 社員（人）の責務（ACT-EXEC-02禁止。GRD-011）

## 3. 自律度・判断権限

### 3.1 自律度レベル

| レベル | 内容 | 適用範囲 |
|------|------|----------|
| 参照 | 社内申請ルールナレッジ（D-002）・経路JSONファイル参照 | ACT-READ-01, ACT-READ-02 |

### 3.2 判断権限の範囲
- **自律判断可能**: 必須情報充足判断・運賃自動計算・申請期限チェック・申請書チェック（必須項目・整合性・ルール適合性・上長承認要否）
- **人間確認必要**: 申請書内容確認（HumanApprovalHook。COND-001/COND-003）。最終確認・提出（社員のみ）
- **実行不可**: 申請書提出（ACT-EXEC-02禁止。GRD-011）

## 4. 処理フロー設計

### 4.1 基本フロー（概要）

```text
開始（AG-001から委譲。invocation_stateで申請者名・申請日・session_id取得）
→ [CF-002: 交通費精算情報収集]
  → [1区間の移動情報収集（Step 1〜2）]
  → [運賃自動計算・駅名正規化（TOOL-001）（Step 3）]
  → [全必須情報充足確認（JD-04）→ 未充足: 次区間へループ]
→ [申請期限チェック（JD-11）（Step 4）→ 超過: 通知・停止]
→ [CF-004: 申請書ドラフト提示（Step 1）]
  → 収集済み申請情報をテキストとして整理・提示する（ツール呼び出しなし）
→ [HumanApprovalHook: 申請者承認待ち（BeforeToolCallEventでトリガー）（Step 2）]
  → [OK選択 → TOOL-002a（generate_transport_application）実行（Step 3）]
  → [修正選択 → CF-002（情報収集）へ戻る]
  → [キャンセル選択 → セッションTERMINATED・申請停止]
→ [CF-005: 申請書チェック（Step 1〜4）]
  → [必須項目チェック（JD-06/JD-07）]
  → [整合性チェック（JD-08/JD-09）]
  → [ルール適合性チェック（BRL-03）]
  → [上長承認要否チェック（BRL-11）]
→ [合格済み申請書提示（Step 5）]
→ 終了（AG-001へ完了を返す）
```

**フローの詳細**: 詳細設計書 agents/transport_agent.py を参照

### 4.2 フロー制御の考え方
- **原則的な処理順序**: 情報収集 → 期限チェック → ドラフト提示 → 承認（HumanApprovalHook） → 申請書生成（TOOL-002a） → チェックの逐次処理
- **処理をスキップする考え方**: 全必須情報が充足した場合（JD-04）は収集ループをスキップして申請期限チェックへ進む
- **同一処理を複数回実行**: 未充足項目がある間は1区間の移動情報収集ループを繰り返す。チェック不備時は修正ループを繰り返す
- **ドラフト提示とツール呼び出しの分離**: 収集済み申請情報はまずテキストとして提示する（ツール呼び出しなし）。HumanApprovalHookのBeforeToolCallEventが発火し、申請者がOKを選択した後にはじめてTOOL-002aを実行する。承認前にTOOL-002aを実行してはならない
- **Human-in-the-Loop承認の発生タイミング**: HumanApprovalHookによる承認は1回のみ発生する（CF-004ステップ内、TOOL-002a呼び出し直前のBeforeToolCallEventで1回のみ）

### 4.3 状態管理
- **セッション状態（D-006）**: SessionManager（session_id=invocation_stateから取得）でdata/sessions/に保存する。flow_step・収集済み申請情報・申請書ファイルパスを管理する
- **invocation_state**: AG-001から`{"applicant_name": str, "application_date": str, "session_id": str}`を受け取る
- **会話履歴**: SlidingWindowConversationManager（window_size=20）で管理する

## 6. ツール設計

### 6.1 利用ツール

AG-002が`tools`リストに登録するツールは業務機能（計算・生成）に特化したツールのみとする。

- **TOOL-001（TL-001: calculate_transport_fare）**: CF-002 Step 3で1区間の移動情報（出発地・目的地・交通手段・移動日）を入力として運賃を計算する。電車はDATA-009（railway_routes.json）を検索、バス/タクシー/飛行機はDATA-010（fixed_fares.json）を参照する
- **TOOL-002a（TL-002a: generate_transport_application）**: CF-004 HumanApprovalHookによる申請者承認（OK）後に、D-005（収集済み申請情報）からExcel申請書（交通費精算申請書）を生成する

TOOL-001は運賃計算（副作用なし）のため承認フック不要。TOOL-002aはファイル生成（副作用あり）のため、HumanApprovalHookのBeforeToolCallEventで申請者承認を取得した後にのみ実行する（COND-001/COND-003）。承認前にTOOL-002aを実行してはならない。

**ツールの詳細**: 交通費計算ツール基本設計書（BD-01a）・申請書生成ツール基本設計書（BD-01b）を参照

## 7. プロンプト設計

### 7.1 システムプロンプトの目的
- AG-002の役割（交通費精算申請の情報収集・運賃計算・申請書作成・チェック）を定義する
- 社内申請ルールナレッジ（D-002）を埋め込み、必須項目・チェックルールを提供する
- 申請期限チェック（BRL-12）で実行時の日付を計算するために動的生成が必要

### 7.2 プロンプト構成要素
- **役割定義**: 交通費精算申請エージェントとしての役割（情報収集・運賃計算・ドラフト提示・承認取得・申請書作成・チェック）
- **処理フロー**: CF-002〜CF-005の処理手順。ドラフト提示（テキスト提示のみ）→ HumanApprovalHook承認待ち → TOOL-002a実行の順序を明示する
- **判断プロトコル**: 必須情報充足判断（JD-04/JD-05）・申請期限チェック（JD-11）・各チェック基準（JD-06〜JD-09/JD-12）
- **社内申請ルールナレッジ埋め込み**: D-002をシステムプロンプトに直接埋め込む
- **動的情報**: 実行時の申請日（application_date）を埋め込む。申請期限判定のためにLLMに今日の日付を渡す
- **エラー発生時の振る舞い**: ツールエラー・システム障害等のシステム系エラーが発生した場合、エラー内容を要約して呼び出し元エージェント（AG-001）に返す

**プロンプト全文**: 詳細設計書 agents/transport_agent.py 2.3.1章を参照

### 7.3 システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 動的生成関数 |
| 動的生成の理由 | CF-002 Step 4（申請期限チェック・BRL-12）でapplication_dateを基準とした90日以内判定が必要。invocation_stateから受け取るapplication_dateを実行時にプロンプトに埋め込む |
| 動的に埋め込む情報 | application_date（YYYY-MM-DD形式）・申請期限日（application_dateの90日前の日付） |

### 7.4 主要な設計意図
- **動的日付埋め込み**: 申請期限チェック（BRL-12）をLLMが正確に判断できるよう、システムプロンプトに実行時の申請日を埋め込む
- **ナレッジ埋め込み方式**: D-002をRAGではなくシステムプロンプトに直接埋め込む
- **1区間ずつ収集**: 移動情報を一区間ずつ収集するループ構造をプロンプトで指示する

## 13. Agent as Tools設計

### 13.1 ツールとしての公開
AG-002はAG-001からAgent as Toolsとして呼び出される。

- **ツール名**: transport_agent_tool
- **デコレータ**: @tool(context=True)
- **入力**: query (str)（委譲メッセージ：申請種別・申請意図テキスト）, tool_context (ToolContext)
- **出力**: str（AG-002からの最終応答：合格済み申請書の提示）

### 13.2 ToolContextの使用
- invocation_stateから`applicant_name`・`application_date`・`session_id`を取得する
- `session_id`を使用してSessionManagerを初期化する
- `applicant_name`・`application_date`をシステムプロンプトの動的生成に使用する
- 専門エージェント内部ではinvocation_stateに`applicant_name`・`application_date`（`session_id`除く）を設定する（マルチエージェント連携設計 7.3準拠）

**詳細**: 詳細設計書 agents/transport_agent.py 13章を参照

## 14. テスト観点

### 14.1 テスト観点の概要
- **機能テスト**: 電車区間の運賃が正しく計算される。ドラフト提示時にTOOL-002aが呼び出されないこと。HumanApprovalHookがBeforeToolCallEventで発火し、OK選択後にTOOL-002aが実行される。申請期限超過時に申請が停止される
- **Human-in-the-Loop承認テスト**: OK選択→TOOL-002a実行・ファイル生成。修正選択→情報収集ステップへ戻ること。キャンセル選択→セッションTERMINATED
- **承認前ツール実行禁止テスト**: HumanApprovalHookのOK前にTOOL-002aが実行されないこと（BeforeToolCallEventで制御されること）
- **異常系テスト**: 存在しない電車区間→手動入力を促す。申請期限91日超過→申請停止。空入力→再入力を促す。ツールエラー発生時→AG-001へ要約を返す
- **性能テスト**: 要件上未定義

## 16. 制約事項・前提条件

### 16.1 技術的制約
- LLMバックエンドはAmazon Bedrock（Claude Sonnet 4.5）のみ
- Strands AgentsのAgent as Toolsパターンで実装する（AG-001からツール関数として呼び出される）
- ストリーミング無効（`callback_handler=None`）

### 16.2 業務的制約
