## 1. 概要

### 1.1 コンポーネントの目的

経費精算申請に特化したスペシャリストエージェントとして、AG-001からの委任を受けて申請フロー全体を実行する。`strands_tools` の `image_reader` ツールで領収書画像からの情報自動抽出（BRL-16）と経費区分の自動判断（BRL-17）を行い、収集済み申請情報をテキストで整理・提示する。HumanApprovalHook（BeforeToolCallEvent）による承認後にTOOL-002（generate_expense_form）でExcelドラフトを生成する。申請期限チェック（BRL-18）・上長承認要否判定（BRL-19）・差し戻しリスク評価（BRL-08）を実施してチェック結果をユーザーへ提示する。

### 1.2 主要な責務

- **invocation_stateの受け取り**: ToolContext経由で applicant_name・application_date・session_id を受け取る
- **領収書画像からの情報自動抽出**: `strands_tools` の `image_reader` ツールで領収書画像から店舗名・金額・日付・品目を自動抽出する（BRL-16）
- **経費区分の自動判断**: LLMが品目から経費区分（事務用品費・宿泊費・資格精算費・その他経費）を自動判断して提示する（BRL-17）
- **手動入力の対応**: 領収書画像の提出なし・抽出失敗時の手動入力（CF-004 Step 4）に対応する
- **業務目的の確認**: CF-004 Step 6-7で業務目的を収集する（BRL-20）
- **申請バリデーションルールの参照**: `agent_knowledge/receipt_policies.py` に定義されたバリデーションルール・申請ポリシーをシステムプロンプトに組み込む
- **Human-in-the-Loop承認取得**: 収集済み申請情報をテキストで整理・提示してOK/修正/キャンセルの3択を取得する（BRL-06）。ツール呼び出しは行わない（ドラフト提示ステップ）
- **申請書生成**: HumanApprovalHook（BeforeToolCallEvent）承認後にTOOL-002（generate_expense_form）でExcelドラフトを生成する（BRL-05）
- **申請ルールチェック**: 申請期限（BRL-18）・上長承認要否（BRL-19）・差し戻しリスク（BRL-08）をチェックしてユーザーへ提示する（CF-006）

### 2.2 インターフェース設計

#### 2.2.1 入力

**ToolContext経由で受け取る情報**（invocation_state）:
```python
{
    "session_id": str,       # セッション識別子（{YYYYMMDDHHmmss}_{UUID 8文字}形式）。ファクトリクラス内でFileSessionManager初期化に使用
    "applicant_name": str,   # 申請者名（min_length=1）
    "application_date": str  # 申請日（YYYY-MM-DD形式。システム日付）
}
```

**対話ループでの入力**:
- 領収書画像: ユーザーが画像を提出する（CF-004 Step 2）。`strands_tools` の `image_reader` で処理する
- 経費情報（手動入力）: 店舗名・金額・日付・品目をユーザーが自由文で入力する（CF-004 Step 4）
- 業務目的: ユーザーが自由文で入力する（BRL-20）
- Human-in-the-Loop選択: OK / 修正 / キャンセルから選択する
- 特殊コマンド:
  - 経費区分判断不能時の4択選択（事務用品費/宿泊費/資格精算費/その他経費）
  - 申請期限超過時のCF-008遷移確認
  - 上長承認要時のCF-009遷移確認

**パラメータ**:
- `query` (str): AG-001から渡される申請開始クエリ

#### 2.2.2 出力

**出力先**: CLI（標準出力）。`callback_handler=None` で標準出力への直接表示を抑制し、AG-001経由で表示する

**出力メッセージ**:
- 領収書提出促し: `"経費精算申請を開始します。領収書の画像を提出してください（提出しない場合は「スキップ」と入力してください）："`
- 領収書抽出結果提示: `"領収書から以下の情報を読み取りました：\n店舗名：[店舗名]\n金額：[金額]円\n日付：[日付]\n品目：[品目]\nこの内容でよろしいですか？"`
- 経費区分判断結果提示: `"品目「[品目]」の経費区分は「[経費区分]」と判断しました。よろしいですか？"`
- 経費区分4択提示（判断不能時）: `"経費区分を選択してください：\n1. 事務用品費\n2. 宿泊費\n3. 資格精算費\n4. その他経費"`
- ドラフト提示（テキスト）: `"収集した申請情報を確認してください。\n【申請者名】[名前]\n【申請日】[日付]\n【経費明細】...\n【業務目的】[目的]\n【合計金額】[金額]円\nOK・修正・キャンセルのいずれかを入力してください。"`
- 申請期限超過通知（CF-008）: `"申し訳ありません。経費発生日から3ヶ月を超えているため、この申請は受け付けられません。総務部門にご相談ください。"`
- 上長承認要警告（CF-009）: `"経費金額が5,000円を超えています。事前に上長の承認を得てください。上長承認済みですか？"`
- 申請書生成完了提示: `"申請書ドラフトを生成しました。ファイルパス：[file_path]\n上記ファイルを確認の上、所定の申請先へご提出ください（提出操作はご自身で実施してください）。"`

**戻り値の形式**: `str`（申請フロー完了メッセージまたはエラーメッセージ）

### 2.3 ビジネスロジック

#### 2.3.1 バリデーションルール参照先

AG-003のシステムプロンプトに組み込む申請バリデーションルールは `agent_knowledge/receipt_policies.py` から読み込む。

**`agent_knowledge/receipt_policies.py` の役割**:
- 経費申請に関するバリデーションルール（申請期限・上長承認閾値・経費区分・差し戻し基準等）を定数として定義する
- `_build_expense_agent_system_prompt()` 関数内でインポートし、プロンプトに埋め込む

**主要な定数（例）**:
```python
# agent_knowledge/receipt_policies.py
DEADLINE_MONTHS = 3                    # 申請期限（経費発生日から3ヶ月以内）
MANAGER_APPROVAL_THRESHOLD = 5000      # 上長承認が必要な経費合計（円）
EXPENSE_CATEGORIES = ["事務用品費", "宿泊費", "資格精算費", "その他経費"]
```

#### 2.3.2 システムプロンプト

##### システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 動的生成関数 `_build_expense_agent_system_prompt(application_date: str) -> str` |
| 動的生成の理由 | CF-006の申請期限チェック（BRL-18）で経費発生日とapplication_dateを比較するため、application_dateをプロンプトに埋め込む必要がある。また `agent_knowledge/receipt_policies.py` のバリデーションルールを動的に組み込む |

##### 動的に埋め込む情報
- `application_date`: 申請日（YYYY-MM-DD形式。invocation_stateから取得）
- `deadline_date`: 申請期限基準日（application_date から3ヶ月前の日付。BRL-18の判断基準）
- `agent_knowledge/receipt_policies.py` のバリデーション定数（DEADLINE_MONTHS・MANAGER_APPROVAL_THRESHOLD・EXPENSE_CATEGORIES 等）

##### プロンプト全文

```
あなたは社内申請AIシステムの「経費精算申請エージェント」です。
申請者名と申請日はすでに設定済みです（invocation_stateから取得しています）。
申請日（application_date）: {application_date}
申請期限基準日（この日付以降の経費発生日のみ申請可能）: {deadline_date}

【申請バリデーションルール（agent_knowledge/receipt_policies.py より）】
- 申請期限: 経費発生日から{deadline_months}ヶ月以内
- 上長承認が必要な閾値: 経費合計 {manager_approval_threshold}円超
- 経費区分: {expense_categories}

【あなたの役割】
経費精算申請フロー全体（領収書読み取り・経費情報収集・経費区分判断・申請書生成・ルールチェック・最終提示）を担当します。
TOOL-001（交通費計算ツール）は使用しません。
領収書画像の読み取りには image_reader ツールを使用します。

【処理フロー】
1. 領収書画像を受け取る（CF-004 Step 1-2）
   - ユーザーへ領収書画像の提出を促す
   - 画像が提出された場合：image_reader ツールで店舗名・金額・日付・品目を自動抽出する（BRL-16）
     - 抽出成功：抽出結果をユーザーへ提示し確認を取る
     - 抽出失敗：Step 4（手動入力）へ遷移する
   - 画像が提出されない場合（「スキップ」等）：Step 4（手動入力）へ遷移する

2. 経費情報の確認または手動入力（CF-004 Step 3-4）
   - 自動抽出の場合：抽出結果（店舗名・金額・日付・品目）をユーザーが確認する
   - 手動入力の場合：店舗名・金額・日付・品目をユーザーが入力する

3. 経費区分の自動判断（CF-004 Step 5、BRL-17）
   - 品目から経費区分（{expense_categories}）を自動判断する
   - 判断結果をユーザーへ提示し確認を取る
   - 判断不能または異議あり：4択（{expense_categories}）をユーザーへ提示する

   【経費区分の判断基準（BRL-17）】
   - 事務用品費：文房具、印刷用紙、コピー用品、事務用品全般
   - 宿泊費：ホテル、旅館、宿泊施設への支払い
   - 資格精算費：資格試験費用、受験料、資格取得に関する教材費
   - その他経費：上記3区分に該当しない経費（会議費、交際費、研修費等）

4. 業務目的を収集する（CF-004 Step 6-7、BRL-20）
   - 業務目的が入力されていない場合は入力を促す

5. 収集済み申請情報をテキストとして整理・提示する（ドラフト提示ステップ）
   - 申請者名・申請日・経費明細（購入日/店舗名/品目/経費区分/金額）・業務目的・合計金額を提示する
   - ※この時点ではTOOL-002を呼び出さない（テキスト提示のみ）
   - OK/修正/キャンセルの3択を取得する（BRL-06）

6. OK承認後にTOOL-002（generate_expense_form）を呼び出す（HumanApprovalHookが承認ゲートとして機能）
   - applicant_name, application_date, items（経費明細リスト）, business_purpose を渡す
   - 成功時：生成されたファイルパスをユーザーへ提示する
   - 失敗時：エラーメッセージをユーザーへ提示する

7. 申請ルールチェックを実施する（CF-006）
   - 申請期限チェック（BRL-18）: 全経費発生日が申請期限基準日（{deadline_date}）以降であることを確認する
     - 超過している場合はCF-008へ遷移し「申し訳ありません。経費発生日から3ヶ月を超えているため、この申請は受け付けられません。総務部門にご相談ください。」と通知する
   - 上長承認要否判定（BRL-19）: 経費金額の合計が{manager_approval_threshold}円を超える場合はCF-009へ遷移し上長承認確認を促す
   - 差し戻しリスク評価（BRL-08）: リスクが高い場合は警告を提示する（判定基準は要件上未定義）

8. 申請書ドラフト・チェック結果・最終提示（CF-007）
   - 申請書ドラフトのファイルパスを提示する
   - 提出操作はユーザーが実施すること（BRL-09。申請書の自動提出禁止）

【修正選択時】
- CF-004 Step 1（領収書提出促し）へ戻り、収集情報を最初からやり直す

【キャンセル選択時】
- 申請を中断し、セッションを終了する

【禁止事項】
- 申請書の自動提出（BRL-09）
- ドラフト提示ステップ（テキスト整理・提示）でのTOOL-002の呼び出し
- AG-001・AG-002への委任（循環呼び出し禁止）
- TOOL-001（交通費計算ツール）の呼び出し（AG-003は使用しない）

【エラー時の振る舞い】
- TOOL-002のテンプレートファイルが見つからない・ファイル書き込みエラー等のシステム系エラーが発生した場合は、エラー内容を要約して呼び出し元エージェント（AG-001）に返すこと
- ループ上限（30回）に達した場合は「処理が複雑すぎるため終了します。最初からやり直すには「reset」と入力してください。」とユーザーへ提示する
```

##### プロンプト設計の意図

| 要素 | 意図 |
|------|------|
| バリデーションルールの外部参照（receipt_policies.py） | ルールの変更時に設計書・ソースコードの一か所のみを修正すれば済むよう、ポリシーをモジュールに集約する |
| 動的な申請日・申請期限基準日の埋め込み | LLMが実行時の申請日を正確に参照し、BRL-18（経費発生日から3ヶ月以内）の期限判断を行えるようにする |
| image_reader ツールの明示 | LLMが領収書画像の読み取りに `strands_tools` の `image_reader` ツールを使用することを明確に指示する（BRL-16） |
| 経費区分判断基準の明示 | 4区分（事務用品費・宿泊費・資格精算費・その他経費）の判断基準をプロンプトに記載し、LLMによる一貫した分類を実現する（BRL-17） |
| 画像抽出とフォールバック（手動入力）の明示 | 領収書画像なし・抽出失敗時の手動入力フローを明確化する（BRL-16例外） |
| ドラフト提示とツール呼び出しの分離 | TOOL-002はHumanApprovalHook承認後にのみ呼び出すことを明示し、テキスト提示とExcel生成を分離する（BRL-06/GRD-016） |
| エラー時の呼び出し元への通知 | システム系エラー時はAG-001へ要約を返し、AG-001がユーザーへエスカレーション案内を提示する |

#### 2.3.3 申請ルールチェック（CF-006）

| ルール | 条件 | 対応 |
|--------|------|------|
| BRL-18（申請期限） | 経費発生日が申請期限基準日（application_date - 3ヶ月）より前の場合 | CF-008へ遷移して申請不可を通知する |
| BRL-19（上長承認） | 経費金額の合計が5,000円超の場合 | CF-009へ遷移して上長承認確認メッセージを提示する（警告フラグ） |
| BRL-08（差し戻しリスク） | 差し戻しリスクが高いと判断した場合（判定基準は要件上未定義） | 警告メッセージをユーザーへ提示する |

#### 2.3.4 処理フロー

```
開始（AG-001から expense_application_agent_tool 経由で呼び出し）
  ↓
invocation_stateから applicant_name, application_date, session_id を取得
  ↓
ExpenseAgentFactory.get_agent(session_id, application_date) でAgentインスタンスを取得（キャッシュまたは新規生成）
  ↓
  invocation_stateのバリデーション（ファクトリ内）
    - applicant_name: min_length=1。不正時 → エラーを返す
    - application_date: YYYY-MM-DD形式。不正時 → エラーを返す
  ↓
対話ループ開始（SlidingWindow(15)）
  ↓
CF-004 Step 1: 領収書画像提出促し
  ↓
CF-004 Step 2-3: 領収書画像受け取り
  - 画像提出あり → image_reader ツールで自動抽出（BRL-16）
    - 抽出成功 → 結果提示・確認 → Step 5へ
    - 抽出失敗 → Step 4（手動入力）へ
  - 画像提出なし（スキップ）→ Step 4（手動入力）へ
  ↓
CF-004 Step 4（手動入力）: 店舗名・金額・日付・品目を収集
  ↓
CF-004 Step 5: 経費区分自動判断（BRL-17）
  - 判断成功 → 判断結果提示・確認
  - 判断不能または異議あり → 4択提示 → ユーザー選択
  ↓
CF-004 Step 6-7: 業務目的収集（BRL-20）
  ↓
ドラフト提示ステップ（テキストのみ。TOOL-002呼び出しなし）
  - 申請情報サマリーをテキストで表示
  - OK/修正/キャンセルを取得
  ↓
  修正 → CF-004 Step 1へ戻る
  キャンセル → セッション終了
  OK → 次へ（HumanApprovalHookがBeforeToolCallEventで承認ゲートとして機能）
  ↓
ツール呼び出しステップ: TOOL-002（generate_expense_form）呼び出し
  - applicant_name, application_date, items（経費明細リスト）, business_purpose を渡す
  - 成功: file_pathをユーザーへ提示
  - 失敗: エラーメッセージを提示し、システム系エラーの場合はAG-001へ返す
  ↓
CF-006: 申請ルールチェック
  - BRL-18（申請期限）チェック → 超過: CF-008遷移
  - BRL-19（上長承認要否）チェック → 要: CF-009遷移（警告フラグ）
  - BRL-08（差し戻しリスク評価）
  ↓
CF-007: 申請書ドラフト・チェック結果・最終提示
  - ファイルパス提示
  - 提出操作はユーザーが実施することを案内（BRL-09）
  ↓
AG-001へ処理完了メッセージを返す
```

### 2.5 連携設計

#### 2.5.1 Agent as Tools設計

AG-003は `@tool(context=True)` デコレータで実装された `expense_application_agent_tool` 関数として AG-001 に公開される。ツール関数内部では `ExpenseAgentFactory` クラスがAgentインスタンスを管理する。

**ツール名**: `expense_application_agent_tool`

**ToolContextの使用**:
- `@tool(context=True)` デコレータを使用
- `tool_context.invocation_state` から `session_id`・`applicant_name`・`application_date` を取得する
- `session_id` はファクトリクラス内でFileSessionManagerの初期化に使用する
- `applicant_name`・`application_date` をAG-003のinvocation_stateとして設定する（session_idは含めない）

#### 2.5.2 invocation_stateの受け渡しとAgentインスタンスのキャッシュ

```python
class ExpenseAgentFactory:
    """AG-003エージェントインスタンスをsession_idキーで管理するファクトリクラス。"""

    _instances: dict[str, Agent] = {}
    _logger = logging.getLogger(__name__)

    @classmethod
    def get_agent(cls, session_id: str, application_date: str) -> Agent:
        if session_id not in cls._instances:
            from agent_knowledge.receipt_policies import (
                DEADLINE_MONTHS, MANAGER_APPROVAL_THRESHOLD, EXPENSE_CATEGORIES
            )
            from strands_tools import image_reader
            cls._instances[session_id] = Agent(
                system_prompt=_build_expense_agent_system_prompt(
                    application_date,
                    DEADLINE_MONTHS,
                    MANAGER_APPROVAL_THRESHOLD,
                    EXPENSE_CATEGORIES,
                ),
                tools=[image_reader, generate_expense_form],
                conversation_manager=SlidingWindowConversationManager(window_size=15),
                hooks=[
                    HumanApprovalHook(tool_names=["generate_expense_form"]),
                    LoopControlHook(max_iterations=30, agent_name="AG-003"),
                ],
                session_manager=FileSessionManager(f"data/sessions/{session_id}"),
                callback_handler=None,
            )
            cls._logger.info(
                "[AG-003] エージェントインスタンス生成: session_id=%s", session_id
            )
        return cls._instances[session_id]

    @classmethod
    def remove(cls, session_id: str) -> None:
        cls._instances.pop(session_id, None)


@tool(context=True)
def expense_application_agent_tool(query: str, tool_context: ToolContext) -> str:
    logger = logging.getLogger(__name__)
    error_handler = ErrorHandler()
    state = tool_context.invocation_state
    session_id = state.get("session_id", "")
    applicant_name = state.get("applicant_name", "")
    application_date = state.get("application_date", "")

    # invocation_stateのバリデーション
    if not applicant_name:
        return "エラー: 申請者名が設定されていません。"
    if not _is_valid_date(application_date):
        return "エラー: 申請日の形式が不正です。"

    agent = ExpenseAgentFactory.get_agent(session_id, application_date)
    agent_state = {
        "applicant_name": applicant_name,
        "application_date": application_date,
    }

    try:
        response = agent(query, invocation_state=agent_state)
        return str(response)
    except LoopLimitError as e:
        logger.warning(
            "[AG-003] ループ上限到達: %s/%s, session_id=%s, query=%.50s",
            e.current_iteration, e.max_iterations, session_id, query
        )
        return error_handler.handle_loop_limit_error(e)
    except ContextWindowOverflowException as e:
        logger.warning(
            "[AG-003] コンテキストウィンドウ超過: session_id=%s, query=%.50s",
            session_id, query
        )
        return error_handler.handle_context_window_error(e)
    except MaxTokensReachedException as e:
        logger.warning(
            "[AG-003] 最大トークン数到達: session_id=%s, query=%.50s",
            session_id, query
        )
        return error_handler.handle_max_tokens_error(e)
    except RuntimeError as e:
        logger.error(
            "[AG-003] RuntimeError: %s, session_id=%s, query=%.50s",
            str(e), session_id, query, exc_info=True
        )
        return error_handler.handle_runtime_error(e)
    except Exception as e:
        logger.error(
            "[AG-003] 予期しないエラー: %s, session_id=%s, query=%.50s",
            str(e), session_id, query, exc_info=True
        )
        return error_handler.handle_unexpected_error(e)
```

## 3. 実装詳細

### 3.1 クラス設計

#### 3.1.1 ExpenseAgentFactoryクラス

##### 説明
AG-003のAgentインスタンスを `session_id` をキーとして管理するファクトリクラス。`agent_knowledge/receipt_policies.py` のバリデーションルールと `strands_tools` の `image_reader` をインポートしてエージェント生成に使用する。

##### クラス変数

| 変数名 | 型 | 説明 |
|--------|-----|------|
| `_instances` | `dict[str, Agent]` | session_idキーのAgentインスタンスキャッシュ辞書 |
| `_logger` | `logging.Logger` | Pythonロガーインスタンス |

##### get_agentメソッド（classmethod）

キャッシュに `session_id` が存在しない場合のみ新しい Agent インスタンスを生成してキャッシュし、Agentインスタンスを返す。

**処理内容**:
1. `session_id` がキャッシュに存在する場合はそのまま返す
2. 存在しない場合: `agent_knowledge/receipt_policies.py` からルール定数をインポートする
3. `strands_tools` から `image_reader` をインポートする
4. `_build_expense_agent_system_prompt()` でシステムプロンプトを生成する
5. Agentインスタンスを生成（`tools=[image_reader, generate_expense_form]`）してキャッシュに格納する
6. INFO ログ出力: `"[AG-003] エージェントインスタンス生成: session_id={session_id}"`

##### removeメソッド（classmethod）

セッションリセット時にキャッシュから指定 `session_id` のエントリを削除する。

#### 3.1.2 _build_expense_agent_system_prompt関数

##### 説明
AG-003の動的システムプロンプトを生成する。application_date・申請期限基準日・`agent_knowledge/receipt_policies.py` のルール定数を埋め込む。

##### 引数
- `application_date` (str): 申請日（YYYY-MM-DD形式）
- `deadline_months` (int): 申請期限（月数）
- `manager_approval_threshold` (int): 上長承認閾値（円）
- `expense_categories` (list[str]): 経費区分リスト

##### 戻り値
- `str`: 動的生成されたシステムプロンプト全文

##### 処理内容
1. `application_date` を `datetime.date` にパースする
2. `deadline_date = application_date_obj - relativedelta(months=deadline_months)` で申請期限基準日を算出する
3. `EXPENSE_AGENT_SYSTEM_PROMPT_TEMPLATE.format(...)` を返す

#### 3.1.3 _is_valid_date関数

##### 説明
文字列がYYYY-MM-DD形式の有効な日付かを検証する。AG-002の同関数と同実装。

##### 引数
- `date_str` (str): 検証対象の日付文字列

##### 戻り値
- `bool`: 有効な場合 True

### 3.2 エラーハンドリング

`expense_application_agent_tool` 関数内で例外を個別にキャッチし、ErrorHandler に委譲してユーザー向けメッセージ文字列を返す。ログ出力はErrorHandler呼び出し前に実施する。すべての例外は `str` を返して呼び出し元（AG-001）へ伝播させる（例外を再 raise しない）。

| 例外クラス | ログレベル | 戻り値 | ユーザー向けメッセージ |
|-----------|----------|--------|---------------------|
| `LoopLimitError` | WARNING | `str` | `ErrorHandler.handle_loop_limit_error(e)` の返却値 |
| `ContextWindowOverflowException` | WARNING | `str` | `ErrorHandler.handle_context_window_error(e)` の返却値 |
| `MaxTokensReachedException` | WARNING | `str` | `ErrorHandler.handle_max_tokens_error(e)` の返却値 |
| `RuntimeError` | ERROR（exc_info=True） | `str` | `ErrorHandler.handle_runtime_error(e)` の返却値 |
| `Exception` | ERROR（exc_info=True） | `str` | `ErrorHandler.handle_unexpected_error(e)` の返却値 |

**例外キャッチ順序**: LoopLimitError → ContextWindowOverflowException → MaxTokensReachedException → RuntimeError → Exception の順で個別にキャッチする。

**ログへのquery埋め込み**: ログメッセージには `query` の先頭50文字（`query[:50]`）を含める（`"query=%.50s"` 形式）。

## 5. 補足情報

### 5.1 実装上の注意点

1. **クラスベース設計**
   - Agentインスタンスの管理を `ExpenseAgentFactory` クラスとして実装する
   - `@tool(context=True)` デコレータの `expense_application_agent_tool` 関数内でファクトリクラスを使用する
   - ファクトリクラスはモジュールレベルのクラス変数（`_instances`）でインスタンスを管理する

2. **`agent_knowledge/receipt_policies.py` からのルール参照**
   - `ExpenseAgentFactory.get_agent()` 内でインポートし、プロンプト生成に渡す
   - ルール変更時はこのモジュールのみ修正すればよい（設計書・ソースコードの変更箇所を局所化する）

3. **`strands_tools` の `image_reader` ツールの使用**
   - Agentのツールリストに `image_reader`（`from strands_tools import image_reader`）を登録する
   - 領収書画像の読み取りは LLM が `image_reader` ツールを呼び出す形式で実行する（BRL-16）
   - `image_reader` による抽出失敗時は手動入力（CF-004 Step 4）へ遷移する

4. **HumanApprovalHookの登録**
   - `HumanApprovalHook(tool_names=["generate_expense_form"])` をAG-003インスタンスに登録する。AG-002の `generate_transport_expense_form` は対象外
   - 承認はドラフト提示（テキスト）後・TOOL-002実行前の1回のみ発生する

5. **ドラフト提示とツール呼び出しの分離の徹底**
   - ドラフト提示ステップ（収集情報のテキスト整理・提示）でTOOL-002を呼び出してはならない
   - Human-in-the-LoopでOKを取得した後にのみTOOL-002を呼び出す（HumanApprovalHookが承認ゲートとして機能）

6. **経費区分の自動判断と4択フォールバック**
   - LLMがBRL-17の判断基準（事務用品費・宿泊費・資格精算費・その他経費）に基づいて自動判断する
   - 判断不能または異議あり時は4択をユーザーへ提示し、選択結果を使用する

7. **callback_handler=None の設定**
   - AG-003のAgentインスタンスには `callback_handler=None` を指定し、AG-001経由の一元表示を実現する

8. **invocation_stateは辞書リテラルで渡す**
   - 専用Pydanticモデルは定義せず、辞書リテラルで渡す（バリデーション方針 3.2節）

9. **applicant_nameのマスキング**
   - ログ出力時は `_mask_applicant_name()` 関数で先頭1文字 + `***` にマスキングする。invocation_stateに渡す値・ユーザーへの表示はマスキングしない

---

### 5.2 パフォーマンス考慮事項

1. **SlidingWindow(15)による会話履歴の管理**
   - 領収書抽出・経費情報収集・申請書生成・ルールチェックが収まるウィンドウサイズを設定する
   - AG-002（SlidingWindow(20)）より少ない理由：移動区間のループ収集がなく、経費明細は1件ずつ収集するため

2. **Agentインスタンスキャッシュによるパフォーマンス確保**
   - `ExpenseAgentFactory._instances` キャッシュにより、複数回の呼び出し時のエージェント初期化コストを削減する

---

### 5.3 セキュリティ考慮事項

1. **invocation_stateのバリデーション**
   - ファクトリ関数内でapplicant_name（min_length=1）・application_date（YYYY-MM-DD形式）を個別チェックする

2. **HumanApprovalHookによる承認制御**
   - TOOL-002（generate_expense_form）はHumanApprovalHookのBeforeToolCallEventによる承認なしには実行されない（GRD-016）

3. **applicant_nameのマスキング**
   - ログ出力時は先頭1文字 + `***` でマスキングし、ログファイルへの個人情報の過剰な記録を防止する

- 申請期限超過（BRL-18）でCF-008へ遷移すること
