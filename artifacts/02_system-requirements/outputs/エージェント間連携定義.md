## 1. 連携ポリシー（概要）

- 連携方式：同期（リクエスト/レスポンス）― AG-001がAG-002/AG-003をAgent-as-Tool（Strands SDK）パターンで呼び出す
- 連携単位：タスク（申請種別ごとに専門エージェントへ委譲）
- 失敗時方針：人へエスカレーション（CF-009に従い管理部門への問い合わせを案内する）
- 相互呼び出し制約：循環呼び出し禁止（AG-002/AG-003はAG-001からのみ呼び出し可）、最大委譲深度：1段、タイムアウト：要件上未定義

## 3. 連携種別の定義（記入ルール）

### 3.1 協調（Collaboration）
- 目的：複数エージェントが**並行/分担**して成果を統合
- 本システムでの適用：なし（申請種別ごとに単一スペシャリストに委譲するため）

### 3.2 委譲（Delegation）
- 目的：起点エージェントがサブタスクを**丸投げ**して結果を受け取る
- 本システムでの適用：LNK-001（AG-001→AG-002）、LNK-002（AG-001→AG-003）
- Strands SDKのAgent-as-Toolパターンで実装

### 3.3 イベント（Event-driven）
- 目的：状態変化（例：申請書生成完了）を発火し、購読側が処理
- 本システムでの適用：EVT-001（AG-002自動チェック）、EVT-002（AG-003自動チェック）
- BRL-16による申請書生成後の自動チェック実行

## 4. 連携インタフェース定義

### 4.1 連携ID：LNK-001（AG-001 → AG-002 委譲）

#### 4.1.1 トリガー条件
- 発火条件：AG-001が申請種別を「交通費精算申請」と判断（JD-01）またはユーザーが「交通費精算申請」を選択（BRL-04）
- 事前条件（Pre-condition）：申請種別判断完了。社内申請ルール（DATA-001）がナレッジベースに登録済み
- 事後条件（Post-condition）：AG-002が交通費精算申請書（下書き）を生成し、チェック結果を提示した

#### 4.1.2 入力スキーマ（構造化推奨）
```json
{
  "request_id": "string",
  "task": "交通費精算申請書作成",
  "context": {
    "application_type": "交通費精算申請",
    "applicant_name": "string",
    "application_date": "string"
  },
  "constraints": {
    "approval_required": true,
    "approval_gate": "APR-001"
  }
}
```

#### 4.1.3 出力スキーマ（構造化推奨）
```json
{
  "request_id": "string",
  "status": "completed | error",
  "output": {
    "application_file_path": "string",
    "check_result": "OK | NG",
    "check_details": []
  },
  "error": {
    "code": "string",
    "message": "string"
  }
}
```

#### 4.1.4 エラー時動作
- AG-002からエラーが返却された場合：AG-001がエラー内容を社員に提示し、管理部門への問い合わせを案内する（CF-009）
- リトライ：要件上未定義

#### 4.1.5 セキュリティ・認可
- 利用可能エージェント：AG-001のみ（ABAC-001：AG-002はAG-001からの委譲でのみAPR-001通過後に申請書生成可能）
- セッション状態（invocation_state）経由でコンテキストを引き継ぐ

### 4.3 連携ID：EVT-001（AG-002 申請書生成後自動チェック）

#### 4.3.1 トリガー条件
- 発火条件：AG-002による交通費精算申請書（下書き）生成完了（BRL-16）
- 事前条件（Pre-condition）：OUT-003が生成済み。社内申請ルール（DATA-001）がナレッジベースに登録済み
- 事後条件（Post-condition）：チェック結果（OK/NG）と修正案が社員に提示された

#### 4.3.2 処理内容
- 申請書（下書き）の全項目を社内申請ルールと照合する（JD-13）
- チェック項目：申請期限（BRL-11）、高額申請（BRL-12）、業務目的（BRL-13）、申請ルール全般（KN-009）
- チェック結果（Lv3根拠：判断条件・数値・適用ルール名）を社員に提示する

#### 4.3.3 エラー時動作
- チェック実行失敗時：エラーを社員に提示し、手動チェックを促す
