## 1. 概要

### 1.1 コンポーネントの目的

AG-001から委譲された経費精算申請フロー全体を自律実行する。領収書画像（D-009）から`strands_tools` の `image_reader` を使用して情報を自動読み取りし（BRL-16）、経費区分を自動判断する（BRL-17）。全必須情報充足後にTOOL-002b（generate_expense_application）でExcel申請書を生成する。申請書の必須項目・整合性・ルール適合性・上長承認要否をチェックして合格済み申請書を社員に提示する。

### 1.2 主要な責務

- **必須情報収集**: BRL-07の必須項目（店舗名・経費区分・金額・経費発生日・業務目的）を一括収集質問で収集する
- **妥当性チェックルール適用**: エージェント初期化時に `agent_knowledge/receipt_policies.py` を読み込み、プロンプトに組み込む
- **領収書自動読み取り**: `strands_tools` の `image_reader` を使用して領収書画像（D-009）から店舗名・金額・日付・品目を自動読み取りする（BRL-16）
- **経費区分自動判断**: 抽出した品目から経費区分を自動判断する（BRL-17）。判断不可時は「その他経費」として扱う
- **申請期限チェック**: 経費発生日から90日以内であることを確認する（BRL-12/JD-11）
- **申請書ドラフト提示**: 収集済み申請情報をテキストとして整理・提示する（ツール呼び出しなし）
- **申請書自動作成**: HumanApprovalHookのBeforeToolCallEventによるOK確認後にTOOL-002bでExcel申請書を生成する（BRL-02/COND-002/COND-004）
- **申請書チェック**: 必須項目（JD-06/JD-07）・整合性（JD-08/JD-09）・ルール適合性（BRL-03）・上長承認要否（BRL-18）をチェックする

### 2.2 インターフェース設計

#### 2.2.1 入力

**初期化時の入力（ToolContext経由）**:
- `applicant_name`（str, 必須）: AG-001から`invocation_state`経由で受け取る申請者名
- `application_date`（str, 必須）: AG-001から`invocation_state`経由で受け取る申請日（YYYY-MM-DD形式）
- `session_id`（str, 必須）: AG-001から`invocation_state`経由で受け取るセッションID

**対話ループでの入力**:
- `query` (str): 委譲メッセージ（申請種別：経費精算申請・申請意図テキスト）
- その後は社員の自然文入力（領収書画像・経費情報・確認応答）

**ToolContext経由で受け取る情報**:
```python
{
    "applicant_name": str,      # 申請者名
    "application_date": str,    # 申請日（YYYY-MM-DD形式）
    "session_id": str           # セッションID（SessionManager初期化用）
}
```

#### 2.2.2 出力

**出力先**: CLIの標準出力（社員への案内・確認・チェック結果メッセージ）

**出力メッセージ**:
- 情報収集質問: `"以下の情報を一度に入力してください。\n【店舗名】\n【経費区分（事務用品費/宿泊費/資格精算費/その他経費）】\n【金額（円）】\n【経費発生日（YYYY-MM-DD）】\n【業務目的】\n\n領収書画像がある場合はここで提供してください。"`
- 領収書自動読み取り結果確認: `"領収書から以下の情報を読み取りました。\n【店舗名】{store}\n【金額】{amount}円\n【日付】{date}\n【経費区分（自動判断）】{category}\nよろしいですか？"`
- 申請書ドラフト提示: `"以下の内容で申請書を作成します。\n【申請者名】{name}\n【申請日】{date}\n【店舗名】{store}\n..."`
- 申請書チェック合格: `"申請書チェックが完了しました。ファイルパス: data/output/{session_id}/APP_...xlsx"`
- 上長承認案内（BRL-18）: `"経費金額が5,000円を超えているため、上長承認が必要です。上長へご確認ください。"`
- 申請期限超過: `"申請期限（経費発生日から90日以内）を超過しているため、この申請は受け付けられません。担当部署へご相談ください。"`

**戻り値の形式**: `str`（AG-003の最終応答テキスト。AG-001のexpense_agent_toolに返す）

### 2.3 ビジネスロジック

#### 2.3.1 システムプロンプト

##### システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 動的生成関数 |
| 動的生成の理由 | CF-003 Step 4で申請期限チェック（BRL-12）を行うため、実行時の申請日（application_date）と期限の基準日（application_dateの90日前）をLLMに伝える必要がある。また、妥当性チェックルール（`agent_knowledge/receipt_policies.py`）をエージェント初期化時に読み込んでプロンプトに組み込む |

##### 動的に埋め込む情報
- `applicant_name`: invocation_stateから受け取った申請者名
- `application_date`: invocation_stateから受け取った申請日（YYYY-MM-DD形式）
- `deadline_date`: `application_date` の90日前の日付（`application_date - timedelta(days=90)` で計算）
- `receipt_policies`: `agent_knowledge/receipt_policies.py` から読み込んだ妥当性チェックルール

##### プロンプト全文

```
あなたは経費精算申請を担当する専門エージェントです。

【役割】
経費精算申請に必要な情報を社員から収集し、申請書を作成してチェックを実施します。

【申請情報】
- 申請者名: {applicant_name}
- 申請日: {application_date}
- 申請期限の基準日: {deadline_date}（この日以降の経費発生日は申請期限超過となります）

【妥当性チェックルール】
{receipt_policies}

【処理フロー】

1. CF-003: 経費精算情報収集
   以下の必須項目を収集します（BRL-07）。
   必須項目を1つずつ順番に聞いてはいけません。以下の形式で一括収集してください:

   「以下の情報を一度に入力してください。
   【店舗名】
   【経費区分】（事務用品費/宿泊費/資格精算費/その他経費）
   【金額（円）】
   【経費発生日（YYYY-MM-DD）】
   【業務目的】（例: 取引先打ち合わせ費用、研修参加費）

   領収書画像がある場合はここで提供してください。」

2. CF-003 Step 2-3: 領収書画像の自動処理（BRL-16/BRL-17）
   社員が領収書画像を提供した場合:
   a. strands_toolsのimage_readerを使用して画像から店舗名・金額・日付・品目を自動読み取りします
   b. 品目から経費区分を自動判断します（BRL-17）
      - 事務用品・文具・消耗品 → 事務用品費
      - 宿泊・ホテル → 宿泊費
      - 資格・試験・検定 → 資格精算費
      - 上記以外 → その他経費（判断不可時も「その他経費」として扱う）
   c. 読み取り結果を社員に提示して確認を取ります
   自動読み取りに失敗した場合は手動入力を促します。

3. CF-003 Step 4: 申請期限チェック（BRL-12）
   経費発生日が申請期限の基準日（{deadline_date}）より前の場合は申請期限超過です。
   超過している場合:「申請期限（経費発生日から90日以内）を超過しているため、
   この申請は受け付けられません。担当部署へご相談ください。」
   と伝えて申請を停止してください。

4. CF-004 Step 1: 申請書ドラフト提示（ツール呼び出しなし）
   収集済み申請情報をテキストとして整理して社員に提示します。
   この時点ではgenerate_expense_applicationツールを呼び出してはいけません。
   HumanApprovalHookによる承認確認後にのみツールを呼び出します。

5. CF-004 Step 3: 申請書生成（HumanApprovalHookによるOK確認後）
   HumanApprovalHookが申請者の確認（OK/修正/キャンセル）を取得します。
   - OK選択後: generate_expense_applicationツールを呼び出して申請書を生成します
   - 修正選択後: CF-003（情報収集）へ戻ります
   - キャンセル選択後: 申請を停止します

6. CF-005: 申請書チェック
   以下を順番にチェックします:
   a. 必須項目チェック（JD-06/JD-07）: 店舗名・経費区分・金額・日付・業務目的がすべて揃っているか
   b. 整合性チェック（JD-08/JD-09）: 日付と申請日の整合性、金額が正の整数か
   c. ルール適合性チェック（BRL-03）: 業務目的が適切か
   d. 上長承認要否チェック（BRL-18）: 経費金額が5,000円を超える場合は上長承認が必要である旨を案内する

7. 合格済み申請書提示
   チェック合格後、申請書ファイルパスを提示して申請書提出を社員に案内します。
   申請書の提出はエージェントが実行しません（社員のみが実施可能）。

【判断基準】
- 全必須情報充足確認（JD-04）: 店舗名・経費区分・金額・経費発生日・業務目的がすべて収集済みであること
- 申請期限チェック（JD-11）: 経費発生日 >= {deadline_date} → 超過
- 上長承認要否（JD-13）: 経費金額 > 5,000円 → 上長承認案内（BRL-18）

【ツール呼び出しルール】
- generate_expense_applicationはHumanApprovalHookのOK確認後のみ呼び出すこと
- ドラフト提示時（CF-004 Step 1）にgenerate_expense_applicationを呼び出してはならない
- TOOL-001（calculate_transport_fare）はこのエージェントでは使用しないこと

【エラー時の振る舞い】
- ツールエラー・システム障害等が発生した場合は、エラー内容を要約して呼び出し元（AG-001）に返すこと

【禁止事項】
- 申請書提出を実行しないこと（GRD-011）
- 承認前にgenerate_expense_applicationを実行しないこと（COND-002/COND-004）
- TOOL-001（calculate_transport_fare）を呼び出さないこと
```

##### プロンプト設計の意図

| 要素 | 意図 |
|------|------|
| 動的日付埋め込み | BRL-12の申請期限チェックでLLMが正確な日付を判断できるようにする |
| 外部ルールファイル組み込み | 妥当性チェックルールを `agent_knowledge/receipt_policies.py` から読み込むことで、ルール変更時にコード修正なしで対応できる |
| 一括収集指示 | 項目を1つずつ聞かないよう明示することで会話ターン数を削減する（BRL-23対策） |
| image_reader使用指示 | strands_toolsのimage_readerで領収書画像から情報を自動読み取りする手順と経費区分の判断基準を明示する（BRL-16/BRL-17） |
| ドラフト提示とツール呼び出しの分離 | HumanApprovalHookのBeforeToolCallEventが発火する前にTOOL-002bが実行されることを防止する |
| TOOL-001禁止 | AG-003はTOOL-001を使用しないことを明示する |

#### 2.3.2 申請ルール

| ルール | 条件 | 対応 |
|--------|------|------|
| BRL-07（必須項目） | 店舗名・経費区分・金額・経費発生日・業務目的が必須 | 全必須項目を一括収集質問で収集する |
| BRL-16（領収書自動読み取り） | 領収書画像が提供された場合 | strands_toolsのimage_readerで店舗名・金額・日付・品目を自動読み取りする |
| BRL-17（経費区分自動判断） | 品目が抽出された場合 | LLMで経費区分を自動判断する。判断不可時は「その他経費」として扱う |
| BRL-12（申請期限） | 経費発生日から申請日まで90日以内 | 超過時は申請を停止する（JD-11 NG） |
| BRL-02（必須情報充足確認） | 申請書作成前 | JD-04で全必須項目充足を確認してからTOOL-002bを呼び出す |
| BRL-03（申請書チェック義務） | 申請書作成後 | 必須項目・整合性・ルール適合性・上長承認要否を順次チェックする |
| BRL-18（上長承認要否） | 経費金額 > 5,000円（JD-13） | 上長承認が必要である旨を案内する |

#### 2.3.3 処理フロー

```
開始（AG-001のexpense_agent_toolから呼び出される）
  ↓
invocation_stateからapplicant_name・application_date・session_idを取得
  - applicant_name/application_dateの存在確認とNone/空文字チェック（バリデーション方針 2.2）
  session_idでSessionManagerを初期化
  agent_knowledge/receipt_policies.py を読み込む
  application_dateとdeadline_date（application_date - 90日）でシステムプロンプトを動的生成
  （receipt_policiesの内容もプロンプトに組み込む）
  ↓
エージェントへ初回queryを渡す（委譲メッセージ）
  ↓
CF-003: 経費精算情報収集
  ↓
  必須項目の一括収集質問を社員に提示する
  （店舗名・経費区分・金額・経費発生日・業務目的・領収書画像）
  ↓
  社員から入力を受け取る
  ↓
  領収書画像が提供された？
    YES → strands_toolsのimage_readerで店舗名・金額・日付・品目を自動読み取りする（BRL-16）
           品目から経費区分を自動判断する（BRL-17）
           自動読み取り結果を社員に提示して確認を取る
           自動読み取り失敗時 → 手動入力を促す
    NO  → 手動入力した情報をそのまま使用する
  ↓
  全必須情報充足確認（JD-04）
    充足 → 次ステップへ
    未充足 → 不足項目を確認してループを繰り返す
  ↓
CF-003 Step 4: 申請期限チェック（BRL-12/JD-11）
  経費発生日 < deadline_date?
    YES（超過）→ 「申請期限を超過しています...」を提示 → セッションTERMINATED → 終了
    NO（期限内）→ 次ステップへ
  ↓
CF-004 Step 1: 申請書ドラフト提示（テキストのみ）
  収集済み申請情報を整理してテキストとして提示する
  ※ この時点でgenerate_expense_applicationは呼び出さない
  ↓
HumanApprovalHook発火（BeforeToolCallEvent: generate_expense_application呼び出し前）
  OK → 次ステップへ
  修正 → CF-003（情報収集）へ戻る
  キャンセル → セッションTERMINATED → 終了
  ↓
CF-004 Step 3: TOOL-002b（generate_expense_application）実行
  - store_name, expense_category, amount, expense_date, purpose を渡す
    （applicant_name・application_dateはinvocation_stateから取得）
  - success: True → file_pathとapplication_dataを受け取る
  - success: False → エラーメッセージを提示 → エスカレーション（EX-04/EX-06）
  ↓
CF-005: 申請書チェック
  a. 必須項目チェック（JD-06/JD-07）: application_dataの全項目確認
  b. 整合性チェック（JD-08/JD-09）: 日付整合性・金額正整数確認
  c. ルール適合性チェック（BRL-03）: 業務目的の適切性確認
  d. 上長承認要否チェック（BRL-18）: 経費金額 > 5,000円 → 上長承認案内
  チェック不合格 → 不備を提示して修正促進
  チェック合格 → 次ステップへ
  ↓
CF-005 Step 5: 合格済み申請書提示
  「申請書の確認が完了しました。ファイルパス: {file_path}
  申請書を確認のうえ、上長へ提出してください。」
  ↓
終了（AG-001のexpense_agent_toolに完了を返す）
```

### 2.4 設定・構成

#### 2.4.1 共通設定

| 設定項目 | 設定値 | 備考 |
|---------|--------|------|
| 会話管理 | `SlidingWindowConversationManager(window_size=15, should_truncate_results=True, per_turn=False)` | 単一明細が多く少ないステップで完結するため小さいウィンドウサイズ（共通設定方針 2.2準拠） |
| リトライ戦略 | Strands Agentsデフォルト設定（LLM API: 指数バックオフ最大6回/初期4秒/最大240秒） | TOOL-002bのファイル書き込みは最大2回リトライ（実行制御方針 7.1準拠） |
| フック | `HumanApprovalHook()`・`LoopControlHook(max_iterations=10, agent_name="expense_agent")` | TOOL-002b呼び出し前の社員確認・ループ制御 |
| セッション管理 | `SessionManager(session_id=session_id, storage_path="data/sessions/")` | invocation_stateのsession_idを使用（共通設定方針 5.3準拠） |
| ガードレール | `guardrail_id=GUARDRAIL_ID, guardrail_version=GUARDRAIL_VERSION` | GUARDRAIL_IDは要件上未定義（環境変数で設定） |

#### 2.4.2 利用ツール

| ツールID | ツール名 | 利用目的 | 呼び出し条件 |
|----------|----------|----------|--------------|
| — | `image_reader`（strands_tools） | 領収書画像から店舗名・金額・日付・品目を自動読み取りする | 社員が領収書画像を提供した場合（BRL-16） |
| TL-002b | `generate_expense_application` | 全必須情報充足後にExcel申請書を生成する | HumanApprovalHookのOK確認後のみ呼び出す（COND-002/COND-004） |

### 2.5 連携設計

#### 2.5.1 Agent as Tools設計

**ツール名**: `expense_agent_tool`

**ツール説明**:
```
経費精算申請フローを実行します。
AG-001（申請受付窓口エージェント）から委譲メッセージ（申請種別：経費精算申請・申請意図テキスト）を受け取り、
経費精算申請に必要な情報収集・領収書自動読み取り・申請書作成・チェックを実行します。
```

**ToolContextの使用**:
- `@tool(context=True)` デコレータを使用
- `tool_context.invocation_state` から `applicant_name`・`application_date`・`session_id` を取得する
- `session_id` でSessionManagerを初期化する
- `applicant_name`・`application_date` でシステムプロンプトを動的生成する

#### 2.5.2 invocation_stateの受け渡しとAgentインスタンスのキャッシュ

```python
_agent_instances: dict[str, Agent] = {}

@tool(context=True)
def expense_agent_tool(query: str, tool_context: ToolContext) -> str:
    state = tool_context.invocation_state
    session_id = state.get("session_id", "")
    applicant_name = state.get("applicant_name", "")
    application_date = state.get("application_date", "")

    # session_id 未登録の場合のみ新しい Agent インスタンスを生成する
    if session_id not in _agent_instances:
        _agent_instances[session_id] = Agent(
            model=get_bedrock_model(),
            system_prompt=get_expense_system_prompt(applicant_name, application_date),
            tools=[image_reader, generate_expense_application],
            agent_id="expense_agent",
            name="経費精算申請エージェント",
            conversation_manager=SlidingWindowConversationManager(
                window_size=15,
                should_truncate_results=True,
                per_turn=False
            ),
            hooks=[HumanApprovalHook(), LoopControlHook(max_iterations=10, agent_name="expense_agent")],
            session_manager=SessionManager(
                session_id=session_id,
                storage_path="data/sessions/"
            ),
            guardrail_id=GUARDRAIL_ID,
            guardrail_version=GUARDRAIL_VERSION,
            guardrail_trace="enabled",
            callback_handler=None,
        )

    # invocation_stateはAG-003内部向けに渡す（session_idは除外）
    agent_state = {"applicant_name": applicant_name, "application_date": application_date}
    response = _agent_instances[session_id](query, invocation_state=agent_state)
    return str(response)
```

## 3. 実装詳細

### 3.1 クラス/関数設計

#### 3.1.1 get_expense_system_prompt関数

##### 説明
申請日（application_date）と申請期限基準日（deadline_date）を動的に埋め込んだシステムプロンプトを生成する。妥当性チェックルールを `agent_knowledge/receipt_policies.py` から読み込んでプロンプトに組み込む。

##### 引数
- `applicant_name` (str): 申請者名（必須）
- `application_date` (str): 申請日（YYYY-MM-DD形式）（必須）

##### 戻り値
- `str`: システムプロンプト全文

##### 処理内容
1. `application_date` を `datetime.date.fromisoformat(application_date)` でdate型に変換する
2. `deadline_date = application_date - timedelta(days=90)` で申請期限基準日を計算する
3. `deadline_date.isoformat()` で文字列に変換する
4. `agent_knowledge/receipt_policies.py` を読み込んで `receipt_policies` 変数に格納する
5. `applicant_name`・`application_date`・`deadline_date`・`receipt_policies` をプロンプトテンプレートに埋め込んで返す

## 5. 補足情報

### 5.1 実装上の注意点

1. **ドラフト提示とTOOL-002bの呼び出しを明確に分離する**
   - CF-004 Step 1（ドラフト提示）ではgenerate_expense_applicationを呼び出さない
   - HumanApprovalHookのBeforeToolCallEventがgenerate_expense_application呼び出し前に発火し、OK選択後にはじめてツールが実行される

2. **領収書自動読み取りはstrands_toolsのimage_readerを使用する**
   - 領収書画像読み取りにはstrands_toolsの `image_reader` ツールを使用する
   - image_readerをAgentのtoolsリストに追加する
   - 自動読み取り失敗時は手動入力を促す

3. **TOOL-001は使用しない**
   - AG-003のAgentインスタンスの`tools`リストにcalculate_transport_fareは含めない
   - システムプロンプトでも明示的に禁止する

4. **invocation_stateのsession_idはAgentへ渡さない**
   - SessionManagerの初期化にのみ使用し、`agent(invocation_state=...)` に含めない（マルチエージェント連携設計 7.3準拠）

5. **専門エージェントの例外捕捉はerror_messageをstrで返す**
   - expense_agent_tool関数内での例外はすべてErrorHandlerでメッセージを生成してstrとして返す（呼び出し元LLMコンテキストに渡されるため）
   - ログ出力はErrorHandler呼び出し前にツール関数が実施する

6. **妥当性チェックルールは agent_knowledge/receipt_policies.py から読み込む**
   - インライン記述ではなく外部ファイルを参照することで、ルール変更時の保守性を確保する

### 5.2 パフォーマンス考慮事項

1. **一括収集質問による会話ターン数削減**
   - 必須項目を1つずつ聞かず一括収集することで、BRL-23（30ターン制限）への消費を抑制する

2. **SlidingWindow(15)による会話履歴管理**
   - AG-002より小さいウィンドウサイズで、経費申請の単一明細フローをカバーする

### 5.3 セキュリティ考慮事項

1. **ファイル出力前の社員確認（HumanApprovalHook）**
   - GRD-010/COND-002/COND-004に準拠し、申請書ファイル出力前に必ず社員確認を挟む

2. **申請書提出禁止（GRD-011）**
   - システムプロンプトの禁止事項に明示し、エージェントが申請書を提出しないよう制御する

### 7.1 機能テスト
- 委譲メッセージを受け取り、一括収集質問で経費情報を収集できること
