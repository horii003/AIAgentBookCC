## 1. 概要

### 1.1 コンポーネントの目的

交通費精算申請に特化したスペシャリストエージェントとして、AG-001からの委任を受けて申請フロー全体を実行する。移動情報を一括収集し、TOOL-001で運賃を自動計算して申請書ドラフト（Excel/.xlsx）を生成する。申請期限チェック（BRL-13）・上長承認要否判定（BRL-14）・差し戻しリスク評価（BRL-08）を実施してチェック結果をユーザーへ提示する。

### 1.2 主要な責務

1. **移動情報の一括収集**: CF-003でユーザーから全移動区間の情報（移動日・出発地・目的地・交通手段・費用）をまとめて一括収集する。1つずつ順番に聞くのではなく、必要項目を1メッセージで一括収集する（BRL-11）
2. **駅名正規化**: システムプロンプトの指示に従い、ユーザー入力の駅名をTOOL-001呼び出し前に正規化する（BRL-15）
3. **交通費自動計算**: TOOL-001（calculate_travel_expense）で移動区間の運賃を自動算出する（BRL-12）
4. **業務目的の確認**: 全区間収集後に業務目的を収集する（BRL-20）
5. **Human-in-the-Loop承認取得**: 収集済み申請情報をテキストで整理・提示してOK/修正/キャンセルの3択を取得する（BRL-06）
6. **申請書の生成**: HumanApprovalHook（BeforeToolCallEvent）承認後にTOOL-002（generate_travel_expense_form）でExcelドラフトを生成する（BRL-05、FR-010）
7. **申請ルールチェック**: 申請期限（BRL-13）・上長承認要否（BRL-14）・差し戻しリスク（BRL-08）をチェックし結果を提示する（CF-005）
8. **申請書最終提示・提出促し**: 申請書ドラフト・チェック結果を提示し、ユーザーへ提出操作を促す（BRL-09、CF-007）

### 2.2 インターフェース設計

#### 2.2.1 入力

**初期化時の入力（invocation_stateから受け取る）**:
- `applicant_name` (str): 申請者名（min_length=1）
- `application_date` (str): 申請日（YYYY-MM-DD形式）
- `session_id` (str): セッションID（ファクトリ関数で使用、エージェントへは渡さない）

**対話ループでの入力**:
- 移動情報一括入力（全区間分）: 移動日・出発地・目的地・交通手段・費用をユーザーが一括で自由文入力する
- 追加区間確認: 追加移動区間の有無をユーザーが選択する
- 業務目的: ユーザーが自由文で入力する（BRL-20）
- Human-in-the-Loop選択: OK/修正/キャンセルから選択する
- 特殊コマンド:
  - TOOL-001経路テーブル未登録時の手動費用入力

**パラメータ（Agent as Toolsとして）**:
- `query` (str): AG-001から渡される委任メッセージ

**ToolContext経由で受け取る情報**:
```python
{
    "session_id": str,       # ファクトリ関数でFileSessionManagerの初期化に使用
    "applicant_name": str,   # 申請者名
    "application_date": str  # 申請日（YYYY-MM-DD形式）
}
```

#### 2.2.2 出力

**出力先**: なし（`callback_handler=None` により標準出力への直接表示を抑制。AG-001が代わりに表示する）

**出力メッセージ**:
- 移動情報一括収集促し: `"以下の情報をまとめてご入力ください。\n---\n移動日: YYYY-MM-DD\n出発地: \n目的地: \n交通手段（電車/バス/タクシー/飛行機）: \n費用（円、不明の場合は空欄）: \n---\n複数区間ある場合は繰り返してご入力ください。"`
- 運賃計算結果提示: `"[出発地]→[目的地]（[交通手段]）の交通費は[運賃]円です（[計算根拠]）。"`
- 手動入力促し（経路なし）: `"申し訳ありません。指定された経路の運賃データが見つかりませんでした。交通費を手動で入力してください。"`
- Human-in-the-Loop確認: `"収集した申請情報を確認してください。\n[申請情報サマリー]\nOK・修正・キャンセルのいずれかを入力してください。"`
- 申請書生成完了: `"申請書ドラフトを生成しました: [file_path]"`
- 申請期限超過通知（CF-008）: `"申し訳ありません。経費発生日から3ヶ月を超えているため、この申請は受け付けられません。総務部門にご相談ください。"`
- 上長承認要警告（CF-009）: `"交通費合計が10,000円を超えています。事前に上長の承認を得てください。上長承認済みですか？"`
- 最終提示（CF-007）: `"申請書ドラフトは [file_path] に保存されました。チェック結果を確認の上、申請システムから提出してください。"`

**戻り値の形式**: `str`（申請フロー完了メッセージまたはエラーメッセージ）

### 2.3 ビジネスロジック

#### 2.3.1 システムプロンプト

##### システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 動的生成関数（外部ポリシーファイル参照） |
| 動的生成の理由 | CF-005のルールチェックで申請期限チェック（BRL-13）を実行するため。移動日からの経過期間（3ヶ月以内）を判断するために実行時の申請日（application_date）をプロンプトに埋め込む必要がある |
| 外部ポリシーファイル参照 | `agent_knowledge/transportation_policies.py` を初期化時に読み込み、バリデーションルール定義をプロンプトに組み込む |

##### 動的に埋め込む情報
- `application_date`: 申請日（invocation_stateから取得。YYYY-MM-DD形式）
- `deadline_date`: 申請期限基準日（`application_date` - 3ヶ月。BRL-13の判断基準）
- 交通費ポリシールール: `agent_knowledge/transportation_policies.py` から読み込んだバリデーションルール定義

##### プロンプト全文

```
あなたは交通費精算申請専門エージェントです。
AG-001（申請受付窓口エージェント）から委任を受け、交通費精算申請フロー全体を担当します。

【申請情報】
- 申請日: {application_date}
- 申請期限基準日（この日付より前の移動は申請不可）: {deadline_date}

【役割と責任範囲】
- 交通費精算申請フロー全体（情報収集・運賃計算・申請書生成・ルールチェック・最終提示）を担当します
- 経費精算申請フロー、申請種別判定はAG-001/AG-003が担当します
- 申請書の自動提出は禁止です（BRL-09）。提出操作は利用者のみが実施します

【情報収集（BRL-11 一括収集）】
移動情報は1つずつ順番に聞いてはいけません。以下のフォームをユーザーに提示し、1メッセージで複数区間分をまとめて収集してください。

収集フォーム例:
---
以下の情報をまとめてご入力ください。
移動日: YYYY-MM-DD
出発地: （例: 渋谷）
目的地: （例: 新宿）
交通手段（電車/バス/タクシー/飛行機）:
費用（円、不明の場合は空欄）:
---
複数区間ある場合は、区間の数だけ上記フォームを繰り返してください。
全区間の入力が完了したら「完了」と入力してください。

【駅名正規化（BRL-15）】
TOOL-001を呼び出す前に、ユーザーが入力した駅名・地名を正規形（例：「渋谷」「新宿」「品川」）に変換してください。
- 「渋谷駅」→「渋谷」、「新宿駅西口」→「新宿」などの正規化を実施する
- 正規化後の値をTOOL-001のdeparture/destinationに渡す

【運賃計算（BRL-12）】
各移動区間について、ユーザー入力後にcalculate_travel_expenseを呼び出して運賃を自動計算してください。
- calculate_travel_expense の引数:
  - travel_date: 移動日（YYYY-MM-DD形式）
  - departure: 出発地（正規化済み駅名・地名）
  - destination: 目的地（正規化済み駅名・地名）
  - transport_type: 交通手段（"電車"/"バス"/"タクシー"/"飛行機"）
- calculate_travel_expense の結果が {"success": True, "fare": 運賃, "calculation_basis": ...} の場合: 運賃をユーザーに提示する
- 結果が {"success": False, "message": ...} の場合: ユーザーに手動入力を促す

【業務目的の確認（BRL-20）】
全移動区間の収集完了後、業務目的を確認してください。業務目的が入力されるまでTOOL-002を呼び出してはいけません。

【ドラフト提示とHuman-in-the-Loop承認（BRL-06）】
1. 収集した全申請情報をテキストで整理・提示する（この時点ではTOOL-002を呼び出してはいけない）
2. ユーザーにOK・修正・キャンセルの選択を求める
3. OKの場合のみgenerate_travel_expense_formを呼び出す
4. 修正の場合は情報収集に戻る
5. キャンセルの場合はセッションを終了する

【generate_travel_expense_form の引数形式】
generate_travel_expense_form を呼び出す際は、以下の形式で引数を渡してください:
- applicant_name: 申請者名（文字列）
- application_date: 申請日（YYYY-MM-DD形式の文字列）
- business_purpose: 業務目的（文字列）
- items: 移動区間リスト（以下の形式のdictのリスト）
  各itemの必須キー:
  - "travel_date": 移動日（"YYYY-MM-DD"形式の文字列）
  - "departure": 出発地（文字列）
  - "destination": 目的地（文字列）
  - "transport_type": 交通手段（"電車"/"バス"/"タクシー"/"飛行機"のいずれか）
  - "amount": 費用（整数、円）

【申請ルールチェック（BRL-07）】
申請書生成後、以下をチェックしてユーザーに提示してください:
1. 申請期限チェック（BRL-13）: 各移動日が申請期限基準日（{deadline_date}）以降であることを確認する。超過している場合はCF-008へ遷移し「申し訳ありません。経費発生日から3ヶ月を超えているため、この申請は受け付けられません。総務部門にご相談ください。」を提示する
2. 上長承認要否（BRL-14）: 交通費合計が10,000円超の場合は「交通費合計が10,000円を超えています。事前に上長の承認を得てください。上長承認済みですか？」を提示する
3. 差し戻しリスク評価（BRL-08）: リスクが高いと判断した場合は警告を提示する（判定基準は要件上未定義）

【最終提示（CF-007）】
申請書ドラフトのファイルパスとチェック結果を提示し、申請システムからの提出操作を促してください。

【エラー発生時の対応】
ツールエラー・システム障害等のシステム系エラーが発生した場合は、エラー内容を要約して呼び出し元エージェント（AG-001）に返してください。AG-001がユーザーへエスカレーション案内を提示します。
```

##### プロンプト設計の意図

| 要素 | 意図 |
|------|------|
| 一括収集フォーム | 1つずつ質問するよりも会話ターン数を削減し、BRL-23のターン数制限リスクを低減する |
| 駅名正規化 | TOOL-001に正規化済み値を渡すことで経路検索の精度を高める（BRL-15） |
| ドラフト提示分離 | ツール呼び出し前にテキスト提示を行い、HumanApprovalHookとの連携を明確化する |
| dict型引数の明示 | generate_travel_expense_formのitemsフィールドの必須キーをすべて明記し、キー省略によるエラーを防ぐ |
| 申請期限の動的埋め込み | LLMが実行時のapplication_dateを正確に判断できるよう deadline_date を埋め込む |
| 外部ポリシーファイル参照 | バリデーションルールを `agent_knowledge/transportation_policies.py` から読み込むことで、ルール変更時のプロンプト修正を不要にする |

#### 2.3.2 申請ルール

| ルール | 条件 | 対応 |
|--------|------|------|
| BRL-11: 一括収集 | 移動情報収集時 | 収集フォームを提示し全区間を1メッセージで一括収集する |
| BRL-12: 運賃自動計算 | 移動区間入力後 | TOOL-001で運賃を自動計算する。経路テーブル未登録時はユーザーへ手動入力を促す |
| BRL-13: 申請期限チェック | CF-005実行時 | 移動日が deadline_date 以降かチェックする。超過時はCF-008へ遷移 |
| BRL-14: 上長承認要否判定 | CF-005実行時 | 交通費合計が10,000円超の場合にCF-009へ遷移 |
| BRL-15: 駅名正規化 | TOOL-001呼び出し前 | ユーザー入力の駅名を正規形に変換する |
| BRL-20: 業務目的必須 | 業務目的が未入力の場合 | 業務目的の入力を促す。業務目的なしでTOOL-002を呼び出さない |

#### 2.3.3 処理フロー

```
開始（travel_application_agent_tool 呼び出し）
  ↓
ToolContextからsession_id・applicant_name・application_dateを取得
  - session_idはファクトリ関数でFileSessionManagerの初期化に使用
  - applicant_name（min_length=1）・application_date（YYYY-MM-DD形式）を個別バリデーション
    → バリデーション失敗 → エラーメッセージを返す
  ↓
_agent_instances[session_id] が存在しない場合のみ Agent インスタンスを生成
  - build_travel_system_prompt(application_date) でシステムプロンプトを動的生成
  - Agent(callback_handler=None, ...) を生成してキャッシュ
  ↓
agent(query, invocation_state={"applicant_name": applicant_name, "application_date": application_date}) を呼び出す
  ↓
【エージェント内部での処理フロー（システムプロンプトに基づくLLMの動作）】
  1. 移動情報一括収集フォームを提示する
  2. ユーザーから全移動区間の情報を一括受け取る
  3. 各区間について駅名正規化（BRL-15）を実施
  4. 各区間について calculate_travel_expense を呼び出す（BRL-12）
     → success=True: 運賃をユーザーに提示
     → success=False: 手動入力を促す
  5. 追加区間の確認・収集をループ
  6. 業務目的を収集する（BRL-20）
  7. 収集済み申請情報をテキストで整理・提示（TOOL-002は呼び出さない）
  8. OK/修正/キャンセルをユーザーから取得
     → 修正: Step 1へ戻る
     → キャンセル: セッション終了メッセージを返す
     → OK: 次ステップへ
  9. generate_travel_expense_form を呼び出す（HumanApprovalHook（BeforeToolCallEvent）が承認後に実行）
     → success=True: ファイルパスを取得
     → success=False: エラーメッセージをAG-001へ返す
  10. 申請期限チェック（BRL-13）・上長承認要否（BRL-14）・差し戻しリスク（BRL-08）を実施
  11. 申請書ドラフト・チェック結果・最終提示をユーザーへ提示（CF-007）
  ↓
処理完了メッセージを AG-001 へ返す
```

### 2.4 設定・構成

#### 2.4.1 共通設定

| 設定項目 | 設定値 | 備考 |
|---------|--------|------|
| 会話管理 | SlidingWindowConversationManager(window_size=20, should_truncate_results=True, per_turn=False) | 複数区間の移動情報収集・交通費計算・申請書生成・ルールチェックを行うため |
| リトライ戦略 | 標準設定（Strands Agents SDK組み込み） | 6回/4秒/240秒。Amazon Bedrockの一時的エラーへのリトライのみ適用 |
| ループ制御 | LoopControlHook(max_iterations=10) | 暴走防止。収集ループ・ツール呼び出しの正常ステップ数に対して十分な余裕 |
| 承認フック | HumanApprovalHook()（generate_travel_expense_form のみ対象） | TOOL-002（generate_travel_expense_form）のBeforeToolCallEventで利用者確認を取得するため（BRL-06、GRD-016） |
| セッション管理 | FileSessionManager(session_id=session_id, storage_path="data/sessions/") | session_idをinvocation_stateから受け取りファクトリ関数内で初期化する |

#### 2.4.2 利用ツール

| ツールID | ツール名 | 利用目的 | 呼び出し条件 |
|----------|----------|----------|--------------|
| TOOL-001 | `calculate_travel_expense` | 移動区間の運賃を自動計算する | CF-003で各移動区間入力後、駅名正規化後に呼び出す（BRL-12） |
| TOOL-002 | `generate_travel_expense_form` | 交通費精算申請書ExcelドラフトをOutput/に生成する | HumanApprovalHook（BeforeToolCallEvent）承認後の1回のみ（BRL-05、GRD-016） |

> 【注意】`generate_expense_form`（AG-003用）は呼び出さない。`generate_travel_expense_form` のみを使用する

### 2.5 連携設計

#### 2.5.1 Agent as Tools設計

**ツール名**: `travel_application_agent_tool`

**ツール説明**:
```
交通費精算申請フロー全体を実行する。
移動情報の収集・運賃計算・申請書Excelドラフト生成・申請ルールチェック・最終提示を担当する。
```

**ToolContextの使用**:
- `@tool(context=True)` デコレータを使用
- `tool_context.invocation_state` から `session_id`・`applicant_name`・`application_date` を取得
- `session_id` はファクトリ関数 `_get_travel_agent(session_id)` 内で FileSessionManager の初期化に直接使用する
- `applicant_name`・`application_date` をエージェントへの invocation_state として設定する

#### 2.5.2 invocation_stateの受け渡しとAgentインスタンスのキャッシュ

```python
# session_id をキーにして Agent インスタンスをキャッシュし、
# セッション内で会話履歴を保持する（情報の再入力を防ぐ）
_agent_instances: dict[str, Agent] = {}

@tool(context=True)
def travel_application_agent_tool(query: str, tool_context: ToolContext) -> str:
    # AG-001から受け取る
    state = tool_context.invocation_state
    session_id = state.get("session_id", "")
    applicant_name = state.get("applicant_name", "")
    application_date = state.get("application_date", "")

    # 個別バリデーション
    if not applicant_name or not applicant_name.strip():
        return "エラー: 申請者名が設定されていません。"
    if not application_date or not _is_valid_date(application_date):
        return "エラー: 申請日の形式が不正です。"

    # session_id 未登録の場合のみ新しい Agent インスタンスを生成する
    # Agent初期化時にcallback_handler=Noneを指定して標準出力への直接表示を抑制する（AG-001による二重出力を防ぐ）
    if session_id not in _agent_instances:
        _agent_instances[session_id] = _get_travel_agent(session_id, application_date)

    internal_state = {
        "applicant_name": applicant_name,
        "application_date": application_date,
    }
    response = _agent_instances[session_id](query, invocation_state=internal_state)
    return str(response)


def _get_travel_agent(session_id: str, application_date: str) -> Agent:
    session_manager = FileSessionManager(
        session_id=session_id,
        storage_path="data/sessions/"
    )
    system_prompt = build_travel_system_prompt(application_date)
    return Agent(
        agent_id="travel_agent",
        name="交通費精算申請エージェント",
        model=get_model(),
        system_prompt=system_prompt,
        tools=[calculate_travel_expense, generate_travel_expense_form],
        conversation_manager=SlidingWindowConversationManager(
            window_size=20,
            should_truncate_results=True,
            per_turn=False
        ),
        hooks=[HumanApprovalHook(), LoopControlHook(max_iterations=10)],
        session_manager=session_manager,
        callback_handler=None,
    )
```

## 3. 実装詳細

### 3.1 クラス/関数設計

#### 3.1.1 travel_application_agent_tool 関数

##### 説明
AG-001から呼び出されるAgent as Toolsのラッパー関数。invocation_stateからセッション情報を取得し、AG-002エージェントに委任する。

##### 引数
- `query` (str): AG-001から渡される委任メッセージ（必須）
- `tool_context` (ToolContext): AG-001のToolContext（context=Trueにより自動渡し）

##### 戻り値
- `str`: 申請フロー完了メッセージまたはエラーメッセージ

##### 処理内容
1. `tool_context.invocation_state` から `session_id`・`applicant_name`・`application_date` を取得する
2. `applicant_name`（min_length=1）と `application_date`（YYYY-MM-DD形式）を個別バリデーションする
3. `session_id` が `_agent_instances` に未登録の場合のみ `_get_travel_agent(session_id, application_date)` を呼び出してエージェントを生成・キャッシュする
4. `_agent_instances[session_id](query, invocation_state={...})` を呼び出して処理を委任する
5. 結果を文字列で返す

#### 3.1.2 build_travel_system_prompt 関数

##### 説明
申請日と申請期限基準日をシステムプロンプトに動的に埋め込んで返す。外部ポリシーファイル（`agent_knowledge/transportation_policies.py`）を読み込み、バリデーションルール定義をプロンプトに組み込む。

##### 引数
- `application_date` (str): 申請日（YYYY-MM-DD形式、必須）

##### 戻り値
- `str`: 動的情報および外部ポリシールールが埋め込まれたシステムプロンプト全文

##### 処理内容
1. `datetime.date.fromisoformat(application_date)` で申請日を `date` 型に変換する
2. `deadline_date` を `application_date - relativedelta(months=3)` で計算する
3. `os.path.exists("agent_knowledge/transportation_policies.py")` で外部ポリシーファイルの存在を事前チェックする
4. ファイルが存在する場合は `open()` でポリシーファイルを読み込み、ポリシー内容を変数に格納する
5. ファイルが存在しない場合はエラーログを出力し、フォールバック用の空ポリシー文字列を使用する
6. TRAVEL_SYSTEM_PROMPT_TEMPLATE の `{application_date}`・`{deadline_date}`・`{transportation_policies}` を置換して返す

### 3.2 エラーハンドリング

`travel_application_agent_tool` 関数内で捕捉する例外と処理を以下に示す。エラー時の戻り値はすべて `str`（呼び出し元LLMコンテキストに渡されるため）。ログにはクエリの先頭50文字を付加情報として含める。

| エラー種別 | 条件 | ログレベル | 対応 |
|-----------|------|-----------|------|
| `LoopLimitError` | エージェントのループ回数がLoopControlHookの上限を超えた場合 | WARNING | WARNINGレベルでログ出力。エラーメッセージ文字列を返す |
| `ContextWindowOverflowException` | コンテキストウィンドウが溢れた場合 | WARNING | WARNINGレベルでログ出力。エラーメッセージ文字列を返す |
| `MaxTokensReachedException` | 最大トークン数に達した場合 | WARNING | WARNINGレベルでログ出力。エラーメッセージ文字列を返す |
| `RuntimeError` | その他のRuntimeError発生時 | ERROR (exc_info=True) | ERRORレベル（exc_info=True）でログ出力。エラーメッセージ文字列を返す |
| `Exception`（その他） | 上記以外の予期しないエラー発生時 | ERROR (exc_info=True) | ERRORレベル（exc_info=True）でログ出力。エラーメッセージ文字列を返す |

## 5. 補足情報

### 5.1 実装上の注意点

1. **callback_handler=Noneの必須指定**
   - AG-002の `Agent` インスタンスは必ず `callback_handler=None` で初期化する
   - これにより専門エージェント自身の標準出力への直接表示を抑制し、AG-001による二重出力を防ぐ
   - `agent()` 呼び出し時に渡すと `TypeError` になるため必ずコンストラクタで指定する

2. **Agentインスタンスのキャッシュ**
   - `_agent_instances: dict[str, Agent] = {}` をモジュールレベルで定義する
   - ツール関数が呼び出されるたびに `Agent(...)` を新規生成してはならない
   - `session_id` をキーとして初回のみ生成・登録し、2回目以降は再利用する

3. **TOOL-002の呼び出し禁止（承認前）**
   - `generate_travel_expense_form` はHumanApprovalHook（BeforeToolCallEvent）による承認後にのみ呼び出す
   - ドラフト提示（テキスト整理）とツール呼び出しステップを明確に分離する

4. **generate_expense_form の呼び出し禁止**
   - AG-002は `generate_expense_form`（AG-003用）を呼び出してはならない

5. **外部ポリシーファイルの読み込み**
   - `agent_knowledge/transportation_policies.py` の存在チェックは `os.path.exists()` による事前チェックで実装すること
   - `open()` 実行時の `FileNotFoundError` 捕捉との併用は禁止（事前チェックと例外捕捉を二重に行わない）
   - ファイルが存在しない場合はフォールバック処理としてエラーログを出力し、空ポリシー文字列を使用する

---

### 5.2 パフォーマンス考慮事項

1. **一括収集による会話ターン数削減**
   - 移動情報を1つずつ聞かずにフォームで一括収集することで、会話ターン数を削減する（BRL-23対策）

2. **Agentインスタンスのキャッシュ**
   - 同一セッション内での複数呼び出し間で `SlidingWindowConversationManager` の会話履歴を保持する

---

### 5.3 セキュリティ考慮事項

1. **invocation_stateの安全な受け渡し**
   - `session_id`・`applicant_name`・`application_date` はLLMプロンプトには含めずinvocation_state経由で渡す

2. **申請書の自動提出禁止**
   - BRL-09に従い、申請書の提出操作はユーザーのみが実施する
