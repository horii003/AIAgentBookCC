## 1. 概要

### 1.1 コンポーネントの目的
APR-001（社員の申請書生成前確認）通過後に、収集済み申請情報と申請書テンプレートを組み合わせて Excel 形式の申請書（下書き）を自動生成する。LLM 推論では実現できない openpyxl によるファイル生成処理とテンプレートへのセル書き込みを担当する。

### 1.2 主要な責務
- `generate_transport_application`: 移動情報リスト（D-013）を交通費精算申請書テンプレート（DATA-003）の各セルに書き込み、Excel ファイルを生成する
- `generate_expense_application`: 経費明細リスト（D-007）を経費精算申請書テンプレート（DATA-002）の各セルに書き込み、Excel ファイルを生成する
- `applicant_name`, `application_date` は LLM パラメータとして受け取らず、`invocation_state` から取得する
- `{申請者名}_{申請書種別}_{タイムスタンプ}.xlsx` 形式でファイルを `data/output/{session_id}/` ディレクトリに保存する（タイムスタンプはツール内部で生成し LLM 引数として受け取らない）
- エラー発生時は例外を再送出せず `{"success": False, "message": str}` の辞書形式で返却する

## 3. generate_transport_application ツール設計

### 3.1 基本情報

#### ツール名（関数名）
`generate_transport_application`

#### 説明（docstring）
```
交通費精算申請書生成ツール

収集済み移動情報リストから交通費精算申請書（下書き）Excel ファイルを生成する。
申請者名・申請日は invocation_state から取得する（LLM パラメータではない）。
APR-001（社員の申請書生成前確認）通過後に呼び出すこと。

Args:
    collected_items (dict): 収集済み移動情報を含む辞書。以下のキーを持つこと:
        - segments (list[dict]): 移動区間リスト（1件以上必須）。各要素は以下のキーを持つ辞書:
            - travel_date (str): 移動日（YYYY-MM-DD 形式）
            - departure (str): 出発地（正規化済み）
            - destination (str): 目的地（正規化済み）
            - transport_type (str): 交通手段（「電車」「バス」「タクシー」「飛行機」のいずれか）
            - fare (int): 費用（円単位、1以上）
            - business_purpose (str): 業務目的（必須）

Returns:
    dict: 以下のキーを持つ辞書。
        - success (bool): 処理成功フラグ。True=生成成功、False=失敗。
        - file_path (str | None): 生成済みファイルパス（成功時のみ）。失敗時は None。
        - message (str | None): エラーメッセージ（失敗時のみ）。成功時は None。
```

### 3.2 インターフェース設計

#### 3.2.1 入力

**パラメータ定義（LLM から渡されるパラメータ）**:

| 引数名 | 型 | 説明 | デフォルト値 |
|--------|-----|------|--------------|
| collected_items | dict | 収集済み移動情報辞書（segments キーに移動区間リストを含む） | なし（必須） |

**invocation_state から取得する値**:

| キー名 | 型 | 説明 |
|--------|-----|------|
| applicant_name | str | 申請者名 |
| application_date | str | 申請日（YYYY-MM-DD 形式） |
| session_id | str | セッションID（出力ディレクトリ名に使用） |

##### バリデーションルール詳細

| フィールド | ルール | エラーメッセージ | 例 |
|-----------|--------|----------------|-----|
| applicant_name（invocation_state） | min_length=1、空文字禁止 | "申請者名が入力されていません。" | "" → ValidationError |
| application_date（invocation_state） | YYYY-MM-DD 形式の date 型変換 | "申請日の形式が不正です。YYYY-MM-DD 形式で入力してください。" | "2026/05/01" → ValidationError |
| collected_items | dict 型、"segments" キーが必須 | "collected_items に 'segments' キーが存在しません。{'segments': [...]} 形式で指定してください。" | {"items": [...]} → キーガードエラー |
| collected_items.segments | list 型、1件以上必須 | "移動情報が1件も収集されていません。移動情報を少なくとも1件入力してください。" | [] → ValidationError |
| segments[i].travel_date | YYYY-MM-DD 形式 | "移動日の形式が不正です。YYYY-MM-DD 形式で入力してください。（区間{i+1}）" | "2026/05/01" → ValidationError |
| segments[i].departure | min_length=1 | "出発地が入力されていません。（区間{i+1}）" | "" → ValidationError |
| segments[i].destination | min_length=1 | "目的地が入力されていません。（区間{i+1}）" | "" → ValidationError |
| segments[i].transport_type | Literal["電車", "バス", "タクシー", "飛行機"]（正規化後） | "交通手段が不正です。（区間{i+1}）" | "バイク" → ValidationError |
| segments[i].fare | int、gt=0 | "費用は1円以上を入力してください。（区間{i+1}）" | 0 → ValidationError |
| segments[i].business_purpose | min_length=1 | "業務目的が入力されていません。（区間{i+1}）（BRL-13）" | "" → ValidationError |

### 3.3 ビジネスロジック

#### 3.3.1 処理フロー

```
開始
  ↓
invocation_state から applicant_name, application_date, session_id を取得する
  ↓
collected_items に "segments" キーが存在するかチェック（必須キーガード）
  - 存在しない → {"success": False, "message": "collected_items に 'segments' キーが存在しません。{'segments': [...]} 形式で指定してください。"} を返す（終了）
  - 存在する → 処理続行
  ↓
Pydantic TransportApplicationInput モデルで入力バリデーション
  - applicant_name: min_length=1、空白除去（invocation_state から取得）
  - application_date: str→date 変換（YYYY-MM-DD）（invocation_state から取得）
  - items: segments を TransportItem リストとしてバリデーション
    - 各 TransportItem: normalize_station_name, normalize_transport_type, check_application_deadline（BRL-11）
  ↓
[ValidationError 発生？]
  - YES → logger.error でログ記録
          → ErrorHandler.handle_validation_error(e) を呼び出す
          → {"success": False, "message": エラーメッセージ} を返す（終了）
  - NO → 処理続行
  ↓
テンプレートファイルの存在確認（os.path.exists() で事前チェック）
  ↓
[テンプレートファイルが存在しない？]
  - YES → logger.error でログ記録（file_path 付き）
          → ErrorHandler.handle_file_save_error(e) を呼び出す
          → {"success": False, "message": エラーメッセージ} を返す（終了）
  - NO → 処理続行
  ↓
openpyxl でテンプレートを読み込む（data/templates/transport_template.xlsx）
  ↓
[読み込み失敗（Exception）？]
  - YES → logger.error でログ記録（file_path 付き）
          → ErrorHandler.handle_unexpected_error(e) を呼び出す
          → {"success": False, "message": エラーメッセージ} を返す（終了）
  - NO → 処理続行
  ↓
openpyxl でシートを開き、セルに申請情報を書き込む
  - B3 ← applicant_name
  - B4 ← application_date（YYYY-MM-DD 形式）
  - A{7+i} ← No（i+1）
  - B{7+i} ← items[i].travel_date（移動日）
  - C{7+i} ← items[i].departure（出発地）
  - D{7+i} ← items[i].destination（目的地）
  - E{7+i} ← items[i].transport_type（交通手段）
  - F{7+i} ← items[i].fare（費用）
  - G{7+i} ← items[i].business_purpose（業務目的）
  - H{7+i} ← 空欄（承認状況）
  - F{7+n+2} ← sum(items[i].fare for i in range(n))  # 合計金額
  ↓
ファイル名・保存先パスを生成する
  - timestamp = datetime.now().strftime("%Y%m%d%H%M%S")
  - file_name = f"{applicant_name}_交通費精算申請書_{timestamp}.xlsx"
  - output_dir = f"data/output/{session_id}/"
  - file_path = f"{output_dir}{file_name}"
  ↓
出力ディレクトリを作成する（os.makedirs(output_dir, exist_ok=True)）
  ↓
ファイルを保存する（workbook.save(file_path)）
  ↓
[ファイル書き込み失敗？]
  - IOError → logger.error でログ記録（file_path 付き）
              → ErrorHandler.handle_file_save_error(e) を呼び出す
              → (False, エラーメッセージ) タプルを返す（終了）
  - PermissionError → logger.error でログ記録（file_path 付き）
                      → ErrorHandler.handle_file_save_error(e) を呼び出す
                      → (False, エラーメッセージ) タプルを返す（終了）
  - Exception → logger.error でログ記録（file_path 付き）
                → ErrorHandler.handle_file_save_error(e) を呼び出す
                → (False, エラーメッセージ) タプルを返す（終了）
  - NO → 処理続行
  ↓
強化監査ログ（LOG-HI）に記録する
  ↓
{"success": True, "file_path": file_path, "message": None} を返す（終了）
```

#### 3.3.2 データの検証方法

**Pydantic モデル**: `TransportApplicationInput`（agents/models/data_models.py）

**検証内容**:
- `applicant_name`: min_length=1、str_strip_whitespace（invocation_state から取得）
- `application_date`: str→date 型変換（YYYY-MM-DD）（invocation_state から取得）
- `items`: List[TransportItem]（1件以上）
  - TransportItem 内バリデーター: normalize_station_name, normalize_transport_type, check_application_deadline（BRL-11）

### 3.4 エラーハンドリング

#### 3.4.1 エラー処理フロー

```
例外発生
  ↓
必須キーガード（segments キー不在）
  → {"success": False, "message": "collected_items に 'segments' キーが存在しません。{'segments': [...]} 形式で指定してください。"}
  → 返す

ValidationError（入力不正）
  → logger.error でログ記録（exc_info=True）
  → ErrorHandler.handle_validation_error(e)
  → {"success": False, "message": "入力内容を確認してください。{詳細}"}
  → 返す

ファイルが存在しない（os.path.exists() チェック失敗）
  → logger.error でログ記録（file_path 付き）
  → ErrorHandler.handle_file_save_error(e)
  → {"success": False, "message": "申請書ファイルの保存中にエラーが発生しました。管理部門（経理部）にお問い合わせください。"}
  → 返す

IOError（ファイル保存失敗）
  → logger.error でログ記録（file_path 付き, exc_info=True）
  → ErrorHandler.handle_file_save_error(e)
  → (False, "申請書ファイルの保存中にエラーが発生しました。管理部門（経理部）にお問い合わせください。") タプル
  → 返す

PermissionError（ファイル保存失敗）
  → logger.error でログ記録（file_path 付き, exc_info=True）
  → ErrorHandler.handle_file_save_error(e)
  → (False, "申請書ファイルの保存中にエラーが発生しました。管理部門（経理部）にお問い合わせください。") タプル
  → 返す

Exception（想定外）
  → logger.error でログ記録（file_path 付き, exc_info=True）
  → ErrorHandler.handle_unexpected_error(e)
  → {"success": False, "message": "処理できませんでした。管理部門（経理部）にお問い合わせください。"}
  → 返す
```

#### 3.4.2 エラーメッセージの詳細

| エラー種別 | 発生条件 | 対応 | ユーザー向けメッセージ |
|-----------|---------|------|---------------------|
| 必須キーガード | collected_items に "segments" キーが存在しない | 即時エラー返却 | "collected_items に 'segments' キーが存在しません。{'segments': [...]} 形式で指定してください。" |
| ValidationError（入力不正） | applicant_name 空文字、application_date 形式不正、segments 空リスト等 | logger + ErrorHandler.handle_validation_error | "入力内容を確認してください。{詳細}" |
| os.path.exists() 失敗 | テンプレートファイルが存在しない | logger + ErrorHandler.handle_file_save_error | "申請書ファイルの保存中にエラーが発生しました。管理部門（経理部）にお問い合わせください。" |
| IOError（ファイル保存失敗） | data/output/{session_id}/ への書き込み失敗（IOError） | logger + ErrorHandler.handle_file_save_error → タプル返却 | "申請書ファイルの保存中にエラーが発生しました。管理部門（経理部）にお問い合わせください。" |
| PermissionError（ファイル保存失敗） | data/output/{session_id}/ への書き込み失敗（PermissionError） | logger + ErrorHandler.handle_file_save_error → タプル返却 | "申請書ファイルの保存中にエラーが発生しました。管理部門（経理部）にお問い合わせください。" |
| Exception（想定外） | 上記以外の未分類例外 | logger + ErrorHandler.handle_unexpected_error | "処理できませんでした。管理部門（経理部）にお問い合わせください。" |

### 4.3 ビジネスロジック

#### 4.3.1 処理フロー

```
開始
  ↓
invocation_state から applicant_name, application_date, session_id を取得する
  ↓
collected_items に "expenses" キーが存在するかチェック（必須キーガード）
  - 存在しない → {"success": False, "message": "collected_items に 'expenses' キーが存在しません。{'expenses': [...]} 形式で指定してください。"} を返す（終了）
  - 存在する → 処理続行
  ↓
Pydantic ExpenseApplicationInput モデルで入力バリデーション
  - applicant_name: min_length=1、空白除去（invocation_state から取得）
  - application_date: str→date 変換（YYYY-MM-DD）（invocation_state から取得）
  - items: expenses を ExpenseItem リストとしてバリデーション
    - 各 ExpenseItem: check_application_deadline（BRL-11）
  ↓
[ValidationError 発生？]
  - YES → logger.error でログ記録
          → ErrorHandler.handle_validation_error(e) を呼び出す
          → {"success": False, "message": エラーメッセージ} を返す（終了）
  - NO → 処理続行
  ↓
テンプレートファイルの存在確認（os.path.exists() で事前チェック）
  ↓
[テンプレートファイルが存在しない？]
  - YES → logger.error でログ記録（file_path 付き）
          → ErrorHandler.handle_file_save_error(e) を呼び出す
          → {"success": False, "message": エラーメッセージ} を返す（終了）
  - NO → 処理続行
  ↓
openpyxl でテンプレートを読み込む（data/templates/expense_template.xlsx）
  ↓
[読み込み失敗（Exception）？]
  - YES → logger.error でログ記録（file_path 付き）
          → ErrorHandler.handle_unexpected_error(e) を呼び出す
          → {"success": False, "message": エラーメッセージ} を返す（終了）
  - NO → 処理続行
  ↓
openpyxl でシートを開き、セルに申請情報を書き込む
  - B3 ← applicant_name

  ↓

  - timestamp = datetime.now().strftime("%Y%m%d%H%M%S")
  - file_name = f"{applicant_name}_経費精算申請書_{timestamp}.xlsx"

  - file_path = f"{output_dir}{file_name}"
  ↓
出力ディレクトリを作成する（os.makedirs(output_dir, exist_ok=True)）
  ↓
ファイルを保存する（workbook.save(file_path)）
  ↓
[ファイル書き込み失敗？]
  - IOError → logger.error でログ記録（file_path 付き）
              → ErrorHandler.handle_file_save_error(e) を呼び出す
              → (False, エラーメッセージ) タプルを返す（終了）
  - PermissionError → logger.error でログ記録（file_path 付き）

                      → (False, エラーメッセージ) タプルを返す（終了）
  - Exception → logger.error でログ記録（file_path 付き）
                → ErrorHandler.handle_file_save_error(e) を呼び出す
                → (False, エラーメッセージ) タプルを返す（終了）
  - NO → 処理続行
  ↓
強化監査ログ（LOG-HI）に記録する
  ↓
```

## 6. 補足情報

### 6.1 実装上の注意点

1. **applicant_name / application_date の取得方式**
   - `applicant_name`, `application_date` は LLM パラメータとして受け取らず、`invocation_state` から取得する
   - 取得方法: `invocation_state.get("applicant_name")` / `invocation_state.get("application_date")`

2. **os.path.exists() による事前チェック**
   - テンプレートファイルの存在確認は `os.path.exists()` で事前チェックを行う
   - `open()` 時の `FileNotFoundError` 捕捉との併用禁止（`os.path.exists()` チェックのみを使用）

3. **タイムスタンプの内部生成**
   - ファイル名のタイムスタンプは `datetime.now().strftime("%Y%m%d%H%M%S")` でツール内部生成すること
   - LLM の引数（ツール関数のパラメータ）としてファイルパスやタイムスタンプを受け取ってはならない

4. **テンプレートの読み取り専用使用**
   - `openpyxl.load_workbook(template_path, read_only=False)` で読み込み後、`workbook.save(output_path)` で新規ファイルへ書き込む（テンプレートを上書きしない）

5. **generate() 例外クラスの制限**
   - `generate()` 関数内で捕捉する例外クラスは `ValidationError` と `Exception` の2種類のみ
   - `FileNotFoundError` を個別 except で捕捉しない（`os.path.exists()` による事前チェックを使用）

6. **ファイル保存例外クラス**
   - ファイル保存処理では `IOError`, `PermissionError`, `Exception` の3種類を個別 except で捕捉する
   - 各クラスで `(False, エラーメッセージ)` タプルを返却する

7. **出力ディレクトリ**
   - 出力先は `data/output/{session_id}/` 形式（旧 `data/outputs/` から変更）
   - `os.makedirs(output_dir, exist_ok=True)` で存在しない場合は作成する

### 6.2 パフォーマンス考慮事項

1. **テンプレートの毎回読み込み**
   - テンプレートは関数呼び出しごとに読み込む（モジュールロード時キャッシュなし）。ファイルサイズが小規模であれば許容範囲内

### 6.3 セキュリティ考慮事項

1. **テンプレートパスの固定化**
   - テンプレートパスは `data/templates/transport_template.xlsx` / `data/templates/expense_template.xlsx` に固定し、外部から注入できないようにする

  - E = 経費区分（expense_category）
  - F = 金額（amount）
  - G = 業務目的（business_purpose）
  - H = 承認状況（空欄）
- 合計金額セル: `F{7+n+2}`（n=経費件数）
