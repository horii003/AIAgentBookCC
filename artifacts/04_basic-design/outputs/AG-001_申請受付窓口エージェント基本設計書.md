## 2. 目的・役割定義

### 2.1 エージェントの目的
- 申請者名を収集した後、申請内容（自由文）から申請種別（交通費精算申請・経費精算申請）を判定し、対応する専門エージェントへ委任する
- 申請フロー全体のオーケストレーターとして、セッション全体を統括する
- 判定不能時は2択の選択肢を提示してユーザーに確認し、申請種別を確定させる

### 2.2 主な責務
- **申請者名の収集**: 専門エージェント委任前に申請者名を取得し、invocation_stateに含めて渡す
- **申請種別の判定**: ユーザーの申請内容（自由文）から社内申請ルールを参照してBRL-01に基づき申請種別を判定する
- **申請種別の確認（判定不能時）**: BRL-02に基づき2択を提示し、ユーザー選択で申請種別を確定する（CF-002）
- **専門エージェントへの委任**: 申請種別確定後にAG-002またはAG-003をツールとして呼び出す（LNK-001/LNK-002）
- **セッション全体の管理**: invocation_stateにsession_id・applicant_name・application_dateを含めて専門エージェントへ渡す

### 2.3 非責務（やらないこと）
- **申請情報の収集**: AG-002/AG-003が担当
- **申請書の生成・チェック**: AG-002/AG-003が担当
- **交通費計算・運賃算出**: AG-002がTOOL-001を通じて担当
- **申請期限チェック（BRL-13/BRL-18）**: AG-002/AG-003が担当
- **上長承認要否判定（BRL-14/BRL-19）**: AG-002/AG-003が担当

## 3. 自律度・判断権限

### 3.1 自律度レベル

| レベル | 内容 | 適用範囲 |
|------|------|----------|
| 参照 | 社内申請ルール（ナレッジ）を参照して申請種別を判定する | 申請種別判定（BRL-01） |
| 提案 | 申請書・申請先をユーザーに提示する | 申請書・申請先の提示（BRL-03） |
| 実行 | 申請種別確定後に専門エージェントへ委任する | 専門エージェントへの委譲（ACT-EXEC-02） |
| 実行（承認必須） | — | 本エージェントに該当なし |

### 3.2 判断権限の範囲
- **自律判断可能**: 申請種別の自動判定（BRL-01、BRL-02）、専門エージェントへの委任判断（ACT-DEC-01）
- **人間確認必要**: 申請種別が判定不能な場合の2択確認（BRL-02、CF-002）
- **実行不可**: 申請情報収集・申請書生成・申請書提出（BRL-09）

## 4. 処理フロー設計

### 4.1 基本フロー（概要）

```text
開始 → 申請者名収集（CF-001 Step1-2）→ 申請内容受付（Step3）→ 申請種別判定（BRL-01）
→ [判定成功] 申請書・申請先提示（BRL-03）→ AG-002/AG-003委任 → 終了
→ [判定不能] CF-002（2択提示）→ ユーザー選択 → 申請書・申請先提示 → AG-002/AG-003委任 → 終了
```

**フローの詳細**: 詳細設計書 2.3章を参照

### 4.2 フロー制御の考え方
- **原則的な処理順序**: 逐次処理。申請者名収集 → 申請内容受付 → 申請種別判定 → 専門エージェント委任の順序を厳守する
- **処理をスキップする考え方**: 申請種別が自動判定できた場合はCF-002（2択確認）をスキップして直接BRL-03へ進む
- **同一処理を複数回実行**: ユーザーが申請内容を再入力した場合のみBRL-01を再実行する

### 4.3 状態管理
- **申請者名（applicant_name）**: アプリ起動時に収集し、invocation_stateでAG-002/AG-003へ引き渡す（セッション終了まで保持）
- **申請日（application_date）**: システム日付を自動取得（YYYY-MM-DD形式）し、invocation_stateでAG-002/AG-003へ引き渡す
- **申請種別リスト（D-003）**: CF-001のStep 4以降でセッション状態として保持し、CF-003/CF-004へ引き継ぐ

## 5. 専門エージェントとの連携設計

### 5.1 連携対象エージェント

| エージェントID | エージェント名 | 連携目的 | 振り分け条件 |
|---------------|---------------|----------|--------------|
| AG-002 | 交通費精算申請エージェント | 交通費精算申請フロー全体の実行を委譲する | 申請種別 = 交通費精算申請が確定した時点（LNK-001） |
| AG-003 | 経費精算申請エージェント | 経費精算申請フロー全体の実行を委譲する | 申請種別 = 経費精算申請が確定した時点（LNK-002） |

### 5.2 振り分け判断基準

- **AG-002（交通費精算申請）**: 申請内容が「電車代精算」「交通費」等の交通費精算申請に該当すると判断できる場合（BRL-01、JD-01）
- **AG-003（経費精算申請）**: 申請内容が「会議費」「文房具」等の経費精算申請に該当すると判断できる場合（BRL-01、JD-01）
- **判定不能時（CF-002）**: 2択をユーザーに提示し、選択結果で振り分け先を確定する（BRL-02、JD-02）

**詳細な判断基準**: 詳細設計書 2.3.4章を参照

### 5.3 invocation_stateの受け渡し

専門エージェントにsession_id・applicant_name・application_dateを辞書リテラルで渡します。

```python
{
    "session_id": str,       # セッション識別子（UUID v4形式）
    "applicant_name": str,   # 申請者名
    "application_date": str  # 申請日（YYYY-MM-DD形式。システム日付自動取得）
}
```

> ※ application_type（申請種別）はinvocation_stateに含めない。専門エージェントは自身の役割を固定で把握しているため
> ※ invocation_stateは辞書リテラルで渡す。専用のPydanticモデルは定義しない

**データ構造の詳細**: 詳細設計書 2.5章および マルチエージェント連携設計 7.2節を参照

## 6. ツール設計

### 6.0 ツール割り当て方針
- AG-001が使用するツールは、業務機能（計算・生成など）に特化したツールのみとする
- AG-002/AG-003（Agent as Tools）はエージェント委譲のための仕組みであり、業務機能ツールとは区別する

### 6.1 利用ツール

- **AG-002（交通費精算申請エージェント）をツールとして呼び出す**: 交通費精算申請フロー全体を委譲する（`@tool(context=True)` で実装）
- **AG-003（経費精算申請エージェント）をツールとして呼び出す**: 経費精算申請フロー全体を委譲する（`@tool(context=True)` で実装）

申請種別確定後に対応するエージェントツールを1回のみ呼び出す。AG-002とAG-003を同時に呼び出すことはない。

> 【注意】TOOL-001（交通費計算ツール）・TOOL-002（申請書生成ツール）はAG-001の利用ツールではない。AG-002/AG-003が担当する

## 7. プロンプト設計

### 7.1 システムプロンプトの目的
- 申請者名収集後に申請内容（自由文）から申請種別を自動判定する役割を定義する
- 社内申請ルール参照による申請種別判定プロトコル（BRL-01/BRL-02）を指示する
- 判定不能時の2択提示とユーザー選択への対応方針を指示する
- 専門エージェントへの委任タイミングと禁止事項（申請書生成・申請書提出・申請期限チェック）を明示する

### 7.2 プロンプト構成要素
- **役割定義**: 申請受付窓口エージェントとして申請種別判定と専門エージェント委任を担う
- **処理フロー**: 申請者名収集 → 申請内容受付 → 申請種別判定 → 申請書・申請先提示 → 専門エージェント委任
- **判断プロトコル**: 社内申請ルール参照によるBRL-01/BRL-02ベースの判定手順
- **主な責任**: 申請種別の判定と委任のみ。申請書生成・提出・申請期限チェックは担当外
- **動的情報**: なし（本エージェントのシステムプロンプトに日付・期限判断は含まない）

**プロンプト全文**: 詳細設計書 2.3.1章を参照

### 7.3 システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 静的定数 |
| 動的生成の理由 | 不要。申請期限チェック（BRL-13/BRL-18）はAG-002/AG-003が担当するため、AG-001のプロンプトに日付・期限判断は含まれない |
| 動的に埋め込む情報 | なし |

> **判断根拠**: CF-001・CF-002の会話ステップに日付・期限を基準とした判断処理が含まれていないため静的定数を選択した

### 7.4 主要な設計意図
- **申請期限チェックの責務分離**: BRL-01/BRL-02のルールのみを扱い、申請期限（BRL-13/BRL-18）はAG-002/AG-003へ委ねることを明示する
- **2択提示プロトコル**: 判定不能時に「交通費精算申請」「経費精算申請」の2択のみを提示し、他の回答を受け付けないことを指示する
- **委任の単純化**: 申請種別確定後は専門エージェントへ全委任し、AG-001が申請フロー詳細に関与しないことを明示する

**プロンプト全文**: 詳細設計書 2.3.1章を参照

## 10. 入力データ設計

### 10.1 初期化時の入力
- **申請者名（applicant_name）**: 必須。アプリ起動時に対話ループ開始前に取得する。min_length=1（空文字禁止）
- 入力方法: アプリケーション起動時にユーザーから収集し、エージェント初期化パラメータとして渡す
- バリデーション: 空文字は再入力を促す（GRD-001）

### 10.2 対話ループでの入力
- **申請内容（自由文）**: ユーザーが申請したい内容を自由文で入力する
- **特殊コマンド**:
  - 申請種別が判定不能の場合の2択選択: 「交通費精算申請」「経費精算申請」から選択
  - 会話履歴・申請者情報のリセット: `reset` / `リセット` / `最初から`（対話ループの冒頭でチェックし、セッションをリセットして申請者名の収集から再開する）

### 10.3 invocation_stateの入力
※ 本エージェントはオーケストレーターのため invocation_state の受信は行わない（起点エージェント）

## 13. Agent as Tools設計

### 13.1 ツールとしての公開（AG-002をツールとして呼び出す側の設計）

AG-001はオーケストレーターとして、AG-002/AG-003をAgent as Toolsパターンで呼び出す。

- **AG-002のツール関数名**: `travel_application_agent_tool`（仮称。詳細設計で確定）
- **デコレータ**: `@tool(context=True)`
- **入力**: `query (str)`, `tool_context (ToolContext)`
- **出力**: str（AG-002からの応答）

- **AG-003のツール関数名**: `expense_application_agent_tool`（仮称。詳細設計で確定）
- **デコレータ**: `@tool(context=True)`
- **入力**: `query (str)`, `tool_context (ToolContext)`
- **出力**: str（AG-003からの応答）

### 13.2 ToolContextの使用
- invocation_stateに `session_id`・`applicant_name`・`application_date` を辞書リテラルで設定して専門エージェントへ渡す
- 専門エージェントのファクトリ関数（`_get_travel_agent(session_id)`・`_get_expense_agent(session_id)`）をツール関数内から呼び出す
- セッションIDは `invocation_state["session_id"]` から取得し、ファクトリ関数内でFileSessionManagerの初期化に使用する

**詳細**: 詳細設計書 2.5章および マルチエージェント連携設計 7.2節を参照

## 16. 制約事項・前提条件

### 16.1 技術的制約
- AG-002とAG-003の直接連携禁止（循環呼び出し禁止）。最大委譲深度は1（マルチエージェント連携設計 10.1節）
- invocation_stateは辞書リテラルで渡す。専用Pydanticモデルは定義しない（マルチエージェント連携設計 7.2節）
- AG-001にHumanApprovalHookは登録しない（申請書生成はAG-002/AG-003の責務）

### 16.2 業務的制約
- 申請書の自動提出は禁止（BRL-09）。提出操作は利用者のみ実施可能
- 申請期限チェック（BRL-13/BRL-18）はAG-001の責務外。AG-002/AG-003が担当する
- エスカレーション先は要件上未定義

### 16.3 前提条件
- 申請者名はアプリ起動時（対話ループ開始前）に取得済みであること
- 申請日（application_date）はシステム日付をYYYY-MM-DD形式で自動取得すること
- 社内申請ルール（ナレッジ: KN-001）がシステム起動時に参照可能な状態にあること
