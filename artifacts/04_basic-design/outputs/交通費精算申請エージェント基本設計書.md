## 2. 目的・役割定義

### 2.1 エージェントの目的
- AG-001 から委譲された交通費精算申請書作成フロー全体を担当し、社員との対話を通じて移動情報を収集・確認し、最終的に交通費精算申請書（下書き）を生成する
- TOOL-001（交通費計算）を使用して自動計算を試み、計算不可の場合は社員に手動入力を促すことで申請フローを停滞させない
- 生成した申請書に対して自動チェック（EVT-001/FR-010）を実施し、業務ルール違反を申請前に検出する

### 2.2 主な責務
- **移動情報の収集**: CF-004 フローに沿って1区間ずつ移動情報（移動日・出発地・目的地・交通手段・費用・業務目的）を収集する（BRL-10/BRL-02）
- **交通費の自動計算**: 各区間の移動情報収集後に TOOL-001 を呼び出して運賃を自動計算する（BRL-17）
- **高額申請の案内**: 10,000円を超える費用がある場合は事前上長承認を案内する（BRL-12/JD-10）
- **APR-001 承認の取得**: 全区間収集完了後、HumanApprovalHook 経由で社員の申請書生成前確認を取得する（ABAC-001）
- **申請書の生成**: APR-001 承認後に TOOL-002（generate_transport_application）を呼び出して申請書を生成する（BRL-07）
- **申請内容の自動チェック**: 申請書生成後に EVT-001 として申請内容チェック（FR-010）を自動実施する（BRL-03/BRL-16）

### 2.3 非責務（やらないこと）
- **申請者名・申請日の収集**: AG-001 が収集し invocation_state で引き継ぐ
- **申請種別の判断**: AG-001 が実施する（委譲時点で交通費精算申請と確定済み）
- **経費情報の収集**: AG-003 が実施する

## 3. 自律度・判断権限

### 3.1 自律度レベル

| レベル | 内容 | 適用範囲 |
|------|------|----------|
| 参照 | 社内申請ルール（D-001）・運賃データ（DATA-004/005）の参照 | TOOL-001 経由で参照 |
| 提案 | 経路が見つからない場合の手動入力案内（JD-16） | TOOL-001 が計算不可を返した場合 |
| 実行 | 移動情報収集・TOOL-001 呼び出し・申請内容チェック（EVT-001） | 情報収集・交通費計算・チェック |
| 実行（承認必須） | 申請書生成（TOOL-002 呼び出し） | APR-001 承認後のみ（HumanApprovalHook） |

### 3.2 判断権限の範囲
- **自律判断可能**: 1区間ずつの情報収集順序（BRL-10）、TOOL-001 の呼び出しタイミング（CF-004 Step 5）、高額判定（JD-10）、申請期限チェック（BRL-11）
- **人間確認必要**: APR-001（申請書生成前の社員確認）、計算不可時の費用手動入力（JD-16）
- **実行不可**: APR-001 承認なしでの TOOL-002 実行（EX-05）

## 4. 処理フロー設計

### 4.1 基本フロー（概要）

```text
開始（AG-001からの委譲受付）
  → invocation_stateから申請者名・申請日を取得
  → 移動情報収集ループ（BRL-10: 1区間ずつ）
    → 交通手段確認（BRL-17）
    → 出発地・目的地確認（BRL-18正規化）
    → 移動日・業務目的確認（BRL-13）
    → TOOL-001呼び出し（交通費自動計算）
      → 計算成功: 費用を移動情報リストに反映
      → 計算不可: 社員に手動入力を促す（JD-16）
    → 高額判定（10,000円超: BRL-12案内）
    → 申請期限チェック（BRL-11）
    → 次区間の有無確認
  → 全区間収集完了
  → 収集済み申請情報をテキストとして整理・提示（ドラフト確認用。ツール呼び出しなし）
  → APR-001承認取得（HumanApprovalHook / BeforeToolCallEvent）
    → OK: はじめてTOOL-002呼び出し（申請書生成）※承認前にTOOL-002を実行してはならない
    → 修正: 修正対象を再収集してAPR-001再実施
    → キャンセル: セッションをTERMINATEDに遷移
  → 申請書生成完了
  → EVT-001: 申請内容自動チェック（FR-010/BRL-03/BRL-16）
  → 結果をAG-001に返却
  → 終了
```

**フローの詳細**: 詳細設計書 2.3.1章を参照

### 4.2 フロー制御の考え方
- **原則的な処理順序**: 1区間ずつ逐次収集（BRL-10）。区間のまとめ入力は認めない
- **処理をスキップする考え方**: TOOL-001 が計算不可を返した場合、費用入力ステップを手動に切り替えてスキップせずフローを継続する（JD-16）
- **同一処理を複数回実行**: 区間数分だけ移動情報収集ループを繰り返す。区間追加の意思確認を各区間完了後に実施する
- **ドラフト提示とツール呼び出しの分離**: 全区間収集完了後、収集済み申請情報をテキストとして整理・提示する（この時点ではツールを呼び出さない）。その後 APR-001 承認（HumanApprovalHook / BeforeToolCallEvent）を取得し、承認（OK）後にはじめて TOOL-002 を呼び出す。Human-in-the-Loop 承認は1回のみ発生する
- **承認前のツール実行禁止**: APR-001 承認なしで TOOL-002 を実行してはならない（ABAC-001）

### 4.3 状態管理
- **移動情報リスト（D-013）**: `List[TransportItem]` として AG-002 のメモリ内で保持。各区間の収集完了ごとに追加する。session_{id}.json にも反映する
- **invocation_state（applicant_name / application_date）**: ToolContext 経由で AG-001 から受け取り、TOOL-002 入力に使用する
- **APR-001 承認済みフラグ**: session_{id}.json の `approval_granted` フィールドに記録

## 6. ツール設計

### 6.1 利用ツール

**ツール割り当て方針**: AG-002 が使用するツールは業務機能（計算・生成）に特化したツールのみとする。

- **TOOL-001（calculate_transport_fare）**: 各区間の移動情報収集完了後に呼び出す。交通手段・正規化済み出発地・正規化済み目的地・移動日を渡し、運賃を取得する。計算不可の場合は社員に手動入力を促す（JD-16）
- **TOOL-002（generate_transport_application）**: 全区間収集完了・APR-001 承認後に呼び出す。申請者名・申請日・移動情報リストを渡して交通費精算申請書 Excel ファイルを生成する。**承認（OK）後にはじめて呼び出すこと。承認前に呼び出してはならない**

TOOL-001 は情報収集ループ内で区間ごとに呼び出す（CF-004 Step 5）。TOOL-002 は APR-001 承認後に1回のみ呼び出す（CF-006 Step 1）。

**ツールの詳細**: 詳細設計書 2.4.2章および各ツール基本設計書を参照

## 7. プロンプト設計

### 7.1 システムプロンプトの目的
- 交通費精算申請書作成フローの手順（CF-004/CF-006）をエージェントに定義する
- 1区間ずつの収集（BRL-10）・駅名正規化（BRL-18）・申請期限チェック（BRL-11）・高額判定（BRL-12）をプロンプト内で指示する
- 申請書生成後の自動チェック（EVT-001/BRL-03/BRL-16）の手順を定義する

### 7.2 プロンプト構成要素
- **役割定義**: 交通費精算申請書作成専門エージェントとしての役割
- **処理フロー**: CF-004 Step 1〜8 / CF-006 Step 1〜3 の手順
- **判断プロトコル**: JD-10（高額判定）、JD-16（計算不可時の手動入力）、BRL-11（申請期限チェック）
- **主な責任**: BRL-10（1区間ずつ収集）、BRL-13（業務目的必須）、BRL-17（自動計算）、BRL-18（駅名正規化）
- **動的情報（本日の日付）**: 申請期限チェック（BRL-11）で「移動日から90日以内」を判定するため、現在日付を実行時に埋め込む
- **エラー発生時の振る舞い**: システム系エラー（ツールエラー・システム障害等）が発生した場合は、呼び出し元エージェント（AG-001）に要約メッセージを返す

**プロンプト全文**: 詳細設計書 2.3.1章を参照

### 7.3 システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 動的生成関数（呼び出しのたびに関数を実行して生成する） |
| 動的生成の理由 | 申請期限チェック（BRL-11）で移動日から90日以内であることを確認するため、現在日付（date.today()）をプロンプトに埋め込む必要がある |
| 動的に埋め込む情報 | 本日の日付（`date.today().isoformat()`）、申請期限の基準日（90日前の日付） |

### 7.4 主要な設計意図
- **動的生成の採用**: 申請期限チェック（BRL-11）はシステム日付に依存する業務ルールであるため、プロンプトに現在日付を埋め込んで LLM が正確な期限判断を行えるようにする
- **1区間ずつの強制**: BRL-10 に従い、システムプロンプトで「1区間ずつ確認する」ことを明示し、複数区間のまとめ入力を防ぐ
- **駅名正規化の明示**: BRL-18 に従い、TOOL-001 呼び出し前に「駅」を除去することをプロンプトで明示する

## 10. 入力データ設計

### 10.1 初期化時の入力
- **なし**: AG-002 は AG-001 からの委譲を受けて起動するため、特別な初期化パラメータは不要

### 10.2 対話ループでの入力
- **社員の回答（D-006）**: 移動情報収集における各項目の回答（移動日・出発地・目的地・交通手段・業務目的・追加区間の有無等）
- **費用の手動入力**: TOOL-001 が計算不可を返した場合の費用（円単位の整数）

### 10.3 invocation_stateの入力
AG-001 から ToolContext 経由で受け取る情報（`InvocationStateModel` でバリデーション）:
```python
{
    "applicant_name": str,    # 申請者名
    "application_date": date  # 申請日（システム日付）
}
```
> `session_id` はファクトリ関数内でのみ使用し、エージェントの LLM プロンプトには含めない。

## 11. 出力データ設計

### 11.1 出力形式
- **出力先**: AG-001 への返却文字列（Agent as Tools の戻り値）および標準出力（社員への経過表示）
- **出力内容**: 申請書生成完了通知（ファイルパス）と申請内容チェック結果（EVT-001）

### 11.2 出力メッセージ

- 申請書生成完了時: `"交通費精算申請書を生成しました。{ファイルパス} に保存されています。"`

### 11.3 生成ファイル
- **ファイル名**: `{申請者名}_交通費精算申請書_{タイムスタンプ}.xlsx`
- **保存先**: `data/outputs/`
- **内容**: 申請者情報・移動情報リストが記入された交通費精算申請書（下書き）

## 12. バリデーション設計

**システム設計書の方針**: システム設計書 バリデーション方針.md（`artifacts/03_system-design/outputs/バリデーション方針.md`）を参照してください。

### 12.1 バリデーション方針
- **交通手段**: `Literal["電車", "バス", "タクシー", "飛行機"]`。`normalize_transport_type` で正規化後に Literal チェック（BRL-17）
- **駅名（出発地・目的地）**: `normalize_station_name` で末尾「駅」を除去（BRL-18）。min_length=1
- **費用**: `gt=0`（1円以上）。高額判定（10,000円超）は ValidationError でブロックせずエージェントが案内
- **業務目的**: min_length=1（BRL-13）
- **申請期限**: `check_application_deadline`（model_validator）で移動日から90日以内チェック（BRL-11）

**バリデーションの詳細**: 詳細設計書 4章を参照

### 12.2 申請ルール一覧

| ルールID | ルール名 | 条件 | 対応 |
|----------|----------|------|------|
| BRL-02 | 不足情報の対話確認 | 未収集の必須項目が1つ以上存在する | 不足項目を対話で一つずつ確認する |
| BRL-07 | 全必要情報収集後の申請書生成 | 全必須項目収集済み・APR-001 承認完了後のみ | TOOL-002 を呼び出す |
| BRL-10 | 交通費移動情報の一区間ずつ収集 | 移動情報収集時（全区間共通） | 1区間ずつ順次確認する |
| BRL-11 | 申請期限超過検出・警告 | 移動日から申請日まで90日超過の場合 | ErrorHandler 経由で超過を提示し管理部門への相談を案内する |
| BRL-12 | 高額申請時の事前上長承認案内 | 費用が10,000円を超える場合（JD-10） | 事前上長承認取得を確認する旨を社員に提示する |
| BRL-13 | 業務目的記載必須 | 申請書生成前（全区間分） | 業務目的が空の場合は記載を促す |
| BRL-16 | 申請書生成後の自動チェック実行 | 申請書生成後（EVT-001） | 社内申請ルールと照合し結果を提示する |
| BRL-17 | 運賃データからの交通費自動計算 | 各区間の移動情報収集完了後（JD-16） | TOOL-001 を呼び出す。計算不可の場合は手動入力を促す |
| BRL-18 | 駅名正規化 | 電車の出発地・目的地入力時 | 末尾の「駅」を除去してから TOOL-001 に渡す |

## 13. Agent as Tools設計

### 13.1 ツールとしての公開
- **ツール名**: handle_transport_expense_application
- **デコレータ**: `@tool(context=True)`
- **入力**: `query: str`（AG-001 からの指示文字列）、`tool_context: ToolContext`
- **出力**: `str`（申請書生成完了通知・申請内容チェック結果を含む文字列）

### 13.2 ToolContextの使用
- `invocation_state` から `applicant_name`・`application_date` を取得し `InvocationStateModel` でバリデーション
- `invocation_state["session_id"]` をファクトリ関数内で使用して `FileBasedSessionManager` を初期化（session_id は LLM には渡さない）
- セッションファイル（session_{id}.json）から既存の移動情報リストを読み込み（会話中断・再開対応）

**詳細**: 詳細設計書 2.5章を参照

## 14. テスト観点

### 14.1 テスト観点の概要
- **機能テスト**: 移動情報収集（2区間）→ TOOL-001 計算 → 申請情報のドラフト提示（テキスト整理、ツール呼び出しなし）→ APR-001 承認（BeforeToolCallEvent 経由で1回のみ発生）→ TOOL-002 生成 → EVT-001 チェックの一連フローが正常に動作すること
- **Human-in-the-Loop テスト**: ドラフト提示後に APR-001 承認が1回だけ発生し、承認（OK）後にはじめて TOOL-002 が実行されること。承認前に TOOL-002 が実行されないこと
- **異常系テスト**: TOOL-001 が計算不可を返した場合に手動入力フローに切り替わること。申請期限超過（BRL-11）の警告が提示されること。高額（10,000円超）の案内（BRL-12）が提示されること。システム系エラー発生時に AG-001 へ要約メッセージが返却されること
- **性能テスト**: LoopControlHook により10回上限でループが終了すること

**詳細なテストケース**: 詳細設計書 7章を参照

## 16. 制約事項・前提条件

### 16.1 技術的制約
- Strands Agents SDK v1.25.0 を使用し、`@tool(context=True)` デコレータで AG-001 からツールとして呼び出されること
- HumanApprovalHook を必須設定すること（TOOL-002 呼び出し前の APR-001 承認のため）
- LoopControlHook（max 10回）を必須設定すること

### 16.2 業務的制約
- APR-001 承認なしでの TOOL-002 実行は EX-05（実行不可）として処理する（ABAC-001）
- 移動情報収集は1区間ずつ逐次実施する（BRL-10）。複数区間のまとめ入力は認めない
- TOOL-001 が計算不可を返した場合もフローを停止せず社員手動入力で継続する（JD-16）

### 16.3 前提条件
- invocation_state に `applicant_name`（str、min_length=1）と `application_date`（date）が含まれていること
- `data/templates/transport_template.xlsx` がファイルストアに存在すること
- `data/fare/fare_routes.json` と `data/fare/fare_fixed.json` がファイルストアに存在すること
