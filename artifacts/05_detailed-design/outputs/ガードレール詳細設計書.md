## 1. 概要

### 1.1 ガードレールの目的

AIエージェントが申請業務で利用される際の誤操作・誤判断・逸脱・リスクを防止する。入力・出力・判断・実行の各段階にガードレールを設け、Amazon Bedrock Guardrailsとシステム側の実装ガードを組み合わせて実現する。

### 1.2 主要な責務

- **入力フィルタリング**: 有害コンテンツ・プロンプト攻撃・個人情報の入力をブロックする（Amazon Bedrock Guardrails）
- **出力フィルタリング**: 有害コンテンツ・Chain-of-Thoughtの出力をブロックし、機密情報をマスキングする（Amazon Bedrock Guardrails + システム実装）
- **業務ルール保護**: 必須情報未充足・申請期限超過・チェック未実施時の申請書作成を禁止する（システム実装）
- **実行制御**: 社員確認前のファイル出力・申請書提出をエージェントが実行することを防止する（システム実装）

### 1.3 適用範囲

- AG-001（申請受付窓口エージェント）・AG-002（交通費精算申請エージェント）・AG-003（経費精算申請エージェント）
- 対象業務：BIZ-01〜BIZ-04
- 適用フェーズ：入力ガード・ルール検証・出力ガード・実行ガード

## 2. ブロック時のメッセージ

- メッセージは日本語で表示する
- エラーコードや技術的な詳細はユーザーに見せない

### 入力ブロック時
```
その話題は有害または不適切である可能性があるため、情報を提供できません。
安全にお手伝いできる別の内容についてご質問ください。
```

### 出力ブロック時
```
そのような種類のコンテンツは、有害または不適切である可能性があるため生成できません。
他にお手伝いできることがあればお知らせください。
```

## 5. 機密情報ポリシー

### PII（個人識別情報）

| 種別 | 入力有効 | 出力有効 | アクション |
|------|----------|----------|------------|
| NAME（氏名） | false | true | ANONYMIZE |
| EMAIL（メールアドレス） | false | true | BLOCK |
| PHONE（電話番号） | false | true | BLOCK |
| ADDRESS（住所） | false | true | ANONYMIZE |
| CREDIT_DEBIT_CARD_NUMBER（クレジットカード番号） | true | true | BLOCK |
| BANK_ACCOUNT_NUMBER（銀行口座番号） | true | true | BLOCK |

> 入力側で氏名・住所をANONYMIZEしない理由: 申請者名・申請先等の業務情報を含むため、入力フィルタリングの対象外とする。ログへの平文出力は出力ガード（GRD-009）のシステム実装で別途マスキングする。

### カスタム正規表現

なし（本要件では標準PIIポリシーのみ対象）

## 9. システム実装ガード詳細

### 9.1 GRD-001/GRD-012（入力バリデーション）

**実現方式**: エージェント対話ループ内でのPythonコード実装

**判定タイミング**: ユーザー入力受付後・LLMへの渡し前

**処理内容**:
- 空文字・Noneチェック: `if not user_input.strip()` → 再入力を促すメッセージを表示して再入力
- 文字数チェック: `if len(user_input) > 500` → `"入力が500文字を超えています。500文字以内で再入力してください。"` を表示して再入力

### 9.2 GRD-002（空入力・矛盾検知）

**実現方式**: AG-002/AG-003のシステムプロンプトへの記述

**判定タイミング**: 移動情報・経費情報の収集後

**処理内容**:
- システムプロンプトに「収集済み情報との矛盾を検知した場合は矛盾箇所を提示して再確認を促すこと」と記述する
- LLMが判断して対応する

### 9.3 GRD-003/GRD-004（業務ルール違反防止）

**実現方式**: AG-002/AG-003のシステムプロンプト + TOOL-002のバリデーション

**判定タイミング**: TOOL-002（generate_transport_application/generate_expense_application）呼び出し前

**処理内容**:
- 必須情報未充足時: TransportApplicationData/ExpenseApplicationDataのバリデーションがValueErrorを発生させ、ツール関数が`success: False`を返す
- 申請期限超過時: ExpenseApplicationDataの`model_validator`がValueErrorを発生させる（BRL-12）

### 9.4 GRD-010（社員確認前ファイル出力禁止）

**実現方式**: HumanApprovalHook（BeforeToolCallEvent）

**判定タイミング**: TOOL-002の呼び出し直前

**処理内容**: ハンドラー詳細設計書の7.4節を参照

### 9.5 GRD-011（エージェントによる申請書提出禁止）

**実現方式**: AG-001〜AG-003のシステムプロンプトへの禁止事項明記

**処理内容**: システムプロンプトに「申請書の提出はエージェントが実行しない（GRD-011）」と明記する

### 9.6 GRD-013（対話回数上限30回）

**実現方式**: エージェント対話ループ内でのカウント実装

**判定タイミング**: ユーザー入力受付後

**処理内容**:
- セッション開始時にカウンタを0で初期化する
- ユーザー入力を受け取るたびにカウンタをインクリメントする
- `counter >= 30` の場合: `"対話回数の上限（30回）に達しました。担当部署へご連絡ください。"` を表示してCF-006へ遷移する
