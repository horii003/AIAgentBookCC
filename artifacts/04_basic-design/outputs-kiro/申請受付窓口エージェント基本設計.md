## 2. 目的・役割定義

### 2.1 エージェントの目的
- 申請者名を収集した後、申請内容テキストを受け付け、社内申請ルールに基づいて申請種別（交通費精算申請・経費精算申請）を判断する
- 判断結果に基づき、対応する専門エージェント（AG-002・AG-003）へ処理を委譲する
- 判断不能時は2択を提示してユーザーに選択させ、申請種別を確定する

### 2.2 主な責務
- **申請者名収集**: アプリ起動時に取得済みの申請者名を初期化パラメータとして受け取る
- **申請内容受付・意図分類**: 申請内容テキスト（D-001）を受け付け、LLMで意図を分類する
- **申請種別判断**: 社内申請ルール（D-002）を参照して申請種別を判断する（BRL-01、JD-01、JD-02）
- **専門エージェントへの委譲**: 申請種別確定後に AG-002 または AG-003 へ invocation_state を渡して委譲する
- **判断不能時の2択提示**: 申請種別が判断不能の場合（JD-03）、2択を提示してユーザーに選択させる（BRL-04）

### 2.3 非責務（やらないこと）
- **移動情報・経費情報の収集**: AG-002・AG-003 が担当する
- **申請書生成・妥当性チェック**: AG-002・AG-003 が担当する
- **セッション状態の永続化**: セッションマネージャ（SM-001）が担当する

## 3. 自律度・判断権限

### 3.1 自律度レベル

| レベル | 内容 | 適用範囲 |
|------|------|----------|
| 参照 | 社内申請ルール（D-002）の参照 | 申請種別判断時（BRL-01） |
| 提案 | 申請種別の候補提示・2択提示 | 判断不能時（JD-03、BRL-04） |
| 実行 | 申請種別の自動判断・専門エージェントへの委譲 | 申請種別が明確に判断できる場合（JD-01、JD-02） |

### 3.2 判断権限の範囲
- **自律判断可能**: 申請内容から申請種別（交通費精算申請・経費精算申請）を判断すること（JD-01、JD-02）
- **人間確認必要**: 申請種別が判断不能の場合の最終選択（APR-001、JD-03）
- **実行不可**: 申請書生成・申請書提出・経費情報収集（AG-002・AG-003 の責務）

## 4. 処理フロー設計

### 4.1 基本フロー（概要）

```text
開始（申請者名受取）
→ 申請内容テキスト受付（CF-001 Step1）
→ 申請種別判断（CF-001 Step2：BRL-01、JD-01、JD-02）
→ [判断可能] 申請種別・申請書・申請先の提示（CF-001 Step3）
→ [判断不能] 2択提示・ユーザー選択（CF-002：BRL-04、JD-03）
→ 専門エージェントへの委譲（AG-002 または AG-003）
→ 終了
```

**フローの詳細**: 詳細設計書 2.3章を参照

### 4.2 フロー制御の考え方
- **原則的な処理順序**: 逐次処理（申請内容受付 → 判断 → 提示 → 委譲）
- **処理をスキップする考え方**: 申請種別が明確に判断できる場合は CF-002（2択選択）をスキップする
- **同一処理を複数回実行**: 入力が空の場合は再入力を促す（EX-01）

### 4.3 状態管理
- **session_id**: AG-001 がセッション開始時に UUID v4 形式で生成し、セッションマネージャ（SM-001）に登録する
- **application_type**: 申請種別確定後に SM-001 経由で更新する（上書き禁止）
- **turn_count**: 会話ターンごとに SM-001 経由でインクリメントする（上限30回、BRL-16）

## 5. 専門エージェントとの連携設計

### 5.1 連携対象エージェント

| エージェントID | エージェント名 | 連携目的 | 振り分け条件 |
|---------------|---------------|----------|--------------|
| AG-002 | 交通費精算申請エージェント | 交通費精算申請の情報収集・申請書生成・妥当性チェックを委譲する | 申請種別が「交通費精算申請」に確定した場合（JD-01 または CF-002 でユーザーが選択） |
| AG-003 | 経費精算申請エージェント | 経費精算申請の情報収集・申請書生成・妥当性チェックを委譲する | 申請種別が「経費精算申請」に確定した場合（JD-02 または CF-002 でユーザーが選択） |

### 5.2 振り分け判断基準

- **AG-002（交通費精算申請）**: 申請内容が電車・バス・タクシー・飛行機等の交通費精算に該当する場合（JD-01）
- **AG-003（経費精算申請）**: 申請内容が領収書・備品・宿泊費等の経費精算に該当する場合（JD-02）

**詳細な判断基準**: 詳細設計書 2.3章を参照

### 5.3 invocation_stateの受け渡し

専門エージェントに申請者名・申請日・セッションIDを渡します。

```python
# AG-001 → 専門エージェント（ToolContext経由）
invocation_state = {
    "applicant_name": applicant_name,       # アプリ起動時取得済み
    "application_date": application_date,   # システム日付自動取得（YYYY-MM-DD）
    "session_id": session_id,               # AG-001がセッション開始時に生成
}
```

**データ構造の詳細**: マルチエージェント連携設計.md 7.2節を参照

## 7. プロンプト設計

### 7.1 システムプロンプトの目的
- 申請者名収集・申請内容受付・申請種別判断・専門エージェントへの振り分けの役割を定義する
- 社内申請ルール（BRL-01〜BRL-16）に基づく判断プロトコルを定義する
- 申請ルール外の推測・補完を禁止する（BRL-14）

### 7.2 プロンプト構成要素
- **役割定義**: 申請受付窓口エージェントとしての役割（申請者名収集・意図分類・申請種別判断・振り分け）
- **処理フロー**: CF-001（申請種別案内）→ CF-002（2択選択、判断不能時）→ 専門エージェント委譲
- **判断プロトコル**: JD-01（交通費精算申請判断）・JD-02（経費精算申請判断）・JD-03（判断不能時の2択提示）
- **主な責任**: 申請ルール参照義務（BRL-01）・推測補完禁止（BRL-14）・ユーザー入力文字数制限（BRL-15）
- **社内申請ルール**: システムプロンプトに直接埋め込む（RAG不使用）

**プロンプト全文**: 詳細設計書 2.3.1章を参照

### 7.3 システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 静的定数 |
| 動的生成の理由 | 不要（AG-001 は申請種別判断・振り分けのみ担当。日付・期限を基準とした判断処理は含まない） |
| 動的に埋め込む情報 | なし |

### 7.4 主要な設計意図
- **社内申請ルールのシステムプロンプト埋め込み**: RAG を使用せず、申請ルールをシステムプロンプトに直接埋め込む（システム基本情報.md 11.3節）
- **最小権限**: AG-001 は申請種別判断・振り分けのみを担当し、情報収集・申請書生成は専門エージェントに委譲する
- **判断不能時の人間委譲**: 申請種別が判断不能の場合は2択を提示し、最終判断を人（R-EMP）に委ねる（APR-001）

**プロンプト全文**: 詳細設計書 2.3.1章を参照

## 10. 入力データ設計

### 10.1 初期化時の入力
- **applicant_name**: 必須、アプリ起動時（対話ループ開始前）に取得済みの申請者名
- 入力方法: main.py でアプリ起動時に取得し、AG-001 の初期化パラメータとして渡す
- バリデーション: 空文字列・空白のみは不正（バリデーション方針 6.3節）

### 10.2 対話ループでの入力
- **申請内容テキスト（D-001）**: 社員が自然言語で入力する申請内容
- **特殊コマンド**:
  - 空入力: 再入力を促す（EX-01）
  - 500文字超過: 再入力を促す（BRL-15）
  - `reset` / `リセット` / `最初から`: 会話履歴と申請者情報をリセットし、セッションを最初から開始する

## 11. 出力データ設計

### 11.1 出力形式
- **出力先**: CLI（標準出力）
- **出力内容**: 申請種別・申請書名・申請先の提示（D-004）、または2択の提示（CF-002）

### 11.2 出力メッセージ
- 申請種別提示: `"申請種別：[申請種別]、申請書：[申請書名]、申請先：[申請先]"`
- 2択提示: `"申請種別を選択してください：1. 交通費精算申請 / 2. 経費精算申請"`

## 12. バリデーション設計

**システム設計書の方針**: `artifacts/03_system-design/outputs/バリデーション方針.md` を参照してください。

- **申請内容テキスト（D-001）**: 空文字列・空白のみは不正（EX-01）。500文字以内（BRL-15）
- **申請者名**: 空文字列・空白のみは不正（バリデーション方針 6.3節）

### 12.2 申請ルール一覧

| ルールID | ルール名 | 条件 | 対応 |
|----------|----------|------|------|
| BRL-01 | 申請ルール参照義務 | 申請種別判断時 | 社内申請ルール（D-002）をシステムプロンプトから参照して判断する |

## 13. Agent as Tools設計

### 13.1 ツールとしての公開
AG-001 は Agent as Tools パターンのオーケストレーターであり、自身はツールとして公開されない。AG-002・AG-003 を Agent as Tools として呼び出す。

- **AG-002 のツール名**: `transport_application_agent`
- **AG-003 のツール名**: `expense_application_agent`
- **デコレータ**: `@tool(context=True)`
- **入力**: `query (str)`, `tool_context (ToolContext)`
- **出力**: `str`（専門エージェントからの応答）

### 13.2 ToolContextの使用
- invocation_state（applicant_name・application_date・session_id）を ToolContext 経由で専門エージェントへ渡す
- 専門エージェントは ToolContext から invocation_state を取得して処理を開始する

**詳細**: 詳細設計書 2.5章を参照

## 14. テスト観点

### 14.1 テスト観点の概要

## 16. 制約事項・前提条件

### 16.1 技術的制約
- LLMバックエンドは Amazon Bedrock（Claude Sonnet 4.5）に固定
- 社内申請ルールはシステムプロンプト埋め込みのみ（RAG不使用）
- strands-agents v1.25.0 を使用する

### 16.2 業務的制約
- 申請者名はアプリ起動時（対話ループ開始前）に取得済みの値を使用する（対話中の収集禁止）
- 申請種別の最終判断は人（R-EMP）が行う（判断不能時）
- AG-001 は申請書生成・申請書提出を行わない

### 16.3 前提条件
- アプリ起動時に申請者名が取得済みであること
- 社内申請ルールがシステムプロンプトに埋め込まれていること
- AG-002・AG-003 が Agent as Tools として定義されていること
