## 1. 概要

### 1.1 コンポーネントの目的

AG-001から委譲された交通費精算申請フロー全体を自律実行する。移動情報を対話形式で一区間ずつ収集してTOOL-001（calculate_transport_fare）で運賃を自動計算し、全必須情報充足後にTOOL-002a（generate_transport_application）でExcel申請書を生成する。申請書の必須項目・整合性・ルール適合性・上長承認要否をチェックして合格済み申請書を社員に提示する。

### 1.2 主要な責務

- **必須情報収集**: BRL-06の必須項目（移動日・出発地・目的地・交通手段・費用・業務目的）を1区間ずつ対話で収集する。1区間の情報は入力内容を一括で情報提示する
- **妥当性チェックルール適用**: エージェント初期化時に `agent_knowledge/transportation_policies.py` を読み込み、プロンプトに組み込む
- **運賃自動計算**: TOOL-001で区間ごとの運賃を計算し、D-005に補完する（BRL-14/BRL-15）
- **申請期限チェック**: 移動日から申請日まで90日以内であることを確認する（BRL-12/JD-11）
- **申請書ドラフト提示**: 収集済み申請情報をテキストとして整理・提示する（ツール呼び出しなし）
- **申請書自動作成**: HumanApprovalHookのBeforeToolCallEventによるOK確認後にTOOL-002aでExcel申請書を生成する（BRL-02/COND-001/COND-003）
- **申請書チェック**: 必須項目（JD-06/JD-07）・整合性（JD-08/JD-09）・ルール適合性（BRL-03）・上長承認要否（BRL-11）をチェックする

### 2.2 インターフェース設計

#### 2.2.1 入力

**初期化時の入力（ToolContext経由）**:
- `applicant_name`（str, 必須）: AG-001から`invocation_state`経由で受け取る申請者名
- `application_date`（str, 必須）: AG-001から`invocation_state`経由で受け取る申請日（YYYY-MM-DD形式）
- `session_id`（str, 必須）: AG-001から`invocation_state`経由で受け取るセッションID（SessionManager初期化に使用）

**対話ループでの入力**（AG-001から渡される委譲メッセージを初回のqueryとして受け取る）:
- `query` (str): 委譲メッセージ（申請種別：交通費精算申請・申請意図テキスト）
- その後は社員の自然文入力（移動情報・業務目的・確認応答）

**ToolContext経由で受け取る情報**:
```python
{
    "applicant_name": str,      # 申請者名
    "application_date": str,    # 申請日（YYYY-MM-DD形式）
    "session_id": str           # セッションID（SessionManager初期化用）
}
```

#### 2.2.2 出力

**出力先**: CLIの標準出力（社員への案内・確認・チェック結果メッセージ）

**出力メッセージ**:
- 移動情報収集質問: `"以下の情報を一度に入力してください。\n【移動日（YYYY-MM-DD）】\n【出発地】\n【目的地】\n【交通手段（電車/バス/タクシー/飛行機）】\n【業務目的】"`
- 運賃確認: `"渋谷→新宿 電車 200円（電車経路テーブル検索）で記録します。よろしいですか？"`
- 申請書ドラフト提示: `"以下の内容で申請書を作成します。\n【申請者名】{name}\n【申請日】{date}\n【移動区間】...\n よろしければOKを入力してください。"`
- 申請書チェック合格: `"申請書チェックが完了しました。ファイルパス: data/output/{session_id}/APP_...xlsx"`
- 上長承認案内（BRL-11）: `"交通費合計が10,000円を超えているため、上長承認が必要です。上長へご確認ください。"`
- 申請期限超過: `"申請期限（経費発生日から90日以内）を超過しているため、この申請は受け付けられません。担当部署へご相談ください。"`

**戻り値の形式**: `str`（AG-002の最終応答テキスト。AG-001のtransport_agent_toolに返す）

### 2.3 ビジネスロジック

#### 2.3.1 システムプロンプト

##### システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 動的生成関数 |
| 動的生成の理由 | CF-002 Step 4で申請期限チェック（BRL-12）を行うため、実行時の申請日（application_date）と期限の基準日（application_dateの90日前）をLLMに伝える必要がある。また、妥当性チェックルール（`agent_knowledge/transportation_policies.py`）をエージェント初期化時に読み込んでプロンプトに組み込む |

##### 動的に埋め込む情報
- `applicant_name`: invocation_stateから受け取った申請者名
- `application_date`: invocation_stateから受け取った申請日（YYYY-MM-DD形式）
- `deadline_date`: `application_date` の90日前の日付（`application_date - timedelta(days=90)` で計算）
- `transportation_policies`: `agent_knowledge/transportation_policies.py` から読み込んだ妥当性チェックルール

##### プロンプト全文

```
あなたは交通費精算申請を担当する専門エージェントです。

【役割】
交通費精算申請に必要な情報を社員から収集し、申請書を作成してチェックを実施します。

【申請情報】
- 申請者名: {applicant_name}
- 申請日: {application_date}
- 申請期限の基準日: {deadline_date}（この日以降の移動日は申請期限超過となります）

【妥当性チェックルール】
{transportation_policies}

【処理フロー】

1. CF-002: 交通費精算情報収集
   以下の必須項目を移動区間ごとに1区間ずつ対話で収集します（BRL-06）。
   1区間の情報は入力内容を一括で情報提示してください:

   「以下の情報を一度に入力してください。
   【移動日（YYYY-MM-DD）】
   【出発地】（例: 渋谷、東京）
   【目的地】（例: 新宿、品川）
   【交通手段】（電車/バス/タクシー/飛行機）
   【業務目的】（例: 取引先訪問、研修参加）」

   情報収集後、calculate_transport_fareツールで運賃を自動計算して社員に提示します。
   複数区間がある場合は全区間の情報を収集します。

2. CF-002 Step 4: 申請期限チェック（BRL-12）
   移動日が申請期限の基準日（{deadline_date}）より前の場合は申請期限超過です。
   超過している場合:「申請期限（経費発生日から90日以内）を超過しているため、
   この申請は受け付けられません。担当部署へご相談ください。」
   と伝えて申請を停止してください。

3. CF-004 Step 1: 申請書ドラフト提示（ツール呼び出しなし）
   収集済み申請情報をテキストとして整理して社員に提示します。
   この時点ではgenerate_transport_applicationツールを呼び出してはいけません。
   HumanApprovalHookによる承認確認後にのみツールを呼び出します。

4. CF-004 Step 3: 申請書生成（HumanApprovalHookによるOK確認後）
   HumanApprovalHookが申請者の確認（OK/修正/キャンセル）を取得します。
   - OK選択後: generate_transport_applicationツールを呼び出して申請書を生成します
   - 修正選択後: CF-002（情報収集）へ戻ります
   - キャンセル選択後: 申請を停止します

5. CF-005: 申請書チェック
   以下を順番にチェックします:
   a. 必須項目チェック（JD-06/JD-07）: 移動日・出発地・目的地・交通手段・費用・業務目的がすべて揃っているか
   b. 整合性チェック（JD-08/JD-09）: 移動日と申請日の整合性、費用が正の整数か
   c. ルール適合性チェック（BRL-03）: 業務目的が適切か
   d. 上長承認要否チェック（BRL-11）: 交通費合計が10,000円を超える場合は上長承認が必要である旨を案内する

6. 合格済み申請書提示
   チェック合格後、申請書ファイルパスを提示して申請書提出を社員に案内します。
   申請書の提出はエージェントが実行しません（社員のみが実施可能）。

【判断基準】
- 全必須情報充足確認（JD-04）: 移動日・出発地・目的地・交通手段・費用・業務目的がすべて収集済みであること
- 申請期限チェック（JD-11）: 移動日 >= {deadline_date} → 超過
- 上長承認要否（JD-12）: 交通費合計 > 10,000円 → 上長承認案内（BRL-11）

【ツール呼び出しルール】
- calculate_transport_fareは1区間の移動情報収集後に必ず呼び出すこと
- generate_transport_applicationはHumanApprovalHookのOK確認後のみ呼び出すこと
- ドラフト提示時（CF-004 Step 1）にgenerate_transport_applicationを呼び出してはならない

【エラー時の振る舞い】
- ツールエラー・システム障害等が発生した場合は、エラー内容を要約して呼び出し元（AG-001）に返すこと

【禁止事項】
- 申請書提出を実行しないこと（GRD-011）
- 承認前にgenerate_transport_applicationを実行しないこと（COND-001/COND-003）
```

##### プロンプト設計の意図

| 要素 | 意図 |
|------|------|
| 動的日付埋め込み | BRL-12の申請期限チェックでLLMが正確な日付を判断できるようにする |
| 外部ルールファイル組み込み | 妥当性チェックルールを `agent_knowledge/transportation_policies.py` から読み込むことで、ルール変更時にコード修正なしで対応できる |
| 1区間ずつ一括収集指示 | 1区間の情報を一括提示することで会話ターン数を削減する（BRL-23対策） |
| ドラフト提示とツール呼び出しの分離 | HumanApprovalHookのBeforeToolCallEventが発火する前にTOOL-002aが実行されることを防止する |
| ツール呼び出しルール | LLMがドラフト提示時に誤ってgenerate_transport_applicationを呼び出さないよう明示的に禁止する |

#### 2.3.2 申請ルール

| ルール | 条件 | 対応 |
|--------|------|------|
| BRL-06（必須項目） | 移動日・出発地・目的地・交通手段・費用・業務目的が必須 | 全必須項目を1区間ずつ一括収集質問で収集する |
| BRL-12（申請期限） | 移動日から申請日まで90日以内 | 超過時は申請を停止する（JD-11 NG） |
| BRL-14（運賃計算） | 移動情報収集後 | TOOL-001で運賃を自動計算する |
| BRL-15（駅名正規化） | 出発地・目的地に接尾語が含まれる場合 | FareCalculationInputの`field_validator`で自動除去する |
| BRL-02（必須情報充足確認） | 申請書作成前 | JD-04で全必須項目充足を確認してからTOOL-002aを呼び出す |
| BRL-03（申請書チェック義務） | 申請書作成後 | 必須項目・整合性・ルール適合性・上長承認要否を順次チェックする |
| BRL-11（上長承認要否） | 交通費合計 > 10,000円（JD-12） | 上長承認が必要である旨を案内する |

#### 2.3.3 処理フロー

```
開始（AG-001のtransport_agent_toolから呼び出される）
  ↓
invocation_stateからapplicant_name・application_date・session_idを取得
  - applicant_name/application_dateの存在確認とNone/空文字チェック（バリデーション方針 2.2）
  session_idでSessionManagerを初期化
  agent_knowledge/transportation_policies.py を読み込む
  application_dateとdeadline_date（application_date - 90日）でシステムプロンプトを動的生成
  （transportation_policiesの内容もプロンプトに組み込む）
  ↓
エージェントへ初回queryを渡す（委譲メッセージ）
  ↓
CF-002: 交通費精算情報収集ループ
  ↓
  移動情報の一括収集質問を社員に提示する（1区間ずつ）
  （移動日・出発地・目的地・交通手段・業務目的を1メッセージで収集）
  ↓
  社員から入力を受け取る
  ↓
  TOOL-001（calculate_transport_fare）で運賃を計算する
    - departure, destination, transport_type, travel_date を渡す
    - success: True → fareをD-005に補完し、計算方法とともに提示する
    - success: False（電車区間なし）→ 手動入力を促す
    - success: False（バリデーションエラー）→ 再入力を促す
  ↓
  全必須情報充足確認（JD-04）
    充足 → 次ステップへ
    未充足 → 次区間の収集質問を提示してループを繰り返す
  ↓
CF-002 Step 4: 申請期限チェック（BRL-12/JD-11）
  全区間の最も古い移動日 >= deadline_date?
    YES（超過）→ 「申請期限を超過しています...」を提示 → セッションTERMINATED → 終了
    NO（期限内）→ 次ステップへ
  ↓
CF-004 Step 1: 申請書ドラフト提示（テキストのみ）
  収集済み申請情報を整理してテキストとして提示する
  ※ この時点でgenerate_transport_applicationは呼び出さない
  ↓
HumanApprovalHook発火（BeforeToolCallEvent: generate_transport_application呼び出し前）
  OK → 次ステップへ
  修正 → CF-002（情報収集ループ）へ戻る
  キャンセル → セッションTERMINATED → 終了
  ↓
CF-004 Step 3: TOOL-002a（generate_transport_application）実行
  - segments（全区間）, purpose を渡す（applicant_name・application_dateはinvocation_stateから取得）
  - success: True → file_pathとapplication_dataを受け取る
  - success: False → エラーメッセージを提示 → エスカレーション（EX-04/EX-06）
  ↓
CF-005: 申請書チェック
  a. 必須項目チェック（JD-06/JD-07）: application_dataの全項目確認
  b. 整合性チェック（JD-08/JD-09）: 日付整合性・金額正整数確認
  c. ルール適合性チェック（BRL-03）: 業務目的の適切性確認
  d. 上長承認要否チェック（BRL-11）: 交通費合計 > 10,000円 → 上長承認案内
  チェック不合格 → 不備を提示して修正促進（情報収集ループへ戻る場合あり）
  チェック合格 → 次ステップへ
  ↓
CF-005 Step 5: 合格済み申請書提示
  「申請書の確認が完了しました。ファイルパス: {file_path}
  申請書を確認のうえ、上長へ提出してください。」
  ↓
終了（AG-001のtransport_agent_toolに完了を返す）
```

### 2.5 連携設計

#### 2.5.1 Agent as Tools設計

**ツール名**: `transport_agent_tool`

**ツール説明**:
```
交通費精算申請フローを実行します。
AG-001（申請受付窓口エージェント）から委譲メッセージ（申請種別：交通費精算申請・申請意図テキスト）を受け取り、
交通費精算申請に必要な情報収集・運賃計算・申請書作成・チェックを実行します。
```

**ToolContextの使用**:
- `@tool(context=True)` デコレータを使用
- `tool_context.invocation_state` から `applicant_name`・`application_date`・`session_id` を取得する
- `session_id` でSessionManagerを初期化する
- `applicant_name`・`application_date` でシステムプロンプトを動的生成する

#### 2.5.2 invocation_stateの受け渡しとAgentインスタンスのキャッシュ

```python
_agent_instances: dict[str, Agent] = {}

@tool(context=True)
def transport_agent_tool(query: str, tool_context: ToolContext) -> str:
    state = tool_context.invocation_state
    session_id = state.get("session_id", "")
    applicant_name = state.get("applicant_name", "")
    application_date = state.get("application_date", "")

    # session_id 未登録の場合のみ新しい Agent インスタンスを生成する
    if session_id not in _agent_instances:
        _agent_instances[session_id] = Agent(
            model=get_bedrock_model(),
            system_prompt=get_transport_system_prompt(applicant_name, application_date),
            tools=[calculate_transport_fare, generate_transport_application],
            agent_id="transport_agent",
            name="交通費精算申請エージェント",
            conversation_manager=SlidingWindowConversationManager(
                window_size=20,
                should_truncate_results=True,
                per_turn=False
            ),
            hooks=[HumanApprovalHook(), LoopControlHook(max_iterations=10, agent_name="transport_agent")],
            session_manager=SessionManager(
                session_id=session_id,
                storage_path="data/sessions/"
            ),
            guardrail_id=GUARDRAIL_ID,
            guardrail_version=GUARDRAIL_VERSION,
            guardrail_trace="enabled",
            callback_handler=None,
        )

    # invocation_stateはAG-002内部向けに渡す（session_idは除外）
    agent_state = {"applicant_name": applicant_name, "application_date": application_date}
    response = _agent_instances[session_id](query, invocation_state=agent_state)
    return str(response)
```

## 3. 実装詳細

### 3.1 クラス/関数設計

#### 3.1.1 get_transport_system_prompt関数

##### 説明
申請日（application_date）と申請期限基準日（deadline_date）を動的に埋め込んだシステムプロンプトを生成する。妥当性チェックルールを `agent_knowledge/transportation_policies.py` から読み込んでプロンプトに組み込む。

##### 引数
- `applicant_name` (str): 申請者名（必須）
- `application_date` (str): 申請日（YYYY-MM-DD形式）（必須）

##### 戻り値
- `str`: システムプロンプト全文

##### 処理内容
1. `application_date` を `datetime.date.fromisoformat(application_date)` でdate型に変換する
2. `deadline_date = application_date - timedelta(days=90)` で申請期限基準日を計算する
3. `deadline_date.isoformat()` で文字列に変換する
4. `agent_knowledge/transportation_policies.py` を読み込んで `transportation_policies` 変数に格納する
5. `applicant_name`・`application_date`・`deadline_date`・`transportation_policies` をプロンプトテンプレートに埋め込んで返す

#### 3.1.2 transport_agent_tool関数

2.5.2に記載の通り。

## 5. 補足情報

### 5.1 実装上の注意点

1. **ドラフト提示とTOOL-002aの呼び出しを明確に分離する**
   - CF-004 Step 1（ドラフト提示）ではgenerate_transport_applicationを呼び出さない
   - HumanApprovalHookのBeforeToolCallEventがgenerate_transport_application呼び出し前に発火し、OK選択後にはじめてツールが実行される
   - この制御はHumanApprovalHookとシステムプロンプトの両方で担保する

2. **get_transport_system_promptはAgentインスタンス生成時のみ呼び出す**
   - Agentインスタンスがキャッシュされているセッションでは、2回目以降の呼び出しでもシステムプロンプトは初回生成時のものが使用される
   - application_dateは初回生成時に固定される

3. **invocation_stateのsession_idはAgentへ渡さない**
   - SessionManagerの初期化にのみ使用し、`agent(invocation_state=...)` に含めない（マルチエージェント連携設計 7.3準拠）

4. **申請期限チェックはシステムプロンプトとLLM判断で実施する**
   - `deadline_date`をシステムプロンプトに埋め込み、LLMが移動日と比較して判断する
   - 追加のPythonコードによる二重チェックは実施しない

5. **専門エージェントの例外捕捉はerror_messageをstrで返す**
   - transport_agent_tool関数内での例外はすべてErrorHandlerでメッセージを生成してstrとして返す（呼び出し元LLMコンテキストに渡されるため）
   - ログ出力はErrorHandler呼び出し前にツール関数が実施する

6. **妥当性チェックルールは agent_knowledge/transportation_policies.py から読み込む**
   - インライン記述ではなく外部ファイルを参照することで、ルール変更時の保守性を確保する

### 5.2 パフォーマンス考慮事項

1. **1区間ずつ一括収集質問による会話ターン数削減**
   - 1区間の必須項目を一括提示して収集することで、BRL-23（30ターン制限）への消費を抑制する

2. **SlidingWindow(20)による会話履歴管理**
   - 複数区間収集の長い対話でも直近20メッセージを保持し、コンテキストを維持する

### 5.3 セキュリティ考慮事項

1. **ファイル出力前の社員確認（HumanApprovalHook）**
   - GRD-010/COND-001/COND-003に準拠し、申請書ファイル出力前に必ず社員確認を挟む

2. **申請書提出禁止（GRD-011）**
   - システムプロンプトの禁止事項に明示し、エージェントが申請書を提出しないよう制御する
