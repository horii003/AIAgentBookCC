## 1. 目的・適用範囲

### 1.1 目的

システム全体として **「例外とは何か」「どう分類し」「どう収束させるか」** の共通方針を定義する

個々のエージェント設計に **一貫した判断軸** を提供する

### 1.2 適用範囲

- マルチエージェントを含む全システム（AG-001/AG-002/AG-003/TOOL-001/TOOL-002）
- 業務処理、外部連携、UI/UX、運用を含む

### 1.3 本方針で定義しないこと

- エージェント内部の例外検知ロジック
- SDK／ライブラリ依存の例外クラス設計
- 具体的な再試行アルゴリズム（実行制御方針参照）

---

## 2. 例外処理の基本原則（システム共通）

- **例外は必ず収束させる（無限リトライ禁止）**
- **失敗は状態として管理する**
- **例外＝即エラー表示ではない**
- **ユーザーに内部失敗を見せない**
- **後続処理に影響を残さない**


---

## 3. 例外の定義と分類（システム観点）

### 3.1 例外の定義

「通常の業務フローから逸脱し、継続判断が必要な状態」

### 3.2 例外分類

| 分類ID | 分類名 | 説明 | 例 |
|-------|-------|------|---|
| EX-01 | 入力例外 | 入力不正・不足 | 入力文字数超過（500文字超過、FR-012）、申請意図が空欄、入力形式不正 |
| EX-02 | 判断不能 | AIが結論を出せない | 申請種別が交通費・経費のどちらか判断不能（JD-03）、ループ上限（10回）到達後も情報収集完了不可 |
| EX-03 | 業務ルール違反 | 定義済ルール不一致 | 申請期限超過（経費発生日から90日超、BRL-11）、高額申請での事前承認未取得（BRL-12）、対象外経費区分、対象外交通手段 |
| EX-04 | 外部I/F例外 | API障害・タイムアウト | Amazon Bedrock APIエラー・タイムアウト、ファイルストアへのアクセス失敗（テンプレート・運賃データ・ナレッジファイル） |
| EX-05 | 実行不可 | 権限・状態不整合 | APR-001 承認未取得状態での TOOL-002 実行試行（ABAC-001 違反）、AG-002/AG-003 が AG-001 を呼び出そうとした場合（循環呼び出し） |
| EX-06 | データ不整合 | データ形式不正 | fare_routes.json / fare_fixed.json のスキーマ不正、申請書テンプレートのセル定義不整合、session_{id}.json の形式エラー |
| EX-07 | システム障害 | 基盤障害 | ファイルストアへの書き込み失敗（session_{id}.json・監査ログ）、Bedrock API が連続失敗しリトライ上限（6回）に到達 |
| EX-08 | 想定外 | 未分類事象 | 上記 EX-01〜07 に該当しない予期しない例外 |

## 4. 例外検知レベル

### 4.1 検知レベルの考え方

| レベル | 説明 | 検知主体 |
|-------|------|---------|
| 業務レベル | 業務定義上の異常（入力不正・ルール違反・判断不能） | エージェント（AG-001/AG-002/AG-003） |
| 処理レベル | 処理失敗（ツール実行エラー・データ不整合） | ツール（TOOL-001/TOOL-002）、image_reader |
| システムレベル | 基盤・インフラ異常（API 障害・ファイル I/O 失敗） | ErrorHandler |

### 4.2 検知責務

- 検知は **エージェントまたはツール**
- **判断・収束はErrorHandler** が行う


---

## 5. 例外発生時の基本対応フロー

### 5.1 共通対応フロー

1. 例外検知（エージェント/ツール）
2. 例外分類（ErrorHandler）
3. 継続可否判定（ErrorHandler）
4. 処理分岐（再試行／代替／停止）
5. 状態更新（セッション管理）
6. ログ・監査記録（ErrorHandler）

## 7. 例外ハンドリング実装

### 7.1 ErrorHandlerの責務方針

エラー伝播の責務は以下のように分担する。

**各モジュール（エージェント・ツール）の責務**:
- 例外発生箇所でログを出力し、ErrorHandler へ処理を委譲する

**ErrorHandler の責務**:
- 例外分類（EX-01〜EX-08）ごとに継続可否を判定し、対応方針（再試行・代替処理・エスカレーション・中断）を決定する
- ユーザー向けの日本語メッセージを生成し、技術的な詳細を隠蔽する
- 生成したユーザー向けメッセージを呼び出し元に返す
- スタックトレースはログ（logs/agent.log）にのみ記録する

**呼び出し元の責務**:
- ErrorHandler から受け取ったメッセージを戻り値または表示として上位レイヤーへ伝播させる

**実装上の規約**:
- 全コンポーネントは ErrorHandler インスタンスを保持し、エラー処理を委譲する（`self._error_handler = ErrorHandler()` またはモジュールレベルで `_error_handler = ErrorHandler()`）
- ツール関数のエラー返却は辞書形式（`{"success": False, "message": エラーメッセージ}`）で行い、例外を再送出しない
- エラーハンドリングの階層は EventLoopException（`__cause__` で LoopLimitError を判定） → Exception の順でキャッチする

具体的なメソッド設計（メソッド名・引数・戻り値）は基本設計（ハンドラー基本設計.md）で定義する。

> ※ 基本設計書への参照: 04_basic-design/outputs/共通コンポーネント基本設計.md


### 7.2 エラーメッセージ設計方針

**ユーザーフレンドリー**:
- 技術的な詳細を隠蔽
- 具体的な対処方法を提示
- 丁寧な言葉遣い

**情報の充実**:
- エラーの原因を明確に説明
- 複数の対処方法を提示
- 必要に応じてシステム管理部門への連絡を促す

**例**:
```
【入力例外（EX-01）例】
入力内容が500文字を超えています。500文字以内に短縮してから再度入力してください。

【外部I/F例外（EX-04）例】
申請書の作成処理中に一時的なエラーが発生しました。
しばらく待ってから再度お試しください。
繰り返しエラーが発生する場合は、管理部門（経理部）にお問い合わせください。

【業務ルール違反（EX-03）例】
申請期限を超過しています。経費発生日から90日以内に申請が必要です。
申請期限を超過している場合は、管理部門（経理部）にご相談ください。
```

## 8. 例外とエスカレーションの関係

### 8.1 エスカレーション対象例外

| 例外分類 | エスカレーション要否 | 行き先 |
|---------|------------------|-------|
| EX-02 判断不能 | △（繰り返し判断不能の場合のみ） | 管理部門（CF-009：「管理部門（経理部）にお問い合わせください」を提示） |
| EX-04 外部I/F | 要（リトライ失敗時） | 管理部門（CF-009） |
| EX-05 実行不可 | 要 | 管理部門（CF-009） |
| EX-06 データ不整合 | 要 | 管理部門（CF-009） |
| EX-07 システム障害 | 要 | 管理部門（CF-009） |
| EX-08 想定外 | 要 | 管理部門（CF-009） |

### 8.2 エスカレーション判定責務

**判定主体**: ErrorHandler

**判定基準**: リトライ上限到達・代替処理不可・権限違反・データ不整合・システム障害が発生し、セッション内での解決が不可能と判断した場合にエスカレーションする

## 9. セッション状態への反映方針

### 9.1 例外時の状態更新

| 状態 | 説明 | トリガー |
|-----|------|---------|
| ACTIVE | 処理中（エラー発生前） | — |
| WAITING | 人対応待ち（代替処理：ユーザー入力要求） | EX-01/EX-02/EX-03 でユーザーへの再入力・確認を要求した時 |
| TERMINATED | 異常終了 | EX-05/EX-06/EX-07/EX-08 が発生し解決不可能と判定した時。EX-04 でリトライ上限到達の場合も同様 |

### 9.2 再開可否

**再開可能な例外**:
- EX-01（入力例外）：ユーザーが正しい入力を行えばセッション継続可能
- EX-02（判断不能）：ユーザーが申請種別を選択すればセッション継続可能
- EX-03（業務ルール違反）：ユーザーが申請内容を修正すればセッション継続可能
- EX-04（外部I/F例外）：リトライ内で解決した場合のみセッション継続可能

**再開不可な例外**:
- EX-04（リトライ上限到達時）
- EX-05（実行不可）
- EX-06（データ不整合）
- EX-07（システム障害でリトライ上限到達）
- EX-08（想定外）

## 10. UX観点の扱い（概要）

**利用者への通知有無**:
- 全例外でユーザーに日本語メッセージを提示する。技術的な例外詳細は提示しない

**エラーメッセージの抽象度**:
- 業務的な原因（「申請期限超過」「入力文字超過」等）は明示する
- 技術的な詳細（「APIタイムアウト」「ファイルI/Oエラー」等）は隠蔽する

**次の行動を必ず提示**:
- 再入力を促す場合は具体的な修正方法を提示する
- 解決不可能な場合は「管理部門（経理部）にお問い合わせください」を提示する（CF-009）

## 11. ログ・監査方針

### 11.1 例外ログ

| 項目 | 内容 | 出力先 |
|-----|------|-------|
| error_type | エラータイプ（EX-01〜EX-08） | 標準監査ログ（audit_log_std.jsonl） |
| error_message | ユーザー向けメッセージ | 標準監査ログ（audit_log_std.jsonl） |
| context | エラー発生時のコンテキスト情報（セッション ID・エージェント ID 等） | 標準監査ログ（audit_log_std.jsonl） |
| exc_info | スタックトレース | アプリケーションログ（logs/agent.log）のみ。ユーザーには提示しない |
| timestamp | タイムスタンプ（ISO8601） | 標準監査ログ（audit_log_std.jsonl） |


### 11.2 監査観点

**再現可能性**:
- セッション ID・エージェント ID・タイムスタンプを全ログに記録し、例外発生時の操作を再現可能にする

**判断経路の説明可能性**:
- ErrorHandler が例外分類（EX-01〜EX-08）・継続可否判定・対応方針を標準監査ログに記録する

## 12. 設計上の留意点・リスク

**例外過多によるUX低下**:
- 過度な例外チェックによりユーザーへのエラー提示が頻発するリスクがある
- 対策: 業務ルール違反（EX-03）は申請書生成前チェック時にまとめて提示する（FR-010）

**例外の握り潰し**:
- ツール関数内でエラーをログに記録せず握り潰すリスクがある
- 対策: 全ツール関数は ErrorHandler 経由でエラーを処理し、戻り値に `{"success": False}` を含めることを必須とする

**想定外例外の扱い**:
- 未対応の例外が発生した場合に処理が停止するリスクがある
- 対策: 全エージェント・ツールの最外部で EX-08 として捕捉し、セッションを TERMINATED に遷移させる
