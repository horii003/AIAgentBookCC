## 1. 概要

本設計書では、申請AIエージェントシステムの品質を検証するための2つの評価スクリプトを定義する。

| 評価スクリプト | 評価指標 | 評価器クラス | 評価レベル | スコア形式 | 実行方式 |
|---|---|---|---|---|---|
| `eval_tool_selection.py` | MET-001（ツール選択精度・EV-001） | `ToolSelectionAccuracyEvaluator` | TOOL_LEVEL | 二値（1.0/0.0） | シングルターン |
| `eval_goal_success.py` | MET-002（ゴール達成率・EV-002） | `GoalSuccessRateEvaluator` | SESSION_LEVEL | 二値（1.0/0.0） | マルチターン（ActorSimulator） |

**実行コマンド**:
```bash
# プロジェクトルートから実行する
python evals/eval_tool_selection.py
python evals/eval_goal_success.py
```

## 3. 初期設定処理

ファイル先頭のモジュールレベルで以下を順に実行する（両スクリプト共通）。

### 3.1 標準入出力のエンコーディング設定

```python
import sys
if hasattr(sys.stdout, "reconfigure"):
    sys.stdout.reconfigure(encoding="utf-8")
if hasattr(sys.stderr, "reconfigure"):
    sys.stderr.reconfigure(encoding="utf-8")
```

`reconfigure` が存在しない環境（リダイレクト時など）はスキップする。Windows環境での日本語出力の文字化けを防ぐ。

### 3.2 sys.path の設定

```python
import os
sys.path.insert(0, os.path.dirname(os.path.dirname(os.path.abspath(__file__))))
```

`evals/` サブディレクトリから `helpers` モジュールを解決するため、プロジェクトルートを `sys.path` 先頭に追加する。

### 3.3 環境変数の読み込み

```python
from dotenv import load_dotenv
load_dotenv()
```

AWS認証情報・モデル設定などを `.env` ファイルから環境変数として読み込む。

### 3.4 フック設定

```python
from helpers import patch_human_approval_hook
patch_human_approval_hook()
```

`load_dotenv()` の直後、エージェント生成より前に実行すること。エージェント実行中の人間承認待ちを自動化し、評価がブロックされないようにする（GRD-010/COND-003/COND-004の評価環境回避）。

## 6. タスク関数（run_eval_task）

### 6.1 シグネチャ（両スクリプト共通）

```python
def run_eval_task(case: Case) -> dict:
```

| 戻り値キー | 型 | 説明 |
|---|---|---|
| `output` | `str` | エージェントの最終応答文字列 |
| `trajectory` | `Session` | Evaluatorが参照する実行軌跡（全ターン・全ツール呼び出しを含む） |

---

### 6.2 eval_tool_selection.py の処理フロー（シングルターン）

```
run_eval_task(case)
  │
  ├── [1] session_id = case.session_id
  │
  ├── [2] logger.info("=== ケース '%s' 開始 ===", case.name)
  │
  ├── [3] memory_exporter.clear()          ← 前ケーススパンの消去（必須）
  │
  ├── [4] agent = create_reception_agent(session_id)
  │
  ├── [5] state = create_invocation_state(session_id)
  │
  ├── [6] ★ シングルターン実行
  │         response = agent(case.input, invocation_state=state)
  │         output = str(response)
  │
  ├── [7] finished_spans = memory_exporter.get_finished_spans()
  │
  ├── [8] session = StrandsInMemorySessionMapper()
  │               .map_to_session(finished_spans, session_id=session_id)
  │
  └── [9] return {"output": output, "trajectory": session}
```

### 6.3 実行方式の根拠（eval_tool_selection.py）

**シングルターン** を採用する。ツール選択精度（MET-001）は「AG-001が最初の申請意図テキストから正しいサブエージェントを選択するか」を評価する。AG-001の最初のLLM推論でtransport_agent_toolまたはexpense_agent_toolを呼び出すため、1ターンで評価が完結する。

---

### 6.4 eval_goal_success.py の処理フロー（マルチターン）

```
run_eval_task(case)
  │
  ├── [1] session_id = case.session_id
  │
  ├── [2] logger.info("=== ケース '%s' 開始 ===", case.name)
  │
  ├── [3] memory_exporter.clear()          ← 前ケーススパンの消去（必須）
  │
  ├── [4] agent = create_reception_agent(session_id)
  │
  ├── [5] state = create_invocation_state(session_id)
  │
  ├── [6] ★ マルチターン実行（ActorSimulator）
  │         turns = run_actor_conversation(agent, case, invocation_state=state)
  │         output = turns[-1]["agent_response"] if turns else ""
  │
  ├── [7] finished_spans = memory_exporter.get_finished_spans()
  │
  ├── [8] session = StrandsInMemorySessionMapper()
  │               .map_to_session(finished_spans, session_id=session_id)
  │
  └── [9] return {"output": output, "trajectory": session}
```

### 6.5 実行方式の根拠（eval_goal_success.py）

**マルチターン（ActorSimulator）** を採用する。ゴール達成率（MET-002）は「申請フロー全体（CF-001〜CF-005）が完了して申請書が生成・提示されるか」を評価する。情報収集・確認・申請書作成の複数ステップが必要なため、ActorSimulatorがユーザー役を担い動的な会話を継続する。

## 7. メイン処理（main 関数）

### 7.1 eval_tool_selection.py の処理フロー

```
main()
  │
  ├── [1] print("=== ツール選択精度評価 (MET-001) 開始 ===")
  │
  ├── [2] evaluator = ToolSelectionAccuracyEvaluator(model=get_model())
  │
  ├── [3] experiment = Experiment(cases=EVAL_CASES, evaluators=[evaluator])
  │
  ├── [4] reports = experiment.run_evaluations(run_eval_task)
  │
  ├── [5] for report in reports:
  │         print(f"ケース: {report.case.name}")
  │         print(f"  スコア: {report.evaluations[0].score}")
  │         print(f"  根拠: {report.evaluations[0].reasoning}")
  │         # JSONレポートをファイルに保存
  │         with open("evals/logs/eval_tool_selection_report.json", "w", encoding="utf-8") as f:
  │             json.dump(report.to_dict(), f, ensure_ascii=False, indent=2)
  │
  └── [6] print("=== 評価完了 ===")
```

### 7.3 レポート保存設定

| 設定 | 値 | 理由 |
|---|---|---|
| `ensure_ascii` | `False` | 日本語文字列（goal・reasoningの日本語テキスト）をエスケープせずに保存する |
| `indent` | `2` | 人間が読みやすい整形済みJSONにする |
| エンコーディング | `"utf-8"` | 日本語を含むファイルの標準エンコーディング |

## 9. エラーハンドリング・制約事項

### 9.1 エラーハンドリング

両スクリプトとも明示的な `try/except` を `run_eval_task` および `main()` に持たない設計とする。例外はstrands_evalsフレームワークが捕捉してレポートに記録する。エージェント実行中の例外（LLM API障害・LoopControlHookによるループ上限等）はフレームワークに委譲する。

### 9.2 制約事項

| 制約 | 内容 |
|---|---|
| 並列実行禁止 | `memory_exporter`（InMemorySpanExporter）はプロセス内シングルトンのため、並列実行するとスパンが混入して評価結果が不正になる。ケースは必ず逐次実行する |
| レポート上書き | 評価スクリプトを再実行するとJSONレポートは上書きされる。過去の結果を保存する場合はファイル名にタイムスタンプを付与すること |
| HumanApprovalHookモック必須 | `patch_human_approval_hook()` を `load_dotenv()` の直後に呼び出さないと、評価実行中に人間承認待ちでブロックされる |
| memory_exporter.clear() 必須 | ケース実行前に `memory_exporter.clear()` を呼び出さないと前ケースのスパンが混入して評価結果が不正になる |
| 評価目標値未定義 | MET-001/MET-002の合格基準目標値は要件上未定義。実装フェーズ完了後にシステムの実測値を基に決定する |
| LLMコスト | ActorSimulator使用のeval_goal_success.pyは複数ターンのLLM呼び出しが発生するためeval_tool_selection.pyより高コストになる |
