## 1. 目的・適用範囲

### 1.1 目的

システム全体で共通するバリデーション方針を定義し、入力データの整合性と業務ルールへの適合を保証する。

### 1.2 適用範囲

- 全エージェント（AG-001, AG-002, AG-003）の入力処理
- 全ツール（TOOL-001, TOOL-002）のパラメータ受取
- `invocation_state` を経由するエージェント間データ伝播
- セッション状態ファイルへの書き込み前

### 1.3 本方針で定義すること

- バリデーションの基本原則とタイミング
- Pydantic v2+ を用いたモデル設計方針
- データ型別（日付・金額・文字列・列挙型）のバリデーション方針
- 業務ルールに基づくカスタムバリデーションルール
- バリデーションエラーメッセージの方針
- バリデーションテストの観点

### 1.4 本方針で定義しないこと

- 各モデルの具体的なフィールド定義（データモデル基本設計書に委ねる）
- エラーメッセージの全種別一覧（基本設計書・詳細設計書に委ねる）
- UI層での入力バリデーション

## 2. バリデーションの基本原則

### 2.1 基本方針

- **早期検証**: システム境界（ツール入口・エージェント間データ受取）でバリデーションを実施し、不正データの内部伝播を防ぐ
- **型安全性**: Pydantic v2+ による型強制で実行時型エラーを防ぐ
- **ユーザーフレンドリー**: バリデーションエラーは日本語で記述し、社員が次の操作を判断できるメッセージを提示する
- **自動正規化**: 駅名の表記揺れ（BRL-15）・列挙型の別表記など、許容範囲内の表記を自動正規化して後続処理に渡す
- **一貫性**: 全コンポーネントで同一バリデーションロジック（`models/data_models.py`）を使用する

### 2.2 バリデーションのタイミング

| タイミング | 対象 | 実装場所 |
|----------|------|---------|
| ツール呼び出し前 | ツールパラメータ（TOOL-001 の交通手段・経路、TOOL-002 の収集済み申請情報） | Pydanticモデル（`models/data_models.py`） |
| エージェント呼び出し前 | `invocation_state` のフィールド（申請種別・申請者名・入力テキスト） | Pydanticモデル（`models/data_models.py`） |
| データ読み込み後 | 外部ファイル（運賃テーブル・テンプレート）の構造チェック。運賃データ（train_fares.json / fixed_fares.json）はJSONファイルからの静的読み込みであり、起動時にPydanticモデルで構造を検証する | ツール内部バリデーション（Pydanticモデル） |
| 入力受付時 | 社員入力テキスト（ガードレール GRD-001〜GRD-004） | エージェント（AG-001/AG-002/AG-003）のシステムプロンプト |

### 2.3 バリデーション失敗時の扱い

**ValidationError 発生時**:
- `ValidationError` をキャッチし、`ErrorHandler.handle_validation_error(e)` に委譲する
- ErrorHandler が EX-01（入力例外）として分類し、社員へ日本語のエラーメッセージを返す
- バリデーションエラーはログ（LOG-STD）に記録する（スタックトレースは LOG-HI のみ）
- ツール関数は `{"success": False, "message": エラーメッセージ}` を返す（例外を再送出しない）

## 3. Pydanticモデルの使用方針

### 3.1 基本原則

- 全てのデータモデルは `BaseModel` を継承する
- 必須フィールドには型アノテーションを明示する
- オプションフィールドには `Optional` と `None` デフォルト値を使用する
- `model_config = ConfigDict(strict=False)` を基本とし、自動型変換を許容する

### 3.2 Pydanticモデルの定義場所

**定義場所**: `models/data_models.py`（全モデルを一元管理）

**バリデーション関連ハンドラーの配置**: `handlers/` 配下（`handlers/error_handler.py`）に配置する

**バリデーションの責務集中方針**:
- 全入出力データのバリデーションは Pydantic モデル（`models/data_models.py`）と ErrorHandler（`handlers/error_handler.py`）に集中させる
- ツール・エージェントは Pydantic モデルを通じてデータを受け取り、ValidationError は ErrorHandler に委譲する

**命名規則**:
- ツール入力モデル: `{ToolName}Input`（例: `TransportToolInput`, `OutputGeneratorInput`）
- エージェント状態モデル: `{AgentName}State`（例: `OrchestratorState`, `SessionContext`）
- マスタデータモデル: `{DataName}Master`（例: `TrainFareMaster`, `FixedFareMaster`）

**モデルカテゴリと用途**:

| カテゴリ | 用途 | 利用先 |
|---------|------|-------|
| マスタデータモデル | `data/` 配下の JSON ファイルの型保証 | `tools/transport_tools.py` |
| ツール入力モデル | ツール関数の入力パラメータの型保証 | `tools/transport_tools.py`, `tools/output_generator.py` |
| エージェント状態モデル | `invocation_state` の型保証 | `agents/`, `tools/` |
| 出力生成モデル | 申請書生成時のデータ型保証 | `tools/output_generator.py` |

### 3.3 Pydanticモデルの基本構造

```python
from pydantic import BaseModel, field_validator, Field
from typing import Optional, Literal
from datetime import date

class TransportToolInput(BaseModel):
    transportation_type: Literal["電車", "バス", "タクシー", "飛行機"]
    departure: str = Field(..., min_length=1, max_length=100)
    destination: str = Field(..., min_length=1, max_length=100)
    travel_date: date
    purpose: str = Field(..., min_length=1)

    @field_validator("departure", "destination", mode="before")
    @classmethod
    def normalize_station_name(cls, v: str) -> str:
        # BRL-15: 駅名の表記揺れ正規化
        return v.replace("駅", "").replace("Station", "").strip()
```

## 4. 日付のバリデーション

### 4.1 基本原則

- **標準形式**: `YYYY-MM-DD`（ISO 8601）を標準形式とする
- **自動変換**: 文字列形式の日付は `date` 型へ自動変換する
- **範囲チェック**: 未来日付・申請期限（90日）の業務ルールチェックを実施する

### 4.2 日付の定義

**標準形式**: `datetime.date`（Python 標準型）

**許容値**: `date.min` 以上 `date.today()` 以下（未来日付は原則不可。経費発生日として当日まで許容）

### 4.3 日付バリデーションの実装

```python
from datetime import date, timedelta
from pydantic import field_validator

@field_validator("expense_date", mode="before")
@classmethod
def validate_expense_date(cls, v) -> date:
    if isinstance(v, str):
        try:
            v = date.fromisoformat(v)
        except ValueError:
            raise ValueError("日付はYYYY-MM-DD形式で入力してください")
    if v > date.today():
        raise ValueError("経費発生日は本日以前の日付を入力してください")
    return v
```

### 4.4 日付の特殊ルール

**申請期限チェック（BRL-14）**:
- 申請ルール: 経費発生日から90日以内
- 判定: `(date.today() - expense_date).days > 90` の場合は EX-03（業務ルール違反）として処理
- 実装場所: ツール入力モデルの `model_validator` または エージェントのシステムプロンプト

```python
from pydantic import model_validator

@model_validator(mode="after")
def check_application_deadline(self) -> "TransportToolInput":
    if (date.today() - self.travel_date).days > 90:
        raise ValueError(
            "申請期限（経費発生日から90日以内）を超過しています。"
            "担当部門にご確認ください。"
        )
    return self
```

## 5. 金額のバリデーション

### 5.1 基本原則

- **数値型**: `int` 型（円単位・小数点なし）を標準とする
- **上限チェック**: 業務ルール（BRL-10）に基づく高額申請通知ラインをチェックする
- **非負制約**: 金額は0以上の整数に制限する

### 5.2 金額の定義

**標準形式**: `int`（非負整数、円単位）

**許容値**: `0` 以上（上限は要件上未定義、実運用は業務ルール通知ラインで対応）

### 5.3 金額バリデーションの実装

```python
from pydantic import field_validator, Field

amount: int = Field(..., ge=0, description="申請金額（円）")

@field_validator("amount", mode="before")
@classmethod
def validate_amount(cls, v) -> int:
    if isinstance(v, str):
        v = v.replace("円", "").replace(",", "").strip()
        try:
            v = int(v)
        except ValueError:
            raise ValueError("金額は数値（円単位）で入力してください")
    if v < 0:
        raise ValueError("金額は0円以上の値を入力してください")
    return v
```

### 5.4 金額の特殊ルール

**高額申請通知ライン（BRL-10）**:
- 交通費精算申請: 10,000円超で上長承認要否を通知
- 経費精算申請: 5,000円超で上長承認要否を通知
- バリデーションエラーではなく、通知として処理する（申請自体は受け付ける）

## 6. 文字列のバリデーション

### 6.1 基本原則

- **必須チェック**: 業務上必須のフィールドは空文字・空白のみを禁止する（GRD-001）
- **長さ制限**: 申請内容テキストは500文字以内（GRD-004）
- **トリミング**: 前後の空白を自動除去する

### 6.2 文字列の定義

**標準形式**: `str`（Unicode）

**許容値**: 業務フィールドごとに異なる（各モデルの `Field` で定義）

### 6.3 文字列バリデーションの実装

```python
from pydantic import field_validator, Field

# 申請内容テキスト（GRD-001, GRD-004）
application_text: str = Field(..., min_length=1, max_length=500)

# 申請者名（必須）
applicant_name: str = Field(..., min_length=1, max_length=50)

# 業務目的（BRL-18 必須）
purpose: str = Field(..., min_length=1)

@field_validator("applicant_name", "purpose", "application_text", mode="before")
@classmethod
def strip_and_validate_not_empty(cls, v: str) -> str:
    v = v.strip()
    if not v:
        raise ValueError("必須項目が入力されていません。入力してください")
    return v
```

### 6.4 文字列の特殊ルール

**申請内容テキストの長さ制限（GRD-004）**:
- 500文字を超える場合は EX-01（入力例外）として処理
- エラーメッセージ例: 「申請内容は500文字以内で入力してください（現在: {文字数}文字）」

## 7. 列挙型のバリデーション

### 7.1 基本原則

- **Literal 型または Enum で定義**: 受け入れ可能な値を明示的に制限する
- **正規化**: LLM が渡す可能性のある別表記（略称・漢字表記など）を正規化して統一表記に変換する
- **正規化後の値を使用**: 入力値をそのまま使用せず、必ず正規化後の値を後続処理に渡す

### 7.2 列挙型の定義

| フィールド | 許容値 | 正規化対象の別表記 |
|----------|-------|----------------|
| 申請種別（application_type） | `"交通費精算申請"` / `"経費精算申請"` | `"交通費"`, `"交通費申請"`, `"経費"`, `"経費申請"` 等 |
| 交通手段（transportation_type） | `"電車"` / `"バス"` / `"タクシー"` / `"飛行機"` | `"JR"`, `"新幹線"`, `"地下鉄"`, `"ハイヤー"`, `"飛機"` 等 |
| 経費区分（expense_category） | `"事務用品費"` / `"宿泊費"` / `"資格精算費"` / `"その他経費"` | `"事務"`, `"文具"`, `"宿泊"`, `"資格"` 等 |

### 7.3 列挙型バリデーションの実装

**申請種別の正規化**:
```python
from typing import Literal
from pydantic import field_validator

application_type: Literal["交通費精算申請", "経費精算申請"]

@field_validator("application_type", mode="before")
@classmethod
def normalize_application_type(cls, v: str) -> str:
    mapping = {
        "交通費": "交通費精算申請",
        "交通費申請": "交通費精算申請",
        "交通費精算": "交通費精算申請",
        "経費": "経費精算申請",
        "経費申請": "経費精算申請",
        "経費精算": "経費精算申請",
    }
    normalized = mapping.get(v, v)
    if normalized not in ("交通費精算申請", "経費精算申請"):
        raise ValueError(
            f"申請種別が不正です: '{v}'。"
            "'交通費精算申請'または'経費精算申請'を選択してください"
        )
    return normalized
```

**交通手段の正規化（BRL-12）**:
```python
transportation_type: Literal["電車", "バス", "タクシー", "飛行機"]

@field_validator("transportation_type", mode="before")
@classmethod
def normalize_transportation_type(cls, v: str) -> str:
    mapping = {
        "JR": "電車", "新幹線": "電車", "地下鉄": "電車", "モノレール": "電車",
        "ハイヤー": "タクシー", "タクシー代": "タクシー",
        "飛機": "飛行機", "航空": "飛行機", "ANA": "飛行機", "JAL": "飛行機",
        "路線バス": "バス",
    }
    normalized = mapping.get(v, v)
    if normalized not in ("電車", "バス", "タクシー", "飛行機"):
        raise ValueError(
            f"対応していない交通手段です: '{v}'。"
            "電車・バス・タクシー・飛行機のいずれかを入力してください"
        )
    return normalized
```

## 8. カスタムバリデーションルール

### 8.1 申請期限チェック（BRL-14）

- 対象: 移動日（交通費）・経費発生日（経費）
- ルール: 経費発生日から申請日（本日）までの差が90日を超える場合はエラー
- 実装: `model_validator(mode="after")` でモデル全体チェック（セクション4.4参照）

### 8.2 必須項目充足チェック（BRL-04）

- 対象: TOOL-002 実行前の収集済み申請情報（D-010）
- ルール: 必須項目リスト（D-008）の全フィールドが入力済みであること
- 実装: `model_validator(mode="after")` で None フィールドをチェック

```python
from pydantic import model_validator

@model_validator(mode="after")
def check_required_fields_complete(self) -> "CollectedInfoModel":
    required_fields = ["travel_date", "departure", "destination",
                       "transportation_type", "amount", "purpose"]
    missing = [f for f in required_fields if getattr(self, f, None) is None]
    if missing:
        raise ValueError(
            f"以下の必須項目が未入力です: {', '.join(missing)}"
        )
    return self
```

### 8.3 高額申請通知（BRL-10）

- 対象: 申請金額（収集済み申請情報）
- ルール: 交通費10,000円超・経費5,000円超で通知フラグを立てる
- 実装: バリデーションエラーではなく `requires_approval` フラグとして返却

### 8.4 駅名正規化（BRL-15）

- 対象: 電車の出発地・目的地（`departure`, `destination`）
- ルール: 「駅」「Station」等の接尾語を除去して正規化する
- 実装: `field_validator(mode="before")` で入力前処理（セクション3.3参照）

## 9. バリデーションエラーメッセージ

### 9.1 基本原則

- **わかりやすさ**: 業務的な言葉で記述し、技術用語・エラーコードは含めない
- **具体性**: 何が問題で、どう修正すればよいかを明示する
- **一貫性**: 全バリデーションエラーは統一フォーマットで記述する

### 9.2 エラーメッセージの形式

**基本形式**:
```
{フィールド名}のエラー: {エラー内容}
対処方法: {対処方法}
```

**エラーメッセージ例**:

| バリデーション | エラーメッセージ例 |
|------------|----------------|
| 申請内容テキスト空入力（GRD-001） | 「申請内容が入力されていません。申請内容を入力してください」 |
| 申請内容500文字超（GRD-004） | 「申請内容は500文字以内で入力してください（現在: {文字数}文字）」 |
| 申請期限90日超過（BRL-14） | 「申請期限（経費発生日から90日以内）を超過しています。担当部門にご確認ください」 |
| 対応外交通手段（BRL-12） | 「対応していない交通手段です。電車・バス・タクシー・飛行機のいずれかを入力してください」 |
| 金額が0未満 | 「金額は0円以上の値を入力してください」 |
| 必須項目未入力 | 「{項目名}が入力されていません。{項目名}を入力してください」 |

### 9.3 バリデーションエラーメッセージの実装

```python
from pydantic import ValidationError

try:
    model = TransportToolInput(**data)
except ValidationError as e:
    error_message = error_handler.handle_validation_error(e)
    return {"success": False, "message": error_message}
```

## 10. バリデーションのテスト

### 10.1 テスト観点

- **正常系**: 正常な入力値でモデルが正常に生成されること
- **異常系**: バリデーションルール違反時に `ValidationError` が発生すること
- **境界値**: 境界値（500文字ちょうど・91日・0円等）で正しく動作すること
- **正規化**: 別表記・略称が正規化されて統一表記に変換されること
- **エラーメッセージ**: 日本語のエラーメッセージが正しく出力されること

### 10.2 テストケースの例

| テストケース | 入力 | 期待結果 |
|------------|------|---------|
| 正常: 申請種別（正式表記） | `"交通費精算申請"` | `"交通費精算申請"` として格納 |
| 正規化: 申請種別（別表記） | `"交通費"` | `"交通費精算申請"` に正規化 |
| 異常: 申請種別（対象外） | `"住宅手当申請"` | ValidationError（申請種別不正） |
| 正常: 日付（ISO形式） | `"2026-04-01"` | `date(2026, 4, 1)` として格納 |
| 異常: 日付（未来日） | `"2030-01-01"` | ValidationError（未来日付） |
| 境界値: 申請期限（90日ちょうど） | 90日前の日付 | 正常に格納 |
| 境界値: 申請期限（91日超過） | 91日前の日付 | ValidationError（期限超過） |
| 正常: 金額（0円） | `0` | `0` として格納 |
| 異常: 金額（負数） | `-100` | ValidationError（0円以上） |
| 正常: 申請内容テキスト（500文字） | 500文字の文字列 | 正常に格納 |
| 異常: 申請内容テキスト（501文字） | 501文字の文字列 | ValidationError（文字数超過） |
| 正規化: 駅名（接尾語あり） | `"渋谷駅"` | `"渋谷"` に正規化 |
| 異常: 交通手段（対象外） | `"自動車"` | ValidationError（対応外交通手段） |

**テスト設計書**: `tests/unit/test_data_models.py`（マーカー: `@pytest.mark.unit`）

## 11. 設計上の留意点・リスク

### 11.1 LLM出力の表記揺れ

**リスク**: LLM が生成したデータが Pydantic モデルの Literal 型定義と一致しない（表記揺れ・別表記）ケースが発生する可能性がある

**対策**: 主要な列挙型フィールド（申請種別・交通手段・経費区分）に `field_validator(mode="before")` で正規化マッピングを定義する。正規化後も一致しない場合のみ ValidationError を発生させる

### 11.2 バリデーションとガードレールの二重チェック

**リスク**: Pydantic バリデーションとガードレール（GRD-001〜GRD-015）が重複して同じ入力チェックを行い、メンテナンスコストが増加する可能性がある

**対策**: 入力バリデーション（型・形式・長さ）は Pydantic モデルで、業務ルール判断（申請種別判定・経費区分判断）はエージェントのシステムプロンプト（ガードレール）で分担する
