## 1. 概要

### 1.1 コンポーネントの目的
移動区間情報（交通手段・出発地・目的地・移動日）と運賃データファイル（`/data/train_routes.json` / `/data/fixed_fares.json`）を照合し、交通費を自動計算して AG-002（交通費精算申請エージェント）へ返却する。エージェントの LLM 推論では代替できない外部ファイルアクセスと演算処理を担当する。

### 1.2 主要な責務
- `/data/train_routes.json` から出発地・目的地に対応する電車運賃をリスト線形探索で検索して返却する
- `/data/fixed_fares.json` から交通手段（バス・タクシー・飛行機）に対応する固定運賃を返却する
- 対応する経路・交通手段のデータが存在しない場合に自動計算不可フラグ（calculable=False）を返却し、例外ではなく正常処理として扱う
- `/data/train_routes.json` / `/data/fixed_fares.json` の読み込みとスキーマ検証（Pydantic モデルにより実施）を行う
- エラー発生時は例外を再送出せず `{"success": False, "message": str}` の辞書形式で返却する

## 3. calculate_transport_fare ツール設計

### 3.1 基本情報

#### ツール名（関数名）
`calculate_transport_fare`

#### 説明（docstring）
```
交通費計算ツール

移動区間情報から交通費を自動計算する。
電車は /data/train_routes.json の経路テーブルをリスト線形探索で検索し、
バス・タクシー・飛行機は /data/fixed_fares.json の固定運賃を返却する。

Args:
    transport_type (str): 交通手段。許容値: 「電車」「バス」「タクシー」「飛行機」。
        英語・略称（train, bus, taxi, cab, airplane, plane）は自動正規化される。
    departure (str): 出発地。電車の場合は駅名（末尾「駅」は自動除去）。
    destination (str): 目的地。電車の場合は駅名（末尾「駅」は自動除去）。
    travel_date (str): 移動日。YYYY-MM-DD 形式の文字列。

Returns:
    dict: 以下のキーを持つ辞書。
        - calculable (bool): 自動計算可否フラグ。True=計算成功、False=計算不可（経路未存在等）。
        - fare (int | None): 計算済み運賃（円単位）。calculable=False の場合は None。
        - success (bool): 処理成功フラグ。エラー発生時は False。
        - message (str | None): エラー時のメッセージ。成功時は None。
```

### 3.2 インターフェース設計

#### 3.2.1 入力

**パラメータ定義**:

| 引数名 | 型 | 説明 | デフォルト値 |
|--------|-----|------|--------------|
| transport_type | str | 交通手段。許容値: 「電車」「バス」「タクシー」「飛行機」（英語・略称を自動正規化） | なし（必須） |
| departure | str | 出発地。電車の場合は駅名（末尾「駅」を自動除去） | なし（必須） |
| destination | str | 目的地。電車の場合は駅名（末尾「駅」を自動除去） | なし（必須） |
| travel_date | str | 移動日（YYYY-MM-DD 形式） | なし（必須） |

##### バリデーションルール詳細

| フィールド | ルール | エラーメッセージ | 例 |
|-----------|--------|----------------|-----|
| transport_type | normalize_transport_type で正規化後、Literal["電車", "バス", "タクシー", "飛行機"] チェック | "交通手段が不正です。「電車」「バス」「タクシー」「飛行機」のいずれかを選択してください。" | "バイク" → ValidationError |
| departure | min_length=1、空文字禁止 | "出発地が入力されていません。出発地を入力してください。" | "" → ValidationError |
| destination | min_length=1、空文字禁止 | "目的地が入力されていません。目的地を入力してください。" | "" → ValidationError |
| travel_date | YYYY-MM-DD 形式の date 型変換 | "移動日の形式が不正です。YYYY-MM-DD 形式で入力してください。" | "2026/05/01" → ValidationError |

##### 正規化処理

| 入力値 | 正規化後 | 処理内容 |
|--------|---------|---------|
| "渋谷駅" | "渋谷" | normalize_station_name: 末尾の「駅」を除去（BRL-18） |
| "新宿駅" | "新宿" | normalize_station_name: 末尾の「駅」を除去（BRL-18） |
| "train" | "電車" | normalize_transport_type: 英語→日本語マッピング |
| "鉄道" | "電車" | normalize_transport_type: 別表記→正規値マッピング |
| "bus" | "バス" | normalize_transport_type: 英語→日本語マッピング |
| "taxi" / "cab" | "タクシー" | normalize_transport_type: 英語→日本語マッピング |
| "airplane" / "plane" | "飛行機" | normalize_transport_type: 英語→日本語マッピング |

#### 3.2.2 出力

**戻り値の型**: `dict`

**説明**: 計算結果を辞書形式で返却する。例外は再送出せず、エラー時は success=False の辞書を返す。

**構造（正常・計算成功時）**:
```python
{
    "calculable": True,   # bool: 自動計算成功
    "fare": 250,          # int: 計算済み運賃（円単位）
    "success": True,      # bool: 処理成功
    "message": None       # None: エラーなし
}
```

**構造（正常・計算不可時）**:
```python
{
    "calculable": False,  # bool: 経路データ未存在等
    "fare": None,         # None: 計算不可
    "success": True,      # bool: 処理自体は成功
    "message": None       # None: エラーなし
}
```

**構造（エラー時）**:
```python
{
    "calculable": False,  # bool: 計算不可
    "fare": None,         # None: 計算不可
    "success": False,     # bool: 処理失敗
    "message": "str"      # str: ユーザー向けエラーメッセージ
}
```

#### 3.2.3 例外

| 例外 | 発生条件 |
|------|---------|
| 例外を再送出しない | ツール関数内で発生した全例外を dict 形式で返却する（例外処理方針.md 7.1準拠） |

#### 3.3.2 データの検証方法

**Pydantic モデル**: `TransportFareInput`（agents/models/data_models.py）

**検証内容**:
- `transport_type`: normalize_transport_type（mode="before"）→ Literal["電車", "バス", "タクシー", "飛行機"] チェック
- `departure` / `destination`: normalize_station_name → min_length=1 チェック
- `travel_date`: str → date 型変換（YYYY-MM-DD）

**Pydantic モデル（内部スキーマ検証用）**:
- `FareRoutesModel`: `/data/train_routes.json` の構造検証（`{"routes": [{"from": str, "to": str, "fare": int}]}`）
- `FareFixedModel`: `/data/fixed_fares.json` の構造検証（`{"バス": int, "タクシー": int, "飛行機": int}`）

### 3.4 エラーハンドリング

#### 3.4.1 エラー処理フロー

```
例外発生
  ↓
ValidationError（入力不正）
  → logger.error でログ記録（exc_info=True）
  → ErrorHandler.handle_validation_error(e)

  → 返す

FileNotFoundError / os.path.exists() 失敗（ファイル不存在 = WARNING）
  → logger.warning でログ記録（file_path 付き）
  → ErrorHandler.handle_fare_data_error(e)
  → (False, エラーメッセージ) タプル
  → 返す

Exception（ファイル読み込み失敗 = ERROR）
  → logger.error でログ記録（file_path 付き, exc_info=True）
  → ErrorHandler.handle_fare_data_error(e)
  → (False, エラーメッセージ) タプル
  → 返す

ValidationError（スキーマ不正）
  → logger.error でログ記録（file_path 付き, exc_info=True）
  → ErrorHandler.handle_calculation_error(e)
  → {"success": False, "message": エラーメッセージ, "calculable": False, "fare": None}
  → 返す

ValueError（経路未存在・キー不存在など計算系エラー）
  → logger.error でログ記録（file_path 付き）
  → ErrorHandler.handle_calculation_error(e)
  → {"success": False, "message": エラーメッセージ, "calculable": False, "fare": None}
  → 返す
```

## 5. 補足情報

### 5.1 実装上の注意点

1. **ファイル読み込みタイミング**
   - `/data/train_routes.json` / `/data/fixed_fares.json` はモジュールロード時ではなく関数呼び出し時に読み込む（実行時の最新データを参照するため）

2. **os.path.exists() による事前チェック**
   - ファイルの存在確認は `os.path.exists()` で事前チェックを行う
   - `open()` 時の `FileNotFoundError` 捕捉との併用禁止（`os.path.exists()` チェックのみを使用）
   - ファイル不存在時のログレベルは WARNING（システム障害ではなくデータ準備不足のため）

3. **例外を再送出しない**
   - ツール関数内で発生した全例外を捕捉し `{"success": False, "message": str}` 形式で返却する（例外処理方針.md 7.1準拠）

4. **リスト線形探索**
   - `/data/train_routes.json` の経路検索はリスト線形探索（`for route in routes: if route["from"] == departure and route["to"] == destination`）を使用する
   - 辞書キー参照（`routes[key]`）は使用しない

5. **FareRouteEntry の from フィールド**
   - Python の予約語回避のため `from_: str = Field(..., alias="from")` として定義し、`model_config = ConfigDict(populate_by_name=True)` を設定する

6. **FareFixedModel の日本語キー**
   - `/data/fixed_fares.json` のキーは日本語交通手段名（バス・タクシー・飛行機）を使用する
   - Pydantic モデルは日本語フィールド名（alias）で定義する

7. **経路未存在の扱い**
   - 電車経路未存在（線形探索でマッチしない）は正常系として `calculable=False` を返す
   - 固定運賃キー不存在は異常系として `ValueError` を発生させ、ErrorHandler.handle_calculation_error() で処理する

8. **エラー戻り値の使い分け**
   - データ読み込み系エラー: `(False, エラーメッセージ)` タプルを返す
   - 計算系エラー（スキーマ不正・ValueError等）: `{"success": False, "message": エラーメッセージ, "calculable": False, "fare": None}` dict を返す

### 5.2 パフォーマンス考慮事項

1. **ファイル読み込みのオーバーヘッド**
   - 1呼び出しにつき最大1ファイルの読み込みが発生する。ファイルサイズが小規模（数KB程度）であれば許容範囲内

### 5.3 セキュリティ考慮事項

1. **ファイルパスの固定化**
   - `/data/train_routes.json` / `/data/fixed_fares.json` のパスは固定し、外部から注入できないようにする

## 7. テスト観点

### 7.1 機能テスト
- 出発地「渋谷」・目的地「品川」・交通手段「電車」・移動日「2026-05-01」を入力し、`/data/train_routes.json` の定義運賃（例: 250円）が返却されること
- 交通手段「バス」を入力し、`/data/fixed_fares.json` の "バス" 値が返却されること
- 交通手段「タクシー」を入力し、`/data/fixed_fares.json` の "タクシー" 値が返却されること
- 交通手段「飛行機」を入力し、`/data/fixed_fares.json` の "飛行機" 値が返却されること

### 7.2 異常系テスト
- `/data/train_routes.json` に存在しない経路（例: 「上野」→「代々木」）を入力し、calculable=False・fare=None・success=True が返却されること（正常系として扱う）
- `/data/train_routes.json` が存在しない状態（os.path.exists()失敗）で呼び出し、(False, エラーメッセージ) タプルが返却されること（WARNING ログ）
- ファイル読み込み中に Exception が発生した場合、(False, エラーメッセージ) タプルが返却されること（ERROR ログ）
- 不正スキーマの `/data/train_routes.json`（routes キーがない）を読み込み、success=False とエラーメッセージが返却されること
- 不正スキーマの `/data/fixed_fares.json`（バスキーがない）を読み込み、success=False とエラーメッセージが返却されること
- `/data/fixed_fares.json` に対応する交通手段キーが存在しない場合に ValueError が発生し、success=False が返却されること
- transport_type に「バイク」を渡し、ValidationError が発生して success=False とエラーメッセージが返却されること
- departure に空文字を渡し、ValidationError が発生して success=False とエラーメッセージが返却されること
- travel_date に「2026/05/01」（スラッシュ形式）を渡し、ValidationError が発生して success=False とエラーメッセージが返却されること

### 7.3 性能テスト
- `/data/train_routes.json` / `/data/fixed_fares.json` の読み込みと検索が 1 秒以内に完了すること（測定指標: 処理時間）

### 7.4 境界値テスト
- 駅名末尾に「駅」が付く入力（「渋谷駅」）が「渋谷」に正規化された上で検索されること
- 「train」「鉄道」「bus」「taxi」「cab」「airplane」「plane」が各正規値に変換されること
- `/data/train_routes.json` に1件のみエントリが存在する場合にマッチ・非マッチがそれぞれ正しく返却されること
