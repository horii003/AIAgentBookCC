## 2. ErrorHandler 基本設計

**システム設計書の方針**: システム設計書 例外処理方針.md（`artifacts/03_system-design/outputs/例外処理方針.md`）を参照してください。

### 2.1 ErrorHandlerの位置づけ

#### システム全体における役割
- 全エージェント（AG-001/AG-002/AG-003）およびツール（TOOL-001/TOOL-002）から発生する例外を EX-01〜EX-08 に分類し、継続可否を判定して対応方針（再試行・代替処理・エスカレーション・中断）を決定する
- ユーザー向けの日本語メッセージを生成して技術的詳細を隠蔽し、呼び出し元へ返す
- 例外発生時の状態（EX-05/06/07/08 は TERMINATED）を FileBasedSessionManager に連携する

#### 主な責務（概要）
- 例外分類（EX-01〜EX-08）ごとの継続可否判定
- ユーザー向けエラーメッセージの生成（日本語、技術詳細を隠蔽）
- スタックトレースを `logs/agent.log` のみへ記録し、監査ログ（audit_log_std.jsonl）に EX 分類・コンテキストを記録
- エスカレーション要否の判定と CF-009 案内

#### 非責務（やらないこと）
- **リトライ実行**: Strands SDK の RetryConfig が実施する
- **セッション状態の更新**: FileBasedSessionManager（SM-001）が実施する
- **フックのトリガー**: HumanApprovalHook（HD-002）・LoopControlHook（HD-003）が実施する

#### 詳細
詳細設計書 1章を参照

### 2.5 ErrorHandlerの設計方針と設計意図

**例外分類による一貫処理**:
- 全コンポーネントが EX-01〜EX-08 の分類を共有することで、例外対応の方針を統一する（例外処理方針.md 3.2）

**スタティックメソッドのみ**:
- ErrorHandler はインスタンス状態を持たない。全メソッドをスタティックメソッドとして実装し、エージェント・ツールが `ErrorHandler.handle_xxx(...)` として直接呼び出せるようにする（インスタンス化不要）

**主要な設計意図**:
- **技術詳細の隠蔽**: ユーザーに APIエラー・スタックトレースを見せず、業務的な原因と次の行動のみを提示する（例外処理方針.md 7.2）
- **ツール関数からの例外非送出**: ツール関数は例外を再送出せず `{"success": False, "message": str}` で返却する。これにより LLM が次のステップを判断できる（例外処理方針.md 7.1）

**詳細**: 詳細設計書 2.1章を参照

## 3. HumanApprovalHook 基本設計

**システム設計書の方針**: システム設計書 実行制御方針.md（`artifacts/03_system-design/outputs/実行制御方針.md`）を参照してください。

### 3.1 HumanApprovalHookの位置づけ

#### システム全体における役割
- AG-002/AG-003 が TOOL-002 を呼び出す直前に介入し、社員（R-EMP）に申請書生成前の確認（APR-001）を求めるフック
- 社員の承認・修正・キャンセルに応じて TOOL-002 の実行可否を制御する（ABAC-001）
- APR-001 承認済みフラグを `session_{id}.json` の `approval_granted` フィールドに記録し、強化監査ログ（LOG-HI）に承認情報を記録する

#### 主な責務（概要）
- APR-001 承認ダイアログ（OK / 修正 / キャンセル）の提示と選択受付
- 承認済みフラグの `session_{id}.json` への記録（FileBasedSessionManager 経由）
- キャンセル時のセッション TERMINATED 遷移指示
- 強化監査ログ（LOG-HI）への承認者・承認日時・対象申請情報の記録

#### 非責務（やらないこと）
- **申請書の生成**: TOOL-002 (generate_transport_application / generate_expense_application) が実施する
- **承認内容の修正反映**: AG-002/AG-003 が修正対話を担当し再度 APR-001 を実施する
- **ループカウントのインクリメント**: LoopControlHook（HD-003）が実施する

#### 詳細
詳細設計書 1章を参照

### 3.2 利用エージェント

**登録対象の特定根拠**:
`自律度・権限定義.md` のエージェント × アクションマトリクスで `△`（承認必須）となっているアクションを持つエージェントを特定した。AG-002 の ACT-GEN-02/ACT-EXEC-03 および AG-003 の ACT-GEN-01/ACT-EXEC-03 が COND-003〜COND-006（APR-001 必須）に該当する。AG-001 は ACT-GEN-01/02 および ACT-EXEC-03 がすべて `×`（禁止）であるため対象外とする。

| エージェントID | エージェント名 | 利用目的 |
|---------------|---------------|---------|
| AG-002 | 交通費精算申請エージェント | TOOL-002 (generate_transport_application) 呼び出し前の APR-001 承認取得（COND-003/COND-005） |
| AG-003 | 経費精算申請エージェント | TOOL-002 (generate_expense_application) 呼び出し前の APR-001 承認取得（COND-004/COND-006） |

### 3.3 フック登録の概要

**フックの目的**: TOOL-002 が APR-001 承認なしに実行されることを防止し（ABAC-001）、社員が申請書の生成前に内容を確認できる機会を設ける（実行制御方針.md 11.1）

**登録するイベント**:
- `BeforeToolCallEvent`: TOOL-002 呼び出し直前に発火。APR-001 確認ダイアログを提示し、承認結果に基づいて実行可否を制御する

### 3.4 フック登録の詳細
詳細設計書 2.4.1章を参照

### 3.5 HumanApprovalHookの処理概要

**処理の流れ**:
1. `BeforeToolCallEvent` で呼び出し対象ツールが TOOL-002 (generate_transport_application または generate_expense_application) であることを確認する
2. 収集済み申請情報を整形してユーザーに提示し、APR-001 ダイアログ（OK / 修正 / キャンセル）を表示する（**この APR-001 承認は1回のみ発生する**）
3. ユーザーの選択を受け取り分岐する:
   - **OK**: `session_{id}.json` の `approval_granted = True` を記録し、LOG-HI に承認情報を記録してツール実行を続行する。**承認（OK）後にはじめて TOOL-002 が実行される**
   - **修正**: ツール実行をキャンセルし、修正対象を AG-002/AG-003 に通知して情報再収集ループへ戻す
   - **キャンセル**: ツール実行をキャンセルし、セッションを TERMINATED に遷移させる
4. 不正入力（OK/修正/キャンセル以外）の場合は最大3回まで再入力を促し、3回後もキャンセルとして処理する

**処理の目的**: 意図しない申請書生成を防止し、社員が申請内容を確認してから申請書生成を実行できるガバナンス要件を満たす（実行制御方針.md 11.1/11.2）。**承認前に TOOL-002 を実行してはならない（ABAC-001）**

### 3.6 処理フローの詳細
詳細設計書 3.1章を参照

## 4. LoopControlHook 基本設計

**システム設計書の方針**: システム設計書 実行制御方針.md（`artifacts/03_system-design/outputs/実行制御方針.md`）を参照してください。

### 4.1 LoopControlHookの位置づけ

#### システム全体における役割
- 全エージェント（AG-001/AG-002/AG-003）の ReActループ回数を監視し、上限（10回）到達時にループを強制停止する
- ループ暴走によるシステムコスト（API コール数）の増加とユーザーへの無応答状態を防止する
- 上限到達時は EventLoopException を発生させ、ErrorHandler が EX-02（判断不能）として処理する（実行制御方針.md 10.3）

#### 主な責務（概要）
- エージェント呼び出し開始時のループカウンタのリセット
- LLM 呼び出し後のループカウンタのインクリメントと上限監視
- 上限到達時の EventLoopException 発生

#### 非責務（やらないこと）
- **EX-02 の例外処理**: ErrorHandler（HD-001）が実施する
- **セッション状態の TERMINATED 遷移**: FileBasedSessionManager（SM-001）が実施する
- **リトライ回数の制御**: Strands SDK の RetryConfig が実施する

#### 詳細
詳細設計書 1章を参照

### 4.2 利用エージェント

| エージェントID | エージェント名 | 利用目的 |
|---------------|---------------|---------|
| AG-001 | 申請受付窓口エージェント | 申請種別判断・委譲ループの上限制御 |
| AG-002 | 交通費精算申請エージェント | 移動情報収集・交通費計算ループの上限制御 |
| AG-003 | 経費精算申請エージェント | 経費情報収集・OCR・経費区分判断ループの上限制御 |

### 4.3 フック登録の概要

**フックの目的**: エージェントの ReActループが10回を超えないよう制御し、無限ループによるコスト増大・応答不能状態を防ぐ（実行制御方針.md 10.1/10.2）

**登録するイベント**:
- `BeforeInvocationEvent`: エージェント呼び出し開始時に発火。ループカウンタを 0 にリセットする
- `AfterModelCallEvent`: LLM 呼び出し（1 ReActステップ）完了後に発火。ループカウンタをインクリメントし、10回到達時に EventLoopException を発生させる

### 4.4 フック登録の詳細
詳細設計書 2.4.1章を参照

### 4.5 LoopControlHookの処理概要

**処理の流れ**:
1. `BeforeInvocationEvent`: エージェント呼び出し開始時にインスタンス変数 `_loop_count` を 0 にリセットする
2. `AfterModelCallEvent`: LLM 呼び出し完了後に `_loop_count` をインクリメントする。例外・エラーが発生した呼び出しはカウントしない（誤カウント防止）
3. `_loop_count >= 10` の場合は EventLoopException を発生させ、ErrorHandler が EX-02 として処理する
4. ErrorHandler が「処理が完了できませんでした。管理部門（経理部）にお問い合わせいただくか、再度最初からお試しください。」を提示しセッションを TERMINATED に遷移させる

**処理の目的**: エージェントの自律処理に上限を設けることで制御不能な自律動作を防止し、システムコストとユーザー体験を保護する（実行制御方針.md 10.2）

### 4.6 処理フローの詳細
詳細設計書 3.1章を参照

## 5. マルチエージェント連携時の扱い

### 5.1 ErrorHandler

**共有方法**:
- ErrorHandler はスタティックメソッドのみで構成するため、インスタンスを共有する必要がない。各エージェント・ツールから `ErrorHandler.handle_xxx(...)` を直接呼び出す

**スレッドセーフ性**:
- スタティックメソッドはインスタンス状態を持たないためスレッドセーフ（例外ログの書き込みは Python logging モジュールのスレッドセーフ性に依存する）

### 5.2 HumanApprovalHook

**共有方法**:
- AG-002 と AG-003 はそれぞれ独立したエージェントとして動作し、HumanApprovalHook インスタンスは各エージェントの `hooks` パラメータに個別に登録する（インスタンス共有しない）

**競合防止**:
- AG-001 は AG-002/AG-003 を同時に呼び出さないため（実行制御方針.md 6.2）、HumanApprovalHook の同時発火は発生しない
- APR-001 承認済みフラグは `session_{id}.json` に記録するため、複数呼び出しの競合は FileBasedSessionManager が排他制御する

### 5.3 LoopControlHook

**共有方法**:
- AG-001/AG-002/AG-003 それぞれが個別の LoopControlHook インスタンスを保持する。各エージェントのループカウンタは独立して管理される（エージェント間でカウンタを共有しない）

**競合防止**:
- 各エージェントが独立したインスタンスを持つためカウンタ競合は発生しない。`BeforeInvocationEvent` でリセットされるため、エージェントの再呼び出し時も正しくリセットされる

## 6. テスト観点

### 6.1 ErrorHandler
- **正常系**: `handle_input_error` に EX-01 を渡すと日本語エラーメッセージが返却され、スタックトレースが `logs/agent.log` に記録されること
- **異常系**: `handle_unknown_error` に未分類例外を渡すと EX-08 として処理され、「処理できませんでした」メッセージが返却されること
- **境界値**: EX-04 に対してリトライ判定フラグが正しく返却されること（リトライ可否の判定）

### 6.2 HumanApprovalHook
- **正常系（OK）**: `BeforeToolCallEvent` で APR-001 ダイアログを提示し、「OK」選択後に `approval_granted = True` が `session_{id}.json` に記録されてツール実行が継続されること
- **Human-in-the-Loop テスト**: APR-001 承認が BeforeToolCallEvent で1回のみ発生し、承認（OK）後にはじめて TOOL-002 が実行されること。承認前に TOOL-002 が実行されないこと
- **正常系（キャンセル）**: 「キャンセル」選択後にツール実行がブロックされ、セッション状態が TERMINATED に遷移すること
- **異常系**: 不正入力が3回続いた場合にキャンセルとして処理されること

### 6.3 LoopControlHook
- **正常系**: ループ10回未満でループが正常継続すること
- **異常系**: ループ10回到達時に EventLoopException が発生し、ErrorHandler が EX-02 として処理してセッションが TERMINATED に遷移すること
- **境界値**: `BeforeInvocationEvent` でループカウンタが 0 にリセットされること（前回のループカウントが持ち越されないこと）

### 6.4 技術的なテスト詳細
詳細設計書 9章を参照

## 7. 詳細設計への引き渡し事項

### 7.1 実装時の識別子

- **クラス名**: `ErrorHandler`、`HumanApprovalHook`、`LoopControlHook`
- **実装ファイル**:
  - `handlers/error_handler.py`（ErrorHandler）
  - `handlers/human_approval_hook.py`（HumanApprovalHook）
  - `handlers/loop_control_hook.py`（LoopControlHook）

### 7.2 SDK実装時の注意事項

- **フックのイベント登録**: Strands Agents SDK で使用可能なイベントは `BeforeInvocationEvent`、`BeforeModelCallEvent`、`AfterModelCallEvent`、`BeforeToolCallEvent`、`AfterToolCallEvent`、`AfterInvocationEvent` の6種。HumanApprovalHook は `BeforeToolCallEvent`、LoopControlHook は `BeforeInvocationEvent` + `AfterModelCallEvent` を登録する
- **LoopControlHookのカウント例外処理**: `AfterModelCallEvent` ハンドラー内で例外が発生した場合（LLM 呼び出し自体がエラーの場合）はカウントをインクリメントしない。ErrorHandler が EX-04/EX-07 として別途処理する
- **HumanApprovalHookの対象ツール判定**: `BeforeToolCallEvent` の `tool_name` が `generate_transport_application` または `generate_expense_application` の場合のみ APR-001 を実施する（TOOL-001 等の他ツールは対象外）
- **ErrorHandlerのスタティックメソッド**: クラスインスタンスを生成せずに `@staticmethod` デコレータで全メソッドを定義する
- **EventLoopException**: Strands SDK 固有の例外クラス。LoopControlHook が raise するとエージェントループが停止する。`__cause__` で `LoopLimitError` を確認してから ErrorHandler に委譲する

### 7.3 モック／スタブ要否

- **ErrorHandler**: 単体テストでは実際の logging モジュールを使用してもよい。`logs/agent.log` の生成確認にはテスト用ディレクトリを使用する
- **HumanApprovalHook**: ユーザー入力を `input()` 関数で受け取る場合は `unittest.mock.patch('builtins.input')` でモックする。`session_{id}.json` は tmpdir のモックファイルを使用する
- **LoopControlHook**: EventLoopException は Strands SDK のテスト用スタブを使用する。`AfterModelCallEvent` のモックでカウンタインクリメントをシミュレートする

## 9. 制約事項・前提条件

### 9.1 技術的制約
- Strands Agents SDK v1.25.0 のフックシステム（`BeforeInvocationEvent`/`AfterModelCallEvent`/`BeforeToolCallEvent`）を使用すること
- ErrorHandler は Python の `logging` モジュールを使用してアプリケーションログ（`logs/agent.log`）に記録すること
- LoopControlHook の最大ループ回数は 10 回とし、全エージェントで共通の値を使用すること（共通設定方針.md 4.2）

### 9.2 業務的制約
- HumanApprovalHook は AG-002 と AG-003 にのみ登録する。AG-001 には登録しない（申請書生成権限なし）
- APR-001 の承認選択肢は「OK / 修正 / キャンセル」の3択とし、不正入力は最大3回まで再入力を促してキャンセルとして処理する
- ErrorHandler はユーザーに技術的な例外詳細（スタックトレース・APIエラーコード等）を提示しない
- **APR-001 承認（BeforeToolCallEvent）は1回のみ発生する**。承認前に TOOL-002 を実行してはならない（ABAC-001）

### 9.3 前提条件
- `logs/` ディレクトリが書き込み可能な状態で存在すること
- `data/sessions/` ディレクトリが書き込み可能な状態で存在すること（HumanApprovalHook の承認フラグ記録のため）
- Strands Agents SDK の HookProvider インターフェースが利用可能であること
