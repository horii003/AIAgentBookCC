## 1. 概要

### 1.1 コンポーネントの目的

経費精算申請に特化した専門エージェント。AG-001から委譲を受け、領収書画像から経費情報を自動抽出・経費区分を自動判断してExcel申請書（下書き）を自動作成する。Human-in-the-Loop承認（OK）後にTOOL-002を呼び出して申請書を生成し、妥当性チェック結果を提示する。

### 1.2 主要な責務

- invocation_stateから申請者名・申請日・セッションIDを受け取る
- 領収書画像（D-014）のアップロードを依頼し、image_readerで経費情報を自動抽出する（BRL-12）
- 品目から経費区分（事務用品費・宿泊費・資格精算費・その他経費）を自動判断する（BRL-13、JD-09）
- 業務目的等の不足情報を対話で確認する（BRL-02、BRL-11）
- 収集済み申請情報をテキストとして整理・提示し（ドラフト提示）、Human-in-the-Loop承認を要求する（APR-002）
- 承認（OK）後にTOOL-002（generate_expense_application）を呼び出して申請書（下書き）を生成する
- 申請期限チェック（BRL-07）・高額承認要否判断（BRL-09）・業務目的記載有無チェック（BRL-11）を実施する

### 2.2 インターフェース設計

#### 2.2.1 入力

**初期化時の入力**:
- ToolContext経由でinvocation_stateを受け取る

**ToolContext経由で受け取る情報**:
```python
{
    "applicant_name": str,

```

**対話ループでの入力**:
- 領収書画像（D-014）: 社員がアップロードするJPEG/PNG形式の領収書画像

**パラメータ**:

#### 2.2.2 出力

**出力先**: CLI（標準出力）

**出力メッセージ**:
- 画像アップロード依頼: `"領収書画像をアップロードしてください。"`

**戻り値の形式**: `str`（エージェントからの応答）

### 2.3 ビジネスロジック

#### 2.3.1 システムプロンプト

##### システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 動的生成関数 |
| 動的生成の理由 | 申請期限チェック（BRL-07）で実行時の申請日と申請期限をプロンプトに埋め込む必要がある |

##### 動的に埋め込む情報
- 申請日（application_date：invocation_stateから取得）
- 申請期限（申請日から3ヶ月後の日付：実行時に計算）
- 妥当性チェックルール（`agent_knowledge/receipt_policies.py` から読み込んでプロンプトに組み込む）

##### プロンプト全文

```
あなたは経費精算AIエージェントシステム（ECAAS）の経費精算申請エージェントです。
申請者の経費精算申請を支援し、領収書画像から経費情報を自動抽出して申請書（下書き）を自動作成します。

【申請情報】
- 申請者名：{applicant_name}
- 申請日：{application_date}
- 申請期限（経費発生日から3ヶ月以内）：{deadline}

【役割】
1. 領収書画像からの経費情報自動抽出（BRL-12）
   - 申請者に領収書画像のアップロードを依頼する
   - image_readerを使用して購入日・店舗名・品目・金額を品目ごとに自動抽出する
   - 画像読み取り失敗時は再アップロードを促す

2. 経費区分の自動判断（BRL-13）
   - 品目から経費区分（事務用品費・宿泊費・資格精算費・その他経費）を自動判断する
   - 判断不能時は「その他経費」として設定する（JD-09）

3. 不足情報の確認（BRL-02、BRL-11）
   - 業務目的等の不足情報を対話で確認する
   - 全情報が揃うまで繰り返す

4. 収集済み申請情報のテキスト整理・提示（ドラフト提示）
   - 全経費情報・業務目的が揃ったら、収集済み申請情報をテキストとして整理して提示する
   - この時点ではツールを呼び出さない
   - OK/修正/キャンセルの3択を提示する（APR-002）

5. Human-in-the-Loop承認（APR-002）
   - OK：generate_expense_applicationツールを呼び出して申請書（下書き）を生成する
   - 修正：修正内容を収集して情報収集フローへ戻る
   - キャンセル：処理を中断してセッションを終了する

6. 申請内容妥当性チェック（BRL-03）
   - 申請期限チェック（BRL-07）：購入日が申請日から3ヶ月以内であること
   - 高額承認要否判断（BRL-09）：経費合計 > 5,000円の場合は上長承認案内
   - 業務目的記載有無チェック（BRL-11）：業務目的が記載されていること

【申請ルール（BRL-01）】
- 対応経費区分：事務用品費・宿泊費・資格精算費・その他経費（BRL-13）
- 申請期限：経費発生日から3ヶ月以内（BRL-07）
- 経費合計 > 5,000円：上長の事前承認が必要（BRL-09）
- 業務目的：必須記載（BRL-11）
- 申請ルール外の推測・補完禁止（BRL-14）

【エラー発生時の振る舞い】
- ツールエラー・システム障害等のシステム系エラーが発生した場合は、エラー内容を要約して呼び出し元エージェント（AG-001）に返す

【禁止事項】
- 承認前にgenerate_expense_applicationツールを呼び出してはならない
- 申請ルールに定義されていない内容を推測・補完してはならない（BRL-14）
```

##### プロンプト設計の意図

| 要素 | 意図 |
|------|------|
| 動的な申請日・申請期限の埋め込み | BRL-07の申請期限チェックをLLMが正確に判断できるよう、実行時の日付情報を提供する |
| 役割定義 | 領収書画像処理→経費情報抽出→経費区分判断→不足情報確認→ドラフト提示→承認→申請書生成→妥当性チェックの順序を明確化する |
| 承認前ツール呼び出し禁止 | APR-002に基づき、Human-in-the-Loop承認前のTOOL-002呼び出しを防止する |

#### 2.3.2 申請ルール

| ルール | 条件 | 対応 |
|--------|------|------|
| BRL-12（領収書画像からの自動抽出） | 経費精算申請の情報収集時 | image_readerを使用して領収書画像から経費情報を品目ごとに自動抽出する |
| BRL-13（経費区分自動判断） | 経費情報抽出後 | 品目から経費区分を自動判断する。判断不能時は「その他経費」として設定する（JD-09） |
| BRL-07（申請期限チェック） | 妥当性チェック時 | 購入日が申請日から3ヶ月以内であることを確認する |
| BRL-09（高額承認案内） | 経費合計 > 5,000円 | 上長の事前承認が必要である旨を案内する（APR-004） |
| BRL-11（業務目的必須） | 申請書作成・妥当性チェック時 | 業務目的が未記載の場合は不足情報として対話で確認する |

#### 2.3.3 処理フロー

```
開始（ToolContext経由でinvocation_stateを受け取る）
  ↓
invocation_stateからapplicant_name・application_date・session_idを取得する
  ↓
application_dateから申請期限（3ヶ月後）を計算する
  ↓
動的システムプロンプトを生成する
  ↓
エージェントを初期化する
  ↓
領収書画像のアップロードを依頼する（CF-004 Step1）
  ↓
image_readerで経費情報を自動抽出する（BRL-12）
  → 画像読み取り失敗（EX-01）: 再アップロードを促す
  ↓
品目から経費区分を自動判断する（BRL-13、JD-09）
  → 判断不能: 「その他経費」として設定する
  ↓
抽出した経費情報・経費区分をユーザーに提示する
  ↓
不足情報の確認（CF-004 Step5）
  - 業務目的が未記載の場合は対話で確認する（BRL-11）
  - 全情報が揃うまで繰り返す（BRL-02）
  ↓
収集済み申請情報をテキストとして整理・提示する（ドラフト提示）
  ※ この時点ではツールを呼び出さない
  ↓
Human-in-the-Loop承認を要求する（APR-002）
  OK → generate_expense_applicationツールを呼び出す（CF-005）
         → エラー（EX-06/EX-07）: エラー内容を要約してAG-001に返す
  修正 → 修正内容を収集してcollected_infoに追記し、不足情報確認フローへ戻る
  キャンセル → 「申請処理をキャンセルしました。」を表示してセッション終了（EX-05）
  ↓
申請書ファイルパスをユーザーに提示する
  ↓
妥当性チェックを実施する（CF-006）
  - 申請期限チェック（BRL-07）
  - 高額承認要否判断（BRL-09）
  - 業務目的記載有無チェック（BRL-11）
  ↓
チェック結果・修正案を提示する
  ↓
処理完了。AG-001に結果を返す
```

## 3. 実装詳細

### 3.1 クラス/関数設計

#### 3.1.1 expense_application_agent関数

##### 説明
経費精算申請エージェントをAgent as Toolsとして公開する関数。AG-001から呼び出される。

##### 引数
- `query` (str): ユーザーからの申請内容テキスト（必須）
- `tool_context` (ToolContext): AG-001から渡されるToolContext（必須）

##### 戻り値
- `str`: エージェントからの応答（申請書ファイルパス・チェック結果・エラーメッセージ等）

##### 処理内容
1. tool_context.invocation_stateからapplicant_name・application_date・session_idを取得する
2. application_dateから申請期限（3ヶ月後）を計算する
3. 動的システムプロンプトを生成する
4. エージェントを初期化する（ツール・フック・会話管理・セッション管理を設定）
5. エージェントにqueryを渡して処理を開始する
6. エージェントの応答を返す

## 5. 補足情報

### 5.1 実装上の注意点

1. **承認前のツール呼び出し禁止**
   - HumanApprovalHookがgenerate_expense_applicationツール呼び出し前に承認を実施する
   - 承認前にTOOL-002を呼び出してはならない（APR-002）

2. **image_readerの使用**
   - 領収書画像の読み取りはstrands-agents-toolsのimage_reader（フレームワーク組み込み機能）を使用する
   - 独自のOCRツールは実装しない

3. **動的プロンプト生成**
   - 妥当性チェックルールはエージェント初期化時に `agent_knowledge/receipt_policies.py` を読み込んでプロンプトに組み込む

4. **session_idの除外**
   - 専門エージェント内部ではsession_idを除外したinvocation_stateを使用する（マルチエージェント連携設計 DEC-004）

### 5.2 パフォーマンス考慮事項

1. **ウィンドウサイズ**
   - SlidingWindowConversationManager(window_size=15)を使用する

## 7. テスト観点

### 7.1 機能テスト
- 領収書画像から経費情報（購入日・店舗名・品目・金額）が正しく抽出されること
- 品目から経費区分が正しく判断されること
- 判断不能な品目が「その他経費」として設定されること
- 業務目的が未記載の場合に対話で確認されること
- 収集済み申請情報がテキストとして整理・提示されること（ツール呼び出しなし）
- Human-in-the-Loop承認（OK）後にgenerate_expense_applicationが呼び出されること
- 申請書（下書き）が正しく生成されること
- 申請期限チェック（BRL-07）が正しく実施されること
- 経費合計 > 5,000円の場合に上長承認案内が表示されること

### 7.2 異常系テスト
- 領収書画像読み取り失敗時に再アップロードを促すメッセージが表示されること
- Human-in-the-Loop承認でキャンセルを選択した場合にセッションが終了すること
- 承認前にgenerate_expense_applicationが呼び出されないこと
- LoopLimitError発生時にWARNINGログが出力され、エラーメッセージ文字列がAG-001に返却されること
- ContextWindowOverflowException発生時にWARNINGログが出力され、エラーメッセージ文字列がAG-001に返却されること
- MaxTokensReachedException発生時にWARNINGログが出力され、エラーメッセージ文字列がAG-001に返却されること
- RuntimeError発生時にERRORログ（exc_info=True）が出力され、エラーメッセージ文字列がAG-001に返却されること
- ログにクエリの先頭50文字が付加情報として含まれること

### 7.3 性能テスト
- 領収書画像処理から申請書生成までが適切な応答時間内に完了すること
- 複数品目（5品目以上）の申請書生成が正常に完了すること
