## 2. 目的・役割定義

### 2.1 エージェントの目的
- AG-001から委譲された交通費精算申請フローを引き受け、移動情報を一区間ずつ対話で収集する
- 収集した移動情報をもとに交通費を自動計算（TOOL-001）し、申請書（下書き）を生成（TL-002a）する
- 申請期限（90日）チェックおよび高額申請時の上長承認要否通知（10,000円超）を担当する

### 2.2 主な責務
- **移動情報の対話収集**: 移動日・出発地・目的地・交通手段・費用・業務目的を一区間ずつ収集する（CF-003 / BRL-04/05）
- **申請期限チェック（90日）**: 移動日から本日までの差が90日を超える場合に申請不可を通知する（BRL-14 / GRD-006）
- **交通費自動計算**: TOOL-001（calculate_transport_fare）を呼び出して交通費を算出する（BRL-13/15）
- **高額申請通知**: 交通費が10,000円超の場合に上長承認が必要な旨を通知する（BRL-10 / GRD-007）
- **申請書生成前の社員確認**: HumanApprovalHookで社員のOK/修正/キャンセルを取得する（GRD-014）
- **申請書（下書き）生成**: TL-002a（generate_transport_expense_form）を呼び出して申請書を生成する（CF-005）

### 2.3 非責務（やらないこと）
- **申請種別判断**: AG-001が担当。AG-002は交通費精算申請と確定済みの状態で起動される
- **申請書の提出**: 社員が手動で実施（GRD-012 / ACT-EXEC-01禁止）
- **上長承認の代行**: 上長承認要否の通知のみ担当。承認取得は社員が行う（GRD-013）
- **経費精算申請フロー**: AG-003が担当

## 3. 自律度・判断権限

### 3.1 自律度レベル

| レベル | 内容 | 適用範囲 |
|------|------|----------|
| 参照（Lv1） | 社内申請ルールファイル・申請先情報の参照 | P-READ-001 / DATA-001, DATA-004 |
| 提案（Lv2） | 不足情報質問の生成、収集済み申請情報の提示 | ACT-GEN-02 |
| 実行（Lv3） | 交通費計算（TOOL-001）、申請書生成（TL-002a）（HumanApprovalHook承認後） | ACT-EXEC-01 / FR-008, FR-011 |

### 3.2 判断権限の範囲
- **自律判断可能**: 交通費の自動計算（TOOL-001）、申請期限の自動チェック（90日）、10,000円超の上長承認要否判定、移動情報の一区間収集完了判断
- **人間確認必要**: 収集情報全体の確認（HumanApprovalHook の OK/修正/キャンセル）（GRD-014）
- **実行不可**: 申請書の提出（GRD-012 / ACT-EXEC-01禁止）、上長承認の代行（GRD-013禁止）

## 4. 処理フロー設計

### 4.1 基本フロー（概要）

```text
開始（AG-001から委譲、invocation_stateにsession_id・applicant_name・application_date設定済み）
→ 移動日の収集（CF-003 Step 1-2）
→ 申請期限チェック（BRL-14 / GRD-006）
  ├── [期限超過] 申請不可通知して終了（EX-03）
  └── [期限内] 出発地・目的地・交通手段の収集（Step 4-9）
→ 交通費計算（TOOL-001 / Step 10）
  ├── [計算成功] 費用を提示（Step 11）
  └── [計算不能] 手動入力を促す（EX-02）
→ 高額チェック・上長承認要否通知（BRL-10 / GRD-007 / Step 12）
→ 業務目的の収集（Step 13-14）
→ 追加区間の確認（Step 15-16）
  ├── [追加あり] Step 1に戻る（多区間ループ）
  └── [追加なし] 収集情報の確認提示（Step 17）
→ 収集情報確認提示（Step 17）: 収集済み申請情報をテキストとして整理・提示（ツール呼び出しなし）
→ HumanApprovalHook（BeforeToolCallEventトリガー）（社員確認: OK/修正/キャンセル）（GRD-014）
  ├── [OK] TL-002a（generate_transport_expense_form）呼び出し
  ├── [修正] 該当項目再収集へ
  └── [キャンセル] フロー終了
→ 申請書（下書き）ファイルパスを社員に提示（CF-005完了）
→ 終了
```

### 4.2 フロー制御の考え方
- **多区間ループ**: 1区間の情報収集（Step 1-14）→ 追加確認（Step 15-16）→ 追加ありなら Step 1に戻るループ構造。LoopControlHookが最大10反復を保証する
- **申請期限チェックのタイミング**: 移動日入力直後（Step 3）に実施。期限超過時はその後の情報収集を行わず即座に終了する（EX-03）
- **高額通知のタイミング**: 交通費確定後（Step 12）に実施。フローは継続するが上長承認要否を明示する
- **ドラフト提示とツール呼び出しの分離**: Step 17 で収集済み申請情報をテキストとして整理・提示（ツール呼び出しなし）した後、社員が申請書生成を指示したタイミングで BeforeToolCallEvent が発火し HumanApprovalHook が介入する。Human-in-the-Loop 承認は1回のみ発生する設計とする。承認前に TL-002a を実行してはならない

### 4.3 CF-003 会話ステップ詳細

| Step | 発話者 | 内容 | 判断/処理 | 次ステップ | 根拠提示 |
|------|--------|------|----------|-----------|---------|
| 1 | Agent | 移動日の確認 | — | Step 2 | Lv0 |
| 2 | 社員 | 移動日の入力 | — | Step 3 | — |
| 3 | Agent | 申請期限チェック | BRL-14（JD-10/11） | Step 4（期限内）/ 終了（期限超過） | **Lv3（超過時）** |
| 4 | Agent | 出発地の確認 | — | Step 5 | Lv0 |
| 5 | 社員 | 出発地の入力 | — | Step 6 | — |
| 6 | Agent | 目的地の確認 | — | Step 7 | Lv0 |
| 7 | 社員 | 目的地の入力 | — | Step 8 | — |

### 4.4 状態管理
- **invocation_state（AG-001から受け取り）**: `session_id`・`applicant_name`・`application_date`
- **収集済み申請情報（D-010）**: セッション状態ファイル（`data/sessions/{session_id}.json`）に永続化。区間ごとの移動情報（D-014）を蓄積する
- **申請者名（applicant_name）**: invocation_stateから取得済み。AG-002は対話で再収集しない
- **申請日（application_date）**: invocation_stateから取得済み。申請書生成時に使用する

## 5. 利用ツール設計

### 5.1 利用ツール一覧

**ツール割り当て方針**: 各エージェントが使用するツールは、業務機能（計算・生成など）に特化したツールのみとする。

| ツールID | ツール名 | 呼び出しタイミング | 目的 |
|---------|---------|-----------------|------|
| TOOL-001 | calculate_transport_fare | CF-003 Step 10（区間情報確定後） | 移動区間の交通費を自動算出する |
| TL-002a | generate_transport_expense_form | CF-005（HumanApprovalHook で OK 取得後） | 収集済み移動情報から交通費申請書（下書き）を生成する |

### 5.2 TOOL-001 利用設計

| 項目 | 内容 |
|------|------|
| 呼び出し条件 | 移動日・出発地・目的地・交通手段が確定した区間ごとに呼び出す（C-001/C-002を満たすこと） |
| 入力 | TransportToolInput（departure, destination, transportation_type, travel_date, purpose） |
| 出力 | `{"success": bool, "fare": int, "calculation_basis": str, "message": str}` |
| 成功時 | fare値をD-014の費用フィールドに格納し、calculation_basisを社員に提示する |
| 失敗時（経路不明 EX-02） | 社員に手動入力を促す（「テーブルに該当区間がないため、交通費を直接入力してください」） |
| 失敗時（ファイル参照失敗 EX-04） | ErrorHandler経由でCF-006に誘導して中断する |
| リトライ | 最大6回・指数バックオフ（冪等のため安全）（実行制御方針 §6.1） |

### 5.3 TL-002a 利用設計

| 項目 | 内容 |
|------|------|
| 呼び出し条件 | HumanApprovalHook（BeforeToolCallEvent）でOKを取得した後のみ（GRD-014）。承認前にツールを実行してはならない。全必須項目収集済みであること |
| 入力 | TransportFormInput（applicant_name, application_date, segments: list[TransportSegment]） |
| 出力 | `{"success": bool, "file_path": str, "message": str}` |
| 成功時 | file_pathを社員に提示し「output/〇〇に申請書（下書き）を生成しました。申請先に提出してください。」と案内する |
| 失敗時（テンプレート不在 EX-04） | CF-006に誘導して担当部門への問い合わせを案内して中断する |
| 失敗時（生成失敗 EX-06） | 1回リトライ後も失敗の場合はCF-006へエスカレーションする |
| リトライ | 最大1回（非冪等のため）（実行制御方針 §6.1） |

## 6. プロンプト設計

### 6.1 システムプロンプトの目的
- 交通費精算申請フローの進行ルール（CF-003）を定義する
- 申請期限チェック（BRL-14 / 90日）の判断基準を組み込む（動的生成が必要な理由）
- 対応交通手段の制限（BRL-11/12）と交通費上限通知（BRL-10）を定義する
- 申請書提出禁止（GRD-012 / ACT-EXEC-01）を明示する
- HumanApprovalHookによる確認承認フロー（GRD-014）を定義する

### 6.2 プロンプト構成要素
- **役割定義**: 交通費精算申請の専門エージェントとして、移動情報収集・交通費計算・申請書生成を担当する
- **処理フロー**: CF-003（17ステップ）の進行手順
- **判断プロトコル**: 申請期限チェック（移動日から本日まで90日以内）、高額チェック（10,000円超で上長承認通知）
- **主な責任**: 移動情報の正確な収集、交通費の透明な算出（根拠提示）、申請書生成前の社員確認
- **動的情報**: 本日の日付（`application_date`）、申請期限日（`deadline_date`）
- **エラー発生時の振る舞い**: システム系エラー（ツールエラー・システム障害等）が発生した場合、呼び出し元エージェント（AG-001）にエラー内容を要約したメッセージを返却する

### 6.3 システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | **動的生成関数** |
| 動的生成の理由 | 申請期限チェック（BRL-14）で移動日から90日以内かを判断するために、本日の日付（`date.today()`）をプロンプトに埋め込む必要がある |
| 動的に埋め込む情報 | `application_date`（本日の日付、YYYY-MM-DD形式）、`deadline_date`（本日 - 90日、YYYY-MM-DD形式） |

```python
def _build_transport_agent_system_prompt(application_date: str) -> str:
    deadline_date = (date.fromisoformat(application_date) - timedelta(days=90)).isoformat()
    return f"""あなたは交通費精算申請の専門エージェントです。...
申請日: {application_date}
申請期限: {deadline_date}以降の移動のみ申請可能（移動日がこれより前の場合は申請不可）
..."""
```

### 6.4 主要な設計意図
- **動的プロンプト**: 申請期限チェックには実行時の日付が必要なため、毎回プロンプトを生成する。ただし埋め込む情報は最小限（日付2つのみ）に抑え生成コストを低減する
- **根拠提示**: 申請期限超過通知・高額通知時はルールID（BRL-14・BRL-10）を提示して社員が確認できるようにする
- **多区間ループの明示**: 1区間ずつ順番に収集するフローをプロンプトで明示し、LLMが複数区間を一度に収集しようとする誤動作を防ぐ

**プロンプト全文**: 詳細設計書 2.3.1章を参照

## 10. 出力データ設計

### 10.1 出力形式

### 10.2 主要な出力メッセージ

- 申請書生成完了: `"申請書（下書き）を「{file_path}」に生成しました。総務部に提出してください。"`

### 10.3 成果物
- **移動情報（D-014）**: 区間ごとの移動情報（移動日・出発地・目的地・交通手段・費用・業務目的）

## 11. バリデーション設計

**システム設計書の方針**: `artifacts/03_system-design/outputs/バリデーション方針.md` を参照。

### 11.1 バリデーション方針
- **交通手段**: `normalize_transportation_type`（BRL-12）でJR・新幹線等の別表記を正規4種に変換。対応外は GRD-002 により拒否
- **駅名**: `normalize_station_name`（BRL-15）で「駅」等の接尾語を除去
- **移動日**: 未来日付禁止（当日以前）、申請期限チェック（BRL-14）はシステムプロンプト内で実施（`check_application_deadline`はTransportToolInput側でも実施）
- **費用**: `int`型・`ge=0`、文字列 `"1,500円"` → `1500` 自動変換

### 11.2 申請ルール一覧

| ルールID | ルール名 | 条件 | エージェントの対応 |
|----------|----------|------|------------------|
| BRL-04 | 申請書生成前の必須項目充足確認 | 全必須項目（D-008）が収集済みの場合のみ申請書生成を許可 | 未収集項目を提示して対話継続 |
| BRL-05 | 不足情報残存時の対話継続 | 必須項目に未収集あり | 不足項目を1つずつ質問して収集を続ける |
| BRL-10 | 高額申請の上長承認要否 | 交通費 > 10,000円 | 上長承認が必要な旨を通知（Lv3根拠提示）。フロー継続 |
| BRL-11 | 対応交通手段の制限 | 入力交通手段 ∈ {電車, バス, タクシー, 飛行機} | 対応外は拒否して再選択を促す（GRD-002） |
| BRL-14 | 申請期限90日チェック | (application_date - travel_date).days > 90 | 申請不可通知してフロー終了（EX-03） |

## 12. Agent as Tools設計（AG-002の被呼び出し側設計）

### 12.1 AG-002のAgent as Tools化
AG-002はAG-001から `@tool(context=True)` のデコレータを持つファクトリ関数 `_get_transport_agent(session_id)` を通じてツールとして呼び出される。

### 12.2 ツール関数設計

| 項目 | 内容 |
|------|------|
| ツール関数名 | transport_agent_tool |
| デコレータ | @tool(context=True) |
| 役割 | AG-002インスタンスを生成してAG-001から呼び出し可能にする |
| 呼び出し元 | AG-001（申請種別 = 交通費精算申請確定後） |

### 12.3 ToolContextからのinvocation_state受け取り

```python
@tool(context=True)
def transport_agent_tool(context: ToolContext, ...) -> str:
    # invocation_stateから状態を取得
    invocation_state = InvocationState.model_validate(context.invocation_state)
    session_id = invocation_state.session_id
    applicant_name = invocation_state.applicant_name
    application_date = invocation_state.application_date
    
    # セッションIDでエージェントインスタンスを生成
    agent = _get_transport_agent(session_id, applicant_name, application_date)
    return agent(...)
```

**詳細**: 詳細設計書 2.5章を参照

## 13. テスト観点

### 13.1 テスト観点の概要
- **機能テスト**: 移動情報の1区間ずつ収集フロー（CF-003の全17ステップ）。多区間ループ（区間追加あり）のフロー。交通費自動計算結果（TOOL-001）の正確性。申請書生成（TL-002a）の成功確認
- **業務ルールテスト**: 申請期限90日超過時のフロー終了確認（BRL-14）。交通費10,000円超の上長承認通知（BRL-10）。対応外交通手段入力時の拒否と再入力要求（BRL-11/GRD-002）
- **HITL テスト**: HumanApprovalHookのOK/修正/キャンセル各分岐のフロー確認。HumanApprovalHook承認前にTL-002aが実行されないこと。Step 17のドラフト提示がツール呼び出しなしのテキスト整理であること
- **異常系テスト**: 電車経路テーブルに存在しない区間での手動入力フォールバック。テーブルファイル不在時のCF-006誘導。申請書生成失敗時のリトライとCF-006エスカレーション
- **境界値テスト**: 移動日が90日ちょうどの場合（期限内）と91日前（期限超過）。交通費がちょうど10,000円の場合（通知なし）と10,001円の場合（通知あり）

**詳細なテストケース**: 詳細設計書 7章を参照

## 15. 制約事項・前提条件

### 15.1 技術的制約
- Strands Agents SDK v1.25.0 の `Agent` クラスを使用する
- LLMバックエンドは Amazon Bedrock（Claude Sonnet 4.5）のみ
- `SlidingWindowConversationManager(window_size=20, should_truncate_results=True)` を使用する
- HumanApprovalHookはAG-002への必須設定とし、TL-002a呼び出し前に介入する
- システムプロンプトは動的生成関数（`_build_transport_agent_system_prompt`）で生成する

### 15.2 業務的制約
- 申請書の提出はAG-002が行わない（P-EXEC-003禁止 / GRD-012）
- 上長承認の代行はAG-002が行わない（GRD-013）
- 対応交通手段は電車・バス・タクシー・飛行機の4種類のみ（BRL-11/12）
- 申請期限は移動日から90日以内（BRL-14）。超過時は申請不可でフロー終了

### 15.3 前提条件
- AG-001が申請種別を「交通費精算申請」に確定した後に起動されること
- `invocation_state` に `session_id`・`applicant_name`・`application_date` がセット済みであること（AG-001が設定）
- TOOL-001用テーブルファイル（train_fares.json / fixed_fares.json）が `data/` 配下に配置されていること
- TL-002a用テンプレート（交通費申請書_template.xlsx）が `templates/` 配下に配置されていること
