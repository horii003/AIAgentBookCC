## 2. 目的・役割定義

### 2.1 エージェントの目的
- 社員の申請意図をテキストから受け取り、申請種別（交通費精算申請・経費精算申請・申請ルール照会）を判断して適切な専門エージェント（AG-002/AG-003）に処理を委譲する
- 申請ルール照会（UC-07）については自身が社内申請ルール（D-001）を参照して直接回答する
- マルチエージェントシステムの入口として、セッション管理・申請者情報収集・ルーティングを一元的に担当する

### 2.2 主な責務
- **申請種別判断**: ユーザーの自然言語入力から申請種別（JD-01/JD-02/JD-03）を判断し、担当エージェントを決定する（BRL-01/BRL-04）
- **申請者情報収集**: 申請者名を収集し、申請日（システム日付）と共に invocation_state に設定する
- **専門エージェントへの委譲**: 申請種別確定後、AG-002（交通費精算）または AG-003（経費精算）へ invocation_state 経由でセッション情報を渡して委譲する（LNK-001/LNK-002）
- **申請ルール照会の回答**: ナレッジベース（D-001）を参照し、照会内容に対応するルールを根拠付きで回答する（BRL-01/BRL-14）
- **セッション管理**: AG-001 起動時に session_id を生成し、FileBasedSessionManager でセッションファイルを作成・管理する

### 2.3 非責務（やらないこと）
- **申請書の生成**: AG-002/AG-003 が実施する（TOOL-002）。AG-001 は TOOL-002 の実行権限を持たない
- **移動情報・経費情報の収集**: AG-002/AG-003 が実施する
- **交通費の計算**: AG-002 が TOOL-001 経由で実施する

## 3. 自律度・判断権限

### 3.1 自律度レベル

| レベル | 内容 | 適用範囲 |
|------|------|----------|
| 参照 | 社内申請ルール（D-001）の参照 | 申請種別判断・申請ルール照会時 |
| 提案 | 申請種別の選択肢提示（BRL-04） | 申請種別が判断できない場合 |
| 実行 | 申請種別判断・AG-002/AG-003 への委譲 | 申請種別確定後の委譲（JD-01/BRL-04確定後） |

### 3.2 判断権限の範囲
- **自律判断可能**: 申請種別判断（JD-01）、申請ルール照会の回答（UC-07）、申請者名の収集タイミング
- **人間確認必要**: 申請種別が判断できない場合（JD-03/BRL-04）。社員に選択肢を提示して確認する
- **実行不可**: 申請書の生成（TOOL-002）、交通費の計算（TOOL-001）

## 4. 処理フロー設計

### 4.1 基本フロー（概要）

```text
開始（ユーザー入力受付）
  → 申請意図の分析（BRL-01）
  → 申請種別判断（JD-01/JD-02）
  → 申請種別確定
    → 交通費精算申請 → 申請者名収集 → AG-002に委譲（LNK-001） → 結果受取 → ユーザーへ提示 → 終了
    → 経費精算申請  → 申請者名収集 → AG-003に委譲（LNK-002） → 結果受取 → ユーザーへ提示 → 終了
    → 申請ルール照会 → D-001参照 → 根拠付き回答 → 終了
    → 判断不能       → BRL-04（選択肢提示） → ユーザー選択 → 再判断
```

**フローの詳細**: 詳細設計書 2.3.1章を参照

### 4.2 フロー制御の考え方
- **原則的な処理順序**: 申請種別判断 → 申請者名収集 → 委譲という逐次処理。申請種別が確定するまで委譲は行わない
- **処理をスキップする考え方**: 申請種別が1回の入力で明確に判断できる場合は BRL-04（選択肢提示）をスキップする
- **同一処理を複数回実行**: 申請種別が判断できない場合（JD-03）は確認ダイアログを最大3回実施。3回後も判断不能の場合は CF-009（管理部門エスカレーション）へ遷移する
- **対話回数上限チェック**: ユーザー入力回数が30回に達した場合、セッションを終了してユーザーに案内する（SlidingWindow(30) と合わせた上限値）。上限到達時は「対話回数の上限に達しました。管理部門（経理部）にお問い合わせいただくか、改めて最初からお試しください。」を提示する

### 4.3 状態管理
- **session_id**: AG-001 起動時に生成し FileBasedSessionManager が `data/sessions/session_{id}.json` で管理。会話終了まで保持
- **applicant_name**: ユーザーとの対話で収集後、session_{id}.json に保存。委譲時に invocation_state に設定
- **application_type**: 申請種別確定後に session_{id}.json に保存（invocation_state には含めない）

## 5. 専門エージェントとの連携設計

### 5.1 連携対象エージェント

| エージェントID | エージェント名 | 連携目的 | 振り分け条件 |
|---------------|---------------|----------|--------------|
| AG-002 | 交通費精算申請エージェント | 交通費精算申請書作成フロー全体を委譲する | 申請種別=「交通費精算申請」が確定（JD-01/BRL-04）かつ申請者名収集済み |
| AG-003 | 経費精算申請エージェント | 経費精算申請書作成フロー全体を委譲する | 申請種別=「経費精算申請」が確定（JD-01/BRL-04）かつ申請者名収集済み |

### 5.2 振り分け判断基準
- **AG-002（交通費精算）**: 「電車で行った」「交通費を申請したい」「出張交通費」など移動・交通費に関する意図（JD-01）
- **AG-003（経費精算）**: 「領収書がある」「備品を買った」「文房具」など購入・経費に関する意図（JD-01）
- **判断不能（BRL-04）**: 意図が不明または複数申請種別に該当する可能性がある場合。「交通費精算申請」「経費精算申請」の2択を提示してユーザーに選択させる

**詳細な判断基準**: 詳細設計書 2.3.2章を参照

### 5.3 invocation_stateの受け渡し
AG-002/AG-003 に以下の情報を `ToolContext.invocation_state` 経由で渡す（マルチエージェント連携設計.md 7.2準拠）。`application_type` は含めない（委譲先エージェントの選択により申請種別が決定される）。

```python
{
    "session_id": str,        # ファクトリ関数内で消費（子エージェントには渡さない）
    "applicant_name": str,    # 申請者名
    "application_date": date  # 申請日（システム日付: date.today()）
}
```

**データ構造の詳細**: `artifacts/04_basic-design/outputs/データモデル基本設計書.md`（InvocationStateModel）を参照

## 6. ツール設計

### 6.1 利用ツール
AG-001 は AG-002/AG-003 を Agent as Tools パターンでツールとして呼び出す。**AG-001 が使用するツールは AG-002/AG-003 のみとし、業務計算・生成系ツール（TOOL-001/TOOL-002）は直接使用しない**（各エージェントが使用するツールは業務機能に特化したツールのみとする方針に準拠）。

- **handle_transport_expense_application（AG-002）**: 交通費精算申請書作成フロー全体を委譲する。入力として invocation_state（セッション情報）を ToolContext 経由で渡す
- **handle_expense_application（AG-003）**: 経費精算申請書作成フロー全体を委譲する。入力として invocation_state（セッション情報）を ToolContext 経由で渡す

社内申請ルール（D-001）はナレッジファイル（`data/knowledge/`）として全文ロードし、システムプロンプトまたは会話コンテキストに含める形で参照する（独立したツール関数は設けない）。

**ツールの詳細**: 詳細設計書 2.4.2章および各エージェント基本設計書を参照

## 7. プロンプト設計

### 7.1 システムプロンプトの目的
- 申請種別の判断基準（交通費精算・経費精算・申請ルール照会）をエージェントに定義する
- 社内申請ルール（D-001）のみを根拠とした回答・判断を徹底させる（BRL-01/BRL-14）
- 判断不能時の選択肢提示手順（BRL-04）を定義する

### 7.2 プロンプト構成要素
- **役割定義**: 申請受付窓口エージェントとしての役割（申請種別判断・委譲・申請ルール照会）
- **処理フロー**: 申請意図受付 → 種別判断 → 委譲またはルール回答の手順
- **判断プロトコル**: JD-01（種別確定）、JD-02（照会判断）、JD-03（判断不能時の選択肢提示）の手順
- **主な責任**: BRL-01（社内申請ルールのみ参照）、BRL-04（判断不能時の2択提示）、BRL-14（推測・補完禁止）
- **動的情報**: なし（申請種別判断・ルール照会に日付・期限は不要）

**プロンプト全文**: 詳細設計書 2.3.1章を参照

### 7.3 システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 静的定数 |
| 動的生成の理由 | 不要（AG-001 の責務には日付・期限に依存する業務ルール判断が含まれない） |
| 動的に埋め込む情報 | なし |

### 7.4 主要な設計意図
- **静的定数**: AG-001 の役割（申請種別判断・委譲・申請ルール照会）に動的な日付情報は不要なため、実行のたびに関数を呼び出すコストを避ける
- **委譲先選択の明確化**: AG-002/AG-003 への委譲判断はシステムプロンプトに明記し、LLM の裁量で変更させない
- **根拠提示の徹底**: 申請ルール照会の回答には参照したルールID・内容を必ず提示する（BRL-01/BRL-14）

**プロンプト全文**: 詳細設計書 2.3.1章を参照

## 10. 入力データ設計

### 10.1 初期化時の入力
- **なし**: AG-001 は CLI からの起動時に特別な初期化パラメータを必要としない。session_id は AG-001 起動時にシステムが自動生成する

### 10.2 対話ループでの入力
- **申請意図テキスト（D-001）**: ユーザーが自然言語で入力する申請意図（例：「出張交通費を申請したい」「文房具の領収書があります」「申請期限を教えて」）
- **入力制限**: 500文字以内（FR-012/GRD-017）。超過時は EX-01 として再入力を促す
- **特殊コマンド**:

| コマンド | 動作 |
|---------|------|
| `reset` / `リセット` / `最初から` | 会話履歴と収集済み申請者情報をリセットし、最初からやり直す。セッションを再初期化して新規セッションを開始する |

### 10.3 invocation_stateの入力
AG-001 はオーケストレーターであり、invocation_state を受け取る側ではなく渡す側。入力として受け取る invocation_state はない。

## 11. 出力データ設計

### 11.1 出力形式
- **出力先**: 標準出力（CLIユーザーへの表示）
- **出力内容**: 申請種別案内文、申請ルール照会の回答（根拠付き）、委譲完了後の結果提示（申請書パス・チェック結果）

### 11.2 出力メッセージ
- 申請種別確定時: `"「{申請種別}」として受け付けました。{専門エージェント名}に引き継ぎます。"`
- 申請種別不明時（BRL-04）: `"申請種別を確認させてください。「交通費精算申請」または「経費精算申請」のどちらになりますか？"`
- 申請ルール照会回答時: `"ご照会の内容について、社内申請ルール（{ルールID}）に基づき回答いたします。{回答内容}"`
- 委譲完了後: `"申請書の作成が完了しました。{ファイルパス} に保存されています。"`

## 14. テスト観点

### 14.1 テスト観点の概要
- **機能テスト**: 申請種別判断（「電車で行った」→ AG-002、「領収書がある」→ AG-003）が正しく動作すること。申請ルール照会回答が D-001 に基づいて正確であること
- **異常系テスト**: 申請種別不明時に BRL-04 の選択肢が提示されること。500文字超過時に EX-01 が発生すること
- **性能テスト**: LoopControlHook により10回上限でループが終了すること

**詳細なテストケース**: 詳細設計書 7章を参照

## 16. 制約事項・前提条件

### 16.1 技術的制約
- Strands Agents SDK v1.25.0 の Agent as Tools パターンを使用すること
- Amazon Bedrock（us.anthropic.claude-sonnet-4-5-20251001）を LLM として使用すること
- AG-001 は TOOL-002（申請書生成）を tools リストに含めないこと（実行権限なし）

### 16.2 業務的制約
- 申請種別は「交通費精算申請」「経費精算申請」「申請ルール照会」の3種類のみ対応する（BRL-01/BRL-14）
- AG-001 は AG-002/AG-003 を同時に呼び出してはならない（同一セッション内の並列実行禁止）
- 申請ルール外の内容について推測・補完した回答を行ってはならない（BRL-14）

### 16.3 前提条件
- 社内申請ルール（D-001）が `data/knowledge/` ディレクトリに配置されていること
- AWS 認証情報（`AWS_ACCESS_KEY_ID`/`AWS_SECRET_ACCESS_KEY`/`AWS_REGION`）が環境変数に設定されていること
- `data/sessions/` ディレクトリが書き込み可能な状態で存在すること
