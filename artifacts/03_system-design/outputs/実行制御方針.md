## 1. 目的・適用範囲

### 1.1 目的

本方針では、エージェント・ツールの **「いつ実行するか」「どう制御するか」「失敗時にどうするか」** の共通方針を定義する。

個々のエージェント・ツール設計に **一貫した実行制御の判断軸** を提供する。

### 1.2 適用範囲

- マルチエージェントを含む全システム（AG-001, AG-002, AG-003）
- 全ツール（TOOL-001, TOOL-002）
- 共通ハンドラー（LoopControlHook, HumanApprovalHook, ErrorHandler, SessionManager）

### 1.3 本方針で定義しないこと

- 具体的な実行クラス・メソッド設計（基本設計書で定義）
- エージェント内部の推論ロジック
- ガードレールの詳細ルール（ガードレール要件定義参照）

## 2. 実行単位の定義

### 2.1 実行単位

| 実行単位 | 説明 | 制御対象 |
|---------|------|---------|
| セッション | 1申請フロー全体（申請開始〜申請書提示または中断・エラー終了） | SessionManager（SM-001） |
| エージェント呼び出し | エージェントへの1回の呼び出し（AG-001 / AG-002 / AG-003） | エージェントインスタンス |
| ReActループ | エージェント内の推論→ツール呼び出し→観察の繰り返しサイクル | LoopControlHook（HD-002） |
| ツール実行 | ツール関数1回の呼び出し（TOOL-001 / TOOL-002） | ツール関数 |
| 人間承認 | 申請書生成前の社員確認（OK / 修正 / キャンセル） | HumanApprovalHook（HD-003） |

### 2.2 実行単位の階層関係

```
セッション
└── エージェント呼び出し（AG-001）
    └── ReActループ × N（最大10回）
        └── ツール実行（AG-002 or AG-003 as tool）
            └── エージェント呼び出し（AG-002 / AG-003）
                └── ReActループ × N（最大10回）
                    ├── ツール実行（TOOL-001 / TOOL-002）
                    └── 人間承認（HumanApprovalHook）
```

## 3. 実行状態管理

### 3.1 実行状態一覧

| 状態 | 説明 | 設定タイミング |
|-----|------|-------------|
| READY | 実行待機中 | セッション開始前 |
| RUNNING | 実行中 | エージェント呼び出し開始時 |
| WAITING | 人間入力待ち | エージェントが社員の入力・承認を待つ時 |
| COMPLETED | 正常完了 | 申請書（下書き）提示完了時（CF-005完了） |
| FAILED | エラー発生 | ErrorHandler が例外を分類した時 |
| TERMINATED | 異常終了 | エスカレーション後フロー終了時・タイムアウト時 |

### 3.2 状態遷移ルール

| 遷移 | トリガー |
|-----|---------|
| READY → RUNNING | 社員がアクセスしてセッション開始（CF-001 Step 0） |
| RUNNING → WAITING | エージェントが社員入力を要求（HITL確認・選択肢提示など） |
| WAITING → RUNNING | 社員が入力・承認を返す |
| RUNNING → COMPLETED | CF-005 完了（申請書下書き提示） |
| RUNNING → FAILED | ErrorHandler が例外を受理 |
| FAILED → RUNNING | 再試行・代替処理により継続可能な場合 |
| FAILED → TERMINATED | 再試行・代替処理で収束不可・エスカレーション後 |
| RUNNING → TERMINATED | タイムアウト・LoopLimitError・セッションタイムアウト |

**状態管理責務**: SessionManager（SM-001）がセッション状態を管理する。エージェントの実行状態は ErrorHandler および LoopControlHook が連携して監視する。

## 4. 実行開始制御

### 4.1 エージェント実行開始条件

| エージェント | 開始条件 | 禁止条件 |
|------------|---------|---------|
| AG-001 | 社員がアクセスしてセッションが開始されていること | なし（エントリーポイント） |
| AG-002 | AG-001が申請種別=交通費精算申請を確定かつ申請者名取得済み（起動時に設定済み） | 申請者名未取得時の起動禁止（GRD-014） |
| AG-003 | AG-001が申請種別=経費精算申請を確定かつ申請者名取得済み（起動時に設定済み） | 申請者名未取得時の起動禁止（GRD-014） |

### 4.2 ツール実行開始条件

| ツール | 開始条件 | 禁止条件 |
|-------|---------|---------|
| TOOL-001（交通費計算） | AG-002が呼び出す。交通手段・区間が入力済み | バリデーション未通過時の実行禁止 |
| TOOL-002（申請書生成） | HumanApprovalHookの承認（OK）取得後 | 社員確認未取得の申請書生成禁止（GRD-014）・申請書提出操作の実行禁止（GRD-012 / ACT-EXEC-01） |

### 4.3 ガードレール適用タイミング

ガードレールは以下のタイミングでチェックを行う。

| タイミング | 適用ガードレール | 対象 |
|----------|---------------|------|
| 入力受付時 | GRD-001〜GRD-004（入力チェック） | AG-001 入力処理 |
| 申請種別判断後 | GRD-002（対応外交通手段拒否） | AG-001 → AG-002 委譲前 |
| 申請書生成前 | GRD-012（提出操作禁止）・GRD-014（確認未取得禁止） | HumanApprovalHook |
| 申請書出力時 | GRD-013〜GRD-015（出力内容・経路） | TOOL-002 |

## 5. 並列・逐次制御

### 5.1 並列実行方針

**本システムでは並列実行を行わない。**

- AG-002とAG-003の同一セッション内での同時起動は禁止（DEC-003）
- 1セッションで対応できる申請種別は1種類のみ
- ツール実行（TOOL-001/TOOL-002）はエージェント内の逐次実行のみ

### 5.2 逐次実行の順序

```
申請者名・申請日取得（アプリケーション起動時・対話ループ開始前に完了）
→ 申請内容テキスト受付（AG-001）
→ 申請種別確定（AG-001）
→ 専門エージェントへ委譲（AG-001 → AG-002 または AG-003）
→ 情報収集（AG-002 / AG-003）
→ バリデーション・計算（TOOL-001 ※交通費のみ）
→ 人間承認（HumanApprovalHook）
→ 申請書生成（TOOL-002）
→ 申請書提示（CF-005 完了）
```

## 6. 再試行方針

### 6.1 再試行ポリシー一覧

| 対象 | 最大試行回数 | 待機戦略 | 初期遅延 | 最大遅延 | 再試行可能条件 |
|-----|------------|---------|---------|---------|--------------|
| Amazon Bedrock（LLM呼び出し） | 6回 | 指数バックオフ | 4秒 | 240秒 | 接続失敗・タイムアウト（EX-07） |
| TOOL-001（交通費計算） | 6回 | 指数バックオフ | 実行制御方針に準拠 | 実行制御方針に準拠 | 参照のみ（冪等）のため再試行可能 |
| TOOL-002（申請書生成） | 1回 | 即時 | — | — | 生成失敗（EX-06）のみ。非冪等のため1回のみ |

### 6.2 再試行後の処理

| 対象 | 上限到達時の処理 |
|-----|--------------|
| Amazon Bedrock（LLM呼び出し） | ErrorHandler が EX-07 として分類し CF-006（エスカレーション案内）へ |
| TOOL-001（交通費計算） | 経路不明の場合は手動入力フローへ切替（EX-02）。それ以外は CF-006 へ |
| TOOL-002（申請書生成） | リトライ後もエラーの場合は ErrorHandler が EX-06 として分類し CF-006 へ |

### 6.3 再試行禁止ケース

- 入力例外（EX-01）：社員への再入力促しで対応（再試行不要）
- 業務ルール違反（EX-03）：ルール上の禁止事項のため再試行不可
- 実行不可（EX-05）：システム設計上の禁止操作のため再試行不可
- 想定外（EX-08）：LoopLimitError・予期しない例外は再試行不可

## 7. 中断・停止制御

### 7.1 正常中断

| 中断種別 | 発生条件 | 処理 |
|---------|---------|------|
| 社員によるキャンセル | CF-005 Step 2 でキャンセル選択 | セッション状態を CLOSED に遷移してフロー終了 |
| 申請書提示完了 | CF-005 完了 | セッション状態を CLOSED に遷移してフロー終了 |

### 7.2 異常停止

| 停止種別 | 発生条件 | 処理 |
|---------|---------|------|
| エスカレーション後停止 | EX-04〜EX-08 でエスカレーション完了 | セッション状態を TERMINATED に遷移 |
| 申請期限超過 | BRL-14：経費発生日から90日超過（EX-03） | フロー終了・セッション TERMINATED |
| LoopLimitError | ReActループ上限（10回）到達 | LoopControlHook が検知→ ErrorHandler が EX-08 として処理→ CF-006 へ |
| 対話ターン上限 | セッションあたり30ターン到達 | セッション TERMINATED |

### 7.3 緊急停止

**最高優先ガードレール（GRD-012）**: 申請書提出操作の実行試行を検知した場合、即時実行禁止。ユーザーに対応方法（手動提出）を案内する。

## 8. タイムアウト制御

### 8.1 タイムアウト設定

| 対象 | タイムアウト値 | 設定場所 |
|-----|-------------|---------|
| セッションタイムアウト | 要件上未定義 | 運用要件確定後に定義する |
| Bedrock APIタイムアウト | 要件上未定義 | 実装時にboto3クライアント設定で定義する |
| ツール実行タイムアウト | 要件上未定義 | 実装時に定義する |

> ※ タイムアウト値の具体的な設定は運用要件確定後に基本設計書で定義する（U-001）

### 8.2 タイムアウト発生時の処理

- セッションタイムアウト発生時: セッション状態を TERMINATED に遷移させる
- APIタイムアウト発生時: EX-07（システム障害）として ErrorHandler に委譲し、再試行方針に従う

## 9. ループ制御

### 9.1 LoopControlHook の責務

LoopControlHook（HD-002）は全エージェントに適用し、ReActループの上限を制御する。

**登録イベント**:
- `BeforeInvocationEvent`: エージェント呼び出し開始時にループカウンタを0にリセットする
- `AfterModelCallEvent`: LLM呼び出し後にループカウンタをインクリメントし、上限（10回）到達を監視する

> ※ LoopControlHook の具体的な実装方針（メソッド名・引数・戻り値）は基本設計書（ハンドラー基本設計.md）で定義する。

### 9.2 ループ制御設定

| 設定項目 | 値 | 適用対象 |
|---------|---|---------|
| 最大ループ回数（max_iterations） | 10回 | AG-001, AG-002, AG-003（全エージェント共通） |
| LoopLimitError 到達時の処理 | EX-08 として ErrorHandler に委譲→ CF-006 へ | 全エージェント |

### 9.3 ループ回数のカウント方針

- ReActループの1回 = 推論1回 + ツール呼び出し1回 + 観察1回 のサイクル
- エージェント委譲（AG-001 → AG-002/AG-003）はAG-001のループカウントに含まれる
- AG-002/AG-003 のループカウントは独立してカウントする

## 10. 人間承認制御

### 10.1 HumanApprovalHook の責務

HumanApprovalHook（HD-003）はAG-002・AG-003の申請書生成前（TOOL-002実行前）に介入し、社員確認を制御する。

> ※ HumanApprovalHook の具体的な実装方針（メソッド名・引数・戻り値・フック発動タイミング）は基本設計書（ハンドラー基本設計.md）で定義する。

### 10.2 適用対象

| エージェント | HumanApprovalHook | 理由 |
|------------|------------------|------|
| AG-001 | 適用しない | 申請書生成は担当しない |
| AG-002 | 適用する | TOOL-002 実行前の社員確認が必要（GRD-014） |
| AG-003 | 適用する | TOOL-002 実行前の社員確認が必要（GRD-014） |

### 10.3 承認フロー

1. AG-002/AG-003 が TOOL-002 を呼び出そうとする直前に HumanApprovalHook が介入
2. 収集済み申請情報（D-010 の全フィールド）を社員に提示
3. 社員が OK / 修正 / キャンセル を選択
4. 選択結果に応じて処理分岐:

| 選択 | 処理 |
|-----|------|
| OK | TOOL-002 実行（申請書生成）へ進む。AUD-004 を LOG-HI に記録 |
| 修正 | 修正対象フィールドの対話ステップ（CF-003/CF-004）に戻る |
| キャンセル | セッション状態を CLOSED に遷移してフロー終了。AUD-004 を LOG-HI に記録 |

## 11. エラー伝播の責務分担

### 11.1 責務チェーン

| 層 | 責務 |
|---|------|
| 各モジュール（エージェント・ツール） | 例外発生箇所でログを出力し、ErrorHandler へ処理を委譲する |
| ErrorHandler | ユーザー向けメッセージを生成して呼び出し元に返す |
| 呼び出し元 | 受け取ったメッセージを戻り値または表示として上位レイヤーへ伝播させる |

### 11.2 伝播の原則

- ツール関数は例外を再送出しない。`{"success": False, "message": エラーメッセージ}` を返す
- エージェントは ErrorHandler から受け取ったメッセージを社員向けレスポンスとして返す
- `main.py`（エントリーポイント）は最終的なエラーメッセージを社員に表示する

## 12. エスカレーション連携

エスカレーション判定は ErrorHandler（HD-001）が行い、CF-006（エラー・エスカレーション案内フロー）に遷移する。

**エスカレーション対象の実行例外**:

| 例外分類 | トリガー | エスカレーション先 |
|---------|---------|----------------|
| EX-04（外部I/F例外）：ナレッジ/テンプレート参照失敗 | ツール実行中のファイル参照失敗 | CF-006 → 担当部門（管理部門） |
| EX-06（データ不整合）：申請書生成失敗（リトライ後） | TOOL-002 のリトライ後も失敗 | CF-006 → 担当部門（管理部門） |
| EX-07（システム障害）：Bedrock接続失敗（リトライ上限後） | LLM呼び出しのリトライ上限到達 | CF-006 → 担当部門（管理部門） |
| EX-08（想定外）：LoopLimitError・予期しない例外 | LoopControlHook の LoopLimitError 検知 | CF-006 → 担当部門（管理部門） |

## 13. セッション管理連携

**実行制御とセッション状態の連携**:

| 実行イベント | セッション状態更新 | 責務 |
|-----------|---------------|------|
| 対話ループ開始 | CREATED → ACTIVE | SM-001 |
| エージェントが社員入力を要求 | ACTIVE → WAITING | SM-001 |
| 社員が入力を返す | WAITING → ACTIVE | SM-001 |
| CF-005 完了（申請書提示） | ACTIVE → CLOSED | SM-001 |
| エスカレーション後フロー終了 | ACTIVE → TERMINATED | SM-001 |
| LoopLimitError 発生 | ACTIVE → TERMINATED | SM-001 |
| 対話ターン上限（30回）到達 | ACTIVE → TERMINATED | SM-001 |

**ターンカウント管理**:
- turn_count は SM-001 が管理し、エージェントが対話ターンを消費するたびに更新する
- turn_count が30に達した時点でセッションを TERMINATED に遷移させる

## 16. 設計上の留意点・リスク

**LoopLimitError の扱い**:
- LoopControlHook が LoopLimitError を検知した後、ErrorHandler が CF-006 に誘導する
- LoopLimitError の `__cause__` を判定して詳細原因を特定する（設計標準参照）
- スタックトレースはログ（LOG-HI）にのみ記録し、社員には表示しない

**TOOL-002 の非冪等性リスク**:
- 申請書生成は1回しかリトライしない（2回以上の同一申請書生成を防ぐ）
- リトライ後も失敗した場合は ErrorHandler が EX-06 として分類し、CF-006 へエスカレーション

**人間承認のスキップリスク**:
- HumanApprovalHook の未適用により GRD-014 を違反するリスク
- 対策: AG-002/AG-003 の全インスタンスに HumanApprovalHook を必須適用し、TOOL-002 の直前に必ず介入する

**タイムアウト未定義リスク**:
- セッションタイムアウト・APIタイムアウトが要件上未定義のため、実装時に適切な値を設定する必要がある
- 対策: 運用要件確定後に基本設計書でタイムアウト値を定義し、本方針を更新する（U-001）
