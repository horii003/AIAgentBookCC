## 2. 目的・役割定義

### 2.1 コンポーネントの目的
- システム全体で使用する Pydantic v2 データモデルを一箇所（`agents/models/data_models.py`）に集約し、バリデーションロジックの重複定義を防ぐ
- ツール入出力・エージェント間連携（invocation_state）・セッション状態・申請情報の各データ構造を型安全に定義する
- バリデーション方針.md に定義されたバリデーションルール（日付・金額・文字列・列挙型・駅名正規化）を Pydantic モデルとして実装し、各コンポーネントから再利用可能にする

### 2.2 主な責務
- **ツール入出力モデルの定義**: TOOL-001（交通費計算）および TOOL-002（申請書生成）の入出力データ構造を Pydantic v2 BaseModel で定義する
- **申請情報モデルの定義**: エージェントが収集・保持する移動情報（TransportItem）および経費情報（ExpenseItem）のデータ構造を定義する
- **invocation_state モデルの定義**: AG-001 から AG-002/AG-003 へ引き継ぐ共有コンテキスト（InvocationStateModel）のスキーマを定義する
- **セッション状態モデルの定義**: FileBasedSessionManager が session_{id}.json に永続化するセッション状態（SessionStateModel）のスキーマを定義する
- **共通バリデーターの実装**: 駅名正規化（BRL-18）・申請期限チェック（BRL-11）・高額判定（BRL-12）等の業務バリデーションを field_validator / model_validator として実装する

### 2.3 非責務（やらないこと）
- **ファイルI/O処理**: TOOL-001/TOOL-002 が実施する。本モジュールはデータ構造定義のみ担当する
- **エラーハンドリング**: ErrorHandler（HD-001）が実施する。バリデーションエラー時は ValidationError を送出するのみ
- **LLM推論・ツール実行**: エージェントおよびツールが実施する。データモデルは受け渡しの型定義のみ担当する

### 2.4 必要性
- 複数のエージェント（AG-001/AG-002/AG-003）とツール（TOOL-001/TOOL-002）が同一データ構造を参照するため、単一定義ファイルへの集約が必要
- Pydantic v2 の型安全性により、LLM 出力の表記揺れ（「渋谷駅」→「渋谷」等）を field_validator で自動正規化し、後続処理の信頼性を確保する
- バリデーション方針.md の方針（早期検証・型安全性・自動正規化・一貫性）を実現するための共通基盤として必要

### 2.5 主要な設計意図
- **単一定義・集中管理**: `agents/models/data_models.py` に全モデルを集約し、各コンポーネントで重複定義しない（バリデーション方針.md 2.1）
- **Pydantic v2 標準設定**: 全モデルに `model_config = ConfigDict(str_strip_whitespace=True)` を適用し、文字列の前後空白を自動除去する
- **invocation_state スキーマの最小化**: マルチエージェント連携設計.md に従い `applicant_name` と `application_date` のみを含む。`application_type` は委譲先エージェントの選択で決定されるため含めない

## 4. 共通バリデーター

### 4.1 normalize_station_name（駅名正規化）
- **目的**: BRL-18 に基づき、駅名末尾の「駅」を除去して正規化する（例：「渋谷駅」→「渋谷」）
- **適用フィールド**: `TransportFareInput.departure`, `TransportFareInput.destination`, `TransportItem.departure`, `TransportItem.destination`
- **引数**: `v: str`（駅名文字列）
- **戻り値**: 末尾「駅」を除去した文字列。末尾が「駅」でない場合はそのまま返す
- **例外**: 送出なし（正規化のみ実施）

### 4.2 normalize_transport_type（交通手段正規化）
- **目的**: LLM が生成する交通手段の表記揺れ（英語・略称等）を正規値に正規化する
- **適用フィールド**: `TransportFareInput.transport_type`, `TransportItem.transport_type`
- **引数**: `v: str`（交通手段文字列）
- **戻り値**: 正規化後の文字列（「電車」「バス」「タクシー」「飛行機」のいずれか）
- **マッピング**: `{"train": "電車", "鉄道": "電車", "bus": "バス", "taxi": "タクシー", "cab": "タクシー", "airplane": "飛行機", "plane": "飛行機"}`

### 4.3 check_application_deadline（申請期限チェック）
- **目的**: BRL-11 に基づき、経費発生日（移動日・購入日）から申請日までが90日以内であることを確認する
- **適用モデル**: `TransportItem`（travel_date と application_date のクロスフィールド）、`ExpenseItem`（purchase_date と application_date のクロスフィールド）
- **引数**: モデルインスタンス（model_validator として実装）
- **戻り値**: バリデーション通過時はモデルインスタンスをそのまま返す
- **例外**: 90日超過時に `ValueError`（バリデーション方針.md 4.3 のメッセージフォーマット準拠）

### 4.4 normalize_application_type（申請種別正規化）
- **目的**: LLM が生成する申請種別の略称を正規値に正規化する
- **適用フィールド**: `SessionStateModel.application_type`
- **引数**: `v: str`
- **戻り値**: 正規化後の文字列（「交通費精算申請」「経費精算申請」のいずれか）
- **マッピング**: `{"交通費": "交通費精算申請", "交通費申請": "交通費精算申請", "経費": "経費精算申請", "経費申請": "経費精算申請"}`

**詳細**: 詳細設計書 4章を参照

## 5. 主要モデルの概要

### 5.1 TransportFareInput
- **目的**: TOOL-001（calculate_transport_fare）の入力パラメータバリデーション
- **フィールド**:
  - `transport_type: Literal["電車", "バス", "タクシー", "飛行機"]` — 交通手段（BRL-17 正規化後）
  - `departure: str` — 出発地（BRL-18 正規化後の駅名・地名。min_length=1）
  - `destination: str` — 目的地（BRL-18 正規化後の駅名・地名。min_length=1）
  - `travel_date: date` — 移動日（YYYY-MM-DD 形式、必須）。申請期限チェック（BRL-11）でツール内部から利用する
- **バリデーション**: `normalize_transport_type`（mode="before"）、`normalize_station_name`（departure・destination に適用）、`ConfigDict(str_strip_whitespace=True)`

### 5.2 TransportFareOutput
- **目的**: TOOL-001 の返却データ構造（AG-002 が結果を解釈する際の型定義）
- **フィールド**:
  - `fare: Optional[int]` — 計算済み運賃（円単位）。計算不可の場合は None
  - `calculable: bool` — 自動計算可否フラグ（True=計算成功、False=経路未存在等）
- **バリデーション**: `fare` が None でない場合は `gt=0`（1円以上）

### 5.3 TransportItem
- **目的**: AG-002 が1区間分の移動情報を収集・保持するモデル（D-013の1件に相当）
- **フィールド**:
  - `travel_date: date` — 移動日（YYYY-MM-DD）
  - `departure: str` — 出発地（BRL-18 正規化済み。min_length=1）
  - `destination: str` — 目的地（BRL-18 正規化済み。min_length=1）
  - `transport_type: Literal["電車", "バス", "タクシー", "飛行機"]` — 交通手段
  - `fare: int` — 費用（円単位。gt=0）
  - `business_purpose: str` — 業務目的（BRL-13。min_length=1）
  - `application_date: date` — 申請日（申請期限チェック用。invocation_state から取得）
- **バリデーション**: `normalize_station_name`、`normalize_transport_type`、`check_application_deadline`（model_validator、BRL-11）

### 5.4 ExpenseItem
- **目的**: AG-003 が1件分の経費情報を収集・保持するモデル（D-007の1件に相当）
- **フィールド**:
  - `purchase_date: date` — 購入日（YYYY-MM-DD）
  - `amount: int` — 金額（円単位。gt=0）
  - `store_name: str` — 店舗名（min_length=1）
  - `item_name: str` — 品目（min_length=1）
  - `expense_category: Literal["事務用品費", "宿泊費", "資格精算費", "その他経費"]` — 経費区分（BRL-09）
  - `business_purpose: str` — 業務目的（BRL-13。min_length=1）
  - `application_date: date` — 申請日（申請期限チェック用。invocation_state から取得）
- **バリデーション**: `check_application_deadline`（model_validator、BRL-11）、`amount` は高額判定閾値（5,000円超）の確認は AG-003 が処理（バリデーションでブロックしない）

### 5.5 TransportApplicationInput
- **目的**: TOOL-002（generate_transport_application）の入力パラメータバリデーション
- **フィールド**:
  - `applicant_name: str` — 申請者名（invocation_state から取得。min_length=1）
  - `application_date: date` — 申請日（invocation_state から取得）
  - `items: List[TransportItem]` — 移動情報リスト（1件以上必須）
- **バリデーション**: `items` の要素数チェック（min_length=1）、BRL-07（全必須項目収集済み確認）

### 5.6 ExpenseApplicationInput
- **目的**: TOOL-002（generate_expense_application）の入力パラメータバリデーション
- **フィールド**:
  - `applicant_name: str` — 申請者名（invocation_state から取得。min_length=1）
  - `application_date: date` — 申請日（invocation_state から取得）
  - `items: List[ExpenseItem]` — 経費明細リスト（1件以上必須）
- **バリデーション**: `items` の要素数チェック（min_length=1）、BRL-07（全必須項目収集済み確認）

### 5.7 ApplicationFormOutput
- **目的**: TOOL-002 の返却データ構造（AG-002/AG-003 が結果を解釈する際の型定義）
- **フィールド**:
  - `success: bool` — 生成結果フラグ（True=成功、False=失敗）
  - `file_path: Optional[str]` — 生成済みファイルパス（成功時のみ）
  - `message: Optional[str]` — エラーメッセージ（失敗時のみ）
- **バリデーション**: `success=True` のとき `file_path` は必須

### 5.8 InvocationStateModel
- **目的**: AG-001 から AG-002/AG-003 へ ToolContext 経由で渡す invocation_state のバリデーション（マルチエージェント連携設計.md 7.2準拠）
- **フィールド**:
  - `applicant_name: str` — 申請者名（min_length=1）
  - `application_date: date` — 申請日（システム日付を自動取得）
- **バリデーション**: `ConfigDict(str_strip_whitespace=True)`。`application_type` は含めない（委譲先エージェントの選択で決定）

### 5.9 SessionStateModel
- **目的**: FileBasedSessionManager が session_{id}.json を読み書きする際のバリデーション
- **フィールド**:
  - `session_id: str` — セッションID（`{YYYYMMDD}_{HHMMSS}_{UUID4前8桁}` 形式）
  - `status: Literal["CREATED", "ACTIVE", "WAITING", "CLOSED", "TERMINATED"]` — セッション状態
  - `applicant_name: Optional[str]` — 申請者名（収集後に設定。min_length=1）
  - `application_date: Optional[date]` — 申請日（収集後に設定）
  - `application_type: Optional[Literal["交通費精算申請", "経費精算申請"]]` — 申請種別（確定後に設定）
  - `approval_granted: bool` — APR-001 承認済みフラグ（default=False）
  - `created_at: datetime` — セッション作成日時（ISO8601）
  - `updated_at: datetime` — 最終更新日時（ISO8601）
- **バリデーション**: `normalize_application_type`（application_type に適用）

**詳細**: 詳細設計書 3.1章〜3.9章を参照

## 6. バリデーションルール

**システム設計書の方針**: システム設計書 バリデーション方針.md（`artifacts/03_system-design/outputs/バリデーション方針.md`）を参照してください。

### 6.1 日付バリデーション
- **フォーマット**: `YYYY-MM-DD`（ISO 8601）。Pydantic が `date` 型に自動変換する
- **例**: OK: `2026-05-01`、NG: `2026/05/01`（スラッシュ区切りは ValidationError）
- **申請期限チェック（BRL-11）**: `application_date - travel_date（または purchase_date）> 90日` の場合は ValidationError（90日は `relativedelta(days=90)` で計算）

### 6.2 交通手段バリデーション
- **許可される値**: `Literal["電車", "バス", "タクシー", "飛行機"]`（BRL-17）
- **正規化**: `mode="before"` の `field_validator` で英語・略称を正規値に変換してから Literal 検証を実施する

### 6.3 金額バリデーション
- **制約**: `int`（円単位）、`gt=0`（1円以上）
- **範囲**: 上限なし（高額判定は警告のみ、申請をブロックしない）
- **高額判定閾値**: 交通費 10,000円超（JD-10/BRL-12）、経費 5,000円超（JD-11/BRL-12）。エージェントが案内メッセージを提示する（ValidationError ではない）

### 6.4 文字列バリデーション
- **制約**: `min_length=1`（空文字禁止）、`ConfigDict(str_strip_whitespace=True)`（前後空白自動除去）
- **最大長**: 申請者名・業務目的・店舗名・品目は要件上未定義（未定義のため上限なし）
- **CLI 入力**: 500文字上限（FR-012/GRD-017）は AG-001 入力受付時にチェック（本モデルには含めない）

**詳細**: 詳細設計書 5章を参照

## 7. マルチエージェント連携時の扱い

### 7.1 データモデルの共有
- 全モデルは `agents/models/data_models.py` に集約し、各エージェント・ツールが `from agents.models.data_models import ...` でインポートして使用する
- モデル定義の変更は全コンポーネントに影響するため、フィールド変更時は全利用箇所のテストを実施すること
- 型安全性は Pydantic v2 の型アノテーションにより実行時（モデル生成時）に保証される

### 7.2 データの受け渡し
- **エージェント間（invocation_state）**: AG-001 が `InvocationStateModel` でバリデーション後、`tool_context.invocation_state` に辞書形式（`.model_dump()` 結果）でセットする。AG-002/AG-003 は受け取った辞書を `InvocationStateModel(**state)` でパースして使用する
- **エージェント → ツール（入力）**: AG-002/AG-003 が Pydantic モデルを生成してからツール関数に渡す。ツール関数内でも同一モデルでバリデーションを実施する（二重チェック）
- **ツール → エージェント（出力）**: ツール関数は `TransportFareOutput` / `ApplicationFormOutput` のインスタンス、またはエラー時は `{"success": False, "message": str}` の辞書形式で返却する

## 9. 詳細設計への引き渡し事項

### 9.1 実装時の識別子
- **モジュール名**: `agents/models/data_models.py`

### 9.2 SDK実装時の注意事項
- Pydantic v2 の `BaseModel` を継承し、`model_config = ConfigDict(str_strip_whitespace=True)` を全モデルに設定すること（バリデーション方針.md 3.1）
- `field_validator` は `mode="before"` で正規化処理を実施し、その後の型チェック（Literal 等）に反映させること
- `model_validator` は `mode="after"` でクロスフィールドバリデーション（申請期限チェック等）を実施すること
- `python-dateutil` の `relativedelta(days=90)` を使用して90日計算を実装すること（バリデーション方針.md 4.3）
- `TransportFareOutput` / `ApplicationFormOutput` の `fare` / `file_path` は `Optional` として定義し、`None` と値を明確に区別すること

### 9.3 モック／スタブ要否
- 単体テストでは本モジュール自体をモックせず、実際の Pydantic モデルを使用してバリデーション動作を検証すること
- 日付依存テスト（申請期限チェック）では `date.today()` を `freezegun` または `pytest-mock` でモックして境界値テストを実施すること

## 11. 制約事項・前提条件

### 11.1 技術的制約
- Pydantic v2+ を使用すること（v1 の互換性は保証しない）
- `python-dateutil` ライブラリを申請期限チェックに使用すること
- `Optional[X]` は `X | None` の型エイリアスとして Python 3.10+ で使用可能だが、互換性のため `Optional[X]` 記法を使用する

### 11.2 業務的制約
- `InvocationStateModel` には `application_type` を含めない（マルチエージェント連携設計.md D-002 の決定事項）
- 高額申請の警告（BRL-12）は ValidationError でブロックせず、エージェントが案内メッセージを提示する設計とする
- `TransportItem.fare` は TOOL-001 の計算結果または社員手動入力値を設定する（JD-16）

### 11.3 前提条件
- Pydantic v2 の `ConfigDict` および `field_validator`、`model_validator` が利用可能な環境であること
- 全モデルは `agents/models/data_models.py` に定義し、他ファイルへの分散定義は行わないこと
- バリデーション方針.md 未決事項（U-001：文字列フィールドの最大長）は要件上未定義のため、詳細設計フェーズで確定する
