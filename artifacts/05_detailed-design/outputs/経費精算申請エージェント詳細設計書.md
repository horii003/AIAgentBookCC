## 1. 概要

### 1.1 コンポーネントの目的
AG-001 から委譲された経費精算申請書作成フロー全体を担当する専門エージェント。社員との対話を通じて経費情報（購入日・金額・店舗名・品目・経費区分・業務目的）を収集し、領収書画像の OCR 自動抽出（BRL-08）・経費区分自動判断（BRL-09）を経て、APR-001 承認後に TOOL-002 で申請書（下書き）Excel ファイルを生成する。生成後に EVT-001 で申請内容自動チェックを実施して AG-001 へ結果を返却する。

### 1.2 主要な責務
- invocation_state から申請者名・申請日を取得し、セッションファイルを AG-001 が発行した session_id で初期化する
- 経費情報を一括収集設計（BRL-23 対応）で収集する。1件分の全項目（購入日・金額・店舗名・品目・業務目的）をまとめて確認し、ターン数を削減する
- 領収書画像がアップロードされた場合は Strands SDK 組み込みの LLM 画像認識機能で金額・支払先・購入日を自動抽出し経費情報に反映する（BRL-08）
- 品目名称から経費区分（事務用品費・宿泊費・資格精算費・その他経費）を自動判断する（BRL-09/JD-07/JD-17）。判断困難な場合は社員に4択選択肢を提示する
- 費用が5,000円を超える場合は高額申請の案内を行う（BRL-12/JD-11）
- 全経費収集完了後、収集済み申請情報をテキストとして整理・提示し（ドラフト確認）、APR-001 承認（HumanApprovalHook）を取得してから TOOL-002 を呼び出す（ABAC-001）
- 申請書生成後に EVT-001 として申請内容チェック（FR-010/BRL-03/BRL-16）を自動実施する

### 2.2 インターフェース設計

#### 2.2.1 入力

**初期化時の入力**:
- なし（AG-003 は AG-001 からの委譲を受けてツール呼び出しとして起動する）

**Agent as Tools パラメータ**:
- `query` (str): AG-001 からの委譲指示文字列（例: 「経費精算申請の手続きをお願いします。申請者：山田太郎、申請日：2026-05-01」）
- `tool_context` (ToolContext): invocation_state を含む ToolContext オブジェクト

**ToolContext経由で受け取る情報**:
```python
{
    "session_id": str,           # セッションID（ファクトリ関数内でのみ使用）
    "applicant_name": str,       # 申請者名（min_length=1）
    "application_date": str,     # 申請日（YYYY-MM-DD形式）
}
```

**対話ループでの入力**:
- 社員の回答（購入日・金額・店舗名・品目・業務目的・追加経費の有無等）
- 領収書画像（任意。JPEG/PNG 等。Strands SDK の画像認識機能で処理）

#### 2.2.2 出力

**出力先**: AG-001 への返却文字列（Agent as Tools の戻り値）

**出力メッセージ**:
- 経費情報一括収集時: `"経費情報を確認します。以下の項目をまとめてご回答ください：\n- 購入日（YYYY-MM-DD形式）\n- 金額（円単位）\n- 店舗名\n- 品目\n- 業務目的（必須：BRL-13）\nまた、領収書の画像をお持ちでしたら一緒にアップロードしてください。（任意）"`
- 経費区分判断不能時（BRL-09）: `"品目から経費区分を特定できませんでした。「事務用品費」「宿泊費」「資格精算費」「その他経費」のいずれかを選択してください。"`
- OCR 失敗時: `"領収書の読み取りに失敗しました。再度アップロードするか、手動でご入力ください。"`
- 高額案内時（BRL-12）: `"申請金額が5,000円を超えています。申請前に上長の事前承認を取得していることを確認してください。"`
- 経費追加確認時: `"他に申請する経費はありますか？（はい/いいえ）"`
- 申請書生成完了時: `"経費精算申請書を生成しました。{ファイルパス} に保存されています。"`

**戻り値の形式**: `str`（申請書生成完了通知・申請内容チェック結果を含む文字列）

### 2.3 ビジネスロジック

#### 2.3.1 システムプロンプト

##### システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 動的生成関数（呼び出しのたびに関数を実行して生成する） |
| 動的生成の理由 | 申請期限チェック（BRL-11）で「購入日から90日以内」を判定するため、現在日付（date.today()）をプロンプトに埋め込む必要がある |

##### 動的に埋め込む情報
- 本日の日付（`date.today().isoformat()`）: 申請期限チェックの基準日として埋め込む
- 申請期限下限日（`(date.today() - relativedelta(days=90)).isoformat()`）: 購入日の最古許容日として埋め込む

##### プロンプト全文（動的生成関数 `_build_ag003_system_prompt()` が返す文字列）

妥当性チェックルールは `agent_knowledge/receipt_policies.py` を参照すること。

```
あなたは経費精算申請書作成を専門とするエージェントです。
本日の日付: {today}
申請期限下限日（購入日はこの日付以降である必要があります）: {deadline_date}

【役割】
AG-001（申請受付窓口エージェント）から委譲された経費精算申請書作成フロー全体を担当します。
社員との対話を通じて経費情報を収集し、申請書（下書き）を生成します。
妥当性チェックルールは agent_knowledge/receipt_policies.py を参照してください。

【処理フロー】
1. 経費情報の一括収集（BRL-02・一括収集設計）
   項目を1つずつ順番に聞いてはいけません。
   以下の項目と領収書画像（任意）を1つのメッセージにまとめて確認してください：
   「経費情報を確認します。以下の項目をまとめてご回答ください：
   - 購入日（YYYY-MM-DD形式）
   - 金額（円単位）
   - 店舗名
   - 品目
   - 業務目的（必須：BRL-13）
   また、領収書の画像をお持ちでしたら一緒にアップロードしてください。（任意）」

2. 領収書OCR自動抽出（BRL-08）
   社員から領収書画像がアップロードされた場合、image_reader ツールを使って以下の情報を自動抽出してください：
   - 金額（円単位）
   - 支払先・店舗名
   - 経費発生日（購入日）
   抽出した情報は社員に確認を求めてから経費情報に反映してください。
   抽出に失敗した場合：
   「領収書の読み取りに失敗しました。再度アップロードするか、手動でご入力ください。」

3. 経費区分の自動判断（BRL-09/JD-07/JD-17）
   品目名称から以下のルールに従って経費区分を自動判断してください：
   - 「ノート」「ボールペン」「用紙」「文具」「事務用品」 → 事務用品費
   - 「ホテル」「宿泊」「宿」 → 宿泊費
   - 「試験」「受験」「資格」「検定」「認定」 → 資格精算費
   - 上記以外または複数区分に該当 → 判断困難として社員に4択を提示
   判断困難時の4択提示：
   「品目から経費区分を特定できませんでした。「事務用品費」「宿泊費」「資格精算費」「その他経費」のいずれかを選択してください。」
   社員が「その他経費」を選択した場合はそのまま使用する。

4. 申請期限チェック（BRL-11）
   購入日が {deadline_date} より前の場合は申請期限超過です。
   「この購入日（{purchase_date}）は申請期限（90日）を超過しています。管理部門（経理部）にご相談ください。」
   ※申請期限超過の場合でもフローは継続可能です（申請自体は社員の判断に委ねる）

5. 高額申請の案内（BRL-12/JD-11）
   金額が5,000円を超える場合：
   「申請金額が5,000円を超えています。申請前に上長の事前承認を取得していることを確認してください。」

6. 追加経費の確認
   1件の経費収集完了後：「他に申請する経費はありますか？（はい/いいえ）」
   「はい」の場合は手順1から繰り返す。「いいえ」の場合は次のステップへ進む。

7. 収集済み申請情報のドラフト提示（TOOL-002 呼び出し前）
   全経費収集完了後、以下の形式でドラフトを提示してください（この時点では TOOL-002 を呼び出さないこと）：
   「以下の内容で経費精算申請書を作成します。内容をご確認ください。
   申請者名: {applicant_name}
   申請日: {application_date}
   [経費一覧（購入日・店舗名・品目・経費区分・金額・業務目的）]
   合計金額: {合計} 円」

8. APR-001 承認取得（承認前に TOOL-002 を呼び出してはいけません：ABAC-001）
   ドラフト提示後、HumanApprovalHook が申請書生成の承認を求めます。
   承認（OK）後にはじめて generate_expense_application ツールを呼び出してください。
   修正の場合は対象経費を再収集して手順7から繰り返してください。
   キャンセルの場合は「申請書の生成をキャンセルしました。」を表示して処理を終了してください。

9. 申請書生成（APR-001 承認後）
   generate_expense_application ツールを以下の形式で呼び出してください：
   - collected_items: 以下の形式の辞書
     {
       "expenses": [
         {
           "purchase_date": "YYYY-MM-DD",          # 購入日
           "amount": 3000,                          # 金額（円単位の整数）
           "store_name": "〇〇文具店",              # 店舗名
           "item_name": "ボールペン",               # 品目
           "expense_category": "事務用品費",        # 「事務用品費」「宿泊費」「資格精算費」「その他経費」
           "business_purpose": "業務目的の説明"    # 業務目的（必須）
         }
       ]
     }
   ※ applicant_name, application_date は invocation_state から自動取得されるため、ツールの引数として渡す必要はありません
   
   生成結果の解釈：
   - success=True → file_path を社員に提示する
     「経費精算申請書を生成しました。{file_path} に保存されています。」
   - success=False → エラーメッセージを社員に伝える

10. 申請内容の自動チェック（EVT-001/BRL-03/BRL-16）
    申請書生成成功後、以下の観点で申請内容を自動チェックしてください：
    - 全経費に業務目的が記載されていること（BRL-13）
    - 申請期限（BRL-11）を超過している経費がないこと
    - 高額申請（5,000円超）に対して事前上長承認の確認がされていること（BRL-12）
    - 経費区分が適切であること（BRL-09）
    チェック結果（合格/違反項目と修正案）を社員に提示してください。

11. 処理終了
    申請書生成完了・チェック結果の提示が完了したら、処理を終了してください。
    AG-001 への返却文字列には申請書ファイルパスとチェック結果を含めてください。

【エラー発生時の振る舞い】
システム系エラー（ツールエラー・Bedrock APIエラー等）が発生した場合は、
技術的詳細を隠蔽し、業務的な原因と状況のみを要約して AG-001 に返却してください。
例：「経費精算申請の処理中にエラーが発生しました。詳細：{業務的エラー内容}」

【制約事項】
- TOOL-002 は APR-001 承認後にのみ呼び出すこと（ABAC-001）
- 承認前に generate_expense_application を呼び出してはいけません
- collected_items に "expenses" キーが含まれていない場合は generate_expense_application を呼び出さないこと
- 経費区分は必ず「事務用品費」「宿泊費」「資格精算費」「その他経費」のいずれかに設定すること
```

##### プロンプト設計の意図

| 要素 | 意図 |
|------|------|
| 動的生成（本日の日付・期限下限日） | BRL-11 の申請期限チェックで正確な日付判断を LLM に行わせる。静的固定すると実行日が変わるたびに誤判断する |
| 一括収集設計 | 経費情報と領収書画像をまとめて1ターンで収集する。BRL-23（対話回数上限）の消費を抑制しユーザー体験を向上させる |
| 経費区分判断ルールの明示 | BRL-09 の判断基準をキーワードリストで明示し、LLM が一貫した経費区分判断を行えるようにする。曖昧なケースは社員選択にフォールバックする |
| TOOL-002 呼び出し形式の明示 | collected_items の "expenses" キーと各フィールドの型・キー名を明示し、LLM がキーを省略して実行時エラーが発生するリスクを排除する |
| ドラフト提示と TOOL-002 の分離 | APR-001 承認フロー（ABAC-001）を確実に守るため、ドラフト提示と TOOL-002 呼び出しを明確に分離する |

#### 2.3.2 申請ルール

| ルール | 条件 | 対応 |
|--------|------|------|
| BRL-02 | 経費情報収集時 | 1件ずつ順次収集する（一括収集設計で1ターンに集約） |
| BRL-08 | 領収書画像がアップロードされた場合 | LLM 画像認識で金額・支払先・購入日を自動抽出する。失敗時は手動入力に切り替える |
| BRL-09/JD-07/JD-17 | 品目名称が入力された場合 | キーワードルールで経費区分を自動判断する。判断困難な場合は4択選択肢を提示する |
| BRL-11 | 購入日が申請期限下限日より前 | 申請期限超過を案内してフローを継続する |
| BRL-12/JD-11 | 金額が5,000円超過 | 事前上長承認取得を確認する旨を社員に提示する |
| BRL-13 | 申請書生成前（全経費分） | 業務目的が空の場合は記載を促す |
| BRL-16/EVT-001 | 申請書生成後 | 申請内容を社内申請ルールと照合し結果を提示する |

#### 2.3.3 処理フロー

```
開始（handle_expense_application が AG-001 から呼び出される）
  ↓
invocation_state から session_id・applicant_name・application_date を取得
InvocationStateModel でバリデーション
  ↓
[session_id が _agent_instances に未登録？]
  - YES → AG-003 Agent インスタンスを生成して _agent_instances[session_id] に登録
           （システムプロンプトは動的生成: _build_ag003_system_prompt()）
  - NO  → キャッシュ済みインスタンスを使用
  ↓
_agent_instances[session_id](query) で AG-003 を呼び出す
  ↓
AG-003 内部の ReAct ループ開始
  ↓
経費情報の一括収集（1件分の全項目と領収書画像を1ターンで確認）
  - 購入日・金額・店舗名・品目・業務目的をまとめて確認
  ↓
[領収書画像がアップロードされた？]
  - YES → LLM 画像認識で金額・支払先・購入日を自動抽出（BRL-08）
           ↓
           [抽出成功？]
             - YES → 抽出情報を社員に確認後、経費情報に反映
             - NO  → 「領収書の読み取りに失敗しました。再度アップロードするか、手動でご入力ください。」
  - NO  → 手動入力フロー継続
  ↓
経費区分の自動判断（BRL-09/JD-07/JD-17）
  - キーワードルール適用: 事務用品費 / 宿泊費 / 資格精算費 の自動判定
  ↓
[経費区分判断成功？]
  - YES → 経費区分を自動設定
  - NO  → 社員に4択を提示（「事務用品費」「宿泊費」「資格精算費」「その他経費」）
  ↓
申請期限チェック（BRL-11）
  - purchase_date < (today - 90日) → 申請期限超過案内（フローは継続）
  ↓
高額チェック（BRL-12）
  - amount > 5,000 → 事前上長承認確認案内
  ↓
経費追加確認
  - はい → 経費情報収集ループ先頭へ
  - いいえ → 次のステップへ
  ↓
全経費収集完了
  ↓
収集済み申請情報をテキストとして整理・提示（ドラフト確認）
  ※この時点では generate_expense_application を呼び出さない
  ↓
APR-001 承認取得（HumanApprovalHook / BeforeToolCallEvent が発火）
  - OK → generate_expense_application 呼び出しへ進む
  - 修正 → 修正対象経費を再収集してドラフト提示に戻る
  - キャンセル → セッションを TERMINATED に遷移 → 「申請書の生成をキャンセルしました。」を返却 → 終了
  ↓
generate_expense_application 呼び出し
  - 入力: applicant_name, application_date, collected_items{"expenses":[...]}
  ↓
[生成結果の判定]
  - success=True → file_path を取得
  - success=False → エラーメッセージを AG-001 に返却 → 終了
  ↓
EVT-001 申請内容自動チェック（FR-010/BRL-03/BRL-16）
  - 全経費の業務目的・申請期限・高額申請・経費区分のルール適合を確認
  - チェック結果（合格/違反項目・修正案）を生成
  ↓
AG-001 への返却文字列を生成
  - 申請書ファイルパス + チェック結果
  ↓
終了（handle_expense_application が str を返す）
```

### 2.5 連携設計

#### 2.5.1 Agent as Tools設計

**ツール名**: `handle_expense_application`

**ツール説明**:
```
経費精算申請書作成エージェント。
AG-001から委譲された経費精算申請フロー全体（経費情報収集・OCR自動抽出・経費区分判断・承認・申請書生成・チェック）を実行する。
```

**ToolContextの使用**:
- `@tool(context=True)` デコレータを使用
- `tool_context.invocation_state` から session_id・applicant_name・application_date を取得
- `InvocationStateModel(**state)` でバリデーションを実施する
- session_id はファクトリ関数内でのみ使用し、AG-003 の LLM プロンプトには含めない

#### 2.5.2 invocation_stateの受け渡しとAgentインスタンスのキャッシュ

```python
# モジュールレベルで Agentインスタンスをキャッシュ（session_id をキーにする）
_agent_instances: dict[str, Agent] = {}

@tool(context=True)
def handle_expense_application(query: str, tool_context: ToolContext) -> str:
    # AG-001 から受け取る
    state = tool_context.invocation_state
    session_id = state.get("session_id", "")
    applicant_name = state.get("applicant_name", "")
    application_date = state.get("application_date", "")

    # InvocationStateModel でバリデーション
    validated_state = InvocationStateModel(
        applicant_name=applicant_name,
        application_date=application_date,
    )

    # FileBasedSessionManager を初期化する（session_id は LLM には渡さない）
    session_manager = FileBasedSessionManager(
        session_id=session_id,
        storage_path=SESSION_STORAGE_PATH,
    )

    # session_id 未登録の場合のみ新しい Agent インスタンスを生成する
    # callback_handler=None: AG-001 経由の二重出力を防ぐ
    if session_id not in _agent_instances:
        _agent_instances[session_id] = Agent(
            name="AG-003",
            model=MODEL_ID,
            instructions=_build_ag003_system_prompt(),  # 動的生成
            tools=[generate_expense_application, image_reader],
            conversation_manager=SlidingWindowConversationManager(
                window_size=WINDOW_SIZE_EXPENSE_AGENT,
                should_truncate_results=True,
            ),
            hooks=[
                LoopControlHook(max_iterations=MAX_LOOP_ITERATIONS, agent_name="expense_agent"),
                HumanApprovalHook(approval_callback=approval_callback),
            ],
            callback_handler=None,  # AG-001 経由の二重出力を防ぐ
        )

    # エージェントを呼び出す（invocation_state は LLM には渡さない）
    response = _agent_instances[session_id](query, invocation_state=state)
    return str(response)
```

## 3. 実装詳細

### 3.1 クラス/関数設計

#### 3.1.1 handle_expense_application 関数

##### 説明
AG-003 の Agent as Tools エントリポイント。AG-001 から呼び出され、経費精算申請書作成フロー全体を実行する。

##### 引数
- `query` (str): AG-001 からの委譲指示文字列（必須）
- `tool_context` (ToolContext): invocation_state を含む ToolContext（必須）

##### 戻り値
- `str`: 申請書生成完了通知・申請内容チェック結果を含む文字列

##### 処理内容
1. `tool_context.invocation_state` から session_id・applicant_name・application_date を取得する
2. `InvocationStateModel` でバリデーションを実施する（ValidationError は RuntimeError として再送出しない。エラーメッセージを str で返す）
3. `FileBasedSessionManager` を初期化する
4. `session_id` をキーに `_agent_instances` を確認し、未登録の場合のみ Agent インスタンスを生成・登録する
5. `_agent_instances[session_id](query)` を呼び出す
6. レスポンスを `str` に変換して返却する

#### 3.1.2 _build_ag003_system_prompt 関数

##### 説明
AG-003 のシステムプロンプトを動的生成する関数。本日の日付と申請期限下限日をプロンプトに埋め込む。

##### 引数
- なし

##### 戻り値
- `str`: システムプロンプト全文（本日の日付・申請期限下限日が埋め込まれた文字列）

##### 処理内容
1. `date.today()` で本日の日付を取得する
2. `date.today() - relativedelta(days=90)` で申請期限下限日を計算する
3. テンプレート文字列に両日付を埋め込んで返却する

## 5. 補足情報

### 5.1 実装上の注意点

1. **callback_handler=None の必須指定**
   - `Agent(...)` の初期化時に必ず `callback_handler=None` を指定する
   - `agent()` 呼び出し時に渡すと `TypeError` が発生するため、コンストラクタで指定すること
   - これにより AG-003 の出力が標準出力に直接表示されず、AG-001 経由での二重出力を防ぐ

2. **Agentインスタンスのモジュールレベルキャッシュ**
   - `_agent_instances: dict[str, Agent] = {}` はモジュールレベルで定義する
   - session_id をキーとして初回のみ生成・登録し、2回目以降は再利用する
   - SlidingWindowConversationManager の会話履歴がセッション内で保持され、収集済み情報の再入力が不要になる

3. **OCR 処理の実装方式**
   - 領収書 OCR は `strands_tools` の `image_reader` ツールを使用する
   - `tools` リストには `generate_expense_application` と `image_reader` を含める
   - `image_reader` が画像ファイルを受け取り、テキストとして抽出結果を返す

4. **妥当性チェックルールの外部ファイル参照**
   - 申請ルール（BRL-08〜BRL-16 等）の詳細は `agent_knowledge/receipt_policies.py` に集約する
   - システムプロンプトはポリシーファイルへの参照のみを記述し、ルール本体を直接埋め込まない

5. **出力ディレクトリ**
   - 申請書ファイルは `data/output/{session_id}/` に保存される
   - session_id は AG-001 が生成した値を使用する

4. **ドラフト提示と TOOL-002 呼び出しの分離**
   - 全経費収集完了後のドラフト提示は LLM の通常出力として行う（ツール呼び出しなし）
   - HumanApprovalHook が BeforeToolCallEvent で generate_expense_application の実行前に介入する

5. **動的プロンプト生成のコスト**
   - `_build_ag003_system_prompt()` は `handle_expense_application` 呼び出し時に実行される
   - session_id が _agent_instances に未登録の場合（初回呼び出し時）のみ実行されるため、セッション内での再呼び出し時はキャッシュ済みインスタンスを使用する

### 5.2 パフォーマンス考慮事項

1. **Agent インスタンスのキャッシュ効果**
   - 同一セッション内での複数呼び出し間で Agent インスタンスとその会話履歴が保持される
   - セッションリセット時（AG-001 が新しい session_id を発行）はキャッシュが自動的に切り替わる

### 5.3 セキュリティ考慮事項

1. **invocation_state の LLM プロンプトへの非公開**
   - session_id・applicant_name・application_date は ToolContext 経由でのみ参照する
   - AG-003 の LLM プロンプトには含めない（情報漏洩・改ざんのリスク排除）
