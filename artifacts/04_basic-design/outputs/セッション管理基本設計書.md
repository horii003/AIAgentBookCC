## 2. 目的・役割定義

### 2.1 コンポーネントの目的
- システム起動時にセッションIDを発行し、全エージェント（AG-001/AG-002/AG-003）で共有されるセッション識別子を管理する
- 収集済み申請情報（D-010）をファイルベースのJSONストア（`data/sessions/{session_id}.json`）に永続化し、セッション中断・再開を可能にする
- セッション状態（CREATED/ACTIVE/WAITING/CLOSED/TERMINATED）の遷移を管理する

### 2.2 主な責務
- **セッションID発行**: `{YYYYMMDD}_{HHMMSS}_{uuid4[:8]}` 形式のセッションIDをアプリケーション起動時に生成する
- **セッション状態の永続化**: ターン単位でセッション状態ファイルを上書き保存する
- **状態遷移管理**: CREATED → ACTIVE → WAITING/CLOSED/TERMINATED の状態遷移を制御する
- **セッション状態の読み取り**: 各エージェント・ハンドラーからのセッション状態参照リクエストに応答する
- **セッションの削除**: CLOSED/TERMINATED 状態への遷移時にセッション状態ファイルを削除する
- **SessionManagerFactoryによる提供**: `SessionManagerFactory.create()` を通じて各エージェントに `FileBasedSessionManager` インスタンスを提供する。各エージェントは取得したインスタンスを `Agent` の `session_manager` パラメータに渡す

### 2.3 非責務（やらないこと）
- **会話履歴の管理**: SlidingWindowConversationManager が各エージェント独立で管理する
- **invocation_stateの設定**: アプリケーション起動コード（main.py）が担当する
- **バリデーション実装**: Pydanticモデル（models/data_models.py）が担当する
- **エラーメッセージ生成**: ErrorHandler（HD-001）が担当する

### 2.4 必要性
- マルチエージェント構成（AG-001 → AG-002/AG-003）でセッションIDを共有し、全エージェントが同一セッションのコンテキストで動作するために必要
- セッション中断時に収集済み申請情報を保持し、再開時に継続できるために永続化が必要
- invocation_stateにセッションIDを設定することで、LLMプロンプトを汚染せずにセッション情報を専門エージェントへ伝達する

## 3. セッションID設計

### 3.1 セッションID形式

```
{YYYYMMDD}_{HHMMSS}_{uuid4[:8]}
例: 20260502_143022_a3f9b1c2
```

| 部分 | 形式 | 目的 |
|-----|------|------|
| `{YYYYMMDD}` | 年月日（8桁） | 日付による識別・ログ検索の容易化 |
| `{HHMMSS}` | 時分秒（6桁） | 同日複数セッションの区別 |
| `{uuid4[:8]}` | UUID v4の先頭8文字 | 一意性の保証（衝突確率を極小化） |

### 3.2 発行タイミング
- アプリケーション起動時（main.py 内）に `SessionManager.create_session()` を呼び出して発行する
- 以降はセッション終了まで同一のセッションIDを使用する。再発行は行わない

## 4. セッション状態設計

### 4.1 状態定義

| 状態 | 説明 | 遷移元 | 遷移先 |
|-----|------|--------|--------|
| CREATED | セッション生成完了。対話開始前 | — | ACTIVE |
| ACTIVE | エージェントが会話ターンを処理中 | CREATED / WAITING | WAITING / CLOSED / TERMINATED |
| WAITING | 社員の入力（承認・修正・選択）を待機中 | ACTIVE | ACTIVE |
| CLOSED | 正常終了（申請書提示完了またはキャンセル） | ACTIVE | — |
| TERMINATED | 異常終了（エラー/エスカレーション/ターン上限） | ACTIVE | — |

### 4.2 状態遷移条件

| 遷移 | トリガー |
|------|---------|
| CREATED → ACTIVE | 対話ループ開始時 |
| ACTIVE → WAITING | エージェントが社員へ入力を要求する発話を行ったとき |
| WAITING → ACTIVE | 社員が入力を返したとき |
| ACTIVE → CLOSED | CF-005完了（申請書提示完了）または社員がキャンセルを選択したとき |
| ACTIVE → TERMINATED | EX-04/05/06/07/08 でCF-006誘導後、またはターン上限30回到達時 |

### 4.3 再開可能条件

| セッション状態 | 再開可否 | 理由 |
|-------------|---------|------|
| ACTIVE / WAITING | ○（再開可） | 未完了セッション。収集済み情報から継続可能 |
| CLOSED | × | 正常終了済み。同じ申請内容の再提出は禁止 |
| TERMINATED | × | 異常終了済み。担当部門対応後に新規セッションで再開する |

## 5. セッション状態ファイル設計

### 5.1 ファイルパス

```
data/sessions/{session_id}.json
例: data/sessions/20260502_143022_a3f9b1c2.json
```

### 5.2 JSONスキーマ

```json
{
  "session_id": "20260502_143022_a3f9b1c2",
  "applicant_name": "山田太郎",
  "application_date": "2026-05-02",
  "application_type": "交通費精算申請",
  "session_status": "ACTIVE",
  "turn_count": 5,
  "collected_info": {
    "segments": [
      {
        "travel_date": "2026-04-28",
        "departure": "渋谷",
        "destination": "新宿",
        "transportation_type": "電車",
        "amount": 200,
        "purpose": "社内会議のため"
      }
    ]
  },
  "output_file_path": null
}
```

**収集済み申請情報（collected_info）の構造**:
- **交通費精算申請（AG-002）**: `{"segments": [TransportSegmentの辞書リスト]}`
- **経費精算申請（AG-003）**: `{"items": [ExpenseItemの辞書リスト]}`

### 5.3 フィールド定義

| フィールド名 | 型 | 必須 | 更新可否 | 説明 |
|------------|---|------|---------|------|
| session_id | str | ○ | 禁止 | セッションID（発行後固定） |
| applicant_name | str | ○ | 禁止 | 申請者名（起動時設定後固定） |
| application_date | str | ○ | 禁止 | 申請日（YYYY-MM-DD、起動時設定後固定） |
| application_type | str | △（AG-001確定後に設定） | AG-001確定後禁止 | 申請種別（AG-001が一度だけ設定） |
| session_status | str | ○ | 遷移条件に従い更新 | セッション状態（5種類） |
| turn_count | int | ○ | ターン処理時に更新 | 対話ターン数（最大30回） |
| collected_info | dict | △（AG-002/AG-003が逐次更新） | 追記・上書き可 | 収集済み申請情報（D-010） |
| output_file_path | str or null | △（申請書生成後に設定） | 生成後固定 | 生成済み申請書ファイルパス |

### 5.4 保存・削除ルール

| 操作 | タイミング | 方式 |
|------|----------|------|
| 作成 | `create_session()` 呼び出し時 | CREATED 状態でファイル新規作成 |
| 更新 | ターン処理完了ごと | 最新状態を上書き保存（バージョン履歴なし） |
| 削除 | CLOSED/TERMINATED 遷移時 | ファイル削除（`data/sessions/{session_id}.json`） |

## 6. メソッド設計

### 6.1 主要メソッド

| メソッド名 | 引数 | 戻り値 | 処理内容 |
|-----------|------|--------|---------|
| create_session(applicant_name, application_date) | applicant_name: str, application_date: str | str（session_id） | セッションIDを発行し、CREATED 状態でファイルを作成する |
| read_session(session_id) | session_id: str | dict（セッション状態全体） | `data/sessions/{session_id}.json` を読み取って辞書で返す。ファイルが存在しない場合は None を返す |
| update_session(session_id, updates) | session_id: str, updates: dict（更新フィールドと値） | None | セッション状態ファイルに updates を反映して上書き保存する。禁止フィールド（session_id/applicant_name/application_date）の上書きは ValueError を raise する |
| update_status(session_id, new_status) | session_id: str, new_status: str（状態） | None | session_status フィールドを new_status に更新する。CLOSED/TERMINATED への遷移時はファイルを削除する |
| delete_session(session_id) | session_id: str | None | `data/sessions/{session_id}.json` を削除する。ファイルが存在しない場合は無視する |
| session_exists(session_id) | session_id: str | bool | セッション状態ファイルの存在確認 |
| _generate_session_id() | — | str | `{YYYYMMDD}_{HHMMSS}_{uuid4[:8]}` 形式のセッションIDを生成して返す |

### 6.2 初期化パラメータ

エージェントは `SessionManagerFactory.create()` を呼び出して `FileBasedSessionManager` インスタンスを取得し、`Agent` の `session_manager` パラメータに渡す:

```python
session_manager = SessionManagerFactory.create()
agent = Agent(
    ...,
    session_manager=session_manager,
)
```

`FileBasedSessionManager` の初期化パラメータ:

| パラメータ | デフォルト値 | 説明 |
|----------|-----------|------|
| storage_path | `"data/sessions"` | セッション状態ファイルの保存ディレクトリ（共通設定方針 §5.3） |

### 6.3 SessionManagerFactory設計

`SessionManagerFactory` は `FileBasedSessionManager` インスタンスの生成を集約するファクトリクラスである。

| 項目 | 内容 |
|------|------|
| クラス名 | `SessionManagerFactory` |
| メソッド | `create()` — クラスメソッド（引数なし） |
| 戻り値 | `FileBasedSessionManager`（`storage_path="data/sessions"` で初期化済み） |
| `create_session` | `FileBasedSessionManager` に委譲。変更なし |

```python
class SessionManagerFactory:
    @classmethod
    def create(cls) -> FileBasedSessionManager:
        return FileBasedSessionManager(storage_path="data/sessions")
```

**設計意図**: エージェントコードで `SessionManager(storage_path="data/sessions")` を直接記述する代わりに `SessionManagerFactory.create()` を使用することで、将来的なストレージ実装の切り替え（例: DBベースの実装）をファクトリ内のみの変更で完結できるようにする。

## 7. マルチエージェント連携時の扱い

### 7.1 invocation_stateへのセッションID設定

main.py 起動時に以下を設定する:

```python
session_id = session_manager.create_session(applicant_name, application_date)
context.invocation_state = {
    "session_id": session_id,
    "applicant_name": applicant_name,
    "application_date": application_date
}
```

- LLMプロンプトには session_id を含めない（マルチエージェント連携設計.md §7.3）
- AG-002/AG-003 のツール関数は `ToolContext.invocation_state["session_id"]` で取得する

### 7.2 エージェント別のセッション状態アクセス

| エージェント | アクセス方法 | 更新対象フィールド |
|------------|-----------|----------------|
| AG-001 | invocation_state に設定済み | application_type（申請種別確定後に一度のみ） |
| AG-002 | ToolContextから session_id を取得後に read/update | collected_info（segments）、output_file_path、session_status |
| AG-003 | ToolContextから session_id を取得後に read/update | collected_info（items）、output_file_path、session_status |

### 7.3 並行アクセス制御

- AG-002 と AG-003 の同時起動は実行制御方針 §5.1 で禁止されているため、同一セッションへの並行書き込みは発生しない
- ファイルロック機構は不要（設計上の直列化を前提とする）

## 8. セッション再開設計

### 8.1 再開フロー

```text
アプリケーション起動時
→ 既存セッションの確認（session_exists）
  ├── [存在する] セッション状態を読み取る（read_session）
  │   ├── [ACTIVE/WAITING] 収集済み情報を社員に提示して継続確認
  │   └── [CLOSED/TERMINATED] 新規セッションを開始する
  └── [存在しない] create_session で新規セッションを開始する
```

### 8.2 再開時の復元情報

| 情報 | 復元要否 | 復元方法 |
|-----|---------|---------|
| applicant_name | ○ | セッション状態ファイルから読み取る |
| application_date | ○ | セッション状態ファイルから読み取る |
| application_type | ○（確定済みの場合） | セッション状態ファイルから読み取る |
| collected_info | ○ | セッション状態ファイルから読み取る |
| turn_count | ○ | セッション状態ファイルから読み取る |
| 会話履歴 | × | SlidingWindowConversationManager がインメモリ管理のため再開時は失われる |

## 9. エラー・例外処理設計

| エラー種別 | 対応方針 |
|-----------|----------|
| ファイル書き込み失敗（IOError） | EX-07（システム障害）として ErrorHandler に委譲。リトライ後も失敗の場合はCF-006へ |
| ファイル読み取り失敗（IOError） | EX-04（外部I/F例外）として ErrorHandler に委譲 |
| JSONパースエラー（json.JSONDecodeError） | EX-06（データ不整合）として ErrorHandler に委譲。セッションファイルの破損と判断 |
| 禁止フィールドへの上書き試行 | ValueError を raise する。ErrorHandler が EX-05 として処理する |
| `data/sessions/` ディレクトリが存在しない | アプリケーション起動時に自動作成する（エラーとしない） |

## 10. セキュリティ設計

### 10.1 個人情報の保護

- **applicant_name（D-3: 個人情報）**: セッション状態ファイルには記録するが、ログ出力時はマスキングする（GRD-011）
- **collected_info（D-2: 機微情報）**: セッション状態ファイルには記録するが、ログ出力時は項目名のみ記録（値は除外または要約）

### 10.2 セッション状態ファイルのアクセス制御

- ファイルパスは固定形式（`data/sessions/{session_id}.json`）とし、session_id 以外のパスアクセスを防ぐ
- session_id の形式バリデーションを実施して任意パスへのアクセスを防ぐ（パストラバーサル対策）

## 11. テスト観点

| 観点 | テスト内容 |
|-----|---------|
| セッションID発行 | 正しい形式（`YYYYMMDD_HHMMSS_xxxxxxxx`）のIDが発行されること。同時刻でもUUIDで一意性が保証されること |
| セッション作成 | CREATED 状態でファイルが作成されること。data/sessions/ ディレクトリが存在しない場合も作成されること |
| 状態遷移 | CREATED→ACTIVE→WAITING→ACTIVE→CLOSED の正常系遷移。CLOSED/TERMINATED 遷移時にファイルが削除されること |
| 禁止フィールド保護 | applicant_name/application_date/session_id の上書き試行で ValueError が発生すること |
| ファイルI/O | write・read が正常に動作すること。JSONの内容が一致すること |
| 再開 | セッション状態ファイルが存在する場合に read_session で正しく復元されること |
| 並行アクセス | 設計上発生しないが、同一ファイルへの同時書き込みが発生しないことを確認する |

**詳細なテストケース**: 詳細設計書 7章を参照

## 13. 制約事項・前提条件

### 13.1 技術的制約
- ファイルベースのJSONストアのみ対応する（データベース不使用）
- Python 標準ライブラリ（`json`、`pathlib`、`uuid`、`datetime`）のみを使用する
- バージョン履歴は保持しない（最新状態のみ）
- 並行書き込みは設計上発生しない（ファイルロック不要）

### 13.2 業務的制約
- セッションIDは申請フロー単位で一意（1申請 = 1セッション）
- CLOSED/TERMINATED 後のセッションファイルは削除される（监査ログは LOG-HI に別途保存）
- applicant_name・application_date は変更禁止（起動時設定後固定）

### 13.3 前提条件
- アプリケーション起動時に `applicant_name` が取得済みであること
- `data/sessions/` ディレクトリが存在しない場合は SessionManager が自動作成すること
- セッション終了後（CLOSED/TERMINATED）の監査情報は LOG-HI（DATA-012）に別途記録済みであること
