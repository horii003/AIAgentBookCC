## 1. 概要

### 1.1 コンポーネントの目的

社員の申請意図テキスト（D-001）を受け取り、社内申請ルールナレッジ（D-002）を根拠に交通費精算申請・経費精算申請のいずれかを判断して案内（D-003）する。申請種別が確定したら申請者名・申請日・セッションIDを`invocation_state`に設定し、適切な専門エージェント（AG-002またはAG-003）へフロー全体を委譲する。

### 1.2 主要な責務

- **申請者名収集**: セッション開始時に申請者名を収集し、`invocation_state`に設定する（main.pyがアプリ起動時に取得）
- **申請意図受付**: 社員の申請意図テキスト（D-001）を受け取り、入力バリデーション（GRD-001/GRD-012）を実施する
- **申請種別判断**: D-002（社内申請ルールナレッジ）をシステムプロンプトに埋め込んだLLM推論で申請種別を判断する（BRL-01/JD-01/JD-02）
- **案内提示**: 申請種別・申請書・申請先をD-003として社員に提示する（根拠提示Lv2）
- **専門エージェントへの委譲**: `transport_agent_tool`（AG-002ラッパー）または`expense_agent_tool`（AG-003ラッパー）を呼び出してフローを委譲する

### 2.2 インターフェース設計

#### 2.2.1 入力

**初期化時の入力**:
- `applicant_name`（必須、空文字禁止）: アプリケーション起動時にCLI引数またはプロンプト入力で取得する
- `session_id`（必須）: SessionManagerがセッション開始時に生成する

**対話ループでの入力**:
- ユーザー入力（申請意図テキスト D-001）
- 特殊コマンド:
  - `exit`, `quit`, `終了`: システム終了
  - `reset`, `リセット`, `最初から`: 会話履歴と申請者情報をリセットして申請フローを最初からやり直す

**パラメータ**:
- `query` (str): ユーザーからの申請意図テキスト

#### 2.2.2 出力

**出力先**: CLIの標準出力（社員への案内メッセージ）

**ウェルカムメッセージ**（対話ループ開始時に表示）:
```
============================================================
こちらは申請受付窓口エージェントです
社内の様々な申請作業をサポートします

最初に申請者名を入力してください。その後、申請したい内容をお知らせください。キーワードでも構いません

※終了するには 'exit' または 'quit' と入力ください
※最初からやり直すには 'reset' と入力ください
============================================================
```

**対話ループ 入力プロンプト**:
```
\n\n入力内容（終了時はquit）: 
```

**出力メッセージ**:
- 申請種別判断完了（JD-01）: `"申請種別：交通費精算申請（根拠：[適用ルールID]）。交通費精算申請エージェントにお繋ぎします。"`
- 判断不可（JD-02）: `"交通費精算申請・経費精算申請のどちらでしょうか？\n1. 交通費精算申請\n2. 経費精算申請"`
- エスカレーション: `"ご申請内容が申請ルール外のため、担当部署へご相談ください。"`
- エラー（空入力）: `"申請内容が入力されていません。どのような精算をされたいか、自由な言葉で入力してください。"`

**戻り値の形式**: `str`（エージェントからの最終応答）

### 2.3 ビジネスロジック

#### 2.3.1 システムプロンプト

##### システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 静的定数 |
| 動的生成の理由 | 不要（AG-001は申請種別判断・案内のみ担当。日付・期限を基準とした判断処理は含まれない） |

##### プロンプト全文

```
あなたは社員からの申請を受け付ける窓口エージェントです。

【役割】
1. 社員の申請意図を受け取り、交通費精算申請・経費精算申請のいずれかを判断して案内する
2. 申請種別が確定したら適切な担当エージェントへフローを引き継ぐ
3. 申請ルール外のケースはエスカレーション案内を行う

【申請種別判断ルール（社内申請ルールナレッジ）】
以下のルールを根拠に申請種別を判断してください。

交通費精算申請に該当するケース（BRL-01）:
- タクシー・電車・バス・飛行機等の交通費の精算
- 通勤・出張・業務上の移動に伴う交通費
- 複数区間の交通費をまとめて申請する場合

経費精算申請に該当するケース（BRL-01）:
- 事務用品費（文具・消耗品等の購入費用）
- 宿泊費（出張時の宿泊費用）
- 資格精算費（業務に関連する資格取得費用）
- その他経費（会食費・接待費等の交通費以外の経費）

判断不能なケース（BRL-01, JD-02）:
- 上記のどちらにも明確に該当しない場合は、2択選択肢を提示して社員に選択させること

申請ルール外のケース:
- 給与・賞与・経費以外の金銭的請求は申請ルール外として担当部署へのエスカレーション案内を行うこと

【処理フロー】
1. 社員の申請意図テキストを受け取る
2. 上記の申請種別判断ルールを参照し、交通費精算申請・経費精算申請のいずれかを判断する
3. 判断できた場合（JD-01）: 申請種別・根拠（適用ルールID）を提示し、担当エージェントへ引き継ぐ
   - 交通費精算申請: transport_agent_toolを呼び出す
   - 経費精算申請: expense_agent_toolを呼び出す
4. 判断不能な場合（JD-02）: 以下の2択を提示して社員に選択させる
   「交通費精算申請・経費精算申請のどちらでしょうか？
   1. 交通費精算申請
   2. 経費精算申請」
   社員が選択後、該当するエージェントへ引き継ぐ
5. 申請ルール外の場合: エスカレーション案内を提示する
   「ご申請内容が申請ルール外のため、担当部署へご相談ください。」

【ツール呼び出しルール】
- transport_agent_toolの呼び出し: 申請種別が「交通費精算申請」に確定した後のみ実行すること
- expense_agent_toolの呼び出し: 申請種別が「経費精算申請」に確定した後のみ実行すること
- 申請種別が未確定の場合はいかなるエージェントツールも呼び出してはならない
- transport_agent_toolとexpense_agent_toolを同時に呼び出してはならない（一方のみ）

【根拠提示ルール（Lv2）】
- 申請種別を判断した際は、必ず適用した判断根拠（ルールID・内容）を提示すること
  例: 「タクシー代の精算は交通費精算申請に該当します（BRL-01: 交通費の精算）」

【禁止事項】
- 申請情報の収集（申請日・移動情報・領収書情報等）は行わないこと（専門エージェントの責務）
- 申請書の生成・ファイル出力は行わないこと（最大自律度Lv2）
- 申請書の提出は行わないこと（GRD-011）
```

##### プロンプト設計の意図

| 要素 | 意図 |
|------|------|
| 申請種別判断ルール埋め込み | D-002（社内申請ルールナレッジ）をRAGではなく静的プロンプトに埋め込む（RAG廃止方針準拠） |
| 処理フロー | LLMが判断→案内→委譲の手順を守るよう明示的に指示する |
| ツール呼び出しルール | 申請種別確定前の誤委譲を防止する |
| 禁止事項 | AG-001の責務範囲を明確に制限し、専門エージェントへの責任分界を徹底する |

#### 2.3.2 申請種別判断ルール

| ルール | 条件 | 対応 |
|--------|------|------|
| BRL-01 JD-01（判断可） | 申請意図が交通費関連（タクシー/電車/バス/飛行機等）と明確に判断できる | AG-002（transport_agent_tool）へ委譲 |
| BRL-01 JD-01（判断可） | 申請意図が経費関連（会食費/事務用品費/宿泊費等）と明確に判断できる | AG-003（expense_agent_tool）へ委譲 |
| BRL-01 JD-02（判断不能） | 申請意図から申請種別が一意に判断できない | 2択選択肢を提示し、選択後に委譲 |
| 申請ルール外 | 給与・賞与等の申請ルール外ケース | CF-006エスカレーション案内を提示 |

#### 2.3.3 処理フロー

```
開始（applicant_nameはアプリ起動時取得済み）
  ↓
ウェルカムメッセージを表示する
  ↓
invocation_stateにapplicant_name・application_date・session_idを設定
  - application_date: datetime.now().strftime("%Y-%m-%d")
  - session_id: {タイムスタンプ（秒単位）}_{UUID（8文字）} 形式で生成
  ↓
対話ループ開始
  ↓
入力プロンプト（"\n\n入力内容（終了時はquit）: "）を表示してユーザー入力受付
  ↓
特殊コマンド確認
  "exit"/"quit"/"終了" → システム終了（break）
  "reset"/"リセット"/"最初から" → 会話履歴リセット → 対話ループ先頭へ
  ↓
入力バリデーション
  空文字チェック（GRD-001）: 空 → "申請内容が入力されていません。..." を提示 → 再入力
  500文字超チェック（GRD-012）: 超過 → 文字数上限を案内 → 再入力
  ↓
エージェント（AG-001）に入力を渡す
  LLMが申請種別を判断（社内申請ルールナレッジ参照）
  ↓
判断結果による分岐
  ↓
交通費精算申請と判断（JD-01）
  → 「申請種別：交通費精算申請（根拠：BRL-01）」を提示
  → transport_agent_toolを呼び出す
    - query: 委譲メッセージ（申請種別・申請意図テキスト）
    - invocation_state: {applicant_name, application_date, session_id}
  → AG-002の処理完了応答をユーザーに提示
  ↓
経費精算申請と判断（JD-01）
  → 「申請種別：経費精算申請（根拠：BRL-01）」を提示
  → expense_agent_toolを呼び出す
    - query: 委譲メッセージ（申請種別・申請意図テキスト）
    - invocation_state: {applicant_name, application_date, session_id}
  → AG-003の処理完了応答をユーザーに提示
  ↓
判断不能（JD-02）
  → 2択選択肢を提示
  → ユーザーが選択 → 選択に応じて上記委譲処理を実行
  ↓
申請ルール外
  → エスカレーション案内を提示
  → 対話ループ先頭へ
  ↓
例外発生時（個別捕捉）
  KeyboardInterrupt
    → logging.info("KeyboardInterrupt: session_id={session_id}")でINFOログ
    → ErrorHandler.handle_keyboard_interrupt(e) でメッセージ生成
    → print(エラーメッセージ)
    → break（ループ終了）
  LoopLimitError
    → logging.warning("LoopLimitError: session_id={session_id}")でWARNINGログ
    → ErrorHandler.handle_loop_limit_error(e) でメッセージ生成
    → print(エラーメッセージ)
    → continue（ループ継続）
  ContextWindowOverflowException
    → logging.warning("ContextWindowOverflowException: session_id={session_id}")でWARNINGログ
    → ErrorHandler.handle_context_window_error(e) でメッセージ生成
    → print(エラーメッセージ)
    → continue（ループ継続）
  MaxTokensReachedException
    → logging.warning("MaxTokensReachedException: session_id={session_id}")でWARNINGログ
    → ErrorHandler.handle_max_tokens_error(e) でメッセージ生成
    → print(エラーメッセージ)
    → continue（ループ継続）
  RuntimeError
    → logging.error("RuntimeError: session_id={session_id}", exc_info=True)でERRORログ
    → ErrorHandler.handle_runtime_error(e) でメッセージ生成
    → print(エラーメッセージ)
    → continue（ループ継続）
  Exception（その他）
    → logging.error("Unexpected error: session_id={session_id}", exc_info=True)でERRORログ
    → ErrorHandler.handle_unexpected_error(e) でメッセージ生成
    → print(エラーメッセージ)
    → continue（ループ継続）
  ↓
対話ループ先頭へ（終了コマンドまで継続）
```

#### 2.5.1 Agent as Tools設計

**transport_agent_tool**:

**ツール名**: `transport_agent_tool`

**ツール説明**:
```
交通費精算申請フローを実行します。
申請者名・申請種別（交通費精算申請）・申請意図テキストを引数として渡してください。
このツールを呼び出す前に申請種別が「交通費精算申請」に確定していることを確認してください。
```

**expense_agent_tool**:

**ツール名**: `expense_agent_tool`

**ツール説明**:
```
経費精算申請フローを実行します。
申請者名・申請種別（経費精算申請）・申請意図テキストを引数として渡してください。
このツールを呼び出す前に申請種別が「経費精算申請」に確定していることを確認してください。
```

**ToolContextの使用**:
- `@tool(context=True)` デコレータを使用
- `tool_context` パラメータでToolContextを受け取る
- `invocation_state` から `applicant_name`・`application_date`・`session_id` を取得して専門エージェントに渡す

#### 2.5.2 invocation_stateの受け渡しとAgentインスタンスのキャッシュ

AG-002・AG-003をAgent as Toolsとして登録する専門エージェントは、モジュールレベルの辞書に `session_id` をキーとして `Agent` インスタンスをキャッシュする。

```python
# session_id をキーにして Agent インスタンスをキャッシュし、
# セッション内で会話履歴を保持する（情報の再入力を防ぐ）
_agent_instances: dict[str, Agent] = {}

@tool(context=True)
def transport_agent_tool(query: str, tool_context: ToolContext) -> str:
    # AG-001から渡されるinvocation_stateを受け取る
    state = tool_context.invocation_state
    session_id = state.get("session_id", "")
    applicant_name = state.get("applicant_name", "")
    application_date = state.get("application_date", "")

    # session_id 未登録の場合のみ新しい Agent インスタンスを生成する
    # Agent初期化時にcallback_handler=Noneを指定して標準出力への直接表示を抑制する（AG-001による二重出力を防ぐ）
    if session_id not in _agent_instances:
        _agent_instances[session_id] = Agent(
            model=get_bedrock_model(),
            system_prompt=get_transport_system_prompt(application_date),
            tools=[calculate_transport_fare, generate_transport_application],
            agent_id="transport_agent",
            name="交通費精算申請エージェント",
            conversation_manager=SlidingWindowConversationManager(
                window_size=20,
                should_truncate_results=True,
                per_turn=False
            ),
            hooks=[HumanApprovalHook(), LoopControlHook(max_iterations=10, agent_name="transport_agent")],
            session_manager=SessionManager(
                session_id=session_id,
                storage_path="data/sessions/"
            ),
            guardrail_id=GUARDRAIL_ID,
            guardrail_version=GUARDRAIL_VERSION,
            guardrail_trace="enabled",
            callback_handler=None,
        )

    # invocation_stateは専門エージェント内部向けに渡す（session_idは含めない）
    agent_state = {"applicant_name": applicant_name, "application_date": application_date}
    response = _agent_instances[session_id](query, invocation_state=agent_state)
    return str(response)
```

`expense_agent_tool`も同様のパターンで実装する（`_agent_instances`は各ツールファイルで独立管理）。

#### 2.5.3 専門エージェントへの振り分けロジック

**振り分け判断基準**:

| 連携先 | キーワード | 特徴 |
|--------|-----------|------|
| AG-002（transport_agent_tool） | タクシー、電車、バス、飛行機、交通費、移動費、通勤費 | 交通機関の利用に伴う費用の精算 |
| AG-003（expense_agent_tool） | 会食、事務用品、文具、宿泊、資格、経費 | 交通費以外の業務上経費の精算 |
| ユーザー選択後に決定 | 精算したい（種別不明） | 申請意図テキストから申請種別が一意に判断できない |

## 5. 補足情報

### 5.1 実装上の注意点

1. **Agent初期化時に`callback_handler=None`を指定する**
   - AG-001がtransport_agent_tool/expense_agent_toolを呼び出す際、専門エージェントが標準出力に直接出力してAG-001と二重出力になることを防ぐ
   - `agent()`呼び出し時に`callback_handler`を渡すと`TypeError`になるため、必ずコンストラクタで指定する

2. **Agentインスタンスはモジュールレベルの辞書でキャッシュする**
   - `_agent_instances: dict[str, Agent] = {}` をモジュールレベルで定義する
   - ツール関数が呼び出されるたびに`Agent(...)`を新規生成してはならない
   - `session_id`をキーとして初回のみ生成・登録し、2回目以降は再利用する

3. **invocation_stateの`session_id`は専門エージェントに渡す際に除外する**
   - AG-001→transport_agent_tool/expense_agent_toolへは`{applicant_name, application_date, session_id}`を渡す
   - ツール関数内でAG-002/AG-003のAgentに渡す`invocation_state`からは`session_id`を除外する（マルチエージェント連携設計 7.3準拠）

4. **申請種別確定前はツールを呼び出さない**
   - システムプロンプトで明示的に禁止するが、実装レベルでも入力バリデーション後にのみエージェントへ渡す

5. **例外は個別に捕捉し、ループ制御を適切に行う**
   - `KeyboardInterrupt` はbreakで終了する
   - `LoopLimitError`, `ContextWindowOverflowException`, `MaxTokensReachedException` はcontinueで継続する
   - `RuntimeError`, `Exception` はERRORログ（exc_info=True）を記録してcontinueで継続する

### 5.2 パフォーマンス考慮事項

1. **Agentインスタンスのキャッシュ**
   - セッション内での複数ターン呼び出しで会話履歴が保持され、収集済み情報の再入力が不要になる

2. **SlidingWindow(30)による会話履歴管理**
   - 古い履歴はメッセージ単位で切り詰められ、コンテキストウィンドウを効率的に使用する

### 5.3 セキュリティ考慮事項

1. **invocation_stateのプロンプト外管理**
   - 申請者名・申請日・セッションIDはプロンプトに含めず`invocation_state`経由で渡す。LLMのコンテキストウィンドウへの不必要な公開を防ぐ

2. **入力文字数制限**
   - 500文字超の入力はバリデーションで拒否し、プロンプトインジェクションのリスクを低減する
