## 2. 目的・役割定義

### 2.1 コンポーネントの目的
- Strands Agents SDK の `FileSessionManager` を使用して、セッションの会話履歴・業務コンテキスト・実行履歴をファイルストレージ（`data/sessions/`）に永続化する
- セッションIDに基づく複数セッションの識別・独立管理を実現し、マルチエージェント環境でのセッション共有を安全に行う
- セッション情報の保持により、CLIプロセス再起動後も同一セッションIDで申請フローを再開可能にする

### 2.2 主な責務
- **会話履歴の永続化**: ユーザー入力・エージェント応答・ツール実行の入出力を `data/sessions/` にセッション単位でファイル保存する
- **セッションIDによる識別**: UUID v4形式の `session_id` を使用してセッションを一意に識別し、複数セッションの混同を防止する
- **window_sizeによる履歴管理**: AG-001: 30、AG-002: 20、AG-003: 15のウィンドウサイズ設定を適用し、コンテキストウィンドウ超過を防止する（`should_truncate_results=True`）
- **セッション状態管理**: CREATED/ACTIVE/WAITING/CLOSED/TERMINATEDの状態遷移をStrands Agents SDKに委譲して管理する

### 2.3 非責務（やらないこと）
- **独自のセッション管理クラスの作成**: Strands Agents SDK の FileSessionManager に全委譲する（セッション管理方針 2.1節）
- **会話ログ・判断ログ・承認ログの記録**: ErrorHandler（`handlers/error_handler.py`）が担当する
- **申請書ドラフトの実体保存**: output/ディレクトリに保存。セッションにはファイルパスのみ保持する
- **セッション状態の直接操作**: Strands Agents SDKが担当する。アプリケーションコードで直接操作しない

### 2.4 必要性
- AG-001起動からAG-002/AG-003の処理完了まで、複数エージェントにまたがる会話履歴・業務コンテキストを一貫して保持する必要がある
- CLIベースのシングルプロセスで複数のエージェント（AG-001/AG-002/AG-003）が同一セッション情報を共有するために、ファイルストレージによる永続化が必要
- Human-in-the-Loop承認待ち（WAITING状態）での中断後に、同一フローを継続するためのセッション復元が必要

### 2.5 主要な設計意図
- **Strands Agents SDK FileSessionManagerへの全委譲**: 独自セッションクラスを作成せず、SDKの FileSessionManager を使用することで実装コストを最小化し、フレームワークの標準機能で一貫したセッション管理を実現する（セッション管理方針 2.1節）
- **ファクトリ関数パターンでの初期化**: `_get_travel_agent(session_id)` / `_get_expense_agent(session_id)` のファクトリ関数内でSessionManagerを初期化し、session_idの混同リスクを防止する（マルチエージェント連携設計 7.2節）
- **invocation_stateによるセッションID受け渡し**: session_idはLLMプロンプトに露出させず、ToolContextのinvocation_state経由でツール関数に渡す

## 3. SessionManagerFactoryクラス

### 3.0 SessionManagerFactoryの導入
- **クラス名**: `SessionManagerFactory`（アプリケーション実装クラス）
- **目的**: FileSessionManagerのインスタンスを生成し、Agentに渡すファクトリクラス
- **利用方法**: `SessionManagerFactory.create(session_id)` を呼び出し、戻り値の `FileSessionManager` インスタンスを `Agent(session_manager=...)` に渡す
- **設計方針**: エージェントは `SessionManagerFactory.create()` を呼び出したら、戻り値を Agent の初期化パラメータ（`session_manager`）として必ず渡すこと

```python
# 利用パターン（概要）
session_manager = SessionManagerFactory.create(session_id)
agent = Agent(..., session_manager=session_manager)
```

### 3.1 FileSessionManagerクラス概要
- **クラス名**: `FileSessionManager`（Strands Agents SDK提供クラス）
- **説明**: セッション情報をJSONファイルとして `data/sessions/` に永続化するStrands Agents SDK標準実装。独自サブクラスは作成しない

### 3.2 FileSessionManagerクラス変数
- **session_id**: セッション識別子（UUID v4形式文字列）。初期化時に指定する
- **storage_dir**: セッションファイルの保存ディレクトリパス（`data/sessions/`）

**詳細**: 詳細設計書 3章を参照

## 4. 主要メソッド

### 4.0 SessionManagerFactory.create（必須エントリポイント）
- **目的**: session_idを指定してFileSessionManagerインスタンスを生成し、Agentに渡すために返す
- **引数**: `session_id: str`（UUID v4形式）
- **戻り値**: `FileSessionManager` インスタンス（必ずAgentの `session_manager` パラメータに渡すこと）
- **呼び出し元**: AG-001のエージェント初期化処理、`_get_travel_agent(session_id)`、`_get_expense_agent(session_id)` ファクトリ関数
- **注意**: 戻り値を破棄してはならない。必ず `Agent(session_manager=SessionManagerFactory.create(session_id))` のように使用する

### 4.1 初期化（ファクトリ関数内）
- **目的**: `SessionManagerFactory.create(session_id)` 経由でFileSessionManagerインスタンスを生成する
- **引数**: `session_id: str`（UUID v4形式）、`storage_dir: str`（`"data/sessions/"`）
- **戻り値**: `FileSessionManager` インスタンス
- **呼び出し元**: `SessionManagerFactory.create()` 経由。直接のFileSessionManager初期化は行わない

### 4.2 load_session
- **目的**: 指定したsession_idのセッションファイルを読み込み、会話履歴・業務コンテキストを復元する
- **引数**: `session_id: str`
- **戻り値**: セッション情報辞書（存在しない場合は新規セッションとして初期化）
- **デフォルトパス**: `data/sessions/{session_id}.json`

### 4.3 save_session
- **目的**: 現在のセッション情報（会話履歴・業務コンテキスト・実行履歴）をファイルに保存する
- **引数**: `session_id: str`、セッションデータ（会話ターン等）
- **戻り値**: なし（ファイルI/OエラーはEX-07として処理）

### 4.4 delete_session
- **目的**: セッション終了時にセッションファイルを削除する（CLOSED/TERMINATED遷移時）
- **引数**: `session_id: str`
- **戻り値**: なし
- **フォーマット**: セッション管理方針 12節に従い、セッション終了時に削除する

**詳細**: 詳細設計書 4.1章〜4.4章を参照

## 5. セッション管理の仕組み

**システム設計書の方針**: システム設計書 セッション管理方針.md（`artifacts/03_system-design/outputs/セッション管理方針.md`）を参照してください。

### 5.1 セッション生成
1. CLIアプリ起動時に申請者名をユーザーから収集する
2. Strands Agents SDKがセッション開始時にUUID v4形式のsession_idを自動生成する
3. AG-001のエージェント初期化時に `SessionManagerFactory.create(session_id)` を呼び出し、戻り値の `FileSessionManager` インスタンスを `Agent(session_manager=...)` に渡す
4. セッション状態がCREATED→ACTIVEに遷移し、会話ループが開始する

### 5.2 セッション更新（AG-002/AG-003委任時）
1. AG-001がAG-002またはAG-003のツール関数を呼び出す際に、ToolContextのinvocation_stateに `session_id`・`applicant_name`・`application_date` を辞書リテラルで設定する
2. ツール関数内でinvocation_stateから `session_id` を取得する
3. ファクトリ関数（`_get_travel_agent(session_id)` または `_get_expense_agent(session_id)`）で `SessionManagerFactory.create(session_id)` を呼び出し、戻り値を `Agent(session_manager=...)` に渡す
4. 専門エージェントがAG-001と同一セッションファイルを参照・更新しながら申請フローを実行する

### 5.3 セッション中断・再開
1. Human-in-the-Loop承認待ち（WAITING状態）でプロセスが中断した場合、FileSessionManagerが現在の会話履歴・業務コンテキストをファイルに保存する
2. 再起動時に同一session_idでFileSessionManagerを初期化すると、Strands Agents SDKが自動的に会話履歴・業務コンテキストを復元する
3. セッション情報（`data/sessions/{session_id}.json`）が存在しない場合は新規セッションとして開始する

### 5.4 セッション終了
1. 申請書ドラフト最終提示・提出案内完了（CLOSED）またはエラー/キャンセル（TERMINATED）でセッションが終了する
2. セッション管理方針 12節に従い、セッション終了時にセッションファイルを削除する
3. LoopControlHookによるループ強制停止（TERMINATED）時もセッションを終了する
4. 対話ターン数30回超過（GRD-018）時もセッションをTERMINATEDに遷移させて終了する

**詳細**: 詳細設計書 5章を参照

## 6. マルチエージェント連携

### 6.1 セッション共有の仕組み

**AG-001（オーケストレーター）**:
1. アプリ起動時に `SessionManagerFactory.create(session_id)` を呼び出し、戻り値を `Agent(session_manager=...)` に渡す
2. 申請種別確定後にinvocation_stateにsession_idを含めてAG-002またはAG-003のツール関数を呼び出す
3. AG-002/AG-003の処理中は同一セッションファイルを共有する（同時書き込みは発生しない）

**AG-002/AG-003（スペシャリスト）**:
1. ツール関数内でinvocation_stateから `session_id` を取得する
2. ファクトリ関数内で `SessionManagerFactory.create(session_id)` を呼び出し、戻り値を `Agent(session_manager=...)` に渡す（AG-001と同一セッションファイルを参照）
3. `applicant_name`・`application_date` をエージェントへのinvocation_stateとして設定し、専門エージェントを起動する

### 6.2 会話履歴の共有範囲
- **共有される情報**: 会話履歴（ユーザー入力・エージェント応答・ツール実行結果）、業務コンテキスト（申請者名・申請種別・収集済み申請情報・申請日）、申請書ドラフトファイルパス
- **共有されない情報**: LLMの中間思考プロセス（推論ログ）、エージェント内部処理状態、session_idのLLMプロンプトへの露出

## 11. 詳細設計への引き渡し事項

### 11.1 実装時の識別子
- **ファクトリクラス名**: `SessionManagerFactory`（アプリケーション実装クラス）
- **ファクトリメソッド名**: `SessionManagerFactory.create(session_id: str) -> FileSessionManager`
- **クラス名**: `FileSessionManager`（Strands Agents SDK提供）
- **会話管理クラス名**: `SlidingWindowConversationManager`（Strands Agents SDK提供）
- **ファクトリ関数名**: `_get_travel_agent(session_id: str)` / `_get_expense_agent(session_id: str)`

### 11.2 SDK実装時の注意事項
- `FileSessionManager` は Strands Agents SDK の標準実装を使用する。独自サブクラスは作成しない
- エージェントは `SessionManagerFactory.create(session_id)` を呼び出したら、戻り値を `Agent(session_manager=...)` に必ず渡すこと。戻り値を無視してはならない
- session_idはinvocation_stateから取得し、`SessionManagerFactory.create()` の引数として使用する。エージェントへのinvocation_stateにはsession_idを含めない
- `SlidingWindowConversationManager` は `per_turn=False` で設定し、`should_truncate_results=True` を指定する
- セッションファイルへの書き込みエラーはEX-07（システム障害）としてErrorHandlerに委譲してセッションを終了する

### 11.3 モック／スタブ要否
- ユニットテストでは `FileSessionManager` をモック化し、ファイルI/Oを抽象化してテストする
- 統合テストでは一時ディレクトリ（`/tmp/sessions/` 等）を使用して実際のファイルI/Oを検証する

## 13. 制約事項・前提条件

### 13.1 技術的制約
- Strands Agents SDK の `FileSessionManager` を使用する。独自セッション管理クラスは実装しない
- エージェントは `SessionManagerFactory.create(session_id)` を呼び出したら、戻り値を `Agent(session_manager=...)` に必ず渡すこと
- session_idはUUID v4形式の文字列とする。Strands Agents SDKが自動生成する
- invocation_stateは辞書リテラルで渡す。専用Pydanticモデルは定義しない（マルチエージェント連携設計 7.2節）

### 13.2 業務的制約
- セッション情報はセッション終了時（CLOSED/TERMINATED）に削除する（セッション管理方針 12節）
- セッション情報（DATA-004）は機密区分D-2（機微）として扱う。ログ出力時はマスキングを適用する
- AG-002とAG-003は同一セッション内で同時実行しない（Agent as Toolsの逐次呼び出し構造で保証）

### 13.3 前提条件
- `data/sessions/` ディレクトリが書き込み可能な状態で存在すること（Strands Agents SDKが自動生成するため通常不要）
- Strands Agents SDK v1.25.0の `FileSessionManager` および `SlidingWindowConversationManager` が利用可能であること
