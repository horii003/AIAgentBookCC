## 業務ルール定義：BRL-01

### 1) 基本情報
- ルールID：BRL-01
- ルール名：申請種別判定
- ルール種別：判断
- 優先度：High
- 管理部門／責任者：要件上未定義

### 2) ルール概要
- ルールの目的：ユーザーが入力した申請内容（自由文）から交通費精算申請・経費精算申請のいずれかを判定する
- このルールが必要な理由：申請ルールを知らない社員でも正しい申請フローへ誘導するため
- 適用される業務上の場面：申請受付の起点（CF-001 Step 2）

### 3) 適用範囲
- 適用業務ID：BIZ-01
- 適用ユースケースID：UC-01
- 適用プロセス／ステップ：ST-001-02
- 適用タイミング：
  - ☐ 業務開始時
  - ☑ 判断前
  - ☐ 実行前
  - ☐ 実行後

### 4) ルール内容

#### 4.1 自然言語定義（業務表現）
ユーザーが入力した申請内容（自由文）をAIエージェントが解析し、社内申請ルールを参照して交通費精算申請・経費精算申請のいずれかを特定する。

#### 4.2 構造化定義（正規形）
```
IF 申請内容（自由文）から「交通費精算申請」と判定できる
THEN 申請種別 = 交通費精算申請（JD-01 OK）→ BRL-03へ
ELSE IF 申請内容（自由文）から「経費精算申請」と判定できる
THEN 申請種別 = 経費精算申請（JD-01 OK）→ BRL-03へ
ELSE（JD-02 NG）→ BRL-02へ（2択提示）
```

### 5) 使用データ

| データID | データ名 | 利用方法 | 必須 |
|---|---|---|---:|
| D-001 | 申請内容（自由文） | 判定条件 | Yes |
| D-002 | 社内申請ルール | 判定根拠のナレッジ参照 | Yes |

### 6) 対応関係

| 種別 | ID | 関係内容 |
|---|---|---|
| 判断基準 | JD-01, JD-02 | 申請種別特定可否の判定に使用 |
| ユースケース | UC-01 | Step 2 |
| 業務プロセス | ST-001-02 | 申請種別判定ノード |
| ナレッジ | KN-001 | 社内申請ルール（申請種別判断） |

### 7) 例外・補足
- 例外条件：入力内容から申請種別が特定できない場合
- 例外時の扱い：BRL-02を適用（JD-02 NG → CF-002へ遷移）
- 判断不可時の対応：2択選択肢をユーザーに提示

## 業務ルール定義：BRL-02

### 1) 基本情報
- ルールID：BRL-02
- ルール名：申請種別確認（判定不能時）
- ルール種別：手続
- 優先度：High
- 管理部門／責任者：要件上未定義

### 2) ルール概要
- ルールの目的：申請種別が特定できない場合に、ユーザーに2択を提示して申請種別を確定させる
- このルールが必要な理由：AIが判定不能な場合でもユーザーの選択により申請フローを継続するため
- 適用される業務上の場面：BRL-01による判定不能時（CF-002）

### 3) 適用範囲
- 適用業務ID：BIZ-01
- 適用ユースケースID：UC-01（AF-1）
- 適用プロセス／ステップ：ST-001-02（代替フロー）
- 適用タイミング：
  - ☐ 業務開始時
  - ☑ 判断前
  - ☐ 実行前
  - ☐ 実行後

### 4) ルール内容

#### 4.1 自然言語定義（業務表現）
申請種別が特定できない場合、エージェントは「交通費精算申請」「経費精算申請」の2択をユーザーに提示し、ユーザーの選択に従って申請種別を確定する。

#### 4.2 構造化定義（正規形）
```
IF BRL-01（JD-02 NG）
THEN 「交通費精算申請」「経費精算申請」の2択を提示する
AND ユーザーが選択する
THEN 申請種別を選択値に確定 → CF-001 Step 3（BRL-03）へ復帰
ELSE（ユーザーが選択しない場合）→ セッション中断
```

### 5) 使用データ

| データID | データ名 | 利用方法 | 必須 |
|---|---|---|---:|
| D-001 | 申請内容（自由文） | 判定不能の根拠 | Yes |
| D-003 | 申請種別リスト | ユーザー選択後に保存 | Yes |

### 6) 対応関係

| 種別 | ID | 関係内容 |
|---|---|---|
| 判断基準 | JD-02 | 判定不能ケースの識別 |
| ユースケース | UC-01 | AF-1 |
| 業務プロセス | ST-001-02 | 代替フロー |
| ナレッジ | KN-001 | 社内申請ルール（申請種別判断） |

### 7) 例外・補足
- 例外条件：ユーザーが選択せず離脱した場合
- 例外時の扱い：セッションを中断・保留する
- 判断不可時の対応：申請担当者へのエスカレーション案内（エスカレーション先は要件上未定義）

## 業務ルール定義：BRL-04

### 1) 基本情報
- ルールID：BRL-04
- ルール名：必須項目不足時の質問
- ルール種別：手続
- 優先度：High
- 管理部門／責任者：要件上未定義

### 2) ルール概要
- ルールの目的：申請書の必須項目が不足している場合に不足項目を特定してユーザーに質問し、全必須項目を揃える
- このルールが必要な理由：申請書作成に必要な全項目が揃うまで対話収集を繰り返すため
- 適用される業務上の場面：申請情報収集フロー（CF-003またはCF-004）

### 3) 適用範囲
- 適用業務ID：BIZ-02
- 適用ユースケースID：UC-02
- 適用プロセス／ステップ：ST-002-01
- 適用タイミング：
  - ☐ 業務開始時
  - ☐ 判断前
  - ☑ 実行前
  - ☐ 実行後

### 4) ルール内容

#### 4.1 自然言語定義（業務表現）
申請書の必須項目定義（D-005）と照合し、未収集の必須項目を特定してユーザーに質問する。全必須項目が揃うまで繰り返す。

#### 4.2 構造化定義（正規形）
```
IF 収集済み申請情報（D-007）に未入力の必須項目がある（JD-04 NG）
THEN 未入力の必須項目をユーザーに質問する → ユーザーの回答を収集
AND 全必須項目の充足確認をループ
ELSE（JD-03 OK）→ 申請書自動作成（BRL-05）へ進む
```

### 5) 使用データ

| データID | データ名 | 利用方法 | 必須 |
|---|---|---|---:|
| D-005 | 申請書必須項目定義 | 必須項目との照合 | Yes |
| D-006 | ユーザー回答 | 収集した回答の保存 | Yes |
| D-007 | 収集済み申請情報 | 充足確認・更新 | Yes |

### 6) 対応関係

| 種別 | ID | 関係内容 |
|---|---|---|
| 判断基準 | JD-03, JD-04 | 必須項目充足確認 |
| ユースケース | UC-02 | 全体 |
| 業務プロセス | ST-002-01 | 情報収集ループ |
| ナレッジ | KN-002 | 申請書必須項目定義 |

### 7) 例外・補足
- 例外条件：ユーザーが回答できない場合
- 例外時の扱い：エスカレーション案内を提示する（BRL-10）
- 判断不可時の対応：申請担当者へのエスカレーション（エスカレーション先は要件上未定義）

## 業務ルール定義：BRL-09

### 1) 基本情報
- ルールID：BRL-09
- ルール名：申請書自動提出禁止
- ルール種別：制約
- 優先度：High
- 管理部門／責任者：要件上未定義

### 2) ルール概要
- ルールの目的：AIエージェントによる申請書の自動提出を禁止し、提出操作を必ず利用者が行うことを保証する
- このルールが必要な理由：申請書の提出は重要な業務行為であり、人の最終確認・責任を担保するため
- 適用される業務上の場面：申請書の提出操作全般（CF-007 Step 3）

### 3) 適用範囲
- 適用業務ID：BIZ-05
- 適用ユースケースID：UC-05
- 適用プロセス／ステップ：ST-005-01
- 適用タイミング：
  - ☐ 業務開始時
  - ☐ 判断前
  - ☐ 実行前
  - ☑ 実行後

### 4) ルール内容

#### 4.1 自然言語定義（業務表現）
申請書の提出操作は必ず利用者（R-EMP）が行う。AIエージェントは申請書の提出操作を行ってはならない。いかなる条件・例外においても自動提出は禁止する。

#### 4.2 構造化定義（正規形）
```
IF 申請書提出操作が発生する
THEN エージェントは提出操作を実行しない（ACT-EXEC-01の自律度 = Lv0）
AND 利用者（R-EMP）に提出操作を促すメッセージを提示する
AND 利用者が提出操作を実施する → 提出完了（D-011生成）
ELSE（利用者が提出操作をキャンセルした場合）→ セッション保留
```

### 5) 使用データ

| データID | データ名 | 利用方法 | 必須 |
|---|---|---|---:|
| D-008 | 申請書ドラフト | 提示の対象 | Yes |
| D-011 | 提出済み申請書 | 提出後に生成される成果物 | Yes |

### 6) 対応関係

| 種別 | ID | 関係内容 |
|---|---|---|
| 判断基準 | — | 例外なし |
| ユースケース | UC-05 | Step 3 |
| 業務プロセス | ST-005-01 | 提出操作ノード |
| ナレッジ | KN-009 | 申請書提出禁止ルール |
| 権限 | ACT-EXEC-01 | 自律度Lv0（禁止）に対応 |

### 7) 例外・補足
- 例外条件：なし（例外は一切認めない）
- 例外時の扱い：適用外
- 判断不可時の対応：適用外

## 業務ルール定義：BRL-12

### 1) 基本情報
- ルールID：BRL-12
- ルール名：交通費：運賃自動計算
- ルール種別：判断
- 優先度：High
- 管理部門／責任者：要件上未定義

### 2) ルール概要
- ルールの目的：移動情報（出発地・目的地・交通手段）から運賃データを参照して交通費を自動計算する
- このルールが必要な理由：ユーザーが運賃を手動入力する手間を省き、正確な運賃を提示するため
- 適用される業務上の場面：交通費情報収集フロー（CF-003 Step 3）

### 3) 適用範囲
- 適用業務ID：BIZ-02
- 適用ユースケースID：UC-02
- 適用プロセス／ステップ：ST-002-01
- 適用タイミング：
  - ☐ 業務開始時
  - ☐ 判断前
  - ☑ 実行前
  - ☐ 実行後

### 4) ルール内容

#### 4.1 自然言語定義（業務表現）
移動情報（出発地・目的地・交通手段）から交通手段種別に応じた運賃データを参照し、交通費を自動計算する。電車は経路テーブル検索、バス・タクシー・飛行機は固定運賃データを参照する。

#### 4.2 構造化定義（正規形）
```
IF 交通手段 = 電車
THEN DATA-011（電車経路・運賃JSON）から（出発地, 目的地）の運賃を検索
  AND 該当データがある → 運賃を自動設定
  AND 該当データなし → エラー：手動入力を促す
ELSE IF 交通手段 IN [バス, タクシー, 飛行機]
THEN DATA-012（固定運賃JSON）から交通手段の固定運賃を取得 → 運賃を自動設定
  （バス:230円, タクシー:10,000円, 飛行機:50,000円）
ELSE 対応外の交通手段 → エラーメッセージを返す
```

### 5) 使用データ

| データID | データ名 | 利用方法 | 必須 |
|---|---|---|---:|
| D-006 | ユーザー回答（移動情報） | 出発地・目的地・交通手段の参照 | Yes |
| DATA-011 | 運賃データ（電車経路・運賃JSON） | 電車の経路・運賃検索（ファイル参照） | Yes |
| DATA-012 | 運賃データ（固定運賃JSON） | バス・タクシー・飛行機の固定運賃参照（ファイル参照） | Yes |

### 6) 対応関係

| 種別 | ID | 関係内容 |
|---|---|---|
| 判断基準 | — | |
| ユースケース | UC-02 | Step 3 |
| 業務プロセス | ST-002-01 | 運賃計算ノード |
| ナレッジ | KN-003 | 運賃計算ルール |
| ツール | TOOL-001 | 交通費計算ツールにより実行 |

### 7) 例外・補足
- 例外条件：経路テーブルに対象経路がない場合
- 例外時の扱い：エラーメッセージを返し手動入力を促す
- 判断不可時の対応：ユーザーが費用を手動入力する

## 業務ルール定義：BRL-13

### 1) 基本情報
- ルールID：BRL-13
- ルール名：交通費：申請期限チェック
- ルール種別：制約
- 優先度：High
- 管理部門／責任者：要件上未定義

### 2) ルール概要
- ルールの目的：交通費精算申請の申請期限（経費発生日から3ヶ月以内）を超過していないか確認する
- このルールが必要な理由：期限超過の申請を受け付けないことで申請ルールを遵守するため
- 適用される業務上の場面：申請チェックフロー（CF-005でのルール照合時）

### 3) 適用範囲
- 適用業務ID：BIZ-04
- 適用ユースケースID：UC-04
- 適用プロセス／ステップ：ST-004-01
- 適用タイミング：
  - ☐ 業務開始時
  - ☐ 判断前
  - ☑ 実行前
  - ☐ 実行後

### 4) ルール内容

#### 4.1 自然言語定義（業務表現）
交通費精算申請は経費発生日から3ヶ月以内に申請しなければならない。3ヶ月を超過した申請は受け付けられない。

#### 4.2 構造化定義（正規形）
```
IF (申請日 - 移動日) <= 3ヶ月（JD-09 OK）
THEN 申請期限チェック = OK → 申請処理を継続
ELSE（JD-10 NG）
THEN 申請期限超過を通知（D-009に記録）→ CF-008（申請期限超過通知会話）へ遷移
```

### 5) 使用データ

| データID | データ名 | 利用方法 | 必須 |
|---|---|---|---:|
| D-007 | 収集済み申請情報 | 移動日の参照 | Yes |
| D-009 | 申請チェック結果 | チェック結果の記録 | Yes |

### 6) 対応関係

| 種別 | ID | 関係内容 |
|---|---|---|
| 判断基準 | JD-09, JD-10 | 3ヶ月以内の判定 |
| ユースケース | UC-04 | チェック処理 |
| 業務プロセス | ST-004-01 | 申請期限チェックノード |
| ナレッジ | KN-004 | 申請期限ルール |

### 7) 例外・補足
- 例外条件：なし（3ヶ月超は例外なく申請不可）
- 例外時の扱い：申請不可を通知し、CF-008へ遷移する
- 判断不可時の対応：適用外

## 業務ルール定義：BRL-14

### 2) ルール概要

- このルールが必要な理由：高額の交通費申請に対する統制・ガバナンスを確保するため
- 適用される業務上の場面：申請チェックフロー（CF-005でのルール照合時）
