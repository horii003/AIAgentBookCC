# 前提条件

- 規約（`.claude/skills/06-code-generation/00_rule_project_conventions.md`）の内容を遵守すること

- コーディング規約・命名規則・ディレクトリ構造等、実装に影響するルールを把握すること

# 出力形式
tasks.md（実装タスク計画書）

## tasks.mdのフォーマット

各タスクは以下の形式で記載すること：

```markdown
## タスク[番号]: [タスク名]

- **ステータス**: [ ] 未着手
- **参照する設計書**: [設計書ファイルパス（artifacts/配下）]
- **参照するスケルトンコード**: [スケルトンコードファイルパス]
- **成果物のファイルパス**: [artifacts/06_code-generation/src/配下の出力先パス]
- **単体テストコードのファイルパス**: [artifacts/06_code-generation/src/tests/unit/配下の出力先パス]
- **実装内容**:
  - [具体的な実装内容を箇条書きで記載]
- **単体テスト内容**:
  - [具体的な単体テスト項目を箇条書きで記載]
```

## 結合テストタスク

全実装タスクの後に、以下の形式で結合テストタスクを記載すること：

```markdown
## タスク[番号]: 結合テスト

- **ステータス**: [ ] 未着手
- **テスト対象**: [結合テスト対象のモジュール・連携パスを箇条書きで記載]
- **結合テストコードのファイルパス**: [artifacts/06_code-generation/src/tests/integration/配下の出力先パス]
- **結合テスト内容**:
  - [具体的な結合テスト項目を箇条書きで記載]
```

### 結合テストの観点
- 複数モジュールを組み合わせた連携動作を検証するテストファイルを作成する

## タスク分割の方針
- 1つのスケルトンコードに対して1タスクを基本とすること
- エージェントやツールなど複数ファイルが生成されるものは、個別のサブタスクとして記載すること
- 依存関係がある場合はタスクの説明に明記すること
- 全実装タスクの後に結合テストタスクを必ず1つ以上配置すること

## E2Eテストタスク

結合テスト完了後に、以下の形式でE2Eテストタスクを記載すること：

```markdown
## タスク[番号]: E2Eテスト

- **ステータス**: [ ] 未着手
- **テスト対象**: [E2Eテストの内容を箇条書きで記載]
- **E2Eテストコードのファイルパス**: [artifacts/06_code-generation/src/tests/e2e/配下の出力先パス]
- **E2Eテスト内容**:
  - [具体的な結合テスト項目を箇条書きで記載]
```

### E2Eテストの観点
- 実装したAIエージェントを起動し、実際にエンドツーエンド（E2E）のテストで動作確認を実施するテストファイルを作成する
- E2Eテストの結果、申請書ファイルが作成されるところまでを動作確認する
- E2Eテストを実施した結果、発見した不具合を修正する


# 受入基準
作成後に以下の受入基準を満たしているか確認すること。
- すべてのスケルトンコード（01～14）に対応するタスクが存在すること
- 各タスクに参照する設計書ファイル名が記載されていること
- 各タスクに参照するスケルトンコードファイル名が記載されていること
- 各タスクに成果物のファイルパスが記載されていること
- 各タスクに実装内容が具体的に記載されていること
- 各タスクにテスト内容が記載されていること
- タスクの順序が依存関係を考慮した順序になっていること
- 設計書のファイルパスがartifacts/配下の実際のパスであること（テンプレートパスを転記していないこと）
- 各タスクの「参照する設計書」が、スケルトンコードの内容と整合していること（無関係な設計書を参照していないこと）
- 規約ファイル（.claude/skills/06-code-generation/SKILL.md）が全タスク共通の参照元として記載されていること
