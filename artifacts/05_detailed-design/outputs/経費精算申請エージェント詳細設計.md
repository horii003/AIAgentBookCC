## 1. 概要

### 1.1 コンポーネントの目的

AG-001（申請受付窓口エージェント）から委譲された経費精算申請フローを引き受け、経費情報（店舗名・金額・経費発生日・品目・経費区分・業務目的）を対話で収集する。領収書画像は `strands_tools` の `image_reader` ツールで情報を自動抽出し、失敗時は手動入力フローへ切り替える。経費区分を自動判断して社員に提案し、確認を経て申請書（下書き）を生成（TL-002b）する。

### 1.2 主要な責務

1. **invocation_stateの受け取り**: ToolContextから`session_id`・`applicant_name`・`application_date`を取得する
2. **Agentインスタンスキャッシュ管理**: `_agent_instances` モジュールレベル辞書で session_id をキーにインスタンスをキャッシュし、SlidingWindowConversationManager の会話履歴を保持する
3. **動的システムプロンプト生成**: `_build_expense_agent_system_prompt(application_date)` で申請日・申請期限日を埋め込んだプロンプトを生成する
4. **領収書画像からの情報抽出（BRL-16）**: `strands_tools` の `image_reader` ツールで店舗名・金額・経費発生日・品目を自動抽出する。失敗時は手動入力フローへ自動切り替え（GRD-003）
5. **申請期限チェック（BRL-14）**: 経費発生日が申請期限（申請日 - 90日）より前の場合は申請不可を通知してフローを終了する
6. **高額申請通知（BRL-10）**: 経費が5,000円超の場合に上長承認要否を通知する（フロー継続）
7. **経費区分の自動判断と提案（BRL-17/GRD-010）**: 品目から経費区分（事務用品費・宿泊費・資格精算費・その他経費）を自動判断して提案し、必ず社員の確認を取得する。自動確定は禁止
8. **申請書生成前社員確認（GRD-014）**: HumanApprovalHook（BeforeToolCallEvent）で OK/修正/キャンセルを取得する。承認前に TL-002b を実行してはならない
9. **申請書（下書き）生成（TL-002b）**: HumanApprovalHook の OK 取得後のみ generate_expense_reimbursement_form を呼び出す

### 2.2 インターフェース設計

#### 2.2.1 入力

**ツール関数引数（AG-001からの呼び出し時）**:
- `application_type` (str): 申請種別（"経費精算申請"確定済み）（必須）
- `applicant_name` (str): 申請者名（必須）
- `user_input_text` (str): 社員の申請内容テキスト（最大500文字）（必須）

**ToolContext経由で受け取る情報**:
```python
{
    "session_id": str,          # セッションID（Agentインスタンスキャッシュのキーとして使用）
    "applicant_name": str,      # 申請者名
    "application_date": str     # 申請日（YYYY-MM-DD形式）
}
```

**対話ループでの入力**:
- 社員からの自然言語入力（領収書アップロード・経費情報・経費区分確認応答等）

#### 2.2.2 出力

**出力先**: AG-001（tool応答として返す）

**戻り値の形式**: `str`（エージェントからの最終応答テキスト）

**主要な出力メッセージ**:
- 領収書アップロード依頼: `"経費精算申請を承ります。領収書の画像をアップロードしてください。画像がない場合は「手動入力」とお伝えください。"`
- LLM抽出失敗案内: `"領収書の読み取りができませんでした。以下の項目を順番に入力してください。\n店舗名・金額・経費発生日（YYYY-MM-DD形式）・品目"`
- 申請期限超過通知: `"経費発生日「{expense_date}」から90日を超えているため、経費精算申請はできません（根拠: BRL-14）。担当部門にご確認ください。"`
- 高額申請通知: `"経費が5,000円を超えているため、上長の承認が必要です（根拠: BRL-10）。フローは継続します。"`
- 経費区分提案: `"品目「{item_name}」から経費区分を「{expense_category}」と判断しました（参照: BRL-17）。よろしければそのまま「OK」とお答えください。修正が必要な場合は正しい経費区分をご入力ください。"`
- 申請書生成完了: `"申請書（下書き）を「{file_path}」に生成しました。申請先に提出してください。"`

### 2.3 ビジネスロジック

#### 2.3.1 システムプロンプト

##### システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 動的生成関数 |
| 動的生成の理由 | 申請期限チェック（BRL-14）で経費発生日から90日以内かを判断するために、申請日（`application_date`）と申請期限日（`deadline_date`）をプロンプトに埋め込む必要がある |

##### 動的に埋め込む情報
- `application_date`（申請日、YYYY-MM-DD形式）: `invocation_state["application_date"]` から取得
- `deadline_date`（申請期限日、YYYY-MM-DD形式）: `(date.fromisoformat(application_date) - timedelta(days=90)).isoformat()`
- 経費精算に関するポリシー情報: `agent_knowledge/receipt_policies.py` から取得して埋め込む

##### プロンプト全文

```python
def _build_expense_agent_system_prompt(application_date: str) -> str:
    from datetime import date, timedelta
    from agent_knowledge.receipt_policies import get_receipt_policies
    deadline_date = (date.fromisoformat(application_date) - timedelta(days=90)).isoformat()
    policies = get_receipt_policies()
    return f"""あなたは経費精算申請の専門エージェントです。経費情報を収集し、経費区分を判断して申請書（下書き）を生成します。

【申請日・申請期限】
- 申請日: {application_date}
- 申請期限: {deadline_date}以降の経費発生日のみ申請可能（経費発生日がこれより前の場合は申請不可）

【役割と責務】
1. 領収書画像を `image_reader` ツールで解析し、店舗名・金額・経費発生日・品目を自動抽出する
2. 抽出情報を社員に確認し、不足情報を対話で収集する
3. 品目から経費区分を自動判断して提案し、社員の確認を得る（自動確定は禁止）
4. すべての経費情報が収集できたら、収集済み情報をテキストとして整理・提示する
5. 社員の確認（OK/修正/キャンセル）を取得後、generate_expense_reimbursement_form を呼び出して申請書（下書き）を生成する

【領収書アップロードの処理（BRL-16）】
- まず社員に領収書画像のアップロードを依頼する
  「経費精算申請を承ります。領収書の画像をアップロードしてください。画像がない場合は「手動入力」とお伝えください。」
- 画像がアップロードされた場合: `image_reader` ツールを呼び出して店舗名・金額・経費発生日・品目を自動抽出する
  - 成功時: 抽出した情報を提示して確認を求める
  - 失敗時（GRD-003）: 「領収書の読み取りができませんでした。以下の項目を順番に入力してください。\n店舗名・金額・経費発生日（YYYY-MM-DD形式）・品目」と案内して手動入力フローへ切り替える
- 「手動入力」と回答された場合: 「店舗名・金額・経費発生日（YYYY-MM-DD形式）・品目を入力してください。」と案内して手動入力フローへ移行する

【申請期限チェック（BRL-14）】
- 経費発生日（expense_date）が確定したら、申請期限（{deadline_date}）より前かどうかを確認する
- expense_date < {deadline_date} の場合は申請不可を通知してフローを終了する
- メッセージ例: 「経費発生日「YYYY-MM-DD」から90日を超えているため、経費精算申請はできません（根拠: BRL-14）。担当部門にご確認ください。」

【高額申請通知（BRL-10）】
- 金額が5,000円を超える場合: 「経費が5,000円を超えているため、上長の承認が必要です（根拠: BRL-10）。」と通知する
- 高額通知後もフローは継続する（フロー終了ではない）

【経費区分の自動判断と提案（BRL-17・GRD-010）】
- 品目が確定したら、以下のルールで経費区分を自動判断して「提案」として提示する
  - 事務用品・文具・消耗品 → 「事務用品費」
  - ホテル・宿泊・ビジネスホテル → 「宿泊費」
  - 資格・検定・試験・受験 → 「資格精算費」
  - 上記以外・判断不能 → 「その他経費」
- 提案メッセージ例:
  「品目「{item_name}」から経費区分を「{expense_category}」と判断しました（参照: BRL-17）。よろしければそのまま「OK」とお答えください。修正が必要な場合は正しい経費区分をご入力ください。」
- 社員が修正した場合はその内容を経費区分として確定する
- 経費区分を社員の確認なしに自動確定してはならない（GRD-010）
- 有効な経費区分: 事務用品費・宿泊費・資格精算費・その他経費 の4種類のみ

【不足情報の収集】
- LLM抽出または手動入力後に不足している必須項目を確認して収集する
- 業務目的は領収書から取得できないため、必ず手動で収集する
- 必須項目が揃った場合はStep 11（収集済み情報の整理・提示）へ進む

【収集済み経費情報の提示（ドラフト提示）】
- すべての経費情報が収集完了したら、次のように整理して提示する（ツール呼び出しなし）:
  「以下の申請情報をご確認ください。
  【経費精算申請】
  申請者: {applicant_name}
  申請日: {application_date}
  明細1:
    経費発生日: {expense_date}  店舗名: {store_name}
    金額: {amount}円  品目: {item_name}
    経費区分: {expense_category}  業務目的: {purpose}
  （以下、収集済み全明細分）
  上記の内容でよろしいですか？」
- この提示はテキスト整理であり、ツール呼び出しを行ってはならない

【申請書生成（TL-002b）】
- 社員から「OK」または承認の意思が示された後に generate_expense_reimbursement_form を呼び出す
- 承認前に generate_expense_reimbursement_form を呼び出してはならない
- generate_expense_reimbursement_form の呼び出しパラメータ:
  {{
    "applicant_name": "申請者名",
    "application_date": "YYYY-MM-DD形式の申請日",
    "items": [
      {{
        "expense_date": "YYYY-MM-DD形式の経費発生日",
        "store_name": "店舗名",
        "amount": 金額（整数）,
        "item_name": "品目",
        "expense_category": "経費区分（事務用品費・宿泊費・資格精算費・その他経費のいずれか）",
        "purpose": "業務目的"
      }}
    ]
  }}
- 申請書生成完了時: 「申請書（下書き）を「{{file_path}}」に生成しました。申請先に提出してください。」と案内する

【禁止事項】
- 申請書の提出操作は絶対に行わない（GRD-012）。提出は社員が行う
- 上長承認の代行は絶対に行わない（GRD-013）。承認取得は社員が行う
- 申請書生成は社員の確認（OK）取得後のみ実施する（GRD-014）
- 経費区分を社員の確認なしに自動確定してはならない（GRD-010）

【エラー対応】
- ツールエラーやシステム障害が発生した場合は、エラー内容を要約して呼び出し元（AG-001）に返す
- 対話継続不能な場合は「担当部門（管理部）にお問い合わせください。」と案内する
"""
```

##### プロンプト設計の意図

| 要素 | 意図 |
|------|------|
| 動的プロンプト（申請日・期限日） | BRL-14の申請期限チェックに実行時日付が必要なため、毎回生成する |
| 領収書処理の2経路 | LLM抽出→手動入力切り替えフローを明示し、GRD-003の要件を確実に実装する |
| 経費区分「提案」の明示 | GRD-010（自動確定禁止）をプロンプトで確実に実装。「提案」「社員確認」のワードを明示する |
| generate_expense_reimbursement_formパラメータ例示 | dict型引数（items）の必須キー・型・構造を明示し、LLMがキーを省略する実行時エラーを防ぐ |
| 禁止事項明示 | GRD-010（自動確定禁止）・GRD-012（提出禁止）・GRD-013（上長承認代行禁止）・GRD-014（確認前生成禁止）を明示する |

#### 2.3.2 申請期限チェック（BRL-14）

| ルール | 条件 | 対応 |
|--------|------|------|
| BRL-14: 申請期限90日チェック | expense_date < deadline_date（申請日 - 90日） | 申請不可を通知してフロー終了。EX-03として処理 |
| BRL-14: 期限内 | expense_date >= deadline_date | フロー継続 |

#### 2.3.3 高額申請チェック（BRL-10）

| ルール | 条件 | 対応 |
|--------|------|------|
| BRL-10: 上長承認要否 | amount > 5000 | 上長承認要否を通知してフロー継続 |
| BRL-10: 通常 | amount <= 5000 | 通知なしでフロー継続 |

#### 2.3.4 経費区分の自動判断ルール（BRL-17）

| 品目のキーワード | 経費区分 |
|---------------|---------|
| 事務用品、文具、消耗品、ボールペン、ノート等 | 事務用品費 |
| ホテル、宿泊、ビジネスホテル、旅館等 | 宿泊費 |
| 資格、検定、試験、受験、TOEIC等 | 資格精算費 |
| 上記以外・判断不能 | その他経費 |

#### 2.3.5 処理フロー（CF-004 11ステップ）

```
開始（expense_agent_tool 呼び出し）
  ↓
【ステップ1】invocation_stateの受け取り
  - tool_context.invocation_state から session_id, applicant_name, application_date を取得
  - InvocationState.model_validate(tool_context.invocation_state) で検証
  ↓
【ステップ2】Agentインスタンスの取得またはキャッシュ生成
  - session_id が _agent_instances に存在するか確認
  - 存在しない場合: Agent(..., system_prompt=_build_expense_agent_system_prompt(application_date), callback_handler=None) を生成して登録
  - 存在する場合: キャッシュを再利用（会話履歴を保持）
  ↓
【ステップ3】エージェント呼び出し（対話ループ開始）
  agent_instance(user_input_text, invocation_state=tool_context.invocation_state) を呼び出す
  ↓
  === 以下はエージェント内部の対話ループ（CF-004） ===
  ↓
【CF-004 Step 1: 領収書アップロード依頼】
  - 「経費精算申請を承ります。領収書の画像をアップロードしてください。」を提示
  ↓
【CF-004 Step 2: 社員の選択】
  - [画像アップロード] → Step 3（LLM抽出）へ進む
  - [「手動入力」回答] → Step 4（手動入力）へ進む
  ↓
【CF-004 Step 3: image_readerツールによる情報抽出（BRL-16）】
  - `image_reader` ツール（`strands_tools`）を呼び出して店舗名・金額・経費発生日・品目を抽出
  - 成功: 抽出情報を提示して確認 → Step 5へ
  - 失敗（GRD-003）: 「領収書の読み取りができませんでした。手動で入力してください。」 → Step 4へ
  ↓
【CF-004 Step 4: 手動入力（LLM抽出失敗時または手動選択時）】
  - 「店舗名・金額・経費発生日（YYYY-MM-DD形式）・品目を入力してください。」を提示
  - 社員から入力を受け取る
  ↓
【CF-004 Step 5: 申請期限チェック（BRL-14）】
  - expense_date を解析
  - expense_date < deadline_date の場合:
    → 申請不可メッセージを提示してフロー終了（EX-03）
  - expense_date >= deadline_date の場合:
    → Step 6へ進む
  ↓
【CF-004 Step 6: 高額チェック・上長承認要否通知（BRL-10）】
  - amount > 5000 の場合: 「経費が5,000円を超えているため、上長の承認が必要です（根拠: BRL-10）。」と通知
  - フロー継続 → Step 7へ
  ↓
【CF-004 Step 7: 経費区分の自動判断と提案（BRL-17）】
  - 品目から経費区分（事務用品費/宿泊費/資格精算費/その他経費）を自動判断
  - 「品目「{item_name}」から経費区分を「{expense_category}」と判断しました...」を提示
  ↓
【CF-004 Step 8: 社員による経費区分の確認（GRD-010）】
  - 社員が OK → 提案の経費区分を確定
  - 社員が修正 → 入力された経費区分を確定
  → Step 9へ
  ↓
【CF-004 Step 9-10: 不足情報の確認・収集ループ】
  - 未収集の必須項目（業務目的等）を確認
  - 不足あり → 不足項目を収集（Step 10）→ Step 9に戻る
  - 全必須項目収集完了 → Step 11へ進む
  ↓
【CF-004 Step 11: 収集済み経費情報の提示（ドラフト提示）】
  - 全収集済み経費情報をテキストとして整理・提示（ツール呼び出しなし）
  - 「以下の申請情報をご確認ください。...上記の内容でよろしいですか？」
  ↓
【HumanApprovalHook: BeforeToolCallEvent介入】
  generate_expense_reimbursement_form 呼び出し直前に HumanApprovalHook が介入:
  - [OK] → generate_expense_reimbursement_form 実行を許可
  - [修正] → 修正対象の項目再収集フローへ戻る
  - [キャンセル] → セッション状態をCLOSED、フロー終了
  ↓
【generate_expense_reimbursement_form 呼び出し（TL-002b）】
  - ExpenseFormInput（applicant_name, application_date, items）を渡す
  - 成功: file_path を取得 → 申請書生成完了メッセージを提示
  - 失敗（EX-04）: CF-006に誘導して中断
  - 失敗（EX-06）: 1回リトライ後も失敗の場合はCF-006にエスカレーション
  ↓
終了（expense_agent_tool の戻り値として応答テキストを返す）
```

### 2.5 連携設計

#### 2.5.1 Agent as Tools設計

**ツール名**: `expense_agent_tool`

**ツール説明**:
```
経費精算申請エージェント（AG-003）を呼び出し、経費情報の収集・経費区分判断・申請書生成を行う。
AG-001が申請種別「経費精算申請」と確定した後に呼び出す。
```

**ToolContextの使用**:
- `@tool(context=True)` デコレータを使用
- `tool_context: ToolContext` パラメータで ToolContext を受け取る
- `tool_context.invocation_state` から `session_id`・`applicant_name`・`application_date` を取得

#### 2.5.2 invocation_stateの受け渡しとAgentインスタンスのキャッシュ

```python
# session_id をキーにして Agent インスタンスをキャッシュし、
# セッション内で会話履歴を保持する（情報の再入力を防ぐ）
_agent_instances: dict[str, Agent] = {}

@tool(context=True)
def expense_agent_tool(
    application_type: str,
    applicant_name: str,
    user_input_text: str,
    tool_context: ToolContext,
) -> str:
    # invocation_state から状態を取得
    state = tool_context.invocation_state
    session_id = state.get("session_id", "")
    application_date = state.get("application_date", "")

    # session_id 未登録の場合のみ新しい Agent インスタンスを生成する
    # Agent初期化時に callback_handler=None を指定して標準出力への直接表示を抑制する
    # （AG-001による二重出力を防ぐ。agent()呼び出し時に渡すと TypeError になるため必ずコンストラクタで指定）
    if session_id not in _agent_instances:
        _agent_instances[session_id] = Agent(
            model=BedrockModel(model_id="jp.anthropic.claude-sonnet-4-5-20250929-v1:0"),
            system_prompt=_build_expense_agent_system_prompt(application_date),
            tools=[image_reader, generate_expense_reimbursement_form],
            conversation_manager=SlidingWindowConversationManager(window_size=15),
            hooks=[LoopControlHook(max_iterations=10), HumanApprovalHook(approval_callback=approval_callback)],
            session_manager=SessionManagerFactory.create(),
            callback_handler=None,
        )

    response = _agent_instances[session_id](user_input_text, invocation_state=state)
    return str(response)
```

## 3. 実装詳細

### 3.1 クラス/関数設計

#### 3.1.1 `_build_expense_agent_system_prompt` 関数

##### 説明
申請日・申請期限日・経費精算ポリシーを埋め込んだ経費精算申請エージェントのシステムプロンプトを動的生成する。

##### 引数
- `application_date` (str): 申請日（YYYY-MM-DD形式）（必須）

##### 戻り値
- `str`: 完全なシステムプロンプトテキスト

##### 処理内容
1. `deadline_date = (date.fromisoformat(application_date) - timedelta(days=90)).isoformat()` を計算する
2. `agent_knowledge/receipt_policies.py` の `get_receipt_policies()` からポリシー情報を取得する
3. `application_date`・`deadline_date`・`policies` を埋め込んだプロンプト文字列を返す

#### 3.1.2 `expense_agent_tool` 関数

##### 説明
AG-003（経費精算申請エージェント）をツールとして公開し、AG-001からの呼び出しを受け付ける。

##### 引数
- `application_type` (str): 申請種別（"経費精算申請"確定済み）（必須）
- `applicant_name` (str): 申請者名（必須）
- `user_input_text` (str): 社員の申請内容テキスト（必須）
- `tool_context` (ToolContext): ToolContext（`@tool(context=True)` による注入）（必須）

##### 戻り値
- `str`: AG-003エージェントからの応答テキスト

##### 処理内容
1. `tool_context.invocation_state` から `session_id`・`application_date` を取得する
2. `_agent_instances` に `session_id` が存在しない場合、新しい Agent インスタンスを生成して登録する
3. `_agent_instances[session_id](user_input_text, invocation_state=state)` を呼び出す
4. 応答を `str` にキャストして返す

##### 例外処理
- RuntimeError（ループ制限）、RuntimeError（その他）、Exception は `3.2 エラーハンドリング` に従い捕捉・処理する

## 5. 補足情報

### 5.1 実装上の注意点

1. **callback_handler=None はコンストラクタで指定**
   - `Agent(..., callback_handler=None)` のようにコンストラクタで指定する。`agent(query, callback_handler=None)` のように呼び出し時に渡すと `TypeError` になる。

2. **_agent_instancesはモジュールレベルで定義**
   - `_agent_instances: dict[str, Agent] = {}` をモジュールレベルで定義し、ツール関数呼び出しのたびに再生成してはならない。同一セッション内で SlidingWindowConversationManager の会話履歴を保持するために必須。

3. **領収書画像読み取りはimage_readerツールを使用**
   - 領収書画像の解析は `strands_tools` の `image_reader` ツールを使用する（LLMネイティブ画像認識ではなく、ツール経由で呼び出す）。エージェントの `tools` リストに `image_reader` を追加すること。外部OCRサービスは不使用。

4. **経費区分の提案・確認フロー（GRD-010）の実装**
   - システムプロンプトに「経費区分を社員の確認なしに自動確定してはならない」と明示することで GRD-010 をプロンプトレベルで実装する。

5. **ドラフト提示とツール呼び出しの分離**
   - Step 11 の収集済み経費情報提示はテキスト整理（ツール呼び出しなし）として設計する。その後 generate_expense_reimbursement_form を呼び出した時点で HumanApprovalHook（BeforeToolCallEvent）が介入する。

---

### 5.2 パフォーマンス考慮事項

1. **LLM画像読み取りのレイテンシ**
   - 領収書画像の読み取りはLLMの推論コストに含まれるため、画像サイズが大きい場合はレイテンシが増加する。合理的なサイズ制限（10MB以下推奨）を実装を検討する。

2. **SlidingWindowConversationManager のウィンドウサイズ**
   - `window_size=15` は1明細単位の収集フローに対応するために設定。複数明細を収集する場合はウィンドウサイズの見直しが必要。

---

### 5.3 セキュリティ考慮事項

1. **invocation_stateの信頼性**
   - `tool_context.invocation_state` は AG-001 が設定した値を受け取る。外部からの直接操作を防ぐため、session_id のフォーマット検証（`YYYYMMDDHHMMSSxxxxxxxx`、例: `20260502143022_a1b2c3d4`）を実施する。

2. **個人情報（applicant_name）のログマスキング**
   - ログ出力時に `applicant_name` は `applicant_name[:1] + "***"` 形式でマスキングする（GRD-011）。

## 7. テスト観点

### 7.1 機能テスト
- 領収書画像アップロード時にLLMが店舗名・金額・経費発生日・品目を正しく抽出すること（CF-004 Step 3）
- LLM抽出失敗時に手動入力フローへ自動切り替えが動作すること（GRD-003）
- 手動入力選択時に手動入力フローへ移行すること（CF-004 Step 4）
- 品目から経費区分が正しく自動判断されること（BRL-17）
- 経費区分提案に対して社員が修正した場合に修正後の経費区分が確定すること（GRD-010）
- 不足情報収集ループ（Step 9-10）で業務目的が収集されること
- 収集済み経費情報のドラフト提示がツール呼び出しなしのテキスト整理として動作すること
- HumanApprovalHook の OK 選択後に generate_expense_reimbursement_form が呼び出されること
- generate_expense_reimbursement_form が成功し、file_path が提示されること
- 同一セッション内で expense_agent_tool を複数回呼び出した場合に、同一の Agent インスタンスが再利用されること

### 7.2 異常系テスト
- 経費発生日が申請期限（90日前）を超えている場合に申請不可メッセージが返却され、フローが終了すること（BRL-14）
- 経費発生日がちょうど90日前の場合にフローが継続すること（BRL-14 境界値）
- 金額が5,000円超の場合に上長承認通知メッセージが返却されること（BRL-10）
- 金額がちょうど5,000円の場合に通知なしでフローが継続すること（BRL-10 境界値）
- 経費区分の自動確定が試みられた場合に社員確認ステップが発生すること（GRD-010）
- HumanApprovalHook の修正選択時に TL-002b が実行されず、対話に戻ること
- HumanApprovalHook のキャンセル選択時に TL-002b が実行されず、フローが終了すること
- LoopControlHook が 10 反復を超えた場合に EX-08 対応メッセージが返却されること
- 承認（OK）前に generate_expense_reimbursement_form が実行されないこと（GRD-014）
