## 1. 概要

### 1.1 コンポーネントの目的

交通費精算申請に特化したスペシャリストエージェントとして、AG-001からの委任を受けて申請フロー全体を実行する。移動情報を一区間ずつ一括入力形式で対話収集し、TOOL-001（calculate_transport_expense）で運賃を自動計算する。収集済み申請情報をテキストで整理・提示し、HumanApprovalHook（BeforeToolCallEvent）による承認後にTOOL-002（generate_transport_expense_form）でExcelドラフトを生成する。申請期限チェック（BRL-13）・上長承認要否判定（BRL-14）・差し戻しリスク評価（BRL-08）を実施してチェック結果をユーザーへ提示する。

### 1.2 主要な責務

- **invocation_stateの受け取り**: ToolContext経由で applicant_name・application_date・session_id を受け取る
- **移動情報の対話収集（一区間ずつ一括入力）**: 移動日・出発地・目的地・交通手段を一区間分まとめて収集する（BRL-11）
- **交通費自動計算**: TOOL-001（calculate_transport_expense）で移動区間の運賃を自動算出する（BRL-12）
- **駅名正規化**: TOOL-001呼び出し前にユーザー入力の駅名を正規形に変換する（BRL-15）
- **業務目的の確認**: CF-003 Step 5-6で業務目的を収集する（BRL-20）
- **申請バリデーションルールの参照**: `agent_knowledge/transportation_policies.py` に定義されたバリデーションルール・申請ポリシーをシステムプロンプトに組み込む
- **Human-in-the-Loop承認取得**: 収集済み申請情報をテキストで整理・提示してOK/修正/キャンセルの3択を取得する（BRL-06）。ツール呼び出しは行わない（ドラフト提示ステップ）
- **申請書生成**: HumanApprovalHook（BeforeToolCallEvent）承認後にTOOL-002（generate_transport_expense_form）でExcelドラフトを生成する（BRL-05）
- **申請ルールチェック**: 申請期限（BRL-13）・上長承認要否（BRL-14）・差し戻しリスク（BRL-08）をチェックしてユーザーへ提示する（CF-005）

### 2.2 インターフェース設計

#### 2.2.1 入力

**ToolContext経由で受け取る情報**（invocation_state）:
```python
{
    "session_id": str,       # セッション識別子（{YYYYMMDDHHmmss}_{UUID 8文字}形式）。ファクトリ関数内でFileSessionManager初期化に使用
    "applicant_name": str,   # 申請者名（min_length=1）
    "application_date": str  # 申請日（YYYY-MM-DD形式。システム日付）
}
```

**対話ループでの入力**:
- 移動情報（一区間一括入力）: 移動日・出発地・目的地・交通手段を1メッセージで一括入力するよう促す
- 業務目的: ユーザーが自由文で入力する（BRL-20）
- Human-in-the-Loop選択: OK / 修正 / キャンセルから選択する
- 特殊コマンド:
  - TOOL-001経路テーブル未登録時の手動費用入力
  - 申請期限超過時のCF-008遷移確認
  - 上長承認要時のCF-009遷移確認

**パラメータ**:
- `query` (str): AG-001から渡される申請開始クエリ

#### 2.2.2 出力

**出力先**: CLI（標準出力）。`callback_handler=None` で標準出力への直接表示を抑制し、AG-001経由で表示する

**出力メッセージ**:
- 移動情報収集開始（一括入力形式）: `"交通費精算申請を開始します。\n1区間分の移動情報をまとめて入力してください。\n（例：2026-04-28 渋谷 新宿 電車）\n移動日・出発地・目的地・交通手段を半角スペースまたは改行で区切ってください："`
- 運賃計算結果提示: `"[出発地]→[目的地]（[交通手段]）の交通費は[運賃]円です（[計算根拠]）。"`
- 追加区間確認: `"他にも移動区間はありますか？（ある場合は「はい」、ない場合は「いいえ」）"`
- ドラフト提示（テキスト）: `"収集した申請情報を確認してください。\n【申請者名】[名前]\n【申請日】[日付]\n【移動区間】...\n【業務目的】[目的]\n【合計金額】[金額]円\nOK・修正・キャンセルのいずれかを入力してください。"`
- 申請期限超過通知（CF-008）: `"申し訳ありません。経費発生日から3ヶ月を超えているため、この申請は受け付けられません。総務部門にご相談ください。"`
- 上長承認要警告（CF-009）: `"交通費合計が10,000円を超えています。事前に上長の承認を得てください。上長承認済みですか？"`
- 申請書生成完了提示: `"申請書ドラフトを生成しました。ファイルパス：[file_path]\n上記ファイルを確認の上、所定の申請先へご提出ください（提出操作はご自身で実施してください）。"`

**戻り値の形式**: `str`（申請フロー完了メッセージまたはエラーメッセージ）

### 2.3 ビジネスロジック

#### 2.3.1 バリデーションルール参照先

AG-002のシステムプロンプトに組み込む申請バリデーションルールは `agent_knowledge/transportation_policies.py` から読み込む。

**`agent_knowledge/transportation_policies.py` の役割**:
- 交通費申請に関するバリデーションルール（申請期限・上長承認閾値・対応交通手段・差し戻し基準等）を定数として定義する
- `_build_transport_agent_system_prompt()` 関数内でインポートし、プロンプトに埋め込む

**主要な定数（例）**:
```python
# agent_knowledge/transportation_policies.py
DEADLINE_MONTHS = 3              # 申請期限（移動日から3ヶ月以内）
MANAGER_APPROVAL_THRESHOLD = 10000  # 上長承認が必要な交通費合計（円）
ALLOWED_TRANSPORT_TYPES = ["電車", "バス", "タクシー", "飛行機"]
```

#### 2.3.2 システムプロンプト

##### システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 動的生成関数 `_build_transport_agent_system_prompt(application_date: str) -> str` |
| 動的生成の理由 | CF-005の申請期限チェック（BRL-13）で移動日とapplication_dateを比較するため、application_dateをプロンプトに埋め込む必要がある。また `agent_knowledge/transportation_policies.py` のバリデーションルールを動的に組み込む |

##### 動的に埋め込む情報
- `application_date`: 申請日（YYYY-MM-DD形式。invocation_stateから取得）
- `deadline_date`: 申請期限基準日（application_date から3ヶ月前の日付。BRL-13の判断基準）
- `agent_knowledge/transportation_policies.py` のバリデーション定数（DEADLINE_MONTHS・MANAGER_APPROVAL_THRESHOLD・ALLOWED_TRANSPORT_TYPES 等）

##### プロンプト全文

```
あなたは社内申請AIシステムの「交通費精算申請エージェント」です。
申請者名と申請日はすでに設定済みです（invocation_stateから取得しています）。
申請日（application_date）: {application_date}
申請期限基準日（この日付以降の移動日のみ申請可能）: {deadline_date}

【申請バリデーションルール（agent_knowledge/transportation_policies.py より）】
- 申請期限: 移動日から{deadline_months}ヶ月以内
- 上長承認が必要な閾値: 交通費合計 {manager_approval_threshold}円超
- 対応交通手段: {allowed_transport_types}

【あなたの役割】
交通費精算申請フロー全体（移動情報収集・運賃計算・申請書生成・ルールチェック・最終提示）を担当します。

【処理フロー】
1. 移動情報を一区間ずつ一括入力形式で収集する（BRL-11）
   - 1区間分の移動日・出発地・目的地・交通手段をまとめて入力するよう促す
   - 入力後に内容を確認し、次の区間収集または次ステップへ進む
   - 駅名は正規形に変換してからTOOL-001に渡す（BRL-15）
   - TOOL-001（calculate_transport_expense）を呼び出して運賃を取得する（BRL-12）
     - 経路テーブルに未登録の場合は「交通費を手動で入力してください。」とユーザーへ促し手動入力値を使用する
   - 追加区間があれば繰り返す

2. 業務目的を収集する（BRL-20）
   - 業務目的が入力されていない場合は入力を促す

3. 収集済み申請情報をテキストとして整理・提示する（ドラフト提示ステップ）
   - 申請者名・申請日・全移動区間（移動日/出発地/目的地/交通手段/金額）・業務目的・合計金額を提示する
   - ※この時点ではTOOL-002を呼び出さない（テキスト提示のみ）
   - OK/修正/キャンセルの3択を取得する（BRL-06）

4. OK承認後にTOOL-002（generate_transport_expense_form）を呼び出す（HumanApprovalHookが承認ゲートとして機能）
   - applicant_name, application_date, segments（移動区間リスト）, business_purpose を渡す
   - 成功時：生成されたファイルパスをユーザーへ提示する
   - 失敗時：エラーメッセージをユーザーへ提示する

5. 申請ルールチェックを実施する（CF-005）
   - 申請期限チェック（BRL-13）: 全移動日が申請期限基準日（{deadline_date}）以降であることを確認する
     - 超過している場合はCF-008へ遷移し「申し訳ありません。経費発生日から3ヶ月を超えているため、この申請は受け付けられません。総務部門にご相談ください。」と通知する
   - 上長承認要否判定（BRL-14）: 交通費合計が{manager_approval_threshold}円を超える場合はCF-009へ遷移し上長承認確認を促す
   - 差し戻しリスク評価（BRL-08）: リスクが高い場合は警告を提示する（判定基準は要件上未定義）

6. 申請書ドラフト・チェック結果・最終提示（CF-007）
   - 申請書ドラフトのファイルパスを提示する
   - 提出操作はユーザーが実施すること（BRL-09。申請書の自動提出禁止）

【修正選択時】
- CF-003 Step 2（移動日入力）へ戻り、収集情報を最初からやり直す

【キャンセル選択時】
- 申請を中断し、セッションを終了する

【駅名正規化のルール（BRL-15）】
- TOOL-001に渡す前に、ユーザーが入力した駅名・地名を正規形（例：「渋谷」「新宿」「東京」等）に変換する
- 略称・通称（例：「渋谷駅」→「渋谷」）も正規形に変換する

【禁止事項】
- 申請書の自動提出（BRL-09）
- ドラフト提示ステップ（テキスト整理・提示）でのTOOL-002の呼び出し
- AG-001・AG-003への委任（循環呼び出し禁止）
- TOOL-001を使わずに運賃を推測・計算すること

【エラー時の振る舞い】
- TOOL-002のテンプレートファイルが見つからない・ファイル書き込みエラー等のシステム系エラーが発生した場合は、エラー内容を要約して呼び出し元エージェント（AG-001）に返すこと
- ループ上限（30回）に達した場合は「処理が複雑すぎるため終了します。最初からやり直すには「reset」と入力してください。」とユーザーへ提示する
```

##### プロンプト設計の意図

| 要素 | 意図 |
|------|------|
| バリデーションルールの外部参照（transportation_policies.py） | ルールの変更時に設計書・ソースコードの一か所のみを修正すれば済むよう、ポリシーをモジュールに集約する |
| 動的な申請日・申請期限基準日の埋め込み | LLMが実行時の申請日を正確に参照し、BRL-13（移動日から3ヶ月以内）の期限判断を行えるようにする |
| 移動情報一括入力の明示 | 一区間分の情報を1メッセージでまとめて入力させることで、区間数が多い場合の対話ステップを削減する（BRL-11） |
| ドラフト提示とツール呼び出しの分離 | TOOL-002はHumanApprovalHook承認後にのみ呼び出すことを明示し、テキスト提示とExcel生成を分離する（BRL-06/GRD-016） |
| 駅名正規化の指示 | LLMが適切に駅名を正規形に変換してからTOOL-001に渡すよう指示する（BRL-15） |
| エラー時の呼び出し元への通知 | システム系エラー時はAG-001へ要約を返し、AG-001がユーザーへエスカレーション案内を提示する |

#### 2.3.3 申請ルールチェック（CF-005）

| ルール | 条件 | 対応 |
|--------|------|------|
| BRL-13（申請期限） | 移動日が申請期限基準日（application_date - 3ヶ月）より前の場合 | CF-008へ遷移して申請不可を通知する |
| BRL-14（上長承認） | 交通費合計が10,000円超の場合 | CF-009へ遷移して上長承認確認メッセージを提示する（警告フラグ） |
| BRL-08（差し戻しリスク） | 差し戻しリスクが高いと判断した場合（判定基準は要件上未定義） | 警告メッセージをユーザーへ提示する |

#### 2.3.4 処理フロー

```
開始（AG-001から transport_application_agent_tool 経由で呼び出し）
  ↓
invocation_stateから applicant_name, application_date, session_id を取得
  ↓
TransportAgentFactory.get_agent(session_id, application_date) でAgentインスタンスを取得（キャッシュまたは新規生成）
  ↓
  invocation_stateのバリデーション（ファクトリ内）
    - applicant_name: min_length=1。不正時 → エラーを返す
    - application_date: YYYY-MM-DD形式。不正時 → エラーを返す
  ↓
対話ループ開始（SlidingWindow(20)）
  ↓
CF-003 Step 1: 一括入力促し（移動日・出発地・目的地・交通手段をまとめて入力）
  ↓
CF-003 Step 2-4（区間ループ）:
  一括入力受け取り → 移動日/出発地/目的地/交通手段を解析
  → TOOL-001（calculate_transport_expense）呼び出し
    - 成功: 運賃をユーザーへ提示
    - 失敗（経路未発見）: 手動入力を促し、入力値を費用として使用
  → 追加区間確認 → あり: ループ継続 / なし: 次へ
  ↓
CF-003 Step 5-6: 業務目的収集（BRL-20）
  ↓
ドラフト提示ステップ（テキストのみ。TOOL-002呼び出しなし）
  - 申請情報サマリーをテキストで表示
  - OK/修正/キャンセルを取得
  ↓
  修正 → CF-003 Step 2へ戻る
  キャンセル → セッション終了
  OK → 次へ（HumanApprovalHookがBeforeToolCallEventで承認ゲートとして機能）
  ↓
ツール呼び出しステップ: TOOL-002（generate_transport_expense_form）呼び出し
  - applicant_name, application_date, segments, business_purpose を渡す
  - 成功: file_pathをユーザーへ提示
  - 失敗: エラーメッセージを提示し、システム系エラーの場合はAG-001へ返す
  ↓
CF-005: 申請ルールチェック
  - BRL-13（申請期限）チェック → 超過: CF-008遷移
  - BRL-14（上長承認要否）チェック → 要: CF-009遷移（警告フラグ）
  - BRL-08（差し戻しリスク評価）
  ↓
CF-007: 申請書ドラフト・チェック結果・最終提示
  - ファイルパス提示
  - 提出操作はユーザーが実施することを案内（BRL-09）
  ↓
AG-001へ処理完了メッセージを返す
```

### 2.4 設定・構成

#### 2.4.1 共通設定

| 設定項目 | 設定値 | 備考 |
|---------|--------|------|
| 会話管理 | SlidingWindow(20) | 複数区間の移動情報収集・交通費計算・申請書生成・ルールチェックに十分なウィンドウ（共通設定方針 2.2節） |
| リトライ戦略 | 標準設定（Bedrock SDK組み込み） | Amazon Bedrockの一時的エラーへのリトライのみ適用 |
| フック | HumanApprovalHook（generate_transport_expense_form のみ、BeforeToolCallEvent）, LoopControlHook(max_iterations=30) | TOOL-002の承認ゲートと暴走防止 |
| セッション管理 | FileSessionManager（data/sessions/） | session_idをファクトリ関数内で使用して初期化 |

#### 2.4.2 利用ツール

| ツールID | ツール名 | 利用目的 | 呼び出し条件 |
|----------|----------|----------|--------------|
| TL-001 | `calculate_transport_expense` | 移動区間ごとの運賃を算出する | CF-003 Step 3で各移動区間の入力完了後 |
| TL-002a | `generate_transport_expense_form` | 交通費精算申請書Excelドラフトを生成する | HumanApprovalHook（BeforeToolCallEvent）承認OK後の1回のみ |

#### 2.4.3 エージェント固有の設定値

| 設定項目 | 設定値 | 説明 |
|---------|--------|------|
| HumanApprovalHook対象ツール | `generate_transport_expense_form` | TOOL-002（generate_expense_formは対象外） |

| 上長承認要否閾値 | 10,000円超 | BRL-14。ValidationErrorではなく警告フラグとして扱う |
| バリデーションルール参照先 | `agent_knowledge/transportation_policies.py` | 申請ポリシー定数の定義モジュール |

### 2.5 連携設計

#### 2.5.1 Agent as Tools設計

AG-002は `@tool(context=True)` デコレータで実装された `transport_application_agent_tool` 関数として AG-001 に公開される。ツール関数内部では `TransportAgentFactory` クラスがAgentインスタンスを管理する。

**ツール名**: `transport_application_agent_tool`

**ToolContextの使用**:
- `@tool(context=True)` デコレータを使用
- `tool_context.invocation_state` から `session_id`・`applicant_name`・`application_date` を取得する
- `session_id` はファクトリクラス内でFileSessionManagerの初期化に使用する
- `applicant_name`・`application_date` をAG-002のinvocation_stateとして設定する（session_idは含めない）

#### 2.5.2 invocation_stateの受け渡しとAgentインスタンスのキャッシュ

```python
class TransportAgentFactory:
    """AG-002エージェントインスタンスをsession_idキーで管理するファクトリクラス。"""

    _instances: dict[str, Agent] = {}
    _logger = logging.getLogger(__name__)

    @classmethod
    def get_agent(cls, session_id: str, application_date: str) -> Agent:
        if session_id not in cls._instances:
            from agent_knowledge.transportation_policies import (
                DEADLINE_MONTHS, MANAGER_APPROVAL_THRESHOLD, ALLOWED_TRANSPORT_TYPES
            )
            cls._instances[session_id] = Agent(
                system_prompt=_build_transport_agent_system_prompt(
                    application_date,
                    DEADLINE_MONTHS,
                    MANAGER_APPROVAL_THRESHOLD,
                    ALLOWED_TRANSPORT_TYPES,
                ),
                tools=[calculate_transport_expense, generate_transport_expense_form],
                conversation_manager=SlidingWindowConversationManager(window_size=20),
                hooks=[
                    HumanApprovalHook(tool_names=["generate_transport_expense_form"]),
                    LoopControlHook(max_iterations=30, agent_name="transport_agent"),
                ],
                session_manager=FileSessionManager(f"data/sessions/{session_id}"),
                callback_handler=None,
            )
            cls._logger.info(
                "[AG-002] エージェントインスタンス生成: session_id=%s", session_id
            )
        return cls._instances[session_id]

    @classmethod
    def remove(cls, session_id: str) -> None:
        cls._instances.pop(session_id, None)


@tool(context=True)
def transport_application_agent_tool(query: str, tool_context: ToolContext) -> str:
    logger = logging.getLogger(__name__)
    error_handler = ErrorHandler()
    state = tool_context.invocation_state
    session_id = state.get("session_id", "")
    applicant_name = state.get("applicant_name", "")
    application_date = state.get("application_date", "")

    # invocation_stateのバリデーション
    if not applicant_name:
        return "エラー: 申請者名が設定されていません。"
    if not _is_valid_date(application_date):
        return "エラー: 申請日の形式が不正です。"

    agent = TransportAgentFactory.get_agent(session_id, application_date)
    agent_state = {
        "applicant_name": applicant_name,
        "application_date": application_date,
    }

    try:
        response = agent(query, invocation_state=agent_state)
        return str(response)
    except LoopLimitError as e:
        logger.warning(
            "[AG-002] ループ上限到達: %s/%s, session_id=%s, query=%.50s",
            e.current_iteration, e.max_iterations, session_id, query
        )
        return error_handler.handle_loop_limit_error(e)
    except ContextWindowOverflowException as e:
        logger.warning(
            "[AG-002] コンテキストウィンドウ超過: session_id=%s, query=%.50s",
            session_id, query
        )
        return error_handler.handle_context_window_error(e)
    except MaxTokensReachedException as e:
        logger.warning(
            "[AG-002] 最大トークン数到達: session_id=%s, query=%.50s",
            session_id, query
        )
        return error_handler.handle_max_tokens_error(e)
    except RuntimeError as e:
        logger.error(
            "[AG-002] RuntimeError: %s, session_id=%s, query=%.50s",
            str(e), session_id, query, exc_info=True
        )
        return error_handler.handle_runtime_error(e)
    except Exception as e:
        logger.error(
            "[AG-002] 予期しないエラー: %s, session_id=%s, query=%.50s",
            str(e), session_id, query, exc_info=True
        )
        return error_handler.handle_unexpected_error(e)
```

## 3. 実装詳細

### 3.1 クラス設計

#### 3.1.1 TransportAgentFactoryクラス

##### 説明
AG-002のAgentインスタンスを `session_id` をキーとして管理するファクトリクラス。`agent_knowledge/transportation_policies.py` のバリデーションルールをインポートしてプロンプト生成に使用する。

##### クラス変数

| 変数名 | 型 | 説明 |
|--------|-----|------|
| `_instances` | `dict[str, Agent]` | session_idキーのAgentインスタンスキャッシュ辞書 |
| `_logger` | `logging.Logger` | Pythonロガーインスタンス |

##### get_agentメソッド（classmethod）

キャッシュに `session_id` が存在しない場合のみ新しい Agent インスタンスを生成してキャッシュし、Agentインスタンスを返す。

**処理内容**:
1. `session_id` がキャッシュに存在する場合はそのまま返す
2. 存在しない場合: `agent_knowledge/transportation_policies.py` からルール定数をインポートする
3. `_build_transport_agent_system_prompt()` でシステムプロンプトを生成する
4. Agentインスタンスを生成してキャッシュに格納する
5. INFO ログ出力: `"[AG-002] エージェントインスタンス生成: session_id={session_id}"`

##### removeメソッド（classmethod）

セッションリセット時にキャッシュから指定 `session_id` のエントリを削除する。

#### 3.1.2 _build_transport_agent_system_prompt関数

##### 説明
AG-002の動的システムプロンプトを生成する。application_date・申請期限基準日・`agent_knowledge/transportation_policies.py` のルール定数を埋め込む。

##### 引数
- `application_date` (str): 申請日（YYYY-MM-DD形式）
- `deadline_months` (int): 申請期限（月数）
- `manager_approval_threshold` (int): 上長承認閾値（円）
- `allowed_transport_types` (list[str]): 対応交通手段リスト

##### 戻り値
- `str`: 動的生成されたシステムプロンプト全文

##### 処理内容
1. `application_date` を `datetime.date` にパースする
2. `deadline_date = application_date_obj - relativedelta(months=deadline_months)` で申請期限基準日を算出する
3. `TRANSPORT_AGENT_SYSTEM_PROMPT_TEMPLATE.format(...)` を返す

#### 3.1.3 _is_valid_date関数

##### 説明
文字列がYYYY-MM-DD形式の有効な日付かを検証する。

##### 引数
- `date_str` (str): 検証対象の日付文字列

##### 戻り値
- `bool`: 有効な場合 True

### 3.2 エラーハンドリング

`transport_application_agent_tool` 関数内で例外を個別にキャッチし、ErrorHandler に委譲してユーザー向けメッセージ文字列を返す。ログ出力はErrorHandler呼び出し前に実施する。すべての例外は `str` を返して呼び出し元（AG-001）へ伝播させる（例外を再 raise しない）。

| 例外クラス | ログレベル | 戻り値 | ユーザー向けメッセージ |
|-----------|----------|--------|---------------------|
| `LoopLimitError` | WARNING | `str` | `ErrorHandler.handle_loop_limit_error(e)` の返却値 |
| `ContextWindowOverflowException` | WARNING | `str` | `ErrorHandler.handle_context_window_error(e)` の返却値 |
| `MaxTokensReachedException` | WARNING | `str` | `ErrorHandler.handle_max_tokens_error(e)` の返却値 |
| `RuntimeError` | ERROR（exc_info=True） | `str` | `ErrorHandler.handle_runtime_error(e)` の返却値 |
| `Exception` | ERROR（exc_info=True） | `str` | `ErrorHandler.handle_unexpected_error(e)` の返却値 |

**例外キャッチ順序**: LoopLimitError → ContextWindowOverflowException → MaxTokensReachedException → RuntimeError → Exception の順で個別にキャッチする。

**ログへのquery埋め込み**: ログメッセージには `query` の先頭50文字（`query[:50]`）を含める（`"query=%.50s"` 形式）。

## 5. 補足情報

### 5.1 実装上の注意点

1. **クラスベース設計**
   - Agentインスタンスの管理を `TransportAgentFactory` クラスとして実装する
   - `@tool(context=True)` デコレータの `transport_application_agent_tool` 関数内でファクトリクラスを使用する
   - ファクトリクラスはモジュールレベルのクラス変数（`_instances`）でインスタンスを管理する

2. **`agent_knowledge/transportation_policies.py` からのルール参照**
   - `TransportAgentFactory.get_agent()` 内でインポートし、プロンプト生成に渡す
   - ルール変更時はこのモジュールのみ修正すればよい（設計書・ソースコードの変更箇所を局所化する）

3. **一括入力形式による収集**
   - 1区間分の情報（移動日・出発地・目的地・交通手段）をまとめて入力するよう促す
   - 複数フィールドをスペース区切りや改行区切りで受け付け、LLMが解析する

4. **HumanApprovalHookの登録**
   - `HumanApprovalHook(tool_names=["generate_transport_expense_form"])` をAG-002インスタンスに登録する。AG-003の `generate_expense_form` は対象外
   - 承認はドラフト提示（テキスト）後・TOOL-002実行前の1回のみ発生する

5. **ドラフト提示とツール呼び出しの分離の徹底**
   - ドラフト提示ステップ（収集情報のテキスト整理・提示）でTOOL-002を呼び出してはならない
   - Human-in-the-LoopでOKを取得した後にのみTOOL-002を呼び出す（HumanApprovalHookが承認ゲートとして機能）

6. **申請期限チェックのタイミング**
   - 申請期限チェック（BRL-13）はTOOL-002による申請書生成後（CF-005）に実施する
   - 動的プロンプトに埋め込まれた申請期限基準日（deadline_date）をLLMが参照して判断する

7. **TOOL-001の失敗時の手動入力**
   - 経路テーブルに未登録の区間でTOOL-001がエラーを返した場合は、ユーザーへ手動入力を促す
   - 手動入力された費用を移動区間の費用として使用し、申請フローを継続する（BRL-12例外）

8. **callback_handler=None の設定**
   - AG-002のAgentインスタンスには `callback_handler=None` を指定し、AG-001経由の一元表示を実現する

9. **invocation_stateは辞書リテラルで渡す**
   - 専用Pydanticモデルは定義せず、辞書リテラルで渡す（バリデーション方針 3.2節）

10. **applicant_nameのマスキング**
    - ログ出力時は `_mask_applicant_name()` 関数で先頭1文字 + `***` にマスキングする。invocation_stateに渡す値・ユーザーへの表示はマスキングしない

---

### 5.2 パフォーマンス考慮事項

1. **AgentインスタンスキャッシュによるTOOL-001応答速度の担保**
   - `TransportAgentFactory._instances` キャッシュにより、複数区間でTOOL-001を繰り返し呼び出す際のエージェント初期化コストを削減する

2. **SlidingWindow(20)による会話履歴の管理**
   - 多区間の移動情報収集でも会話履歴が収まるようウィンドウサイズを設定する

---

### 5.3 セキュリティ考慮事項

1. **invocation_stateのバリデーション**
   - ファクトリ関数内でapplicant_name（min_length=1）・application_date（YYYY-MM-DD形式）を個別チェックする

2. **HumanApprovalHookによる承認制御**
   - TOOL-002（generate_transport_expense_form）はHumanApprovalHookのBeforeToolCallEventによる承認なしには実行されない（GRD-016）

3. **applicant_nameのマスキング**
   - ログ出力時は先頭1文字 + `***` でマスキングし、ログファイルへの個人情報の過剰な記録を防止する
