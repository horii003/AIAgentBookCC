## 1. 目的・適用範囲

### 1.1 目的

システム全体として **処理の開始・継続・中断・停止・再開** を一貫したルールで制御するための方針を定義する

### 1.2 適用範囲

- マルチエージェント処理（AG-001, AG-002, AG-003）
- ツール実行（TOOL-001, TOOL-002）
- 人間承認（Human-in-the-Loop）を含む業務処理
- ファイルシステムI/O（セッション・申請書・ログ）

### 1.3 本方針で定義しないこと

- 個々のエージェントの内部制御ロジック（基本設計書に委ねる）
- SDK固有のキュー／スレッド制御
- 実装コードレベルの具体的な設定値（共通設定方針で定義）

---

## 2. 実行制御の基本原則

- **制御はシステムが行う（エージェント任せにしない）**：LoopControlHook・HumanApprovalHookによりフレームワークレベルで制御する
- **処理は必ず状態を持つ**：Strands Agents SDKセッションで実行状態を管理する
- **再実行可能性を前提にする**：TOOL-001は冪等。TOOL-002は利用者確認後にのみ呼び出す
- **制御不能な自律動作をさせない**：最大ループ回数・対話ターン上限・申請書自動提出禁止で制御する
- **人が介入できる余地を残す**：HumanApprovalHookで申請書生成前に必ず利用者確認を行う

## 3. 実行単位の定義

### 3.1 実行単位の種類

| 単位 | 説明 | 識別子 |
|-----|------|-------|
| セッション | 1件の申請フロー全体（AG-001起動〜最終提示完了または異常終了） | session_id |
| エージェント呼び出し | AG-001の1回の推論サイクル、またはAG-002/AG-003への1回の委任（Agent as Toolsの1呼び出し） | invocation_id |
| ループ | ReActループの1イテレーション（思考→ツール選択→ツール実行→観察） | iteration |
| ツール実行 | TOOL-001またはTOOL-002の1回の実行 | tool_name + execution_id |


### 3.2 実行単位の識別子

**session_id**:
- 形式: UUID v4（文字列）
- 例: `"a1b2c3d4-e5f6-7890-abcd-ef1234567890"`
- 生成: セッション開始時にStrands Agents SDKが自動生成

## 4. 実行状態管理

### 4.1 実行状態定義

| 状態 | 説明 | 適用対象 |
|-----|------|---------|
| READY | 実行待ち | セッション（CLIからの入力を待機中） |
| RUNNING | 実行中 | エージェント呼び出し・ループ・ツール実行 |
| WAITING | 外部／人待ち | Human-in-the-Loop承認待ち、追加情報入力待ち |
| COMPLETED | 正常終了 | セッション（最終提示完了後） |
| FAILED | 失敗 | ツール実行（エラーで戻り値を返した場合） |
| TERMINATED | 強制終了 | セッション（ループ上限超過・エラー・キャンセル） |


### 4.2 状態遷移の責務

**状態管理主体**: Strands Agents SDK（セッション・エージェント呼び出し）、LoopControlHook（ループ）、HumanApprovalHook（承認待ち）

**エージェントの役割**: 状態の直接操作は行わない。ツール関数の戻り値・フックの制御を通じてSDKに状態変更を委譲する

---

## 5. 実行開始制御

### 5.1 実行開始条件

**前提条件充足**:
- CLIからのユーザー入力受信
- セッション情報（data/sessions/）の読み込み成功（または新規セッション生成）
- 環境変数（AWS認証情報・モデルID・ガードレールID等）のロード成功

**権限確認完了**:
- エージェントロール（R-AG）の権限確認（自律度・権限定義参照）
- 申請書生成（TOOL-002）はHuman-in-the-Loop承認（OK）取得後にのみ実行可能

**ガードレールチェック通過**:
- 入力文字数が500文字以下であること（GRD-017）
- 対話ターン数が30回以下であること（GRD-018）


### 5.2 実行抑止条件

**同一セッションでの重複実行**:
- Strands Agents SDKのセッション管理機能が同一セッションへの二重アクセスを防止する

**排他対象リソース使用中**:
- TOOL-002（申請書生成）：利用者のHuman-in-the-Loop承認前は実行不可（GRD-016）
- 申請書提出操作：AIエージェントによる実行は全面禁止（BRL-09）

**システム負荷超過**:
- 要件上未定義（Amazon Bedrockのレート制限はStrands Agents SDKのリトライ機構で対応）

## 6. 並列・逐次制御方針

### 6.1 並列実行可否の判断軸

**データ競合有無**:
- AG-002とAG-003は同一セッション内で同時に実行しない（同一の申請書ドラフト生成対象が競合するため）

**外部I/F制約**:
- Amazon Bedrock（LLM）への同時呼び出しは1エージェントにつき1リクエスト（Strands Agents SDKの制約に従う）

**業務整合性**:
- 1件の申請フローで交通費精算申請と経費精算申請が同時に発生しないため、並列実行は不要

### 6.2 排他制御対象

| 対象 | 排他理由 | 実装方法 |
|-----|---------|---------|
| TOOL-002（申請書生成） | HumanApprovalHookによる承認前の誤実行防止 | HumanApprovalHookがツール実行前にブロック |
| AG-002とAG-003の同時実行 | 同一セッションへの競合書き込み防止 | Agent as Toolsの逐次呼び出し構造で自然に排他 |

## 7. 再試行（Retry）方針

### 7.1 再試行対象

| 対象 | 再試行可否 | 最大回数 | 理由 |
|-----|----------|---------|------|
| Amazon Bedrock（LLM）呼び出し | ○ | 6回（指数バックオフ：初期4秒・最大240秒） | Strands Agents SDKの組み込みリトライ機構を使用 |
| TOOL-001（交通費計算） | × | — | ファイルI/Oエラーは代替処理（手動入力）で対応 |
| TOOL-002（申請書生成） | × | — | 申請書生成失敗はエラーメッセージ返却のみ。重複生成リスクあり |
| ファイルI/O（セッション・ログ） | × | — | ファイルシステム障害はシステム障害（EX-07）として中断 |

### 7.2 再試行制約

**最大回数**: 6回（Amazon Bedrock呼び出しのみ。Strands Agents SDKの設定で制御）

**クールダウン**: 指数バックオフ（初期4秒、最大240秒）

**冪等性前提**: TOOL-001は同一入力で同一結果を返す（冪等）。TOOL-002は再実行で申請書ドラフトを上書き生成する（冪等ではないためHuman-in-the-Loop承認後にのみ呼び出す）

---

## 8. 中断・停止制御

### 8.1 中断（Pause）

**中断可能条件**:
- Human-in-the-Loop承認待ち（WAITING状態）：HumanApprovalHookが承認待ちでブロック

**中断時の状態保持**:
- Strands Agents SDKのセッション管理機能がファイルストレージ（data/sessions/）にセッション情報を保持する
- 収集済み申請情報・現在の会話状態が保存され、再開時に復元される

### 8.2 停止（Terminate）

**停止条件**:
- LoopControlHookによるループ上限（10回）到達（自動停止）
- FR-017 対話ターン数30回超過（GRD-018）
- EX-03 申請期限超過（BRL-13/BRL-18）：業務ルール違反で中断
- EX-04 LLMリトライ全失敗（6回）
- EX-07 ファイルシステム障害
- EX-08 想定外例外

**停止後の復旧可否**:
- ループ上限到達: 新規セッションで再開可能
- 対話ターン数超過: 新規セッションで再開可能
- 申請期限超過: 申請自体が不可のため再開不可
- LLMリトライ全失敗: インフラ復旧後に新規セッションで再開可能
- システム障害: インフラ復旧後に新規セッションで再開可能

## 9. タイムアウト制御

### 9.1 タイムアウト対象

| 対象 | タイムアウト時間 | 実装状況 |
|-----|---------------|---------|
| Amazon Bedrock（LLM）呼び出し | 要件上未定義 | Strands Agents SDK のデフォルト設定を使用 |
| TOOL-001（交通費計算） | 要件上未定義 | 未設定（ファイルI/Oのデフォルトタイムアウト） |
| TOOL-002（申請書生成） | 要件上未定義 | 未設定（ファイルI/Oのデフォルトタイムアウト） |
| Human-in-the-Loop承認待ち | 要件上未定義（タイムアウトなし） | HumanApprovalHookは明示的な応答があるまで待機 |

### 9.2 タイムアウト時の扱い

**例外分類**: EX-04（外部I/F例外）

**再試行 or エスカレーション**: Strands Agents SDKのリトライ機構（最大6回）が再試行。全失敗後はEX-04としてエスカレーション案内→セッション終了

## 10. ループ制御

### 10.1 ループ制御の目的

- ReActループの無限ループ・暴走を防止する
- LLMが同一アクションを繰り返すことを防止する
- システムコスト・レスポンス時間を一定範囲内に抑制する

### 10.2 ループ制御の仕組み

**最大ループ回数**: 10回（デフォルト）

**実装クラス**: `LoopControlHook`（HookProvider として実装）

**制御方式**: LoopControlHookがStrands Agents SDKのコールバックフックとして実装される。全エージェント（AG-001, AG-002, AG-003）に適用する

**カウント対象**: 1ReActイテレーション（思考→ツール選択→ツール実行→観察）を1カウントとする

**登録イベント**:

| イベント | 発火タイミング | 処理内容 |
|---------|-------------|---------|
| BeforeInvocationEvent | エージェント呼び出し開始時 | ループカウンタを0にリセットする |
| AfterModelCallEvent | LLM呼び出し後 | ループカウンタをインクリメントし、上限（10回）到達を監視する |
| AfterInvocationEvent | エージェント呼び出し完了後 | ループカウンタを0にリセットする |

### 10.3 ループ制限到達時の処理

**処理**: LoopControlHookがループを強制停止し、エラーメッセージをユーザーに提示する

**ユーザーへの通知**: 「申し訳ありません。処理が複雑になりすぎたため、一旦終了します。もう一度最初から申請をやり直してください。」

**セッション状態**: TERMINATED に遷移

> ※ 具体的な実装はハンドラー基本設計.md を参照

## 11. 人間承認制御

### 11.1 承認制御の目的

- AIによる申請書の自動生成・自動提出を防止する（BRL-09）
- 利用者が申請書内容を確認・承認した上でのみExcel生成を許可する（GRD-016）
- 上長承認要否・差し戻しリスクを利用者が確認した上で提出操作に進む

### 11.2 承認制御の仕組み

**発火タイミング**: AG-002またはAG-003がTOOL-002（申請書生成）を呼び出す直前。HumanApprovalHookがツール実行をブロックし、利用者に確認を求める

**承認選択肢**:
1. OK（承認）：申請書内容に問題なし。TOOL-002の実行を許可する
2. 修正（修正要望）：申請書内容に修正が必要。修正対象項目の再収集ループへ戻す
3. キャンセル：申請を中止する。セッションをTERMINATEDに遷移する

### 11.3 承認結果の処理

| 選択肢 | 処理内容 |
|-------|---------|
| OK | HumanApprovalHookがブロックを解除しTOOL-002の実行を許可する。申請書ドラフトを生成してExcelファイルを保存する |
| 修正 | HumanApprovalHookがブロックを継続し、修正対象を収集するループへ戻す（ACTIVE維持） |
| キャンセル | セッションをTERMINATEDに遷移し「申請をキャンセルしました」を提示する |

> ※ 具体的な実装はハンドラー基本設計.md を参照

## 12. エスカレーション連携

**実行不能状態の扱い**:
- EX-04〜EX-08（エスカレーション対象例外）発生時は、ErrorHandlerがユーザー向けエスカレーションメッセージを生成しセッションを終了する

**実行中断 → エスカレーションの流れ**:
1. ErrorHandlerが例外を検知し継続不能と判定する
2. ユーザーへのエスカレーションメッセージを生成する（「システム管理者にご連絡ください」等）
3. ログにエラー詳細（スタックトレース含む）を記録する
4. セッション状態をTERMINATEDに遷移させてセッションを終了する

**実行再開条件**:
- エスカレーション先（システム管理者）による障害対応後、新規セッションで再試行する

---

## 13. セッション管理との連携

**実行状態とセッション状態の関係**:
- 実行状態（RUNNING/WAITING/COMPLETED/TERMINATED）はStrands Agents SDKのセッション状態に対応する
- 実行状態の変更はSDKが自動で行うため、アプリケーションコードで直接操作しない

**再開時の制御ポイント**:
- セッション情報（data/sessions/）が存在する場合、同一セッションIDでセッションを復元する
- 復元後は前回の会話状態・収集済み申請情報・承認状態を引き継いで処理を継続する

---

## 14. ガードレール連携

**実行前チェック**:
- 入力文字数チェック（GRD-017）：500文字超の場合は入力拒否メッセージを提示（EX-01）
- 対話ターン数チェック（GRD-018）：30回超の場合はセッション終了

**実行中監視**:
- LoopControlHookによるReActループ上限監視（10回）
- HumanApprovalHookによるTOOL-002実行前の承認ゲート（GRD-016）
- Amazon Bedrockガードレールによるコンテンツポリシー制御（設定値は共通設定方針参照）

**実行後検証**:
- ツール実行結果（`{"success": True/False, ...}`）を確認し、失敗の場合はErrorHandlerに委譲する
- 申請書ドラフト生成後の必須項目チェック（ツール内のPydanticバリデーション）

## 15. ログ・監査方針

### 15.1 実行制御ログ

| 項目 | 内容 | 出力先 |
|-----|------|-------|
| event_type | イベントタイプ（loop_start/loop_end/tool_call/approval_request/session_start/session_end等） | logs/error.log（ErrorHandlerのログメソッド経由） |
| agent_name | エージェント名（AG-001/AG-002/AG-003） | logs/error.log |
| iteration | ループ回数（LoopControlHookが記録） | logs/error.log |
| tool_name | ツール名（TOOL-001/TOOL-002） | logs/error.log |
| timestamp | タイムスタンプ | logs/error.log |

### 15.2 監査観点

**実行判断の説明可能性**:
- ループ回数・ツール実行回数・承認操作をログに記録し、実行制御の判断経路を後から説明できる状態を維持する

**人間介入の記録**:
- HumanApprovalHookによる承認操作（OK/修正/キャンセル）の結果と実行時刻をDATA-009（承認ログ）として記録する

## 16. 設計上の留意点・リスク

**過剰制御による性能劣化**:
- LoopControlHookによる頻繁な割り込みでReActループのスループットが低下するリスク
- 対策: LoopControlHookはループ回数チェックのみを行い、軽量な実装を維持する

**制御不足による暴走**:
- LoopControlHookが正常に動作しない場合、ReActループが無限に継続するリスク
- 対策: Strands Agents SDKの組み込みループ制限機能をバックアップとして活用する

**再試行ループ**:
- Strands Agents SDKのリトライ機構とLoopControlHookが相互に干渉し、意図しない再試行が発生するリスク
- 対策: リトライ（Amazon Bedrock呼び出しのみ・最大6回）とループ制限（全エージェント・10回）の対象を明確に分離する
