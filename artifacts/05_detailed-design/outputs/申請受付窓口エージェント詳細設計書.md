## 1. 概要

### 1.1 コンポーネントの目的
社員の申請意図をテキストから受け取り、申請種別（交通費精算申請・経費精算申請・申請ルール照会）を LLM により分類し、適切な専門エージェント（AG-002/AG-003）に処理を委譲する。マルチエージェントシステムの入口として、セッション管理・申請者情報収集・ルーティングを一元的に担当する。

### 1.2 主要な責務
- 社員の自然言語入力から申請種別（JD-01/JD-02/JD-03）を判断し、担当エージェントを決定する（BRL-01/BRL-04）
- 申請者名を収集し、申請日（システム日付）と共に invocation_state に設定して AG-002/AG-003 へ委譲する（LNK-001/LNK-002）
- 申請ルール照会（UC-07）についてはナレッジ（data/knowledge/）を参照して直接回答する（BRL-01/BRL-14）
- AG-001 起動時に session_id を生成し、FileBasedSessionManager でセッションファイルを作成・管理する
- CLI 対話ループを制御し、特殊コマンド（reset/exit）・入力文字数制限（500文字）・対話回数上限（30回）を管理する

### 2.2 インターフェース設計

#### 2.2.1 入力

**初期化時の入力**:
- なし（AG-001 は CLI 起動時に特別な初期化パラメータを必要としない。session_id は AG-001 起動時に FileBasedSessionManager が自動生成する）

**対話ループでの入力**:
- ユーザー入力（自然言語での申請意図、または申請ルール照会）
- 入力制限: 500文字以内（FR-012/GRD-017）。超過時は EX-01 として再入力を促す
- 特殊コマンド:

| コマンド | 動作 |
|---------|------|
| `reset` / `リセット` / `最初から` | 会話履歴と収集済み申請者情報をリセットし、新規セッションを開始する |
| `exit` / `quit` / `終了` | CLI を終了する |

**パラメータ（対話ループ）**:
- `user_input` (str): ユーザーからの自然言語入力（最大500文字）

**ToolContext経由で受け取る情報**:
- なし（AG-001 はオーケストレーターであり invocation_state を受け取る側ではない）

#### 2.2.2 出力

**出力先**: 標準出力（CLI ユーザーへの表示）

**出力メッセージ**:
- 起動時ウェルカムメッセージ:
  ```
  ============================================================
  こちらは申請受付窓口エージェントです
  社内の様々な申請作業をサポートします

  最初に申請者名を入力してください。その後、申請したい内容をお知らせください。キーワードでも構いません

  ※終了するには 'exit' または 'quit' と入力ください
  ※最初からやり直すには 'reset' と入力ください
  ============================================================
  ```
- 入力プロンプト: `\n\n入力内容（終了時はquit）: `
- 申請種別確定時: `"「{申請種別}」として受け付けました。担当エージェントに引き継ぎます。"`
- 申請種別不明時（BRL-04）: `"申請種別を確認させてください。「交通費精算申請」または「経費精算申請」のどちらになりますか？"`
- 申請ルール照会回答時: `"ご照会の内容について、社内申請ルール（{ルールID}）に基づき回答いたします。{回答内容}"`
- 委譲完了後: `"申請書の作成が完了しました。{ファイルパス} に保存されています。"`
- 対話回数上限到達時: `"対話回数の上限に達しました。管理部門（経理部）にお問い合わせいただくか、改めて最初からお試しください。"`

**戻り値の形式**: なし（標準出力への表示のみ）

### 2.3 ビジネスロジック

#### 2.3.1 システムプロンプト

##### システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 静的定数 |
| 動的生成の理由 | 不要（AG-001 の責務には日付・期限に依存する業務ルール判断が含まれない） |

##### プロンプト全文

```
あなたは経費精算申請システムの申請受付窓口エージェントです。

【役割】
社員から申請意図を受け取り、以下の3種類の対応を行います。
1. 交通費精算申請 → 交通費精算申請エージェント（handle_transport_expense_application）に委譲
2. 経費精算申請 → 経費精算申請エージェント（handle_expense_application）に委譲
3. 申請ルール照会 → 社内申請ルール（data/knowledge/）を参照して直接回答

【申請者情報の収集】
処理を開始する前に、必ず申請者名を確認してください。
「申請者のお名前を教えていただけますか？」と確認してください。
申請者名が収集できたら、申請種別を判断して対応を進めてください。
申請日はシステム日付（本日の日付）を自動使用します。社員への確認は不要です。

【申請種別の判断基準（JD-01〜JD-03）】
以下の基準に従い申請種別を判断してください：
- 「電車で行った」「交通費を申請したい」「出張交通費」「交通費」「移動費」など
  移動・交通費に関する意図 → 交通費精算申請（handle_transport_expense_application を呼び出す）
- 「領収書がある」「備品を買った」「文房具」「事務用品」「宿泊費」「資格試験費」など
  購入・経費に関する意図 → 経費精算申請（handle_expense_application を呼び出す）
- 「申請ルールを教えて」「期限はいつ？」「上限はいくら？」「経費区分とは」など
  照会に関する意図 → 申請ルール照会（自身で社内申請ルールを参照して回答）
- 意図が不明または複数種別に該当する可能性がある場合 → BRL-04（選択肢提示）

【判断不能時の対応（BRL-04）】
申請種別が判断できない場合は、以下の2択を提示してください：
「申請種別を確認させてください。「交通費精算申請」または「経費精算申請」のどちらになりますか？」
同じ状況が3回繰り返された場合は、以下のように案内してください：
「申請種別の特定が難しい状況です。管理部門（経理部）にお問い合わせいただくことをお勧めします。」

【申請ルール照会の回答（BRL-01/BRL-14）】
- 社内申請ルール（data/knowledge/ に配置されたファイル）に記載された内容のみを根拠として回答してください
- 参照したルールIDまたはルール名を必ず回答に含めてください
  例：「社内申請ルール（BRL-11：申請期限）に基づき、申請は経費発生日から90日以内に行う必要があります。」
- 社内申請ルールに該当しない内容については推測・補完した回答を行ってはいけません
  その場合は「該当するルールが見つかりませんでした。管理部門（経理部）にお問い合わせください。」と案内してください

【委譲時の対応（LNK-001/LNK-002）】
申請種別が確定し申請者名が収集済みの場合、対応するツールを呼び出してください。
- 交通費精算申請 → handle_transport_expense_application ツールを呼び出す
  queryには申請者名・申請日・申請種別を含めた委譲メッセージを渡してください
  例：「交通費精算申請の手続きをお願いします。申請者：{申請者名}、申請日：{申請日}」
- 経費精算申請 → handle_expense_application ツールを呼び出す
  queryには申請者名・申請日・申請種別を含めた委譲メッセージを渡してください
  例：「経費精算申請の手続きをお願いします。申請者：{申請者名}、申請日：{申請日}」
委譲後は専門エージェントからの結果をそのまま社員に提示してください。

【特殊コマンド】
- 「reset」「リセット」「最初から」 → 会話をリセットして新しいセッションを開始します
- 「exit」「quit」「終了」 → システムを終了します

【入力制限】
500文字を超える入力は受け付けません。以下のように案内してください：
「入力が500文字を超えています。500文字以内に短縮してから再入力してください。」

【対話回数上限】
対話回数が30回に達した場合は、以下のメッセージを表示して終了してください：
「対話回数の上限に達しました。管理部門（経理部）にお問い合わせいただくか、改めて最初からお試しください。」

【制約事項】
- 社内申請ルール以外の情報で判断・回答してはいけません（BRL-01/BRL-14）
- 交通費精算申請（handle_transport_expense_application）と
  経費精算申請（handle_expense_application）を同時に呼び出してはいけません
- 申請書生成ツール（TOOL-002）を直接呼び出してはいけません
- 申請者名が収集できていない状態で専門エージェントに委譲してはいけません
```

##### プロンプト設計の意図

| 要素 | 意図 |
|------|------|
| 静的定数 | AG-001 の役割に動的な日付情報は不要なため、実行コストを削減する |
| 委譲先選択の明確化 | LLM の裁量で委譲先を変更させないよう、ツール名を明示する |
| 根拠提示の徹底 | 申請ルール照会の回答には参照ルールID・内容を必ず含めさせる（BRL-01/BRL-14） |
| 一括収集指示 | 申請者名と申請種別の確認は最初のターンで同時に収集する設計とする |

#### 2.3.2 申請種別ルール

| ルール | 条件 | 対応 |
|--------|------|------|
| BRL-01 | 申請種別判断・ルール照会の全場面 | 社内申請ルール（D-001）のみを参照して判断・回答する |
| BRL-04 | 申請意図から申請種別が特定できない場合 | 「交通費精算申請」「経費精算申請」の2択を提示して選択させる |
| BRL-14 | 社内申請ルールに該当しない照会内容の場合 | 推測・補完せず、管理部門への問い合わせを案内する |

#### 2.3.3 処理フロー

```
開始（CLI 起動）
  ↓
session_id 生成（FileBasedSessionManager）
session_{id}.json 作成（status=CREATED）
  ↓
AG-001 Agent インスタンス生成
  - モデル: MODEL_ID（us.anthropic.claude-sonnet-4-5-20251001）
  - システムプロンプト: AG001_SYSTEM_PROMPT（静的定数）
  - ツール: [handle_transport_expense_application, handle_expense_application]
  - SlidingWindowConversationManager(window_size=30)
  - LoopControlHook(max_iterations=30, agent_name="AG-001")
  ↓
ウェルカムメッセージを標準出力に表示（対話ループ開始前）
  ↓
対話ループ開始（session status=ACTIVE）
  ↓
ユーザー入力受付（プロンプト: `\n\n入力内容（終了時はquit）: `）
  ↓
[特殊コマンド？]
  - "reset"/"リセット"/"最初から" → 新規セッション生成・エージェント再初期化 → ループ先頭へ
  - "exit"/"quit"/"終了" → セッション status=CLOSED → 終了
  - その他 → 処理続行
  ↓
[入力文字数 > 500？]
  - YES → "入力が500文字を超えています。500文字以内に短縮してから再入力してください。" を表示 → ループ先頭へ
  - NO → 処理続行
  ↓
[対話回数 >= 30？]
  - YES → "対話回数の上限に達しました..." を表示 → セッション status=TERMINATED → 終了
  - NO → 処理続行
  ↓
Agent に入力を渡す（agent(user_input, invocation_state=state)）
  ↓
[LLM が申請者名未収集と判断？]
  - YES → 申請者名の確認メッセージを表示 → ループ先頭へ
  ↓
[LLM が申請種別を判断？]
  - 交通費精算申請 → handle_transport_expense_application ツール呼び出し
      invocation_state: {"session_id": str, "applicant_name": str, "application_date": str}
      → 結果をユーザーに提示 → ループ先頭へ
  - 経費精算申請 → handle_expense_application ツール呼び出し
      invocation_state: {"session_id": str, "applicant_name": str, "application_date": str}
      → 結果をユーザーに提示 → ループ先頭へ
  - 申請ルール照会 → ナレッジ参照・回答生成 → 結果をユーザーに提示 → ループ先頭へ
  - 判断不能（BRL-04） → 2択選択肢を提示 → ループ先頭へ（3回繰り返し後は CF-009）
  ↓
ループ先頭へ（次のユーザー入力待ち）
```

### 2.5 連携設計

#### 2.5.1 Agent as Tools設計

AG-001 はオーケストレーターとして AG-002/AG-003 を Agent as Tools パターンで呼び出す。  
AG-001 自身は上位エージェントから呼び出されるツールとして公開されない。

**利用するツール関数**:

| ツール関数名 | 定義場所 | 説明 |
|------------|---------|------|
| `handle_transport_expense_application` | `tools/transport_expense_agent.py` | 交通費精算申請フロー全体を AG-002 に委譲する |
| `handle_expense_application` | `tools/expense_agent.py` | 経費精算申請フロー全体を AG-003 に委譲する |

#### 2.5.2 invocation_stateの受け渡しとAgentインスタンスのキャッシュ

AG-001 から AG-002/AG-003 へ渡す invocation_state:

```python
# AG-001 が委譲時に設定する invocation_state
state = {
    "session_id": session_id,        # str: FileBasedSessionManager が生成したセッションID
    "applicant_name": applicant_name, # str: 収集済み申請者名（min_length=1）
    "application_date": str(date.today()),  # str: 申請日（YYYY-MM-DD形式）
}

# AG-001 から AG-002 への委譲例（LNK-001）
response = agent(
    "交通費精算申請の手続きをお願いします。",
    invocation_state=state
)
# handle_transport_expense_application ツールが LLM により呼び出される
```

> session_id はファクトリ関数内で消費し、子エージェントの LLM プロンプトには含めない。

#### 2.5.3 連携先への振り分けロジック

**振り分け判断基準（JD-01〜JD-03）**:

| 連携先 | キーワード・特徴 | 判断基準 |
|--------|----------------|---------|
| AG-002（交通費精算申請） | 「電車」「交通費」「出張」「移動費」「定期」「乗車」 | 移動・交通手段・運賃に関する意図（JD-01） |

## 3. 実装詳細

### 3.1 クラス/関数設計

#### 3.1.1 モジュール構成（agents/reception_agent.py）

**モジュールレベル定数**:
```python
AG001_SYSTEM_PROMPT: str  # 静的システムプロンプト（定数）
```

#### 3.1.2 run 関数

##### 説明
CLI のエントリポイント。session_id を生成し、Agent インスタンスを初期化して対話ループを実行する。

##### 引数
- なし

##### 戻り値
- なし

##### 処理内容
1. `FileBasedSessionManager` を初期化して session_id を生成する
2. AG-001 Agent インスタンスを生成する（`_create_ag001_agent(session_id)`）
3. 対話ループ（`_run_repl(agent, session_id, session_manager)`）を実行する
4. 終了時に session status を CLOSED に更新する

#### 3.1.3 _create_ag001_agent 関数

##### 説明
AG-001 の Agent インスタンスを生成するファクトリ関数。

##### 引数
- `session_id` (str): セッションID

##### 戻り値
- `Agent`: 初期化済みの AG-001 Agent インスタンス

##### 処理内容
1. `SlidingWindowConversationManager(window_size=30, should_truncate_results=True)` を生成する
2. `LoopControlHook(max_iterations=MAX_LOOP_ITERATIONS)` を生成する
3. `Agent(name="AG-001", model=MODEL_ID, instructions=AG001_SYSTEM_PROMPT, tools=[handle_transport_expense_application, handle_expense_application], conversation_manager=..., hooks=[LoopControlHook(max_iterations=MAX_LOOP_ITERATIONS, agent_name="AG-001")])` を生成して返す

#### 3.1.4 _run_repl 関数

##### 説明
対話ループを実行する。ユーザー入力を受け取り、バリデーション後に Agent に渡す。

##### 引数
- `agent` (Agent): AG-001 Agent インスタンス
- `session_id` (str): セッションID
- `session_manager` (FileBasedSessionManager): セッション管理インスタンス

##### 戻り値
- なし

##### 処理内容
1. 対話カウンタを 0 に初期化する
2. ウェルカムメッセージを標準出力に表示する（ループ開始前）
3. ループ開始（対話カウンタが30未満の間）:
   a. ユーザー入力を `input("\n\n入力内容（終了時はquit）: ")` で受け取る
   b. 特殊コマンド（reset/exit等）を判定して処理する
   c. 入力文字数が 500 を超える場合はエラーメッセージを表示してループ先頭へ
   d. invocation_state を組み立てる（session_id, applicant_name, application_date）
   e. `agent(user_input, invocation_state=state)` を呼び出す
   f. レスポンスを標準出力に表示する
   g. 対話カウンタをインクリメントする
3. 対話カウンタが30に達した場合、上限到達メッセージを表示して終了する

## 4. データ設計

### 4.1 invocation_state

**エージェントに渡すデータ（AG-002/AG-003 への委譲時）**:
```python
{
    "session_id": str,           # セッションID（形式: YYYYMMDDHHMMSS_{uuid4().hex[:8]}）
    "applicant_name": str,       # 申請者名（min_length=1）
    "application_date": str,     # 申請日（YYYY-MM-DD形式。date.today().isoformat()）
}
```

**session_id 生成方式**: `datetime.now().strftime("%Y%m%d%H%M%S") + "_" + uuid.uuid4().hex[:8]`

> `application_type` は invocation_state に含めない（委譲先エージェントの選択により申請種別が決定）

## 5. 補足情報

### 5.1 実装上の注意点

1. **applicant_name の保持タイミング**
   - LLM が申請者名を収集したら、session_{id}.json に保存する
   - invocation_state 組み立て時は session_{id}.json から読み出して設定する

2. **AG-002/AG-003 との分離**
   - AG-001 は handle_transport_expense_application / handle_expense_application を `tools` リストに含める
   - AG-002/AG-003 のインスタンスキャッシュ（_agent_instances）は各モジュール（tools/transport_expense_agent.py, tools/expense_agent.py）で管理する。AG-001 は関知しない

3. **特殊コマンドの処理**
   - reset コマンド受信時は新しい session_id を生成し、AG-001 Agent インスタンスを再生成する
   - SlidingWindowConversationManager の会話履歴は再生成により自動リセットされる

4. **ウェルカムメッセージの出力タイミング**
   - ウェルカムメッセージは対話ループ開始直前に1回だけ出力する
   - reset コマンド受信時（セッション再生成後）も再度出力する

5. **session_id 生成方式**
   - `datetime.now().strftime("%Y%m%d%H%M%S") + "_" + uuid.uuid4().hex[:8]` で生成する
   - `datetime` と `uuid` の両モジュールが必要

### 5.2 パフォーマンス考慮事項

1. **ナレッジ読み込みのオーバーヘッド**
   - data/knowledge/ のファイルはシステムプロンプト生成時に一度読み込む。対話ターンごとに再読み込みしない

### 5.3 セキュリティ考慮事項

1. **入力サニタイズ**
   - ユーザー入力は500文字上限でトリミングする（GRD-017）
   - システムプロンプトへのユーザー入力の直接埋め込みは行わない

### 7.1 機能テスト
- 「交通費を申請したい」と入力したとき handle_transport_expense_application が呼び出されること

### 7.2 異常系テスト
- 501文字の入力に対して EX-01 エラーメッセージが表示され再入力が促されること

- MaxTokensReachedException 発生時に WARNING ログが出力され、ループが continue されること
- RuntimeError 発生時に ERROR ログ（exc_info=True）が出力され、ループが continue されること
