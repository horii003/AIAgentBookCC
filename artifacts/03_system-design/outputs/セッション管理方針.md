## 1. 目的・設計方針

### 1.1 目的

- AIエージェントが **セッション単位で保持・参照すべき情報の範囲と方針**を定義する
- 会話継続、再開、引き継ぎ、監査に耐える **一貫した状態管理** を実現する

### 1.2 本方針で定義すること

- セッションの単位・ライフサイクル
- 管理対象情報（会話履歴、判断結果、成果物参照 等）
- メモリの種類と役割分担
- 再開・中断・終了時の扱い

### 1.3 本方針で定義しないこと

- メモリの具体的な保存技術（Strands Agents SDK の `FileSessionManager` を使用。実装詳細は基本設計書に委ねる）
- エージェント内部の推論用メモリ構造
- 実装クラス／SDK依存設計の詳細

## 2. セッションの定義

### 2.1 セッションとは

**本システムにおける「セッション」の定義**:
- 1名の利用者が1件の申請（交通費精算申請または経費精算申請）を行うための一連の対話フロー全体を指す
- AG-001が起動してから申請書ドラフトの最終提示・利用者の提出操作案内が完了するまで（または異常終了するまで）の期間を1セッションとする
- セッション状態管理は Strands Agents SDK の機能へ委譲し、独自のセッション管理クラスは作成しない

### 2.2 セッション識別子

| 項目 | 内容 | 形式 | 例 |
|-----|------|------|---|
| session_id | セッション識別子 | UUID v4（文字列） | `"a1b2c3d4-e5f6-7890-abcd-ef1234567890"` |

## 3. セッションライフサイクル

### 3.1 状態遷移

| 状態 | 説明 | トリガー |
|-----|------|---------|
| CREATED | セッション生成 | 利用者がCLIを起動し最初のメッセージを送信した時点 |
| ACTIVE | 処理中 | エージェントがユーザー入力を受け付け処理を開始した時点 |
| WAITING | 人間対応待ち | Human-in-the-Loop確認（申請書内容OK/修正/キャンセル選択待ち）または追加情報入力待ち |
| CLOSED | 正常終了 | 申請書ドラフト最終提示・提出案内が完了し、利用者が会話を終了した時点 |
| TERMINATED | 異常終了 | 対話回数上限（30回）超過・エラー発生・利用者が途中でキャンセルした時点 |

### 3.2 状態遷移条件

**状態遷移のトリガー**:
- CLIからの最初のユーザー入力受信: CREATED → ACTIVE
- Human-in-the-Loop承認依頼発生: ACTIVE → WAITING
- 利用者がOK/修正/キャンセルを選択: WAITING → ACTIVE（修正は継続）または WAITING → CLOSED（OKで完了）
- 申請書最終提示完了・提出案内: ACTIVE → CLOSED
- 対話回数上限（30回）超過: ACTIVE → TERMINATED
- システムエラー発生（回復不能）: ACTIVE → TERMINATED
- 利用者からのキャンセル操作: ACTIVE → TERMINATED

**状態管理責務**:
- Strands Agents SDK がセッション状態をファイルストレージ（data/sessions/）に自動管理する
- セッション状態の明示的な変更はSDKの機能に委譲し、アプリケーションコードで直接操作しない

## 4. セッションで管理する情報の分類

### 4.1 管理情報カテゴリ

| カテゴリ | 説明 | 保存場所 |
|---------|------|---------|
| 会話履歴 | ユーザーとエージェントの対話ターン（入力・応答）の履歴 | Strands Agents SDK セッション（data/sessions/） |
| 業務コンテキスト | 申請者名・申請種別・収集済み申請情報・申請日等の業務データ | Strands Agents SDK セッション（data/sessions/） |
| 実行履歴 | ツール実行結果（TOOL-001/TOOL-002）・ルーティング結果・承認結果 | Strands Agents SDK セッション（data/sessions/） |
| 成果物参照 | 申請書ドラフトファイルパス・申請チェック結果 | Strands Agents SDK セッション（data/sessions/） |
| 状態管理 | 現在のセッション状態（CREATED/ACTIVE/WAITING/CLOSED/TERMINATED）・対話ターン数 | Strands Agents SDK セッション（data/sessions/） |

## 5. 会話履歴管理方針

### 5.1 会話履歴の範囲

**保存対象**:
- ユーザー入力メッセージ（申請内容・回答・確認操作）
- エージェント応答メッセージ（案内・質問・提示内容）
- ツール実行の入力パラメータと返却値

**保存しないもの**:
- LLMの中間思考プロセス（推論ログ）：プロンプト非表示・ユーザーへの非表示（根拠提示方針に従う）
- システム内部のデバッグログ（会話履歴には含めず、ログファイルに別途記録）


### 5.2 会話履歴の粒度

| 粒度 | 方針 |
|-----|------|
| フル履歴 | Strands Agents SDK が会話ターンごとに自動保存する。window_size（AG-001: 30、AG-002: 20、AG-003: 15）でコンテキストウィンドウ内の参照範囲を制御する |
| 要約 | 本システムでは会話要約は行わない（window_sizeでの切り詰めのみ） |
| 重要発言抽出 | 申請種別・申請者名・収集済み申請情報は業務コンテキストとして別途保持する |

---

## 6. 業務コンテキスト管理方針

### 6.1 管理対象

- 申請者名（applicant_name）
- 申請種別（application_type）
- 申請日（application_date）
- セッションID（session_id）
- 収集済み申請情報（移動情報または経費情報）
- 申請ルール照合結果・上長承認要否フラグ・差し戻しリスクフラグ
- 申請書ドラフトファイルパス


### 6.2 更新ルール

**誰が更新できるか**:
- アプリケーション起動時：申請者名（対話ループ開始前にCLIで取得）・申請日（システム日付を自動取得。YYYY-MM-DD形式）
- AG-001：申請種別の判定と確定
- AG-002：交通費精算申請の収集済み申請情報・ルール照合結果の更新
- AG-003：経費精算申請の収集済み申請情報・ルール照合結果の更新
- 利用者（R-EMP）：Human-in-the-Loop承認時（OK/修正/キャンセル）の状態変更のトリガーのみ

**上書き／追記のルール**:
- 申請者名・申請種別・申請日：一度確定したら上書き禁止（修正はユーザーからの明示的な修正指示時のみ）
- 収集済み申請情報（移動情報・経費情報）：区間・経費ごとに追記する。利用者の修正指示で上書き可能
- ルール照合結果・警告フラグ：最新の申請書内容に基づき都度上書きする

**差分管理の考え方**:
- Strands Agents SDK のセッション管理機能に委譲する。明示的な差分管理は行わない

## 7. 実行・判断履歴管理方針

### 7.1 実行履歴

| 項目 | 内容 | 保存場所 |
|-----|------|---------|
| tool_name | ツール名（TOOL-001 / TOOL-002） | Strands Agents SDK セッション（data/sessions/） |
| 実行内容 | ツールパラメータ（移動情報・申請種別等） | Strands Agents SDK セッション（data/sessions/） |
| 実行結果 | ツール戻り値（交通費計算結果・申請書ドラフトパス等） | Strands Agents SDK セッション（data/sessions/） |
| 実行時刻 | タイムスタンプ | Strands Agents SDK セッション（data/sessions/） |
| 再実行可否 | TOOL-001は冪等（同一入力で同一結果）。TOOL-002は再実行でドラフトを上書き生成 | — |

### 7.2 判断履歴

**ルーティング結果**:
- 申請種別の判定結果（交通費精算申請 / 経費精算申請 / 判定不能）と委任先エージェントIDをセッション内に保持する

**エスカレーション判定**:
- エラー発生時・申請期限超過時・判定不能時のエスカレーション案内内容をセッション内に記録する

**人間承認の結果**:
- Human-in-the-Loop承認時（OK/修正/キャンセル）の選択結果をセッション内に記録する

## 8. 成果物・ナレッジ参照管理

### 8.1 管理対象

- 生成ドキュメントID: 申請書ドラフトファイルパス（output/配下のExcelファイルパス）
- 参照ナレッジID: 社内申請ルール（システムプロンプト埋め込み。ファイルパスは参照不要）
- 外部I/F参照ID: 要件上未定義（外部システム連携なし）

### 8.2 管理方針

**実体は保持しない**:
- 申請書テンプレート・申請書ドラフトのExcelファイル実体はファイルストレージ（data/・output/）に保存し、セッションには保持しない

**参照IDのみを保持**:
- 申請書ドラフトのファイルパスのみをセッション内に保持する

**再取得可能性を前提**:
- ファイルパスが存在すれば申請書ドラフトを再参照可能。ファイルが消失した場合はエラーとして処理する

## 9. メモリの種類と役割分担（概念）

### 9.1 メモリ分類

| 種類 | 用途 | 実装 |
|-----|------|------|
| セッションメモリ | 会話・状態の継続（会話履歴・業務コンテキスト・実行履歴） | Strands Agents SDK の FileSessionManager（data/sessions/へのファイル永続化） |
| ワーキングメモリ | LLMのコンテキストウィンドウ内一時処理用（エージェントの推論・応答生成） | Strands Agents SDK のコンテキストウィンドウ（window_size で制御） |
| ロングターム参照 | ナレッジ参照（社内申請ルール・運賃データ） | システムプロンプト埋め込み（ナレッジ） / ファイルI/O（運賃データJSON） |

## 10. 再開・引き継ぎ方針

### 10.1 再開条件

- セッション情報（data/sessions/）が存在する場合: 同一セッションIDで申請フローを再開可能
- セッション情報が消失した場合: 新規セッションを開始する（既存フローの復元不可）
- AG-001からAG-002/AG-003への委任後に中断した場合: セッション情報が存在すれば専門エージェントの途中から再開可能

### 10.2 再開時に復元する情報

| 情報 | 復元要否 | 実装状況 |
|-----|---------|---------|
| 会話履歴 | 要 | Strands Agents SDK が自動復元 |
| 申請者名・申請種別・申請日 | 要 | Strands Agents SDK が自動復元 |
| 収集済み申請情報 | 要 | Strands Agents SDK が自動復元 |
| ルール照合結果・警告フラグ | 要 | Strands Agents SDK が自動復元 |
| 申請書ドラフトファイルパス | 要 | Strands Agents SDK が自動復元 |
| LLM中間思考プロセス | 不要 | 保存対象外 |

## 11. エスカレーション・承認との連携

**エスカレーション発生時に保持すべき情報**:
- エラー種別・エラー発生時のセッション状態
- 申請者名・申請種別・収集済み申請情報（エスカレーション案内提示に利用）

**人間承認時に提示する情報**:
- 申請書ドラフトの内容（申請者名・申請種別・明細行・合計金額）
- 申請ルール照合結果（不備・警告・期限チェック結果）
- 上長承認要否フラグ・差し戻しリスク評価結果

**承認結果の反映方法**:
- 承認（OK）: TOOL-002（申請書生成ツール）を呼び出しExcelドラフトを生成。セッションに申請書ドラフトパスを記録してCLOSEDへ遷移
- 修正要望: 修正対象項目の再収集ループへ戻す（ACTIVEを維持）
- 拒否（キャンセル）: セッションをTERMINATEDに遷移しキャンセルメッセージを提示

## 12. セキュリティ・ガードレール観点

**個人情報の保持方針**:
- セッション情報（DATA-004）は機密区分D-2（機微）として扱う
- 申請内容・申請情報等の個人情報（D-3）を含む会話履歴はセッションに保持するが、ログ出力時はマスキングを適用する

**セッション内秘匿情報の扱い**:
- `invocation_state` でのセッション情報受け渡しはLLMプロンプトに含めない（コンテキストウィンドウ非消費）
- セッションIDはLLMプロンプトに露出させない

**マスキング／削除方針**:
- ログ出力時（DATA-007/008/009）：申請内容・申請者情報等の個人情報をマスキングして記録する
- セッション終了後の保持期間：セッション終了時に削除

---

## 13. ログ・監査との関係

**セッション情報とログの関係**:
- セッション情報（DATA-004）はStrands Agents SDKが管理するファイルセッション
- ログ（DATA-007/008/009）はエラーハンドラー（ErrorHandler）のログメソッド経由で別途logs/に記録する

**監査時に参照可能な情報**:
- 会話ログ（DATA-007）：エージェントと利用者の会話履歴（ターン単位）
- 判断ログ（DATA-008）：申請種別判定・ルールチェック・経費区分判断等の判断根拠・適用ルールID
- 承認ログ（DATA-009）：利用者による最終確認・承認操作・提出操作の記録

**保持期間の考え方**:
- セッション情報（DATA-004）：セッション終了時に削除
- ログ（DATA-007/008/009）：要件上未定義

## 14. 設計上の留意点・リスク

**セッション肥大化のリスク**:
- 長い会話履歴がコンテキストウィンドウを超過するリスク
- 対策: `window_size`（AG-001: 30、AG-002: 20、AG-003: 15）と `should_truncate_results=True` で切り詰め制御を行う

**メモリ誤参照のリスク**:
- 複数セッションが存在する場合にセッションIDを混同するリスク
- 対策: `invocation_state` でセッションIDを渡し、ファクトリ関数でセッションマネージャーを生成する際に必ず正しいセッションIDを使用する

**再開時の誤動作リスク**:
- セッション情報が不整合状態で保存されている場合の不正な復元リスク
- 対策: Strands Agents SDK のファイルセッション機能に委譲し、不整合が発生した場合はErrorHandlerを通じてエラーとして処理する
