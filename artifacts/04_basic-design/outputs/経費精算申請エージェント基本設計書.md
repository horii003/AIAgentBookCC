## 2. 目的・役割定義

### 2.1 エージェントの目的
- AG-001から委譲された経費精算申請フローを引き受け、経費情報を対話で収集する
- 領収書画像をLLMの画像読み取り機能で解析し、店舗名・金額・経費発生日・品目を自動抽出する
- 経費区分を自動判断して社員に提案し、確認を経て申請書（下書き）を生成（TL-002b）する
- 申請期限（90日）チェックおよび高額申請時の上長承認要否通知（5,000円超）を担当する

### 2.2 主な責務
- **領収書画像からの情報抽出**: LLMの画像読み取り機能で店舗名・金額・経費発生日・品目を抽出する（BRL-16 / CAP-APP-009）。失敗時は手動入力フローへ切り替える（GRD-003）
- **経費情報の対話収集**: 店舗名・金額・経費発生日・品目・経費区分・業務目的を一問一答で収集する（CF-004 / BRL-04/05）
- **申請期限チェック（90日）**: 経費発生日から本日までの差が90日を超える場合に申請不可を通知する（BRL-14 / GRD-006）
- **高額申請通知**: 経費が5,000円超の場合に上長承認が必要な旨を通知する（BRL-10 / GRD-007）
- **経費区分の自動判断と提案**: 品目から経費区分（事務用品費・宿泊費・資格精算費・その他経費）を自動判断して提案し、社員の確認を得る（BRL-17 / GRD-010）
- **申請書生成前の社員確認**: HumanApprovalHookで社員のOK/修正/キャンセルを取得する（GRD-014）
- **申請書（下書き）生成**: TL-002b（generate_expense_reimbursement_form）を呼び出して申請書を生成する（CF-005）

### 2.3 非責務（やらないこと）
- **申請種別判断**: AG-001が担当。AG-003は経費精算申請と確定済みの状態で起動される
- **申請書の提出**: 社員が手動で実施（GRD-012 / ACT-EXEC-01禁止）
- **上長承認の代行**: 上長承認要否の通知のみ担当。承認取得は社員が行う（GRD-013）
- **交通費精算申請フロー**: AG-002が担当
- **経費区分の自動確定**: 提案のみで自動確定は禁止。社員の確認が必須（GRD-010）

## 3. 自律度・判断権限

### 3.1 自律度レベル

| レベル | 内容 | 適用範囲 |
|------|------|----------|
| 参照（Lv1） | 社内申請ルールファイル・申請先情報の参照 | P-READ-001 / DATA-001, DATA-004 |
| 提案（Lv2） | 経費区分の自動判断・提案（社員確認が必要） | ACT-DEC-04 / FR-009 / GRD-010 |
| 実行（Lv3） | 申請書生成（TL-002b）（HumanApprovalHook承認後） | ACT-EXEC-01 / FR-011 |

### 3.2 判断権限の範囲
- **自律判断可能**: 領収書画像からの情報抽出（Lv3）、申請期限の自動チェック（90日）、5,000円超の上長承認要否判定
- **提案のみ（人間確認必要）**: 経費区分の自動判断（提案として提示、GRD-010）、収集情報全体の確認（HumanApprovalHook の OK/修正/キャンセル）（GRD-014）
- **実行不可**: 申請書の提出（GRD-012 / ACT-EXEC-01禁止）、上長承認の代行（GRD-013禁止）、経費区分の自動確定（GRD-010禁止）

## 4. 処理フロー設計

### 4.1 基本フロー（概要）

```text
開始（AG-001から委譲、invocation_stateにsession_id・applicant_name・application_date設定済み）
→ 領収書アップロード要求（CF-004 Step 1）
→ 社員: アップロードまたは手動入力を選択（Step 2）
  ├── [アップロード] LLM画像読み取りで情報抽出（BRL-16 / Step 3）
  │   ├── [成功] 抽出情報を取得（Step 5へ）
  │   └── [失敗] 手動入力フローへ切り替え（GRD-003 / Step 4へ）
  └── [手動入力] 一問一答で経費情報を収集（Step 4）
→ 申請期限チェック（BRL-14 / GRD-006 / Step 5）
  ├── [期限超過] 申請不可通知して終了（EX-03）
  └── [期限内] 高額チェック・上長承認要否通知（BRL-10 / GRD-007 / Step 6）
→ 経費区分の自動判断と提案（BRL-17 / GRD-010 / Step 7）
→ 社員: 経費区分を確認・修正（Step 8）
→ 不足情報の確認・収集（Step 9-10）
→ 収集情報確認提示（Step 11）: 収集済み経費情報をテキストとして整理・提示（ツール呼び出しなし）
→ HumanApprovalHook（BeforeToolCallEventトリガー）（社員確認: OK/修正/キャンセル）（GRD-014）
  ├── [OK] TL-002b（generate_expense_reimbursement_form）呼び出し
  ├── [修正] 該当項目再収集へ
  └── [キャンセル] フロー終了
→ 申請書（下書き）ファイルパスを社員に提示（CF-005完了）
→ 終了
```

### 4.2 フロー制御の考え方
- **領収書処理の2経路**: 画像アップロード（LLM抽出）と手動入力の2経路を持つ。LLM抽出失敗時は手動入力に自動切り替え（GRD-003）
- **申請期限チェックのタイミング**: 経費発生日が確定した直後（Step 5）に実施。期限超過時はその後の処理を行わず即座に終了する（EX-03）
- **経費区分の確認ステップ**: 自動判断結果を提案として提示し、必ず社員の確認（Step 8）を経てから確定する（GRD-010）
- **不足情報収集ループ**: Step 9-10 で不足項目を検出・収集するループ構造。全必須項目が揃った時点で Step 11 へ進む
- **ドラフト提示とツール呼び出しの分離**: Step 11 で収集済み経費情報をテキストとして整理・提示（ツール呼び出しなし）した後、社員が申請書生成を指示したタイミングで BeforeToolCallEvent が発火し HumanApprovalHook が介入する。Human-in-the-Loop 承認は1回のみ発生する設計とする。承認前に TL-002b を実行してはならない

### 4.3 CF-004 会話ステップ詳細

| Step | 発話者 | 内容 | 判断/処理 | 次ステップ | 根拠提示 |
|------|--------|------|----------|-----------|---------|
| 1 | Agent | 領収書アップロード要求（手動入力も可） | — | Step 2 | Lv0 |
| 2 | 社員 | 画像アップロードまたは手動入力を選択 | — | Step 3（画像）/ Step 4（手動） | — |
| 3 | Agent | LLM画像読み取りで情報抽出 | CAP-APP-009（BRL-16）/ JD-14/15 | Step 5（成功）/ Step 4（失敗: GRD-003） | Lv0 |
| 4 | Agent | 手動入力要求（一問一答） | DIALOG-01/02 | Step 5 | Lv0 |
| 5 | Agent | 申請期限チェック | BRL-14（JD-10/11） | Step 6（期限内）/ 終了（期限超過） | **Lv3（超過時）** |
| 6 | Agent | 高額チェック・上長承認要否通知 | ABAC-003（BRL-10 / JD-09） | Step 7 | **Lv3（超過時）** |
| 7 | Agent | 経費区分の自動判断と提案 | ACT-DEC-04（BRL-17 / EXPENSE-02） | Step 8 | Lv1（提案） |
| 8 | 社員 | 経費区分を確認・修正（GRD-010） | — | Step 9 | — |
| 9 | Agent | 不足情報の確認 | JD-04/05 | Step 10（不足あり）/ Step 11（全収集完了） | Lv0 |
| 10 | 社員 | 不足情報の入力 | — | Step 9（不足残）/ Step 11（完了）/ CF-006（不明・中断） | — |
| 11 | Agent | 収集済み経費情報をテキストとして整理・提示（ツール呼び出しなし） | ACT-GEN-02 | CF-005（HumanApprovalHook: BeforeToolCallEventトリガー） | Lv0 |

### 4.4 状態管理
- **invocation_state（AG-001から受け取り）**: `session_id`・`applicant_name`・`application_date`
- **収集済み申請情報（D-010）**: セッション状態ファイル（`data/sessions/{session_id}.json`）に永続化。経費明細情報（D-015）を蓄積する
- **申請者名（applicant_name）**: invocation_stateから取得済み。AG-003は対話で再収集しない
- **申請日（application_date）**: invocation_stateから取得済み。申請書生成時に使用する

## 6. プロンプト設計

### 6.1 システムプロンプトの目的
- 経費精算申請フローの進行ルール（CF-004）を定義する
- 申請期限チェック（BRL-14 / 90日）の判断基準を組み込む（動的生成が必要な理由）
- 経費区分の自動判断ルール（BRL-17）と確認必須（GRD-010）を定義する
- 高額申請通知（BRL-10 / 5,000円超）のルールを定義する
- 申請書提出禁止（GRD-012 / ACT-EXEC-01）を明示する
- HumanApprovalHookによる確認承認フロー（GRD-014）を定義する

### 6.2 プロンプト構成要素
- **役割定義**: 経費精算申請の専門エージェントとして、経費情報収集・経費区分判断・申請書生成を担当する
- **処理フロー**: CF-004（11ステップ）の進行手順
- **判断プロトコル**: 申請期限チェック（経費発生日から本日まで90日以内）、経費区分自動判断（4種類）、高額チェック（5,000円超で上長承認通知）
- **主な責任**: 経費情報の正確な収集、経費区分の提案（確定は社員の確認後）、申請書生成前の社員確認
- **動的情報**: 本日の日付（`application_date`）、申請期限日（`deadline_date`）
- **エラー発生時の振る舞い**: システム系エラー（ツールエラー・システム障害等）が発生した場合、呼び出し元エージェント（AG-001）にエラー内容を要約したメッセージを返却する

### 6.3 システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | **動的生成関数** |
| 動的生成の理由 | 申請期限チェック（BRL-14）で経費発生日から90日以内かを判断するために、本日の日付（`date.today()`）をプロンプトに埋め込む必要がある |
| 動的に埋め込む情報 | `application_date`（本日の日付、YYYY-MM-DD形式）、`deadline_date`（本日 - 90日、YYYY-MM-DD形式） |

```python
def _build_expense_agent_system_prompt(application_date: str) -> str:
    deadline_date = (date.fromisoformat(application_date) - timedelta(days=90)).isoformat()
    return f"""あなたは経費精算申請の専門エージェントです。...
申請日: {application_date}
申請期限: {deadline_date}以降の経費のみ申請可能（経費発生日がこれより前の場合は申請不可）
..."""
```

### 6.4 主要な設計意図
- **動的プロンプト**: AG-002と同様に申請期限チェックには実行時の日付が必要なため毎回プロンプトを生成する。埋め込む情報は日付2つのみに限定する
- **経費区分提案の明示**: LLMが経費区分を「確定」せず「提案」するようプロンプトで明示し、GRD-010を確実に実装する
- **根拠提示**: 申請期限超過通知・高額通知時はルールID（BRL-14・BRL-10）を提示して社員が確認できるようにする

**プロンプト全文**: 詳細設計書 2.3.1章を参照

## 10. 出力データ設計

### 10.1 出力形式
- **出力先**: 社員（チャットインターフェース）
- **出力内容**: 各ステップでの質問・確認メッセージ、LLM抽出結果の提示、経費区分の提案メッセージ、高額通知、申請書ファイルパスの案内

### 10.2 主要な出力メッセージ
- 申請期限超過通知: `"経費発生日「{expense_date}」から90日を超えているため、経費精算申請はできません（根拠: BRL-14）。"`
- 高額申請通知: `"経費が5,000円を超えているため、上長の承認が必要です（根拠: BRL-10）。"`
- 経費区分提案: `"品目「{item_name}」から経費区分を「{expense_category}」と判断しました（参照: BRL-17）。よろしければそのままでお答えください。修正が必要な場合はご入力ください。"`
- LLM抽出失敗: `"領収書の読み取りができませんでした。店舗名・金額・経費発生日・品目を順番に入力してください。"`
- 申請書生成完了: `"申請書（下書き）を「{file_path}」に生成しました。申請先に提出してください。"`

### 10.3 成果物
- **経費情報（D-015）**: 店舗名・金額・経費発生日・品目・経費区分・業務目的
- **収集済み申請情報（D-010）**: セッション状態ファイルに永続化された経費明細集約情報
- **申請書（下書き）（D-011）**: generate_expense_reimbursement_formが生成したExcelファイル

## 11. バリデーション設計

**システム設計書の方針**: `artifacts/03_system-design/outputs/バリデーション方針.md` を参照。

### 11.1 バリデーション方針
- **経費区分**: `normalize_expense_category`（BRL-17）で別表記（"ホテル"→"宿泊費"等）を正規4種に変換。対応外は ValidationError（EX-01）
- **経費発生日**: 未来日付禁止（当日以前）、申請期限チェック（BRL-14）はシステムプロンプト内で実施（`check_application_deadline`は ExpenseItem 側でも実施）
- **金額**: `int`型・`ge=0`、文字列 `"1,500円"` → `1500` 自動変換
- **業務目的**: `min_length=1`・空白のみ禁止（BRL-18）

### 11.2 申請ルール一覧

| ルールID | ルール名 | 条件 | エージェントの対応 |
|----------|----------|------|------------------|
| BRL-04 | 申請書生成前の必須項目充足確認 | 全必須項目（D-008）が収集済みの場合のみ申請書生成を許可 | 未収集項目を提示して対話継続 |
| BRL-05 | 不足情報残存時の対話継続 | 必須項目に未収集あり | 不足項目を確認して収集を続ける（Step 9-10ループ） |
| BRL-10 | 高額申請の上長承認要否 | 経費 > 5,000円 | 上長承認が必要な旨を通知（Lv3根拠提示）。フロー継続 |
| BRL-14 | 申請期限90日チェック | (application_date - expense_date).days > 90 | 申請不可通知してフロー終了（EX-03） |
| BRL-16 | 領収書LLM画像読み取り | 領収書画像が提供された場合 | LLMの画像読み取り機能で情報抽出（失敗時は手動入力へ） |
| BRL-17 | 経費区分の自動判断 | 品目が入力済みの場合 | 品目から経費区分を自動判断して提案（GRD-010: 確定は社員確認後） |

## 12. Agent as Tools設計（AG-003の被呼び出し側設計）

### 12.1 AG-003のAgent as Tools化
AG-003はAG-001から `@tool(context=True)` のデコレータを持つファクトリ関数 `_get_expense_agent(session_id)` を通じてツールとして呼び出される。

### 12.2 ツール関数設計

| 項目 | 内容 |
|------|------|
| ツール関数名 | expense_agent_tool |
| デコレータ | @tool(context=True) |
| 役割 | AG-003インスタンスを生成してAG-001から呼び出し可能にする |
| 呼び出し元 | AG-001（申請種別 = 経費精算申請確定後） |

### 12.3 ToolContextからのinvocation_state受け取り

```python
@tool(context=True)
def expense_agent_tool(context: ToolContext, ...) -> str:
    # invocation_stateから状態を取得
    invocation_state = InvocationState.model_validate(context.invocation_state)
    session_id = invocation_state.session_id
    applicant_name = invocation_state.applicant_name
    application_date = invocation_state.application_date
    
    # セッションIDでエージェントインスタンスを生成
    agent = _get_expense_agent(session_id, applicant_name, application_date)
    return agent(...)
```

## 13. テスト観点

### 13.1 テスト観点の概要
- **機能テスト**: CF-004の11ステップ全体の正常系フロー（LLM抽出経路）。CF-004の手動入力経路。経費区分の自動判断と提案確認フロー（Step 7-8）。申請書生成（TL-002b）の成功確認
- **業務ルールテスト**: 申請期限90日超過時のフロー終了確認（BRL-14）。経費5,000円超の上長承認通知（BRL-10）。LLM抽出失敗時の手動入力切り替え（GRD-003）。経費区分の自動確定禁止（GRD-010）

- **境界値テスト**: 経費発生日が90日ちょうど（期限内）と91日前（期限超過）。金額がちょうど5,000円（通知なし）と5,001円（通知あり）

## 15. 制約事項・前提条件

### 15.1 技術的制約
- Strands Agents SDK v1.25.0 の `Agent` クラスを使用する
- LLMバックエンドは Amazon Bedrock（Claude Sonnet 4.5）のみ
- `SlidingWindowConversationManager(window_size=15, should_truncate_results=True)` を使用する
- HumanApprovalHookはAG-003への必須設定とし、TL-002b呼び出し前に介入する
- システムプロンプトは動的生成関数（`_build_expense_agent_system_prompt`）で生成する
- LLM画像読み取りはBedrockのClaude Sonnet 4.5のネイティブマルチモーダル機能を使用する

### 15.2 業務的制約
- 申請書の提出はAG-003が行わない（P-EXEC-003禁止 / GRD-012）
- 上長承認の代行はAG-003が行わない（GRD-013）
- 経費区分の自動確定は禁止。提案→社員確認の2段階が必須（GRD-010）
- 申請期限は経費発生日から90日以内（BRL-14）。超過時は申請不可でフロー終了
- 高額通知の閾値は5,000円超（AG-002の10,000円とは異なる）（BRL-10）

### 15.3 前提条件
- AG-001が申請種別を「経費精算申請」に確定した後に起動されること
- `invocation_state` に `session_id`・`applicant_name`・`application_date` がセット済みであること（AG-001が設定）
- TL-002b用テンプレート（経費精算申請書_template.xlsx）が `templates/` 配下に配置されていること
