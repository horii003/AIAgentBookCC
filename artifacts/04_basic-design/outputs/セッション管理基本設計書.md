## 2. 目的・役割定義

### 2.1 コンポーネントの目的
- マルチエージェント間でセッション情報（業務コンテキスト・セッション状態・成果物参照）を共有する唯一の永続化ストアを提供する
- セッション識別子（session_id）を生成・管理し、AG-001/AG-002/AG-003が同一セッション内で一貫した状態を参照できるようにする
- セッションのライフサイクル（CREATED → ACTIVE → WAITING → CLOSED/TERMINATED）を管理し、異常終了・正常終了を適切に記録する

### 2.2 主な責務
- **SessionManagerFactory の提供**: `SessionManagerFactory.create()` を呼び出すことで `FileBasedSessionManager` インスタンスを返す。各エージェント（AG-001/AG-002/AG-003）はファクトリ経由でインスタンスを取得して Agent に渡す
- **セッション生成**: AG-001 起動時に session_id を採番し、`data/sessions/session_{id}.json` を作成して初期状態（CREATED）を記録する
- **業務コンテキストの永続化**: 申請者名・申請日・申請種別・収集済み移動情報リスト・収集済み経費情報リスト・申請書ファイルパスを session_{id}.json に保存・更新する
- **セッション状態管理**: CREATED/ACTIVE/WAITING/CLOSED/TERMINATED の状態遷移を管理し、各遷移時にファイルへ書き込む
- **APR-001 承認フラグの記録**: `approval_granted` フィールドを session_{id}.json に記録し、TOOL-002 実行前にフックが参照できるようにする
- **セッション終了処理**: CLOSED（正常終了）または TERMINATED（異常終了）時にセッションファイルを最終更新する

### 2.3 非責務（やらないこと）
- **会話履歴の永続化**: `SlidingWindowConversationManager`（Strands SDK）が担当する
- **ツール実行ログの記録**: 監査ログ（audit_log_std.jsonl / audit_log_hi.jsonl）が担当する
- **ループ制御・承認制御**: LoopControlHook（HD-003）・HumanApprovalHook（HD-002）が担当する
- **セッションタイムアウトの自動処理**: 要件上未定義（実装フェーズで定義）

### 2.4 必要性
- AG-001 が生成した申請者名・申請日を AG-002/AG-003 が参照するためのファイルストア上の共有媒体が必要
- APR-001 承認フラグを invocation_state 外に永続化してフック・エージェント間で共有する仕組みが必要
- セッション状態（ACTIVE/WAITING/CLOSED/TERMINATED）を永続化し、中断・再開時に処理を継続できるようにする必要がある

### 2.5 主要な設計意図
- **ファイルベース採用**: 本システムは単一プロセス・単一マシン運用を前提とし、外部DBを必要としないファイルI/Oでの実装を選択する（セッション管理方針.md 9.1）
- **session_id のファクトリ関数内消費**: session_id は AG-002/AG-003 のファクトリ関数内でのみ `FileBasedSessionManager` の生成に使用し、LLMプロンプトには含めない（マルチエージェント連携設計.md 6.1/7.2）
- **最新状態の上書き保存**: バージョン管理・差分管理は行わず、常に最新状態を上書き保存することで実装を単純化する（セッション管理方針.md 6.2）

## 3. SessionManagerFactoryクラス

### 3.1 クラス概要
- **クラス名**: `SessionManagerFactory`
- **説明**: `FileBasedSessionManager` インスタンスの生成を一元管理するファクトリクラス。各エージェントは直接 `FileBasedSessionManager` をインスタンス化せず、必ず `SessionManagerFactory.create()` 経由で取得する。将来的な依存性注入や実装差し替え（例: DBベースセッション管理への移行）に対応できるよう Factory パターンを採用する

### 3.2 主要メソッド

| メソッド名 | 目的 | 引数 | 戻り値 |
|-----------|-----|------|-------|
| `create()` | `FileBasedSessionManager` インスタンスを生成して返す | なし | `FileBasedSessionManager` |

**使用例**:
```python
# AG-001: セッション生成時
session_manager = SessionManagerFactory.create()
session_id = session_manager.create_session(applicant_name, application_date)

# AG-002/AG-003: ファクトリ関数内
session_manager = SessionManagerFactory.create()
session_data = session_manager.load_session(session_id)
```

**詳細**: 詳細設計書 2.1章を参照

## 4. FileBasedSessionManagerクラス

### 4.1 クラス概要
- **クラス名**: `FileBasedSessionManager`
- **説明**: セッション情報をJSONファイルとして `data/sessions/` ディレクトリに永続化するセッション管理クラス。各エージェントは `SessionManagerFactory.create()` 経由でインスタンスを取得し、同一ファイルを読み書きする

### 4.2 クラス変数
- **`session_id`**: セッションを一意に識別する文字列（形式: `{YYYYMMDD}_{HHMMSS}_{UUID4前8桁}`）（初期値: なし）
- **`session_dir`**: セッションファイルの保存先ディレクトリパス（初期値: `data/sessions/`）
- **`session_file_path`**: セッションファイルのフルパス（`data/sessions/session_{id}.json`）（初期値: session_id から自動生成）

**詳細**: 詳細設計書 2章を参照

## 5. 主要メソッド

### 5.1 create_session
- **目的**: 新規セッションを生成し、初期状態で session_{id}.json を作成する
- **引数**: `applicant_name: str`（申請者名）、`application_date: date`（申請日）
- **戻り値**: `str`（生成した session_id）
- **フォーマット**: session_id は `{YYYYMMDD}_{HHMMSS}_{UUID4前8桁}` 形式で採番する（例: `20260501_143022_a1b2c3d4`）

### 5.2 load_session
- **目的**: 指定した session_id のセッションファイルを読み込み、業務コンテキストを返す
- **引数**: `session_id: str`（読み込み対象のセッションID）
- **戻り値**: `dict`（セッションデータ全体。ファイルが存在しない場合は `None`）
- **デフォルトパス**: `data/sessions/session_{session_id}.json`

### 5.3 save_session
- **目的**: 業務コンテキストの更新内容を session_{id}.json に上書き保存する
- **引数**: `data: dict`（保存するセッションデータ）
- **戻り値**: `bool`（保存成功: `True`、失敗: `False`）

### 5.4 update_status
- **目的**: セッション状態（status フィールド）を更新して session_{id}.json に保存する
- **引数**: `status: str`（遷移先の状態: `CREATED` / `ACTIVE` / `WAITING` / `CLOSED` / `TERMINATED`）
- **戻り値**: `bool`（更新成功: `True`、失敗: `False`）
- **フォーマット**: 更新日時（ISO8601）を `updated_at` フィールドに同時記録する

### 5.5 set_approval_granted
- **目的**: APR-001 承認済みフラグを session_{id}.json に記録する
- **引数**: `granted: bool`（承認: `True`、キャンセル: `False`）
- **戻り値**: `bool`（更新成功: `True`、失敗: `False`）

### 5.6 close_session
- **目的**: セッションを正常終了（CLOSED）させ、申請書ファイルパスを記録する
- **引数**: `generated_file_path: str`（生成した申請書ファイルパス）
- **戻り値**: `bool`（終了処理成功: `True`、失敗: `False`）

### 5.7 terminate_session
- **目的**: セッションを異常終了（TERMINATED）させ、終了理由を記録する
- **引数**: `reason: str`（終了理由。例外種別 EX-XX の文字列）
- **戻り値**: `bool`（終了処理成功: `True`、失敗: `False`）

**詳細**: 詳細設計書 3.1章〜3.7章を参照

## 6. セッション管理の仕組み

**システム設計書の方針**: システム設計書 セッション管理方針.md（`artifacts/03_system-design/outputs/セッション管理方針.md`）を参照してください。

### 6.1 セッション生成（AG-001 起動時）
1. AG-001 が最初のユーザー入力を受け付けた時点で `SessionManagerFactory.create()` を呼び出し、返却された `FileBasedSessionManager` インスタンスで `create_session()` を実行する
2. session_id を `{YYYYMMDD}_{HHMMSS}_{UUID4前8桁}` 形式で採番する
3. `data/sessions/session_{id}.json` を作成し、状態を `CREATED` に設定する
4. session_id を invocation_state に格納して AG-002/AG-003 のファクトリ関数へ渡す

### 6.2 セッション更新（AG-002/AG-003 での情報収集中）
1. AG-002/AG-003 のファクトリ関数内で `SessionManagerFactory.create()` を呼び出し、返却された `FileBasedSessionManager` インスタンスを使用する。invocation_state から session_id を取得し `load_session(session_id)` で既存セッションを読み込む
2. 対話で収集した移動情報（AG-002）または経費情報（AG-003）を `save_session()` で session_{id}.json に追記保存する
3. ユーザーへの入力待ち時は `update_status(WAITING)` を呼び出す
4. ユーザーの入力再開時は `update_status(ACTIVE)` を呼び出す

### 6.3 APR-001 承認フラグの記録（HumanApprovalHook 経由）
1. HumanApprovalHook が APR-001 ダイアログを提示してユーザー選択を取得する
2. OK 選択時: `set_approval_granted(True)` を呼び出し、`approval_granted = True` を session_{id}.json に記録する
3. キャンセル選択時: `terminate_session(reason="APR-001 キャンセル")` を呼び出す
4. 修正選択時: `approval_granted` フラグは更新せず情報再収集ループへ戻す

### 6.4 セッション終了
1. TOOL-002 による申請書生成完了 → `close_session(generated_file_path)` を呼び出し、状態を `CLOSED` に遷移させる
2. 例外発生（EX-05/06/07/08 等）→ `terminate_session(reason=EX分類)` を呼び出し、状態を `TERMINATED` に遷移させる
3. ユーザーによるキャンセル（APR-001 拒否）→ `terminate_session(reason="ユーザーキャンセル")` を呼び出す
4. セッション終了後はファイルを最終状態のまま保持する（削除タイミングは要件上未定義）

**詳細**: 詳細設計書 4章を参照

## 7. マルチエージェント連携

### 7.1 セッション共有の仕組み

**AG-001（オーケストレーター）**:
1. 起動時に `SessionManagerFactory.create()` を呼び出して `FileBasedSessionManager` インスタンスを取得し、`create_session()` を実行する
2. 申請者名・申請日・申請種別を `save_session()` で session_{id}.json に保存する
3. session_id を `invocation_state["session_id"]` に格納して AG-002/AG-003 のツール関数へ渡す
4. AG-002/AG-003 の処理結果を受け取り、session_{id}.json から最終状態を参照して結果をユーザーへ提示する

**AG-002/AG-003（専門エージェント）**:
1. ツール関数（`@tool(context=True)`）内で `tool_context.invocation_state["session_id"]` を取得する
2. `SessionManagerFactory.create()` を呼び出して `FileBasedSessionManager` インスタンスを取得し、`load_session(session_id)` で既存セッション情報を読み込む
3. 情報収集完了ごとに `save_session()` で収集済み情報を追記保存する
4. 処理完了時に `close_session()` または `terminate_session()` を呼び出す

### 7.2 会話履歴の共有範囲
- **共有される情報**: 業務コンテキスト（申請者名・申請日・申請種別・収集済み申請情報・承認フラグ・申請書ファイルパス）を session_{id}.json で全エージェント間共有
- **共有されない情報**: 会話履歴（各エージェントの `SlidingWindowConversationManager` がエージェント内メモリで独立管理）、監査ログの内容（各エージェントが個別記録）

## 12. 詳細設計への引き渡し事項

### 12.1 実装時の識別子
- **クラス名**: `SessionManagerFactory`、`FileBasedSessionManager`
- **実装ファイル**:
  - `session/session_manager_factory.py`（SessionManagerFactory）
  - `session/file_based_session_manager.py`（FileBasedSessionManager）

### 12.2 SDK実装時の注意事項
- **ファイルI/O の排他制御**: AG-001 が session_{id}.json を書き込み中に AG-002/AG-003 が読み込むシナリオは発生しない（AG-001 は invocation_state で session_id を渡してから AG-002/AG-003 を呼び出す同期処理のため）。ただし将来の並列化に備えてファイルロック機構を検討する
- **JSON シリアライズ**: `date` 型は ISO8601 文字列（`YYYY-MM-DD`）に変換して保存する。読み込み時は文字列を `date` 型に復元する
- **`session_id` の LLM 非公開**: session_id は LLM プロンプトには含めない。ファクトリ関数内のみで `FileBasedSessionManager` の生成に使用する（マルチエージェント連携設計.md 7.2）
- **ディレクトリ自動作成**: `create_session()` 実行時に `data/sessions/` が存在しない場合は `os.makedirs()` で自動作成する

### 12.3 モック／スタブ要否
- **単体テスト**: `tmp_path` フィクスチャ（pytest）で一時ディレクトリを作成し、`session_dir` を差し替えてテストする。実ファイルI/Oを使用する（モック不要）
- **統合テスト**: 実際の `data/sessions/` ディレクトリに書き込むテスト用セッションIDを使用し、テスト後にファイルを削除するクリーンアップ処理を実装する

## 14. 制約事項・前提条件

### 14.1 技術的制約
- `data/sessions/` ディレクトリが書き込み可能な状態で存在すること（または自動作成されること）
- JSON シリアライズ／デシリアライズには Python 標準の `json` モジュールを使用すること
- session_{id}.json のファイルI/O は Python 標準の `open()` 関数を使用し、外部DBは使用しないこと

### 14.2 業務的制約
- session_id は AG-001 のみが採番し、AG-002/AG-003 は受け取った session_id を使用する（重複採番禁止）
- 同一セッション内で AG-002 と AG-003 が並列実行されることはない（マルチエージェント連携設計.md 6.2/6.3）
- セッション状態の更新権限: 状態を CLOSED に遷移させるのは AG-002/AG-003 のみ。TERMINATED は AG-002/AG-003 および ErrorHandler が実施する

### 14.3 前提条件
- invocation_state に `session_id`（str）が含まれていること（AG-001 が採番して設定する）
- `data/sessions/` ディレクトリが OS 上に存在し、プロセスが書き込み権限を持つこと
- セッションファイルのエンコーディングは UTF-8 とすること
