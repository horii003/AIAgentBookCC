## 1. 概要

### 1.1 コンポーネントの目的

収集済み申請情報をExcelテンプレートに書き込み、申請書（下書き）ファイルを生成する。交通費精算申請用（`generate_transport_expense_form`）と経費精算申請用（`generate_expense_reimbursement_form`）の2ツールに分割し、それぞれ AG-002・AG-003 から呼び出される。出力ファイルパスはツール内部で自律生成し、LLM のパラメータとして受け取らない。申請書の提出は社員が行い、ツールは生成のみ担当する。

### 1.2 主要な責務

1. **入力バリデーション**: `TransportFormInput` / `ExpenseFormInput` (Pydantic v2) で入力データを検証する
2. **テンプレート読み込み**: `openpyxl` でExcelテンプレートを読み込む（元テンプレートを直接上書きしない）
3. **フィールド書き込み**: 収集済み申請情報を指定セル位置に書き込む
4. **ファイルパス生成**: `datetime.now().strftime("%Y%m%d_%H%M%S")` ベースでユニークなファイルパスをツール内部で生成する
5. **ファイル保存**: `output/` ディレクトリに申請書（下書き）を保存する（ディレクトリ不存在時は自動作成）
6. **エラー返却**: 例外は再送出せず `{"success": False, "message": ...}` を返す

## 3. generate_transport_expense_formツール設計

### 3.1 基本情報

#### ツール名（関数名）
`generate_transport_expense_form`

#### 説明（docstring）
```
交通費精算申請書（下書き）を Excel ファイルとして生成する。

HumanApprovalHook による社員確認（OK）取得後のみ呼び出すこと（GRD-014）。
申請者名・申請日は invocation_state から取得する。LLM はパラメータとして渡さない。
出力ファイルパスはツール内部で自動生成する。

Args:
    segments (list[dict]): 移動情報リスト（1区間以上）。各要素は以下のキーを持つ辞書:
        - travel_date (str): 移動日（YYYY-MM-DD形式）
        - departure (str): 出発地（空文字不可）
        - destination (str): 目的地（空文字不可）
        - transportation_type (str): 交通手段（電車/バス/タクシー/飛行機）
        - amount (int): 費用（円、0以上）
        - purpose (str): 業務目的（空文字不可）

Returns:
    dict: 以下のキーを持つ辞書:
        - success (bool): 処理成否。
        - file_path (str): 生成ファイルのパス。success=False の場合は空文字。
        - message (str): エラーメッセージ。success=True の場合は空文字。
```

---

### 3.2 インターフェース設計

#### 3.2.1 入力

**パラメータ定義**:

| 引数名 | 型 | 説明 | デフォルト値 |
|--------|-----|------|--------------|
| `segments` | `list[dict]` | 移動情報リスト（1区間以上） | なし（必須） |

> **注意**: `applicant_name` および `application_date` は LLM からのパラメータとして受け取らず、`invocation_state` から取得する。

##### バリデーションルール詳細

| フィールド | ルール | エラーメッセージ | 例 |
|-----------|--------|----------------|-----|
| `applicant_name` | `min_length=1`、空白のみ禁止 | "申請者名が入力されていません。申請者名を入力してください" | NG: `""`, `"   "` |
| `application_date` | YYYY-MM-DD形式、当日以前 | "申請日はYYYY-MM-DD形式で入力してください" | NG: `"20260502"` |
| `segments` | `min_length=1`（1区間以上） | "移動情報が入力されていません。移動情報を入力してください" | NG: `[]` |
| `segments[*].travel_date` | YYYY-MM-DD形式、90日以内（BRL-14） | "移動日はYYYY-MM-DD形式で入力してください" / "申請期限（経費発生日から90日以内）を超過しています。担当部門にご確認ください。" | NG: `""` |
| `segments[*].departure` | `min_length=1`、空白のみ禁止 | "出発地が入力されていません。出発地を入力してください" | NG: `""` |
| `segments[*].destination` | `min_length=1`、空白のみ禁止 | "目的地が入力されていません。目的地を入力してください" | NG: `""` |
| `segments[*].transportation_type` | `["電車", "バス", "タクシー", "飛行機"]` の1つ（BRL-12） | "対応していない交通手段です" | NG: `"自動車"` |
| `segments[*].amount` | `ge=0`（0以上の整数） | "費用は0円以上の値を入力してください" | NG: `-100` |
| `segments[*].purpose` | `min_length=1`、空白のみ禁止（BRL-18） | "業務目的が入力されていません。業務目的を入力してください" | NG: `""` |

##### dict型パラメータの必須キーガード

`segments` リストの各要素は以下の6キーを全て含むこと。不足しているキーがある場合は以下のエラーメッセージを返す:

```
"移動情報の必須キーが不足しています。以下のキーをすべて含めてください: travel_date, departure, destination, transportation_type, amount, purpose"
```

#### 3.2.2 出力

**戻り値の型**: `dict[str, bool | str]`

**説明**: 処理成否・生成ファイルパス・エラーメッセージを含む辞書。例外は再送出しない。

**構造**:
```python
{
    "success": bool,     # 処理成否
    "file_path": str,    # 生成ファイルのパス。success=False の場合は ""
    "message": str,      # エラーメッセージ。success=True の場合は ""
}
```

#### 3.2.3 例外

| 例外 | 発生条件 |
|------|---------|
| `ValidationError` (Pydantic) | 入力値がバリデーションルールに違反した場合（EX-01/EX-03として処理） |
| `FileNotFoundError` | テンプレートファイルが存在しない場合（EX-04として処理） |
| `openpyxl.utils.exceptions.InvalidFileException` | テンプレートファイルが破損・不正な場合（EX-06として処理） |
| `OSError` | ファイル保存時の書き込み失敗（EX-06として処理） |

### 3.3 ビジネスロジック

#### 3.3.1 処理フロー

```
開始: generate_transport_expense_form(context, segments) 呼び出し
  ↓
【ステップ0】invocation_state から申請者情報取得
  - applicant_name = context.invocation_state.get("applicant_name", "")
  - application_date = context.invocation_state.get("application_date", "")
  ↓
【ステップ1】必須キーガード（dict型パラメータ）
  - segments が空リストでないことを確認（空の場合は EX-01エラーを返す）
  - 各 segment dict に必須キー(travel_date, departure, destination, transportation_type, amount, purpose) が存在するか確認
  - 不足キーがある場合: return {"success": False, "file_path": "", "message": "移動情報の必須キーが不足しています。以下のキーをすべて含めてください: travel_date, departure, destination, transportation_type, amount, purpose"}
  ↓
【ステップ2】TransportFormInput バリデーション
  - TransportFormInput を生成（Pydantic v2 モデル）
  - segments を TransportSegment のリストとして検証（各フィールド型・制約・BRL-12/14/18）
  - ValidationError 発生時 → 【エラーフロー1: バリデーションエラー】へ
  ↓
【ステップ3】テンプレート読み込み
  - os.path.exists(TRANSPORT_TEMPLATE_PATH) でファイル存在を確認
  - ファイルが存在しない場合 → 【エラーフロー2: テンプレート不在（EX-04）】へ
  - TRANSPORT_TEMPLATE_PATH のファイルを openpyxl.load_workbook() で読み込む
  - ファイル破損 (InvalidFileException) → 【エラーフロー3: ファイル保存エラー（EX-06）】へ
  ↓
【ステップ4】フィールド書き込み
  - ws["B3"] = applicant_name（申請者名）
  - ws["B4"] = application_date（申請日）
  - 各 segment（インデックス i, 0始まり）を行 (7 + i) に書き込む:
    - ws[f"A{7+i}"] = i + 1（No）
    - ws[f"B{7+i}"] = segment.travel_date（移動日）
    - ws[f"C{7+i}"] = segment.departure（出発地）
    - ws[f"D{7+i}"] = segment.destination（目的地）
    - ws[f"E{7+i}"] = segment.transportation_type（交通手段）
    - ws[f"F{7+i}"] = segment.amount（費用）
    - ws[f"G{7+i}"] = segment.purpose（業務目的）
    - ws[f"H{7+i}"] = ""（承認状況: 空欄）
  - 書き込みエラー (Exception) → 【エラーフロー3: ファイル保存エラー（EX-06）】へ
  ↓
【ステップ5】出力ファイルパス生成
  - session_id = context.invocation_state.get("session_id", "unknown")
  - output_dir = f"data/output/{session_id}/"
  - os.makedirs(output_dir, exist_ok=True)
  - safe_name = re.sub(r'[\\/:*?"<>|]', '_', applicant_name)
  - file_path = f"{output_dir}{safe_name}_交通費精算申請書_{datetime.now().strftime('%Y%m%d_%H%M%S')}.xlsx"
  ↓
【ステップ6】ファイル保存
  - try: wb.save(file_path)
  - except IOError as e: → 【エラーフロー3: ファイル保存エラー】へ
  - except PermissionError as e: → 【エラーフロー3: ファイル保存エラー】へ
  - except Exception as e: → 【エラーフロー3: ファイル保存エラー】へ
  ↓
【ステップ7】成功レスポンス生成
  - logger.info(f"[OPE-002] generate_transport_expense_form 完了: file_path={file_path}")
  - return {"success": True, "file_path": file_path, "message": ""}
  ↓
終了
```

#### 3.3.3 データの検証方法（Pydantic モデル使用）

**Pydanticモデル**: `TransportFormInput`

**検証内容**:
- `applicant_name`: `strip_and_validate_not_empty` → `min_length=1`
- `application_date`: 文字列→`date`自動変換 → 未来日付禁止
- `segments`: `min_length=1`（1件以上）
- 各 `TransportSegment`:
  - `travel_date`: `check_application_deadline` (model_validator mode="after") → 90日超過チェック
  - `departure`, `destination`: `normalize_station_name` (mode="before")
  - `transportation_type`: `normalize_transportation_type` (mode="before") → `Literal["電車", "バス", "タクシー", "飛行機"]`
  - `amount`: `ge=0`
  - `purpose`: `strip_and_validate_not_empty` → `min_length=1`

### 3.4 エラーハンドリング

#### 3.4.1 エラー処理フロー

```
【エラーフロー1: バリデーションエラー（EX-01/EX-03）】
ValidationError 発生
  ↓
error_handler.handle_validation_error(e) に委譲
  ↓

【エラーフロー2: テンプレート不在（EX-04）】

  ↓

    "申請書テンプレートが見つかりませんでした。担当部門（管理部）にお問い合わせください。"}

【エラーフロー3: ファイル保存エラー（EX-06）】
InvalidFileException / IOError / PermissionError / Exception 発生
  ↓
except IOError as e:
    logger.error(f"[ERR-006] 申請書保存失敗(IOError): {e}")

except PermissionError as e:
    logger.error(f"[ERR-006] 申請書保存失敗(PermissionError): {e}")
    return {"success": False, "file_path": "", "message": error_handler.handle_file_save_error(e)}
except Exception as e:
    logger.error(f"[ERR-006] 申請書生成失敗: {e}")
    return {"success": False, "file_path": "", "message": error_handler.handle_file_save_error(e)}
```

#### 3.4.2 エラーメッセージの詳細

| エラー種別 | 発生条件 | 対応 | ユーザー向けメッセージ |
|-----------|---------|------|---------------------|
| EX-01（入力例外） | バリデーションルール違反・必須キー不足 | ErrorHandler に委譲 / 直接返却 | "移動情報の必須キーが不足しています。以下のキーをすべて含めてください: ..." / フィールド別メッセージ |
| EX-03（業務ルール違反）: 申請期限超過 | 移動日から本日まで90日超過 | ErrorHandler に委譲、フロー終了 | "申請期限（経費発生日から90日以内）を超過しています。担当部門にご確認ください。" |
| EX-04（外部I/F例外）: テンプレート不在 | os.path.exists() が False | logger.error後にsuccess=False | "申請書テンプレートが見つかりませんでした。担当部門（管理部）にお問い合わせください。" |
| EX-06（データ不整合）: ファイル保存失敗 | テンプレート破損・IOError・PermissionError・Exception | logger.error後にErrorHandler.handle_file_save_error(e)でsuccess=False | ErrorHandler.handle_file_save_error の返却メッセージ |

#### 4.3.1 処理フロー

```
開始: generate_expense_reimbursement_form(context, items) 呼び出し
  ↓
【ステップ0】invocation_state から申請者情報取得
  - applicant_name = context.invocation_state.get("applicant_name", "")
  - application_date = context.invocation_state.get("application_date", "")
  ↓
【ステップ1】必須キーガード（dict型パラメータ）
  - items が空リストでないことを確認（空の場合は EX-01エラーを返す）
  - 各 item dict に必須キー(store_name, amount, expense_date, item_name, expense_category, purpose) が存在するか確認
  - 不足キーがある場合: return {"success": False, "file_path": "", "message": "経費情報の必須キーが不足しています。以下のキーをすべて含めてください: store_name, amount, expense_date, item_name, expense_category, purpose"}
  ↓
【ステップ2】ExpenseFormInput バリデーション
  - ExpenseFormInput を生成（Pydantic v2 モデル）
  - items を ExpenseItem のリストとして検証（各フィールド型・制約・BRL-14/17/18）
  - ValidationError 発生時 → 【エラーフロー1: バリデーションエラー】へ
  ↓
【ステップ3】テンプレート読み込み
  - os.path.exists(EXPENSE_TEMPLATE_PATH) でファイル存在を確認
  - ファイルが存在しない場合 → 【エラーフロー2: テンプレート不在（EX-04）】へ
  - EXPENSE_TEMPLATE_PATH のファイルを openpyxl.load_workbook() で読み込む
  - ファイル破損 (InvalidFileException) → 【エラーフロー3: ファイル保存エラー（EX-06）】へ
  ↓
【ステップ4】フィールド書き込み
  - ws["B3"] = applicant_name（申請者名）
  - ws["B4"] = application_date（申請日）
  - 各 item（インデックス i, 0始まり）を行 (7 + i) に書き込む:
    - ws[f"A{7+i}"] = i + 1（No）
    - ws[f"B{7+i}"] = item.expense_date（購入日）
    - ws[f"C{7+i}"] = item.store_name（店舗名）
    - ws[f"D{7+i}"] = item.item_name（品目）
    - ws[f"E{7+i}"] = item.expense_category（経費区分）
    - ws[f"F{7+i}"] = item.amount（金額）
    - ws[f"G{7+i}"] = item.purpose（業務目的）
    - ws[f"H{7+i}"] = ""（承認状況: 空欄）
  - 書き込みエラー (Exception) → 【エラーフロー3: ファイル保存エラー（EX-06）】へ
  ↓
【ステップ5】出力ファイルパス生成
  - session_id = context.invocation_state.get("session_id", "unknown")
  - output_dir = f"data/output/{session_id}/"
  - os.makedirs(output_dir, exist_ok=True)
  - safe_name = re.sub(r'[\\/:*?"<>|]', '_', applicant_name)
  - file_path = f"{output_dir}{safe_name}_経費精算申請書_{datetime.now().strftime('%Y%m%d_%H%M%S')}.xlsx"
  ↓
【ステップ6】ファイル保存
  - try: wb.save(file_path)
  - except IOError as e: → 【エラーフロー3: ファイル保存エラー】へ
  - except PermissionError as e: → 【エラーフロー3: ファイル保存エラー】へ
  - except Exception as e: → 【エラーフロー3: ファイル保存エラー】へ
  ↓
【ステップ7】成功レスポンス生成
  - logger.info(f"[OPE-002] generate_expense_reimbursement_form 完了: file_path={file_path}")
  - return {"success": True, "file_path": file_path, "message": ""}
  ↓
終了
```

#### 4.3.3 データの検証方法（Pydantic モデル使用）

**Pydanticモデル**: `ExpenseFormInput`

**検証内容**:
- `applicant_name`: `strip_and_validate_not_empty` → `min_length=1`
- `application_date`: 文字列→`date`自動変換 → 未来日付禁止
- `items`: `min_length=1`（1件以上）
- 各 `ExpenseItem`:
  - `expense_date`: `check_application_deadline` (model_validator mode="after") → 90日超過チェック
  - `store_name`: `strip_and_validate_not_empty` → `min_length=1`
  - `item_name`: `strip_and_validate_not_empty` → `min_length=1`
  - `expense_category`: `normalize_expense_category` (mode="before") → `Literal["事務用品費", "宿泊費", "資格精算費", "その他経費"]`
  - `amount`: `ge=0`
  - `purpose`: `strip_and_validate_not_empty` → `min_length=1`

## 6. 補足情報

### 6.1 実装上の注意点

1. **元テンプレートの保護**
   - `openpyxl.load_workbook(template_path)` で読み込んだ後、別パスに `wb.save(output_path)` として保存する。元テンプレートファイルを上書きしない。

2. **出力ファイルパスはLLMから受け取らない**
   - ツール関数の引数に `file_path` を含めてはならない。ツール内部で `datetime.now().strftime("%Y%m%d_%H%M%S")` を使って生成する。

3. **output/ ディレクトリの自動作成**
   - `os.makedirs(f"data/output/{session_id}/", exist_ok=True)` でディレクトリが存在しない場合に自動作成する。session_id は invocation_state から取得する。

4. **HumanApprovalHook との連携**
   - 本ツール（`generate_transport_expense_form` / `generate_expense_reimbursement_form`）はHumanApprovalHook の `BeforeToolCallEvent` 対象。ツール呼び出し前に社員確認が必ず挟まれる設計。

5. **非冪等性への対応**
   - ファイル生成は副作用を持つため、リトライは1回のみ（実行制御方針 §6.1）。呼び出し元（AG-002/AG-003）がリトライ制御を行う。

---

### 6.2 パフォーマンス考慮事項

1. **テンプレートの読み込み**
   - 通常のExcelファイルサイズであれば、読み込みによる遅延は許容範囲内。

2. **output/ ディレクトリの肥大化**
   - `data/output/{session_id}/` 配下に生成ファイルが蓄積する。定期的なクリーンアップは運用で対応する（アーカイブポリシーは要件上未定義）。

---

### 6.3 セキュリティ考慮事項

1. **ファイルパスのハードコーディング禁止**
   - テンプレートパスは `config/settings.py` から取得する。

2. **申請者名のファイル名への埋め込み**
   - 申請者名にパス区切り文字（`/`, `\`, `..`）が含まれないことを確認する。ファイル名生成時に `re.sub(r'[\\/:*?"<>|]', '_', applicant_name)` でサニタイズする。

## 8. テスト観点

### 8.1 機能テスト
- 有効な移動情報（3区間）を渡して交通費申請書が生成され、`success=True`・`file_path`（存在するパス）が返されること
- 有効な経費情報（2明細）を渡して経費精算申請書が生成され、`success=True`・`file_path`（存在するパス）が返されること
- 生成された Excelファイルを開き、申請者名・申請日・各区間（明細）情報が正しいセルに書き込まれていること
- output/ディレクトリが存在しない状態でツールを呼び出した際に自動作成されてファイルが保存されること
- 申請者名に特殊文字（`/`, `\` 等）が含まれる場合にサニタイズされたファイル名で保存されること

### 8.2 異常系テスト
- テンプレートファイルが存在しない場合に `success=False` と EX-04 メッセージが返されること
- テンプレートファイルが破損している場合に `success=False` と EX-06 メッセージが返されること
- `segments=[]`（空リスト）の場合に `success=False` と EX-01 メッセージが返されること
- `segments` の各要素に必須キーが不足する場合に `success=False` と必須キー一覧のメッセージが返されること
- `items` の各要素に必須キーが不足する場合に `success=False` と必須キー一覧のメッセージが返されること
- 申請期限超過（91日前の移動日/経費発生日）で `success=False` と EX-03 メッセージが返されること
- 対応外経費区分（「不明」等）で `success=False` と EX-01 メッセージが返されること

### 8.3 性能テスト
- 20区間（明細）を持つデータで申請書生成が5秒以内に完了すること

### 8.4 境界値テスト
- `segments`（`items`）が1件の場合に正常に生成されること
- `segments`（`items`）が20件の場合に正常に生成されること（行オフセットの確認）
- `amount=0` の場合に正常に生成されること
