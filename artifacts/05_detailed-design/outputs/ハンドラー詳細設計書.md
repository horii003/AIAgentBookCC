## 1. 概要

### 1.1 コンポーネントの目的

`handlers/error_handler.py` に `ErrorHandler`・`HumanApprovalHook`・`LoopControlHook` の3クラスと `LoopLimitError` カスタム例外クラスを集約し、全エージェント（AG-001〜AG-003）と全ツール（TOOL-001・TOOL-002）の例外処理・人間承認制御・ループ制御を一元管理する。

### 1.2 主要な責務

- **ErrorHandler**: 例外種別に対応するユーザー向け日本語メッセージを生成して返す。ログ出力・セッション状態更新はErrorHandlerの責務外とし、呼び出し元モジュールが行う
- **HumanApprovalHook**: AG-002/AG-003のTOOL-002（generate_transport_application/generate_expense_application）呼び出し前にBeforeToolCallEventを受けて社員確認（OK/修正/キャンセル）を挟む。ツール中止は `event.cancel_tool` にメッセージをセットすることで行う
- **LoopControlHook**: 全エージェント（AG-001〜AG-003）のReActループ回数を監視し、上限（10回）到達時に `LoopLimitError` を発生させる
- **LoopLimitError**: LoopControlHookがループ上限到達時に発生させるカスタム例外クラス

### 1.3 非責務
- 例外検知（各エージェント・ツールの責務）
- セッション状態の更新（SessionManagerの責務）
- ガードレールの判定（Amazon Bedrockガードレールの責務）
- 申請書チェックロジック（AG-002/AG-003の責務）
- ファイル書き込み（TOOL-002a/TL-002bの責務）
- ログ出力（各エージェント・各ツールが ErrorHandler 呼び出し前に実施する）

## 2. LoopLimitError 定義

### 2.1 クラス基本情報

#### クラス名
`LoopLimitError`

#### 定義場所
`handlers/error_handler.py`（ErrorHandler・HumanApprovalHook・LoopControlHookと同ファイルに定義する）

#### 説明
LoopControlHookがReActループ上限に到達したときに発生させるカスタム例外クラス。`current_iteration`・`max_iterations`・`agent_name` の3フィールドを保持する。

### 2.2 定義

```python
class LoopLimitError(Exception):
    """ReActループ上限到達時のカスタム例外。"""

    def __init__(self, current_iteration: int, max_iterations: int, agent_name: str):
        self.current_iteration = current_iteration
        self.max_iterations = max_iterations
        self.agent_name = agent_name
        super().__init__(
            f"Loop limit reached: {current_iteration}/{max_iterations} iterations in agent '{agent_name}'"
        )
```

### 2.3 フィールド

| フィールド名 | 型 | 説明 |
|------------|-----|------|
| `current_iteration` | `int` | 上限到達時の現在のループ回数 |
| `max_iterations` | `int` | 設定されたループ上限回数 |
| `agent_name` | `str` | LoopLimitErrorが発生したエージェント名 |

### 2.4 使用例

```python
from handlers.error_handler import LoopLimitError

try:
    response = agent(query, invocation_state=state)
except LoopLimitError as e:
    logging.warning(
        f"LoopLimitError: {e.current_iteration}/{e.max_iterations} iterations, agent={e.agent_name}"
    )
    error_message = error_handler.handle_loop_limit_error(e)
    return error_message
```

## 3. ErrorHandler 設計詳細

### 3.1 クラス基本情報

#### クラス名
`ErrorHandler`

#### 説明

## 4. ErrorHandler ビジネスロジック

### 4.1 ValidationErrorの日本語変換

#### 処理フロー

```
開始（ValidationErrorを受け取る）
  ↓
e.errors() でPydanticエラーリストを取得する
  ↓
エラーが1件？

  ↓

```

## 5. ErrorHandler エラーハンドリング

### 5.1 処理されるエラー

| エラー種別 | 対応メソッド | メッセージ概要 |
|-----------|------------|-------------|
| ModelThrottledException | `handle_throttling_error` | APIレート制限の案内 |
| MaxTokensReachedException | `handle_max_tokens_error` | トークン上限到達の案内 |
| ContextWindowOverflowException | `handle_context_window_error` | 会話履歴上限の案内 |
| FileNotFoundError / Exception（運賃データ） | `handle_fare_data_error` | 運賃データ読み込み失敗の案内 |
| Exception（運賃計算） | `handle_calculation_error` | 運賃計算失敗の案内 |
| IOError / PermissionError / Exception（ファイル保存） | `handle_file_save_error` | ファイル保存失敗の案内 |
| ValidationError | `handle_validation_error` | フィールドごとのエラーメッセージ |
| KeyboardInterrupt | `handle_keyboard_interrupt` | 中断の案内 |
| LoopLimitError | `handle_loop_limit_error` | ループ上限到達の案内 |
| RuntimeError | `handle_runtime_error` | システムエラーの案内 |
| Exception（その他） | `handle_unexpected_error` | 予期しないエラーの案内 |

**注意**: 各メソッドはユーザー向けメッセージ文字列を返すのみ。ログ出力は行わない。呼び出し元が ErrorHandler 呼び出し前にログ出力を実施すること。

## 6. ErrorHandler ログ出力

ErrorHandlerクラス自体はログ出力を行わない。ログ出力は呼び出し元モジュール（各エージェント・各ツール）が ErrorHandler 呼び出し前に実施する。

**呼び出し元でのログ出力パターン（参考）**:

| レベル | タイミング | メッセージ例 |
|--------|-----------|-----------|
| WARNING | LoopLimitError発生時 | `"LoopLimitError: {e.current_iteration}/{e.max_iterations}, agent={e.agent_name}"` |
| WARNING | ContextWindowOverflowException発生時 | `"ContextWindowOverflowException: session_id={session_id}"` |
| WARNING | MaxTokensReachedException発生時 | `"MaxTokensReachedException: session_id={session_id}"` |
| ERROR | RuntimeError発生時 | `"RuntimeError: session_id={session_id}"` （exc_info=True）|
| ERROR | 予期しない例外発生時 | `"Unexpected error: session_id={session_id}"` （exc_info=True）|

## 7. ErrorHandler 使用例

### 7.1 基本的な使用方法

```python
from handlers.error_handler import ErrorHandler, LoopLimitError
from pydantic import ValidationError
import logging

logger = logging.getLogger(__name__)
_error_handler = ErrorHandler()

# ValidationError処理（ツール関数内）
try:
    data = FareCalculationInput(**params)
except ValidationError as e:
    logger.warning(f"ValidationError in calculate_transport_fare: {e}")
    error_message = _error_handler.handle_validation_error(e)
    return {"success": False, "fare": 0, "calculation_method": "", "message": error_message}
except Exception as e:
    logger.error(f"Unexpected error in calculate_transport_fare: {e}", exc_info=True)
    error_message = _error_handler.handle_unexpected_error(e)
    return {"success": False, "fare": 0, "calculation_method": "", "message": error_message}
```

### 7.2 LoopLimitError の処理（エージェント内）

```python
try:
    response = agent(query, invocation_state=state)
except LoopLimitError as e:
    logger.warning(
        f"LoopLimitError: {e.current_iteration}/{e.max_iterations} iterations, agent={e.agent_name}, query={query[:50]}"
    )
    error_message = _error_handler.handle_loop_limit_error(e)
    return error_message
except RuntimeError as e:
    logger.error(f"RuntimeError in agent: {e}, query={query[:50]}", exc_info=True)
    error_message = _error_handler.handle_runtime_error(e)
    return error_message
except Exception as e:
    logger.error(f"Unexpected error in agent: {e}, query={query[:50]}", exc_info=True)
    error_message = _error_handler.handle_unexpected_error(e)
    return error_message
```

## 8. HumanApprovalHook 設計詳細

### 8.1 クラス基本情報

#### クラス名
`HumanApprovalHook`

#### 説明
AG-002/AG-003がTOOL-002（generate_transport_application/generate_expense_application）を呼び出す前にBeforeToolCallEventを受けて社員確認（OK/修正/キャンセル）を挟む。HookProviderインタフェースを実装する。AG-002/AG-003のみに登録する（AG-001には登録しない）。承認フロー内部で `_approval_callback` を呼び出し、戻り値に応じて `event.cancel_tool` にメッセージをセットしてツールをキャンセルする。

---

### 8.2 初期化

#### `__init__(self)`

**インスタンス変数**:
- なし（ステートレスクラス。承認状態は各呼び出しで即座に処理する）

---

### 8.3 主要メソッド（HumanApprovalHook）

#### 8.3.1 register_hooks

##### 説明
HookRegistryにイベントハンドラーを登録する（HookProviderインタフェース実装）。

##### 引数
- `registry` (`HookRegistry`): Strands AgentsのHookRegistry
- `**kwargs` (`Any`): 追加パラメータ（未使用）

##### 戻り値
- `None`

##### 処理内容
1. `registry.add_callback(BeforeToolCallEvent, self._before_tool_call)` を呼び出す

---

### 8.4 フック設計（HumanApprovalHook）

#### 8.4.1 フック登録

##### `register_hooks(self, registry: HookRegistry, **kwargs: Any) -> None`

**登録するイベント**:
- `BeforeToolCallEvent`: `_before_tool_call`
  - **登録タイミング**: AG-002/AG-003のAgentインスタンス生成時（`hooks=[HumanApprovalHook(), LoopControlHook(...)]` の引数として渡す）
  - **依存関係**: LoopControlHookとの順序は問わない

**登録順序**: `hooks=[HumanApprovalHook(), LoopControlHook(max_iterations=10)]` の順序で登録する

---

#### 8.4.2 イベントハンドラー

##### _before_tool_call

**説明**: BeforeToolCallEventでツール名を確認し、承認対象ツール（申請書生成ツール詳細設計書に定義された全ツール関数名: `generate_transport_application`/`generate_expense_application`）の場合のみ `_approval_callback` を呼び出して社員確認を実施する。

**処理内容**:
1. `event.tool_name` でツール名を取得する
2. ツール名が `"generate_transport_application"` または `"generate_expense_application"` でない場合はスキップする（他のツールへの介入禁止）
3. `_approval_callback(tool_name=event.tool_name, tool_params=event.tool_params)` を呼び出す
4. 戻り値 `(approved: bool, message: str)` に応じて処理を分岐する:
   - `(True, "")` → そのまま処理を継続する（ツール実行へ進む）
   - `(False, "CANCEL")` → `event.cancel_tool = "申請をキャンセルしました。"` をセットしてツール実行を中断する
   - `(False, その他メッセージ)` → `event.cancel_tool = message` をセットしてツール実行を中断し、情報収集への戻りをエージェントに通知する

**ログ出力**:
- スキップ: `"HumanApprovalHook: skipped for tool={tool_name}"`
- OK: `"HumanApprovalHook: approved for tool={tool_name}"`
- 修正: `"HumanApprovalHook: modification requested for tool={tool_name}"`
- キャンセル: `"HumanApprovalHook: cancelled for tool={tool_name}"`

---

#### 8.4.3 承認コールバック

##### `_approval_callback`

**シグネチャ**:
```python
def _approval_callback(self, tool_name: str, tool_params: dict) -> tuple[bool, str]:
```

**引数**:
- `tool_name` (`str`): 呼び出し対象のツール名
- `tool_params` (`dict`): ツールに渡されるパラメータ

**戻り値**:
- `(True, "")` → OK（何もしない → ツール実行される）
- `(False, "修正内容")` → 修正要望（`event.cancel_tool` にメッセージをセット → ツールキャンセル）
- `(False, "CANCEL")` → キャンセル（`event.cancel_tool` にキャンセルメッセージをセット → ツールキャンセル）

**処理内容**:
1. CLIで以下の選択肢を表示する:
   ```
   申請書を生成してよろしいですか？
   1. OK（生成する）
   2. 修正（情報収集に戻る）
   3. キャンセル（申請を終了する）
   選択してください（1/2/3）:
   ```
2. ユーザーの入力を受け取る
3. 選択に応じて戻り値を返す:
   - `"1"` (OK) → `(True, "")` を返す
   - `"2"` (修正) → `(False, "修正します。もう一度情報を入力してください。")` を返す
   - `"3"` (キャンセル) → `(False, "CANCEL")` を返す
   - 無効な入力 → 再入力を促して再度選択肢を表示する（ループ）

## 9. HumanApprovalHook ビジネスロジック

### 9.1 承認フロー

#### 処理フロー

```
開始（BeforeToolCallEventを受け取る）
  ↓
event.tool_nameを取得する
  ↓
ツール名が "generate_transport_application" または "generate_expense_application"？
  NO → スキップして次のフックへ（return）
  YES → 次のステップへ
  ↓
_approval_callback(tool_name, tool_params) を呼び出す
  ↓
戻り値の判定
  (True, "")（OK）
    → logging.info("HumanApprovalHook: approved...")
    → return（ツール実行へ進む）
  (False, "CANCEL")（キャンセル）
    → logging.warning("HumanApprovalHook: cancelled...")
    → event.cancel_tool = "申請をキャンセルしました。"
    → return（ツール実行をキャンセル）
  (False, message)（修正要望）
    → logging.warning("HumanApprovalHook: modification requested...")
    → event.cancel_tool = message
    → return（ツール実行をキャンセルし、エージェントに修正メッセージを通知）
  ↓
終了
```

## 10. HumanApprovalHook エラーハンドリング

### 10.1 処理されるエラー

| エラー種別 | 発生条件 | 対応 | メッセージ |

## 13. LoopControlHook 設計詳細

### 13.1 クラス基本情報

#### クラス名
`LoopControlHook`

#### 説明
全エージェント（AG-001〜AG-003）のReActループ回数を監視し、上限（max_iterations=10）に到達した場合に `LoopLimitError` を発生させる。HookProviderインタフェースを実装する。AfterModelCallEventでカウントするが、`event.exception` が存在する場合はスキップする。BeforeInvocationEventでカウンタをリセットし、AfterInvocationEventでは合計ループ回数をINFOログに出力する（カウンタはリセットしない）。

---

### 13.2 初期化

#### `__init__(self, max_iterations: int = 10, agent_name: str = "")`

**引数**:
- `max_iterations` (`int`): ReActループの最大繰り返し回数（デフォルト: `10`）
- `agent_name` (`str`): エージェント名（LoopLimitErrorのフィールドに使用。デフォルト: `""`）

**インスタンス変数**:
- `max_iterations` (`int`): ReActループの最大繰り返し回数
- `agent_name` (`str`): エージェント名
- `iteration_count` (`int`): 現在のループカウンタ（初期値: `0`）

---

### 13.3 主要メソッド（LoopControlHook）

#### 13.3.1 register_hooks

##### 説明
HookRegistryにイベントハンドラーを登録する（HookProviderインタフェース実装）。

##### 引数
- `registry` (`HookRegistry`): Strands AgentsのHookRegistry
- `**kwargs` (`Any`): 追加パラメータ（未使用）

##### 戻り値
- `None`

##### 処理内容
1. `registry.add_callback(BeforeInvocationEvent, self._before_invocation)` を呼び出す
2. `registry.add_callback(BeforeModelCallEvent, self._before_model_call)` を呼び出す
3. `registry.add_callback(AfterModelCallEvent, self._after_model_call)` を呼び出す
4. `registry.add_callback(AfterInvocationEvent, self._after_invocation)` を呼び出す
5. `registry.add_callback(BeforeToolCallEvent, self._before_tool_call_log)` を呼び出す
6. `registry.add_callback(AfterToolCallEvent, self._after_tool_call_log)` を呼び出す

---

### 13.4 フック設計（LoopControlHook）

#### 13.4.1 フック登録

**登録するイベント**:
- `BeforeInvocationEvent`: `_before_invocation`（ループカウンタのリセット）
- `BeforeModelCallEvent`: `_before_model_call`（ループ回数のINFOログ出力）
- `AfterModelCallEvent`: `_after_model_call`（ループカウンタのインクリメント・上限監視）
- `AfterInvocationEvent`: `_after_invocation`（合計ループ回数のINFOログ出力）
- `BeforeToolCallEvent`: `_before_tool_call_log`（ツール名のINFOログ出力）
- `AfterToolCallEvent`: `_after_tool_call_log`（ツール名のINFOログ出力）

---

#### 13.4.2 イベントハンドラー

##### _before_invocation

**説明**: エージェント呼び出し開始時にループカウンタを0にリセットする。

**処理内容**:
1. `self.iteration_count = 0` でカウンタをリセットする
2. `logging.debug(f"LoopControlHook: iteration_count reset to 0")` でログを記録する

**ログ出力**: `"LoopControlHook: iteration_count reset to 0"`

---

##### _before_model_call

**説明**: LLM呼び出し前にループ回数をINFOレベルでログ出力する。

**処理内容**:
1. `logging.info(f"LoopControlHook: starting iteration {self.iteration_count + 1}")` でログを記録する

**ログ出力**: `"LoopControlHook: starting iteration {n}"`

---

##### _after_model_call

**説明**: LLM呼び出し後にループカウンタをインクリメントし、上限到達を監視する。`event.exception` が存在する場合はカウントをスキップする。

**処理内容**:
1. `event.exception` が存在する場合はカウントをスキップして return する
2. `self.iteration_count += 1` でカウンタをインクリメントする
3. `logging.debug(f"LoopControlHook: iteration_count={self.iteration_count}")` でログを記録する
4. `self.iteration_count >= self.max_iterations` の場合:
   - `logging.warning(f"LoopControlHook: max iterations reached ({self.max_iterations}) in agent '{self.agent_name}'")` でWARNINGログを記録する
   - `raise LoopLimitError(current_iteration=self.iteration_count, max_iterations=self.max_iterations, agent_name=self.agent_name)` を発生させる

**ログ出力**:
- 通常: `"LoopControlHook: iteration_count={n}"`（DEBUGレベル）
- 上限到達: `"LoopControlHook: max iterations reached ({max_iterations}) in agent '{agent_name}'"` （WARNINGレベル）

---

##### _after_invocation

**説明**: エージェント呼び出し完了後に合計ループ回数をINFOレベルでログ出力する。カウンタのリセットは行わない。

**処理内容**:
1. `logging.info(f"LoopControlHook: invocation completed, total iterations={self.iteration_count}")` でログを記録する

**ログ出力**: `"LoopControlHook: invocation completed, total iterations={n}"`

---

##### _before_tool_call_log

**説明**: ツール呼び出し前にツール名をINFOレベルでログ出力する。

**処理内容**:
1. `logging.info(f"LoopControlHook: tool_call tool={event.tool_name}")` でログを記録する

**ログ出力**: `"LoopControlHook: tool_call tool={tool_name}"`

---

##### _after_tool_call_log

**説明**: ツール呼び出し後にツール名をINFOレベルでログ出力する。

**処理内容**:
1. `logging.info(f"LoopControlHook: tool_call_completed tool={event.tool_name}")` でログを記録する

**ログ出力**: `"LoopControlHook: tool_call_completed tool={tool_name}"`

---

## 14. LoopControlHook ビジネスロジック

### 14.1 ループ監視

#### 処理フロー

```
エージェント呼び出し開始（BeforeInvocationEvent）
  ↓
iteration_count = 0（リセット）
  ↓
[LLM呼び出し前（BeforeModelCallEvent）]
  ↓
logging.info("LoopControlHook: starting iteration {iteration_count + 1}")
  ↓
[LLM呼び出し（AfterModelCallEvent）]
  ↓
event.exception が存在する？
  YES → カウントをスキップして次のイベントへ
  NO  → iteration_count += 1
  ↓
iteration_count >= max_iterations（10）？
  YES → logging.warning("LoopControlHook: max iterations reached...")
         raise LoopLimitError(current_iteration, max_iterations, agent_name)
         → 呼び出し元エージェントがLoopLimitErrorをキャッチしてエラーメッセージを返す
  NO  → LLM呼び出しループを継続
  ↓
[ツール呼び出し前（BeforeToolCallEvent）]
  ↓
logging.info("LoopControlHook: tool_call tool={tool_name}")
  ↓
[ツール呼び出し後（AfterToolCallEvent）]
  ↓
logging.info("LoopControlHook: tool_call_completed tool={tool_name}")
  ↓
エージェント呼び出し完了（AfterInvocationEvent）
  ↓
logging.info("LoopControlHook: invocation completed, total iterations={iteration_count}")
（カウンタリセットは行わない）
  ↓
終了
```

## 17. LoopControlHook 使用例

### 17.1 基本的な使用方法

```python
from handlers.error_handler import LoopControlHook, LoopLimitError, ErrorHandler
from strands import Agent
import logging

logger = logging.getLogger(__name__)
_error_handler = ErrorHandler()

# 全エージェント（AG-001〜AG-003）でLoopControlHookを登録する
loop_control_hook = LoopControlHook(max_iterations=10, agent_name="orchestrator_agent")

agent = Agent(
    model=get_bedrock_model(),
    system_prompt=system_prompt,
    tools=tools,
    hooks=[loop_control_hook],   # AG-001はLoopControlHookのみ
    callback_handler=None,
)

# エージェント呼び出し時のLoopLimitError処理
try:
    response = agent(query, invocation_state=state)
except LoopLimitError as e:
    # 呼び出し元がWARNINGでログを記録する（ErrorHandlerはログ出力しない）
    logger.warning(
        f"LoopLimitError: {e.current_iteration}/{e.max_iterations}, agent={e.agent_name}, session_id={session_id}"
    )
    error_message = _error_handler.handle_loop_limit_error(e)
    return error_message
except RuntimeError as e:
    logger.error(f"RuntimeError in agent: {e}, session_id={session_id}", exc_info=True)
    error_message = _error_handler.handle_runtime_error(e)
    return error_message
except Exception as e:
    logger.error(f"Unexpected error in agent: {e}, session_id={session_id}", exc_info=True)
    error_message = _error_handler.handle_unexpected_error(e)
    return error_message
```

## 18. 補足情報

### 18.1 実装上の注意点

1. **4クラスを1ファイルに集約する**
   - `handlers/error_handler.py` に `LoopLimitError`・`ErrorHandler`・`HumanApprovalHook`・`LoopControlHook` を集約する

2. **HumanApprovalHookはAG-002/AG-003のみに登録する**
   - AG-001にはHumanApprovalHookを登録しない（AG-001はTOOL-002を呼び出さないため）

3. **LoopControlHookのループカウントはexception時はスキップする**
   - `event.exception` が存在する場合（LLM呼び出し自体がエラーになった場合）はカウントをスキップする
   - 正常なLLM応答後にのみカウントアップすることで、エラーループで誤ってLoopLimitErrorが発生しないようにする

4. **LoopControlHookのAfterInvocationEventはカウンタをリセットしない**
   - 合計ループ回数のログ出力のみ行う。BeforeInvocationEventでリセットするため、AfterInvocationEventでのリセットは不要

5. **HumanApprovalHookは event.cancel_tool を使用する**
   - ツール実行のキャンセルは `StopIteration` ではなく `event.cancel_tool` にメッセージ文字列をセットすることで行う

6. **LoopControlHookはインスタンスごとにカウンタを管理する**
   - AG-001・AG-002・AG-003がそれぞれ独立したLoopControlHookインスタンスを保持するため、ループカウンタが相互に干渉しない

7. **LoopLimitErrorは定義済みの全フィールドを引数として渡すこと**
   - `raise LoopLimitError(current_iteration=..., max_iterations=..., agent_name=...)` の形式で全フィールドを指定する

### 18.2 パフォーマンス考慮事項

1. **ErrorHandlerはステートレスで同期処理**
   - インスタンス変数を持たないため、スレッドセーフで軽量な実装が可能

2. **HumanApprovalHookのCLI入力はブロッキング**
   - 社員の入力を待つためのブロッキング処理。非同期処理には対応しない（本システムは同期処理のため問題なし）

### 18.3 セキュリティ考慮事項

1. **スタックトレースはログにのみ記録する**
   - 各エージェント・ツールが `logging.error(..., exc_info=True)` でスタックトレースをログに記録するが、ユーザーには返さない

2. **エラーメッセージに内部情報を含めない**
   - ユーザー向けメッセージにはシステム内部の詳細（ファイルパス・クラス名・エラーコード等）を含めない
