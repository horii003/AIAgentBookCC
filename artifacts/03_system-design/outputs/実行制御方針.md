## 1. 目的・適用範囲

### 1.1 目的

システム全体として **処理の開始・継続・中断・停止・再開** を一貫したルールで制御するための方針を定義する

### 1.2 適用範囲

- マルチエージェント処理（AG-001/AG-002/AG-003）
- 外部I/F実行（Amazon Bedrock LLM、ファイルシステム）
- 人間承認を含む業務処理（APR-001）

### 1.3 本方針で定義しないこと

- 個々のエージェントの内部制御ロジック
- SDK固有のキュー／スレッド制御
- 実装コードレベルの設定値（共通設定方針参照）

---

## 2. 実行制御の基本原則

- **制御はシステムが行う（エージェント任せにしない）**
- **処理は必ず状態を持つ**
- **再実行可能性を前提にする**
- **制御不能な自律動作をさせない**
- **人が介入できる余地を残す**


---

## 3. 実行単位の定義

### 3.1 実行単位の種類

| 単位 | 説明 | 識別子 |
|-----|------|-------|
| セッション | 1 件の申請書作成フローまたは申請ルール照会に対応する最上位の実行単位 | session_id |
| エージェント呼び出し | AG-001/AG-002/AG-003 の 1 回の invoke 呼び出し。Agent as Tools パターンでの委譲も 1 呼び出し | invocation_id |
| ループ | エージェント内の 1 回の ReActループ（1 ステップの LLM 推論 + ツール呼び出し） | iteration（0 から連番） |
| ツール実行 | TOOL-001/TOOL-002/image_reader などの 1 回のツール呼び出し | tool_name + execution_id |


### 3.2 実行単位の識別子

**session_id**:
- 形式: `{YYYYMMDD}_{HHMMSS}_{UUID4前8桁}`
- 例: `20260501_143022_a1b2c3d4`
- 生成: AG-001 の初期化時にシステムが生成し、FileBasedSessionManager が管理する

## 4. 実行状態管理

### 4.1 実行状態定義

| 状態 | 説明 | 適用対象 |
|-----|------|---------|
| READY | 実行待ち（前提条件チェック完了） | セッション・エージェント呼び出し |
| RUNNING | 実行中（LLM 推論・ツール実行中） | セッション・エージェント呼び出し・ループ・ツール実行 |
| WAITING | ユーザー入力待ち・APR-001 承認待ち | セッション・エージェント呼び出し |
| COMPLETED | 正常終了 | セッション・エージェント呼び出し・ループ・ツール実行 |
| FAILED | 例外発生による失敗（再試行可能） | エージェント呼び出し・ツール実行 |
| TERMINATED | 強制終了（例外処理方針の停止条件に該当） | セッション・エージェント呼び出し |


### 4.2 状態遷移の責務

**状態管理主体**: `FileBasedSessionManager`（セッション状態）、`LoopControlHook`（ループ状態）、`ErrorHandler`（例外時の状態遷移）

**エージェントの役割**: 実行開始・完了・例外を ErrorHandler および FileBasedSessionManager に通知し、状態遷移のトリガーを提供する

---

## 5. 実行開始制御

### 5.1 実行開始条件

**前提条件充足**:
- AG-001 へのユーザー入力が存在すること
- セッション状態が READY または WAITING（再開時）であること
- 必要なナレッジファイル・運賃データ・テンプレートファイルがファイルストアに存在すること

**権限確認完了**:
- エージェント種別に対応するロール（R-AG-001/R-AG-002/R-AG-003）の権限を確認する
- TOOL-002 実行前に APR-001 承認済みフラグが session_{id}.json に記録されていること（ABAC-001）

**ガードレールチェック通過**:
- 入力文字数が 500 文字以内であること（FR-012）
- Amazon Bedrock ガードレール（GUARDRAIL_ID）が有効な場合は実行前チェックを通過すること


### 5.2 実行抑止条件

**同一セッションでの重複実行**:
- 同一 session_id のセッションが RUNNING 状態の場合は新規実行を抑止する（FileBasedSessionManager が状態を確認）

**排他対象リソース使用中**:
- session_{id}.json への書き込み中に同一ファイルへの同時書き込みを禁止する
- TOOL-002（申請書生成）の APR-001 承認前実行を禁止する（ABAC-001）

**システム負荷超過**:
- 要件上未定義


---

## 6. 並列・逐次制御方針

### 6.1 並列実行可否の判断軸

**データ競合有無**:
- session_{id}.json の読み書き競合が生じるため、同一セッション内の並列実行は禁止する

**外部I/F制約**:
- Amazon Bedrock API のレート制限により並列呼び出しはリスクがある

**業務整合性**:
- 申請書作成フローは逐次処理（情報収集 → APR-001 → 申請書生成 → チェック）が必須であり、ステップ間の並列実行は不可

### 6.2 排他制御対象

| 対象 | 排他理由 | 実装方法 |
|-----|---------|---------|
| session_{id}.json | 同一セッションでの同時読み書き競合防止 | FileBasedSessionManager が排他制御する（詳細は基本設計で定義） |
| TOOL-002（申請書生成） | APR-001 承認前実行の防止 | HumanApprovalHook が APR-001 承認済みフラグを確認してから TOOL-002 を実行させる |
| AG-002 / AG-003 の同時起動 | 同一セッション内での複数申請種別の同時処理防止 | AG-001 はいずれか 1 つのみに委譲し、同時委譲しない |


---

## 7. 再試行（Retry）方針

### 7.1 再試行対象

| 対象 | 再試行可否 | 最大回数 | 理由 |
|-----|----------|---------|------|
| Amazon Bedrock API 呼び出し（LLM 推論・OCR） | ○ | 6 回 | 一時的なAPIエラー・レート制限はリトライで回復可能。指数バックオフ（初期4秒、最大240秒） |
| TOOL-001（交通費計算） | ○ | 要件上未定義 | ファイル参照が冪等なためリトライ可能 |
| TOOL-002（申請書生成） | △ | 1 回のみ（APR-001 再取得後） | APR-001 承認を再度取得した場合のみリトライ可能。承認未取得状態での再試行は禁止 |
| ファイルストアへのログ書き込み | × | — | 書き込み失敗時は要件上未定義。TERMINATED に遷移 |

### 7.2 再試行制約

**最大回数**: 6 回（Bedrock API、指数バックオフ）

**クールダウン**: 初期遅延 4 秒、指数バックオフ、最大遅延 240 秒

**冪等性前提**: TOOL-001/Bedrock API は冪等（同一入力で同一結果）。TOOL-002 は APR-001 承認後のみ実行可であり、再実行前に APR-001 承認を再取得する必要がある

---

## 8. 中断・停止制御

### 8.1 中断（Pause）

**中断可能条件**:
- WAITING 状態（ユーザー入力待ち・APR-001 承認待ち）（セッション管理方針のライフサイクル参照）

**中断時の状態保持**:
- セッション状態・業務コンテキストは session_{id}.json に保存する（FileBasedSessionManager）
- 会話履歴は SlidingWindowConversationManager のインメモリに保持（再開時の復元は要件上未定義）

### 8.2 停止（Terminate）

**停止条件**:
- 例外処理方針に定義する EX-05/EX-06/EX-07/EX-08 が発生し解決不可能（ErrorHandler 判定）
- Bedrock API リトライ上限（6回）到達（EX-04 → TERMINATED）
- ループ制限（10回）到達後も処理完了不可（EX-02）

**停止後の復旧可否**:
- EX-01/EX-02/EX-03（ユーザー起因）: セッション再開可（WAITING → ACTIVE）
- EX-04 リトライ失敗: セッション再開不可（TERMINATED）
- EX-05/EX-06/EX-07/EX-08: セッション再開不可（TERMINATED）

---

## 9. タイムアウト制御

### 9.1 タイムアウト対象

| 対象 | タイムアウト時間 | 実装状況 |
|-----|---------------|---------|
| Amazon Bedrock API 呼び出し | 要件上未定義 | 要件上未定義 |
| TOOL-001（交通費計算）実行 | 要件上未定義 | 要件上未定義 |
| TOOL-002（申請書生成）実行 | 要件上未定義 | 要件上未定義 |
| ユーザー入力待ち（WAITING 状態） | 要件上未定義 | 要件上未定義 |
| APR-001 承認待ち | 要件上未定義 | 要件上未定義 |

### 9.2 タイムアウト時の扱い

**例外分類**: EX-04（外部I/F例外）または EX-07（システム障害）

**再試行 or エスカレーション**: Bedrock API タイムアウトはリトライ（最大6回）。リトライ上限到達後は管理部門案内（CF-009）してセッションを TERMINATED に遷移

---

## 10. ループ制御

### 10.1 ループ制御の目的

- ReActループの無限継続（暴走）を防ぐ
- 情報収集の過剰な繰り返しによる UX 低下を防ぐ
- システムコスト（API コール数）の上限を設ける

### 10.2 ループ制御の仕組み

**最大ループ回数**: 10 回（全エージェント共通）

**実装クラス**: `LoopControlHook`（HookProvider として実装）

**制御方式**: `LoopControlHook` を全エージェントに設定する。フックがループカウントを監視し、上限到達時に例外を発生させる

**登録イベント**:
- `BeforeInvocationEvent`（エージェント呼び出し開始時にループカウンタを 0 にリセット）
- `AfterModelCallEvent`（LLM 呼び出し後にループカウンタをインクリメントし上限到達を監視）

**カウント対象**: エージェントの 1 回の ReActループ（1 ステップの LLM 推論 + ツール呼び出し）を 1 カウントとする

### 10.3 ループ制限到達時の処理

**処理**: `LoopControlHook` が EventLoopException を発生させ、ErrorHandler が EX-02（判断不能）として処理する

**ユーザーへの通知**: 「処理が完了できませんでした。管理部門（経理部）にお問い合わせいただくか、再度最初からお試しください。」を提示する

**セッション状態**: TERMINATED に遷移

---

## 11. 人間承認制御

### 11.1 承認制御の目的

- 申請書生成（TOOL-002 実行）前に社員が申請内容を確認する機会を設ける（APR-001）
- 意図しない申請書生成を防ぐ
- 承認操作を強化監査ログ（LOG-HI）に記録しガバナンス要件を満たす

### 11.2 承認制御の仕組み

**発火タイミング**: AG-002/AG-003 が全申請情報の収集完了を確認し、TOOL-002 を呼び出す直前

**承認選択肢**:
1. OK（承認）（申請書の内容で申請書を生成する）
2. 修正（特定の項目を修正して再度確認する）
3. キャンセル（申請書の生成を中止する）

### 11.3 承認結果の処理

| 選択肢 | 処理内容 |
|-------|---------|
| OK（承認） | APR-001 承認済みフラグを session_{id}.json に記録し、TOOL-002 を実行する。承認者・承認日時・申請情報を LOG-HI に記録する |
| 修正 | 修正対象の情報を再収集し、APR-001 を再度実施する |
| キャンセル | TOOL-002 実行をキャンセルし、セッションを TERMINATED に遷移させる |

## 12. エスカレーション連携

**実行不能状態の扱い**:
- ErrorHandler が継続不可と判断した場合、ユーザーに「管理部門（経理部）にお問い合わせください」を提示する（CF-009）

**実行中断 → エスカレーションの流れ**:
1. ErrorHandler が例外を分類し継続不可と判定する
2. セッション状態を TERMINATED に遷移させる
3. 標準監査ログ（audit_log_std.jsonl）に例外分類・セッション ID を記録する
4. ユーザーにエスカレーション案内メッセージを提示する

**実行再開条件**:
- TERMINATED に遷移したセッションは再開不可。新規セッションを開始する（セッション管理方針参照）

---

## 13. セッション管理との連携

**実行状態とセッション状態の関係**:
- 実行状態（RUNNING/WAITING/COMPLETED/FAILED/TERMINATED）はセッション状態（ACTIVE/WAITING/CLOSED/TERMINATED）と対応する。FileBasedSessionManager が session_{id}.json の status フィールドを更新する

**再開時の制御ポイント**:
- session_{id}.json の status が ACTIVE または WAITING の場合のみ処理再開を許可する
- 業務コンテキスト（申請者名・申請日・収集済み申請情報）を session_{id}.json から復元してから処理を継続する

---

## 14. ガードレール連携

**実行前チェック**:
- 入力文字数チェック（500文字上限、FR-012）を AG-001 の入力受付時に実施する
- Amazon Bedrock ガードレール（GUARDRAIL_ID/GUARDRAIL_VERSION 環境変数で設定）が有効な場合は各エージェントの LLM 呼び出し前にコンテンツポリシーチェックを実施する

**実行中監視**:
- `LoopControlHook` が ReActループ回数を監視し、上限（10回）到達時に実行を停止する
- Bedrock リトライ回数（最大6回）を Strands SDK が監視し、上限到達時に例外を発生させる

**実行後検証**:
- 申請書生成完了後に EVT-001/EVT-002（申請内容チェック、FR-010）を自動実行し、業務ルール適合を検証する

## 15. ログ・監査方針

### 15.1 実行制御ログ

| 項目 | 内容 | 出力先 |
|-----|------|-------|
| event_type | イベントタイプ（session_start/loop_iteration/tool_exec/approval/session_end 等） | 標準監査ログ（audit_log_std.jsonl） |
| agent_name | エージェント名（AG-001/AG-002/AG-003） | 標準監査ログ（audit_log_std.jsonl） |
| iteration | ループ回数（0 から連番） | 標準監査ログ（audit_log_std.jsonl） |
| tool_name | ツール名（TOOL-001/TOOL-002/image_reader 等） | 標準監査ログ（audit_log_std.jsonl） |
| timestamp | タイムスタンプ（ISO8601） | 標準監査ログ（audit_log_std.jsonl） |

### 15.2 監査観点

**実行判断の説明可能性**:
- ループカウント・ツール実行・例外分類・エスカレーション判定を標準監査ログに記録し、後から実行経路を説明可能にする

**人間介入の記録**:
- APR-001 の承認者・承認日時・対象申請情報を強化監査ログ（audit_log_hi.jsonl）に記録する

## 16. 設計上の留意点・リスク

**過剰制御による性能劣化**:
- ガードレールチェックやループ制御のオーバーヘッドが性能に影響するリスクがある
- 対策: 入力文字数チェックは軽量なため実装コストは低い。Bedrock ガードレールはオプション（GUARDRAIL_ID 未設定時は無効）

**制御不足による暴走**:
- LoopControlHook が設定されない場合にループが無限継続するリスクがある
- 対策: 全エージェント初期化時に LoopControlHook を必須設定とし、設定漏れをコードレビューで確認する

**再試行ループ**:
- Bedrock API リトライが上限（6回）まで続いた場合に応答時間が最大 4×(2^6-1)=252 秒（約4分）かかるリスクがある
- 対策: 最大遅延を 240 秒に制限する。ユーザーへの「処理中」メッセージを定期的に出力することを検討する（詳細は基本設計フェーズで定義）
