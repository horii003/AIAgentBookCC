## 1. 目的・適用範囲

### 1.1 目的

システム全体で共通する設定方針を定義し、各エージェントとコンポーネントで一貫した設定を適用する。

### 1.2 適用範囲

- AG-001（申請受付窓口エージェント）

- AG-003（経費精算申請エージェント）

- 会話管理の方針

- 個々のエージェントの具体的な設定値

- SDK固有の設定パラメータ

## 2. 会話管理の方針

### 2.1 基本原則

- **会話履歴の保持**: 直近ウィンドウサイズ分のメッセージを保持し、エージェントがコンテキストを維持しながら申請フローを進行できるようにする
- **メモリ効率**: `should_truncate_results=True`・`per_turn=False` を設定し、ウィンドウサイズ超過時はメッセージ単位で古い履歴を切り詰める
- **コンテキストの継続性**: 業務コンテキスト（申請種別・収集済み申請情報）はセッション状態ファイルに永続化し、会話履歴が切り詰められてもフローを継続できるようにする

### 2.2 ウィンドウサイズの基準

| エージェント種別 | ウィンドウサイズ | 理由 |
|----------------|----------------|------|
| オーケストレーター（AG-001） | 30（大） | 複数の専門エージェントとのやり取りを保持する必要があるため |
| 交通費精算申請エージェント（AG-002） | 20（中） | 交通費は複数区間の収集・運賃計算・申請書作成・チェックの多ステップが発生するため |
| 経費精算申請エージェント（AG-003） | 15（小） | 経費は単一明細が多く、より少ないステップで完結するため |
| ツール（TOOL-001/TOOL-002） | 適用なし | ツール関数はステートレスのため会話管理は不要 |

### 2.3 会話管理の実装

**使用するクラス**: `SlidingWindowConversationManager`（Strands Agents提供）

**設定方法**:
```python
from strands.agent.conversation_manager import SlidingWindowConversationManager

conversation_management = SlidingWindowConversationManager(
    window_size=30,              # エージェント種別に応じた値を設定する
    should_truncate_results=True,
    per_turn=False
)
```

**設定パラメータ**:
- `window_size`: LLMに渡す会話履歴の保持メッセージ件数。エージェント種別ごとに基準値が異なる（上記2.2参照）
- `should_truncate_results`: ウィンドウ超過時にツール結果を切り詰める（True固定）
- `per_turn`: カウント単位（Falseでメッセージ単位。True にするとターン単位）

## 3. リトライ戦略の方針

### 3.1 基本原則

- **一時的なエラーへの対応**: LLM APIの一時的な障害・レート制限エラーにはリトライで対応する
- **指数バックオフ**: リトライ間隔を指数的に増加させることで、APIへの過負荷を防止する
- **最大試行回数の制限**: 無限リトライを防止するため最大試行回数を設定する

### 3.2 リトライ対象

| 対象 | リトライ可否 | 理由 |
|-----|----------|------|
| Amazon Bedrock LLM API（EXT-001） | ○ | 一時的なAPI障害・レート制限はリトライで解消できる場合がある。最大6回・初期4秒・最大240秒のバックオフ |
| TOOL-002 ファイル書き込み（EXT-002） | ○ | 一時的なファイルシステムエラーはリトライで解消できる場合がある。最大2回 |
| TOOL-001 JSONファイル読み込み | × | データ不整合（EX-06）のため再試行しても解消しない。手動入力促しの代替処理を実施する |
| 業務ルール違反（申請期限超過等） | × | 再試行しても解消しないため即座に通知・停止する |

### 3.3 リトライ戦略の実装

**使用するクラス**: Strands Agentsのデフォルト設定を使用（RetryConfigに準拠）

**設定方法**: Strands Agentsがモデル呼び出しに対して自動的に指数バックオフリトライを処理する。明示的な設定は `config/model_config.py` で管理する

## 4. ループ制御の方針

### 4.1 基本原則

- **無限ループの防止**: ReActループの最大回数を設定し、エージェントが停止条件を満たさないまま動作し続けることを防止する
- **処理の可視化**: ループ回数をログに記録し、処理の進捗と上限到達を追跡可能にする
- **ユーザーへの通知**: ループ上限到達時はユーザーにエスカレーション案内を提示する

### 4.2 最大ループ回数の基準

| エージェント種別 | 最大ループ回数 | 理由 |
|----------------|--------------|------|
| 全エージェント（AG-001〜AG-003） | 10回 | 申請フローの各ステップはツール呼び出し1〜2回程度で完結するため、10回あれば十分であり、無限ループのリスクも低い |

### 4.3 ループ制御の実装

**使用するクラス**: `LoopControlHook`（`HookProvider` インタフェースを実装したカスタムクラス）

**設定方法**:
```python
from handlers.error_handler import LoopControlHook

loop_control_hook = LoopControlHook(max_iterations=10)
```

**設定パラメータ**:
- `max_iterations`: ReActループの最大繰り返し回数。全エージェント共通で10回を基準値とする

**登録イベント**:
- `BeforeInvocationEvent`: エージェント呼び出し開始時にループカウンタを0にリセットする
- `AfterModelCallEvent`: LLM呼び出し後にループカウンタをインクリメントし、上限（`max_iterations`）に到達していないか監視する
- `AfterInvocationEvent`: エージェント呼び出し完了後にループカウンタを0にリセットする

> ※ LoopControlHookの具体的な実装設計は基本設計（ハンドラー基本設計.md）で定義する

## 5. セッション管理の方針

### 5.1 基本原則

- **会話履歴の永続化**: セッション状態をJSONファイル（`data/sessions/<session_id>.json`）に保存し、揮発性ストレージは使用しない
- **セッションIDの一意性**: SessionManagerがセッション開始時にUUID v4またはタイムスタンプベースの一意なIDを生成する
- **エージェント間の共有**: session_idはAG-001がSessionManagerで生成し、`invocation_state` 経由でAG-002/AG-003へ伝播する

### 5.2 セッションIDの形式

**形式**: `sess_YYYYMMDD_<ランダム8文字>` またはUUID v4

**例**: `sess_20260506_a1b2c3d4`

### 5.3 セッション管理の実装

**使用するクラス**: `SessionManager`（カスタム実装）

**設定方法**:
```python
from session.session_manager import SessionManager

session_manager = SessionManager(
    session_id=session_id,
    storage_path="data/sessions/"
)
```

**詳細**: セッション管理方針（`artifacts/03_system-design/outputs/セッション管理方針.md`）を参照

## 6. 承認フックの方針

### 6.1 基本原則

- **重要な操作の承認**: 申請書生成（TOOL-002呼び出し前）・ファイル出力（申請書チェック合格後）は社員確認を必須とする（GRD-010/COND-003/COND-004）
- **ユーザーの選択肢**: OK（承認）・修正・キャンセル（申請フロー終了）の3択を提示し、社員が次のアクションを選択できるようにする
- **誤操作の防止**: GRD-011（申請書提出はエージェントが実行しない）を徹底し、最終提出は社員のみが実施できるようにする

### 6.2 承認対象ツールの基準

| ツール種別 | 承認要否 | 理由 |
|----------|---------|------|
| TOOL-002（申請書生成・ファイル出力） | 要 | 自律度Lv3のアクション（ACT-GEN-01/ACT-EXEC-01）。社員確認後のみ実行（COND-003/COND-004） |
| TOOL-001（交通費計算） | 不要 | 自律度Lv5の自動実行（ACT-GEN-02）。参照のみで副作用がない |
| LLM推論（EXT-001） | 不要 | Strands Agentsのフレームワーク組み込み機能。独立した承認制御は不要 |

### 6.3 承認フックの実装

**使用するクラス**: `HumanApprovalHook`（カスタム実装）

**設定方法**:
```python
from handlers.error_handler import HumanApprovalHook

human_approval_hook = HumanApprovalHook()
```

**設定パラメータ**:
- 承認対象ツールをツール関数内の `@tool(context=True)` デコレータと組み合わせて制御する

> ※ HumanApprovalHookの具体的な実装設計は基本設計（ハンドラー基本設計.md）で定義する

## 7. 設定の適用方法

### 7.1 エージェント作成時の設定

**オーケストレーター（AG-001）の基本パターン**:
```python
from strands import Agent
from strands.agent.conversation_manager import SlidingWindowConversationManager
from handlers.error_handler import LoopControlHook
from session.session_manager import SessionManager
from config.model_config import get_bedrock_model

agent = Agent(
    model=get_bedrock_model(),
    system_prompt=ORCHESTRATOR_SYSTEM_PROMPT,
    tools=[transport_agent_tool, expense_agent_tool],
    agent_id="orchestrator_agent",
    name="受付窓口エージェント",
    conversation_manager=SlidingWindowConversationManager(
        window_size=30,
        should_truncate_results=True,
        per_turn=False
    ),
    hooks=[LoopControlHook(max_iterations=10)],
    session_manager=self._session_manager,
    guardrail_id=GUARDRAIL_ID,
    guardrail_version=GUARDRAIL_VERSION,
    guardrail_trace="enabled",
    callback_handler=None
)
```

**専門エージェント（AG-002/AG-003）の基本パターン**:
```python
agent = Agent(
    model=get_bedrock_model(),
    system_prompt=SPECIALIST_SYSTEM_PROMPT,
    tools=[domain_tools...],
    agent_id="transport_agent",   # またはexpense_agent
    name="交通費精算申請エージェント",  # またはエージェント名
    conversation_manager=SlidingWindowConversationManager(
        window_size=20,           # AG-002: 20、AG-003: 15
        should_truncate_results=True,
        per_turn=False
    ),
    hooks=[
        HumanApprovalHook(),
        LoopControlHook(max_iterations=10)
    ],
    session_manager=SessionManager(
        session_id=session_id,
        storage_path="data/sessions/"
    ),
    guardrail_id=GUARDRAIL_ID,
    guardrail_version=GUARDRAIL_VERSION,
    guardrail_trace="enabled",
    callback_handler=None
)
```

### 7.2 設定のカスタマイズ

**カスタマイズが必要な場合**:
- エージェントごとのウィンドウサイズ: AG-001=30、AG-002=20、AG-003=15
- フックの組み合わせ（オーケストレーターはLoopControlHookのみ、専門エージェントはHumanApprovalHook + LoopControlHook）は実行制御方針に従う

**カスタマイズ方法**: 各エージェントの基本設計書（エージェント基本設計.md）を参照

## 8. ログ設計方針

### 8.1 基本原則

- **一元管理**: ログ設定（レベル・出力先・フォーマット）は `main.py`（エントリーポイント）で一元管理する
- **分散出力**: ログ出力は各モジュール（エージェント・ツール・ハンドラー）が責任を持って行う
- **環境変数制御**: ログレベルは環境変数（`LOG_LEVEL`）で制御し、未設定時のデフォルトは `WARNING`
- **ユーザーメッセージと技術ログの分離**: ユーザー向けメッセージ（案内・フィードバック等）と技術ログ（エラーログ・監査ログ・トレースログ）は分離する。ユーザーにはスタックトレース等の技術的詳細を表示しない
- **コンテキスト情報**: 各モジュールが付加するコンテキスト情報の詳細（フィールド定義・マスキング対象等）は基本設計・詳細設計で定義する

### 8.2 設計判断

| DEC-ID | 決定事項 | 理由 |
|--------|---------|------|
| DEC-LOG-01 | ログ設定はエントリーポイント（`main.py`）で一元管理する | 設定の重複・分散を防止し、変更箇所を1か所に集約するため |
| DEC-LOG-02 | ユーザー向けメッセージと技術ログを分離する | ユーザーへの情報漏洩リスク（スタックトレース・機密情報）を防止し、UXを改善するため |
| DEC-LOG-03 | デフォルトログレベルをWARNINGとする | 通常運用時に不要なログ出力を抑制し、重要なログが埋もれないようにするため |

## 9. 設計上の留意点・リスク

### 8.1 会話管理

**リスク**: ウィンドウサイズが小さすぎると古い会話履歴が切り詰められ、エージェントが申請フローの文脈を失うリスクがある

**対策**: 業務コンテキスト（申請種別・収集済み申請情報）はセッション状態ファイルに別途永続化し、会話履歴が切り詰められてもフローを継続できるようにする

### 8.2 ループ制御

**リスク**: LoopControlHookが設定されていない場合、ReActループが無限に継続するリスクがある

**対策**: 全エージェント（AG-001〜AG-003）にLoopControlHookを必須設定として適用する。設定漏れをコードレビューで確認する

### 8.3 セッション管理

**リスク**: セッション状態ファイルが破損・不完全な状態で保存されると再開時に誤動作するリスクがある

**対策**: SessionManagerがセッション状態ファイルのバリデーション（JSONスキーマ検証）を実施してから復元する

### 8.4 承認フック

**リスク**: HumanApprovalHookが設定されていない専門エージェントがTOOL-002を社員確認なしで実行するリスクがある

**対策**: AG-002/AG-003には必ずHumanApprovalHookを設定する。GRD-010のガードレールと組み合わせて二重に防止する
