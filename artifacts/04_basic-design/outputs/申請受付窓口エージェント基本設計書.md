## 2. 目的・役割定義

### 2.1 エージェントの目的
- 社員の申請意図から交通費精算申請・経費精算申請のいずれかを判断して案内する
- 申請フローの起点として申請者名を収集し、適切な専門エージェントへ委譲する
- 社内申請ルールナレッジを根拠に申請種別判断の一貫性を保証する

### 2.2 主な責務
- **申請者名収集**: セッション開始時に申請者名を収集し、invocation_stateに設定する
- **申請意図受付**: 社員の申請意図テキスト（D-001）を受け取る
- **申請種別判断**: D-002（社内申請ルールナレッジ）を根拠に交通費精算申請・経費精算申請を判断する（BRL-01）
- **案内提示**: 申請種別・申請書・申請先をD-003として社員に提示する（根拠提示Lv2）
- **専門エージェントへの委譲**: 申請種別確定後にAG-002またはAG-003へフロー全体を委譲する

### 2.3 非責務（やらないこと）
- **申請情報収集**: AG-002（交通費）またはAG-003（経費）の責務
- **申請書生成・ファイル出力**: AG-002/AG-003の責務（最大自律度Lv2）
- **申請書チェック**: AG-002/AG-003の責務

## 3. 自律度・判断権限

### 3.1 自律度レベル

| レベル | 内容 | 適用範囲 |
|------|------|----------|
| 参照 | 社内申請ルールナレッジ（D-002）の参照 | ACT-READ-01 |
| 提案 | 申請種別の判断・案内（最終確認は人） | ACT-DEC-01（申請種別判断） |
| 実行 | 専門エージェントへの委譲（申請種別確定後） | LNK-001/LNK-002の発火 |

### 3.2 判断権限の範囲
- **自律判断可能**: 社内申請ルールナレッジを根拠とした申請種別の判断（JD-01/JD-02）
- **人間確認必要**: 申請種別が一意に判断できない場合の2択選択（BRL-01）。申請ルール外のケースのエスカレーション確認
- **実行不可**: 申請書生成・ファイル出力・申請書提出（ACT-EXEC-02禁止。GRD-011）

## 4. 処理フロー設計

### 4.1 基本フロー（概要）

```text
開始（申請者名はアプリ起動時に取得済み）
→ [申請者名受付] → [申請意図受付（D-001）]
→ [申請種別判断（D-002参照・JD-01/JD-02）]
→ [判断可: 案内提示（D-003）→ 専門エージェント委譲（LNK-001またはLNK-002）]
→ [判断不可: 2択提示（BRL-01）→ 選択後に委譲]
→ 終了
```

**フローの詳細**: 詳細設計書 agents/orchestrator_agent.py を参照

### 4.2 フロー制御の考え方
- **原則的な処理順序**: 申請者名収集 → 申請意図受付 → 申請種別判断 → 委譲の逐次処理
- **処理をスキップする考え方**: 申請種別が一意に判断できた場合（JD-01）は選択肢提示をスキップし直接委譲する
- **同一処理を複数回実行**: 判断不可時（JD-02）は選択肢提示→選択確認のループを繰り返す（社員が選択するまで）

### 4.3 状態管理
- **セッション状態（D-006）**: SessionManagerがdata/sessions/にJSONファイルとして保存する。申請者名・申請種別・session_idを保持する
- **invocation_state**: applicant_name・application_date・session_idを辞書リテラルで専門エージェントへ渡す
- **会話履歴**: SlidingWindowConversationManager（window_size=30）で管理する

## 5. 専門エージェントとの連携設計

### 5.1 連携対象エージェント

| エージェントID | エージェント名 | 連携目的 | 振り分け条件 |
|---------------|---------------|----------|--------------|
| AG-002 | 交通費精算申請エージェント | 交通費精算申請フロー全体の実行 | 申請種別が「交通費精算申請」に確定した場合（JD-01またはユーザー選択後）|
| AG-003 | 経費精算申請エージェント | 経費精算申請フロー全体の実行 | 申請種別が「経費精算申請」に確定した場合（JD-01またはユーザー選択後）|

### 5.2 振り分け判断基準

- **AG-002への委譲**: 申請意図がタクシー・電車・バス・飛行機等の交通費に関連すると判断（社内申請ルールナレッジ参照）
- **AG-003への委譲**: 申請意図が会食費・事務用品費・宿泊費等の経費に関連すると判断（社内申請ルールナレッジ参照）
- **フォールバック**: 判断不可時（JD-02）は2択選択肢を提示してユーザーに選択させる（BRL-01）

**詳細な判断基準**: 詳細設計書 agents/orchestrator_agent.py のシステムプロンプト参照

### 5.3 invocation_stateの受け渡し

専門エージェントへ申請者名・申請日・セッションIDを渡す（委譲メッセージには含めない）。

```python
{
    "applicant_name": str,      # アプリケーション起動時に取得した申請者名
    "application_date": str,    # システム日付（YYYY-MM-DD形式）
    "session_id": str           # SessionManagerが生成したセッションID
}
```

**データ構造の詳細**: マルチエージェント連携設計 7.3章を参照

## 7. プロンプト設計

### 7.1 システムプロンプトの目的
- AG-001の役割（申請種別判断・案内・委譲）を定義する
- 社内申請ルールナレッジ（D-002）を埋め込み、申請種別判断の根拠を提供する
- 判断不可時（JD-02）の選択肢提示手順（BRL-01）を指示する

### 7.2 プロンプト構成要素
- **役割定義**: 申請受付窓口エージェントとしての役割（申請者名収集・申請意図受付・申請種別判断・案内・委譲）
- **処理フロー**: CF-001（申請種別判断・案内）のStep 0〜3の手順
- **判断プロトコル**: 社内申請ルールナレッジを根拠に判断する手順（JD-01/JD-02/BRL-01）
- **根拠提示方針**: 判断根拠（適用ルールID・内容）をLv2で提示する
- **社内申請ルールナレッジ埋め込み**: D-002をシステムプロンプトに直接埋め込む

**プロンプト全文**: 詳細設計書 agents/orchestrator_agent.py 2.3.1章を参照

### 7.3 システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 静的定数 |
| 動的生成の理由 | 不要（AG-001は申請種別判断のみ担当。日付・期限を基準とした判断処理は含まれない） |
| 動的に埋め込む情報 | なし |

### 7.4 主要な設計意図
- **ナレッジ埋め込み方式**: D-002（社内申請ルールナレッジ）をRAGではなくシステムプロンプトに直接埋め込む（RAG廃止方針準拠）
- **単一責任**: 申請種別判断・案内のみを担い、申請情報収集・申請書作成は担当しない
- **委譲前確認**: 申請種別確定（JD-01またはユーザー選択）後にのみ専門エージェントへ委譲する

## 10. 入力データ設計

### 10.1 初期化時の入力
- **applicant_name**: 必須、空文字禁止
  - 入力方法: アプリケーション起動時にCLI引数またはプロンプト入力で取得する
  - バリデーション: 空文字・Noneは拒否して再入力を促す

### 10.2 対話ループでの入力
- **申請意図テキスト（D-001）**: 社員が自然文で入力する申請意図
- **特殊コマンド**:
  - 空入力: 再入力を促す（GRD-001）
  - 500文字超: 文字数上限を案内して再入力を促す（GRD-012）
  - `reset` / `リセット` / `最初から`: 会話履歴と申請者情報をリセットして申請フローを最初からやり直す

## 11. 出力データ設計

### 11.1 出力形式
- **出力先**: CLIの標準出力（社員への案内メッセージ）
- **出力内容**: D-003（申請種別・申請書・申請先の案内結果）。根拠（適用ルールID・内容）を付記する（Lv2）

### 11.2 出力メッセージ
- 申請種別判断完了: `"申請種別：交通費精算申請（根拠：[適用ルールID]）"`
- 判断不可: `"交通費精算申請・経費精算申請のどちらでしょうか？"`
- エスカレーション: `"ご申請内容が申請ルール外のため、担当部署へご相談ください。"`

## 16. 制約事項・前提条件

### 16.1 技術的制約
- LLMバックエンドはAmazon Bedrock（Claude Sonnet 4.5）のみ
- Strands AgentsのAgent as Toolsパターンで実装する
- ストリーミング無効（`callback_handler=None`）

### 16.2 業務的制約
- 申請書生成・ファイル出力・申請書提出はAG-001が実行しない（最大自律度Lv2）
- 申請種別確定前に専門エージェントへ委譲しない（委譲条件：マルチエージェント連携設計 6.2準拠）

### 16.3 前提条件
- 社内申請ルールナレッジ（D-002）がシステムプロンプトに埋め込まれていること
- SessionManagerが正常に初期化されていること
- アプリケーション起動時に申請者名を取得済みであること（main.pyで管理）
