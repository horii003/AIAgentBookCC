## 1. 概要

### 1.1 コンポーネントの目的

AG-001（申請受付窓口エージェント）から委譲された交通費精算申請フローを引き受け、移動情報（移動日・出発地・目的地・交通手段・費用・業務目的）を一区間ずつ対話で収集する。収集した情報をもとに交通費を自動計算（TOOL-001）し、社員確認（HumanApprovalHook）を取得した後に申請書（下書き）を生成（TL-002a）する。

### 1.2 主要な責務

1. **invocation_stateの受け取り**: ToolContextから`session_id`・`applicant_name`・`application_date`を取得する
2. **Agentインスタンスキャッシュ管理**: `_agent_instances` モジュールレベル辞書で session_id をキーにインスタンスをキャッシュし、SlidingWindowConversationManager の会話履歴を保持する
3. **動的システムプロンプト生成**: `_build_transport_agent_system_prompt(application_date)` で申請日・申請期限日を埋め込んだプロンプトを生成する
4. **移動情報の一括収集**: 1区間分の全必須項目（移動日・出発地・目的地・交通手段・費用・業務目的）を1つのメッセージで一括収集し、ターン数を最小化する
5. **申請期限チェック（BRL-14）**: 移動日が申請期限（申請日 - 90日）より前の場合は申請不可を通知してフローを終了する
6. **交通費自動計算（TOOL-001）**: calculate_transport_fare を呼び出して交通費を算出し、算出根拠（calculation_basis）とともに社員に提示する
7. **高額申請通知（BRL-10）**: 交通費が10,000円超の場合に上長承認要否を通知する（フロー継続）
8. **申請書生成前社員確認（GRD-014）**: HumanApprovalHook（BeforeToolCallEvent）で OK/修正/キャンセルを取得する。承認前に TL-002a を実行してはならない
9. **申請書（下書き）生成（TL-002a）**: HumanApprovalHook の OK 取得後のみ generate_transport_expense_form を呼び出す

### 2.2 インターフェース設計

#### 2.2.1 入力

**ツール関数引数（AG-001からの呼び出し時）**:
- `application_type` (str): 申請種別（"交通費精算申請"確定済み）（必須）
- `applicant_name` (str): 申請者名（必須）
- `user_input_text` (str): 社員の申請内容テキスト（最大500文字）（必須）

**ToolContext経由で受け取る情報**:
```python
{
    "session_id": str,          # セッションID（Agentインスタンスキャッシュのキーとして使用）
    "applicant_name": str,      # 申請者名
    "application_date": str     # 申請日（YYYY-MM-DD形式）
}
```

**対話ループでの入力**:
- 社員からの自然言語入力（移動情報・確認応答等）

#### 2.2.2 出力

**出力先**: AG-001（tool応答として返す）

**戻り値の形式**: `str`（エージェントからの最終応答テキスト）

**主要な出力メッセージ**:
- 移動情報一括収集依頼: `"交通費精算申請の移動情報をご入力ください。\n以下の形式で1区間分をまとめてご入力ください。\n...\n（例）\n移動日：2026-04-28\n出発地：渋谷\n目的地：新宿\n交通手段：電車\n費用：（自動計算）\n業務目的：社内会議のため"`
- 申請期限超過通知: `"移動日「{travel_date}」から90日を超えているため、交通費精算申請はできません（根拠: BRL-14）。担当部門にご確認ください。"`
- 交通費計算結果提示: `"交通費は{fare}円です（根拠: {calculation_basis}）。"`
- 高額申請通知: `"交通費が10,000円を超えているため、上長の承認が必要です（根拠: BRL-10）。フローは継続します。"`
- 申請書生成完了: `"申請書（下書き）を「{file_path}」に生成しました。総務部に提出してください。"`
- エラー（経路不明）: `"該当する経路の運賃テーブルが見つかりませんでした。交通費を直接入力してください。"`

### 2.3 ビジネスロジック

#### 2.3.1 システムプロンプト

##### システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 動的生成関数 |
| 動的生成の理由 | 申請期限チェック（BRL-14）で移動日から90日以内かを判断するために、申請日（`application_date`）と申請期限日（`deadline_date`）をプロンプトに埋め込む必要がある |

##### 動的に埋め込む情報
- `application_date`（申請日、YYYY-MM-DD形式）: `invocation_state["application_date"]` から取得
- `deadline_date`（申請期限日、YYYY-MM-DD形式）: `(date.fromisoformat(application_date) - timedelta(days=90)).isoformat()`
- 交通費申請に関するポリシー情報: `agent_knowledge/transportation_policies.py` から取得して埋め込む

##### プロンプト全文

```python
def _build_transport_agent_system_prompt(application_date: str) -> str:
    from datetime import date, timedelta
    from agent_knowledge.transportation_policies import get_transportation_policies
    deadline_date = (date.fromisoformat(application_date) - timedelta(days=90)).isoformat()
    policies = get_transportation_policies()
    return f"""あなたは交通費精算申請の専門エージェントです。移動情報を収集し、交通費を計算して申請書（下書き）を生成します。

【申請日・申請期限】
- 申請日: {application_date}
- 申請期限: {deadline_date}以降の移動日のみ申請可能（移動日がこれより前の場合は申請不可）

【役割と責務】
1. 移動情報を1区間ずつ対話で収集する
2. 各区間の移動日・出発地・目的地・交通手段が確定したら、calculate_transport_fare を呼び出して交通費を自動計算する
3. すべての区間情報が収集できたら、収集済み申請情報をテキストとして整理・提示する
4. 社員の確認（OK/修正/キャンセル）を取得後、generate_transport_expense_form を呼び出して申請書（下書き）を生成する

【移動情報の一括収集（必須）】
- 項目を1つずつ順番に聞いてはいけません。
- 1区間分の必要項目をまとめて1つのメッセージで依頼してください。
- 一括収集メッセージの例:
  「交通費精算申請の移動情報をご入力ください。
  以下の形式で1区間分をまとめてご入力ください。

  移動日：（YYYY-MM-DD形式）
  出発地：（駅名または地点名）
  目的地：（駅名または地点名）
  交通手段：（電車／バス／タクシー／飛行機のいずれか）
  業務目的：（申請理由を具体的に）

  ※費用は交通手段と区間に基づいて自動計算します。

  （入力例）
  移動日：2026-04-28
  出発地：渋谷
  目的地：新宿
  交通手段：電車
  業務目的：社内会議のため」

【申請期限チェック（BRL-14）】
- 各区間の移動日を受け取ったら、申請期限（{deadline_date}）より前かどうかを確認する
- 移動日 < {deadline_date} の場合は申請不可を通知してフローを終了する
- メッセージ例: 「移動日「{travel_date_example}」から90日を超えているため、交通費精算申請はできません（根拠: BRL-14）。担当部門にご確認ください。」

【交通手段（BRL-11/12）】
- 対応交通手段: 電車・バス・タクシー・飛行機の4種類のみ
- 別表記（JR・新幹線・地下鉄・モノレール → 電車、ハイヤー・タクシー代 → タクシー、航空・ANA・JAL・飛機 → 飛行機、路線バス → バス）は自動的に正規化される
- 対応外の交通手段（自動車・徒歩等）が入力された場合は、対応外である旨を伝えて再選択を促す

【駅名の正規化（BRL-15）】
- 駅名に「駅」「Station」等の接尾語が含まれていても、calculate_transport_fare に渡す際に自動的に除去される

【交通費の自動計算（TOOL-001）】
- 各区間の移動日・出発地・目的地・交通手段が確定したら、calculate_transport_fare を呼び出す
- 計算成功時: 「交通費は{fare}円です（根拠: {calculation_basis}）。」と提示する
- 計算不能時（経路テーブルに該当なし）: 「該当する経路の運賃テーブルが見つかりませんでした。交通費を直接入力してください。」と案内し、社員に手動入力を求める
- calculate_transport_fare の呼び出しパラメータ:
  {{
    "departure": "出発地（接尾語除去後）",
    "destination": "目的地（接尾語除去後）",
    "transportation_type": "交通手段（正規化後）",
    "travel_date": "YYYY-MM-DD形式の移動日",
    "purpose": "業務目的"
  }}

【高額申請通知（BRL-10）】
- 交通費が10,000円を超える場合: 「交通費が10,000円を超えているため、上長の承認が必要です（根拠: BRL-10）。」と通知する
- 高額通知後もフローは継続する（フロー終了ではない）

【多区間ループ】
- 1区間の情報収集・計算が完了したら、「他に申請する移動区間はありますか？」と確認する
- 追加区間がある場合は、次の区間の情報一括収集に戻る
- すべての区間が完了したら、収集済み申請情報をテキストとして整理して提示する

【収集済み申請情報の提示（ドラフト提示）】
- すべての区間情報が収集完了したら、次のように整理して提示する（ツール呼び出しなし）:
  「以下の申請情報をご確認ください。
  【交通費精算申請】
  申請者: {applicant_name}
  申請日: {application_date}
  区間1:
    移動日: {travel_date}  出発地: {departure}  目的地: {destination}
    交通手段: {transportation_type}  費用: {amount}円
    業務目的: {purpose}
  （以下、収集済み全区間分）
  上記の内容でよろしいですか？」
- この提示はテキスト整理であり、ツール呼び出しを行ってはならない

【申請書生成（TL-002a）】
- 社員から「OK」または承認の意思が示された後に generate_transport_expense_form を呼び出す
- 承認前に generate_transport_expense_form を呼び出してはならない
- generate_transport_expense_form の呼び出しパラメータ:
  {{
    "applicant_name": "申請者名",
    "application_date": "YYYY-MM-DD形式の申請日",
    "segments": [
      {{
        "travel_date": "YYYY-MM-DD形式の移動日",
        "departure": "出発地",
        "destination": "目的地",
        "transportation_type": "交通手段",
        "amount": 費用（整数）,
        "purpose": "業務目的"
      }}
    ]
  }}
- 申請書生成完了時: 「申請書（下書き）を「{{file_path}}」に生成しました。総務部に提出してください。」と案内する

【禁止事項】
- 申請書の提出操作は絶対に行わない（GRD-012）。提出は社員が行う
- 上長承認の代行は絶対に行わない（GRD-013）。承認取得は社員が行う
- 申請書生成は社員の確認（OK）取得後のみ実施する（GRD-014）

【エラー対応】
- ツールエラーやシステム障害が発生した場合は、エラー内容を要約して呼び出し元（AG-001）に返す
- 対話継続不能な場合は「担当部門（管理部）にお問い合わせください。」と案内する
"""
```

##### プロンプト設計の意図

| 要素 | 意図 |
|------|------|
| 動的プロンプト（申請日・期限日） | BRL-14の申請期限チェックに実行時日付が必要なため、毎回生成する |
| 一括収集指示 | 項目を1つずつ聞く誤動作を防ぎ、ターン数を削減してユーザー体験を向上させる |
| 申請期限チェック | BRL-14の判断基準（deadline_date）をプロンプトに直接埋め込み、LLMが正確に判断できるようにする |
| calculate_transport_fareパラメータ例示 | dict型引数の必須キー・型を明示し、LLMがキーを省略する実行時エラーを防ぐ |
| generate_transport_expense_formパラメータ例示 | dict型引数（segments）の必須キー・型・構造を明示し、LLMがキーを省略する実行時エラーを防ぐ |
| 禁止事項明示 | GRD-012（提出禁止）・GRD-013（上長承認代行禁止）・GRD-014（確認前生成禁止）を明示する |

#### 2.3.2 申請期限チェック（BRL-14）

| ルール | 条件 | 対応 |
|--------|------|------|
| BRL-14: 申請期限90日チェック | 移動日 < deadline_date（申請日 - 90日） | 申請不可を通知してフロー終了。EX-03として処理 |
| BRL-14: 期限内 | 移動日 >= deadline_date | フロー継続 |

#### 2.3.3 高額申請チェック（BRL-10）

| ルール | 条件 | 対応 |
|--------|------|------|
| BRL-10: 上長承認要否 | fare > 10000 | 上長承認要否を通知してフロー継続 |
| BRL-10: 通常 | fare <= 10000 | 通知なしでフロー継続 |

```
  ↓
【ステップ3】エージェント呼び出し（対話ループ開始）
  agent_instance(user_input_text, invocation_state=tool_context.invocation_state) を呼び出す
  ↓
  === 以下はエージェント内部の対話ループ（CF-003） ===
  ↓
【CF-003 Step 1-2: 移動情報の一括収集】
  - 1区間分の必要項目をまとめて依頼するメッセージを提示
    （移動日・出発地・目的地・交通手段・業務目的の一括入力フォーマット）
  - 社員が入力を返す
  ↓
【CF-003 Step 3: 申請期限チェック（BRL-14）】
  - 移動日（travel_date）を解析
  - travel_date < deadline_date の場合:
    → 申請不可メッセージを提示してフロー終了（EX-03）
  - travel_date >= deadline_date の場合:
    → Step 4へ進む
  ↓
【CF-003 Step 4-9: 出発地・目的地・交通手段の確認】
  - 一括収集データから出発地・目的地・交通手段を取得
  - 対応外交通手段の場合: 再選択を促す（EX-01）
  - 確認後、Step 10へ進む
  ↓
【CF-003 Step 10: 交通費計算（TOOL-001）】
  - calculate_transport_fare を呼び出す
  - 成功: fare, calculation_basis を取得 → Step 11へ
  - 失敗（EX-02 経路不明）: 手動入力を促すメッセージを提示 → 社員から費用を受け取る
  - 失敗（EX-04 ファイル参照失敗）: ErrorHandler経由でCF-006に誘導して中断
  ↓
【CF-003 Step 11: 交通費の提示】
  - 「交通費は{fare}円です（根拠: {calculation_basis}）。」
  ↓
【CF-003 Step 12: 高額チェック・上長承認要否通知（BRL-10）】
  - fare > 10000 の場合: 「交通費が10,000円を超えているため、上長の承認が必要です（根拠: BRL-10）。」と通知
  - フロー継続
  ↓
【CF-003 Step 13-14: 業務目的の確認】
  - 一括収集データに業務目的が含まれている場合は確認のみ
  - 含まれていない場合は追加収集
  ↓
【CF-003 Step 15-16: 追加区間の確認】
  - 「他に申請する移動区間はありますか？」と確認
  - 追加あり → Step 1に戻る（多区間ループ）
  - 追加なし → Step 17へ進む
  ↓
【CF-003 Step 17: 収集済み申請情報の提示（ドラフト提示）】
  - 全収集済み区間情報をテキストとして整理・提示（ツール呼び出しなし）
  - 「以下の申請情報をご確認ください。...上記の内容でよろしいですか？」
  ↓
【HumanApprovalHook: BeforeToolCallEvent介入】
  generate_transport_expense_form 呼び出し直前に HumanApprovalHook が介入:
  - [OK] → generate_transport_expense_form 実行を許可
  - [修正] → 修正対象の区間再収集フローへ戻る
  - [キャンセル] → セッション状態をCLOSED、フロー終了
  ↓
【generate_transport_expense_form 呼び出し（TL-002a）】
  - TransportFormInput（applicant_name, application_date, segments）を渡す
  - 成功: file_path を取得 → 申請書生成完了メッセージを提示
  - 失敗（EX-04）: CF-006に誘導して中断
  - 失敗（EX-06）: 1回リトライ後も失敗の場合はCF-006にエスカレーション
  ↓
終了（apply_agent_tool の戻り値として応答テキストを返す）
```

### 2.5 連携設計

#### 2.5.1 Agent as Tools設計

**ツール名**: `transport_agent_tool`

**ツール説明**:
```
交通費精算申請エージェント（AG-002）を呼び出し、移動情報の収集・交通費計算・申請書生成を行う。
AG-001が申請種別「交通費精算申請」と確定した後に呼び出す。
```

**ToolContextの使用**:
- `@tool(context=True)` デコレータを使用
- `tool_context: ToolContext` パラメータで ToolContext を受け取る
- `tool_context.invocation_state` から `session_id`・`applicant_name`・`application_date` を取得

#### 2.5.2 invocation_stateの受け渡しとAgentインスタンスのキャッシュ

```python
# session_id をキーにして Agent インスタンスをキャッシュし、
# セッション内で会話履歴を保持する（情報の再入力を防ぐ）
_agent_instances: dict[str, Agent] = {}

@tool(context=True)
def transport_agent_tool(
    application_type: str,
    applicant_name: str,
    user_input_text: str,
    tool_context: ToolContext,
) -> str:
    # invocation_state から状態を取得
    state = tool_context.invocation_state
    session_id = state.get("session_id", "")
    application_date = state.get("application_date", "")

    # session_id 未登録の場合のみ新しい Agent インスタンスを生成する
    # Agent初期化時に callback_handler=None を指定して標準出力への直接表示を抑制する
    # （AG-001による二重出力を防ぐ。agent()呼び出し時に渡すと TypeError になるため必ずコンストラクタで指定）
    if session_id not in _agent_instances:
        _agent_instances[session_id] = Agent(
            model=BedrockModel(model_id="jp.anthropic.claude-sonnet-4-5-20250929-v1:0"),
            system_prompt=_build_transport_agent_system_prompt(application_date),
            tools=[calculate_transport_fare, generate_transport_expense_form],
            conversation_manager=SlidingWindowConversationManager(window_size=20),
            hooks=[LoopControlHook(max_iterations=10), HumanApprovalHook(approval_callback=approval_callback)],
            session_manager=SessionManagerFactory.create(),
            callback_handler=None,
        )

    response = _agent_instances[session_id](user_input_text, invocation_state=state)
    return str(response)
```

## 3. 実装詳細

### 3.1 クラス/関数設計

#### 3.1.1 `_build_transport_agent_system_prompt` 関数

##### 説明
申請日・申請期限日・交通費申請ポリシーを埋め込んだ交通費精算申請エージェントのシステムプロンプトを動的生成する。

##### 引数
- `application_date` (str): 申請日（YYYY-MM-DD形式）（必須）

##### 戻り値
- `str`: 完全なシステムプロンプトテキスト

##### 処理内容
1. `deadline_date = (date.fromisoformat(application_date) - timedelta(days=90)).isoformat()` を計算する
2. `agent_knowledge/transportation_policies.py` の `get_transportation_policies()` からポリシー情報を取得する
3. `application_date`・`deadline_date`・`policies` を埋め込んだプロンプト文字列を返す

#### 3.1.2 `transport_agent_tool` 関数

##### 説明
AG-002（交通費精算申請エージェント）をツールとして公開し、AG-001からの呼び出しを受け付ける。

##### 引数
- `application_type` (str): 申請種別（"交通費精算申請"確定済み）（必須）
- `applicant_name` (str): 申請者名（必須）
- `user_input_text` (str): 社員の申請内容テキスト（必須）
- `tool_context` (ToolContext): ToolContext（`@tool(context=True)` による注入）（必須）

##### 戻り値
- `str`: AG-002エージェントからの応答テキスト

##### 処理内容
1. `tool_context.invocation_state` から `session_id`・`application_date` を取得する
2. `_agent_instances` に `session_id` が存在しない場合、新しい Agent インスタンスを生成して登録する
3. `_agent_instances[session_id](user_input_text, invocation_state=state)` を呼び出す
4. 応答を `str` にキャストして返す

##### 例外処理
- RuntimeError（ループ制限）、RuntimeError（その他）、Exception は `3.2 エラーハンドリング` に従い捕捉・処理する

## 5. 補足情報

### 5.1 実装上の注意点

1. **callback_handler=None はコンストラクタで指定**
   - `Agent(..., callback_handler=None)` のようにコンストラクタで指定する。`agent(query, callback_handler=None)` のように呼び出し時に渡すと `TypeError` になる。

2. **_agent_instancesはモジュールレベルで定義**
   - `_agent_instances: dict[str, Agent] = {}` をモジュールレベルで定義し、ツール関数呼び出しのたびに再生成してはならない。同一セッション内で SlidingWindowConversationManager の会話履歴を保持するために必須。

3. **動的プロンプトは新規インスタンス生成時のみ**
   - `_build_transport_agent_system_prompt(application_date)` はキャッシュが存在しない（初回呼び出し時）のみ呼び出す。既存キャッシュを再利用する場合はプロンプト再生成を行わない。

4. **一括収集指示のプロンプト設計**
   - システムプロンプトに「項目を1つずつ順番に聞いてはいけません」と明示し、一括収集メッセージの例をプロンプト内に記載する。これにより LLM が1問1答型の対話を行う誤動作を防ぐ。

5. **ドラフト提示とツール呼び出しの分離**
   - Step 17 の収集済み申請情報提示はテキスト整理（ツール呼び出しなし）として設計する。その後 generate_transport_expense_form を呼び出した時点で HumanApprovalHook（BeforeToolCallEvent）が介入する。この2ステップの分離により HITL 承認は1回のみ発生する。

---

### 5.2 パフォーマンス考慮事項

1. **SlidingWindowConversationManager のウィンドウサイズ**
   - `window_size=20` は多区間収集（複数区間のやり取り）に対応するために設定。収集区間数が増加する場合はウィンドウサイズの見直しが必要。

2. **Agentインスタンスキャッシュ**
   - セッション単位でキャッシュするため、同時セッション数に比例してメモリ使用量が増加する。長期セッション未使用のインスタンスは積極的にクリアする設計を将来的に検討する。

---

### 5.3 セキュリティ考慮事項

1. **invocation_stateの信頼性**
   - `tool_context.invocation_state` は AG-001 が設定した値を受け取る。外部からの直接操作を防ぐため、session_id のフォーマット検証（`YYYYMMDDHHMMSSxxxxxxxx`、例: `20260502143022_a1b2c3d4`）を実施する。

2. **個人情報（applicant_name）のログマスキング**
   - ログ出力時に `applicant_name` は `applicant_name[:1] + "***"` 形式でマスキングする（GRD-011）。

## 7. テスト観点

### 7.1 機能テスト
- 1区間の移動情報を一括収集フォーマットで入力した場合に、5項目すべてが正しく収集されること
- 複数区間（2区間以上）のループが正常に動作し、全区間情報が収集されること
- calculate_transport_fare が各区間で正しく呼び出され、fare・calculation_basis が提示されること
- 収集済み申請情報のドラフト提示がツール呼び出しなしのテキスト整理として動作すること
- HumanApprovalHook の OK 選択後に generate_transport_expense_form が呼び出されること
- generate_transport_expense_form が成功し、file_path が提示されること
- 同一セッション内で transport_agent_tool を複数回呼び出した場合に、同一の Agent インスタンスが再利用されること（会話履歴が保持されること）
- session_id が異なる場合に別の Agent インスタンスが生成されること

### 7.2 異常系テスト
- 移動日が申請期限（90日前）を超えている場合に申請不可メッセージが返却され、フローが終了すること（BRL-14）
- 移動日がちょうど90日前の場合にフローが継続すること（BRL-14 境界値）
- 対応外交通手段（「自動車」等）入力時に再選択を促すメッセージが返却されること（BRL-11/12）
- 電車経路テーブルに存在しない区間で手動入力フォールバックが動作すること（EX-02）
- train_fares.json が存在しない場合に EX-04 対応メッセージが返却されること
- HumanApprovalHook の修正選択時に TL-002a が実行されず、対話に戻ること
- HumanApprovalHook のキャンセル選択時に TL-002a が実行されず、フローが終了すること
- LoopControlHook が 10 反復を超えた場合に EX-08 対応メッセージが返却されること
- 承認（OK）前に generate_transport_expense_form が実行されないこと（GRD-014）

### 7.3 性能テスト
- 3区間の移動情報収集から申請書生成まで、合計10ターン以内で完了すること
- Agentインスタンスキャッシュにより、同一セッション内での2回目以降の呼び出し初期化コストが発生しないこと
- SlidingWindowConversationManager(window_size=20) で会話履歴が正しく切り詰められること（21ターン目で先頭が削除されること）
