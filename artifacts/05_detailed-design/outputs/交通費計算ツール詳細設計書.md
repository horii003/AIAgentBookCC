## 1. 概要

### 1.1 コンポーネントの目的

出発地・目的地・交通手段・移動日の情報から正確な運賃を算出してAG-002に返す。電車は `data/templates/train_routes.json`（DATA-011）を検索して区間運賃を取得し、バス・タクシー・飛行機は `data/templates/fixed_fares.json`（DATA-012）の固定運賃を参照する。エージェント単独の文章生成では正確な運賃を保証できないため、確定的なデータ参照による計算を提供する。

### 1.2 主要な責務

1. **電車運賃の検索**: アプリ起動時に静的読み込み済みのDATA-011（`data/templates/train_routes.json`）から出発駅・到着駅に対応する区間運賃を検索して返す
2. **固定運賃の参照**: DATA-012（`data/templates/fixed_fares.json`）からバス（230円）・タクシー（10,000円）・飛行機（50,000円）の固定運賃を参照して返す
3. **入力バリデーション**: `TravelExpenseCalculatorInput` Pydanticモデルで移動日・出発地・目的地・交通手段を検証する（`mode="before"` の正規化バリデーター経由）
4. **エラー結果の返却**: 経路未発見・対応外交通手段の場合は `{"success": False, "message": ...}` を返す（例外の再送出禁止）
5. **ファイルデータの静的読み込み**: アプリ起動時に `load_fare_data()` 補助関数を呼び出し、DATA-011/DATA-012を `TrainFareRecord` Pydanticモデルで検証した上でメモリ上に保持する。`load_fare_data()` のエラー時戻り値は `tuple[bool, str]` 型の `(False, エラーメッセージ文字列)` を返す
6. **申請者名・申請日の取得**: `@tool(context=True)` デコレーターを使用し、`invocation_state` から申請者名・申請日を取得する（LLMがツールパラメータとして渡す値ではない）

## 3. calculate_travel_expenseツール設計

### 3.1 基本情報

#### ツール名（関数名）
`calculate_travel_expense`

#### 説明（docstring）
```
1区間分の移動情報から運賃を算出して返す。

電車の場合はDATA-011（data/templates/train_routes.json）の区間運賃テーブルを参照し、
バス・タクシー・飛行機の場合はDATA-012（data/templates/fixed_fares.json）の固定運賃を返す。

申請者名・申請日はLLMがツールパラメータとして渡す値ではなく、
invocation_stateからツール関数内部で取得する（@tool(context=True) デコレーター使用）。

Args:
    travel_date (str): 移動日（YYYY-MM-DD形式。例: "2026-04-28"）
    departure (str): 出発地（正規化済み駅名または地名。例: "渋谷"）
    destination (str): 目的地（正規化済み駅名または地名。例: "新宿"）
    transport_type (str): 交通手段。許容値: "電車", "バス", "タクシー", "飛行機"。
        英語表記（"train", "bus", "taxi", "airplane", "plane"）および別表記
        （"鉄道", "地下鉄", "cab"）は正規化処理で変換される。

Returns:
    dict: 成功時は {"success": True, "fare": int, "calculation_basis": str}、
          失敗時は {"success": False, "message": str}
```

---

### 3.2 インターフェース設計

#### 3.2.1 入力

**パラメータ定義**:

| 引数名 | 型 | 説明 | デフォルト値 |
|--------|-----|------|--------------|
| `travel_date` | `str` | 移動日（YYYY-MM-DD形式。`parse_date` バリデーターで `datetime.date` 型に変換。BRL-13の期限チェックに使用） | なし（必須） |
| `departure` | `str` | 出発地（正規化済み駅名または地名。AG-002のシステムプロンプトがBRL-15で正規化済みの値を渡す） | なし（必須） |
| `destination` | `str` | 目的地（正規化済み駅名または地名。AG-002のシステムプロンプトがBRL-15で正規化済みの値を渡す） | なし（必須） |
| `transport_type` | `str` | 交通手段（TRANSPORT_TYPE_MAPで正規化後にLiteral["電車","バス","タクシー","飛行機"]で検証） | なし（必須） |

> **注意**: `applicant_name`・`application_date` はLLMがツールパラメータとして渡す値ではない。`@tool(context=True)` デコレーターを使用し、ツール関数内部で `invocation_state` から取得する。

##### バリデーションルール詳細

| フィールド | ルール | エラーメッセージ | 例 |
|-----------|--------|----------------|-----|
| `travel_date` | YYYY-MM-DD形式または dateutil.parser.parse 変換可能な文字列。空文字禁止 | "移動日のエラー: 日付形式が不正です。\n対処方法: YYYY-MM-DD形式（例: 2026-04-28）で入力してください。" | OK: `"2026-04-28"` / NG: `"20260428"`, `""` |
| `departure` | min_length=1（空文字・空白のみ禁止）。strip()後に検証 | "出発地が入力されていません。出発地を入力してください。" | OK: `"渋谷"` / NG: `""`, `"   "` |
| `destination` | min_length=1（空文字・空白のみ禁止）。strip()後に検証 | "目的地が入力されていません。目的地を入力してください。" | OK: `"新宿"` / NG: `""`, `"   "` |
| `transport_type` | TRANSPORT_TYPE_MAPで正規化後にLiteral["電車","バス","タクシー","飛行機"]で検証。マップ外の値はそのままLiteralバリデーションに渡す | "交通手段のエラー: 指定された交通手段が認識できません。\n対処方法: 「電車」「バス」「タクシー」「飛行機」のいずれかで入力してください。" | OK: `"電車"`, `"train"` / NG: `"自転車"` |

##### 正規化処理

| 入力値 | 正規化後 | 処理内容 |
|--------|---------|---------|
| `"train"`, `"鉄道"`, `"地下鉄"` | `"電車"` | `TRANSPORT_TYPE_MAP` による変換（`field_validator(mode="before")`） |
| `"bus"` | `"バス"` | `TRANSPORT_TYPE_MAP` による変換 |
| `"taxi"`, `"cab"` | `"タクシー"` | `TRANSPORT_TYPE_MAP` による変換 |
| `"airplane"`, `"plane"` | `"飛行機"` | `TRANSPORT_TYPE_MAP` による変換 |

#### 3.2.2 出力

**戻り値の型**: `dict[str, bool | int | str]`

**説明**: 成功時は運賃と計算根拠を含む辞書、失敗時はエラーメッセージを含む辞書を返す

**構造**:
```python
# 成功時
{
    "success": bool,           # True
    "fare": int,               # 区間運賃（円、0以上）
    "calculation_basis": str   # "電車経路テーブル参照" または "固定運賃参照"
}

# 失敗時（ValidationError・経路未発見・対応外交通手段）
{
    "success": bool,   # False
    "message": str     # ユーザー向けエラーメッセージ（日本語）
}
```

#### 3.2.3 例外

| 例外 | 発生条件 |
|------|---------|
| `ValidationError` | `TravelExpenseCalculatorInput` のバリデーション失敗時。ツール関数内でキャッチし `{"success": False, "message": ...}` を返す（再送出しない） |
| `ValueError` | 経路テーブルに指定経路が存在しない場合。ツール関数内でキャッチし `{"success": False, "message": ...}` を返す（`KeyError` は使用しない） |

> **注意**: `load_fare_data()` のファイルI/Oエラーは戻り値 `(False, エラーメッセージ文字列)` で返す。例外を再送出しない。

### 3.3 ビジネスロジック

#### 3.3.1 処理フロー

```
開始
  ↓
入力バリデーション（TravelExpenseCalculatorInput）
  - travel_date: parse_date バリデーター（mode="before"）で datetime.date 型に変換
  - transport_type: normalize_transport_type バリデーター（mode="before"）で正規化 → Literal型チェック
  - departure, destination: min_length=1 チェック（strip()後）
  ↓
ValidationError 発生？
  Yes → ErrorHandler.handle_validation_error(e) を呼び出しエラーメッセージを取得
        → {"success": False, "message": error_message} を返す
  No  → 次ステップへ
  ↓
transport_type の判定
  ↓
"電車" → 電車運賃テーブル検索
  - _train_fares（list[TrainFareRecord]）から departure と destination が一致するレコードを検索
  - 一致するレコードあり → fare を取得
  - 一致するレコードなし → ValueError を送出（KeyError は使用しない）
                          → ErrorHandler.handle_data_inconsistency(EX-06) を呼び出しログ記録
                          → {"success": False, "message": "指定された経路の運賃データが見つかりませんでした。交通費を手動で入力してください。"} を返す
  ↓ （一致レコードあり）
{"success": True, "fare": fare, "calculation_basis": "電車経路テーブル参照"} を返す

"バス" / "タクシー" / "飛行機" → 固定運賃テーブル参照
  - _fixed_fares（dict）から transport_type に対応する運賃を取得
    バス: 230円、タクシー: 10000円、飛行機: 50000円
  ↓
{"success": True, "fare": fare, "calculation_basis": "固定運賃参照"} を返す
```

#### 3.3.2 データキャッシュ機構

**キャッシュ機構**:

**用途**: アプリ起動時に DATA-011/DATA-012 を1回だけ読み込み、モジュールレベルの変数 `_train_fares` / `_fixed_fares` としてメモリ上に保持する。ツール関数実行時は毎回ファイルI/Oせずメモリ上のデータを参照する（冪等性・パフォーマンスの確保）。

**実装**:
```python
# モジュールレベル変数（起動時に load_fare_data() で初期化）
_train_fares: list[TrainFareRecord] = []
_fixed_fares: dict[str, int] = {}

def load_fare_data() -> tuple[bool, str]:
    """アプリ起動時に1回だけ呼び出す。失敗時は (False, エラーメッセージ) を返す。"""
    global _train_fares, _fixed_fares
    # ファイル存在チェック（os.path.exists() による事前チェック）
    if not os.path.exists("data/templates/train_routes.json"):
        return (False, "運賃データファイルが見つかりません: data/templates/train_routes.json")
    with open("data/templates/train_routes.json", encoding="utf-8") as f:
        raw = json.load(f)
    _train_fares = [TrainFareRecord(**r) for r in raw]
    # ファイル存在チェック（os.path.exists() による事前チェック）
    if not os.path.exists("data/templates/fixed_fares.json"):
        return (False, "固定運賃データファイルが見つかりません: data/templates/fixed_fares.json")
    with open("data/templates/fixed_fares.json", encoding="utf-8") as f:
        _fixed_fares = json.load(f)
    return (True, "")
```

#### 3.3.3 データの検証方法

**Pydanticモデル**: `TravelExpenseCalculatorInput`

**検証内容**:
- `travel_date`: `parse_date` field_validator（mode="before"）で文字列 → `datetime.date` 型に変換。変換失敗時は `ValueError`（EX-01）
- `transport_type`: `normalize_transport_type` field_validator（mode="before"）で `TRANSPORT_TYPE_MAP` による正規化 → `Literal["電車","バス","タクシー","飛行機"]` で検証。正規化後も Literal 外の場合は `ValidationError`（EX-01）
- `departure`, `destination`: `Field(..., min_length=1)` + strip() バリデーターで空文字・空白のみを拒否

**Pydanticモデル**: `TrainFareRecord`

**検証内容**:
- `departure`, `destination`: `min_length=1` で空文字禁止
- `fare`: `ge=0` で非負整数チェック。アプリ起動時の `load_fare_data()` 内で検証。失敗時は EX-06

### 3.4 エラーハンドリング

#### 3.4.1 エラー処理フロー

```
エラー発生
  ↓
エラー種別の判定
  ↓
ValidationError（EX-01）
  → ErrorHandler.handle_validation_error(e) を呼び出し
  → error_message を取得
  → ログ記録（ERRORレベル）
  → {"success": False, "message": error_message} を返す

経路テーブル未登録（EX-06）
  → ValueError を送出（KeyError は使用しない）
  → ErrorHandler.handle_data_inconsistency() を呼び出し
  → ログ記録（ERRORレベル）
  → {"success": False, "message": "指定された経路の運賃データが見つかりませんでした。交通費を手動で入力してください。"} を返す

ファイルI/Oエラー（EX-07、load_fare_data()内のみ）
  → ErrorHandler.handle_system_error(e) を呼び出し
  → ログ記録（WARNINGレベル）
  → (False, エラーメッセージ文字列) を返す（例外を再送出しない）

想定外例外（EX-08）
  → ErrorHandler.handle_unexpected_error(e) を呼び出し
  → ログ記録（CRITICALレベル、スタックトレース必須）
  → {"success": False, "message": "申し訳ありません。予期しないエラーが発生しました。システム管理者にご連絡ください。"} を返す
```

#### 3.4.2 エラーメッセージの詳細

| エラー種別 | 発生条件 | 対応 | ユーザー向けメッセージ |
|-----------|---------|------|---------------------|
| E-001（EX-06）: 経路テーブル未登録 | DATA-011に指定経路のレコードなし（ValueError） | 代替処理（手動入力促し）。ErrorHandler.handle_data_inconsistency() でEX-06ログ記録 | "申し訳ありません。指定された経路の運賃データが見つかりませんでした。交通費を手動で入力してください。" |
| E-002（EX-01）: 対応外交通手段 | transport_type が Literal 外の値（TRANSPORT_TYPE_MAP変換後も不一致） | ValidationErrorをキャッチしErrorHandler.handle_validation_error()でEX-01ログ記録 | "交通手段のエラー: 指定された交通手段が認識できません。\n対処方法: 「電車」「バス」「タクシー」「飛行機」のいずれかで入力してください。" |
| E-003（EX-01）: 出発地・目的地が空 | departure または destination が空文字・空白のみ | ValidationErrorをキャッチしErrorHandler.handle_validation_error()でEX-01ログ記録 | "出発地が入力されていません。出発地を入力してください。" または "目的地が入力されていません。目的地を入力してください。" |
| E-004（EX-07）: ファイルI/Oエラー | load_fare_data() でDATA-011/DATA-012が読み込み不可 | ErrorHandler.handle_system_error()でEX-07ログ記録（WARNINGレベル）。戻り値 `(False, エラーメッセージ文字列)` を返す（例外を再送出しない） | "申し訳ありません。システムエラーが発生しました。システム管理者にご連絡いただくか、しばらく時間をおいて再度お試しください。" |

### 4.2 データモデル

#### TravelExpenseCalculatorInputモデル

| フィールド名 | 型 | 必須/任意 | 説明 | 制約 |
|--------------|-----|-----------|------|------|
| `travel_date` | `datetime.date` | 必須 | 移動日（parse_dateバリデーターでstr→date変換） | parse_date field_validator(mode="before")。YYYY-MM-DD形式または dateutil パース可能な文字列 |

| `destination` | `str` | 必須 | 目的地（正規化済み駅名・地名） | `Field(..., min_length=1)`。strip()後に空文字チェック |

#### TrainFareRecordモデル

| フィールド名 | 型 | 必須/任意 | 説明 | 制約 |
|--------------|-----|-----------|------|------|
| `departure` | `str` | 必須 | 出発駅名 | `min_length=1` |
| `destination` | `str` | 必須 | 到着駅名 | `min_length=1` |
| `fare` | `int` | 必須 | 区間運賃（円） | `ge=0`（非負整数） |

**バリデーション**:
- `validate_fare`: `@field_validator("fare", mode="before")` — `isinstance(v, int) and v >= 0` をチェック。失敗時は `ValueError`（EX-06）

## 5. 補助関数

### 5.1 load_fare_data

**関数シグネチャ**: `def load_fare_data() -> tuple[bool, str]`

**目的**: アプリ起動時に1回だけ呼び出し、DATA-011/DATA-012をJSONファイルから静的読み込みし、`TrainFareRecord` Pydanticモデルで検証する

**戻り値**:
- 成功時: `(True, "")` を返す
- 失敗時: `(False, エラーメッセージ文字列)` を返す（例外を再送出しない）

**処理フロー**:
```
開始
  ↓
os.path.exists("data/templates/train_routes.json") でファイル存在チェック
  → ファイルなし → WARNINGログ記録 → (False, "運賃データファイルが見つかりません: data/templates/train_routes.json") を返す
  ↓ （ファイルあり）
"data/templates/train_routes.json" をオープン（encoding="utf-8"）
  ↓
json.load() で読み込み
  ↓
各レコードを TrainFareRecord(**record) でバリデーション
  → ValidationError → ErrorHandler.handle_data_inconsistency(e) → EX-06ログ記録（ERROR）→ (False, エラーメッセージ) を返す
  ↓
_train_fares グローバル変数に格納
  ↓
os.path.exists("data/templates/fixed_fares.json") でファイル存在チェック
  → ファイルなし → WARNINGログ記録 → (False, "固定運賃データファイルが見つかりません: data/templates/fixed_fares.json") を返す
  ↓ （ファイルあり）
"data/templates/fixed_fares.json" をオープン（encoding="utf-8"）
  ↓
json.load() で読み込み
  ↓
_fixed_fares グローバル変数に格納
  ↓
INFO ログ出力: "Fare data loaded: {len(_train_fares)} train routes, {len(_fixed_fares)} fixed fares"
  ↓
(True, "") を返す
```

**エラー処理**:
- ファイル未存在（EX-07）: `os.path.exists()` による事前チェックで検出し、`(False, エラーメッセージ文字列)` を返す。例外の再送出は行わない
- TrainFareRecord バリデーションエラー（EX-06）: `(False, エラーメッセージ文字列)` を返す。例外の再送出は行わない
- ログレベルはいずれも WARNING とする

## 6. 補足情報

### 6.1 実装上の注意点

1. **例外の再送出禁止**
   - ツール関数 `calculate_travel_expense` 内で ValidationError・その他例外を再送出しない
   - 全例外を `try/except` でキャッチし `{"success": False, "message": ...}` を返す
   - `load_fare_data()` も例外を再送出せず、必ず `tuple[bool, str]` 型の戻り値でエラー情報を返す

2. **戻り値形式の統一**
   - データ読み込み系メソッド（`load_fare_data()` 等）のエラー時戻り値は `tuple[bool, str]` 型の `(False, エラーメッセージ文字列)` とする
   - 計算系メソッド（`calculate_travel_expense` 等）のエラー時戻り値は `{"success": False, "message": エラーメッセージ文字列}` の dict 形式とする

3. **ファイル存在チェックの実装方針**
   - ファイル存在チェックは `os.path.exists()` による事前チェックで実装すること
   - `open()` 実行時の `FileNotFoundError` 捕捉との併用は禁止し、事前チェックを優先する

4. **経路テーブル未登録時の例外クラス**
   - 経路テーブルに経路が存在しない場合の例外クラスは `ValueError` とする（`KeyError` は使用しない）

5. **申請者名・申請日の取得方法**
   - `@tool(context=True)` デコレーターを使用し、ツール関数内部で `invocation_state` から申請者名・申請日を取得する
   - `applicant_name`・`application_date` はLLMがツールパラメータとして渡す値ではないため、関数の引数一覧に含めない

6. **Strands SDKのデコレーター**
   - `@tool(context=True)` デコレーターを使用する
   - モジュール: `tools/travel_tools.py`

7. **モジュールレベル変数の初期化**
   - `_train_fares: list[TrainFareRecord] = []` と `_fixed_fares: dict[str, int] = {}` をモジュールレベルで宣言する
   - アプリエントリーポイント（`main.py`）で `load_fare_data()` を呼び出し、戻り値の `(bool, str)` を確認して初期化状態を判定する

---

### 6.2 パフォーマンス考慮事項

1. **静的読み込みによるファイルI/O最小化**
   - DATA-011/DATA-012をアプリ起動時に1回だけ読み込む。ツール実行時は毎回ファイルI/Oしない
   - 実行制御方針 7.1節：TOOL-001のリトライは行わない。冪等性の保証

2. **線形検索の許容**
   - 電車運賃テーブル（DATA-011）の検索は線形探索で実装する。要件上の経路数は少ないため、辞書キャッシュ等の最適化は実施しない

---

### 6.3 セキュリティ考慮事項

1. **ファイルパスのハードコード**
   - `data/templates/train_routes.json` / `data/templates/fixed_fares.json` のパスは定数として定義し、LLMからのパラメータとして受け取らない
   - パス操作によるディレクトリトラバーサルのリスクを排除する

2. **出力ファイルの非生成**
   - 本ツールはファイルを生成しない。計算結果のみを返す

## 8. テスト観点

### 8.1 機能テスト
- 有効な移動日・電車区間（出発駅・到着駅がDATA-011に存在）でツールを呼び出し、`success=True`・正しい運賃（fare）・`calculation_basis="電車経路テーブル参照"` が返ること
- 有効な移動日・バスでツールを呼び出し、`success=True`・`fare=230`・`calculation_basis="固定運賃参照"` が返ること
- 有効な移動日・タクシーでツールを呼び出し、`success=True`・`fare=10000` が返ること
- 有効な移動日・飛行機でツールを呼び出し、`success=True`・`fare=50000` が返ること
- `transport_type="train"` を渡し `"電車"` に正規化されて正常処理されること

### 8.2 異常系テスト
- DATA-011に存在しない電車区間でツールを呼び出し、`success=False`・手動入力促しメッセージが返ること（EX-06、ValueError）
- 対応外の交通手段（`"自転車"`）でツールを呼び出し、`success=False`・エラーメッセージが返ること（EX-01）
- `departure=""` の空文字でツールを呼び出し、`success=False`・バリデーションエラーメッセージが返ること（EX-01）
- `destination=""` の空文字でツールを呼び出し、`success=False`・バリデーションエラーメッセージが返ること（EX-01）
- `travel_date="20260428"` の不正形式でツールを呼び出し、`success=False`・日付形式エラーメッセージが返ること（EX-01）
- `travel_date=""` の空文字でツールを呼び出し、`success=False`・バリデーションエラーメッセージが返ること（EX-01）
- `load_fare_data()` でDATA-011が存在しない場合に `(False, エラーメッセージ)` が返ること（EX-07）。例外が再送出されないこと
- `load_fare_data()` でDATA-012が存在しない場合に `(False, エラーメッセージ)` が返ること（EX-07）。例外が再送出されないこと

### 8.3 性能テスト
- `load_fare_data()` がアプリ起動時に1回だけ呼び出されること（ツール実行時には呼び出されないこと）
- 100件の電車区間検索が1秒以内に完了すること

### 8.4 境界値テスト
- `transport_type` が TRANSPORT_TYPE_MAP に存在しない英語（`"bicycle"`）を渡した場合、Literal バリデーションエラーになること
- `fare=0` の電車区間が存在する場合、`success=True`・`fare=0` が返ること（0円は有効）
- `travel_date` として `"2026-04-28"` のように正確なISO形式を渡した場合に正常に変換されること
- `departure` / `destination` が空白のみ（`"   "`）の場合、strip()後に min_length=1 チェックで ValidationError になること
