## 2. 目的・役割定義

### 2.1 エージェントの目的
- 交通費精算申請に特化し、移動情報を一区間ずつ対話で収集・交通費を自動計算してExcel申請書（下書き）を自動作成する
- Excel生成前に Human-in-the-Loop 承認（OK/修正/キャンセル）を実施し、社員が最終確認した上で申請書を確定する
- 申請期限チェックを含む妥当性チェック結果を提示する

### 2.2 主な責務
- **移動情報の一区間ずつ収集**: 移動日・出発地・目的地・交通手段・業務目的を一区間ずつ収集する（BRL-05）
- **交通費の自動計算**: TOOL-001 を呼び出して各区間の交通費を計算する（BRL-06）
- **駅名正規化**: 社員が入力した駅名を正規化してから TOOL-001 に渡す（BRL-10）
- **申請書自動作成（Excel）**: Human-in-the-Loop 承認（OK）後に TOOL-002（generate_transport_application）を呼び出して申請書（下書き）を生成する（BRL-03）
- **申請内容妥当性チェック**: 申請期限チェック（BRL-07）・高額承認要否判断（BRL-08）・業務目的記載有無チェック（BRL-11）を実施する

### 2.3 非責務（やらないこと）
- **申請種別判断・振り分け**: AG-001 が担当する
- **経費精算申請の処理**: AG-003 が担当する
- **申請書の最終提出**: 人（R-EMP）が担当する

## 3. 自律度・判断権限

### 3.1 自律度レベル

| レベル | 内容 | 適用範囲 |
|------|------|----------|
| 参照 | 社内申請ルール（D-002）・経路テーブル（D-015）・固定運賃マスタ（D-016）の参照 | 申請ルール確認・交通費計算時 |
| 提案 | 不足情報の質問・チェック結果・修正案の提示 | 情報収集・妥当性チェック時 |
| 実行 | 移動情報収集・交通費計算・申請書生成・妥当性チェック | 通常処理 |
| 実行（承認必須） | Excel申請書（下書き）生成（TOOL-002 呼び出し） | Human-in-the-Loop 承認（OK）後のみ（APR-002） |

### 3.2 判断権限の範囲
- **自律判断可能**: 不足情報の判定（JD-07）・申請期限チェック（JD-04）・高額承認要否判断（JD-05）・申請内容妥当性判断（JD-08）
- **人間確認必要**: 申請書（下書き）の内容確認・Human-in-the-Loop 承認（OK/修正/キャンセル）（APR-002）
- **実行不可**: 申請書の最終提出・申請種別の変更

## 4. 処理フロー設計

### 4.1 基本フロー（概要）

```text
開始（invocation_state 受取：申請者名・申請日・セッションID）
→ 移動情報の一区間ずつ収集（CF-003：BRL-05）
  → 各区間の交通費計算（TOOL-001：BRL-06）
  → 不足情報がある場合は再収集（BRL-02、JD-07）
→ 収集済み申請情報をテキストとして整理・提示（ドラフト提示：ツール呼び出しなし）
→ Human-in-the-Loop 承認要求（OK/修正/キャンセル）（APR-002）
  ※ 承認前にツールを実行してはならない
  → OK: 申請書生成ツール呼び出し（TOOL-002：generate_transport_application）（CF-005）
  → 修正: 修正内容を収集して再収集フローへ戻る
  → キャンセル: セッション終了（EX-05）
→ 申請内容妥当性チェック（CF-006：BRL-03、BRL-07、BRL-08、BRL-11）
→ チェック結果・修正案の提示
→ 終了
```

**フローの詳細**: 詳細設計書 2.3章を参照

### 4.2 フロー制御の考え方
- **原則的な処理順序**: 逐次処理（情報収集 → 交通費計算 → ドラフト提示 → 承認 → 申請書生成ツール呼び出し → 妥当性チェック）
- **ドラフト提示とツール呼び出しの分離**: 収集済み申請情報のテキスト整理・提示はツール呼び出しなしで行う。Human-in-the-Loop 承認（OK）後にはじめて TOOL-002 を呼び出す。承認前にツールを実行してはならない
- **処理をスキップする考え方**: 全情報が収集済みの場合は不足情報確認をスキップする（JD-07）
- **同一処理を複数回実行**: 不足情報がある限り情報収集を繰り返す（BRL-02）。修正要望時は情報収集フローへ戻る

### 4.3 状態管理
- **collected_info（D-007）**: 収集済み移動情報を SM-001 経由で追記する（上書き禁止）
- **current_flow**: 現在の会話フロー位置（CF-003・CF-005・CF-006）を SM-001 経由で更新する
- **turn_count**: 会話ターンごとに SM-001 経由でインクリメントする（上限30回、BRL-16）

## 6. ツール設計

### 6.1 利用ツール

各エージェントが使用するツールは、業務機能（計算・生成など）に特化したツールのみとする。

- **TOOL-001（交通費計算ツール）**: 各移動区間の出発地・目的地・交通手段・移動日を入力として交通費を計算する（CF-003 Step3）
- **TOOL-002（generate_transport_application）**: Human-in-the-Loop 承認（OK）後に収集済み移動情報から Excel 申請書（下書き）を生成する（CF-005 Step1）

TOOL-001 は各区間の移動情報収集後に呼び出す。TOOL-002 は Human-in-the-Loop 承認（OK）後のみ呼び出す（APR-002）。承認前にツールを実行してはならない。

**ツールの詳細**: 詳細設計書 2.4.2章および各ツールの設計書を参照

## 7. プロンプト設計

### 7.1 システムプロンプトの目的
- 交通費精算申請エージェントとしての役割（移動情報収集・交通費計算・申請書生成・妥当性チェック）を定義する
- 社内申請ルール（BRL-01〜BRL-16）に基づく処理プロトコルを定義する
- 申請期限チェック（BRL-07）を含む妥当性チェックの判断基準を定義する

### 7.2 プロンプト構成要素
- **役割定義**: 交通費精算申請エージェントとしての役割
- **処理フロー**: CF-003（移動情報収集）→ ドラフト提示（テキスト整理・ツール呼び出しなし）→ Human-in-the-Loop 承認 → CF-005（申請書生成ツール呼び出し）→ CF-006（妥当性チェック）
- **判断プロトコル**: JD-07（不足情報判定）・JD-04（申請期限）・JD-05（高額承認要否）・JD-08（妥当性判断）
- **主な責任**: BRL-05（一区間ずつ収集）・BRL-06（交通費計算）・BRL-10（駅名正規化）・BRL-11（業務目的必須）
- **動的情報**: 申請日（application_date）・申請期限計算用の現在日付（申請期限チェック BRL-07 のため）
- **エラー発生時の振る舞い**: ツールエラー・システム障害等のシステム系エラーが発生した場合は、エラー内容を要約して呼び出し元エージェント（AG-001）に返す

**プロンプト全文**: 詳細設計書 2.3.1章を参照

### 7.3 システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 動的生成関数 |
| 動的生成の理由 | 申請期限チェック（BRL-07）で「申請日 > 経費発生日 + 3ヶ月」を判断するため、実行時の申請日（application_date）をプロンプトに埋め込む必要がある |
| 動的に埋め込む情報 | 申請日（application_date：invocation_state から取得）、申請期限（申請日から3ヶ月後の日付） |

### 7.4 主要な設計意図
- **動的プロンプト生成**: 申請期限チェック（BRL-07）のために申請日・申請期限をプロンプトに埋め込む
- **社内申請ルールのシステムプロンプト埋め込み**: RAG を使用せず、申請ルールをシステムプロンプトに直接埋め込む
- **Human-in-the-Loop 承認の強制**: HD-003 が TOOL-002 呼び出し前に承認を実施し、OK 選択時のみ申請書を生成する

**プロンプト全文**: 詳細設計書 2.3.1章を参照

## 10. 入力データ設計

### 10.1 invocation_stateの入力
受付エージェント（AG-001）から渡される情報:
```python
{
    "applicant_name": str,       # 申請者名（アプリ起動時取得済み）
    "application_date": str,     # 申請日（YYYY-MM-DD、システム日付自動取得）
    "session_id": str            # セッションID（UUID v4形式）
}
```

### 10.2 対話ループでの入力
- **移動情報（D-006）**: 社員が回答する移動日・出発地・目的地・交通手段・業務目的
- **Human-in-the-Loop 承認**: OK / 修正 / キャンセル の3択
- **特殊コマンド**:
  - `reset` / `リセット` / `最初から`: 会話履歴と申請者情報をリセットし、セッションを最初から開始する

## 12. バリデーション設計

**システム設計書の方針**: `artifacts/03_system-design/outputs/バリデーション方針.md` を参照してください。

### 12.1 バリデーション方針
- **移動日**: ISO 8601 形式（YYYY-MM-DD）に変換・検証する（バリデーション方針 4.3節）
- **交通手段**: `Literal["電車", "バス", "タクシー", "飛行機"]` で検証・正規化する（バリデーション方針 7.2節）
- **業務目的**: 空文字列・空白のみは不正（BRL-11、バリデーション方針 6.3節）
- **申請期限**: 申請日が経費発生日から3ヶ月以内であることを検証する（BRL-07、バリデーション方針 4.4節）

**バリデーションの詳細**: 詳細設計書 4章を参照

### 12.2 申請ルール一覧

| ルールID | ルール名 | 条件 | 対応 |
|----------|----------|------|------|
| BRL-01 | 申請ルール参照義務 | 申請書作成・妥当性チェック時 | 社内申請ルール（D-002）をシステムプロンプトから参照して処理する |
| BRL-02 | 不足情報の対話確認 | 申請書作成に必要な情報が不足している場合（JD-07） | 不足情報を社員に質問し、全情報が揃うまで繰り返す |
| BRL-03 | 申請書作成前の妥当性チェック | 申請書（下書き）生成後 | 申請内容の妥当性チェック（JD-04、JD-05、JD-08）を実施し、チェック結果・修正案を提示する |
| BRL-05 | 交通費精算：移動情報の一区間ずつ収集 | 交通費精算申請の情報収集時 | 移動情報（移動日・出発地・目的地・交通手段・業務目的）を一区間ずつ収集する |
| BRL-06 | 交通費精算：交通費の自動計算 | 移動情報収集後 | TOOL-001 を呼び出して交通費を計算する（電車：経路テーブル、バス・タクシー・飛行機：固定運賃） |
| BRL-07 | 申請期限チェック（3ヶ月以内） | 妥当性チェック時 | 申請日が経費発生日から3ヶ月以内であることを確認し、超過時は通知する |
| BRL-08 | 交通費高額申請の上長承認案内 | 交通費合計 > 10,000円（JD-05） | 上長の事前承認が必要である旨を案内する（APR-003） |
| BRL-10 | 駅名の正規化 | 電車の移動情報収集時 | 社員が入力した駅名を正規化してから TOOL-001 に渡す |
| BRL-11 | 業務目的の記載必須 | 申請書作成・妥当性チェック時 | 業務目的が未記載の場合は不足情報として対話で確認する |
| BRL-14 | 申請ルール外の推測・補完禁止 | 全処理 | 申請ルールに定義されていない内容は推測・補完せず、管理部門への問い合わせを案内する |
| BRL-15 | ユーザー入力文字数制限 | ユーザー入力受付時 | 500文字超過の場合は再入力を促す |
| BRL-16 | 対話回数上限 | 全処理 | 対話回数が30回に達した場合はセッションを終了する |

## 13. Agent as Tools設計

### 13.1 ツールとしての公開
- **ツール名**: `transport_application_agent`
- **デコレータ**: `@tool(context=True)`
- **入力**: `query (str)`, `tool_context (ToolContext)`

### 13.2 ToolContextの使用
- invocation_state から `applicant_name`・`application_date`・`session_id` を取得する

- 専門エージェント内部では `session_id` を除外した invocation_state を使用する

**詳細**: 詳細設計書 2.5章を参照

## 14. テスト観点

### 14.1 テスト観点の概要
- **機能テスト**: 移動情報が一区間ずつ収集されること。交通費が正しく計算されること。収集済み申請情報がテキストとして整理・提示されること（ツール呼び出しなし）。Human-in-the-Loop 承認（OK）後にはじめて TOOL-002 が呼び出されること。申請書（下書き）が正しく生成されること。妥当性チェックが正しく実施されること
- **異常系テスト**: 必須情報回答拒否・経路テーブル不存在・テンプレート取得失敗・Human-in-the-Loop 承認でキャンセル時にエラー内容が呼び出し元エージェントへ要約返却されること。承認前に TOOL-002 が呼び出されないこと
- **性能テスト**: 申請書生成が適切な応答時間内に完了すること

**詳細なテストケース**: 詳細設計書 7章を参照

## 16. 制約事項・前提条件

### 16.1 技術的制約
- LLMバックエンドは Amazon Bedrock（Claude Sonnet 4.5）に固定
- strands-agents v1.25.0 を使用する
- TOOL-002 は Human-in-the-Loop 承認（OK）後のみ呼び出し可能（APR-002）

### 16.2 業務的制約
- 申請書の最終提出は人（R-EMP）が行う
- 申請書は下書きのみ生成する
- 対話回数上限：セッション全体で30回（BRL-16）

### 16.3 前提条件
- AG-001 から invocation_state（applicant_name・application_date・session_id）が渡されていること
- data/fare_tables/route_fares.json・fixed_fares.json が存在すること
- data/templates/ に交通費精算申請書テンプレートが存在すること
