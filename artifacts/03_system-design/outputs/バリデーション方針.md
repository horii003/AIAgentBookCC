## 1. 目的・適用範囲

### 1.1 目的

システム全体で一貫したバリデーション方針を定義し、業務ルール違反・入力不正・データ不整合を早期に検出してユーザーに分かりやすくフィードバックする。

### 1.2 適用範囲

- 全エージェント（AG-001/AG-002/AG-003）の入力受付時
- 全ツール（TOOL-001/TOOL-002/image_reader）のパラメータ検証時
- エージェント間連携（invocation_state）の受け渡し時
- 外部データ（fare_routes.json/fare_fixed.json）の読み込み時

### 1.3 本方針で定義すること

- バリデーションの基本原則とタイミング
- Pydantic v2+ を用いたモデルの使用方針
- 本システムで使用するデータ型別バリデーション（日付・金額・文字列・列挙型）
- 業務ルールに基づくカスタムバリデーション
- バリデーションエラーメッセージの方針
- テストケースの観点

### 1.4 本方針で定義しないこと

- 各データモデルの具体的なフィールド定義（データモデル基本設計書で定義）
- 個々のツール関数の内部実装詳細（ツール基本設計書で定義）
- フロントエンド（CLI）側の入力補助・フォーマット

## 2. バリデーションの基本原則

### 2.1 基本方針

- **早期検証**: ツール呼び出し前・エージェント呼び出し前に入力を検証し、無駄な LLM 推論を防ぐ
- **型安全性**: Pydantic v2+ の型アノテーションにより、型エラーを実行時ではなくモデル生成時に検出する
- **ユーザーフレンドリー**: バリデーションエラーメッセージは日本語で記述し、具体的な対処方法を提示する
- **自動正規化**: 列挙型・駅名等の表記揺れは `field_validator` で自動正規化する（BRL-18）
- **一貫性**: バリデーションロジックは `agents/models/data_models.py` に集約し、各コンポーネントで重複定義しない

### 2.2 バリデーションのタイミング

| タイミング | 対象 | 実装場所 |
|----------|------|---------|
| ユーザー入力受付時（FR-012） | 入力文字数（500文字上限） | AG-001 の入力処理前 |
| ツール呼び出し前 | TOOL-001/TOOL-002 のパラメータ | Pydantic モデル（data_models.py） |
| エージェント呼び出し前（invocation_state） | 申請者名・申請日 | invocation_state 受け渡し時 |
| 外部データ読み込み後 | fare_routes.json / fare_fixed.json のスキーマ | TOOL-001 内部のファイル読み込み処理 |
| 申請書生成前（BRL-07） | 収集済み申請情報の必須項目 | AG-002/AG-003 の TOOL-002 呼び出し前 |

### 2.3 バリデーション失敗時の扱い

**ValidationError 発生時**:
- `ErrorHandler` 経由でエラーを処理する（EX-01: 入力例外）
- ユーザー向けの日本語エラーメッセージを生成し、再入力を促す
- スタックトレースは `logs/agent.log` にのみ記録し、ユーザーには提示しない
- ツール関数内では `{"success": False, "message": エラーメッセージ}` の辞書形式で返却する（例外を再送出しない）

## 3. Pydantic モデルの使用方針

### 3.1 基本原則

- Pydantic v2+ の `BaseModel` を継承したモデルクラスを使用する
- `field_validator`（`@classmethod`）で業務ルールに基づくカスタムバリデーションを定義する
- `model_validator` はクロスフィールドバリデーション（申請期限チェック等）に限定して使用する
- モデル設定は `model_config = ConfigDict(str_strip_whitespace=True)` を標準設定とし、文字列の前後空白を自動除去する

### 3.2 Pydantic モデルの定義場所

**定義場所**: `agents/models/data_models.py`

**命名規則**:
- ツールの入力モデル: `{ツール用途}Input`（例: `TransportFareInput`, `ApplicationFormInput`）
- ツールの出力モデル: `{ツール用途}Output`（例: `TransportFareOutput`）
- エージェント間連携: `InvocationStateModel`
- 収集済み申請情報: `TransportItem`, `ExpenseItem`

### 3.3 Pydantic モデルの基本構造

```python
from pydantic import BaseModel, ConfigDict, Field, field_validator, model_validator
from typing import Optional, Literal
from datetime import date


class TransportFareInput(BaseModel):
    model_config = ConfigDict(str_strip_whitespace=True)

    departure: str = Field(..., description="出発地（正規化済み駅名）")
    destination: str = Field(..., description="目的地（正規化済み駅名）")
    transport_type: Literal["電車", "バス", "タクシー", "飛行機"] = Field(
        ..., description="交通手段"
    )
    travel_date: date = Field(..., description="移動日（YYYY-MM-DD）")
    business_purpose: str = Field(..., min_length=1, description="業務目的")

    @field_validator("departure", "destination")
    @classmethod
    def normalize_station_name(cls, v: str) -> str:
        """BRL-18: 駅名末尾の「駅」を除去して正規化する"""
        return v.rstrip("駅") if v.endswith("駅") else v
```

## 4. 日付のバリデーション

### 4.1 基本原則

- 日付は `date` 型（Python `datetime.date`）として管理する
- 文字列入力（`YYYY-MM-DD`）は Pydantic が自動変換する
- 申請期限チェック（BRL-11）は `model_validator` で実装する

### 4.2 日付の定義

**標準形式**: `YYYY-MM-DD`（ISO 8601）

**許容値**:
- 申請日: システム起動時の日付（`date.today()`）を使用する。過去・未来日付の許容/拒否は要件上未定義
- 経費発生日・移動日: 任意の日付を許容する（申請期限チェックは別途実施）

### 4.3 日付バリデーションの実装

```python
from pydantic import field_validator, model_validator
from datetime import date
from dateutil.relativedelta import relativedelta


class TransportItemInput(BaseModel):
    travel_date: date = Field(..., description="移動日（YYYY-MM-DD）")
    application_date: date = Field(..., description="申請日（YYYY-MM-DD）")

    @model_validator(mode="after")
    def check_application_deadline(self) -> "TransportItemInput":
        """BRL-11: 申請期限（移動日から90日以内）チェック"""
        deadline = self.travel_date + relativedelta(days=90)
        if self.application_date > deadline:
            raise ValueError(
                f"申請期限を超過しています。移動日（{self.travel_date}）から90日以内（{deadline}まで）に申請が必要です。"
            )
        return self
```

### 4.4 日付の特殊ルール

| ルール | 内容 | 参照 |
|-------|------|------|
| BRL-11 | 申請日 - 経費発生日 > 90日の場合は警告を提示する | GRD-004 |
| 3ヶ月の解釈 | 90日として計算する（`python-dateutil` の `relativedelta` を使用） | BRL-11 補足 |

## 5. 金額のバリデーション

### 5.1 基本原則

- 金額は `int` 型（円単位）として管理する
- 0以下の値は不正とする
- 高額申請閾値（BRL-12）は `field_validator` で警告フラグを設定する（申請をブロックしない）

### 5.2 金額の定義

**標準形式**: 非負整数（円単位）

**許容値**:
- 下限: 1（0以下は不正）
- 上限: 要件上未定義（GRD-005/006 の閾値超過は警告のみ、ブロックなし）

**高額判定閾値**:
- 交通費精算: 10,000円超（JD-10/BRL-12/GRD-005）
- 経費精算: 5,000円超（JD-11/BRL-12/GRD-006）

### 5.3 金額バリデーションの実装

```python
from pydantic import field_validator


class ExpenseItemInput(BaseModel):
    amount: int = Field(..., gt=0, description="金額（円単位、1以上）")
    expense_category: Literal["事務用品費", "宿泊費", "資格精算費", "その他経費"]

    @field_validator("amount")
    @classmethod
    def check_high_amount(cls, v: int) -> int:
        """BRL-12/GRD-006: 経費精算高額申請の確認（5,000円超は警告対象）"""
        # バリデーションはパスさせ、エージェントが警告メッセージを提示する
        # ここでは型チェックのみ実施
        return v
```

### 5.4 金額の特殊ルール

| ルール | 交通費精算 | 経費精算 |
|-------|----------|---------|
| 高額申請警告閾値 | 10,000円超（JD-10） | 5,000円超（JD-11） |
| 対応 | エージェントが案内提示（申請はブロックしない） | エージェントが案内提示（申請はブロックしない） |

## 6. 文字列のバリデーション

### 6.1 基本原則

- 必須文字列フィールドは `min_length=1` を設定して空文字を拒否する
- CLI 入力は 500 文字上限（FR-012/GRD-017）を AG-001 入力受付時にチェックする
- `model_config = ConfigDict(str_strip_whitespace=True)` で前後の空白を自動除去する

### 6.2 文字列の定義

**標準形式**: UTF-8 文字列

**許容値**:

| フィールド | 最小長 | 最大長 | 備考 |
|----------|-------|-------|------|
| CLI 入力（FR-012） | 1 | 500 | GRD-017 対応 |
| 申請者名 | 1 | 要件上未定義 | 空欄禁止 |
| 業務目的 | 1 | 要件上未定義 | BRL-13/GRD-007 対応 |
| 店舗名・品目 | 1 | 要件上未定義 | 空欄禁止 |

### 6.3 文字列バリデーションの実装

```python
from pydantic import Field


class ApplicationInput(BaseModel):
    model_config = ConfigDict(str_strip_whitespace=True)

    # CLI 入力の文字数上限チェック（FR-012）
    user_input: str = Field(..., min_length=1, max_length=500)

    # 業務目的（BRL-13）
    business_purpose: str = Field(..., min_length=1, description="業務目的（必須）")

    # 申請者名
    applicant_name: str = Field(..., min_length=1, description="申請者名（必須）")
```

### 6.4 文字列の特殊ルール

| ルール | 内容 | 参照 |
|-------|------|------|
| FR-012/GRD-017 | CLI 入力は 500 文字以内。超過時はエラーを提示し再入力を促す | EX-01 |
| BRL-13/GRD-007 | 業務目的は必須。空欄の場合は申請書生成をブロックし記載を促す | — |

## 7. 列挙型のバリデーション

### 7.1 基本原則

- 列挙型フィールドは `Literal` 型で定義し、受け入れ可能な値を明示する
- LLM が渡す表記バリエーション（略称・別表記）は `field_validator` で正規化する
- 正規化後の値を後続処理で使用し、入力値をそのまま使用しない

### 7.2 列挙型の定義

| フィールド | 許容値 | 正規化対象 |
|----------|-------|---------|
| 申請種別 | `交通費精算申請`, `経費精算申請` | 「交通費」→「交通費精算申請」等 |
| 交通手段 | `電車`, `バス`, `タクシー`, `飛行機` | 「train」→「電車」等 |
| 経費区分 | `事務用品費`, `宿泊費`, `資格精算費`, `その他経費` | 「文房具」→「事務用品費」はエージェント判断（BRL-09） |

### 7.3 列挙型バリデーションの実装

```python
from typing import Literal
from pydantic import field_validator


class ApplicationTypeInput(BaseModel):
    application_type: Literal["交通費精算申請", "経費精算申請"]

    @field_validator("application_type", mode="before")
    @classmethod
    def normalize_application_type(cls, v: str) -> str:
        """申請種別の表記揺れを正規化する"""
        mapping = {
            "交通費": "交通費精算申請",
            "交通費申請": "交通費精算申請",
            "経費": "経費精算申請",
            "経費申請": "経費精算申請",
        }
        return mapping.get(v, v)


class TransportTypeInput(BaseModel):
    transport_type: Literal["電車", "バス", "タクシー", "飛行機"]

    @field_validator("transport_type", mode="before")
    @classmethod
    def normalize_transport_type(cls, v: str) -> str:
        """交通手段の表記揺れを正規化する"""
        mapping = {
            "train": "電車",
            "鉄道": "電車",
            "bus": "バス",
            "taxi": "タクシー",
            "cab": "タクシー",
            "airplane": "飛行機",
            "plane": "飛行機",
            "飛行機": "飛行機",
        }
        return mapping.get(v, v)
```

## 8. カスタムバリデーションルール

### 8.1 駅名正規化（BRL-18）

```python
@field_validator("departure", "destination", mode="before")
@classmethod
def normalize_station_name(cls, v: str) -> str:
    """BRL-18: 駅名末尾の「駅」を除去して正規化する（例: 渋谷駅 → 渋谷）"""
    if isinstance(v, str) and v.endswith("駅"):
        return v[:-1]
    return v
```

### 8.2 全必要情報収集チェック（BRL-07/GRD-008）

```python
from pydantic import model_validator


class ApplicationFormReadyCheck(BaseModel):
    applicant_name: str = Field(..., min_length=1)
    application_date: date
    application_type: Literal["交通費精算申請", "経費精算申請"]
    items: list  # TransportItem または ExpenseItem のリスト

    @model_validator(mode="after")
    def check_all_required_fields(self) -> "ApplicationFormReadyCheck":
        """BRL-07/GRD-008: 全必要情報が収集済みであることを確認する"""
        if not self.items:
            raise ValueError(
                "申請情報が収集されていません。移動情報または経費情報を少なくとも1件入力してください。"
            )
        for item in self.items:
            if not getattr(item, "business_purpose", "").strip():
                raise ValueError(
                    "業務目的が未記入の申請情報があります。全ての申請情報に業務目的を記載してください。（BRL-13）"
                )
        return self
```

## 9. バリデーションエラーメッセージ

### 9.1 基本原則

- **わかりやすさ**: 技術的な詳細を隠蔽し、業務的な言葉で記述する
- **具体性**: エラーの原因と具体的な対処方法を提示する
- **一貫性**: エラーメッセージは `ErrorHandler` を経由して日本語に変換し、統一したフォーマットで提示する

### 9.2 エラーメッセージの形式

**基本形式**:
```
{フィールド名}のエラー: {エラー内容}
対処方法: {対処方法}
```

### 9.3 エラーメッセージ例

| バリデーション対象 | エラーメッセージ |
|----------------|----------------|
| 入力文字数超過（FR-012） | 入力文字数が500文字を超えています。500文字以内に短縮してから再度入力してください。 |
| 申請期限超過（BRL-11） | 申請期限を超過しています。経費発生日から90日以内に申請が必要です。管理部門（経理部）にご相談ください。 |
| 業務目的未記入（BRL-13） | 業務目的が入力されていません。業務目的を入力してから申請書を生成してください。 |
| 申請種別不正 | 申請種別が不正です。「交通費精算申請」または「経費精算申請」を選択してください。 |
| 交通手段不正 | 交通手段が不正です。「電車」「バス」「タクシー」「飛行機」のいずれかを選択してください。 |
| 経費区分不正 | 経費区分が不正です。「事務用品費」「宿泊費」「資格精算費」「その他経費」のいずれかを選択してください。 |
| 金額不正（0以下） | 金額は1円以上を入力してください。 |

### 9.4 バリデーションエラーの実装

```python
from pydantic import ValidationError


def handle_validation_error(e: ValidationError) -> str:
    """Pydantic ValidationError を日本語エラーメッセージに変換する"""
    messages = []
    for error in e.errors():
        field = " > ".join(str(loc) for loc in error["loc"])
        msg = error["msg"]
        messages.append(f"{field}のエラー: {msg}")
    return "\n".join(messages)
```

## 10. バリデーションのテスト

### 10.1 テスト観点

- 正常系（有効な入力値が受け入れられること）
- 異常系（無効な入力値が拒否されること）
- 境界値（上限・下限の値が正しく扱われること）
- 正規化（表記揺れが正規化されること）
- エラーメッセージ（日本語メッセージが正しく返されること）

### 10.2 テストケース例

| テストID | テストケース | 入力 | 期待結果 |
|---------|------------|------|---------|

| VAL-003 | 境界値: 申請期限ちょうど90日 | `travel_date - application_date = 90` | 変換成功（警告なし） |

| VAL-006 | 異常系: 金額0 | `amount=0` | ValidationError |
| VAL-007 | 正規化: 駅名末尾「駅」あり | `departure="渋谷駅"` | `departure="渋谷"`（正規化後） |

## 11. 設計上の留意点・リスク

### 11.1 LLM 出力のバリデーション

**リスク**: LLM が生成した申請情報が正しい型・形式でない場合にバリデーションエラーが頻発する

**対策**: `field_validator` の `mode="before"` を使用して LLM 出力の表記揺れを正規化してからバリデーションを実施する。正規化でも対応できない場合はエージェントが再確認ダイアログを実施する

### 11.2 外部データスキーマ不整合

**リスク**: `fare_routes.json` / `fare_fixed.json` のスキーマが変更された場合にバリデーションエラーが発生する（EX-06: データ不整合）

**対策**: ファイル読み込み後に Pydantic モデルでスキーマ検証を実施する。検証失敗時は EX-06 として処理し、管理部門へのエスカレーションを案内する

### 11.3 日付の3ヶ月解釈

**リスク**: BRL-11 の「3ヶ月」の解釈が暦月か日数かで結果が異なる

**対策**: 本バリデーション方針では 90 日として実装する（`relativedelta(days=90)` を使用）。業務要件変更時は `data_models.py` の定数を変更する
