## 2. 会話管理の方針

### 2.1 基本原則

- **会話履歴の保持**: エージェントと社員の会話ターン（メッセージ）履歴をウィンドウサイズの範囲でインメモリ保持する
- **メモリ効率**: ウィンドウ超過時は古いメッセージを切り落とす（`should_truncate_results=True`）。会話履歴はセッション状態ファイルに保存しない
- **コンテキストの継続性**: 会話履歴はメッセージ単位（`per_turn=False`）でカウントし、ウィンドウサイズ内の全メッセージを LLM に渡す

### 2.2 ウィンドウサイズの基準

| エージェント種別 | ウィンドウサイズ | 理由 |
|----------------|----------------|------|
| オーケストレーター（AG-001） | 30 | 申請種別判断・専門エージェント委譲まで複数ターンの対話を保持する必要があるため |
| 専門エージェント AG-002 | 20 | 情報収集・計算・確認の複数ステップ処理を行うため。オーケストレーターより小さいサイズで十分 |
| 専門エージェント AG-003 | 15 | 経費情報収集は比較的シンプルな対話フローのため、AG-002よりさらに小さいサイズで十分 |
| ツール（TOOL-001/TOOL-002） | - | ツールは会話管理を持たない（呼び出し元エージェントの会話コンテキストを利用） |

### 2.3 会話管理の実装

**使用するクラス**: `SlidingWindowConversationManager`（Strands Agents 提供）

**設定方法**:
```python
from strands.agent.conversation_manager import SlidingWindowConversationManager

conversation_management = SlidingWindowConversationManager(
    window_size=30,  # AG-001の場合
    should_truncate_results=True
)
```

**設定パラメータ**:
- `window_size`: LLMに渡す会話履歴の保持件数（メッセージ数）。AG-001は30、AG-002は20、AG-003は15
- `should_truncate_results`: ウィンドウ超過時に古いメッセージを切り落とすかどうか（`True` 固定）
- `per_turn`: ターン単位でカウントするかどうか（`False` 固定 = メッセージ単位）

## 3. リトライ戦略の方針

### 3.1 基本原則

- **一時的なエラーへの対応**: ネットワーク障害・APIタイムアウト等の一時的なエラーに対してリトライを行い、システムの堅牢性を確保する
- **指数バックオフ**: リトライ間隔は指数バックオフで延長し、システムへの過負荷を防ぐ（初期遅延4秒、最大遅延240秒）
- **最大試行回数の制限**: リトライ上限を設けて無限リトライを防止し、上限到達後は ErrorHandler に委譲する

### 3.2 リトライ対象

| 対象 | リトライ可否 | 最大試行回数 | 理由 |
|-----|------------|------------|------|
| Amazon Bedrock（LLM呼び出し） | ○ | 6回 | 接続失敗・タイムアウトは一時的なエラーのため |
| TOOL-001（交通費計算） | ○ | 6回 | 参照のみで冪等性が保証されるため |
| TOOL-002（申請書生成） | △（1回のみ） | 1回 | 非冪等（ファイル生成）のため重複生成防止 |
| 入力例外（EX-01）・業務ルール違反（EX-03） | × | 0回 | 業務的な理由によるため再試行しても解決しない |

### 3.3 リトライ戦略の実装

**使用するクラス**: Strands Agents のデフォルトリトライ設定（`boto3`/`botocore` のリトライ機構）を使用

**設定方法**:
```python
# config/model_config.py に集約する
RETRY_CONFIG = {
    "max_attempts": 6,
    "initial_delay": 4,   # 秒
    "max_delay": 240,     # 秒
    "multiplier": 2       # 指数バックオフの乗数
}
```

**詳細**: 実行制御方針（`artifacts/03_system-design/outputs/実行制御方針.md`）の再試行方針を参照

## 4. ループ制御の方針

### 4.1 基本原則

- **無限ループの防止**: 全エージェントに最大ループ回数を設定し、ReActループの無限繰り返しを防止する
- **処理の可視化**: LoopControlHook がループ回数を監視し、ループ回数をログ（OPE-004）に記録する
- **ユーザーへの通知**: LoopLimitError 到達時は ErrorHandler が EX-08 として処理し、CF-006 経由で社員に通知する

### 4.2 最大ループ回数の基準

| エージェント種別 | 最大ループ回数 | 理由 |
|----------------|--------------|------|
| 全エージェント（AG-001/AG-002/AG-003） | 10回 | 申請フローの全ステップをカバーするのに十分な回数。無限ループを防止するための安全上限 |

### 4.3 ループ制御の実装

**使用するクラス**: `LoopControlHook`（カスタム実装、`handlers/loop_control_hook.py`）

**設定方法**:
```python
from handlers.loop_control_hook import LoopControlHook

loop_control_hook = LoopControlHook(max_iterations=10)
```

**設定パラメータ**:
- `max_iterations`: ReActループの最大実行回数（全エージェント共通で10回）

**登録イベント**:
- `BeforeInvocationEvent`: エージェント呼び出し開始時にループカウンタを0にリセットする
- `AfterModelCallEvent`: LLM呼び出し後にループカウンタをインクリメントし、上限（10回）到達を監視する

**詳細**: ハンドラー基本設計_HD-002.md を参照

## 5. セッション管理の方針

### 5.1 基本原則

- **会話履歴の永続化**: 業務コンテキスト（申請者名・申請種別・収集済み申請情報等）を `data/sessions/{session_id}.json` に永続化する。会話履歴はインメモリのみで保持し、ファイルには保存しない
- **セッションIDの一意性**: セッションID は日時 + UUID の組み合わせで生成し、セッション間の衝突を防止する
- **エージェント間の共有**: セッションID は `invocation_state` 経由で専門エージェントに伝播し、LLMのプロンプトには含めない
- **保持期間**: セッション状態ファイルはセッション終了時（CLOSED/TERMINATED）に削除する

### 5.2 セッションIDの形式

**形式**: `{YYYYMMDD}_{HHMMSS}_{uuid4[:8]}`

**例**: `20260502_143022_a3f9b1c2`

### 5.3 セッション管理の実装

**使用するクラス**: `SessionManager`（カスタム実装、`session/session_manager.py`）

**設定方法**:
```python
from session.session_manager import SessionManager

session_manager = SessionManager(
    session_id=session_id,
    storage_path="data/sessions"
)
```

**詳細**: セッション管理方針（`artifacts/03_system-design/outputs/セッション管理方針.md`）を参照

## 6. 承認フックの方針

### 6.1 基本原則

- **重要な操作の承認**: 申請書（下書き）生成前に社員確認を必ず取得する（GRD-014）。承認なしの申請書生成は絶対禁止
- **ユーザーの選択肢**: 社員は OK / 修正 / キャンセル の3択から選択できる。選択に応じて処理を分岐する
- **誤操作の防止**: 申請書提出操作はシステムが実行しない（GRD-012 / ACT-EXEC-01）。提出は社員が手動で実施する

### 6.2 承認対象ツールの基準

| ツール種別 | 承認要否 | 理由 |
|----------|---------|------|
| TOOL-002（申請書生成） | 要 | 申請書（下書き）の生成は不可逆的な出力操作のため事前確認が必要（GRD-014） |
| TOOL-001（交通費計算） | 不要 | 参照のみの操作であり、出力・変更を伴わないため |
| AG-001（委譲呼び出し） | 不要 | エージェント委譲はルーティング操作のみで、永続的な変更を伴わないため |

### 6.3 承認フックの実装

**使用するクラス**: `HumanApprovalHook`（カスタム実装、`handlers/human_approval_hook.py`）

**設定方法**:
```python
from handlers.human_approval_hook import HumanApprovalHook

human_approval_hook = HumanApprovalHook(
    approval_callback=approval_callback
)
```

**設定パラメータ**:
- `approval_callback`: 社員への確認提示と OK/修正/キャンセル 選択結果の取得を行うコールバック関数

**適用対象**: AG-002（交通費精算申請エージェント）・AG-003（経費精算申請エージェント）のみ

**詳細**: ハンドラー基本設計_HD-003.md を参照

## 7. 設定の適用方法

### 7.1 エージェント作成時の設定

**専門エージェント（AG-002/AG-003）の基本パターン**:
```python
from strands import Agent
from strands.agent.conversation_manager import SlidingWindowConversationManager
from handlers.loop_control_hook import LoopControlHook
from handlers.human_approval_hook import HumanApprovalHook
from session.session_manager import SessionManager

def _get_transport_agent(session_id: str) -> Agent:
    return Agent(
        model=model,
        system_prompt=TRANSPORT_AGENT_SYSTEM_PROMPT,
        tools=[transport_expense_tool, output_generator_tool],
        conversation_manager=SlidingWindowConversationManager(
            window_size=20,  # AG-002
            should_truncate_results=True
        ),
        hooks=[
            HumanApprovalHook(approval_callback=approval_callback),
            LoopControlHook(max_iterations=10)
        ],
        session_manager=SessionManager(
            session_id=session_id,
            storage_path="data/sessions"
        ),
        agent_id="transport_agent",
        name="交通費精算申請エージェント"
    )
```

**オーケストレーター（AG-001）の基本パターン**:
```python
def _get_orchestrator_agent(session_id: str) -> Agent:
    return Agent(
        model=model,
        system_prompt=ORCHESTRATOR_SYSTEM_PROMPT,
        tools=[transport_agent_tool, expense_agent_tool],
        conversation_manager=SlidingWindowConversationManager(
            window_size=30,  # AG-001
            should_truncate_results=True
        ),
        hooks=[
            LoopControlHook(max_iterations=10)
        ],
        session_manager=SessionManager(
            session_id=session_id,
            storage_path="data/sessions"
        ),
        agent_id="orchestrator_agent",
        name="申請受付窓口エージェント"
    )
```

### 7.2 設定のカスタマイズ

**カスタマイズが必要な場合**:
- エージェント固有のウィンドウサイズ（AG-001: 30、AG-002: 20、AG-003: 15）は各エージェント基本設計書で定義する
- 承認フックの適用有無はエージェント種別により異なる（AG-001は不要、AG-002/AG-003は必須）
- モデル設定（モデルID・ガードレール設定・リトライ設定）は `config/model_config.py` に集約する

**カスタマイズ方法**: 各エージェントの基本設計書（`artifacts/04_basic-design/outputs/エージェント基本設計_AG-00*.md`）を参照

## 8. 設計上の留意点・リスク

### 8.1 会話管理

**リスク**: ウィンドウサイズを超えた会話履歴が切り落とされることで、エージェントが過去のコンテキストを参照できなくなる可能性がある

**対策**: 業務コンテキスト（申請者名・申請種別・収集済み申請情報）は `data/sessions/{session_id}.json` に別途保存し、会話履歴の消失による影響を軽減する

### 8.2 ループ制御

**リスク**: LoopLimitError 到達時に ErrorHandler に委譲されず、エラーが握り潰される可能性がある

**対策**: LoopControlHook は EventLoopException（`__cause__` で LoopLimitError を判定）を検知して ErrorHandler に委譲する。全エージェントで LoopControlHook を必須適用する

### 8.3 セッション管理

**リスク**: 複数エージェント（AG-001/AG-002/AG-003）が同一 `session_id` のファイルを参照する際に状態不整合が発生する可能性がある

**対策**: applicant_name・application_type は AG-001 が確定後に固定し、AG-002/AG-003 は変更しない。各専門エージェントは自身のセッション状態フィールドのみを更新する

### 8.4 承認フック

**リスク**: HumanApprovalHook の未適用により GRD-014（確認未取得禁止）を違反するリスク

**対策**: AG-002/AG-003 の全インスタンスに HumanApprovalHook を必須適用する。ファクトリ関数 `_get_{agent_name}_agent(session_id)` 内で hooks リストに必ず含める

## 9. ログ設計方針

### 9.1 基本原則

- **エントリーポイントで一元管理**: ログ設定（レベル・出力先・フォーマット）は `main.py` で一元管理する
- **各モジュールが責任を持って出力**: ログ出力は各モジュール（エージェント・ツール・ハンドラー）が責任を持って行う
- **LOG_LEVEL 環境変数で制御**: ログレベルは環境変数 `LOG_LEVEL` で制御し、未設定時のデフォルトは `WARNING`
- **ユーザー向けメッセージと技術ログの分離**: 社員に表示するメッセージと技術ログ（スタックトレース等）を明確に分離する
- **設計判断の記録**: 各モジュールが付加するコンテキスト情報の詳細は基本設計・詳細設計で定義する

## 10. invocation_state フィールド仕様

### 10.1 オーケストレーター → 専門エージェント（ToolContext 経由）

| フィールド名 | 型 | 内容 |
|------------|---|------|
| `applicant_name` | str | 申請者名（アプリケーション起動時に取得済み） |
| `application_date` | str | 申請日（YYYY-MM-DD 形式、システム日付から自動取得） |
| `session_id` | str | セッションID（`{YYYYMMDD}_{HHMMSS}_{uuid4[:8]}` 形式） |

### 10.2 専門エージェント内部でエージェントに渡す invocation_state

| フィールド名 | 型 | 内容 |
|------------|---|------|
| `applicant_name` | str | 申請者名 |
| `application_date` | str | 申請日（YYYY-MM-DD 形式） |

> ※ `session_id` は専門エージェント内部の下位エージェントへは引き継がない（本システムでは AG-002/AG-003 からのさらなる委譲はないため）

## 11. 申請者情報の取得タイミング

### 11.1 申請者名

- 申請者名（`applicant_name`）はユーザーとの対話で収集するのではなく、**アプリケーション起動時（対話ループ開始前）に取得**し、エージェントの初期化パラメータとして渡す
- 取得方法: `main.py` でのプロンプト入力または環境変数から取得する（実装詳細は詳細設計で定義）

### 11.2 申請日

- 申請日（`application_date`）はユーザーとの対話で収集するのではなく、**システム日付（実行時の日付、YYYY-MM-DD 形式）を自動取得**する
- 取得方法: `date.today().isoformat()` で取得し、`invocation_state` 経由でエージェントに渡す
