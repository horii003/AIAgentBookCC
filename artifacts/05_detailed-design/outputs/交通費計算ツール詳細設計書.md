## 1. 概要

### 1.1 コンポーネントの目的

出発地・目的地・交通手段・移動日の情報から正確な区間運賃を算出してAG-002に返す。電車は `template/train_routes.json` を辞書キー検索して区間運賃を取得し、バス・タクシー・飛行機は `template/fixed_fares.json` の固定運賃を辞書キー検索で参照する。LLMの文章生成では正確な運賃を保証できないため、確定的なデータ参照による計算を提供する。

### 1.2 主要な責務

- **電車運賃の検索**: `template/train_routes.json`（辞書形式）から出発駅・到着駅をキーとして区間運賃を検索する
- **固定運賃の参照**: `template/fixed_fares.json`（辞書形式）からバス・タクシー・飛行機の固定運賃を辞書キー検索で参照する
- **申請者名・申請日の取得**: LLMがパラメータとして渡すのではなく、ツール関数内部で `invocation_state` から取得する
- **入力バリデーション**: `TransportExpenseCalculatorInput` Pydanticモデルで移動日・出発地・目的地・交通手段を検証する
- **エラー結果の返却**: 経路未発見・対応外交通手段の場合は `{"success": False, "message": ...}` を返す。例外を raise しない

## 3. calculate_transport_expenseツール設計

### 3.1 基本情報

#### ツール名（関数名）
`calculate_transport_expense`

#### 説明（docstring）
```
1区間分の移動情報（移動日・出発地・目的地・交通手段）から区間運賃（円）を算出する。

電車の場合は template/train_routes.json の経路辞書をキー検索し、バス・タクシー・飛行機は
template/fixed_fares.json の固定運賃辞書をキー検索する。経路が見つからない場合は success=False を返す。
申請者名・申請日はツール関数内部で invocation_state から取得する（LLMがパラメータとして渡さない）。

Args:
    transport_date (str): 移動日（YYYY-MM-DD形式）。申請期限チェック（BRL-13）に使用。
    departure (str): 出発地（正規化済み駅名・地名）。
    destination (str): 目的地（正規化済み駅名・地名）。
    transport_type (str): 交通手段。「電車」「バス」「タクシー」「飛行機」または英語表記・別表記
        （例: "train" → "電車", "taxi" → "タクシー"）を正規化して受け入れる。

Returns:
    dict: {
        "success": bool,
        "fare": int,              # 成功時: 区間運賃（円）
        "calculation_basis": str, # 成功時: 計算根拠
        "message": str            # 失敗時: ユーザー向けエラーメッセージ
    }
```

| 引数名 | 型 | 説明 | デフォルト値 |

| `transport_type` | `str` | 交通手段（「電車」「バス」「タクシー」「飛行機」または英語・別表記） | なし（必須） |

### 3.3 ビジネスロジック

#### 3.3.1 処理フロー

```
開始
  ↓
invocation_state から applicant_name・application_date を取得
  ↓
入力バリデーション（TransportExpenseCalculatorInput.model_validate()）
  ↓
  ValidationError → ErrorHandler.handle_validation_error() → {"success": False, "message": ...} を返す
  ↓
交通手段の判定
  ↓
  「電車」の場合
    ↓
    train_routes_data（辞書）から (departure, destination) をキーとして検索
    ↓
    キー発見
      → {"success": True, "fare": fare, "calculation_basis": "電車経路テーブル参照"} を返す
    ↓
    キー未発見（E-001）
      → {"success": False, "message": "申し訳ありません。指定された経路の運賃データが見つかりませんでした。交通費を手動で入力してください。"} を返す
  ↓
  「バス」「タクシー」「飛行機」の場合
    ↓
    fixed_fares_data（辞書）から交通手段名をキーとして固定運賃を取得
      - キー: "バス" → 230, "タクシー" → 10000, "飛行機" → 50000
    ↓
    {"success": True, "fare": fare, "calculation_basis": "固定運賃参照"} を返す
  ↓
  上記以外（正規化後も許容値外）
    → バリデーション段階で排除済みのため到達しない
```

#### 3.3.2 起動時データ読み込みキャッシュ

**FareDataLoader クラス（データ読み込み担当）**:

**用途**: アプリ起動時に `template/train_routes.json` と `template/fixed_fares.json` をJSONファイルから読み込み、辞書形式でメモリに保持する。以後はメモリ上の辞書を参照する。

**データ構造**:
- `train_routes_data`: `dict[tuple[str, str], int]` 形式（`{("出発地", "目的地"): 運賃}` の辞書）
- `fixed_fares_data`: `dict[str, int]` 形式（`{"バス": 230, "タクシー": 10000, "飛行機": 50000}` の辞書）

**load_train_routes メソッド（戻り値: `tuple[bool, str]`）**:
```
1. os.path.exists("./template/train_routes.json") でファイル存在チェックを行う
2. ファイルが存在しない場合: WARNING ログ出力後に (False, エラーメッセージ文字列) を返す
3. ファイルが存在する場合: json.load() で読み込み、辞書形式に変換して (True, "") を返す
4. JSONDecodeError / Exception 発生時: WARNING ログ出力後に (False, エラーメッセージ文字列) を返す
```

**load_fixed_fares メソッド（戻り値: `tuple[bool, str]`）**:
```
1. os.path.exists("./template/fixed_fares.json") でファイル存在チェックを行う
2. ファイルが存在しない場合: WARNING ログ出力後に (False, エラーメッセージ文字列) を返す
3. ファイルが存在する場合: json.load() で辞書形式として読み込み (True, "") を返す
4. JSONDecodeError / Exception 発生時: WARNING ログ出力後に (False, エラーメッセージ文字列) を返す
```

> **実装上の注意**: ファイル存在チェックは `os.path.exists()` による事前チェックで実装する。`open()` 実行時の `FileNotFoundError` 捕捉との併用は禁止し、事前チェックを優先する。

#### 3.3.3 データの検証方法

**Pydanticモデル**: `TransportExpenseCalculatorInput`

**検証内容**:
- `transport_date`: `parse_date` field_validator（mode="before"）でYYYY-MM-DD文字列を `datetime.date` 型に変換する。変換失敗時は ValueError
- `departure`: `min_length=1` の Field 制約。前後空白は strip() で除去（mode="before"）
- `destination`: `min_length=1` の Field 制約。前後空白は strip() で除去（mode="before"）
- `transport_type`: `normalize_transport_type` field_validator（mode="before"）で `TRANSPORT_TYPE_MAP` による正規化を行い、その後 `Literal["電車", "バス", "タクシー", "飛行機"]` で値を制限する

### 3.4 エラーハンドリング

#### 3.4.1 エラー処理フロー

```
エラー発生
  ↓
ValidationError（TransportExpenseCalculatorInput）
  ↓
  呼び出し元でログ出力後、ErrorHandler.handle_validation_error(e) を呼び出す
  → {"success": False, "message": ユーザー向けメッセージ} を返す
  ↓
電車経路未発見（train_routes_data にキーなし）
  ↓
  ValueError を発生させ内部でキャッチ
  → {"success": False, "message": "申し訳ありません。指定された経路の運賃データが見つかりませんでした。交通費を手動で入力してください。"} を返す
  ↓
データ読み込み失敗（FileNotFoundError / Exception）
  ↓
  WARNINGレベルでログ出力後、ErrorHandler.handle_fare_data_error(e) を呼び出す
  → (False, エラーメッセージ文字列) を返す（load系メソッドの戻り値形式）
  ↓
運賃計算失敗（Exception）
  ↓
  呼び出し元でログ出力後、ErrorHandler.handle_calculation_error(e) を呼び出す
  → {"success": False, "message": ユーザー向けメッセージ} を返す（計算系メソッドの戻り値形式）
  ↓
想定外の例外（Exception）
  ↓
  呼び出し元でERRORレベル（exc_info=True）ログ出力後、ErrorHandler.handle_unexpected_error(e) を呼び出す
  → {"success": False, "message": "申し訳ありません。予期しないエラーが発生しました。システム管理者にご連絡ください。"} を返す
```

#### 3.4.2 例外テーブル

| 例外クラス | 発生条件 | 戻り値形式 | ユーザー向けメッセージ |
|-----------|---------|-----------|---------------------|
| `ValidationError` | TransportExpenseCalculatorInput バリデーション失敗 | `{"success": False, "message": str}` | `"申請情報に不足している項目があります。{フィールド名}を入力してください。"` |
| `ValueError` | 電車経路テーブルにキーが存在しない場合 | `{"success": False, "message": str}` | `"申し訳ありません。指定された経路の運賃データが見つかりませんでした。交通費を手動で入力してください。"` |

### 5.3 データモデル

#### TransportExpenseCalculatorInputモデル

| フィールド名 | 型 | 必須/任意 | 説明 | 制約 |
|--------------|-----|-----------|------|------|
| `transport_date` | `datetime.date` | 必須 | 移動日。`parse_date` field_validator（mode="before"）で文字列から変換 | YYYY-MM-DD形式。変換失敗時は ValueError |
| `departure` | `str` | 必須 | 出発地（正規化済み駅名・地名） | `min_length=1`。前後空白はstrip()で除去 |
| `destination` | `str` | 必須 | 目的地（正規化済み駅名・地名） | `min_length=1`。前後空白はstrip()で除去 |
| `transport_type` | `Literal["電車", "バス", "タクシー", "飛行機"]` | 必須 | 交通手段。`normalize_transport_type` field_validator（mode="before"）で正規化 | 正規化後に列挙値のいずれかであること |

**バリデーション**:
- `parse_date`: transport_date を `datetime.date` 型に変換する。`dateutil.parser.parse(v).date()` を使用。変換失敗時は ValueError
- `normalize_transport_type`: `TRANSPORT_TYPE_MAP` によるマッピングで正規形に変換する。マッピング外の値はそのまま Literal バリデーションに渡す

## 6. 補足情報

### 6.1 実装上の注意点

1. **申請者名・申請日は invocation_state から取得**
   - `applicant_name`・`application_date` はLLMがツールパラメータとして渡す値ではなく、ツール関数内部で `invocation_state` から取得する
   - `@tool(context=True)` デコレータを使用し、`tool_context.invocation_state.get("applicant_name", "")` で取得する

2. **ファイル存在チェックは os.path.exists() による事前チェック**
   - `open()` 実行時の `FileNotFoundError` 捕捉との併用は禁止し、事前チェックを優先する

3. **例外の raise 禁止**
   - ツール関数内で例外を raise しない。すべての例外をキャッチし `{"success": False, "message": ...}` を返す（例外処理方針 2節）
   - データ読み込み系メソッドは `(False, エラーメッセージ文字列)` のタプルを返す

4. **データ構造は辞書形式、検索は辞書キー検索**
   - `train_routes.json`・`fixed_fares.json` はどちらも辞書形式で読み込む
   - 検索は線形探索ではなく辞書キー検索（O(1)）を使用する

5. **冪等性の保証**
   - 同一入力に対して常に同一結果を返す（データファイルが変わらない限り）

6. **単一区間処理**
   - 1回の呼び出しで1区間のみを処理する。複数区間の合計計算はAG-002が担当する

7. **モジュールレベル変数のスレッドセーフ**
   - `train_routes_data` と `fixed_fares_data` はアプリ起動時の1回読み込みで以降は読み取り専用。書き込みが発生しないため排他制御は不要

---

### 6.2 パフォーマンス考慮事項

1. **静的読み込みによるI/O最小化**
   - 両ファイルをアプリ起動時に1回だけ読み込む。ツール呼び出しごとのファイルI/Oを排除することでレスポンスタイムを短縮する

2. **辞書キー検索による高速化**
   - 辞書形式でデータを保持するため、O(1)で経路・運賃を検索できる

---

### 6.3 セキュリティ考慮事項

1. **入力値のサニタイズ**
   - `departure` と `destination` は前後空白を strip() で除去する

2. **ファイルパスのハードコード**
   - データファイルパスは `./template/train_routes.json` と `./template/fixed_fares.json` に固定する
