## 1. 目的・適用範囲

### 1.1 目的

システム全体で共通する設定方針を定義し、各エージェントとコンポーネントで一貫した設定を適用する。

### 1.2 適用範囲

- マルチエージェント（AG-001/AG-002/AG-003）
- 共通コンポーネント（ErrorHandler/HumanApprovalHook/LoopControlHook/FileBasedSessionManager）
- 環境設定（.env/config/model_config.py）

### 1.3 本方針で定義すること

- 会話管理の方針
- リトライ戦略の方針
- ループ制御の方針
- セッション管理の方針
- 承認フックの方針
- 共通環境設定・モデル設定の方針

### 1.4 本方針で定義しないこと

- 個々のエージェントの具体的な設定値（各エージェント基本設計書に委ねる）
- SDK固有の内部パラメータ
- 実装クラスのメソッド設計（基本設計・詳細設計フェーズで定義）

## 2. 会話管理の方針

### 2.1 基本原則

- **会話履歴の保持**: エージェントの推論品質を維持するために直近の会話ターンを保持する。全ターンを永続化するのではなく、スライディングウィンドウで管理する
- **メモリ効率**: ウィンドウサイズを超えた古いメッセージは切り詰めてトークン消費を抑制する（`should_truncate_results=True`）
- **コンテキストの継続性**: 収集済み申請情報は会話履歴とは独立して session_{id}.json に永続化し、ウィンドウ超過時もコンテキストが失われないようにする

### 2.2 ウィンドウサイズの基準

| エージェント種別 | 対象 | ウィンドウサイズ | 理由 |
|----------------|------|----------------|------|
| オーケストレーター | AG-001 | 30 | 申請種別判断・委譲・ルール照会のやり取りが続くため多めに保持する |
| 専門エージェント（交通費） | AG-002 | 20 | 移動情報収集（複数件）・APR-001 承認のやり取りに対応できる範囲 |
| 専門エージェント（経費） | AG-003 | 15 | 経費情報収集・領収書確認・APR-001 承認のやり取りに対応できる範囲 |

> ウィンドウサイズはトークン消費とコンテキスト維持のバランスで決定する値であり、ReActループ最大回数・対話回数上限とは無関係。

### 2.3 会話管理の実装

**使用するクラス**: `SlidingWindowConversationManager`（Strands Agents SDK 提供）

**設定方法**:
```python
from strands.agent.conversation_manager import SlidingWindowConversationManager

# AG-001（オーケストレーター）
conversation_management = SlidingWindowConversationManager(
    window_size=30,
    should_truncate_results=True
)

# AG-002（交通費専門エージェント）
conversation_management = SlidingWindowConversationManager(
    window_size=20,
    should_truncate_results=True
)

# AG-003（経費専門エージェント）
conversation_management = SlidingWindowConversationManager(
    window_size=15,
    should_truncate_results=True
)
```

**設定パラメータ**:
- `window_size`: LLMプロンプトに含める会話メッセージの最大保持件数
- `should_truncate_results`: `True` を必須設定。ウィンドウ超過時に古いメッセージを切り詰める

## 3. リトライ戦略の方針

### 3.1 基本原則

- **一時的なエラーへの対応**: Amazon Bedrock API の一時的なエラー・レート制限はリトライで回復可能なため、指数バックオフで再試行する
- **指数バックオフ**: 初期遅延 4 秒、最大遅延 240 秒、最大 6 回試行（実行制御方針 §7.2 参照）
- **最大試行回数の制限**: 6 回を超えるリトライは禁止。上限到達時は EX-04/EX-07 として処理し、セッションを TERMINATED に遷移させる

### 3.2 リトライ対象

| 対象 | リトライ可否 | 最大回数 | 理由 |
|-----|----------|---------|------|
| Amazon Bedrock API 呼び出し（LLM 推論・OCR） | ○ | 6 回 | 一時的なAPIエラー・レート制限はリトライで回復可能。指数バックオフ使用 |
| TOOL-001（交通費計算） | ○ | 要件上未定義 | ファイル参照が冪等なためリトライ可能 |
| TOOL-002（申請書生成） | △ | 1 回（APR-001 再取得後のみ） | APR-001 承認を再取得した場合のみ再実行可。承認未取得での再試行は禁止（ABAC-001） |
| ファイルストアへのログ書き込み | × | — | 書き込み失敗時は TERMINATED に遷移（要件上未定義のため代替手段なし） |

### 3.3 リトライ戦略の実装

**使用する仕組み**: Strands Agents SDK のデフォルトリトライ機構（`botocore` の `RetryConfig`）

**設定方法**: Bedrock API リトライは Strands SDK が内部で処理する。指数バックオフパラメータは `config/model_config.py` で集約管理する

```python
# config/model_config.py
RETRY_CONFIG = {
    "max_attempts": 6,
    "initial_delay": 4,      # 秒
    "max_delay": 240,         # 秒
    "backoff_factor": 2,
}
```

## 4. ループ制御の方針

### 4.1 基本原則

- **無限ループの防止**: ReActループが無限継続（暴走）するリスクを排除するため、全エージェントに `LoopControlHook` を必須設定する
- **処理の可視化**: ループカウントを標準監査ログ（audit_log_std.jsonl）に記録し、実行経路を後から説明可能にする
- **ユーザーへの通知**: ループ上限到達時は「処理が完了できませんでした。管理部門（経理部）にお問い合わせいただくか、再度最初からお試しください。」を提示する

### 4.2 最大ループ回数の基準

| 適用範囲 | 最大ループ回数 | 理由 |
|---------|--------------|------|
| 全エージェント共通（AG-001/AG-002/AG-003） | 10 回 | 情報収集・ツール呼び出し・承認フローを含む 1 回の申請書作成フローに十分な回数 |

> ReActループ最大回数（10回）は会話ウィンドウサイズ・対話回数上限とは独立した設定値。同じ数値に設定する根拠はない。

### 4.3 ループ制御の実装

**使用するクラス**: `LoopControlHook`（HookProvider として実装、`handlers/loop_control_hook.py`）

**設定方法**:
```python
from handlers.loop_control_hook import LoopControlHook

loop_control_hook = LoopControlHook(max_iterations=10)
```

**設定パラメータ**:
- `max_iterations`: 1 回のエージェント呼び出しで許容する ReActループの最大回数（デフォルト: 10）

**登録イベント**:
- `BeforeInvocationEvent`（エージェント呼び出し開始時にループカウンタを 0 にリセット）
- `AfterModelCallEvent`（LLM 呼び出し後にループカウンタをインクリメントし上限到達を監視）

**注意**: `LoopControlHook` を設定しないエージェントはループが無限継続するリスクがある。全エージェント初期化時に必須設定とし、設定漏れをコードレビューで確認すること

## 5. セッション管理の方針

### 5.1 基本原則

- **業務コンテキストの永続化**: 申請者名・申請日・申請種別・収集済み申請情報はファイルストア（data/sessions/session_{id}.json）に永続化する。会話履歴（SlidingWindowConversationManager のインメモリ）とは分離して管理する
- **session_id の一意性**: session_id は AG-001 の初期化時にシステムが生成する。同一 session_id の重複生成を禁止する
- **エージェント間の共有**: session_id は `invocation_state` 経由で AG-002/AG-003 のファクトリ関数に渡す。LLM プロンプトには含めない

### 5.2 session_id の形式

**形式**: `{YYYYMMDD}_{HHMMSS}_{UUID4前8桁}`

**例**: `20260501_143022_a1b2c3d4`

**生成場所**: AG-001 の初期化時（`FileBasedSessionManager` が生成・管理）

### 5.3 セッション管理の実装

**使用するクラス**: `FileBasedSessionManager`（カスタム実装、`agents/session/file_based_session_manager.py`）

**設定方法**:
```python
from agents.session.file_based_session_manager import FileBasedSessionManager

session_manager = FileBasedSessionManager(
    session_id=session_id,
    storage_path="data/sessions"
)
```

**設定パラメータ**:
- `session_id`: セッション識別子（形式: `{YYYYMMDD}_{HHMMSS}_{UUID4前8桁}`）
- `storage_path`: セッションファイルの保存先ディレクトリ（デフォルト: `data/sessions`）

**詳細**: セッション管理方針（`artifacts/03_system-design/outputs/セッション管理方針.md`）を参照

## 6. 承認フックの方針

### 6.1 基本原則

- **重要な操作の承認**: 申請書生成（TOOL-002）は不可逆な操作であるため、実行前に社員（APR-001）の承認を必須とする（ABAC-001）
- **ユーザーの選択肢**: 承認・修正要望・拒否の 3 択を提示し、誤操作を防ぐ
- **誤操作の防止**: APR-001 承認済みフラグが session_{id}.json に記録されていない場合、`HumanApprovalHook` が TOOL-002 の実行を抑止する

### 6.2 承認対象ツールの基準

| ツール種別 | 承認要否 | 理由 |
|----------|---------|------|
| TOOL-002（申請書生成） | 要（APR-001） | ファイル生成という不可逆な操作。意図しない申請書生成を防ぐ（ABAC-001） |
| TOOL-001（交通費計算） | 不要 | 読み取り専用の計算処理。不可逆な副作用がない |
| image_reader（OCR） | 不要 | 画像解析のみ。ファイル生成・書き込みを行わない |

### 6.3 承認フックの実装

**使用するクラス**: `HumanApprovalHook`（カスタム実装、`agents/handlers/human_approval_hook.py`）

**設定方法**:
```python
from agents.handlers.human_approval_hook import HumanApprovalHook

human_approval_hook = HumanApprovalHook()
```

**承認選択肢**:
1. OK（承認） → APR-001 承認済みフラグを session_{id}.json に記録し、TOOL-002 を実行する
2. 修正 → 修正対象の情報を再収集し、APR-001 を再度実施する
3. キャンセル → TOOL-002 実行をキャンセルし、セッションを TERMINATED に遷移させる

**適用対象**: AG-002/AG-003 のみ（AG-001 は TOOL-002 の実行権限を持たないため不要）

## 7. 共通環境設定・モデル設定の方針

### 7.1 環境変数管理

**管理方式**: `.env.template` → `.env` の形式で AWS 認証情報・設定値を管理する（`python-dotenv`）

**主要環境変数**:

| 変数名 | 説明 | デフォルト値 |
|-------|------|------------|
| `AWS_REGION` | Amazon Bedrock のリージョン | `ap-northeast-1` |
| `AWS_ACCESS_KEY_ID` | AWS アクセスキー ID | — |
| `AWS_SECRET_ACCESS_KEY` | AWS シークレットアクセスキー | — |
| `LOG_LEVEL` | ログ出力レベル | `WARNING` |
| `GUARDRAIL_ID` | Bedrock ガードレール ID（未設定時は無効） | — |
| `GUARDRAIL_VERSION` | Bedrock ガードレールバージョン（未設定時は無効） | — |

### 7.2 モデル設定の集約管理

**管理ファイル**: `config/model_config.py`

**集約する設定値**:

```python
# config/model_config.py

# LLM モデル設定
MODEL_ID = "us.anthropic.claude-sonnet-4-5-20251001"
AWS_REGION = "ap-northeast-1"

# リトライ設定（実行制御方針 §7.2 準拠）
RETRY_CONFIG = {
    "max_attempts": 6,
    "initial_delay": 4,    # 秒
    "max_delay": 240,      # 秒
    "backoff_factor": 2,
}

# ループ制御設定（実行制御方針 §10.2 準拠）
MAX_LOOP_ITERATIONS = 10

# 会話管理設定（セッション管理方針 §5.2 準拠）
WINDOW_SIZE_ORCHESTRATOR = 30
WINDOW_SIZE_TRANSPORT_AGENT = 20
WINDOW_SIZE_EXPENSE_AGENT = 15

# セッション管理設定（セッション管理方針 §2.2 準拠）
SESSION_STORAGE_PATH = "data/sessions"

# ガードレール設定（環境変数優先）
GUARDRAIL_ID = None       # 環境変数 GUARDRAIL_ID で上書き
GUARDRAIL_VERSION = None  # 環境変数 GUARDRAIL_VERSION で上書き

# ログ設定
LOG_FILE_PATH = "logs/agent.log"
LOG_LEVEL = "WARNING"     # 環境変数 LOG_LEVEL で上書き（未設定時のデフォルトは WARNING）
```

### 7.3 ログ設定の方針

**出力先**:
- ファイル: `logs/agent.log`（アプリケーションログ・スタックトレース）
- 標準出力: WARNING レベル以上のメッセージ

**ログレベル**: `LOG_LEVEL` 環境変数で制御する。未設定時のデフォルトは `WARNING`

**ユーザー向けメッセージとログの分離**: ユーザー向けメッセージは標準出力（print 等）で出力し、技術ログは logging モジュール経由で出力する。両者を混在させない

**スタックトレース**: `logs/agent.log` にのみ出力する。ユーザーには提示しない

## 8. 設定の適用方法

### 8.1 エージェント作成時の基本パターン（ファクトリ関数）

```python
from strands import Agent
from strands.agent.conversation_manager import SlidingWindowConversationManager
from strands_tools import image_reader
from agents.handlers.loop_control_hook import LoopControlHook
from agents.handlers.human_approval_hook import HumanApprovalHook
from agents.session.file_based_session_manager import FileBasedSessionManager
from config.model_config import (
    MODEL_ID, AWS_REGION,
    WINDOW_SIZE_ORCHESTRATOR, MAX_LOOP_ITERATIONS, SESSION_STORAGE_PATH
)


def _get_ag001_agent(session_id: str) -> Agent:
    """AG-001（申請受付窓口エージェント）のファクトリ関数"""
    session_manager = FileBasedSessionManager(
        session_id=session_id,
        storage_path=SESSION_STORAGE_PATH
    )
    return Agent(
        name="AG-001",
        model=MODEL_ID,
        instructions=AG001_SYSTEM_PROMPT,
        tools=[transport_application_agent, expense_application_agent],  # Agent as Tools
        conversation_manager=SlidingWindowConversationManager(
            window_size=WINDOW_SIZE_ORCHESTRATOR,
            should_truncate_results=True
        ),
        hooks=[LoopControlHook(max_iterations=MAX_LOOP_ITERATIONS)],
    )


def _get_ag002_agent(session_id: str) -> Agent:
    """AG-002（交通費精算申請エージェント）のファクトリ関数"""
    session_manager = FileBasedSessionManager(
        session_id=session_id,
        storage_path=SESSION_STORAGE_PATH
    )
    return Agent(
        name="AG-002",
        model=MODEL_ID,
        instructions=AG002_SYSTEM_PROMPT,
        tools=[calculate_transport_expense, generate_application_form],
        conversation_manager=SlidingWindowConversationManager(
            window_size=WINDOW_SIZE_TRANSPORT_AGENT,
            should_truncate_results=True
        ),
        hooks=[
            LoopControlHook(max_iterations=MAX_LOOP_ITERATIONS),
            HumanApprovalHook(),
        ],
    )
```

> AG-001 は TOOL-002 の実行権限を持たないため `HumanApprovalHook` を設定しない。
> AG-002/AG-003 は申請書生成前の APR-001 承認が必須なため `HumanApprovalHook` を必須設定する。

### 8.2 設定のカスタマイズ

**カスタマイズが必要な場合**:
- 個別エージェントのウィンドウサイズ変更: 各エージェント基本設計書に記載する
- Bedrock ガードレールの有効化: `GUARDRAIL_ID` / `GUARDRAIL_VERSION` 環境変数を設定する

**カスタマイズ方法**: 個々の設定値は `config/model_config.py` を変更する。各エージェントのファクトリ関数はこのファイルを参照するため、設定変更はここに集約する

## 9. 設計上の留意点・リスク

### 9.1 会話管理

**リスク**: ウィンドウサイズを超えた古いメッセージが切り詰められ、エージェントが過去の情報収集内容を参照できなくなる

**対策**: 収集済み申請情報は session_{id}.json に永続化し、会話履歴から独立して参照可能にする（セッション管理方針 §6 参照）

### 9.2 ループ制御

**リスク**: `LoopControlHook` が設定されない場合にループが無限継続するリスクがある

**対策**: 全エージェント初期化時に `LoopControlHook` を必須設定とし、設定漏れをコードレビューで確認する

### 9.3 セッション管理

**リスク**: 異なるセッションの session_{id}.json を参照するリスクがある

**対策**: `FileBasedSessionManager` の生成はファクトリ関数内でのみ行い、session_id を必須引数として渡す

### 9.4 承認フック

**リスク**: APR-001 承認済みフラグが正しく記録されない場合に TOOL-002 が実行されるリスクがある

**対策**: `HumanApprovalHook` が TOOL-002 呼び出し前に session_{id}.json の承認済みフラグを確認する。フラグが未設定の場合は承認フローを強制的に発火させる（ABAC-001）

### 9.5 モデル設定の集約

**リスク**: `config/model_config.py` を参照せず個別ハードコードされると設定値が分散する

**対策**: コードレビューで `config/model_config.py` 参照を必須とする。エージェントファクトリ関数内でのハードコードを禁止する
