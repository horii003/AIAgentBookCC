## 1. 概要

### 1.1 コンポーネントの目的

申請フロー全体のオーケストレーターとして、申請者名収集後にユーザーの申請内容（自由文）から申請種別（交通費精算申請・経費精算申請）を判定し、対応する専門エージェント（AG-002またはAG-003）へ処理を委任する。判定不能時は2択の選択肢をユーザーに提示して申請種別を確定する（BRL-02）。セッション全体を統括し、invocation_stateに session_id・applicant_name・application_date を含めて専門エージェントへ引き渡す。

### 1.2 主要な責務

- **申請者名の収集**: アプリ起動時（対話ループ開始前）に申請者名を取得し、全セッションを通じて保持する
- **申請種別の判定（BRL-01）**: ユーザーの申請内容自由文から申請種別を自動判定する
- **申請種別の確認（BRL-02）**: 判定不能時は2択（交通費精算申請 / 経費精算申請）を提示してユーザー選択で確定する
- **申請書・申請先の提示（BRL-03）**: 申請種別確定後に申請書名と申請先をユーザーへ提示する
- **専門エージェントへの委任**: 申請種別確定後に `transport_application_agent_tool`（AG-002）または `expense_application_agent_tool`（AG-003）を呼び出す
- **セッション管理**: session_id・applicant_name・application_date を invocation_state で専門エージェントへ渡す

### 2.2 インターフェース設計

#### 2.2.1 入力

**初期化時の入力**:
- `applicant_name` (str): 申請者名。必須。アプリ起動時に対話ループ開始前に取得。空文字禁止（GRD-001）
- `application_date` (str): 申請日。システム日付から自動取得（YYYY-MM-DD形式）
- `session_id` (str): セッション識別子（`{YYYYMMDDHHmmss}_{UUID 8文字}` 形式）。例: `"20260323153045_a1b2c3d4"`

**対話ループでの入力**:
- ユーザー入力（自然言語での申請内容自由文）
- 特殊コマンド:
  - `exit`, `quit`, `終了`: システム終了
  - `reset`, `リセット`, `最初から`: 申請者名収集からリセット

**パラメータ（対話ループ）**:
- `user_input` (str): ユーザーからの申請内容自由文

#### 2.2.2 出力

**出力先**: CLI（標準出力）

**起動時ウェルカムメッセージ**（アプリ起動時に1回表示）:
```
============================================================
こちらは申請受付窓口エージェントです
社内の様々な申請作業をサポートします

最初に申請者名を入力してください。その後、申請したい内容をお知らせください。キーワードでも構いません

※終了するには 'exit' または 'quit' と入力ください
※最初からやり直すには 'reset' と入力ください
============================================================
```

**入力プロンプト**: `\n\n入力内容（終了時はquit）: `

**その他出力メッセージ**:
- 申請者名入力促し: `"申請者名を入力してください："`
- 申請受付開始: `"申請内容を入力してください（例：「交通費を精算したい」「会議費の経費申請をしたい」）："`
- 申請書・申請先提示: `"[申請種別]の申請が必要です。申請書：[申請書名]、申請先：[申請先名]"`
- 2択提示（判定不能時）: `"申請種別が特定できませんでした。以下から選択してください：\n1. 交通費精算申請\n2. 経費精算申請"`
- 専門エージェント委任開始: `"[申請種別]の手続きを開始します。"`
- エスカレーション案内（AG-002/AG-003内部エラー時）: `"申し訳ありません。処理中にエラーが発生しました。システム管理者にご連絡ください。"`

**戻り値の形式**: `str`（エージェントからの応答）

### 2.3 ビジネスロジック

#### 2.3.1 システムプロンプト

##### システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 静的定数 |
| 動的生成の理由 | 不要。申請期限チェック（BRL-13/BRL-18）はAG-002/AG-003が担当するため、AG-001のプロンプトに日付・期限判断は含まれない |

##### プロンプト全文

```
あなたは社内申請AIシステムの「申請受付窓口エージェント」です。
申請者名はすでに収集済みです（invocation_stateではなく、初期化パラメータとして受け取っています）。
申請日（application_date）はシステムが自動取得し、専門エージェントへ引き渡します。

【あなたの役割】
申請内容（自由文）から申請種別を判定し、対応する専門エージェントへ処理を委任することです。
申請情報の収集・申請書の生成・申請期限チェック・申請書の提出は担当しません。これらはすべて専門エージェントが担当します。

【処理フロー】
1. ユーザーから申請内容（自由文）を受け取る
2. 申請内容から申請種別を判定する（BRL-01）
   - 交通費精算申請に該当する場合：transport_application_agent_tool を呼び出す前にBRL-03（申請書・申請先提示）を実施する
   - 経費精算申請に該当する場合：expense_application_agent_tool を呼び出す前にBRL-03（申請書・申請先提示）を実施する
   - 判定不能の場合：2択（交通費精算申請 / 経費精算申請）を提示してユーザーの選択を取得する（BRL-02 → CF-002）
3. 申請書・申請先をユーザーへ提示する（BRL-03）
   - 交通費精算申請：申請書「交通費精算申請書」
   - 経費精算申請：申請書「経費精算申請書」
   - ※申請先は要件上未定義のため、現時点では提示を省略する
4. 対応する専門エージェントへ委任する（Agent as Toolsパターン）
   - 交通費精算申請 → transport_application_agent_tool を呼び出す
   - 経費精算申請 → expense_application_agent_tool を呼び出す

【申請種別の判定基準（BRL-01/BRL-02）】
- 交通費精算申請のキーワード例：「電車代」「交通費」「バス代」「タクシー代」「飛行機代」「移動費」「出張交通費」「定期外」
- 経費精算申請のキーワード例：「会議費」「文房具」「事務用品」「宿泊費」「資格費用」「研修費」「経費」「立替」
- 上記キーワードから明確に判定できない場合は、2択（交通費精算申請 / 経費精算申請）をユーザーに提示する（BRL-02）
- 2択提示後はユーザーの選択（「1」「2」「交通費精算申請」「経費精算申請」等）のみを受け付け、他の回答は再選択を促す

【禁止事項】
- 申請書の自動生成・自動提出（BRL-09）
- 申請情報（移動区間・経費明細）の収集
- 申請期限チェック（BRL-13/BRL-18）
- TOOL-001（交通費計算ツール）・TOOL-002（申請書生成ツール）の直接呼び出し
- AG-002とAG-003の同時呼び出し

【特殊コマンド】
- ユーザーが「reset」「リセット」「最初から」と入力した場合は、申請者名収集からやり直す旨を案内し、セッションをリセットする

【エラー時の振る舞い】
- AG-002またはAG-003からエラーが返された場合は、「申し訳ありません。処理中にエラーが発生しました。システム管理者にご連絡ください。」とユーザーへ案内する
- ループ上限（30回）に達した場合は、「処理が複雑すぎるため終了します。最初からやり直すには「reset」と入力してください。」とユーザーへ提示する
```

##### プロンプト設計の意図

| 要素 | 意図 |
|------|------|
| 役割定義 | オーケストレーターとして申請種別判定・委任のみを担い、申請フロー詳細（情報収集・書類生成）には介入しないことを明示する |
| 判定基準表（キーワード例） | LLMが申請種別を一貫した基準で判定できるよう、具体的なキーワード例をプロンプトに埋め込む（BRL-01） |
| 2択提示プロトコル | 判定不能時に「交通費精算申請」「経費精算申請」の2択のみを受け付け、他回答を排除することで判定の確実性を担保する（BRL-02） |
| 禁止事項の明示 | AG-001がTOOL-001/TOOL-002を誤って呼び出すことを防止する |
| エラー時の振る舞い | AG-002/AG-003の内部エラーをAG-001がユーザーへエスカレーション案内として提示する責任を明確化する |

#### 2.3.2 申請種別判定ルール（BRL-01/BRL-02）

| ルール | 条件 | 対応 |
|--------|------|------|
| BRL-01（自動判定） | 申請内容から交通費精算申請または経費精算申請が明確に判定できる場合 | 申請種別を確定してBRL-03→委任へ進む |
| BRL-02（2択確認） | 申請種別が自動判定できない場合 | 2択提示→ユーザー選択で申請種別を確定する（CF-002） |
| BRL-03（提示） | 申請種別確定後 | 申請書名・申請先をユーザーへ提示する |

#### 2.3.3 処理フロー

```
アプリ起動
  ↓
ウェルカムメッセージ表示（2.2.2節のメッセージ）
  ↓
申請者名の入力促し（「申請者名を入力してください：」）
  ↓
申請者名受け取り（min_length=1チェック。空文字は再入力促し）
  ↓
  ※ application_date = datetime.date.today().isoformat()（システム日付自動取得）
  ※ session_id = datetime.now().strftime("%Y%m%d%H%M%S") + "_" + uuid4().hex[:8]
  ↓
対話ループ開始（SlidingWindow(30)）
  ↓
入力プロンプト表示（「\n\n入力内容（終了時はquit）: 」）
  ↓
ユーザー入力受け取り
  ↓
特殊コマンド判定
  - "reset" / "リセット" / "最初から" → 申請者名収集からリセット（セッション再初期化）
  - "exit" / "quit" / "終了" → システム終了
  ↓
申請内容受け取り（自由文）
  ↓
申請種別判定（BRL-01）
  ↓
  判定成功（交通費精算申請）
    → BRL-03：申請書・申請先提示
    → transport_application_agent_tool 呼び出し（invocation_state引き渡し）
    → AG-002からの応答を受け取り表示
    ↓
  判定成功（経費精算申請）
    → BRL-03：申請書・申請先提示
    → expense_application_agent_tool 呼び出し（invocation_state引き渡し）
    → AG-003からの応答を受け取り表示
    ↓
  判定不能
    → 2択提示（CF-002）：「1. 交通費精算申請 / 2. 経費精算申請」
    → ユーザー選択受け取り（1 or 2 or 「交通費」or「経費」等）
    → 申請種別確定
    → BRL-03：申請書・申請先提示
    → 対応ツール呼び出し
  ↓
委任完了後、次の申請内容入力待ちへ（対話ループ継続）
```

### 2.5 連携設計

#### 2.5.1 Agent as Tools設計

AG-001は AG-002・AG-003 を `@tool(context=True)` デコレータで実装されたツール関数（`transport_application_agent_tool`・`expense_application_agent_tool`）を通じて呼び出す。各ツール関数は `_agent_instances` 辞書でAgentインスタンスをキャッシュし、同一セッション内での会話履歴を保持する。

**ツール名**: `transport_application_agent_tool`

**ツール説明（docstring）**:
```
交通費精算申請フロー全体（移動情報収集・運賃計算・申請書生成・ルールチェック・最終提示）を実行する。
AG-001が交通費精算申請と判定した後に呼び出す。
invocation_stateにsession_id・applicant_name・application_dateが設定されていること。
```

**ツール名**: `expense_application_agent_tool`

**ツール説明（docstring）**:
```
経費精算申請フロー全体（経費情報収集・申請書生成・ルールチェック・最終提示）を実行する。
AG-001が経費精算申請と判定した後に呼び出す。
invocation_stateにsession_id・applicant_name・application_dateが設定されていること。
```

#### 2.5.2 invocation_stateの受け渡しとAgentインスタンスのキャッシュ

専門エージェント（AG-002・AG-003）は、モジュールレベルの辞書に `session_id` をキーとして Agent インスタンスをキャッシュする。これにより同一セッション内で複数回呼び出されても会話履歴が保持される。

```python
# session_id をキーにして Agent インスタンスをキャッシュし、
# セッション内で会話履歴を保持する（情報の再入力を防ぐ）
_transport_agent_instances: dict[str, Agent] = {}

@tool(context=True)
def transport_application_agent_tool(query: str, tool_context: ToolContext) -> str:
    state = tool_context.invocation_state
    session_id = state.get("session_id", "")
    applicant_name = state.get("applicant_name", "")
    application_date = state.get("application_date", "")

    # session_id 未登録の場合のみ新しい Agent インスタンスを生成する
    # callback_handler=None を指定して標準出力への直接表示を抑制する（AG-001による二重出力を防ぐ）
    if session_id not in _transport_agent_instances:
        _transport_agent_instances[session_id] = Agent(
            system_prompt=_build_transport_agent_system_prompt(application_date),
            tools=[calculate_transport_expense, generate_transport_expense_form],
            conversation_manager=SlidingWindowConversationManager(window_size=20),
            callback_handler=None,
        )

    response = _transport_agent_instances[session_id](query, invocation_state=state)
    return str(response)
```

```python
_expense_agent_instances: dict[str, Agent] = {}

@tool(context=True)
def expense_application_agent_tool(query: str, tool_context: ToolContext) -> str:
    state = tool_context.invocation_state
    session_id = state.get("session_id", "")
    applicant_name = state.get("applicant_name", "")
    application_date = state.get("application_date", "")

    if session_id not in _expense_agent_instances:
        _expense_agent_instances[session_id] = Agent(
            system_prompt=_build_expense_agent_system_prompt(application_date),
            tools=[generate_expense_form],
            conversation_manager=SlidingWindowConversationManager(window_size=15),
            callback_handler=None,
        )

    response = _expense_agent_instances[session_id](query, invocation_state=state)
    return str(response)
```

#### 2.5.3 専門エージェントへの振り分けロジック

**振り分け判断基準**:

| 連携先 | 判定キーワード例 | 特徴 |
|--------|----------------|------|

```python
def __init__(self):
    self._logger = logging.getLogger(__name__)
    self._error_handler = ErrorHandler()
    self._applicant_name = ""
    self._session_id = ""
    self._application_date = ""
    self._agent = None
```

##### runメソッド

アプリケーションのメインループ。以下の順で実行する:

1. ウェルカムメッセージを標準出力に表示する（2.2.2節のメッセージ）
2. `_collect_applicant_name()` で申請者名を収集する
3. `_initialize_session()` でセッションを初期化する（session_id・application_date生成、Agentインスタンス化）
4. 対話ループを実行する:
   - `\n\n入力内容（終了時はquit）: ` プロンプトでユーザー入力を受け取る
   - 特殊コマンド判定（reset / exit / quit）
   - AG-001エージェントに渡す（invocation_stateに `{session_id, applicant_name, application_date}` を設定）
   - 例外処理（3.2節参照）

##### _collect_applicant_nameメソッド

申請者名を標準入力から収集する。空文字の場合は再入力を促す。

##### _initialize_sessionメソッド

セッション情報を初期化する:

1. `self._application_date = datetime.date.today().isoformat()` でシステム日付を取得する
2. `self._session_id = datetime.now().strftime("%Y%m%d%H%M%S") + "_" + uuid4().hex[:8]` でセッションIDを生成する
3. FileSessionManagerを `session_id` で初期化する
4. AG-001エージェントをインスタンス化する（tools=[transport_application_agent_tool, expense_application_agent_tool]、LoopControlHook(max_iterations=30)登録）

##### _reset_sessionメソッド

リセットコマンド受信時にセッションを再初期化する:

1. WARNING ログ出力: `"[AG-001] リセットコマンド受信: セッション再初期化: session_id={session_id}"`
2. `_transport_agent_instances` と `_expense_agent_instances` から該当 session_id のエントリを削除する
3. `_collect_applicant_name()` を呼び出して申請者名を再収集する
4. `_initialize_session()` を呼び出してセッションを再初期化する

### 3.2 エラーハンドリング

対話ループ内の例外は個別にキャッチし、ErrorHandler に委譲してユーザー向けメッセージを生成する。ログ出力はErrorHandler呼び出し前に実施する。

| 例外クラス | ログレベル | ループ制御 | ユーザー向けメッセージ |
|-----------|----------|------------|---------------------|
| `KeyboardInterrupt` | INFO | break（ループ終了） | `ErrorHandler.handle_keyboard_interrupt(e)` の返却値 |
| `LoopLimitError` | WARNING | continue（ループ継続） | `ErrorHandler.handle_loop_limit_error(e)` の返却値 |
| `ContextWindowOverflowException` | WARNING | continue（ループ継続） | `ErrorHandler.handle_context_window_error(e)` の返却値 |
| `MaxTokensReachedException` | WARNING | continue（ループ継続） | `ErrorHandler.handle_max_tokens_error(e)` の返却値 |
| `RuntimeError` | ERROR（exc_info=True） | continue（ループ継続） | `ErrorHandler.handle_runtime_error(e)` の返却値 |
| `Exception` | ERROR（exc_info=True） | continue（ループ継続） | `ErrorHandler.handle_unexpected_error(e)` の返却値 |

**例外キャッチ順序**: KeyboardInterrupt → LoopLimitError → ContextWindowOverflowException → MaxTokensReachedException → RuntimeError → Exception の順で個別にキャッチする。

**実装パターン**:
```python
try:
    response = self._agent(user_input, invocation_state=invocation_state)
    print(str(response))
except KeyboardInterrupt as e:
    self._logger.info("[AG-001] ユーザーによる中断: session_id=%s", self._session_id)
    print(self._error_handler.handle_keyboard_interrupt(e))
    break
except LoopLimitError as e:
    self._logger.warning(
        "[AG-001] ループ上限到達: %s/%s, session_id=%s",
        e.current_iteration, e.max_iterations, self._session_id
    )
    print(self._error_handler.handle_loop_limit_error(e))
    continue
except ContextWindowOverflowException as e:
    self._logger.warning("[AG-001] コンテキストウィンドウ超過: session_id=%s", self._session_id)
    print(self._error_handler.handle_context_window_error(e))
    continue
except MaxTokensReachedException as e:
    self._logger.warning("[AG-001] 最大トークン数到達: session_id=%s", self._session_id)
    print(self._error_handler.handle_max_tokens_error(e))
    continue
except RuntimeError as e:
    self._logger.error(
        "[AG-001] RuntimeError: %s, session_id=%s", str(e), self._session_id, exc_info=True
    )
    print(self._error_handler.handle_runtime_error(e))
    continue
except Exception as e:
    self._logger.error(
        "[AG-001] 予期しないエラー: %s, session_id=%s", str(e), self._session_id, exc_info=True
    )
    print(self._error_handler.handle_unexpected_error(e))
    continue
```

### 3.3 ログ出力

ログメッセージはすべて日本語で記述する。applicant_name はログ出力時に先頭1文字 + `***` でマスキングする（例: `"山田太郎"` → `"山***"`）。

| レベル | タイミング | メッセージ |
|--------|-----------|-----------|
| INFO | AG-001エージェント初期化時 | `"[AG-001] 申請受付窓口エージェント起動: session_id={session_id}, applicant_name={マスク済み申請者名}, application_date={application_date}"` |
| INFO | 申請種別確定・委任開始時 | `"[AG-001] 委任開始: 申請種別={申請種別}, tool={tool_name}, session_id={session_id}"` |
| INFO | 委任完了時 | `"[AG-001] 委任完了: tool={tool_name}, session_id={session_id}"` |
| INFO | ユーザー中断時（KeyboardInterrupt） | `"[AG-001] ユーザーによる中断: session_id={session_id}"` |
| WARNING | 申請種別判定不能・2択提示時 | `"[AG-001] 申請種別判定不能: 2択提示（CF-002）, session_id={session_id}"` |
| WARNING | リセットコマンド実行時 | `"[AG-001] リセットコマンド受信: セッション再初期化: session_id={session_id}"` |
| WARNING | LoopLimitError発生時 | `"[AG-001] ループ上限到達: {current_iteration}/{max_iterations}, session_id={session_id}"` |
| WARNING | コンテキストウィンドウ超過時 | `"[AG-001] コンテキストウィンドウ超過: session_id={session_id}"` |
| WARNING | 最大トークン数到達時 | `"[AG-001] 最大トークン数到達: session_id={session_id}"` |
| ERROR | RuntimeError発生時 | `"[AG-001] RuntimeError: {error_detail}, session_id={session_id}"` （スタックトレースを含む） |

**applicant_nameのマスキング処理**:
```python
def _mask_applicant_name(self, name: str) -> str:
    if not name:
        return ""
    return name[0] + "***"
```

### 4.1 invocation_state

**AG-001が専門エージェントに渡すデータ**:
```python
{
    "session_id": str,       # セッション識別子（{YYYYMMDDHHmmss}_{UUID 8文字}形式）。例: "20260323153045_a1b2c3d4"
```

## 5. 補足情報

### 5.1 実装上の注意点

1. **クラスベース設計**
   - アプリケーション全体を `OrchestratorApp` クラスとして実装する
   - `__init__`・`run`・`_collect_applicant_name`・`_initialize_session`・`_reset_session`・`_mask_applicant_name` の各メソッドで責務を分割する
   - エントリーポイントは `main()` 関数から `OrchestratorApp().run()` を呼び出す形式とする

2. **applicant_nameの収集タイミング**
   - 申請者名はアプリ起動時（対話ループ開始前）に収集する。エージェントの invocation_state ではなく、アプリケーション起動スクリプトで収集し、その後の対話ループで再利用する

3. **application_dateの自動取得**
   - `datetime.date.today().isoformat()` でシステム日付を取得する。ユーザーへの問い合わせは不要。申請書生成時のタイムスタンプはAG-002/AG-003が処理する

4. **HumanApprovalHookは登録しない**
   - AG-001には HumanApprovalHook を登録しない。申請書生成（TOOL-002）はAG-002/AG-003の責務であり、AG-001がHumanApprovalHookを持つ必要はない（基本設計書 16.1節）

5. **callback_handler=None の設定**
   - `transport_application_agent_tool` と `expense_application_agent_tool` 内で生成するAgentインスタンスには `callback_handler=None` を指定する。これにより専門エージェントの標準出力への直接表示を抑制し、AG-001経由での表示に統一する（二重出力防止）

6. **リセット時のセッション再初期化**
   - `reset`/`リセット`/`最初から` コマンド受信時は、`_transport_agent_instances` と `_expense_agent_instances` から該当 session_id のエントリを削除し、新しい session_id を生成する。applicant_name の再収集から再開する

7. **invocation_stateは辞書リテラルで渡す**
   - 専用Pydanticモデルは定義せず、辞書リテラルで渡す（バリデーション方針 3.2節・マルチエージェント連携設計 7.2節）

8. **applicant_nameのマスキング**
   - ログ出力時は `_mask_applicant_name()` メソッドで先頭1文字 + `***` にマスキングする。invocation_stateに渡す値・ユーザーへの表示はマスキングしない

---

### 5.2 パフォーマンス考慮事項

1. **Agentインスタンスキャッシュによる応答速度向上**
   - `_transport_agent_instances` と `_expense_agent_instances` にセッション単位でインスタンスをキャッシュすることで、同一セッション内の再呼び出し時に初期化コストを削減する

2. **FileSessionManagerの遅延**
   - ファイルI/Oによるセッション保存はレスポンスに軽微な遅延を与えるが、許容範囲内とする

---

### 5.3 セキュリティ考慮事項

1. **セッションIDの安全性**
   - session_id は `{YYYYMMDDHHmmss}_{uuid4().hex[:8]}` で生成する。8文字のUUID hex は予測困難であり、セッション識別の一意性を担保する

2. **申請者名のマスキング**
   - 申請者名はログ出力時に先頭1文字 + `***` でマスキングする（例: `"山田太郎"` → `"山***"`）。invocation_state 経由での専門エージェントへの受け渡し・ユーザーへの表示はマスキングしない

## 7. テスト観点

### 7.1 機能テスト
- 「電車代を精算したい」等の交通費キーワードを含む申請内容に対して、BRL-03（申請書・申請先提示）実施後に `transport_application_agent_tool` が呼び出されること
- 「会議費を精算したい」等の経費キーワードを含む申請内容に対して、`expense_application_agent_tool` が呼び出されること
- 判定不能な申請内容（例：「申請したい」）に対して2択（CF-002）が提示され、ユーザー選択後に対応ツールが呼び出されること
- applicant_name が空文字の場合に再入力が促されること
- application_date が YYYY-MM-DD 形式でシステム日付から自動取得されること
- invocation_state に session_id・applicant_name・application_date が正しく設定されること
- アプリ起動時にウェルカムメッセージが表示されること
- `\n\n入力内容（終了時はquit）: ` 形式の入力プロンプトが表示されること
- session_id が `{YYYYMMDDHHmmss}_{UUID 8文字}` 形式（例: `"20260323153045_a1b2c3d4"`）で生成されること

### 7.2 異常系テスト
- `reset`/`リセット`/`最初から` コマンド受信時に申請者名収集からリセットされること
- `exit`/`quit`/`終了` コマンド受信時にシステムが終了すること
- AG-002または AG-003の内部エラー時にエスカレーション案内が表示されること
- KeyboardInterrupt発生時にINFOログ出力後にループを終了すること
- LoopLimitError発生時にWARNINGログ出力後にエラーメッセージを表示してループを継続すること
- ContextWindowOverflowException発生時にWARNINGログ出力後にエラーメッセージを表示してループを継続すること
- MaxTokensReachedException発生時にWARNINGログ出力後にエラーメッセージを表示してループを継続すること
- RuntimeError発生時にERRORログ（exc_info=True）出力後にエラーメッセージを表示してループを継続すること
- LoopControlHookが `max_iterations=30` で動作し、ループ上限到達時に `LoopLimitError` が発生すること
- AG-002とAG-003が同時に呼び出されないこと（申請種別は1回で1種類）
- ログ出力時に applicant_name が先頭1文字 + `***` でマスキングされること

### 7.3 性能テスト
- 申請種別判定から専門エージェント委任開始まで3秒以内に完了すること
- LoopControlHookが30回上限で正しく動作すること
