## 1. 目的・適用範囲

### 1.1 目的

システム全体で共通するバリデーション方針を定義し、外部から受け取るデータの検証・正規化を一貫した方針で実施する。

### 1.2 適用範囲

- 全エージェント（AG-001, AG-002, AG-003）
- 全ツール（TOOL-001, TOOL-002）
- invocation_stateによるエージェント間データ受け渡し
- ユーザー入力（CLIからのメッセージ）

### 1.3 本方針で定義すること

- バリデーションの基本原則とタイミング
- Pydanticモデルの使用方針（定義場所・命名規則・基本構造）
- データ型別バリデーション（日付・金額・文字列・列挙型）
- 業務ルールに基づくカスタムバリデーションルール
- バリデーションエラーメッセージの方針

### 1.4 本方針で定義しないこと

- 各Pydanticモデルの完全なフィールド定義（基本設計書に委ねる）
- LLMプロンプト内のバリデーション指示
- ツール内部の実装詳細

## 2. バリデーションの基本原則

### 2.1 基本方針

- **外部入力は全て検証する**: ユーザー入力・ツールパラメータ・invocation_state・ファイルデータはPydanticモデルで検証する
- **早期検証**: システム境界（ツール入口・エージェント入口）で検証し、不正データを内部に持ち込まない
- **自動正規化**: 列挙型フィールドは正規化関数で統一表記に変換してから後続処理に渡す
- **エラーの委譲**: ValidationErrorはErrorHandlerに委譲し、ツール関数から例外を再送出しない
- **ユーザーフレンドリー**: バリデーションエラーメッセージは日本語で具体的な対処方法を含める

### 2.2 バリデーションのタイミングと責務

バリデーションの実施タイミングと責務はすべて **Pydanticモデル**（`models/data_models.py`）と **ErrorHandler**（`handlers/error_handler.py`）に割り当てる。

| タイミング | 対象 | 責務 | 対応ガードレール |
|----------|------|------|----------------|
| ユーザー入力受信時 | CLIからのユーザーメッセージ（文字数） | `UserInputText` Pydanticモデルで検証。エラー時は ErrorHandler（`handlers/error_handler.py`）に委譲 | GRD-017 |
| invocation_state受取時 | セッション引き継ぎデータ（辞書リテラル: session_id・申請者名・申請日）の各値を個別バリデーション | 専門エージェント（AG-002/AG-003）ファクトリ関数内で個別チェック（Pydanticモデルは使わない）。エラー時は ErrorHandler に委譲 | — |
| ツール呼び出し時（TOOL-001） | 移動情報（出発地・目的地・交通手段） | `TravelExpenseCalculatorInput` Pydanticモデルで検証。エラー時は ErrorHandler に委譲 | GRD-002 |
| ツール呼び出し時（TOOL-002） | 収集済み申請情報（申請種別・明細・申請者名・申請日） | 申請書生成ツール専用 Pydanticモデルで検証。エラー時は ErrorHandler に委譲 | GRD-002, GRD-016 |
| ファイルデータ読み込み後（静的） | 運賃データJSON（DATA-011/DATA-012）：アプリ起動時にJSONファイルから静的に読み込む | `TrainFareRecord` Pydanticモデルで検証。エラー時は ErrorHandler に委譲（EX-06） | EX-06対応 |

### 2.3 バリデーション失敗時の扱い

**ValidationError発生時**:
- ツール関数内でキャッチし `{"success": False, "message": エラーメッセージ}` を返す（例外を再送出しない）
- ErrorHandler（`handlers/error_handler.py`）経由でEX-01（入力例外）またはEX-06（データ不整合）として分類しログに記録する
- ユーザーへは日本語エラーメッセージと再入力方法を提示する

## 3. Pydanticモデルの使用方針

### 3.1 基本原則

- Pydantic v2+の `BaseModel` を全モデルの基底クラスとして使用する
- バリデーションは `field_validator`（`mode="before"` または `mode="after"`）で実装する
- モデル生成時に `model_validate()` を使用し、辞書データを安全に変換する
- `Field()` で各フィールドの制約（最小値・最大値・最大文字数）を宣言する

### 3.2 Pydanticモデルの定義場所とErrorHandlerの配置

**Pydanticモデル定義場所**: `models/data_models.py`

**ErrorHandler配置**: `handlers/error_handler.py`

**命名規則**:
- ツールパラメータ: `{ツール名}Input`（例: `TravelExpenseCalculatorInput`）
- ユーザー入力: `UserInputText`
- ファイルデータ: `{データ名}Record`（例: `TrainFareRecord`）

> **注意**: `invocation_state` は辞書リテラルで渡すため、専用の Pydantic モデル（`InvocationState` クラス等）は定義しない

### 3.3 Pydanticモデルの基本構造

```python
from pydantic import BaseModel, field_validator, Field
from typing import Literal, Optional
from datetime import date

class TravelExpenseCalculatorInput(BaseModel):
    departure: str = Field(..., min_length=1, description="出発地（正規化済み）")
    destination: str = Field(..., min_length=1, description="目的地（正規化済み）")
    transport_type: Literal["電車", "バス", "タクシー", "飛行機"]

    @field_validator("transport_type", mode="before")
    @classmethod
    def normalize_transport_type(cls, v: str) -> str:
        mapping = {
            "train": "電車", "鉄道": "電車", "地下鉄": "電車",
            "bus": "バス", "taxi": "タクシー", "cab": "タクシー",
            "airplane": "飛行機", "plane": "飛行機", "飛行機": "飛行機",
        }
        return mapping.get(v, v)
```

## 4. 日付のバリデーション

### 4.1 基本原則

- 標準形式（YYYY-MM-DD）を使用する
- 文字列入力は `date.fromisoformat()` または `dateutil.parser.parse()` で自動変換する
- 未来日付（申請日より後の移動日・経費発生日）は拒否する
- 申請期限チェック（3ヶ月以内）はカスタムバリデーターで実施する（BRL-13/BRL-18）

### 4.2 日付の定義

**標準形式**: `YYYY-MM-DD`（`datetime.date` 型）

**許容値**:
- 移動日・経費発生日: 申請日以前の日付
- 申請日: 当日以前の日付

### 4.3 日付バリデーションの実装

```python
from datetime import date
from dateutil.relativedelta import relativedelta
from pydantic import field_validator, model_validator

class ApplicationDateFields(BaseModel):
    application_date: date
    expense_date: date  # 移動日 or 経費発生日

    @field_validator("application_date", "expense_date", mode="before")
    @classmethod
    def parse_date(cls, v):
        if isinstance(v, str):
            from dateutil import parser as dateutil_parser
            return dateutil_parser.parse(v).date()
        return v

    @model_validator(mode="after")
    def validate_deadline(self):
        if self.expense_date > self.application_date:
            raise ValueError("移動日・経費発生日は申請日以前である必要があります")
        limit = self.expense_date + relativedelta(months=3)
        if self.application_date > limit:
            raise ValueError(
                f"申請期限を超過しています。経費発生日から3ヶ月以内（{limit.isoformat()}まで）に申請してください"
            )
        return self
```

### 4.4 日付の特殊ルール

| ルールID | 条件 | 判定結果 | 対応アクション |
|---------|------|---------|--------------|
| BRL-13 | 移動日から3ヶ月超過（JD-10） | 申請不可 | EX-03 → CF-008へ遷移 |
| BRL-18 | 経費発生日から3ヶ月超過（JD-14） | 申請不可 | EX-03 → CF-008へ遷移 |

## 5. 金額のバリデーション

### 5.1 基本原則

- 金額は非負整数（`int`、0以上）で保持する
- 上長承認要否の閾値チェックはバリデーション後に実施する（EX-03扱いではなく警告）
- 閾値超過はPydanticバリデーションエラーではなく、エージェントが警告メッセージとして提示する

### 5.2 金額の定義

**標準形式**: `int`（円単位、0以上）

**許容値**: 0以上の整数

### 5.3 金額バリデーションの実装

```python
from pydantic import field_validator, Field

class AmountField(BaseModel):
    amount: int = Field(..., ge=0, description="金額（円）")

    @field_validator("amount", mode="before")
    @classmethod
    def parse_amount(cls, v):
        if isinstance(v, str):
            # カンマ区切りや「円」表記を除去
            v = v.replace(",", "").replace("円", "").strip()
        return int(v)
```

### 5.4 金額の特殊ルール（警告判定）

| ルールID | 条件 | 判定結果 | 対応アクション |
|---------|------|---------|--------------|
| BRL-14 | 交通費合計 > 10,000円（JD-12） | 上長承認要 | 警告メッセージ提示 → CF-009へ遷移（ABAC-001） |
| BRL-19 | 経費金額 > 5,000円（JD-16） | 上長承認要 | 警告メッセージ提示 → CF-009へ遷移（ABAC-002） |

> 上長承認要否はPydanticバリデーションエラーではない。エージェントが閾値チェック後に警告を生成する。

## 6. 文字列のバリデーション

### 6.1 基本原則

- 必須文字列フィールドは `min_length=1` で空文字列・空白のみを拒否する
- ユーザー入力（CLIメッセージ）は500文字以内に制限する（GRD-017）
- 前後の空白は `strip()` で自動除去する（`mode="before"`）
- 業務目的（BRL-20）は必須文字列として定義する

### 6.2 文字列の定義

**標準形式**: `str`

**制約**:
- ユーザー入力: `max_length=500`（GRD-017）
- 申請者名: `min_length=1`, `max_length=100`
- 業務目的: `min_length=1`（BRL-20）
- 駅名・地名: `min_length=1`

### 6.3 文字列バリデーションの実装

```python
from pydantic import field_validator, Field

class UserInputText(BaseModel):
    text: str = Field(..., min_length=1, max_length=500)

    @field_validator("text", mode="before")
    @classmethod
    def strip_whitespace(cls, v: str) -> str:
        v = v.strip()
        if not v:
            raise ValueError("入力が空です。申請内容を入力してください")
        return v

class ApplicationFormBase(BaseModel):
    applicant_name: str = Field(..., min_length=1, max_length=100)
    business_purpose: str = Field(..., min_length=1, description="業務目的（BRL-20）")

    @field_validator("applicant_name", "business_purpose", mode="before")
    @classmethod
    def strip_and_validate(cls, v: str) -> str:
        v = v.strip()
        if not v:
            raise ValueError("必須項目が空です")
        return v
```

### 6.4 文字列の特殊ルール

| ルールID/GRD | 条件 | 判定結果 | 対応アクション |
|-------------|------|---------|--------------|
| GRD-017 | ユーザー入力 > 500文字 | 入力拒否 | EX-01 → 再入力を促す |
| GRD-001 | 入力が空欄・空白のみ | 入力拒否 | EX-01 → 再入力を促す |
| BRL-20 | 業務目的が未記入 | 申請書生成禁止 | EX-01 → 業務目的の入力を促す（GRD-010） |

## 7. 列挙型のバリデーション

### 7.1 基本原則

- 列挙型フィールドは `Literal` 型で受け入れ可能な値を明示する
- LLMが渡す可能性のある表記バリエーション（別表記・英語表記・略称）を `field_validator(mode="before")` の正規化マッピングで統一表記に変換する
- 正規化後の値を後続処理で使用する（入力値をそのまま使用しない）

### 7.2 列挙型の定義

| フィールド | 許容値 | 正規化対象の別表記例 |
|----------|--------|-------------------|
| 申請種別 | `"交通費精算申請"` / `"経費精算申請"` | `"交通費"` → `"交通費精算申請"`, `"経費"` → `"経費精算申請"` |
| 交通手段 | `"電車"` / `"バス"` / `"タクシー"` / `"飛行機"` | `"train"` → `"電車"`, `"taxi"` → `"タクシー"`, `"airplane"` → `"飛行機"` |
| 経費区分 | `"事務用品費"` / `"宿泊費"` / `"資格精算費"` / `"その他経費"` | `"文房具"` → `"事務用品費"`, `"ホテル"` → `"宿泊費"`, `"資格"` → `"資格精算費"` |

### 7.3 列挙型バリデーションの実装

```python
from typing import Literal
from pydantic import BaseModel, field_validator

APPLICATION_TYPE_MAP = {
    "交通費": "交通費精算申請",
    "交通費申請": "交通費精算申請",
    "経費": "経費精算申請",
    "経費申請": "経費精算申請",
}

TRANSPORT_TYPE_MAP = {
    "train": "電車", "鉄道": "電車", "地下鉄": "電車", "電車": "電車",
    "bus": "バス", "バス": "バス",
    "taxi": "タクシー", "cab": "タクシー", "タクシー": "タクシー",
    "airplane": "飛行機", "plane": "飛行機", "飛行機": "飛行機",
}

EXPENSE_CATEGORY_MAP = {
    "文房具": "事務用品費", "事務用品": "事務用品費", "消耗品": "事務用品費",
    "ホテル": "宿泊費", "旅館": "宿泊費", "宿泊": "宿泊費",
    "資格": "資格精算費", "検定": "資格精算費",
    "その他": "その他経費",
}

class TravelExpenseCalculatorInput(BaseModel):
    departure: str
    destination: str
    transport_type: Literal["電車", "バス", "タクシー", "飛行機"]

    @field_validator("transport_type", mode="before")
    @classmethod
    def normalize_transport_type(cls, v: str) -> str:
        normalized = TRANSPORT_TYPE_MAP.get(v, v)
        return normalized

# ※ invocation_state は辞書リテラルで渡すため、InvocationState Pydantic モデルは定義しない。
#    ファクトリ関数内で session_id・applicant_name・application_date を個別にチェックする。
```

## 8. カスタムバリデーションルール

### 8.1 申請期限チェック（BRL-13/BRL-18）

```python
from datetime import date
from dateutil.relativedelta import relativedelta
from pydantic import model_validator

class ApplicationDeadlineValidator(BaseModel):
    application_date: date
    expense_date: date

    @model_validator(mode="after")
    def check_application_deadline(self):
        deadline = self.expense_date + relativedelta(months=3)
        if self.application_date > deadline:
            raise ValueError(
                f"申請期限を超過しています。"
                f"経費発生日（{self.expense_date}）から3ヶ月以内（{deadline}まで）に申請してください"
            )
        return self
```

### 8.2 必須項目充足チェック（BRL-04/GRD-002）

```python
from pydantic import model_validator
from typing import Optional

class ApplicationFormInput(BaseModel):
    applicant_name: str
    application_date: date
    business_purpose: str
    items: list  # 移動情報または経費情報の明細リスト

    @model_validator(mode="after")
    def check_required_items(self):
        if not self.items:
            raise ValueError("申請明細が1件以上必要です。移動情報または経費情報を入力してください")
        return self
```

### 8.3 運賃データ整合性チェック（EX-06対応）

```python
from pydantic import BaseModel, field_validator

class TrainFareRecord(BaseModel):
    departure: str
    destination: str
    fare: int

    @field_validator("fare", mode="before")
    @classmethod
    def validate_fare(cls, v) -> int:
        if not isinstance(v, int) or v < 0:
            raise ValueError(f"運賃データが不正です: {v}。正の整数である必要があります")
        return v
```

## 9. バリデーションエラーメッセージ

### 9.1 基本原則

- **わかりやすさ**: 技術的な詳細（フィールド名・クラス名）を隠蔽し、業務用語で説明する
- **具体性**: エラーの原因と対処方法を明確に記載する
- **一貫性**: 全バリデーションエラーメッセージを日本語で統一する

### 9.2 エラーメッセージの形式

**基本形式**:
```
{フィールド名}のエラー: {エラー内容}
対処方法: {対処方法}
```

### 9.3 エラーメッセージの実装

```python
from pydantic import ValidationError
from handlers.error_handler import ErrorHandler

error_handler = ErrorHandler()

try:
    model = TravelExpenseCalculatorInput(**data)
except ValidationError as e:
    error_message = error_handler.handle_validation_error(e)
    return {"success": False, "message": error_message}
```

**具体的なエラーメッセージ例**:

| エラー種別 | エラーメッセージ例 |
|----------|-----------------|
| ユーザー入力文字数超過（GRD-017） | 「入力が500文字を超えています。もう少し短くして、もう一度入力してください。」 |
| 入力が空欄（GRD-001） | 「入力が空です。申請内容を入力してください。」 |
| 申請期限超過（BRL-13/BRL-18） | 「申し訳ありません。経費発生日から3ヶ月を超えているため、この申請は受け付けられません。総務部門にご相談ください。」 |
| 業務目的未記入（BRL-20） | 「業務目的が入力されていません。申請の業務目的を入力してください。」 |
| 交通手段不明（TOOL-001） | 「交通手段のエラー: 指定された交通手段が認識できません。\n対処方法: 「電車」「バス」「タクシー」「飛行機」のいずれかで入力してください。」 |
| 申請種別不明 | 「申請種別のエラー: 申請種別が認識できません。\n対処方法: 「交通費精算申請」または「経費精算申請」のいずれかを選択してください。」 |
| 運賃データ不正（EX-06） | 「申し訳ありません。運賃データの形式が不正です。システム管理者にご連絡ください。」 |

## 10. バリデーションのテスト

### 10.1 テスト観点

- 正常系: 正常な入力値でモデルが生成されること
- 異常系: 不正な入力値でValidationErrorが発生すること
- 境界値: 上限・下限の境界値で正しく動作すること
- 正規化: 別表記が正規形に変換されること
- エラーメッセージ: 日本語メッセージが生成されること

### 10.2 テストケースの例

| テストケース | 入力 | 期待結果 |
|------------|------|---------|
| 正常系: TOOL-001入力 | `{"departure": "渋谷", "destination": "新宿", "transport_type": "電車"}` | モデル生成成功 |
| 正常系: 申請期限チェック | 経費発生日2026-01-28, 申請日2026-04-28 | バリデーション通過（ちょうど3ヶ月） |
| 異常系: ユーザー入力文字数超過 | 501文字の文字列 | ValidationError（GRD-017） |
| 異常系: 申請期限超過 | 経費発生日2026-01-01, 申請日2026-04-15（3ヶ月超） | ValidationError（BRL-13） |
| 異常系: 金額負値 | `{"amount": -100}` | ValidationError |
| 異常系: 交通手段不明 | `{"transport_type": "自転車"}` | ValidationError |
| 境界値: 申請期限ちょうど3ヶ月 | 経費発生日2026-01-28, 申請日2026-04-28 | バリデーション通過 |
| 境界値: 申請期限1日超過 | 経費発生日2026-01-27, 申請日2026-04-28 | ValidationError（3ヶ月+1日超） |
| 境界値: ユーザー入力500文字 | ちょうど500文字の文字列 | バリデーション通過 |
| 正規化: 交通手段（英語表記） | `{"transport_type": "train"}` | `"電車"` に正規化されモデル生成成功 |
| 正規化: 申請種別（略称） | `{"application_type": "交通費"}` | `"交通費精算申請"` に正規化されモデル生成成功 |
| エラーメッセージ: 業務目的なし | `{"business_purpose": ""}` | ValidationError・日本語メッセージ生成 |

> テスト設計の詳細は詳細設計フェーズのテスト設計書で定義する。

## 11. 設計上の留意点・リスク

### 11.1 列挙型正規化の網羅性

**リスク**: LLMが想定外の表記で値を渡した場合、正規化マッピングに含まれず `Literal` 型のバリデーションエラーになるリスク

**対策**: `field_validator` でマッピングにない値はそのままLiteralバリデーションに渡し、エラーメッセージで許容値を明示する。基本設計フェーズでマッピングテーブルを拡充する

### 11.2 日付パース失敗

**リスク**: LLMが非標準形式（「令和8年4月28日」等）で日付を渡した場合、`dateutil.parser.parse()` が失敗するリスク

**対策**: `dateutil.parser.parse()` のパース失敗時は `ValueError` を発生させ、ErrorHandler経由でEX-01（入力例外）として再入力を促す。YYYY-MM-DD形式での入力をエージェントのシステムプロンプトで指示する

### 11.3 ValidationError握り潰し

**リスク**: ツール関数内でValidationErrorをキャッチせず例外が伝播するリスク

**対策**: 全ツール関数の入口で `try/except ValidationError` を実装し、`{"success": False, "message": ...}` を返す。ErrorHandler（`handlers/error_handler.py`）の `handle_validation_error()` に委譲してログを記録する
