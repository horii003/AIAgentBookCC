## 2. 会話管理の方針

### 2.1 基本原則

- **会話履歴の保持**: 全エージェントはウィンドウサイズ内の会話履歴をLLMプロンプトに渡す。ウィンドウサイズを超えた場合は `should_truncate_results=True` で切り詰めて処理を継続する
- **メモリ効率**: `per_turn=False`（メッセージ単位カウント）を全エージェントに適用し、不要なメッセージの保持を防止する
- **コンテキストの継続性**: ウィンドウサイズ内の会話履歴により、申請フロー内の前後の文脈を適切に維持する

### 2.2 ウィンドウサイズの基準

| エージェント | ウィンドウサイズ | 理由 |
|------------|----------------|------|
| AG-001（申請受付窓口エージェント） | 30 | 複数の専門エージェントとのやり取りを保持する必要があるため |
| AG-002（交通費精算申請エージェント） | 20 | 複数区間の移動情報収集・交通費計算・申請書生成・ルールチェックを行うため |
| AG-003（経費精算申請エージェント） | 15 | 経費情報収集・申請書生成・ルールチェックを行うため |
| ツール | — | ツールは独立した関数として実行されるため会話管理は不要 |

### 2.3 会話管理の実装

**使用するクラス**: `SlidingWindowConversationManager`（Strands Agents SDK提供）

**設定方法**:
```python
from strands.agent.conversation_manager import SlidingWindowConversationManager

conversation_management = SlidingWindowConversationManager(
    window_size=30,           # AG-001の場合
    should_truncate_results=True,
    per_turn=False
)
```

**設定パラメータ**:
- `window_size`: LLMに渡す会話履歴の保持件数（AG-001: 30、AG-002: 20、AG-003: 15）
- `should_truncate_results`: ウィンドウ超過時に結果を切り詰める（全エージェント: True）
- `per_turn`: ターン単位でカウントするか（全エージェント: False = メッセージ単位）

## 3. リトライ戦略の方針

### 3.1 基本原則

- **一時的なエラーへの対応**: Amazon Bedrock（LLM）への接続エラー・レート制限等の一時的エラーにのみリトライを適用する
- **指数バックオフ**: 初期遅延4秒・最大遅延240秒の指数バックオフでリトライする
- **最大試行回数の制限**: 最大6回でリトライを打ち切り、ErrorHandlerに委譲する

### 3.2 リトライ対象

| 対象 | リトライ可否 | 理由 |
|-----|----------|------|
| Amazon Bedrock（LLM）呼び出し | ○ | 一時的なAPI障害・レート制限に対応するため。Strands Agents SDKの組み込みリトライ機構を使用 |
| TOOL-001（交通費計算） | × | ファイルI/Oエラーは代替処理（手動入力）で対応するため |
| TOOL-002（申請書生成） | × | 重複生成リスクがあり、HumanApprovalHook承認後の1回のみ実行するため |
| ファイルI/O（セッション・ログ） | × | ファイルシステム障害はシステム障害（EX-07）として中断するため |

### 3.3 リトライ戦略の実装

**使用するクラス**: Strands Agents SDKのデフォルトリトライ設定を使用

**設定方法**: `config/model_config.py` でモデル設定を通じて適用する（明示的なリトライクラスの追加設定は不要）

## 4. ループ制御の方針

### 4.1 基本原則

- **無限ループの防止**: ReActループが最大ループ回数を超えた場合、LoopControlHookが強制的にループを停止する
- **処理の可視化**: LoopControlHookがループ回数をErrorHandler経由でログに記録する
- **ユーザーへの通知**: ループ上限到達時はユーザーに「処理が複雑すぎるため終了します」とメッセージを提示する

### 4.2 最大ループ回数の基準

| エージェント種別 | 最大ループ回数 | 理由 |
|----------------|--------------|------|
| 全エージェント（AG-001, AG-002, AG-003） | 10回 | 申請フローの複雑さに対して十分なループ数を確保しつつ、暴走を防止するため |

### 4.3 ループ制御の実装

**使用するクラス**: `LoopControlHook`（HookProvider として実装: `hooks/loop_control_hook.py`）

**設定方法**:
```python
from hooks.loop_control_hook import LoopControlHook

loop_control_hook = LoopControlHook(max_iterations=10)
```

**設定パラメータ**:
- `max_iterations`: 最大ループ回数（全エージェント共通: 10回）

**登録イベント**:

| イベント | 処理内容 |
|---------|---------|
| BeforeInvocationEvent | エージェント呼び出し開始時にループカウンタを0にリセットする |
| AfterModelCallEvent | LLM呼び出し後にループカウンタをインクリメントし上限到達を監視する |
| AfterInvocationEvent | エージェント呼び出し完了後にループカウンタを0にリセットする |

## 5. セッション管理の方針

### 5.1 基本原則

- **会話履歴の永続化**: セッション情報はファイルストレージ（data/sessions/）に永続化する。メモリ・揮発性ストレージは使用しない
- **セッションIDの一意性**: UUID v4形式のセッションIDを使用し、セッションの一意性を保証する
- **エージェント間の共有**: AG-001から専門エージェント（AG-002/AG-003）へのセッション引き継ぎは、`invocation_state` でセッションIDを渡し、ファクトリ関数内でセッションマネージャーを生成する

### 5.2 セッションIDの形式

**形式**: `UUID v4`（文字列）

**例**: `"a1b2c3d4-e5f6-7890-abcd-ef1234567890"`

### 5.3 セッション管理の実装

**使用するクラス**: `FileSessionManager`（Strands Agents SDK提供）

**設定方法**:
```python
from strands.agent.session import FileSessionManager

session_manager = FileSessionManager(
    session_id=session_id,
    storage_path="data/sessions/"
)
```

**詳細**: セッション管理方針（`artifacts/03_system-design/outputs/セッション管理方針.md`）を参照

## 6. 承認フックの方針

### 6.1 基本原則

- **重要な操作の承認**: TOOL-002（申請書生成）の実行前に必ず利用者の確認を取得する（BRL-09, GRD-016）
- **ユーザーの選択肢**: OK（承認）・修正（修正要望）・キャンセルの3択をユーザーに提示する
- **誤操作の防止**: HumanApprovalHookがTOOL-002実行をブロックし、承認前の自動実行を防止する

### 6.2 承認対象ツールの基準

| ツール種別 | 承認要否 | 理由 |
|----------|---------|------|
| TOOL-002（申請書生成） | 要 | 申請書のドラフト生成は利用者確認後にのみ実行する（BRL-09, GRD-016）ため |
| TOOL-001（交通費計算） | 不要 | 参照のみで副作用がないため（自律度Lv3。承認不要） |
| image_reader（strands_tools提供） | 不要 | 参照のみで副作用がないため |

### 6.3 承認フックの実装

**使用するクラス**: `HumanApprovalHook`（カスタム実装: `hooks/human_approval_hook.py`）

**設定方法**:
```python
from hooks.human_approval_hook import HumanApprovalHook

human_approval_hook = HumanApprovalHook()
```

**設定パラメータ**:
- 承認対象ツール: `TOOL-002`（申請書生成ツール）のみに適用

## 7. 設定の適用方法

### 7.1 エージェント作成時の設定

**オーケストレーター（AG-001）のパターン**:
```python
from strands import Agent
from strands.agent.conversation_manager import SlidingWindowConversationManager
from strands.agent.session import FileSessionManager
from hooks.loop_control_hook import LoopControlHook
from config.model_config import get_model

agent = Agent(
    agent_id="orchestrator_agent",
    name="申請受付窓口エージェント",
    model=get_model(),
    system_prompt=ORCHESTRATOR_SYSTEM_PROMPT,
    tools=[travel_application_tool, expense_application_tool],
    conversation_manager=SlidingWindowConversationManager(
        window_size=30,
        should_truncate_results=True,
        per_turn=False
    ),
    hooks=[LoopControlHook(max_iterations=10)],
    session_manager=self._session_manager,
    callback_handler=None
)
```

**専門エージェント（AG-002）のファクトリ関数パターン**:
```python
from strands import Agent
from strands.agent.conversation_manager import SlidingWindowConversationManager
from strands.agent.session import FileSessionManager
from hooks.loop_control_hook import LoopControlHook
from hooks.human_approval_hook import HumanApprovalHook
from config.model_config import get_model

def _get_travel_agent(session_id: str) -> Agent:
    session_manager = FileSessionManager(
        session_id=session_id,
        storage_path="data/sessions/"
    )
    return Agent(
        agent_id="travel_agent",
        name="交通費精算申請エージェント",
        model=get_model(),
        system_prompt=TRAVEL_SYSTEM_PROMPT,
        tools=[calculate_travel_expense, generate_application_form],
        conversation_manager=SlidingWindowConversationManager(
            window_size=20,           # AG-002の場合
            should_truncate_results=True,
            per_turn=False
        ),
        hooks=[HumanApprovalHook(), LoopControlHook(max_iterations=10)],
        session_manager=session_manager,
        callback_handler=None
    )
```

> ※ AG-003（経費精算申請エージェント）は `window_size=15` を使用する。それ以外の設定は AG-002 と同様

### 7.2 設定のカスタマイズ

**カスタマイズが必要な場合**:
- エージェントごとに `window_size` が異なる（AG-001: 30、AG-002: 20、AG-003: 15）
- オーケストレーターには `HumanApprovalHook` を適用しない（申請書生成は専門エージェントの責務のため）
- 専門エージェントのセッションマネージャーはファクトリ関数内で都度生成する（`invocation_state` から渡された `session_id` を使用）

**カスタマイズ方法**: 各エージェントの基本設計書を参照

## 8. 設計上の留意点・リスク

### 8.1 会話管理

**リスク**: ウィンドウサイズを超えた会話履歴が切り詰められ、必要なコンテキストが失われるリスク

**対策**: 重要な業務コンテキスト（申請者名・申請種別・収集済み申請情報）は `invocation_state` または Strands Agents SDK のセッション管理で別途保持する

### 8.2 ループ制御

**リスク**: LoopControlHookが誤作動し、正常な処理がループ上限前に停止するリスク

**対策**: max_iterations=10 は申請フローの正常な処理ステップ数（収集→計算→生成→チェック→提示）に対して十分な余裕を持たせた値。BeforeInvocationEvent / AfterInvocationEvent でループカウンタをリセットすることで、エージェント呼び出し間でカウンタが引き継がれないようにしている

### 8.3 セッション管理

**リスク**: ファクトリ関数で誤ったセッションIDを使用し、セッション情報が混在するリスク

**対策**: セッションIDは必ず `invocation_state` から受け取り、直接文字列を指定しない

### 8.4 承認フック

**リスク**: HumanApprovalHookの承認待ち中にセッションがタイムアウトし、承認操作が無効になるリスク

**対策**: タイムアウト時の扱いは要件上未定義。要件確定後に設定する

## 9. ログ設計の方針

### 9.1 基本原則

- **ログ設定の一元管理**: ログレベル・出力先・フォーマットはエントリーポイント（`main.py`）で一元設定する
- **モジュール別責任**: ログ出力は各モジュール（エージェント・ツール・ハンドラー）が責任を持って行う
- **ログレベルの環境変数制御**: ログレベルは環境変数 `LOG_LEVEL` で制御し、未設定時のデフォルトは `WARNING` とする
- **ユーザー向けメッセージの分離**: ユーザー向けメッセージ（CLI表示）と技術ログ（ファイル出力）を明確に分離する

### 9.2 ログレベル設定

| 環境変数 `LOG_LEVEL` の値 | 適用ログレベル | 主な用途 |
|-------------------------|-------------|---------|
| DEBUG | logging.DEBUG | 開発・デバッグ時の詳細ログ出力 |
| INFO | logging.INFO | 通常運用時の情報ログ |
| WARNING（デフォルト） | logging.WARNING | 警告以上のみ出力（未設定時） |
| ERROR | logging.ERROR | エラー以上のみ出力 |

### 9.3 コンテキスト情報の付加

各モジュールが付加するコンテキスト情報の詳細は基本設計・詳細設計で定義する。

> ※ 具体的なコンテキストフィールド（session_id・agent_id・tool_name等の付加方式）は基本設計フェーズで定義する
