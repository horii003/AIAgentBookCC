## 1. 概要

### 1.1 コンポーネントの目的

AG-002（交通費精算申請エージェント）から受け取った移動区間情報（出発地・目的地・交通手段・移動日・業務目的）をもとに、対応する交通費を算出して返す。電車の場合は `data/train_routes.json` の経路テーブルを双方向検索し、バス・タクシー・飛行機の場合は `data/fixed_fares.json` の固定運賃テーブルを参照する。算出根拠（参照テーブル・エントリの要約）を必ず返し、AG-002がユーザーへ透明な説明を行えるようにする。

### 1.2 主要な責務

1. **TransportToolInput のバリデーション**: 入力パラメータを Pydantic v2 モデルで検証し、駅名・交通手段の正規化を実施する
2. **マスタデータ読み込み**: `_load_train_routes()` / `_load_fixed_fares()` で各 JSON ファイルを読み込み、Pydantic モデルで構造検証する
3. **電車運賃検索**: `train_routes.json` の `routes` キー配下から（出発地, 目的地）の完全一致を双方向で検索する
4. **固定運賃取得**: `fixed_fares.json` から交通手段（バス/タクシー/飛行機）の固定運賃を取得する
5. **エラー返却**: 経路不明・ファイル参照失敗等の場合、例外を再送出せず `{"success": False, "message": ...}` を返す
6. **算出根拠の記録**: 参照テーブル名・エントリ内容を `calculation_basis` フィールドとして返す

## 3. calculate_transport_fareツール設計

### 3.1 基本情報

#### ツール名（関数名）
`calculate_transport_fare`

#### 説明（docstring）
```
移動区間の交通費を算出する。

電車の場合は train_routes.json の経路テーブルから出発地・目的地の組み合わせを双方向検索し、
バス・タクシー・飛行機の場合は fixed_fares.json の固定運賃テーブルから取得する。
申請者名・申請日は invocation_state から取得する。LLM はパラメータとして渡さない。
算出根拠（参照テーブル・エントリの要約）を result["calculation_basis"] として返す。

Args:
    departure (str): 出発地（駅名または地点名）。「駅」「Station」等の接尾語は自動除去される。
    destination (str): 目的地（駅名または地点名）。「駅」「Station」等の接尾語は自動除去される。
    transportation_type (str): 交通手段。「電車」「バス」「タクシー」「飛行機」のいずれか。
        以下の別表記は自動正規化される:
        - 「JR」「新幹線」「地下鉄」「モノレール」→「電車」
        - 「ハイヤー」「タクシー代」→「タクシー」
        - 「飛機」「航空」「ANA」「JAL」→「飛行機」
        - 「路線バス」→「バス」
    travel_date (str): 移動日（YYYY-MM-DD形式）。未来日付・90日超過は拒否される。
    purpose (str): 業務目的（空文字不可）。

Returns:
    dict: 以下のキーを持つ辞書:
        - success (bool): 処理成否。
        - fare (int): 算出運賃（円）。success=False の場合は 0。
        - calculation_basis (str): 算出根拠の説明文。success=False の場合は空文字。
        - message (str): エラーメッセージ。success=True の場合は空文字。
```

| `"モノレール"` | `"電車"` | 同上 |

| `"飛機"` | `"飛行機"` | 同上 |

| `"路線バス"` | `"バス"` | 同上 |

#### 3.2.2 出力

**戻り値の型**: `dict[str, bool | int | str]`

**説明**: 処理成否・算出運賃・算出根拠・エラーメッセージを含む辞書。例外は再送出しない。

**構造**:
```python
{
    "success": bool,            # 処理成否
    "fare": int,                # 算出運賃（円）。success=False の場合は 0
    "calculation_basis": str,   # 算出根拠の説明文。success=False の場合は ""
    "message": str,             # エラーメッセージ。success=True の場合は ""
}
```

#### 3.2.3 例外

| 例外 | 発生条件 |
|------|---------|
| `ValidationError` (Pydantic) | 入力値がバリデーションルールに違反した場合（EX-01/EX-03として処理） |
| `ValueError` | train_routes.json に経路が存在しない場合 / fixed_fares.json に交通手段が存在しない場合（EX-02として処理） |
| `json.JSONDecodeError` | JSON ファイルのパースに失敗した場合（EX-06として処理） |

> **注意**: `FileNotFoundError` は raise しない。ファイル存在確認は `os.path.exists()` で事前チェックし、存在しない場合は即座にエラーメッセージを返す。

### 3.3 ビジネスロジック

#### 3.3.1 処理フロー

```
開始: calculate_transport_fare(context, departure, destination, transportation_type, travel_date, purpose) 呼び出し
  ↓
【ステップ0】invocation_state から申請者情報取得
  - applicant_name = context.invocation_state.get("applicant_name", "")
  - application_date = context.invocation_state.get("application_date", "")
  ↓
【ステップ1】TransportToolInput バリデーション
  - TransportToolInput(departure=departure, destination=destination,
      transportation_type=transportation_type, travel_date=travel_date, purpose=purpose) を生成
  - normalize_station_name (mode="before"): departure, destination から接尾語除去・空白トリム
  - normalize_transportation_type (mode="before"): transportation_type を許容4種類に正規化
  - travel_date: 文字列→date変換。未来日付禁止チェック
  - check_application_deadline (model_validator mode="after"): 90日超過チェック
  - ValidationError 発生時 → 【エラーフロー1: バリデーションエラー】へ
  ↓
【ステップ2】交通手段による分岐
  - 正規化済み transportation_type が「電車」か？
    YES → 【ステップ3a: 電車運賃検索】
    NO  → 【ステップ3b: 固定運賃取得】
  ↓
【ステップ3a: 電車運賃検索】
  - os.path.exists(TRAIN_ROUTES_PATH) でファイル存在確認
  - ファイルが存在しない場合 → 【エラーフロー2: ファイル参照失敗（ファイル不在）】へ
  - _load_train_routes() で train_routes.json を読み込む
  - JSONパース失敗 → 【エラーフロー3: データ不整合】へ
  - TrainRouteMaster モデルで構造検証
  - routes リストを走査し (departure, destination) または (destination, departure) が一致するエントリを検索
  - 一致エントリが見つかった → fare = entry.fare, calculation_basis = "電車経路テーブル（train_routes.json）より: {departure}→{destination} {fare}円"
  - 一致エントリが見つからない → 【エラーフロー4: 経路不明（EX-02）】へ（ValueError raise）
  ↓ (fare, calculation_basis 確定)
【ステップ4】成功レスポンス生成
  - return {"success": True, "fare": fare, "calculation_basis": calculation_basis, "message": ""}
  ↓
終了

【ステップ3b: 固定運賃取得】
  - os.path.exists(FIXED_FARES_PATH) でファイル存在確認
  - ファイルが存在しない場合 → 【エラーフロー2: ファイル参照失敗（ファイル不在）】へ
  - _load_fixed_fares() でfixed_fares.jsonを読み込む
  - JSONパース失敗 → 【エラーフロー3: データ不整合】へ
  - FixedFareMaster モデルで構造検証
  - entries リストを走査し transportation_type が一致するエントリを検索
  - 一致エントリが見つかった → fare = entry.fare, calculation_basis = "固定運賃テーブル（fixed_fares.json）より: {transportation_type} {fare}円"
  - 一致エントリが見つからない → 【エラーフロー4: 交通手段不明（EX-02）】へ（ValueError raise）
  ↓ (fare, calculation_basis 確定)
→ 【ステップ4】へ
```

#### 3.3.3 データの検証方法（Pydantic モデル使用）

**Pydanticモデル**: `TransportToolInput`

**検証内容**:
- `departure`, `destination`: `normalize_station_name` (mode="before") → `min_length=1` チェック
- `transportation_type`: `normalize_transportation_type` (mode="before") → `Literal["電車", "バス", "タクシー", "飛行機"]` 型チェック
- `travel_date`: 文字列→`date`自動変換 → 未来日付禁止チェック
- モデル全体: `check_application_deadline` (model_validator mode="after") → 90日超過チェック
- `purpose`: `strip_and_validate_not_empty` (mode="before") → `min_length=1` チェック

**Pydanticモデル**: `TrainRouteMaster`

**検証内容**:
- `routes: list[TrainRouteEntry]`: ファイル読み込み後に全エントリの型・制約を一括検証
- `TrainRouteEntry.fare`: `ge=0`

**Pydanticモデル**: `FixedFareMaster`

**検証内容**:
- `entries: list[FixedFareEntry]`: ファイル読み込み後に全エントリの型・制約を一括検証
- `FixedFareEntry.transportation_type`: `Literal["バス", "タクシー", "飛行機"]`
- `FixedFareEntry.fare`: `ge=0`

### 5.2 データモデル

#### TransportToolInputモデル

| フィールド名 | 型 | 必須/任意 | 説明 | 制約 |
|--------------|-----|-----------|------|------|
| `departure` | `str` | 必須 | 出発地（正規化前後ともに使用） | `min_length=1`; `normalize_station_name` (mode="before") |
| `destination` | `str` | 必須 | 目的地 | `min_length=1`; `normalize_station_name` (mode="before") |
| `transportation_type` | `Literal["電車", "バス", "タクシー", "飛行機"]` | 必須 | 交通手段 | `normalize_transportation_type` (mode="before") |
| `travel_date` | `date` | 必須 | 移動日 | 当日以前; 文字列→date自動変換 |
| `purpose` | `str` | 必須 | 業務目的 | `min_length=1`; `strip_and_validate_not_empty` (mode="before") |

**バリデーション**:
- `normalize_station_name`: `departure`, `destination` の接尾語除去・空白トリム (mode="before")
- `normalize_transportation_type`: 別表記→正規4種類のマッピング変換 (mode="before")
- `strip_and_validate_not_empty`: `purpose` の前後空白トリム・空文字禁止 (mode="before")
- `check_application_deadline`: `travel_date` が本日から90日以内であることを確認 (model_validator mode="after")
- `model_config = ConfigDict(strict=False)`: 自動型変換を許容

## 6. 補足情報

### 6.1 実装上の注意点

1. **ファイル読み込みタイミング**
   - ツール呼び出しのたびに毎回ファイルを読み込む（キャッシュなし）。ファイルパスは `config/settings.py` の設定値（`TRAIN_FARES_PATH`, `FIXED_FARES_PATH`）から取得する。

2. **双方向検索**
   - 電車運賃テーブルの検索は（departure, destination）と（destination, departure）の両方向を確認する。往路・復路のどちらかにエントリがあれば運賃を返す。

3. **例外の非再送出**
   - ツール関数は例外を再送出しない。全例外を `{"success": False, "message": ...}` 形式で返し、呼び出し元（AG-002）に処理を委ねる。

4. **TransportToolInput への変換**
   - 受け取った文字列パラメータを `TransportToolInput(**kwargs)` でインスタンス化し、ValidationError が発生した場合は `error_handler.handle_validation_error(e)` に委譲する。

5. **@tool デコレータ**
   - `@tool(context=True)` デコレータを付与し、Strands Agents SDK に認識させる。`context=True` により invocation_state を参照できる。

---

### 6.2 パフォーマンス考慮事項

1. **JSONファイルサイズ**
   - 現時点で運賃テーブルのエントリ数は少量を想定しており、毎回読み込みによる遅延は許容範囲。大規模化する場合はキャッシュを検討する。

2. **テーブル検索の計算量**
   - 線形検索（`O(n)`）。運賃テーブルが数千件規模になる場合は辞書インデックスへの変換を検討する。

---

### 6.3 セキュリティ考慮事項

1. **ファイルパスのハードコーディング禁止**
   - ファイルパスは `config/settings.py` から取得し、実行時の外部入力によるパストラバーサルを防ぐ。

2. **入力サニタイジング**
   - Pydantic バリデーションにより、不正な型・長さ・値の入力を早期に拒否する。ツール内部でのデータ操作に不正値が届かないことを保証する。

## 8. テスト観点

### 8.1 機能テスト
- 電車区間（渋谷→新宿等）でテーブルに存在する区間の運賃が正しく返されること（success=True, fare=正しい値）
- バスで固定運賃テーブルから正しい値が返されること
- タクシーで固定運賃テーブルから正しい値が返されること
- 飛行機で固定運賃テーブルから正しい値が返されること
- 電車の復路（テーブルが往路のみ登録されている場合）で双方向検索により運賃が取得できること
- 駅名に「駅」接尾語がついている場合（「渋谷駅」→「渋谷」）に正規化後の検索が成功すること
- 交通手段の別表記（「JR」→「電車」等）が正規化されてテーブル検索に使用されること
- `calculation_basis` に参照テーブル名とエントリ情報が含まれること

### 8.2 異常系テスト
- 電車テーブルに存在しない区間でsuccess=False・手動入力促しメッセージが返されること（EX-02）
- 固定運賃テーブルに存在しない交通手段でsuccess=False・手動入力促しメッセージが返されること（EX-02）
- train_fares.json が存在しない場合にsuccess=False・EX-04メッセージが返されること
- fixed_fares.json が存在しない場合にsuccess=False・EX-04メッセージが返されること
- JSONフォーマット不正のtrain_fares.jsonでsuccess=False・EX-06メッセージが返されること
- 対応外交通手段（「自動車」等）でValidationError（EX-01）が発生し、適切なメッセージが返されること
- 申請期限超過（91日前の移動日）でValidationError（EX-03）が発生し、適切なメッセージが返されること
- 出発地・目的地が空文字の場合にValidationError（EX-01）が返されること
- 業務目的が空文字の場合にValidationError（EX-01）が返されること
- 未来日付の移動日でValidationError（EX-01）が返されること

### 8.3 性能テスト
- train_fares.jsonが1000エントリの場合に応答時間1秒以内であること（線形検索の性能確認）
- 連続100回呼び出しでメモリリークがないこと（毎回ファイル読み込みの場合）

### 8.4 境界値テスト
- 申請期限ちょうど（90日前）の移動日でsuccess=Trueが返されること
- 申請期限超過（91日前）の移動日でValidationError（EX-03）が返されること
- 運賃が0円のエントリでsuccess=True, fare=0が正しく返されること
- 駅名がスペースのみの入力（「　」）でValidationError（EX-01）が返されること
