## 1. 目的・適用範囲

### 1.1 目的

システム全体として **「例外とは何か」「どう分類し」「どう収束させるか」** の共通方針を定義する

個々のエージェント設計に **一貫した判断軸** を提供する

### 1.2 適用範囲

- マルチエージェントを含む全システム（AG-001, AG-002, AG-003, TOOL-001, TOOL-002）
- 業務処理、CLIインタフェース、ファイルI/O、運用を含む

### 1.3 本方針で定義しないこと

- エージェント内部の例外検知ロジック（基本設計書に委ねる）
- SDK／ライブラリ依存の例外クラス設計
- 具体的な再試行アルゴリズムの実装詳細

---

## 2. 例外処理の基本原則（システム共通）

- **例外は必ず収束させる（無限リトライ禁止）**：エージェントのReActループは最大10回で LoopControlHook が強制停止する
- **失敗は状態として管理する**：例外発生時はセッション状態（TERMINATED等）に反映し、再起動後も状態を把握できるようにする
- **例外＝即エラー表示ではない**：入力例外・判断不能はユーザーへの質問返しで継続する
- **ユーザーに内部失敗を見せない**：スタックトレース・SDK内部エラーはログにのみ記録し、ユーザーには抽象化したメッセージを提示する
- **後続処理に影響を残さない**：ツール関数は例外を再送出せず `{"success": False, "message": エラーメッセージ}` を返す


---

## 3. 例外の定義と分類（システム観点）

### 3.1 例外の定義

「通常の業務フローから逸脱し、継続判断が必要な状態」

### 3.2 例外分類

| 分類ID | 分類名 | 説明 | 例 |
|-------|-------|------|---|
| EX-01 | 入力例外 | 入力不正・不足 | 500文字超の入力（GRD-017）・必須項目未入力・駅名が存在しない |
| EX-02 | 判断不能 | AIが結論を出せない | 申請種別が交通費・経費のいずれとも判定できない・経費区分が4区分のいずれとも判断できない |
| EX-03 | 業務ルール違反 | 定義済みルール不一致 | 申請期限超過（BRL-13/BRL-18）・申請書自動提出禁止違反（BRL-09）・上長承認未確認で申請書生成試行 |
| EX-04 | 外部I/F例外 | API障害・タイムアウト | Amazon Bedrock（LLM）への接続エラー・タイムアウト |
| EX-05 | 実行不可 | 権限・状態不整合 | TOOL-002を利用者確認前に呼び出し（GRD-016）・AG-003がTOOL-001を呼び出し |
| EX-06 | データ不整合 | データ形式不正 | 運賃データJSON（DATA-011/012）の形式不正・申請書テンプレートExcelが存在しない・Pydanticバリデーションエラー |
| EX-07 | システム障害 | 基盤障害 | ファイルストレージへのI/Oエラー（セッション/ログ/申請書）・logsディレクトリへの書き込み不可 |
| EX-08 | 想定外 | 未分類事象 | 上記に分類できない予期しない例外（Exception） |

## 4. 例外検知レベル

### 4.1 検知レベルの考え方

| レベル | 説明 | 検知主体 |
|-------|------|---------|
| 業務レベル | 業務定義上の異常（申請期限超過・入力文字数超過・申請種別不明等） | エージェント（AG-001, AG-002, AG-003） |
| 処理レベル | 処理失敗（交通費計算エラー・申請書生成エラー・Pydanticバリデーションエラー等） | ツール（TOOL-001, TOOL-002） |
| システムレベル | 基盤・インフラ異常（LLM接続エラー・ファイルI/Oエラー等） | ErrorHandler |

### 4.2 検知責務

- 検知は **エージェントまたはツール**
- **判断・収束はErrorHandler** が行う

## 5. 例外発生時の基本対応フロー

### 5.1 共通対応フロー

1. 例外検知（エージェント/ツール）
2. 例外分類（ErrorHandler）
3. 継続可否判定（ErrorHandler）
4. 処理分岐（再試行／代替処理／停止）
5. 状態更新（Strands Agents SDKセッション管理）
6. ログ・監査記録（ErrorHandler）

## 7. 例外ハンドリング実装

### 7.1 ErrorHandlerの責務方針

**エラー伝播の責務分担**:
- **各モジュール（エージェント・ツール）**: 例外発生箇所でログを出力し、ErrorHandler へ処理を委譲する
- **ErrorHandler**: ユーザー向けメッセージを生成して呼び出し元に返す
- **呼び出し元**: 受け取ったメッセージを戻り値または表示として上位レイヤーへ伝播させる

ErrorHandlerは以下の責務を担う。

- 例外分類（EX-01〜EX-08）ごとに継続可否を判定し、対応方針（再試行・代替処理・エスカレーション・中断）を決定する
- ユーザー向けの日本語メッセージを生成し、技術的な詳細を隠蔽する
- 例外発生時のログ出力（ERROR/CRITICALレベル）を行い、スタックトレースはログにのみ記録する
- EventLoopException（`__cause__` で LoopLimitError を判定）→ Exception の2層構造でキャッチする
- ドメイン固有のエラーハンドラー（申請期限超過・上長承認未確認・運賃データ未対応等）を提供する

具体的なメソッド設計（メソッド名・引数・戻り値）は基本設計（ハンドラー基本設計.md）で定義する。

> ※ 基本設計書への参照: ハンドラー基本設計.md 2.3節


### 7.2 エラーメッセージ設計方針

**ユーザーフレンドリー**:
- 技術的な詳細を隠蔽（スタックトレース・SDK内部エラーは非表示）
- 具体的な対処方法を提示
- 丁寧な日本語の言葉遣い

**情報の充実**:
- エラーの原因を明確に説明
- 複数の対処方法を提示（再入力・再試行・システム管理者連絡）
- 次のアクションを必ず提示する

**例**:
```
入力例外（EX-01）:
「入力が500文字を超えています。もう少し短くして、もう一度入力してください。」

申請期限超過（EX-03）:
「申し訳ありません。経費発生日から3ヶ月を超えているため、この申請は受け付けられません。総務部門にご相談ください。」

運賃データなし（EX-06）:
「申し訳ありません。指定された経路の運賃データが見つかりませんでした。交通費を手動で入力してください。」

システム障害（EX-07）:
「申し訳ありません。システムエラーが発生しました。システム管理者にご連絡いただくか、しばらく時間をおいて再度お試しください。」

想定外エラー（EX-08）:
「申し訳ありません。予期しないエラーが発生しました。システム管理者にご連絡ください。」
```

## 8. 例外とエスカレーションの関係

### 8.1 エスカレーション対象例外

| 例外分類 | エスカレーション要否 | 行き先 |
|---------|------------------|-------|
| EX-04 外部I/F例外（LLMリトライ失敗後） | 要 | ユーザーへのシステム管理者連絡案内（エスカレーション先は要件上未定義） |
| EX-05 実行不可 | 要 | ユーザーへのシステム管理者連絡案内 |
| EX-06 データ不整合（テンプレートなし） | 要 | ユーザーへのシステム管理者連絡案内 |
| EX-07 システム障害 | 要 | ユーザーへのシステム管理者連絡案内 |
| EX-08 想定外 | 要 | ユーザーへのシステム管理者連絡案内 |

### 8.2 エスカレーション判定責務

**判定主体**: ErrorHandler

**判定基準**: 以下の条件のいずれかを満たす場合にエスカレーションと判定する
- 再試行・代替処理でも継続不能な例外（EX-04再試行失敗後、EX-05、EX-07、EX-08）
- ユーザー操作で解決不能な内部障害（EX-06テンプレートなし）

## 9. セッション状態への反映方針

### 9.1 例外時の状態更新

| 状態 | 説明 | トリガー |
|-----|------|---------|
| ACTIVE | 処理中（例外発生後も継続可能） | EX-01, EX-02（代替処理で継続）, EX-06（代替処理で継続） |
| WAITING | Human-in-the-Loop承認待ち | EX-03の一部（申請内容修正後の再確認待ち） |
| TERMINATED | 異常終了 | EX-03（申請期限超過）, EX-04（再試行失敗）, EX-05, EX-07, EX-08 |

### 9.2 再開可否

**再開可能な例外**:
- EX-01（入力例外）：セッション継続中のため、修正入力後に再開可能
- EX-02（判断不能）：ユーザー選択後に再開可能
- EX-06（データ不整合：バリデーションエラー）：再入力後に再開可能

**再開不可な例外**:
- EX-03（申請期限超過）：申請自体が不可のため再開不可
- EX-04（LLMリトライ全失敗）：インフラ障害のため再開不可
- EX-05（実行不可）：権限・状態不整合のため再開不可
- EX-07（システム障害）：ファイルI/Oエラー等のため再開不可
- EX-08（想定外）：原因不明のため再開不可

## 10. UX観点の扱い（概要）

**利用者への通知有無**:
- 全例外に対してユーザーへのメッセージを提示する（エラー発生を隠蔽しない）
- ただし技術的な詳細（スタックトレース・エラーコード）は非表示とする

**エラーメッセージの抽象度**:
- 業務レベルの例外（EX-01〜EX-03）：原因と次のアクションを具体的に提示する
- システムレベルの例外（EX-04〜EX-08）：「システムエラー」として抽象化し、システム管理者への連絡を促す

**次の行動を必ず提示**:
- 全てのエラーメッセージに次のアクション（再入力・システム管理者連絡・再試行等）を含める
- CLIの場合は操作例（入力例）を提示することで再試行を促す

## 11. ログ・監査方針

### 11.1 例外ログ

| 項目 | 内容 | 出力先 |
|-----|------|-------|
| error_type | エラータイプ（EX-01〜EX-08） | logs/error.log（ERROR/CRITICALレベル） |
| error_message | エラーメッセージ（ユーザー向けとは別に内部詳細も記録） | logs/error.log |
| context | エラーコンテキスト（セッションID・エージェントID・ツール名等） | logs/error.log |
| exc_info | スタックトレース（EX-07/EX-08の場合は必ず記録） | logs/error.log |
| timestamp | タイムスタンプ | logs/error.log |


### 11.2 監査観点

**再現可能性**:
- エラーコンテキスト（セッションID・入力情報）をログに記録し、後から再現・調査が可能な状態を維持する

**判断経路の説明可能性**:
- ErrorHandlerがエラー分類・継続可否判定の根拠（適用した業務ルールID・エラー種別）をログに記録する

## 12. 設計上の留意点・リスク

**例外過多によるUX低下**:
- 過度に厳格な入力バリデーションにより、頻繁にエラーが発生するリスク
- 対策: EX-01（入力例外）は再入力を促す質問返しで継続し、ユーザーに対してエラーではなく「次のステップ」として提示する

**例外の握り潰し**:
- ツール関数内でエラーを適切に返却せずに無視するリスク
- 対策: ツール関数は例外を再送出せず `{"success": False, "message": エラーメッセージ}` 形式で返す。ErrorHandlerを通じてログ記録を必ず行う

**想定外例外の扱い**:
- 未分類の例外（EX-08）が握り潰されるリスク
- 対策: 全コンポーネントで `except Exception` をキャッチし、ErrorHandlerの `handle_unexpected_error()` に委譲する。ログにはスタックトレースを必ず記録する
