## 1. 概要

### 1.1 コンポーネントの目的

AG-001（申請受付窓口エージェント）は、社員が入力した申請内容テキストから申請種別（交通費精算申請・経費精算申請）を自動判断し、適切な申請先を提示するオーケストレーターエージェントである。申請種別確定後は、対応する専門エージェント（AG-002またはAG-003）へ申請フローを委譲する。社員にとっての「申請受付窓口」として、どの申請フローを進めるべきかをガイドする役割を担う。

### 1.2 主要な責務

1. **申請内容テキスト受付**: 社員が入力した申請内容テキスト（最大500文字）を受け取る（GRD-001, GRD-004）
2. **申請種別判断**: 社内申請ルール（DATA-001）を参照して申請種別（交通費精算申請・経費精算申請）を判断する（BRL-01）
3. **判断不能時の選択肢提示**: 申請種別を自動判定できない場合は選択肢を提示して社員に確認する（BRL-09）
4. **申請書・申請先提示**: 申請種別が確定したら対応する申請書・申請先を根拠ルールIDとともに提示する（BRL-02）
5. **専門エージェントへの委譲**: 申請種別確定後、AG-002またはAG-003をツールとして呼び出して申請フローを委譲する（LNK-001/LNK-002）
6. **特殊コマンド処理**: `reset`/`リセット`/`最初から` 入力時は会話履歴と申請者情報をリセットして初期状態に戻す
7. **対話回数管理**: ターン数が上限（30回: GRD-005）に達した場合はセッション状態をTERMINATEDに遷移してフローを終了する

### 2.2 インターフェース設計

#### 2.2.1 入力

**初期化時の入力**（`main.py` が起動時に設定済み）:
- `applicant_name`（str, 必須）: 申請者名。アプリケーション起動時に取得済み。`invocation_state` にセット済み
- `application_date`（str, 必須）: 申請日（YYYY-MM-DD形式）。`date.today().isoformat()` で自動取得。`invocation_state` にセット済み
- `session_id`（str, 必須）: SM-001が発行済み。`invocation_state` にセット済み

**対話ループでの入力**:
- ユーザー入力（自然言語での申請内容テキスト）: 最大500文字（GRD-004）、空入力禁止（GRD-001）
- 特殊コマンド（通常入力より優先して処理）:
  - `reset` / `リセット` / `最初から`: 会話履歴と申請者情報をリセットして初期状態に戻す（§4.5参照）

**ToolContext経由で受け取る情報**（AG-001自身が参照する情報）:
```python
{
    "session_id": str,         # セッションID（SM-001が発行）
    "applicant_name": str,     # 申請者名（アプリケーション起動時に設定済み）
    "application_date": str    # 申請日（YYYY-MM-DD形式）
}
```

#### 2.2.2 出力

**出力先**: コンソール（チャットインターフェース）

**出力メッセージ**:
- ウェルカムメッセージ（初回起動時）: `"社内申請AIアシスタントへようこそ。交通費精算申請・経費精算申請のお手続きをサポートします。"`
- 申請種別確定時（交通費）: `"申請内容を確認しました。「交通費精算申請」に該当します（根拠: BRL-01）。交通費申請書_template.xlsx を使用し、総務部に申請してください。"`
- 申請種別確定時（経費）: `"申請内容を確認しました。「経費精算申請」に該当します（根拠: BRL-01）。経費精算申請書_template.xlsx を使用し、総務部に申請してください。"`
- 複数種別該当時（BRL-03）: `"申請内容に複数の申請種別が該当します（根拠: BRL-03）。\n1. 交通費精算申請（交通費申請書_template.xlsx、総務部へ）\n2. 経費精算申請（経費精算申請書_template.xlsx、総務部へ）\nどちらの申請フローを進めますか？"`
- 判断不能時（BRL-09）: `"申請種別の判断ができませんでした。以下からお選びください：\n1. 交通費精算申請\n2. 経費精算申請"`
- 委譲完了後の申請先案内: `"申請書（下書き）が生成されました。総務部（submit@example.com）に提出してください。申請書の提出はご自身で行ってください。"`
- リセット完了時: `"リセットしました。最初からやり直します。お名前を入力してください。"`

**戻り値の形式**: `str`（エージェントからの応答）

### 2.3 ビジネスロジック

#### 2.3.1 システムプロンプト

##### システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 静的定数 |
| 動的生成の理由 | 不要。AG-001は申請種別判断のみを担当し、申請期限チェック等の日付ベースのルール判断はAG-002/AG-003が担当するため |

##### プロンプト全文

```
あなたは社内申請AIアシスタントシステムの「申請受付窓口エージェント」です。
社員が入力した申請内容テキストから申請種別を判断し、適切な専門エージェントへ申請フローを委譲します。

【役割】
1. 社員が入力した申請内容テキストを受け取り、申請種別（交通費精算申請・経費精算申請）を自動判断する
2. 申請種別が確定したら、対応する申請書名・申請先を根拠ルールIDとともに提示する
3. 申請種別判断不能の場合は選択肢を提示して社員に確認する
4. 申請種別が確定したら、対応する専門エージェントへ申請フローを委譲する
5. あなた自身は申請書の生成・提出を一切行わない（禁止事項）

【申請種別判断ルール（BRL-01）】
以下のキーワードや特徴をもとに申請種別を判断する:

交通費精算申請に該当するキーワード・特徴:
- 「出張」「交通費」「電車」「移動」「往復」「バス」「タクシー」「飛行機」「新幹線」
- 交通機関を利用した移動に関する内容

経費精算申請に該当するキーワード・特徴:
- 「領収書」「経費」「食事」「宿泊」「接待費」「事務用品」「ホテル」「会食」「消耗品」「資格」「試験代」
- 交通費以外の業務経費に関する内容

判断できない場合・複数に該当する場合:
- 選択肢を提示して社員に確認する（BRL-09）
- 複数種別に該当する場合は全て提示する（BRL-03）

【申請書・申請先提示ルール（BRL-02）】
申請種別確定後に必ず以下を提示する:
- 交通費精算申請: 使用テンプレート「交通費申請書_template.xlsx」、申請先「総務部」
- 経費精算申請: 使用テンプレート「経費精算申請書_template.xlsx」、申請先「総務部」
- 判断根拠のルールID（BRL-01等）を必ず含める

【専門エージェントへの委譲（LNK-001/LNK-002）】
申請種別が確定したら次のルールに従って委譲する:
- 交通費精算申請 → transport_agent_tool を呼び出す（LNK-001）
- 経費精算申請 → expense_agent_tool を呼び出す（LNK-002）
- 委譲時は必ず申請種別（application_type）・申請者名（applicant_name）・申請内容テキスト（user_input_text）の3つのパラメータを設定すること
- 同一セッション内で両ツールを同時に呼び出してはならない（禁止）

【ツール呼び出し時のパラメータ構造】
transport_agent_tool または expense_agent_tool を呼び出す際は、以下のJSON構造のパラメータを必ず指定すること:
```json
{
  "application_type": "交通費精算申請" または "経費精算申請",  // 必須: 申請種別
  "applicant_name": "<申請者名>",                             // 必須: 申請者名（空文字不可）
  "user_input_text": "<社員の申請内容テキスト>"               // 必須: 500文字以内
}
```
パラメータのキーを省略したり、JSON構造を変更したりしてはならない。

【禁止事項】
- 申請書の生成（P-EXEC-002禁止）
- 申請書の提出（P-EXEC-003禁止 / ACT-EXEC-01）
- 申請者名の対話での収集（起動時に取得済みのため不要）
- AG-002/AG-003の同時呼び出し（1セッション1申請種別のみ）

【対話回数上限（GRD-005）】
対話ターン数が30回に達した場合は、以下のメッセージを返してフローを終了する:
「申し訳ありません。対話回数の上限（30回）に達したため、申請フローを終了します。
お手数ですが、担当部門（管理部）にお問い合わせください。」

【特殊コマンド】
社員が以下のいずれかを入力した場合は、通常の処理フローに入る前に優先して処理する:
- reset / リセット / 最初から: セッションをリセットして初期状態に戻す
```

##### プロンプト設計の意図

| 要素 | 意図 |
|------|------|
| 役割定義 | AG-001がオーケストレーターとして申請種別判断と委譲のみを担当することを明確化し、申請書生成・提出を禁止する |
| BRL-01判断ルール | キーワード一覧を明示することでLLMが誤ったエージェントへ委譲するリスクを低減する |
| BRL-02提示ルール | 申請書名・申請先・根拠ルールIDを必ず提示させ、社員が確認できるようにする（根拠提示方針準拠） |
| 委譲パラメータ構造 | dict型パラメータの必須キーを明示し、LLMがキーを省略した際の実行時エラーを防止する |
| 禁止事項 | 自律度Lv2（提案）の範囲を超える行動を明示的に禁止する |
| 静的プロンプト | AG-001は日付依存のルールを扱わないため静的プロンプトで十分。動的生成のオーバーヘッドを排除する |

#### 2.3.2 申請種別判断ルール

| ルール | 条件 | 対応 |
|--------|------|------|
| BRL-01（申請種別判断） | 申請内容テキストが存在し社内申請ルールが参照可能 | 社内申請ルール照合で申請種別（交通費/経費）を判断 |
| BRL-02（申請書・申請先提示） | 申請種別が確定した直後 | 申請書名・申請先部門を根拠ルールIDとともに提示 |
| BRL-03（複数申請種別の全提示） | 申請内容に複数申請種別が該当する場合 | 該当する全申請種別・申請書・申請先を提示 |
| BRL-09（判断不能時の選択肢提示） | 申請種別が自動判定できない場合（JD-02） | 交通費精算申請/経費精算申請の選択肢を提示して社員に確認 |

#### 2.3.3 処理フロー

```

  ↓
【ループ開始: 社員からの入力受付】
  ↓
【ステップ1: 入力受付と前処理】
  - 社員からの入力テキストを受け取る（プロンプト: "\n\n入力内容（終了時はquit）: "）
  - "quit" 入力時はループを終了する
  - 特殊コマンド判定（reset / リセット / 最初から）
    → 特殊コマンドの場合 → 【特殊コマンド処理フロー】へ
    → 通常入力の場合 → ステップ2へ
  ↓
【ステップ2: 入力バリデーション】
  - 空入力チェック（GRD-001）
    → 空の場合 → EX-01: "申請内容を入力してください。" を返してループに戻る
  - 文字数チェック（GRD-004: 最大500文字）
    → 超過の場合 → EX-01: "申請内容は500文字以内で入力してください（現在{n}文字）。" を返してループに戻る
  ↓
【ステップ3: 申請種別判断（BRL-01）】
  - LLMが申請内容テキストを解析して申請種別を判断する
  - 判断結果:
    → 交通費精算申請に確定 → ステップ4a
    → 経費精算申請に確定 → ステップ4b
    → 判断不能・複数該当 → ステップ3a
  ↓
【ステップ3a: 判断不能時フロー（BRL-09/BRL-03）】
  - 選択肢を社員に提示:
    "申請種別の判断ができませんでした。以下からお選びください：\n1. 交通費精算申請\n2. 経費精算申請"
  - 社員の選択を待つ（ループ）
  - 社員が選択 → 申請種別確定 → ステップ4a または ステップ4b
  ↓
【ステップ4a: 交通費精算申請確定時】
  - 申請書・申請先を提示（BRL-02）:
    "申請内容を確認しました。「交通費精算申請」に該当します（根拠: BRL-01）。
    交通費申請書_template.xlsx を使用し、総務部に申請してください。"
  - invocation_state に application_type = "交通費精算申請" を設定
  - SessionManager で session_status を ACTIVE に更新
  - transport_agent_tool を呼び出す（LNK-001）
    パラメータ: {application_type, applicant_name, user_input_text}
  - 委譲完了後 → ステップ5
  ↓
【ステップ4b: 経費精算申請確定時】
  - 申請書・申請先を提示（BRL-02）:
    "申請内容を確認しました。「経費精算申請」に該当します（根拠: BRL-01）。
    経費精算申請書_template.xlsx を使用し、総務部に申請してください。"
  - invocation_state に application_type = "経費精算申請" を設定
  - SessionManager で session_status を ACTIVE に更新
  - expense_agent_tool を呼び出す（LNK-002）
    パラメータ: {application_type, applicant_name, user_input_text}
  - 委譲完了後 → ステップ5
  ↓
【ステップ5: 委譲完了後の申請先案内】
  - 申請先案内を提示:
    "申請書（下書き）が生成されました。総務部（submit@example.com）に提出してください。
    申請書の提出はご自身で行ってください。"
  - SessionManager で session_status を CLOSED に更新
  - フロー終了
  ↓
終了

【特殊コマンド処理フロー（reset / リセット / 最初から）】
  - SlidingWindowConversationManager の会話バッファをクリアする
  - session_id を SM-001 で新規発行し、新しいセッション状態ファイルを作成する（CREATED状態）
  - invocation_state を更新:
    - session_id: 新規発行したもの
    - applicant_name: "" （再入力を促すため空にする）
    - application_date: date.today().isoformat()（現在日付に更新）
  - "リセットしました。最初からやり直します。お名前を入力してください。" を返す
  - ループ先頭に戻る（通常の入力受付へ）
```

### 2.4 設定・構成

#### 2.4.1 共通設定

| 設定項目 | 設定値 | 備考 |
|---------|--------|------|
| 会話管理 | SlidingWindowConversationManager(window_size=30) | 申請種別判断・専門エージェント委譲まで複数ターンの対話を保持するため（共通設定方針 §2.2） |
| リトライ戦略 | 標準設定（Bedrock最大6回・指数バックオフ） | 共通設定方針 §3.2に準拠 |
| フック | LoopControlHook(max_iterations=10) | 全エージェント共通設定（共通設定方針 §4.2）。AG-001はHumanApprovalHookは適用しない |
| セッション管理 | SessionManagerFactory.create() で FileBasedSessionManager を取得し session_manager パラメータに渡す | セッション状態の永続化（storage_path="data/sessions"） |

#### 2.4.2 利用ツール

#### 2.5.2 invocation_stateの受け渡しとAgentインスタンスのキャッシュ

AG-002/AG-003のAgentインスタンスはモジュールレベルの辞書に `session_id` をキーとしてキャッシュする。これにより同一セッション内で複数回呼び出されても会話履歴が保持される。

```python
# session_id をキーにして Agent インスタンスをキャッシュし、
# セッション内で会話履歴を保持する（情報の再入力を防ぐ）
_agent_instances: dict[str, Agent] = {}

@tool(context=True)
def transport_agent_tool(
    application_type: str,
    applicant_name: str,
    user_input_text: str,
    tool_context: ToolContext
) -> str:
    # invocation_state を取得
    state = tool_context.invocation_state
    session_id = state.get("session_id", "")

    # session_id 未登録の場合のみ新しい Agent インスタンスを生成する
    # Agent初期化時にcallback_handler=Noneを指定して標準出力への直接表示を抑制する（AG-001による二重出力を防ぐ）
    if session_id not in _agent_instances:
        _agent_instances[session_id] = Agent(
            model=BedrockModel(model_id="jp.anthropic.claude-sonnet-4-5-20250929-v1:0"),
            system_prompt=build_transport_agent_system_prompt(),
            tools=[calculate_transport_fare, generate_transport_expense_form],
            conversation_manager=SlidingWindowConversationManager(window_size=20),
            hooks=[LoopControlHook(max_iterations=10), HumanApprovalHook(approval_callback=approval_callback)],
            session_manager=SessionManagerFactory.create(),
            callback_handler=None,
        )

    response = _agent_instances[session_id](user_input_text, invocation_state=state)
    return str(response)


@tool(context=True)
def expense_agent_tool(
    application_type: str,
    applicant_name: str,
    user_input_text: str,
    tool_context: ToolContext
) -> str:
    state = tool_context.invocation_state
    session_id = state.get("session_id", "")

    if session_id not in _agent_instances:
        _agent_instances[session_id] = Agent(
            model=BedrockModel(model_id="jp.anthropic.claude-sonnet-4-5-20250929-v1:0"),
            system_prompt=build_expense_agent_system_prompt(),
            tools=[generate_expense_reimbursement_form],
            conversation_manager=SlidingWindowConversationManager(window_size=15),
            hooks=[LoopControlHook(max_iterations=10), HumanApprovalHook(approval_callback=approval_callback)],
            session_manager=SessionManagerFactory.create(),
            callback_handler=None,
        )

    response = _agent_instances[session_id](user_input_text, invocation_state=state)
    return str(response)
```

## 5. 補足情報

### 5.1 実装上の注意点

1. **callback_handler=None の必須設定**
   - `Agent(...)` 初期化時に必ず `callback_handler=None` を指定すること（`agent()` 呼び出し時に渡すと `TypeError` になるため必ずコンストラクタで指定する）
   - これを設定しない場合、AG-002/AG-003の出力がAG-001の出力と重複して表示される（二重出力問題）

2. **_agent_instances キャッシュの管理**
   - `_agent_instances` はモジュールレベルの辞書であり、プロセス起動中はメモリに保持される
   - リセット時は新しい `session_id` で新規インスタンスが生成されるため、古いキャッシュは自然に参照されなくなる（ただしGCされるまでメモリに残る）
   - 長時間運用時はメモリ使用量に注意すること

3. **invocation_state の受け渡し方式**
   - `session_id`, `applicant_name`, `application_date` はLLMのプロンプトには含めず、`invocation_state` 経由で専門エージェントに渡す（マルチエージェント連携設計 §7.3 準拠）
   - ツール関数の引数（`application_type`, `applicant_name`, `user_input_text`）はLLMが設定する

4. **AG-002/AG-003の同時呼び出し禁止**
   - システムプロンプトで明示しているが、実装上も `_agent_instances` の1 `session_id` に対して1エージェントのみキャッシュすること

5. **HumanApprovalHookはAG-001に登録しない**
   - AG-001は申請書生成ツールを呼び出さないため、HumanApprovalHookの登録は不要
   - AG-002/AG-003の初期化時にHumanApprovalHookを登録する

### 5.2 パフォーマンス考慮事項

1. **静的プロンプト使用**
   - AG-001は日付依存のルールを含まない静的プロンプトのため、プロンプト生成のオーバーヘッドが発生しない

2. **Agentインスタンスキャッシュ**
   - `_agent_instances` によりセッション内でAG-002/AG-003のインスタンスが再利用される
   - `SlidingWindowConversationManager` の会話履歴もセッション内で保持されるため、再入力のやり取りが不要になる

### 5.3 セキュリティ考慮事項

1. **申請者名のマスキング**
   - ログ出力時に申請者名（`applicant_name`）をマスキングする（GRD-011）
   - ログメッセージに `applicant_name=***` のように記録する

2. **申請書提出の禁止**
   - システムプロンプトで申請書提出操作を明示的に禁止する（GRD-012 / P-EXEC-003）
   - AG-001のツールリストには申請書提出に関するツールを含めない

  - `build_transport_agent_system_prompt()`: AG-002のシステムプロンプト動的生成
- `agents.expense_agent`: AG-003 Agentインスタンス生成関数
  - `build_expense_agent_system_prompt()`: AG-003のシステムプロンプト動的生成
- `tools.transport_tools`: AG-002が使用するツール
  - `calculate_transport_fare`: 交通費計算ツール（TL-001）
- `tools.output_generator`: 申請書生成ツール
  - `generate_transport_expense_form`: 交通費申請書生成（TL-002a）
  - `generate_expense_reimbursement_form`: 経費申請書生成（TL-002b）
- `handlers.error_handler`: エラーハンドリング
  - `ErrorHandler`: エラー分類・メッセージ生成
- `handlers.loop_control_hook`: LoopControlHook
  - `LoopControlHook`: ループ制御フック（max_iterations=10）
- `handlers.human_approval_hook`: HumanApprovalHook
  - `HumanApprovalHook`: 人間承認フック（AG-002/AG-003専用）
- `session.session_manager`: セッション管理
  - `SessionManagerFactory`: FileBasedSessionManagerの生成ファクトリ
- `models.data_models`: データモデル
  - `InvocationState`: invocation_stateスキーマ
  - `AgentDelegationMessage`: エージェント間メッセージスキーマ

## 7. テスト観点

### 7.1 機能テスト
- 交通費精算申請に該当するキーワード（「出張」「交通費」等）を含む入力でtransport_agent_toolが呼び出されること
- 経費精算申請に該当するキーワード（「領収書」「経費」等）を含む入力でexpense_agent_toolが呼び出されること
- 複数キーワードを含む入力でBRL-03（複数種別全提示）が動作すること
- 申請種別提示時に根拠ルールID（BRL-01）が含まれること（根拠提示方針準拠）
- 申請先提示時にテンプレート名・申請先部門が含まれること（BRL-02）
- BRL-09: 判断不能ケースで選択肢提示後に社員が選択して委譲されること
- リセットコマンド（reset/リセット/最初から）で会話履歴がクリアされ初期状態に戻ること
- リセット後に新しいsession_idで新規セッションが開始されること
- invocation_stateに session_id/applicant_name/application_date が正しく設定されていること
- transport_agent_tool/expense_agent_toolがinvocation_stateを受け取り専門エージェントに引き渡すこと

### 7.2 異常系テスト
- 空入力でEX-01エラーメッセージが返り対話が継続すること
- 501文字以上の入力でEX-01エラーメッセージが返り文字数が表示されること
- 500文字の入力（境界値）で正常処理されること
- LoopLimitError（EX-08）でCF-006誘導メッセージが返りTERMINATEDに遷移すること
- 対話30回到達（GRD-005）でTERMINATEDに遷移してフローが終了すること
- _agent_instancesが正しくキャッシュされ、2回目の呼び出しで既存インスタンスが再利用されること
- transport_agent_toolとexpense_agent_toolが同一セッションで同時呼び出しされないこと

### 7.3 性能テスト
- SlidingWindowConversationManager(window_size=30)で30ターンの会話履歴が正しく管理されること
- セッションリセット後にAgentインスタンスキャッシュが新しいsession_idで切り替わること
