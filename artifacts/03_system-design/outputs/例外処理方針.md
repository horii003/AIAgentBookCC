## 1. 目的・適用範囲

### 1.1 目的

システム全体として **「例外とは何か」「どう分類し」「どう収束させるか」** の共通方針を定義する

個々のエージェント設計に **一貫した判断軸** を提供する

### 1.2 適用範囲

- マルチエージェントを含む全システム（AG-001〜AG-003、TOOL-001、TOOL-002）
- 業務処理、外部連携、UI/UX、運用を含む

### 1.3 本方針で定義しないこと

- エージェント内部の例外検知ロジック
- SDK／ライブラリ依存の例外クラス設計
- 具体的な再試行アルゴリズム

---

## 2. 例外処理の基本原則（システム共通）

- **例外は必ず収束させる（無限リトライ禁止）**: 再試行は設定回数（最大6回）以内で上限を設ける
- **失敗は状態として管理する**: 例外発生時はセッション状態ファイルにエラー状態を記録する
- **例外＝即エラー表示ではない**: 代替処理が可能な場合は代替処理を試みてからユーザーに通知する
- **ユーザーに内部失敗を見せない**: スタックトレース・システム内部情報はログにのみ記録し、ユーザーには抽象化したメッセージを提示する
- **後続処理に影響を残さない**: 例外処理後はセッション状態を確定させ、再開可能な状態に整える


---

## 3. 例外の定義と分類（システム観点）

### 3.1 例外の定義

「通常の業務フローから逸脱し、継続判断が必要な状態」

### 3.2 例外分類

| 分類ID | 分類名 | 説明 | 例 |
|-------|-------|------|---|
| EX-01 | 入力例外 | 入力不正・不足 | 申請意図テキストが空（GRD-001）、500文字超の入力（GRD-012）、移動情報の矛盾入力（GRD-002） |
| EX-02 | 判断不能 | AIが結論を出せない | 申請種別が交通費精算申請・経費精算申請のいずれとも判断できない（JD-02）、LLMの応答が不正形式 |
| EX-03 | 業務ルール違反 | 定義済ルール不一致 | 申請期限90日超過（GRD-004/JD-11）、必須情報未充足での申請書作成試行（GRD-003）、チェック未実施での申請書確定試行（GRD-005） |
| EX-04 | 外部I/F例外 | API障害・タイムアウト | LLMバックエンド（EXT-001）の呼び出し失敗（ERR-002）、ファイルストレージ（EXT-002）への書き込み失敗（ERR-003） |
| EX-05 | 実行不可 | 権限・状態不整合 | 申請書提出（ACT-EXEC-02）をエージェントが実行しようとした（GRD-011）、全チェック合格前のファイル出力試行（GRD-010） |
| EX-06 | データ不整合 | データ形式不正 | 運賃マスタJSONファイル（DATA-009/DATA-010）の破損・パース失敗、申請書テンプレートExcelファイルの破損 |
| EX-07 | システム障害 | 基盤障害 | セッション状態ファイルへの読み書き失敗、ログファイル出力失敗 |
| EX-08 | 想定外 | 未分類事象 | 上記EX-01〜EX-07に分類できない予期しない例外（ERR-004）、LoopControlHookによるReActループ上限到達 |

## 4. 例外検知レベル

### 4.1 検知レベルの考え方

| レベル | 説明 | 検知主体 |
|-------|------|---------|
| 業務レベル | 業務定義上の異常（申請期限超過・必須情報不足・申請種別判断不能等） | エージェント（AG-001〜AG-003） |
| 処理レベル | 処理失敗（ツール実行エラー・ファイル生成失敗等） | ツール（TOOL-001/TOOL-002） |
| システムレベル | 基盤・インフラ異常（LLM障害・ファイルシステム障害・EventLoopException等） | ErrorHandler |

### 4.2 検知責務

- 検知は **エージェントまたはツール**
- **判断・収束はErrorHandler** が行う

## 5. 例外発生時の基本対応フロー

### 5.1 共通対応フロー

1. 例外検知（エージェント/ツール）
2. 例外分類（ErrorHandler）
3. 継続可否判定（ErrorHandler）
4. 処理分岐（再試行／代替／停止）
5. 状態更新（セッション管理）
6. ログ・監査記録（ErrorHandler）

| EX-01 | 入力例外 | × | ○ | × | × | 再入力を促すメッセージを提示する（500文字超は文字数上限を案内する） |
| EX-02 | 判断不能 | × | ○ | △ | × | 2択選択肢を提示してユーザーに選択させる。申請ルール外のケースはCF-006へエスカレーション |

## 7. 例外ハンドリング実装

### 7.1 ErrorHandlerの責務方針

エラー伝播の責務は以下の3層で分担する。

**各モジュール（エージェント・ツール）**:
- 例外発生箇所でログを出力する（エラー内容・コンテキスト・スタックトレース）
- ErrorHandler へ処理を委譲する

**ErrorHandler**:
- 例外分類（EX-01〜EX-08）ごとに継続可否を判定し、対応方針（再試行・代替処理・エスカレーション・中断）を決定する
- ユーザー向けの日本語メッセージを生成して呼び出し元に返す
- 技術的な詳細を隠蔽する（スタックトレースはログにのみ記録し、ユーザーには非表示）

**呼び出し元**:
- ErrorHandlerから受け取ったメッセージを戻り値または表示として上位レイヤーへ伝播させる
- ツール関数は `{"success": False, "message": エラーメッセージ}` を返却する（例外を再送出しない）
- エージェント間の例外伝播は `EventLoopException`（`__cause__` で `LoopLimitError` を判定）→ `Exception` の2層構造でキャッチする

具体的なメソッド設計（メソッド名・引数・戻り値）は基本設計（ハンドラー基本設計.md）で定義する。

> ※ 基本設計書への参照: ハンドラー基本設計.md


### 7.2 エラーメッセージ設計方針

**ユーザーフレンドリー**:
- 技術的な詳細を隠蔽し、社員が理解できる日本語で記述する
- 具体的な対処方法（再入力・エスカレーション等）を提示する
- 丁寧な言葉遣い（敬体）を使用する

**情報の充実**:
- エラーの原因を業務的な言葉で簡潔に説明する
- 可能な場合は複数の対処方法を提示する
- 解消できない場合はエスカレーション先（要件上未定義）への連絡を促す

**例**:
```
# 入力例外（EX-01）の場合
「申請意図が入力されていません。どのような精算をされたいか、自由な言葉で入力してください。」

# 業務ルール違反（EX-03 申請期限超過）の場合
「申請期限（経費発生日から90日以内）を超過しているため、この申請は受け付けられません。
担当部署へご相談ください。」

# 外部I/F例外（EX-04）の場合
「申請処理中にシステムエラーが発生しました。しばらくしてから再度お試しいただくか、
担当部署へご連絡ください。」

# 想定外例外（EX-08）の場合
「申請処理中に予期しないエラーが発生しました。担当部署へご連絡ください。」
```

## 8. 例外とエスカレーションの関係

### 8.1 エスカレーション対象例外

| 例外分類 | エスカレーション要否 | 行き先 |
|---------|------------------|-------|
| EX-02 判断不能（申請ルール外） | 要 | CF-006（エスカレーション案内）。エスカレーション先は要件上未定義 |
| EX-03 業務ルール違反（修正不可不備） | 要（修正不可と判定した場合） | CF-006（エスカレーション案内） |
| EX-04 外部I/F例外（リトライ失敗） | 要 | CF-006（エスカレーション案内）。LLM障害は全機能停止のため必ずエスカレーション |
| EX-05 実行不可 | 要 | CF-006（エスカレーション案内） |
| EX-06 データ不整合（テンプレート破損） | 要 | CF-006（エスカレーション案内） |
| EX-07 システム障害（リトライ失敗） | 要 | CF-006（エスカレーション案内） |
| EX-08 想定外 | 要 | CF-006（エスカレーション案内） |

### 8.2 エスカレーション判定責務

**判定主体**: ErrorHandler

**判定基準**: 再試行・代替処理を試みてもフローを継続できないと判断した場合、またはGRD-010/GRD-011のガードレール違反を検知した場合にCF-006へのエスカレーションを決定する

## 9. セッション状態への反映方針

### 9.1 例外時の状態更新

| 状態 | 説明 | トリガー |
|-----|------|---------|
| ACTIVE | 処理中（代替処理・再入力促しで継続可能な例外の場合） | EX-01（再入力促し）・EX-02（2択提示）・EX-06（手動入力促し）発生時 |
| WAITING | 人対応待ち（HumanApprovalHookが申請書確認待ちを検知した状態） | TOOL-002による申請書生成完了後・社員確認待ち |
| TERMINATED | 異常終了（エスカレーション・申請フロー停止・LoopControlHookによるループ上限到達） | EX-03（申請期限超過）・EX-04〜EX-08（フロー継続不可）・EX-08（ループ上限到達） |

### 9.2 再開可否

**再開可能な例外**:
- EX-01（入力例外）: 再入力後に申請フローを継続する
- EX-02（判断不能）: 2択選択後に申請フローを継続する
- EX-06（データ不整合 - 運賃マスタJSONパース失敗）: 手動入力後に申請フローを継続する

**再開不可な例外**:
- EX-03（申請期限90日超過）: セッションをTERMINATEDに更新して再開不可
- EX-04（外部I/F例外 - リトライ失敗）: セッションをTERMINATEDに更新して再開不可
- EX-05（実行不可）: セッションをTERMINATEDに更新して再開不可
- EX-07（システム障害 - リトライ失敗）: セッションをTERMINATEDに更新して再開不可
- EX-08（想定外・ループ上限到達）: セッションをTERMINATEDに更新して再開不可

## 10. UX観点の扱い（概要）

**利用者への通知有無**:
- 全例外においてユーザーへの通知を行う。サイレント失敗（ユーザーへの通知なし）は禁止する
- EX-01は即時フィードバックとして再入力を促す。EX-03は申請停止を通知する

**エラーメッセージの抽象度**:
- ユーザーには業務的な言葉（申請期限・必須情報等）で記述したメッセージを提示する
- スタックトレース・エラーコード・システム内部情報はユーザーに見せない

**次の行動を必ず提示**:
- 再試行可能な場合：「再度お試しください」または具体的な再入力内容を提示する
- エスカレーションが必要な場合：CF-006のエスカレーション案内メッセージを提示する（エスカレーション先は要件上未定義）

## 11. ログ・監査方針

### 11.1 例外ログ

| 項目 | 内容 | 出力先 |
|-----|------|-------|
| error_type | エラータイプ（EX-01〜EX-08） | エラーログ（ERR-001〜ERR-004） |
| error_message | ユーザー向けメッセージ（マスキング済み） | エラーログ |
| context | エラーコンテキスト（request_id・エージェントID・発生フロー） | エラーログ |
| exc_info | スタックトレース（要約） | エラーログのみ（ユーザーには非表示） |
| timestamp | タイムスタンプ | エラーログ |


### 11.2 監査観点

**再現可能性**:
- request_id（= session_id）を全ログに付与することで、例外発生から復旧までの経緯を追跡できるようにする

**判断経路の説明可能性**:
- 業務ルール違反（EX-03）・エスカレーション判定（EX-04〜EX-08）については適用ルールID（JD-xx/BRL-xx）を監査ログ（AUD-005）に記録する

## 12. 設計上の留意点・リスク

**例外過多によるUX低下**:
- 入力例外（EX-01）が多発すると社員の利便性が低下するリスクがある
- 対策: GRD-001/GRD-002のガードレールで入力検証を行い、適切なフィードバックメッセージで再入力を促す

**例外の握り潰し**:
- ツール関数内でエラーを無視して処理を続行するリスクがある
- 対策: ツール関数は必ず `{"success": False, "message": エラーメッセージ}` を返却し、エージェントがエラー状態を認識できるようにする

**想定外例外の扱い**:
- EX-08（想定外）は設計段階では未知の例外のため、対応が不完全になるリスクがある
- 対策: 全エージェント・ツールに `except Exception` のキャッチを設け、必ずCF-006へのエスカレーションを実施する
