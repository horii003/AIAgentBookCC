## 2. 目的・役割定義

### 2.1 エージェントの目的
- 交通費精算申請に特化したスペシャリストエージェントとして、AG-001からの委任を受けて申請フロー全体を実行する
- 移動情報を一区間ずつ対話収集し、TOOL-001で運賃を自動計算して申請書ドラフト（Excel/.xlsx）を生成する
- 申請期限チェック（BRL-13）・上長承認要否判定（BRL-14）・差し戻しリスク評価（BRL-08）を実施してチェック結果をユーザーへ提示する

### 2.2 主な責務
- **移動情報の対話収集（一区間ずつ）**: CF-003で移動日・出発地・目的地・交通手段・費用を一区間ずつ収集する（BRL-11）
- **交通費自動計算**: TOOL-001（calculate_travel_expense）で移動区間の運賃を自動算出する（BRL-12）
- **駅名正規化**: システムプロンプトの指示に従い、ユーザー入力の駅名をTOOL-001呼び出し前に正規化する（BRL-15）
- **業務目的の確認**: CF-003 Step 5-6で業務目的を収集する（BRL-20）
- **Human-in-the-Loop承認取得**: 収集済み申請情報をテキストで整理・提示してOK/修正/キャンセルの3択を取得する（BRL-06）。ツール呼び出しは行わない（ドラフト提示ステップ）
- **申請書（交通費精算申請書）の生成**: HumanApprovalHook（BeforeToolCallEvent）承認後にTOOL-002（generate_travel_expense_form）でExcelドラフトを生成する（BRL-05、FR-010）
- **申請ルールチェック**: 申請期限（BRL-13）・上長承認要否（BRL-14）・差し戻しリスク（BRL-08）をチェックし結果を提示する（CF-005）
- **申請書最終提示・提出促し**: 申請書ドラフト・チェック結果を提示し、ユーザーへ提出操作を促す（BRL-09、CF-007）

### 2.3 非責務（やらないこと）
- **経費精算申請フロー**: AG-003が担当
- **申請種別の判定**: AG-001が担当
- **申請書の自動提出**: BRL-09により禁止。提出操作は利用者のみ実施可能
- **他エージェントへの委任**: AG-002はAG-001・AG-003への委任を行わない

## 3. 自律度・判断権限

### 3.1 自律度レベル

| レベル | 内容 | 適用範囲 |
|------|------|----------|
| 参照 | 運賃データ（DATA-011/DATA-012）をTOOL-001経由で参照する | 交通費計算（BRL-12） |
| 提案 | 申請書ドラフト・チェック結果・差し戻しリスク評価をユーザーへ提示する | 申請書提示（BRL-06）、差し戻しリスク警告（BRL-08） |
| 実行 | 移動情報収集・交通費計算・申請書生成（承認後）・申請ルールチェックを実行する | CF-003〜CF-007の全ステップ |
| 実行（承認必須） | 申請書Excelドラフトの生成（HumanApprovalHook（BeforeToolCallEvent）承認が必須）。承認前にTOOL-002を呼び出してはならない | TOOL-002の呼び出し（BRL-06、GRD-016） |

### 3.2 判断権限の範囲
- **自律判断可能**: 交通費計算（BRL-12）、駅名正規化（BRL-15）、経費区分自動判断は非対象（交通費専門）、申請期限チェック（BRL-13）、上長承認要否判定（BRL-14）、差し戻しリスク評価（BRL-08）
- **人間確認必要**: Human-in-the-Loop承認（ドラフト提示後・TOOL-002実行前に1回）、申請書最終確認（CF-007）
- **実行不可**: 申請書の自動提出（BRL-09、Lv0）

## 4. 処理フロー設計

### 4.1 基本フロー（概要）

```text
開始（AG-001からのLNK-001委任）
→ invocation_stateからapplicant_name・application_dateを取得
→ CF-003: 移動情報収集（一区間ずつ。TOOL-001で運賃計算）
→ CF-003: 業務目的確認（BRL-20）
→ [ドラフト提示ステップ] 収集済み申請情報をテキストとして整理・提示（ツール呼び出しなし）
→ [Human-in-the-Loop承認] OK/修正/キャンセルの選択を取得
→ [OK] HumanApprovalHook（BeforeToolCallEvent）でTOOL-002呼び出しを承認
       → [ツール呼び出しステップ] TOOL-002で申請書生成
       → CF-005: 申請期限チェック（BRL-13）→ 上長承認要否判定（BRL-14）→ 差し戻しリスク評価（BRL-08）
→ CF-007: 申請書ドラフト・チェック結果・最終提示
→ AG-001へ処理完了を返す
→ [修正] CF-003 Step 2へ戻る
→ [キャンセル] セッション終了
```

> **重要**: ドラフト提示ステップ（収集情報のテキスト整理・提示）とツール呼び出しステップ（TOOL-002実行）は明確に分離する。Human-in-the-Loop承認（HumanApprovalHook）は1回のみ発生し、承認前にTOOL-002を実行してはならない

**フローの詳細**: 詳細設計書 2.3章を参照

### 4.2 フロー制御の考え方
- **原則的な処理順序**: 逐次処理。移動情報収集 → 運賃計算 → 業務目的確認 → ドラフト提示（テキスト整理・ツール呼び出しなし）→ Human-in-the-Loop承認（1回）→ 申請書生成（TOOL-002）→ ルールチェック → 最終提示の順序を厳守する
- **ドラフト提示とツール呼び出しの分離**: 収集済み申請情報の提示はテキストで行う（ツール呼び出しなし）。TOOL-002の呼び出しはHumanApprovalHook（BeforeToolCallEvent）による承認後に1回のみ実行する
- **Human-in-the-Loop承認の発生タイミング**: 収集済み申請情報のテキスト提示後にOK/修正/キャンセルを取得する。承認は申請フロー全体で1回のみ発生する
- **処理をスキップする考え方**: TOOL-001で経路データが存在しない場合は自動計算をスキップし、ユーザーへ手動入力を促す（BRL-12例外）
- **同一処理を複数回実行**: 移動区間が複数の場合、CF-003 Step 2-4をループする（BRL-11）。Human-in-the-Loop承認で修正を選択した場合はStep 2へ戻る

### 4.3 状態管理
- **収集済み申請情報（D-007）**: セッション内で保持。applicant_name・application_date・移動区間リスト・業務目的を管理する
- **会話履歴**: SlidingWindow(20)で管理。ウィンドウ超過時はshould_truncate_results=Trueで切り詰める
- **invocation_state（受信）**: `{"applicant_name": str, "application_date": str}` を受信し、申請書生成の入力として利用する

## 6. ツール設計

### 6.0 ツール割り当て方針
- AG-002が使用するツールは、業務機能（計算・生成など）に特化したツールのみとする
- 具体的には TOOL-001（交通費計算）と TOOL-002（申請書生成）のみが対象

### 6.1 利用ツール

- **TOOL-001（calculate_travel_expense）**: CF-003 Step 3で移動区間ごとに呼び出す。出発地・目的地・交通手段を渡し、運賃（円）を受け取る。経路テーブル未登録の場合はエラーを返し手動入力を促す（BRL-12）
- **TOOL-002（generate_travel_expense_form）**: HumanApprovalHook承認OK後に呼び出す。収集済み申請情報からExcel申請書ドラフトを生成し、output/に保存してファイルパスを返す（BRL-05）

TOOL-001は区間ごとに逐次呼び出す。TOOL-002はHumanApprovalHook承認後の1回のみ呼び出す（重複生成防止）。

> 【注意】TOOL-002は `generate_travel_expense_form` 関数のみを使用する。`generate_expense_form`（AG-003用）は呼び出さない

**ツールの詳細**: 各ツール基本設計書を参照

## 7. プロンプト設計

### 7.1 システムプロンプトの目的
- 交通費精算申請専門エージェントとして移動情報を一区間ずつ収集する役割と手順を定義する
- 駅名正規化（BRL-15）をTOOL-001呼び出し前に実施する指示を含める
- 申請期限（BRL-13）・上長承認要否（BRL-14）のチェック手順と、エージェントが実行日（application_date）を基準にどの時点で期限超過かを判断するための基準日を埋め込む（動的生成）
- BRL-09（申請書自動提出禁止）の制約を明示する

### 7.2 プロンプト構成要素
- **役割定義**: 交通費精算申請専門エージェントとして申請フロー全体を担う
- **処理フロー**: 移動情報収集（一区間ずつ） → TOOL-001呼び出し → 全区間完了後に業務目的確認 → 収集済み情報のテキスト整理・提示（ツール呼び出しなし） → Human-in-the-Loop承認（1回） → TOOL-002呼び出し（承認後のみ） → ルールチェック → 最終提示
- **判断プロトコル**: 申請期限チェック（BRL-13：移動日から3ヶ月以内）、上長承認要否（BRL-14：合計10,000円超）、差し戻しリスク評価（BRL-08）の判断手順
- **主な責任**: 交通費精算申請フロー全体。申請書自動提出禁止（BRL-09）。ドラフト提示前のTOOL-002呼び出し禁止
- **動的情報**: 申請日（application_date）= システム日付（実行時に埋め込む）、申請期限の基準日（application_date - 3ヶ月）
- **エラー発生時の振る舞い**: ツールエラー・システム障害等のシステム系エラーが発生した場合は、エラー内容を要約して呼び出し元エージェント（AG-001）に返すこと

**プロンプト全文**: 詳細設計書 2.3.1章を参照

### 7.3 システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 動的生成関数 |
| 動的生成の理由 | CF-005のルールチェックで申請期限チェック（BRL-13）を実行するため。移動日からの経過期間（3ヶ月以内）を判断するために実行時の申請日（application_date）をプロンプトに埋め込む必要がある |
| 動的に埋め込む情報 | application_date（申請日。invocation_stateから取得）、申請期限基準日（application_date - 3ヶ月。BRL-13の判断基準） |

> **判断根拠**: CF-005の会話ステップに申請期限チェック（BRL-13）が含まれており、移動日とapplication_dateを比較して期限判断を行うため動的生成関数を選択した

### 7.4 主要な設計意図
- **駅名正規化の一元化**: 正規化はシステムプロンプトで指示し、TOOL-001に渡す前に完了させる（BRL-15）
- **申請期限の動的埋め込み**: 申請期限チェック（BRL-13）の基準日をプロンプトに埋め込み、LLMが実行時の日付を正確に判断できるようにする
- **Human-in-the-Loop承認の位置付け**: TOOL-002呼び出し前のHumanApprovalHook（BeforeToolCallEvent）ブロック（GRD-016）との連携を明示し、承認なしのExcel生成を防止する。ドラフト提示（テキスト）とツール実行を分離し、承認は1回のみ発生させる
- **エラー時の呼び出し元への通知**: システム系エラー発生時はエラー内容を要約してAG-001に返す。AG-001がユーザーへエスカレーション案内を提示する

**プロンプト全文**: 詳細設計書 2.3.1章を参照

## 10. 入力データ設計

### 10.1 初期化時の入力
※ Agent as Toolsとして呼び出されるため、ToolContextのinvocation_stateから受け取る

### 10.2 対話ループでの入力
- **移動情報（一区間）**: 移動日・出発地・目的地・交通手段・費用をユーザーが自由文で入力する
- **業務目的**: ユーザーが自由文で入力する（BRL-20）
- **Human-in-the-Loop選択**: OK/修正/キャンセルから選択する（CF-003 Step 8）
- **特殊コマンド**:
  - TOOL-001経路テーブル未登録時の手動費用入力
  - 申請期限超過時のCF-008遷移確認

### 10.3 invocation_stateの入力
受付エージェント（AG-001）から渡される情報:
```python
{
    "applicant_name": str,   # 申請者名（min_length=1）
    "application_date": str  # 申請日（YYYY-MM-DD形式。システム日付）
}
```

> ※ session_idはファクトリ関数（_get_travel_agent）で直接受け取り、FileSessionManagerの初期化に使用する。エージェントへのinvocation_stateには含めない
> ※ ファクトリ関数内で applicant_name（min_length=1）・application_date（YYYY-MM-DD形式）を個別にバリデーションする（専用Pydanticモデルは使用しない）

## 11. 出力データ設計

### 11.1 出力形式
- **出力先**: CLI（標準出力）
- **出力内容**: 移動情報確認・運賃計算結果・申請書ドラフト（Excelファイルパス）・申請チェック結果・最終提示

### 11.2 出力メッセージ
- 運賃計算結果提示: `"[出発地]→[目的地]（[交通手段]）の交通費は[運賃]円です（[計算根拠]）。"`
- Human-in-the-Loop確認: `"収集した申請情報を確認してください。[申請情報サマリー]\nOK・修正・キャンセルのいずれかを入力してください。"`
- 申請期限超過通知（CF-008）: `"申し訳ありません。経費発生日から3ヶ月を超えているため、この申請は受け付けられません。総務部門にご相談ください。"`
- 上長承認要警告（CF-009）: `"交通費合計が10,000円を超えています。事前に上長の承認を得てください。上長承認済みですか？"`

### 11.3 生成ファイル

- **保存先**: `output/`

## 13. Agent as Tools設計

### 13.1 ツールとしての公開
- **ツール名**: `travel_application_agent_tool`（仮称。詳細設計で確定）
- **デコレータ**: `@tool(context=True)`
- **入力**: `query (str)`, `tool_context (ToolContext)`
- **出力**: str（申請フロー完了メッセージまたはエラーメッセージ）

### 13.2 ToolContextの使用
- `tool_context.invocation_state` から `session_id`・`applicant_name`・`application_date` を取得する
- `session_id` はファクトリ関数（`_get_travel_agent(session_id)`）内でFileSessionManagerの初期化に直接使用する
- `applicant_name`・`application_date` を専門エージェントへのinvocation_stateとして設定する
- ファクトリ関数内でapplicant_name（min_length=1）・application_date（YYYY-MM-DD形式）を個別バリデーションする

## 14. テスト観点

### 14.1 テスト観点の概要
- **機能テスト**: 移動情報が一区間ずつ正確に収集されること。TOOL-001で正確な運賃が算出されること。収集済み申請情報がテキストとして正確に整理・提示されること（ツール呼び出しなし）。Human-in-the-Loop承認（BeforeToolCallEvent）後にTOOL-002でExcelドラフトが生成されること。申請期限・上長承認要否が正確にチェックされること
- **異常系テスト**: TOOL-001経路テーブル未登録時の手動入力促しが正常に機能すること。申請期限超過（BRL-13）でCF-008へ遷移すること。上長承認要（BRL-14）でCF-009へ遷移すること。Human-in-the-Loop承認なしでTOOL-002が呼び出されないこと。ドラフト提示ステップでTOOL-002が呼び出されないこと（承認前の実行禁止）。システム系エラー（EX-06/EX-07）発生時にAG-001へ要約メッセージが返却されること
- **性能テスト**: LoopControlHookが10回上限で正しく動作すること。30ターン超でセッション終了すること（FR-017）

**詳細なテストケース**: 詳細設計書 7章を参照

## 16. 制約事項・前提条件

### 16.1 技術的制約
- 申請書の自動提出は禁止（BRL-09）。提出操作は利用者のみ実施可能
- TOOL-002（generate_travel_expense_form）はHumanApprovalHook（BeforeToolCallEvent）による承認後にのみ呼び出し可能（GRD-016）
- ドラフト提示ステップ（テキスト提示）とツール呼び出しステップ（TOOL-002実行）は明確に分離する。Human-in-the-Loop承認は1回のみ発生する
- AG-001・AG-003への委任は禁止（循環呼び出し禁止）
- invocation_stateは辞書リテラルで受け取る。専用Pydanticモデルは使用しない（バリデーション方針 3.2節）

### 16.2 業務的制約
- 申請期限チェック（BRL-13：経費発生日から3ヶ月以内）はAG-002の責務
- 上長承認要否閾値（BRL-14：10,000円超）はPydanticバリデーションエラーではなく警告フラグとして扱う
- 差し戻しリスク閾値の詳細は要件上未定義

### 16.3 前提条件
- AG-001からinvocation_stateでapplicant_name・application_date・session_idが渡されること
- DATA-011（train_fares.json）・DATA-012（fixed_fares.json）がアプリ起動時に静的読み込み済みであること
- DATA-002（交通費申請書テンプレート）が`data/交通費申請書_template.xlsx`として存在すること
