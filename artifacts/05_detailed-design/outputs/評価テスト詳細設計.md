# Part 1: eval_tool_selection.py 詳細設計

## 1. 概要

AG-001（申請受付窓口エージェント）が申請内容テキストに対して正しい専門エージェント（AG-002 または AG-003）を選択しているかを自動評価するスクリプト（MET-010）。`ToolSelectionAccuracyEvaluator` による LLM-as-Judge で各ケースの trajectory を評価し、スコア（1.0/0.0）と reasoning を JSON レポートに出力する。

### 評価方式

| 項目 | 内容 |
|------|------|
| 評価器クラス名 | `ToolSelectionAccuracyEvaluator` |
| 評価レベル | TOOL_LEVEL |
| スコア形式 | 二値スコア（1.0 = 正解 / 0.0 = 不正解） |
| 実行方式 | シングルターン（`agent(case.input, ...)`） |
| 実行コマンド | `python evals/eval_tool_selection.py` |

## 6. タスク関数（run_eval_task）

### 6.1 シグネチャ

```python
def run_eval_task(case: Case) -> dict:
```

| 戻り値キー | 型 | 説明 |
|-----------|------|------|
| `output` | `str` | AG-001 の最終応答文字列 |
| `trajectory` | `Session` | ToolSelectionAccuracyEvaluator が参照する実行軌跡（ツール呼び出し履歴を含む） |

### 6.2 処理フロー

```
run_eval_task(case)
  │
  ├── [1] session_id = case.session_id
  │
  ├── [2] logger.info("=== ケース '%s' 開始 ===", case.name)
  │
  ├── [3] memory_exporter.clear()   ← 前ケーススパンを消去（必須）
  │
  ├── [4] agent = create_reception_agent(session_id)
  │
  ├── [5] state = create_invocation_state(session_id)
  │
  ├── [6] response = agent(case.input, invocation_state=state.model_dump())
  │         ← シングルターン実行
  │
  ├── [7] finished_spans = memory_exporter.get_finished_spans()
  │
  ├── [8] session = StrandsInMemorySessionMapper()
  │               .map_to_session(finished_spans, session_id=session_id)
  │
  └── [9] return {"output": str(response), "trajectory": session}
```

### 6.3 実行方式

シングルターン（`agent(case.input, ...)`）。MET-010 はツール選択の一発判定であり、最初の応答でどの専門エージェントツールを呼び出したかのみを評価するためマルチターンは不要。

### 6.4 ログ出力

| タイミング | レベル | メッセージ形式 |
|-----------|-------|---------------|
| ケース開始時 | INFO | `"=== ケース '%s' 開始 ===" % case.name` |

## 7. メイン処理（main 関数）

### 7.1 処理フロー

```
main()
  │
  ├── [1] logger.info("=== eval_tool_selection 開始 ===")
  │
  ├── [2] evaluator = ToolSelectionAccuracyEvaluator(model=get_model())
  │
  ├── [3] experiment = Experiment(cases=EVAL_CASES, evaluators=[evaluator])
  │
  ├── [4] reports = experiment.run_evaluations(run_eval_task)
  │
  └── [5] for report in reports:
              output_path = "evals/logs/eval_tool_selection_report.json"
              with open(output_path, "w", encoding="utf-8") as f:
                  json.dump(report.to_dict(), f, ensure_ascii=False, indent=2)
              logger.info("レポート保存: %s", output_path)
              print(report)
```

### 7.2 レポート保存設定

| 設定 | 値 | 理由 |
|------|---|------|
| `ensure_ascii` | `False` | 日本語文字列（goal・reasoning 等）をエスケープせずに保存する |
| `indent` | `2` | 人間が読みやすい整形済み JSON にする |
| エンコーディング | `UTF-8` | 日本語を含むファイルの標準エンコーディング |

## 9. エラーハンドリング・制約事項

### 9.1 エラーハンドリング

明示的な try/except は持たない設計。AWS Bedrock の接続エラー・タイムアウトは Strands SDK の再試行機構に委譲する。評価スクリプト全体として未捕捉例外が発生した場合は Python のデフォルト動作（スタックトレース出力 + 終了）に従う。

### 9.2 制約事項

| 制約 | 内容 |
|------|------|
| 並列実行禁止 | `memory_exporter` はプロセス内シングルトンのため、ケースは逐次実行する。並列実行すると前ケースのスパンが混入する |
| レポート上書き | `eval_tool_selection_report.json` は実行ごとに上書きされる。履歴保持が必要な場合はファイル名にタイムスタンプを付与すること |
| 評価再現性 | LLM-as-Judge の判断には非決定性があるため、同一入力でも評価結果が変わる場合がある |
| コスト | シングルターン実行のため、エージェント LLM 呼び出し（AG-001）+ 評価器 LLM 呼び出し（Judge）がケースごとに発生する |

| 項目 | 内容 |
|------|------|
| 評価器クラス名 | `GoalSuccessRateEvaluator` |
| 評価レベル | SESSION_LEVEL |
| スコア形式 | 二値スコア（1.0 = ゴール達成 / 0.0 = ゴール未達成） |
| 実行方式 | マルチターン（`run_actor_conversation`） |
| 実行コマンド | `python evals/eval_goal_success_rate.py` |

```
  │
  ├── [3] memory_exporter.clear()   ← 前ケーススパンを消去（必須）
  │
  ├── [4] agent = create_reception_agent(session_id)
  │
  ├── [5] state = create_invocation_state(session_id)
  │
  ├── [6] turns = run_actor_conversation(agent, case, state.model_dump())
  │         ← マルチターン実行（ActorSimulator が社員役を担う）
  │         ← turns は [{"user_input": ..., "agent_response": ...}, ...] のリスト
  │
  ├── [7] final_response = turns[-1]["agent_response"] if turns else ""
  │
  ├── [8] finished_spans = memory_exporter.get_finished_spans()
  │
  ├── [9] session = StrandsInMemorySessionMapper()
  │               .map_to_session(finished_spans, session_id=session_id)
  │
  └── [10] return {"output": final_response, "trajectory": session}
```

### 6.3 実行方式

マルチターン（`run_actor_conversation`）。MET-011 はセッション全体のゴール達成を評価するため、`ActorSimulator` が社員役として AG-001 と複数ターンの対話を行い申請フローを完結させる必要がある。`ActorSimulator` は `case.metadata["goal"]` を参照してゴール達成を判定し、達成時に会話を終了する。

## 9. エラーハンドリング・制約事項

### 9.1 エラーハンドリング

明示的な try/except は持たない設計。AWS Bedrock の接続エラー・タイムアウトは Strands SDK の再試行機構に委譲する。`ActorSimulator` がゴール未達成のままターン上限に達した場合は、最終ターンの応答を `output` として返し、`GoalSuccessRateEvaluator` が 0.0（ゴール未達成）と判定する。

### 9.2 制約事項

| 制約 | 内容 |
|------|------|
| 並列実行禁止 | `memory_exporter` はプロセス内シングルトンのため、ケースは逐次実行する。並列実行すると前ケースのスパンが混入する |
| レポート上書き | `eval_goal_success_rate_report.json` は実行ごとに上書きされる。履歴保持が必要な場合はファイル名にタイムスタンプを付与すること |
| 評価再現性 | ActorSimulator と LLM-as-Judge の両方に非決定性があるため、同一入力でも評価結果が変わる場合がある |
| コスト | マルチターン実行のため、AG-001/AG-002/AG-003 の LLM 呼び出し + ActorSimulator の LLM 呼び出し + 評価器 LLM 呼び出しが複数ターン分発生する。eval_tool_selection.py より大幅にコストが高い |
| ファイル生成副作用 | TC-001/TC-002 では申請書 Excel ファイル（output/ 配下）が実際に生成される。評価実行後にファイルが残存する |
