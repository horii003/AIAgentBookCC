## 1. 評価システム概要

### 1.1 目的

社内申請AIアシスタントシステム（IAAS）において、受付エージェント（AG-001）が申請内容を正しく分類し適切な専門エージェントへ振り分けていること（EV-001）、および対話を通じて申請書（下書き）の生成またはエスカレーション案内が達成されること（EV-002）を継続的に自動検証する。

### 1.2 評価方式：LLM-as-Judge

全評価スクリプトで共通して **LLM-as-Judge** を採用する。評価専用の LLM が、エージェントの実行軌跡（trajectory）を参照して品質を判定する。

```
エージェント実行
  │
  ▼
trajectory（実行軌跡）
  │  ツール呼び出し履歴・LLM 呼び出し履歴・会話ターンを含む
  ▼
評価用 LLM（Judge）
  │  trajectory と評価基準（expected_tool / goal）を照合
  ▼
スコア（Yes=1.0 / No=0.0）＋ reasoning（判定根拠テキスト）
```

人手評価と比較した LLM-as-Judge の特徴は以下のとおり。

| 観点 | 人手評価 | LLM-as-Judge |
|---|---|---|
| スケール | ケース数に比例してコスト増 | ケース数に応じてスケール可能 |
| 再現性 | 評価者によって基準がばらつく | 評価基準をプロンプトで固定できる |
| スピード | 時間がかかる | 自動実行・即時結果 |
| コスト | 人件費 | LLM API 呼び出しコスト |
| 注意点 | — | 評価用 LLM の判断にも誤りが生じうる |

### 1.3 スコア形式

全評価スクリプトで共通して **二値スコア** を採用する。

| スコア | 意味 |
|---|---|
| `1.0`（Yes） | 評価基準を満たしている |
| `0.0`（No） | 評価基準を満たしていない |

## 3. 評価フレームワーク構成

### 3.1 使用フレームワーク

評価フレームワークとして **strands_evals**（Strands Agents 公式評価ライブラリ）を使用する。

### 3.2 主要コンポーネントと役割

```
strands_evals
  ├── Case            テストケースの定義（入力・メタデータ・session_id）
  ├── Experiment      ケース群の実行管理・評価器の呼び出し
  ├── Evaluator       LLM-as-Judge による品質判定（種類は各スクリプトで選択）
  └── mappers
        └── StrandsInMemorySessionMapper
                      OpenTelemetry スパン → Session 変換
```

### 3.3 Case オブジェクト

全評価スクリプトで共通して使用するテストケース定義オブジェクト。

```python
Case(
    name="<ケース識別子>",          # 評価レポート上の識別名
    input="<エージェントへの入力>",  # 最初のターンで送信するユーザー発話
    metadata={                       # 評価器・人間が参照する補足情報
        "task_description": "...",   # このケースの目的（人間向け説明）
        # 評価器固有のキー（各スクリプト設計書を参照）
    },
)
```

`session_id` は `strands_evals` フレームワークが Case ごとに自動生成する。

### 3.4 Experiment の実行モデル

```
Experiment.run_evaluations(run_eval_task)
  │
  ├── ケース 1: run_eval_task(case_1) を実行 → {"output": ..., "trajectory": ...}
  │     └── evaluator で trajectory を判定 → score + reasoning
  │
  ├── ケース 2: run_eval_task(case_2) を実行 → {"output": ..., "trajectory": ...}
  │     └── evaluator で trajectory を判定 → score + reasoning
  │
  └── ...（EVAL_CASES を逐次実行）
        └── reports[] を返却
```

ケースは **逐次実行**される。並列実行は `memory_exporter` の共有によるスパン混入リスクがあるため行わない。

### 3.5 run_eval_task の戻り値仕様

全評価スクリプトの `run_eval_task` は以下の辞書を返す。この形式は `Experiment` フレームワークとの契約である。

| キー | 型 | 説明 |
|---|---|---|
| `output` | `str` | エージェントの最終応答文字列 |
| `trajectory` | `Session` | Evaluator が参照する実行軌跡（全ターン・全ツール呼び出しを含む） |

## 4. テレメトリ設計

評価における trajectory（実行軌跡）の収集は、OpenTelemetry ベースのインメモリ計装で実現する。

### 4.1 テレメトリフロー

```
エージェント実行（agent() / run_actor_conversation()）
  │
  │  Strands SDK が自動的に OpenTelemetry スパンを生成
  │    - LLM 呼び出しスパン
  │    - ツール呼び出しスパン（サブエージェント呼び出しを含む）
  │    - 会話ターンスパン
  ▼
memory_exporter（InMemorySpanExporter）
  │  プロセス内メモリにスパンを蓄積
  ▼
memory_exporter.get_finished_spans()
  │  完了済みスパンを全件取得
  ▼
StrandsInMemorySessionMapper.map_to_session(spans, session_id)
  │  スパンを Evaluator が解釈できる Session 形式に変換
  ▼
Session オブジェクト（trajectory）
  │  Evaluator（LLM-as-Judge）の入力として使用
  ▼
評価結果（score + reasoning）
```

### 4.2 memory_exporter の性質と注意事項

`memory_exporter` はプロセス内でシングルトンとして共有されるインメモリエクスポーターである。複数のケースを逐次実行する際、前のケースで生成されたスパンがそのまま蓄積されているため、各ケースの実行開始前に必ず `memory_exporter.clear()` を呼び出す。

```python
# ★ 必須：前ケースのスパンを消去してからエージェントを実行する
memory_exporter.clear()
agent = create_reception_agent(session_id)
```

`clear()` を省略した場合の影響:

```
ケース 1 実行 → スパン A, B, C が蓄積
ケース 2 実行 → スパン D, E が追加（clear しないと A, B, C, D, E が混在）
             → Session にケース 1 のスパンが混入
             → ケース 2 の評価結果が不正になる
```

### 4.3 session_id によるスパンの紐付け

`create_reception_agent(session_id)` は `session_id` を `trace_attributes` に付与してエージェントを生成する。これにより、`StrandsInMemorySessionMapper` がスパンを正しいセッションに紐付けることができる。

## 5. 共通モジュール設計（helpers）

両評価スクリプトが依存する `helpers` モジュールのインターフェース定義を以下に示す。実装は `helpers.py` に集約する。

### 5.1 提供する関数・オブジェクト一覧

| 名前 | 種別 | シグネチャ | 説明 |
|---|---|---|---|
| `patch_human_approval_hook` | 関数 | `() → None` | `HumanApprovalHook` をテスト用モックに差し替える |
| `create_reception_agent` | 関数 | `(session_id: str) → Agent` | 受付エージェントのインスタンスを生成して返す |
| `run_actor_conversation` | 関数 | `(agent, case, invocation_state: dict) → list[dict]` | `ActorSimulator` によるマルチターン会話を実行し、ターンリストを返す |
| `memory_exporter` | オブジェクト | `InMemorySpanExporter` | OpenTelemetry スパンのインメモリ収集器（シングルトン） |
| `get_model` | 関数 | `() → Model` | 評価用 LLM モデルのインスタンスを返す |
| `create_invocation_state` | 関数 | `(session_id: str) → InvocationState` | エージェント呼び出しに必要な状態オブジェクトを生成して返す |

### 5.2 各関数の責務

**`patch_human_approval_hook()`**
評価実行中にエージェントが `HumanApprovalHook` に到達した場合、人間の承認なしで自動的に処理を継続させるモックに差し替える。`load_dotenv()` の直後、エージェント生成より前に呼び出す必要がある。自律度・権限定義（ABAC-001）で定義された申請書生成前の社員確認ステップを評価時にバイパスするために必要である。

**`create_reception_agent(session_id)`**
`session_id` を `trace_attributes` に含めた受付エージェント（AG-001）を生成する。テレメトリスパンへの `session_id` 付与が目的であり、評価ケースごとに独立したエージェントインスタンスを返す。

**`run_actor_conversation(agent, case, invocation_state)`**
`ActorSimulator` がユーザー役を担い、エージェントとの動的マルチターン会話を実行する。`eval_goal_success_rate.py` のみが使用する。戻り値は以下の形式のリスト。

```python
[
    {"user_input": "<ユーザー発話>", "agent_response": "<エージェント応答>"},
    ...
]
```

**`memory_exporter`**
プロセス内でシングルトンとして保持する `InMemorySpanExporter` インスタンス。両評価スクリプトで同一インスタンスを参照する。`clear()` メソッドで蓄積スパンを全件消去できる。

**`get_model()`**
LLM-as-Judge として使用する評価用モデルを返す。使用するモデルは環境変数または `.env` で設定する。

**`create_invocation_state(session_id)`**
`session_id` を含む `InvocationState` オブジェクトを生成する。エージェント間連携定義（LNK-001・LNK-002）で定義された invocation_state 経由のデータ伝播（申請者名・申請日）を評価環境で再現するために必要である。`agent()` 呼び出し時に `invocation_state=state.model_dump()` の形で渡す。

## 6. 共通初期設定処理

両評価スクリプトのモジュールレベルで、以下の処理を同一の順序で実行する。

### 6.1 処理順序

```
[1] 標準入出力の UTF-8 設定
      ↓
[2] sys.path へのプロジェクトルート追加
      ↓
[3] load_dotenv()
      ↓
[4] patch_human_approval_hook()   ← load_dotenv() の直後に実行すること
      ↓
[5] ログ設定
      ↓
[6] warnings.filterwarnings("ignore")
      ↓
[7] SDK ログの抑制
```

### 6.2 各処理の目的

| 処理 | 目的 |
|---|---|
| UTF-8 設定 | Windows 環境での日本語出力の文字化けを防ぐ。`reconfigure` が存在しない環境（リダイレクト時など）はスキップする |
| sys.path 設定 | `evals/` サブディレクトリから `helpers` モジュールを解決するため、プロジェクトルートを `sys.path` 先頭に追加する |
| load_dotenv() | AWS 認証情報・モデル設定などを `.env` ファイルから環境変数として読み込む |
| patch_human_approval_hook() | エージェント実行中の人間承認待ちを自動化し、評価がブロックされないようにする |
| ログ設定 | 評価の進行状況をコンソールとファイルの両方に記録する |
| warnings 抑制 | サードパーティライブラリからの非本質的な警告を抑制し、評価結果の視認性を高める |
| SDK ログ抑制 | Strands SDK の詳細ログが評価出力に混入しないよう抑制する |

## 7. 共通ログ設計

### 7.1 ログ設定の共通仕様

| 設定項目 | 値 |
|---|---|
| ルートログレベル | INFO |
| フォーマット | `%(asctime)s [%(levelname)s] %(name)s - %(message)s` |
| 出力先 | 標準出力（コンソール）＋ファイル（`evals/logs/` 配下） |
| ファイルエンコーディング | UTF-8 |

### 7.2 SDK ログの抑制設定

| ロガー名 | 設定レベル | 理由 |
|---|---|---|
| `strands` | WARNING | SDK 全体の INFO ログを抑制する |
| `strands.event_loop.event_loop` | CRITICAL | イベントループ内の冗長ログを完全に抑制する |

### 7.3 ログファイルの命名規則

```
evals/logs/<スクリプト名>.log
evals/logs/<スクリプト名>_report.json
```

`evals/logs/` ディレクトリはスクリプト実行時に `os.makedirs(..., exist_ok=True)` で自動生成する。

## 8. run_eval_task の共通構造

両評価スクリプトの `run_eval_task` 関数は以下の共通構造を持つ。スクリプト固有の処理はステップ [6] のエージェント実行方式のみ異なる。

```
run_eval_task(case: Case) → dict
  │
  ├── [1] session_id = case.session_id
  │
  ├── [2] logger.info("=== ケース '%s' 開始 ===", ...)
  │
  ├── [3] memory_exporter.clear()          ← 共通：前ケーススパンの消去
  │
  ├── [4] agent = create_reception_agent(session_id)
  │
  ├── [5] state = create_invocation_state(session_id)
  │
  ├── [6] ★ エージェント実行（スクリプトごとに異なる）
  │         eval_tool_selection    : agent(case.input, ...)        シングルターン
  │         eval_goal_success_rate : run_actor_conversation(...)   マルチターン
  │
  ├── [7] finished_spans = memory_exporter.get_finished_spans()
  │
  ├── [8] session = StrandsInMemorySessionMapper()
  │               .map_to_session(finished_spans, session_id=session_id)
  │
  └── [9] return {"output": <最終応答>, "trajectory": session}
```

### 9.2 出力ファイル一覧

| ファイル | 生成タイミング | 上書き |
|---|---|---|
| `evals/logs/eval_tool_selection.log` | `eval_tool_selection.py` 実行時 | 追記（append） |

### 9.3 レポート JSON の共通仕様

```python
json.dump(report.to_dict(), f, ensure_ascii=False, indent=2)
```

| 設定 | 値 | 理由 |
|---|---|---|
| `ensure_ascii=False` | True | 日本語文字列（goal・reasoning 等）をエスケープせずに保存する |
| `indent=2` | 2スペース | 人間が読みやすい整形済み JSON にする |
| エンコーディング | UTF-8 | 日本語を含むファイルの標準エンコーディング |

## 11. 評価スクリプト一覧と位置づけ

### 11.1 スクリプト比較

| 項目 | `eval_tool_selection.py` | `eval_goal_success_rate.py` |
|---|---|---|
| 評価器 | `ToolSelectionAccuracyEvaluator` | `GoalSuccessRateEvaluator` |
| 評価レベル | TOOL_LEVEL | SESSION_LEVEL |
| 実行方式 | シングルターン | マルチターン（ActorSimulator） |
| 評価の着眼点 | 各ツール呼び出しの妥当性 | セッション全体でのゴール達成 |
| metadata の判定キー | `expected_tool` | `goal` |
| LLM 呼び出し箇所 | 評価器のみ | ActorSimulator ＋ エージェント ＋ 評価器 |

### 11.2 評価の補完関係

2 つのスクリプトは互いに補完する関係にある。

```
ツール選択評価（TOOL_LEVEL）
  └── 「正しいサブエージェントを呼んでいるか」を検証
       → ルーティングのバグを早期に検出できる
       → ただし、ルーティングが正しくても成果物が生成されない場合は検出できない

ゴール達成評価（SESSION_LEVEL）
  └── 「最終的に申請書が作成されたか」を検証
       → エンドツーエンドの品質を確認できる
       → ただし、失敗時にどのステップで問題が起きたかは判別しにくい
```

両スクリプトを組み合わせることで、ルーティング層の精度とエンドツーエンドの品質を両面から評価できる。
