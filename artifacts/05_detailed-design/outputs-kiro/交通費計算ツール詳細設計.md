## 1. 概要

### 1.1 コンポーネントの目的

出発地・目的地・交通手段・移動日を入力として、経路テーブル（route_fares.json）または固定運賃マスタ（fixed_fares.json）を参照し、交通費（円）と計算根拠を返却する。AG-002（交通費精算申請エージェント）が移動区間ごとの交通費を自動算出するために使用する。

### 1.2 主要な責務

- 電車の場合は経路テーブル（`./template/train_routes.json`）を検索して運賃を取得する
- バス・タクシー・飛行機の場合は固定運賃マスタ（`./template/fixed_fares.json`）から運賃を取得する
- 参照したデータソース名を計算根拠として出力に含める
- Pydanticモデル（TransportCalculatorInput）で入力値を検証する
- 運賃データはアプリ起動時にモジュールレベル変数にキャッシュし、実行時の再読み込みを行わない
- 申請者名・申請日は LLM がツールパラメータとして渡す値ではなく、ツール関数内部で invocation_state から取得する

## 3. calculate_transport_fareツール設計

### 3.1 基本情報

#### ツール名（関数名）
`calculate_transport_fare`

#### 説明（docstring）
```
出発地・目的地・交通手段・移動日から交通費（円）を計算する。

電車の場合は経路テーブル（route_fares.json）を検索し、
バス・タクシー・飛行機の場合は固定運賃マスタ（fixed_fares.json）を参照する。
計算根拠（参照したデータソース名）を結果に含めて返却する。

Args:
    departure (str): 正規化済みの出発駅名または地名（「駅」接尾語除去済み）。
    destination (str): 正規化済みの到着駅名または地名（「駅」接尾語除去済み）。
    transport_type (str): 交通手段。以下の値を受け付ける（別表記は自動正規化）:
        - "電車"（"train"、"鉄道"、"JR"、"地下鉄" も正規化される）
        - "バス"（"bus" も正規化される）
        - "タクシー"（"taxi"、"cab" も正規化される）
        - "飛行機"（"airplane"、"plane"、"flight"、"航空" も正規化される）
    travel_date (str): 移動日。YYYY-MM-DD形式の文字列。申請期限チェック（経費発生日から3ヶ月以内）に使用する。

Returns:
    dict: 以下のキーを持つ辞書:
        - fare (int): 交通費（円）
        - data_source (str): 計算根拠（参照したデータソース名）

Raises:
    ToolExecutionError: 経路テーブル・固定運賃マスタに該当データが存在しない場合（EX-06）
    ToolExecutionError: 入力バリデーションエラーの場合（EX-01）
    ToolExecutionError: 運賃データJSONファイル読み込み失敗の場合（EX-07）
```

#### 3.2.3 例外

| 例外 | 発生条件 | 戻り値 |

### 3.3 ビジネスロジック

#### 3.3.1 処理フロー

```
開始
  ↓
TransportCalculatorInputでバリデーション実行
  - departure: normalize_station_name適用
  - destination: normalize_station_name適用
  - transport_type: normalize_transport_type適用
  - travel_date: validate_date適用
  ↓
バリデーション失敗？
  → YES: {"success": False, "message": エラーメッセージ} を返す
  ↓
transport_typeを判定する
  ↓
transport_type == "電車"？
  → YES: 経路テーブル検索処理へ
  → NO:  固定運賃マスタ参照処理へ
  ↓
【経路テーブル検索処理】
  モジュールレベルキャッシュ（_route_fares）から検索する
  キー: "{departure}_{destination}"
  該当経路が存在しない？
    → YES: {"success": False, "message": エラーメッセージ} を返す（ValueError相当）
  fare = _route_fares["{departure}_{destination}"]
  data_source = "train_routes.json"
  ↓
【固定運賃マスタ参照処理】
  モジュールレベルキャッシュ（_fixed_fares）から参照する
  キー: transport_type（"バス"・"タクシー"・"飛行機"）
  該当交通手段が存在しない？
    → YES: {"success": False, "message": エラーメッセージ} を返す（ValueError相当）

  data_source = "fixed_fares.json"
  ↓
INFOログを出力する
  ↓
{"fare": fare, "data_source": data_source}を返す
```

#### 3.3.2 起動時キャッシュ機構

**キャッシュ機構**:

**用途**: 運賃データ（train_routes.json・fixed_fares.json）をアプリ起動時に一度だけ読み込み、モジュールレベル変数にキャッシュする。実行時の再読み込みを行わないことで、ツール呼び出しごとのI/Oコストを排除する。

**実装**:
```python
# モジュールレベルキャッシュ変数
_route_fares: dict[str, int] = {}
_fixed_fares: dict[str, int] = {}

ROUTE_FARES_PATH = "./template/train_routes.json"
FIXED_FARES_PATH = "./template/fixed_fares.json"

def _load_fare_data() -> tuple[bool, str]:
    """アプリ起動時に運賃データをキャッシュに読み込む"""
    global _route_fares, _fixed_fares
    # ファイル存在チェック（os.path.exists()による事前チェック）
    if not os.path.exists(ROUTE_FARES_PATH):
        logger.warning("運賃データファイルが存在しません path=%s", ROUTE_FARES_PATH)
        return (False, f"運賃データファイルが見つかりません: {ROUTE_FARES_PATH}")
    if not os.path.exists(FIXED_FARES_PATH):
        logger.warning("固定運賃データファイルが存在しません path=%s", FIXED_FARES_PATH)
        return (False, f"固定運賃データファイルが見つかりません: {FIXED_FARES_PATH}")
    try:
        with open(ROUTE_FARES_PATH, encoding="utf-8") as f:
            _route_fares = json.load(f)
        with open(FIXED_FARES_PATH, encoding="utf-8") as f:
            _fixed_fares = json.load(f)
        return (True, "")
    except Exception as e:
        logger.warning("運賃データの読み込みに失敗しました error=%s", str(e))
        return (False, "運賃データの読み込みに失敗しました。管理部門にお問い合わせください。")

# モジュールインポート時に実行
_load_result = _load_fare_data()
```

#### 3.3.3 データの検証方法

**Pydanticモデル**: `TransportCalculatorInput`

**検証内容**:
- `departure`: normalize_station_nameバリデーターで「駅」「Station」等の接尾語を除去し、空文字列チェックを行う
- `destination`: normalize_station_nameバリデーターで「駅」「Station」等の接尾語を除去し、空文字列チェックを行う
- `transport_type`: normalize_transport_typeバリデーターで別表記を正規化し、Literal["電車", "バス", "タクシー", "飛行機"]に含まれない場合はValueErrorを発生させる

### 3.4 エラーハンドリング

#### 3.4.1 エラー処理フロー

```
エラー発生
  ↓
エラー種別の判定
  ↓
ValidationError（Pydantic）
  → 呼び出し元でWARNINGログを出力する
  → ErrorHandler.handle_validation_error(e)でユーザー向けメッセージを生成
  → {"success": False, "message": エラーメッセージ} を返す
  ↓
ValueError（経路テーブル・固定運賃マスタ検索失敗）
  → 呼び出し元でERRORログを出力する
  → ErrorHandler.handle_calculation_error(e)でユーザー向けメッセージを生成
  → {"success": False, "message": エラーメッセージ} を返す
  ↓
FileNotFoundError（JSONファイル不存在）
  → WARNINGログを出力する（file_path付加情報を含める）
  → ErrorHandler.handle_fare_data_error(e)でユーザー向けメッセージを生成
  → (False, エラーメッセージ文字列) を返す
  ↓
Exception（その他のデータ読み込み失敗）
  → WARNINGログを出力する（file_path付加情報を含める）
  → ErrorHandler.handle_fare_data_error(e)でユーザー向けメッセージを生成
  → (False, エラーメッセージ文字列) を返す
  ↓
例外を raise せず、必ず戻り値としてエラー情報を返す
```

| エラー種別 | 対象例外 | 発生条件 | 戻り値 | ユーザー向けメッセージ |

## 6. 補足情報

### 6.1 実装上の注意点

1. **キャッシュの初期化タイミング**
   - `_load_fare_data()` はモジュールインポート時（アプリ起動時）に実行する
   - ツール関数内では再読み込みを行わない

2. **ファイル存在チェックの方法**
   - ファイル存在チェックは `os.path.exists()` による事前チェックで実装する
   - `open()` 実行時の `FileNotFoundError` 捕捉との併用は禁止し、事前チェックを優先する

3. **経路テーブルのキー形式**
   - キーは `"{departure}_{destination}"` 形式（正規化済み駅名を使用）
   - 逆方向（destination → departure）は別キーとして定義する必要がある

4. **エラー時の戻り値形式**
   - データ読み込み系メソッドのエラー時戻り値: `tuple[bool, str]` 型の `(False, エラーメッセージ文字列)`
   - 計算系メソッドのエラー時戻り値: `{"success": False, "message": エラーメッセージ文字列}` の dict 形式
   - 例外を raise せず、必ず戻り値としてエラー情報を返す

5. **申請者名・申請日の取得**
   - 申請者名・申請日は LLM がツールパラメータとして渡す値ではなく、ツール関数内部で invocation_state から取得する

6. **再試行不要**
   - TOOL-001は参照のみ・副作用なしのため、再試行は実施しない（実行制御方針 7.1節）

---

### 6.2 パフォーマンス考慮事項

1. **起動時静的読み込み**
   - 運賃データはアプリ起動時に一度だけ読み込むため、ツール呼び出しごとのI/Oコストは発生しない

2. **辞書検索の計算量**
   - 経路テーブル・固定運賃マスタはPython辞書（dict）で管理するため、検索はO(1)

---

### 6.3 セキュリティ考慮事項

1. **ファイルパスの固定**
   - 運賃データのファイルパスは定数として管理し、外部から変更できないようにする

2. **入力値のサニタイズ**
   - Pydanticモデルによるバリデーションで不正な入力値を早期に排除する

## 8. テスト観点

### 8.1 機能テスト
- 電車の経路テーブルに存在する出発駅・到着駅を入力し、正しい運賃と "route_fares.json" が返却されること
- バスの固定運賃マスタの値が正しく返却され、"fixed_fares.json" が返却されること
- タクシーの固定運賃マスタの値が正しく返却されること
- 飛行機の固定運賃マスタの値が正しく返却されること
- 駅名に「駅」が含まれる場合（例：「東京駅」）に正規化されて正しく検索されること
- 交通手段に別表記（例：「train」）が入力された場合に正規化されて正しく処理されること

### 8.2 異常系テスト
- 経路テーブルに存在しない経路を入力した場合に `{"success": False, "message": ...}` が返却されること（ValueError相当）
- 固定運賃マスタに存在しない交通手段を入力した場合に `{"success": False, "message": ...}` が返却されること（ValueError相当）
- 空文字列の出発地を入力した場合に `{"success": False, "message": ...}` が返却されること
- 空文字列の目的地を入力した場合に `{"success": False, "message": ...}` が返却されること
- 不正な交通手段（例：「自転車」）を入力した場合に `{"success": False, "message": ...}` が返却されること
- 不正な日付形式（例：「2026/04/19」）を入力した場合に `{"success": False, "message": ...}` が返却されること
- train_routes.jsonが存在しない場合に `(False, エラーメッセージ)` が返却されること（WARNINGログ出力）
- fixed_fares.jsonが存在しない場合に `(False, エラーメッセージ)` が返却されること（WARNINGログ出力）
- いずれのエラーケースでも例外が raise されないこと

### 8.3 性能テスト
- ツール呼び出し1回あたりの応答時間が100ms以内であること（キャッシュ済みデータ参照のため）
- 連続100回呼び出し時に応答時間の劣化がないこと

### 8.4 境界値テスト
- 経路テーブルのキーが完全一致する場合のみ検索成功すること（部分一致は不可）
- 駅名の「駅」除去後に空文字列になる場合にバリデーションエラーが発生すること
